# 思考框重复文字问题审查报告

## 1. 问题现象

在基线文档对某个模块进行扩写时，前端“思考”区域出现明显重复文本，例如：

- NowNow I I have have all all the the information information ...
- KeyKey elements elements from from the the source source document ...
- LetLet me me now now produce produce ...

从现象上看，重复粒度主要发生在词或短语级别，且重复内容处于同一段文本流中，不是两段完整思考卡片并排出现，也不是整段内容被渲染两次。

## 2. 审查结论

本次审查结论如下：

1. 这不是前端显示层把同一条思考文本重复渲染两次的问题。
2. 这更像是上游输出侧问题，具体有两种高概率来源：
   - 模型自身在 reasoning/thinking 流中重复吐词。
   - 上游供应商把 reasoning_content 按“累计文本”而不是“增量文本”返回，而当前接入层按 delta 语义直接透传，导致前端拼接后出现重复。
3. 因此前问题应归类为“模型/上游输出语义问题”，而不是“前端展示问题”。

更严格地说，这里最可能不是纯模型能力问题，也不属于前端 DOM/CSS/组件重复渲染问题，而是“上游 thinking 流输出 + 当前接入层未做累计片段去重”的兼容性问题。

## 3. 审查范围

本次按最短链路审查了以下路径：

1. 模型接入层如何解析 reasoning/thinking 流。
2. Agent 后端如何把思考内容转成事件。
3. SSE 层是否可能把同一 chunk 重复写入流。
4. 前端收到 thinking 事件后，是“覆盖”还是“追加”。
5. 前端思考组件是否存在二次渲染、重复合并或历史重建重复。

## 4. 关键证据

### 4.1 前端不是“重复显示同一条思考事件”

前端 Agent 流处理位于 frontend/src/views/chat/index.vue。

在 thinking 事件分支中，前端使用 event_id 作为键，将同一个思考事件保存在 _eventMap 中：

- 如果 event_id 首次出现，只创建一次 thinkingEvent，并 push 到 agentEventStream。
- 后续同 event_id 到来时，不会再次 push 新事件，而是直接执行 thinkingEvent.content += data.content。

这说明前端对同一思考块采用的是“单对象增量累加”，而不是“每个 chunk 新建一条卡片”。

同时，AgentStreamDisplay 组件会基于 eventStream 构建最终展示树，并且还额外做了 thinking/answer 重复隐藏逻辑，用来避免自然收口时思考内容与最终答案重复展示。

因此，从组件逻辑上没有发现“同一思考 chunk 被显示两次”的路径。

### 4.2 SSE 层没有把 thinking 全文重复回放成增量

后端 Agent SSE 转换位于 internal/handler/session/agent_stream_handler.go。

在 handleThought 中，写入 StreamEvent 时使用的是：

- Type = thinking
- Content = data.Content
- 注释明确写的是 no accumulation - frontend will accumulate

也就是说，SSE 层发送的是当前 chunk，而不是每次都把完整思考全文重新发送给前端。

这一点与前端的增量拼接语义是匹配的。

### 4.3 Agent 层发送的也是 thinking chunk，而不是累计全文

后端 Agent 事件生成位于 internal/agent/think.go。

在 streamThinkingToEventBus 中：

- 遇到 ResponseTypeThinking 时，发送 EventAgentThought。
- 发送的数据内容是 chunk.Content。
- 同一轮思考使用同一个 thinkingID，但内容仍然是按 chunk 发送。

因此，在 Agent 这一层也没有看到“每次把完整 reasoning 全量再发一遍”的逻辑。

### 4.4 真正未做去重的是模型接入层的 reasoning_content 透传

模型接入层位于 internal/models/chat/remote_api.go。

这里有一个关键事实：

1. processStreamDelta 接收到 reasoningContent 后，会直接构造成 ResponseTypeThinking 发出去。
2. 当前逻辑默认 reasoningContent 是增量片段。
3. 代码中没有保存“上一段 reasoningContent”，也没有做最长公共前缀去重或累计片段裁剪。

这意味着：

- 如果上游返回的是标准 delta thinking chunk，当前实现没有问题。
- 如果上游返回的是“累计到当前为止的 reasoning 文本”，当前实现会把累计文本再次 append 到前端，最终表现为词级或短语级重复。

这类现象与截图中的重复模式是一致的。

## 5. 为什么判断不是显示问题

如果是前端显示问题，通常会出现以下一种或多种特征：

1. 同一条完整思考卡片出现两份。
2. 同一段文本在 DOM 中被重复插入，形成整段复制。
3. 刷新或历史恢复后重复，但实时流中不重复。
4. 同一个 chunk 经多条分支重复进入 handleAgentChunk。

本次代码审查没有发现上述路径。相反，当前链路是：

1. 前端按 event_id 只维护一个 thinkingEvent。
2. SSE 层按 chunk 发送。
3. Agent 层按 chunk 转发。
4. 唯一缺失的防御点在模型接入层：没有对累计式 reasoning_content 做去重兼容。

因此，问题不应定性为“显示层重复渲染”。

## 6. 更精确的根因判断

结合现象和代码，更精确的判断如下：

### 场景 A：模型原始输出就重复

如果模型自身在 reasoning 流中吐出类似：

- Now
- Now I
- I have
- have all

或者本身就生成重复词序列，那么系统会原样展示，表现为当前截图效果。

### 场景 B：供应商返回累计 reasoning_content

如果某个供应商实际返回的是：

- 第 1 chunk: Now
- 第 2 chunk: Now I
- 第 3 chunk: Now I have

但当前接入层仍把它当成 delta 逐段 append，那么前端最终会看到：

- NowNow INow I have ...

当累计边界刚好落在词级时，就会出现与截图非常接近的“NowNow I I have have”模式。

### 最终判断

从代码链路来看，场景 B 的工程概率更高；从截图的词级重复形态看，场景 A 与场景 B 都可能，但两者都属于输出侧问题，不属于显示层问题。

## 7. 建议的验证方式

为了把“模型重复”与“累计片段重复”彻底区分，建议补以下观测：

1. 在 internal/models/chat/remote_api.go 的 reasoningContent 首次出现和后续每个 chunk 到达时，临时打印：chunk 长度、前 80 字符、与上一 chunk 的公共前缀长度。
2. 若发现后续 chunk 始终以前一 chunk 为前缀，则说明供应商返回的是累计文本。
3. 若相邻 chunk 本身并非前缀关系，但 chunk 内容里就有词级重复，则更接近模型原始输出重复。
4. 同时记录 provider、model、base_url，判断是否只在某一供应商或某一模型上出现。

## 8. 建议的修正方向

虽然本次任务是审查而不是直接改代码，但如果后续需要修复，建议按以下优先级处理：

### 方案一：在接入层对 reasoning_content 做累计片段去重

位置：internal/models/chat/remote_api.go

思路：

1. 为 reasoning_content 维护上一段已发送内容。
2. 若当前 chunk 以上一段为前缀，则仅发送新增尾部。
3. 若当前 chunk 与上一段部分重叠，则按最长公共前后缀裁剪后再发送。
4. 若无明显重叠，则按原样发送，避免误裁剪真实内容。

这是最稳妥的工程修复，能够兼容“增量式”和“累计式”两类供应商。

### 方案二：按 provider/model 做兼容开关

如果确认只有个别供应商返回累计 reasoning_content，可以仅对该 provider/model 开启去重逻辑，避免对所有模型统一处理带来误伤。

### 方案三：保留前端不动

前端当前语义是正确的：收到 delta 就 append。只要上游语义统一或接入层完成裁剪，前端无需为这个问题承担兜底复杂度。

## 9. 最终结论

本问题结论可以直接表述为：

思考框里的重复文字不是前端显示问题。现有前端和 SSE/Agent 渲染链路没有发现重复渲染缺陷。问题更可能出在上游思考流输出本身，或者模型接入层将累计式 reasoning_content 当作增量文本透传，导致前端正常追加后出现重复。

如果要落地修复，应优先在 internal/models/chat/remote_api.go 对 reasoning_content 增加累计片段去重兼容，而不是修改前端思考框展示逻辑。
