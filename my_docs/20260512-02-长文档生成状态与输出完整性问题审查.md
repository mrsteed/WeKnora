# 长设计文档迭代生成问题审查与设计方案

## 审查结论

本次仅修正问题审查与设计方案文档，不修改业务代码。

结合 `my_docs/test/16.md` 日志、前端 `AgentStreamDisplay.vue` 和后端 `session_agent_qa.go` 的实现，本次问题不是单一的前端展示问题，也不是只因为模型输出太短导致的普通截断。

日志显示本次请求进入 `knowledge_grounded_full_document` 路径，完整生成流程共规划 8 个章节，运行阶段也执行了 8 个章节，但最终被后端完整性校验判定为 `outline_or_section_incomplete`，因此持久化为：

- `completion_status: partial`
- `finish_reason: outline_or_section_incomplete`
- `failure_reason: outline_or_section_incomplete`
- `document_generation_status: continuing`
- `runtime_sections: 8`
- `content_len: 16927`
- `artifact_registered: true`
- `allow_indexing: false`

因此，用户看到的“输出不完整”更准确地说是：正文流式生成了较多内容，但最终文档结构未通过严格大纲完整性校验，系统把结果标记为未完成，并且最终内容中存在标题粘连、标题层级异常、章节内小节不匹配等格式问题。

后续修复不应把“格式修复异常”等同于“长文档生成失败”。长设计文档生成的首要目标是持续产出可用正文，格式问题应尽量自动修复；无法完全修复时，应保留已生成正文，并通过质量告警、待审标记、结构化问题列表提示用户，而不是直接把任务判定为失败或阻断文档产物。

## 总体设计原则

### 1. 文档生成优先

长文档任务的核心价值是生成完整、可迭代的设计文档。格式校验、标题修复、完整性检查都应服务于“提升文档质量”，而不是成为轻易中断任务的硬门槛。

建议将终态分为：

- `completed`：正文完整，结构和格式校验通过。
- `completed_with_warnings`：正文完整，但存在可人工复核的格式或结构告警。
- `continuing`：正文未覆盖全部计划章节，需要自动续写或手动继续。
- `blocked`：缺少关键输入、知识库无可用内容、模型调用失败等真正无法继续的问题。

格式修复失败原则上只应进入 `completed_with_warnings` 或 `needs_review`，不应直接进入失败态。

### 2. 结构由用户意图驱动

文档大纲应优先由大模型根据用户提示词规划，而不是使用固定内置模板强行套用。用户明确给出章节、标题、编号或文档类型时，应以用户提示词为最高优先级。

如果用户提示词缺少结构要求，系统应生成一个最基本的大纲建议，并在生成前将该大纲作为规划结果进入后续章节生成。基础大纲至少应包含：

- 项目背景与建设目标
- 需求分析与建设范围
- 总体架构设计
- 核心功能设计
- 数据与接口设计
- 安全与权限设计
- 部署实施方案
- 运维保障与验收建议

### 3. 内容详细但不编造

章节内容要尽量详细，但不能把未知事实写成确定事实。应采用分层表达：

- **已确认事实：** 来自用户提示词、知识库证据、已上传资料或上下文明确内容。
- **设计推导：** 基于已确认事实推导出的架构、流程、接口边界或实施建议，需要保持保守表述。
- **待补充项：** 知识库或用户输入缺失的信息，明确列为待确认、待补充或建议甲方/项目团队确认。

这样既能避免编造，又能让长设计文档保持完整、详细和可交付。

### 4. 上下文采用滚动摘要

后续章节不应携带完整已生成正文。完整正文会让 prompt 越来越长，挤压当前章节的证据与写作空间，也会把前面章节的格式瑕疵传播到后续章节。

建议将 `Completed content so far` 改为结构化滚动摘要，包括：

- 已完成章节标题与小节清单。
- 每章核心事实和设计结论。
- 已定义的术语、系统模块、接口边界和约束。
- 待补充问题与风险项。
- 最近一章的短摘要，而不是完整正文。

完整正文只保存在 artifact 或生成运行状态中，供最终拼装、修订和导出使用。

## 问题一：底部“生成中”状态缺少当前章节标题

### 现象

用户希望最下面的生成中状态显示当前正在生成的大概标题，例如：

- 正在生成第 7/8 章：AR眼镜智能作业系统
- 正在检索第 7/8 章：AR眼镜智能作业系统
- 正在整理最终文档

当前前端底部 loading 区域由 `AgentStreamDisplay.vue` 控制，模板中分别展示：

- `loadingStatusLabel`
- `loadingProgressLabel`

当前逻辑在回答正文开始流式输出后，`loadingStatusLabel` 优先返回通用的“生成中”，章节标题主要依赖 `loadingProgressLabel` 从最近一条 thinking/progress 文本中正则解析。

### 代码定位

- `frontend/src/views/chat/components/AgentStreamDisplay.vue`
  - 底部 loading 模板：约 402-410 行
  - `extractDocumentSectionProgress`：约 906 行
  - `activeDocumentSectionProgress` / `loadingProgressLabel`：约 1184、1267 行
  - `loadingStatusLabel`：约 1249 行

- `internal/application/service/session_agent_qa.go`
  - `fullDocumentProgressReporter.UpdateStage`：约 1027 行
  - 章节生成进度文案：约 4075、4094、4149 行
  - `fullDocumentProgressReporter.Close`：约 1070 行

### 根因分析

1. **前端主状态文案过于粗粒度**

   一旦 `hasAnswerStarted` 为真，底部主状态直接显示通用“生成中”，不再把章节编号、章节标题合并到主状态中。即使 `loadingProgressLabel` 有章节信息，视觉上也容易让用户只看到最下面的“生成中”。

2. **章节进度依赖文本正则解析，不够稳定**

   后端发送的是自然语言进度，例如：

   - `已检索到 6 条证据，正在生成第 7/8 章“AR眼镜智能作业系统”。`
   - `正在检索第 7/8 章“AR眼镜智能作业系统”的本地证据。`

   前端依赖正则从文本中提取章节进度。只要后端文案变化、引号变化、空格变化或 done/replace 事件覆盖最近 thinking 内容，底部就可能退化成通用状态。

3. **关闭进度事件可能覆盖最后一条非空进度**

   `fullDocumentProgressReporter.Close` 使用同一个 `eventID` 发送 `Done: true, Replace: true`，但内容为空、stage 为空。前端如果把它当作最新 thinking 事件，就可能丢失最后一个章节标题上下文。

### 修正建议

- 后端进度事件增加结构化 metadata，不只发送自然语言文本。
- 建议字段包括：`stage`、`section_current`、`section_total`、`section_title`、`query_current`、`query_total`、`progress_label`。
- 前端维护“最后一条非空章节进度”，即使收到空的 close 事件也不清空当前标题。
- 底部主状态建议直接展示组合文案：`正在生成第 7/8 章：AR眼镜智能作业系统`，而不是主标题只显示“生成中”。

## 问题二：`my_docs/test/16.md` 输出不完整的直接原因

### 运行事实

`my_docs/test/16.md` 日志末尾显示：

- 大纲标题：`北海电厂二期智慧电厂项目投标技术方案`
- 大纲章节数：8
- 运行章节数：8
- 最终状态：`partial`
- 结束原因：`outline_or_section_incomplete`
- 文档状态：`continuing`

这说明后端并不是在第 1 章或第 7 章硬中断，而是在全部章节生成后执行最终完整性校验时被降级为未完成。

从产品目标看，这类场景不应被理解为“文档生成失败”。更合理的处理是：正文已经生成，系统应保存并展示可用文档，同时标记结构质量问题，允许用户继续修订、自动修复或人工审阅。

### 代码定位

- `internal/application/service/session_agent_qa.go`
  - `runKnowledgeGroundedFullDocumentGenerationPath`：约 3961 行开始
  - 章节循环生成：约 4058-4168 行
  - 最终完整性校验：约 4177-4183 行
  - `completedFullDocumentIntegrityFailureReason`：约 2194 行
  - `renderedFullDocumentSubsectionsMatchOutline`：约 2145 行
  - `extractRenderedFullDocumentSubsections`：约 2100 行

- `internal/application/service/full_document_markdown_quality.go`
  - `normalizeGeneratedMarkdown`：约 28 行
  - `validateGeneratedSectionMarkdown`：约 106 行
  - `applyGeneratedSectionMarkdownQualityGate`：约 224 行
  - `repairGeneratedMarkdown`：约 274 行
  - `buildSectionMarkdownRepairMessages`：约 305 行

### 根因分析

1. **模型经常输出标题正文粘连**

   日志中可以看到修复请求内的原始片段存在类似问题：

   - `### 1.1 能源转型与数字化转型背景当前，全球能源格局...`
   - `### 7 .1 AR终端与语音交互信息检索**建设内容：** ...`

   这些格式会导致小节标题被解析成错误标题，例如把“当前，全球能源格局...”也当作标题的一部分。

2. **当前 normalize 只能修一部分格式问题**

   `normalizeGeneratedMarkdown` 能修复 `###3.1` 这类缺少空格的问题，也能修复部分列表和加粗标签间距问题，但对“计划标题后正文粘连”的场景处理不足。

   这类问题需要结合已知大纲小节标题进行确定性拆分，例如把：

   `### 7.1 AR终端与语音交互信息检索**建设内容：** ...`

   修成：

   `### 7.1 AR终端与语音交互信息检索`

   空行后再输出正文。

3. **LLM 修复失败后，缺少文档优先的降级策略**

   `applyGeneratedSectionMarkdownQualityGate` 在修复后仍可能返回 `qualityOK=false`。但 `runKnowledgeGroundedFullDocumentGenerationPath` 中只记录 warning，仍会把 `normalizedSectionContent` 写入 `finalContent`，随后继续生成下一章。

   这会造成两个副作用：一方面，系统没有中断，说明当前链路已经具备“继续生成”的意识；另一方面，不合格 Markdown 会污染最终文档，并且进入后续章节上下文。日志中后续 prompt 的 `Completed content so far` 出现了 `## # 1.1 ...` 这类明显异常结构，说明修复结果或归一化结果已经传播到下一章。

   更合理的策略不是因为格式修复异常而判定任务失败，而是把格式问题降级为质量告警，并保证进入后续上下文的是结构化摘要和已校正标题，而不是带有异常 Markdown 的完整正文。

4. **最终完整性校验按大纲严格比对小节，异常标题会导致整篇判为 incomplete**

   `completedFullDocumentIntegrityFailureReason` 会检查：

   - 大纲是否合法。
   - 已完成章节数量是否等于大纲章节数量。
   - 最终渲染出的 H1/H2 是否完整。
   - 最终渲染出的小节是否与规划大纲一致。

   当标题粘连、标题层级异常或小节标题被 LLM 修复改写后，`renderedFullDocumentSubsectionsMatchOutline` 无法匹配大纲，于是最终返回 `outline_or_section_incomplete`。

   这类校验适合作为质量信号，但不适合作为唯一的任务成功/失败判定。对于长设计文档，正文覆盖章节、事实不编造、可继续迭代，应比 Markdown 细节完全匹配更重要。

5. **持久化摘要中的 outline 元数据也不完整**

   日志末尾的 persistence summary 显示 `outline_sections: 0`，而生成完成摘要里是 `outline_sections: 8`。这说明生成侧已有完整大纲，但持久化/索引侧没有完整拿到或解析到大纲元数据。

   这会影响后续 artifact 展示、续写判断、索引调度和问题排查。

### 修正建议

- 在 LLM 修复前增加确定性 Markdown 修复器，优先依据当前章节的规划小节标题处理标题粘连、编号空格异常、H3/H4 层级异常。
- 当 LLM 修复后仍 `qualityOK=false` 时，不直接判定任务失败。可选策略：
   - 回退到确定性修复后的内容。
   - 保留正文并标记 `quality_issues`。
   - 将该章节标记为需审查，但确保标题结构尽量满足大纲。
   - 对无法修复的小节标题降级为普通段落或列表说明，保留章节正文。
- 完整文档收尾前增加一次整篇级确定性结构修复，再执行完整性校验。
- 校验失败时，如果章节正文已覆盖主要大纲，应保存 artifact 并展示 `completed_with_warnings` 或 `needs_review`，而不是把任务打成失败。
- 持久化 artifact 时补齐大纲 metadata，保证生成摘要与持久化摘要中的 `outline_sections` 一致。

## 问题三：`max_completion_tokens: 2048` 的来源

### 结论

本次日志中出现的 `max_completion_tokens: 2048` 主要来自配置 fallback 与长文档预算逻辑，不是前端和模型服务商临时协商出来的固定值。

系统确实存在“预算协商”能力，但它只有在模型能力元数据可用时才会触发。当前日志中没有看到 `[DocumentBudget] source=negotiated` 或类似协商成功记录，并且模型列表中 `deepseek-v4-pro` 的 `extra_config` 为空，因此本次更像是走了 fallback 预算。

### 配置来源

1. 全局会话摘要配置：

   `config/config.yaml` 中：

   - `conversation.summary.max_completion_tokens: 2048`

2. 内置智能体配置：

   `config/builtin_agents.yaml` 中：

   - `builtin-quick-answer.max_completion_tokens: 2048`
   - `builtin-smart-reasoning.max_completion_tokens: 2048`

3. 前端设置默认值：

   - `frontend/src/views/settings/AgentSettings.vue` 默认值 2048
   - `frontend/src/views/agent/AgentEditorModal.vue` 默认值 2048

4. 长文档 fallback 预算：

   `fallbackFullDocumentSectionMaxCompletionTokens` 会优先使用 `cfg.Conversation.Summary.MaxCompletionTokens`，低于 2048 时提升到 `fullDocumentSectionMinCompletionTokens`。

   因此在默认配置下，长文档每章生成预算就是 2048。

### 协商逻辑

`session_agent_qa.go` 中存在 `negotiateDocumentGenerationBudget`：

- 如果模型能力中提供了 `context_window_tokens` 或 `max_output_tokens`，且请求是 `full_document`，就会让模型用一次短调用返回预算 JSON。
- 协商返回字段包括：
  - `outline_max_completion_tokens`
  - `section_max_completion_tokens`
  - `continuation_max_completion_tokens`
  - `outline_evidence_top_k`
  - `section_evidence_top_k`
  - `continuation_evidence_top_k`
  - `section_call_timeout_seconds`

但本次未看到协商成功日志，且模型配置没有能力元数据，因此不能认为本次的 2048 是协商结果。

### 修正建议

- 在模型配置中补齐长上下文与最大输出能力，例如 `context_window_tokens`、`max_output_tokens`。
- 在日志和 SSE metadata 中显式透出最终采用的预算来源：`fallback`、`capability`、`negotiated`、`runtime_adjusted`。
- 对 `full_document` 场景不要直接复用普通问答的 2048，建议按模型能力提升到 4096 或 8192。
- Markdown 修复调用可以继续使用较小预算，但章节正文生成需要单独预算，不应和普通摘要/问答共用一个默认上限。

## 问题四：章节内容不够符合设计文档要求、不够详细

### 现象

用户希望每个章节更像正式设计文档或投标技术方案，越详细越好。当前输出虽然包含章节和小节，但内容更偏“概述型方案”，对技术架构、数据流程、接口边界、部署方式、权限安全、实施步骤、验收指标等设计文档要素覆盖不足。

### 根因分析

1. **章节结构没有充分由用户提示词驱动**

   当前完整文档路径会生成大纲，但大纲约束更偏“章节数量和 Markdown 格式”，没有充分把用户提示词中的文档类型、目标读者、章节要求、详细程度要求转化为结构化设计约束。

   当用户明确要求“投标技术方案”“业务部门和技术专家均需要详细阐述”时，大纲应体现业务价值、技术架构、功能设计、实施方案、运维保障和验收口径，而不是只生成概述型章节。

   当用户没有给出明确大纲时，系统应先给出基础大纲建议，并以该建议作为后续章节生成契约。

2. **章节提示词更偏通用投标文风，不是严格设计文档模板**

   当前 `buildKnowledgeGroundedFullDocumentSectionMessages` 主要要求：

   - 使用本地知识事实。
   - 不编造项目事实。
   - 输出当前章节正文。
   - 使用正式投标方案文风。
   - 标题格式必须正确。

   但它没有强制每节包含设计文档常见结构，例如：

   - 建设目标
   - 业务范围
   - 功能架构
   - 数据流程
   - 系统集成
   - 部署拓扑
   - 安全策略
   - 运维保障
   - 实施步骤
   - 交付物
   - 验收标准
   - 风险与边界

3. **知识库证据粒度偏摘要，限制了细节展开**

   每个证据 chunk 只截取约 1200 字符，章节证据默认 topK 为 6，后续还可能因首 token 慢被 runtime feedback 下调到 5 或 4。对于“越详细越好”的长文档，这会限制模型能拿到的事实材料。

4. **严格“不编造事实”与“越详细越好”之间缺少分层表达策略**

   当前策略是证据不足就提示“本地知识不足”。这对事实准确性是安全的，但会让实施方案、运维保障、质量控制等章节偏空。

   更合理的写法应区分：

   - 已有知识库明确支持的事实。
   - 可作为通用方法论的建议框架。
   - 需要甲方或项目团队补充确认的待定项。

5. **章节上下文包含完整已生成正文，后续章节 prompt 越来越长**

   每生成一章，后续章节都会带上 `Completed content so far`。当前每章输出预算只有 2048，提示上下文逐步膨胀，会进一步压缩模型用于当前章节详细展开的空间。

   这也是长文档迭代生成的关键瓶颈。完整正文不仅消耗上下文，还会把前序章节中的格式问题、异常标题、重复内容继续传给模型，导致后续章节越来越短、越来越保守，甚至发生结构污染。

   建议将完整正文从章节 prompt 中移除，替换为“已完成内容结构化摘要”。摘要可由模型或确定性逻辑在每章结束后更新，但必须保留完整 artifact 作为最终拼接来源。

### 修正建议

- 为 `full_document` 增加“用户意图驱动的大纲规划”步骤。优先按用户提示词生成章节结构；用户提示词缺少结构时，输出基础大纲建议。
- 为 `full_document` 增加“设计文档深度模板”，按章节类型生成能力组、实施步骤、接口边界、数据流、部署与安全、验收指标等内容。
- 按章节类型配置不同写作框架，例如：
  - 数据湖章节：数据源、采集链路、数据治理、存储模型、权限、服务目录、API、质量规则。
  - 智慧运行章节：监盘对象、参数接入、预警规则、模型推理、联动流程、闭环处置。
  - 安防章节：人员/车辆/访客流程、审批、核验、追溯、异常告警。
  - AR 作业章节：终端选型、语音识别、信息检索、AR 投射、远程协同、作业留痕。
- 证据检索在“详细方案”模式下提升 topK 和 chunk 长度，并避免因首 token 慢过早降低 evidence topK。
- 将 `Completed content so far` 改成结构化摘要，而不是完整正文，减少上下文挤压。
- 对证据不足章节输出“待补充项清单 + 建议方案框架”，既避免编造，又能让投标方案可读、完整。

## 推荐生成流程

### 阶段一：理解用户意图并规划文档结构

系统先从用户提示词中提取文档目标、目标读者、文档类型、明确章节、详细程度、事实约束和输出格式约束。

如果用户已经给出章节结构，应以用户结构为准，只做轻量规范化，例如补齐编号、去重、修复明显标题格式问题。

如果用户没有给出章节结构，应让大模型生成基础大纲建议。大纲建议不应直接进入最终正文，而应作为后续章节生成的结构契约。

### 阶段二：构建证据包与待补充清单

每个章节生成前，系统按章节标题、用户目标和已知术语检索知识库，形成章节级证据包。

证据包应区分：

- 可直接引用的事实。
- 可用于设计推导的事实。
- 缺失但影响方案完整性的事实。

缺失事实不应由模型编造。系统应在章节中以“待确认事项”“建议补充资料”或“实施前置条件”的方式表达。

### 阶段三：逐章节生成详细内容

章节生成应使用“章节合同 + 证据包 + 已完成摘要”，而不是完整历史正文。

章节合同包括：

- 当前章节编号和标题。
- 当前章节要覆盖的小节。
- 本章必须使用的事实边界。
- 允许输出的推导型内容范围。
- 需要明确标注的待补充项。

章节正文应尽量详细，优先使用项目符号描述能力组、实施步骤、数据流、接口边界、安全策略、验收口径和价值点。

### 阶段四：格式修复与质量标记

每章生成后先执行确定性 Markdown 修复，再视情况调用 LLM 修复。

格式修复应遵循：

- 能修就修。
- 修不好也保留正文。
- 不因格式修复异常把任务标为失败。
- 将问题写入 `quality_issues`，供前端展示和后续修订使用。

只有在章节正文为空、模型调用失败、知识库完全无可用证据且用户又要求严格基于知识库时，才考虑阻断任务。

### 阶段五：滚动摘要更新

每章完成后更新结构化摘要，摘要用于后续章节 prompt。

摘要建议包含：

- 已完成章节及小节。
- 每章核心事实。
- 关键系统模块和术语。
- 已确定的设计约束。
- 待补充问题。
- 已发生的质量告警。

这样既能保持跨章节一致性，又能避免 prompt 随完整正文线性膨胀。

### 阶段六：最终拼装与宽容校验

最终文档应从 artifact 中保存的章节正文拼装，而不是从 prompt 摘要反推。

完整性校验应分级处理：

- 章节缺失：进入 `continuing`，触发续写。
- 正文完整但标题格式存在告警：进入 `completed_with_warnings` 或 `needs_review`。
- 事实缺失：保留正文并展示待补充清单。
- 模型或系统错误导致无正文：进入失败或 blocked。

这能保证“生成文档依然是最重要的事情”，同时保留格式正确性和质量可追踪性。

## 建议修复优先级

### P0：保证文档产物优先可用

- 调整终态策略：格式修复异常不直接标记失败，正文可用时保存 artifact。
- 增加确定性标题修复，修复 `### 7 .1`、标题正文粘连、H1/H2 污染等问题。
- LLM 修复失败后保留正文并记录 `quality_issues`。
- 最终完成前执行整篇结构校验和修复；校验不完全通过时，优先进入 `completed_with_warnings` 或 `needs_review`。
- 修复持久化侧 `outline_sections: 0` 的 metadata 丢失问题。

### P1：改善生成状态展示

- 后端发送结构化章节进度 metadata。
- 前端底部 loading 主文案直接显示当前章节标题。
- 前端缓存最后一条非空章节进度，避免 close 事件清空展示。

### P1：按用户提示词规划结构并提升详细度

- 大纲由模型根据用户提示词生成；提示词缺少结构时，生成基础大纲建议。
- 增加设计文档写作模板，覆盖功能、数据、接口、安全、部署、实施、验收和风险。
- 长文档章节预算从普通问答的 2048 中解耦。
- 补齐模型能力元数据，让预算协商真正生效。
- 提高详细模式下的 evidence topK、chunk 字符上限和章节输出 token 上限。

### P1：解决章节上下文膨胀

- 将章节 prompt 中的 `Completed content so far` 从完整正文改为结构化滚动摘要。
- 完整正文只写入 artifact，不再作为后续章节的直接上下文。
- 摘要中保留事实、术语、设计约束和待补充项，保证跨章节一致性。
- 避免前序章节的异常 Markdown 继续污染后续章节。

### P2：增强可观测性

- 日志中输出最终预算来源和每章采用的 token/evidence 配置。
- completion summary 与 persistence summary 对齐大纲章节数、artifact_id、run_id、quality_issues。
- 前端在 partial 状态下展示明确原因，例如“文档结构校验未通过：小节标题不匹配”，而不是只显示泛化“部分完成”。

## 本次不修改代码的后续落地建议

如果进入代码修复阶段，建议按以下顺序实施：

1. 先调整任务终态：格式修复异常不直接判失败，正文可用时保存 artifact 并进入质量告警态。
2. 再补 progress metadata 与前端底部标题展示。
3. 然后补确定性 Markdown 修复、整篇结构修复和大纲 metadata 持久化。
4. 接着把章节上下文改为结构化滚动摘要，解决 prompt 越来越长的问题。
5. 最后增加用户意图驱动的大纲规划、设计文档深度模板和详细模式检索策略。
