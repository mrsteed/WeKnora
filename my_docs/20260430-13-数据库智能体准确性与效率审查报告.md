# 数据库智能体准确性与效率审查报告

> 日期：2026-04-30  
> 审查对象：当前 `database-analysis` 数据库智能体实现  
> 审查重点：自然语言到 SQL 的准确性、查询与上下文效率、运行时资源开销  
> 验证命令：`go test ./internal/agent/tools ./internal/application/service -run 'ExternalDatabase|StructuredQuery|SchemaRegistry|DatabaseSchema|DatabaseTool'`

## 一、结论摘要

当前数据库智能体的主链路设计方向是成立的：

1. `database-analysis` 预设只开放数据库相关工具，默认绑定兼容的 Database KB。
2. `external_database_schema` 与 `external_database_query` 分离，避免复用系统主库 `database_query` 造成职责混淆。
3. Database KB、DataSource、Schema Snapshot、StructuredQueryService 的边界清晰。
4. SQL 执行链路具备只读、单语句、限表、限列、敏感列拦截、超时、审计等基础保护。
5. 当前相关测试通过，说明基础工具注册、Schema 获取、查询校验和审计路径处于可用状态。

但在准确性和效率上仍有优化空间，尤其是以下四类问题：

1. SQL Guard 与提示词约束不完全一致，会造成可回答问题被先拒绝，增加无效轮次。
2. 字段权限校验覆盖面不足，目前主要覆盖 SELECT / WHERE 字段，对 JOIN ON、GROUP BY、HAVING、ORDER BY、窗口表达式中的字段缺少统一校验。
3. schema 与查询结果为了准确性保留得较完整，但缺少自适应预算，面对大库、宽表、大结果集时容易放大模型上下文、SSE 载荷和历史上下文成本。
4. 外部数据库工具结果未纳入历史检索结果裁剪，跨轮对话可能复用过期 schema / 旧查询结果，也会持续扩大上下文。

建议优先做“自适应压缩 + 全字段引用校验 + 历史结果摘要化”，不要简单回退到固定 schema 摘要或固定查询结果截断。此前代码已经经历过“压缩过度导致选错表、结论丢失”的问题，当前更适合做按规模、按用途、按阶段的分层输出。

## 二、当前实现概览

### 2.1 预设与提示词

相关文件：

- `config/agent_type_presets.yaml`
- `config/prompt_templates/agent_system_prompt.yaml`

当前 `database-analysis` 预设：

1. `system_prompt_id` 使用 `database_analyst`。
2. `temperature` 为 `0.2`，有利于 SQL 生成稳定性。
3. `max_iterations` 为 `30`，给复杂数据库分析留足多轮修正空间。
4. `allowed_tools` 包含 `thinking`、`todo_write`、`external_database_schema`、`external_database_query`、`final_answer`。
5. `kb_selection_mode` 为 `all`，并通过 `kb_filter.all_of: [database]` 约束只进入 Database KB。

这套配置的准确性基础较好：工具面较窄，减少了模型在 RAG、Wiki、外部数据库之间误路由的概率。

### 2.2 工具实现

相关文件：

- `internal/agent/tools/external_database_schema.go`
- `internal/agent/tools/external_database_query.go`
- `internal/agent/tools/registry.go`

当前 schema 工具行为：

1. 要求传入 `knowledge_base_id`。
2. 校验 KB 必须在当前 Agent scope 内。
3. 调用 `BuildPromptSchema` 给模型输出完整 prompt schema。
4. 再调用 `GetDatabaseSchema` 组装前端 `Data`。
5. `Data["tables"]` 已做表数量、字段数量、索引数量和注释长度预算，避免完整 schema 原样进入前端结构化载荷。

当前 query 工具行为：

1. 要求 `knowledge_base_id`、`sql`、`purpose`。
2. 校验 KB 必须在当前 Agent scope 内。
3. 调用 `StructuredQueryService.ExecuteQuery`。
4. 将完整查询结果逐行逐列写入 `Output`，并把结构化结果写入 `Data`。
5. `ToolRegistry.shouldSkipOutputTruncation` 对 `external_database_query` 返回 `true`，因此查询结果不会走统一工具输出截断。

这说明当前实现是在“准确性优先”的状态：schema 与 query 输出给模型的信息较完整，能降低模型因结果缺失而误答或反复补查的概率。

### 2.3 Schema 与查询服务

相关文件：

- `internal/application/service/database_schema.go`
- `internal/application/service/structured_query.go`
- `internal/application/service/sql_guard.go`
- `internal/databaseconnector/mysql/connector.go`
- `internal/databaseconnector/postgres/connector.go`

当前链路：

1. 创建 Database KB 时同步创建 DataSource。
2. 创建成功后立即 `RefreshSchema`，保存 schema snapshot。
3. 查询时从 schema snapshot 解析当前可查询表字段。
4. SQL Guard 校验 SQL 只读、单语句、显式 LIMIT、表白名单、字段白名单、敏感字段和 denylist。
5. MySQL / PostgreSQL connector 复用 `*sql.DB` 连接池，避免每次查询重新打开连接。
6. 查询成功、失败、拒绝都会进入审计日志。

这条链路的工程完整度较高，已经不是纯原型状态。

## 三、准确性审查

### 3.1 已做得较好的点

#### 3.1.1 工具边界清晰

`external_database_schema` 和 `external_database_query` 独立于系统主库 `database_query`，避免外部业务库 SQL 与 WeKnora 内部库 SQL 混用。这个边界对准确性和安全性都很关键。

#### 3.1.2 Scope 约束有效

两个外部数据库工具都会检查 `searchTargets.ContainsKB(kbID)`，模型不能直接构造任意 KB ID 去探测不在当前智能体范围内的数据库。

#### 3.1.3 Database KB 能力过滤已经贯通

后端 `registerTools` 会基于 `SearchTargets` 判断是否存在 Database KB；运行时 `KBSelectionMode=all` 也会按工具能力过滤知识库。前端 `tool-capabilities.ts` 同步声明了 `external_database_schema/query -> database` 能力要求。

#### 3.1.4 查询前置校验较完整

`sql_guard.go` 已具备：

1. 空 SQL 拦截。
2. 单语句拦截。
3. SELECT-only 拦截。
4. 危险函数和注入风险检查。
5. 显式 LIMIT 要求。
6. LIMIT 上限检查。
7. 表必须来自 schema snapshot。
8. SELECT / WHERE 字段必须来自可见列。
9. 敏感列和 denylist 拦截。

这些能力可以防住大部分“模型生成不受控 SQL”的基础风险。

### 3.2 仍建议优化的准确性问题

#### 3.2.1 P1：提示词允许“聚合或 LIMIT”，SQL Guard 实际强制所有查询必须 LIMIT

现象：

`external_database_query` 工具描述写的是“Always include aggregation or an explicit LIMIT clause”，数据库智能体提示词也强调优先聚合和采样。但 `sqlGuard.Validate` 当前无条件执行：没有显式 `LIMIT \d+` 就返回 `ErrStructuredQueryLimitRequired`。

影响：

1. `SELECT COUNT(*) FROM orders`、`SELECT SUM(amount) FROM orders WHERE ...` 这类天然只返回 1 行的查询会被拒绝。
2. 模型收到拒绝后通常会重试 `SELECT COUNT(*) ... LIMIT 10`，结果最终可用，但多消耗一次工具调用和一次模型轮次。
3. 对复杂问题，多个聚合子查询都可能经历“先拒绝、再补 LIMIT”的过程，效率和稳定性都会下降。

建议：

1. 将规则改为“原始行查询必须 LIMIT；无 GROUP BY 的全局聚合可免 LIMIT”。
2. 对 `GROUP BY` 聚合继续要求 LIMIT，因为分组结果可能很大。
3. SQL Guard 使用 AST 判断是否为纯聚合查询，不建议用字符串正则判断。
4. 同步更新工具描述和提示词，保持“模型看到的规则”和“后端执行规则”一致。

#### 3.2.2 P0：字段权限校验只覆盖 SELECT / WHERE 抽取结果，未覆盖所有字段引用位置

现象：

`utils.ValidateSQL` 内部的 AST validator 已经会遍历 SELECT、WHERE、JOIN、GROUP BY、HAVING、ORDER BY 等节点做通用安全校验，例如危险函数、系统列、锁语句、复合查询等。但 `sqlGuard.Validate` 用来做“字段白名单、敏感列、denylist”判断的 `referencedFields` 仍来自 `parseResult.SelectFields` 和 `parseResult.WhereFields`。也就是说，安全遍历有，但权限字段收集不完整。

从当前 `utils.SQLParseResult` 结构看，它只显式记录 `SelectFields` 和 `WhereFields`，没有记录 JOIN ON、GROUP BY、HAVING、ORDER BY、窗口表达式、CTE 内部 SELECT 等位置的字段引用。

影响：

1. 敏感字段可能通过 `ORDER BY sensitive_col`、`GROUP BY sensitive_col`、`HAVING` 条件、JOIN 条件被引用，形成侧信道推断风险。
2. 模型可能使用 schema 中不存在或不可见的字段作为排序、分组或连接条件，SQL 在数据库执行阶段失败，降低准确性并增加重试。
3. 对多表 join 问题，JOIN ON 字段没有强校验会让错误连接条件更晚暴露。

建议：

1. 基于 `pg_query` AST 做统一 ColumnRef 收集，覆盖 SELECT、WHERE、JOIN ON、GROUP BY、HAVING、ORDER BY、WINDOW、子查询和 CTE。
2. SQL Guard 不只校验 `SelectFields + WhereFields`，而是校验“全语句所有字段引用”。
3. 对字段引用保留 table alias 到真实 table 的映射，避免同名字段在多表场景下被错误放行。
4. 对无法归属表的非限定字段，在单表场景可按当前表解析；在多表同名字段场景应要求模型补充表别名。
5. 增加测试：敏感字段出现在 ORDER BY / GROUP BY / HAVING / JOIN ON / WINDOW / CTE 时必须被拒绝。

#### 3.2.3 P1：MySQL SQL 仍主要走 PostgreSQL parser，方言兼容性有限

现象：

当前 SQL 解析和校验核心使用 `pg_query`。MySQL 分支主要做了反引号转双引号，能支持简单 MySQL 查询，但 MySQL 特有语法不一定能通过 PostgreSQL parser。

影响：

1. 用户问“最近 7 天”“按月统计”时，模型可能生成 MySQL 风格的 `DATE_SUB(NOW(), INTERVAL 7 DAY)`、`DATE_FORMAT` 等表达式，校验阶段可能被拒绝。
2. 提示词要求“Follow the SQL dialect implied by the schema/database type”，但后端校验器并非真正 dialect-aware，会造成模型按要求生成方言 SQL 却被拒绝。
3. 每次拒绝都会增加重试轮次，且模型可能反复在 MySQL / PostgreSQL 方言之间摇摆。

建议：

1. 短期：在数据库 schema 输出中明确“当前校验器支持的 SQL 子集”，提示模型优先使用兼容表达式。
2. 中期：引入或封装 MySQL AST parser，至少覆盖常见日期函数、字符串函数、LIMIT/OFFSET、标识符引用。
3. 为 MySQL 常见分析 SQL 增加回归测试，尤其是日期窗口、分组统计、字符串匹配和分页。

#### 3.2.4 P1：Join Hints 目前是启发式推断，可能误导模型

现象：

`inferJoinHints` 基于主键、`id`、`xxx_id`、单复数表名做启发式关联推断。MySQL / PostgreSQL schema discovery 当前主要拉取表、字段、主键、索引，没有拉取外键关系。

影响：

1. 没有外键的业务库里，`xxx_id` 命名不一定代表真实关系。
2. 多租户、组织树、层级表等场景中，启发式 join 可能把不同语义的 `id` / `parent_id` / `org_id` 错连。
3. 模型看到 `Join Hints` 可能把它当成强事实，生成看似合理但业务语义错误的 SQL。

建议：

1. connector schema discovery 增加外键抓取，优先输出真实 FK。
2. 启发式结果改名为 `Possible join hints`，并在输出中说明“需结合字段注释和业务语义确认”。
3. 对没有真实 FK 的库，允许管理员在 datasource config 里维护业务 join 规则。

#### 3.2.5 P1：Schema Snapshot 可能过期，当前缺少自动刷新策略

现象：

创建 Database KB 时会立即刷新 schema；DataSource handler 也提供手动 `refresh-schema`。但 `SchemaRefreshCron` 目前只是配置字段，未看到实际调度器使用。

影响：

1. 外部数据库表结构变更后，智能体仍基于旧 schema 生成 SQL。
2. 新字段、新表不可见，旧字段被删除则查询失败。
3. 模型会把失败归因于 SQL 写法问题，可能多轮重试而不是刷新 schema。

建议：

1. schema 工具输出 `refreshed_at` 和 `schema_hash`，让模型和前端都能感知新旧程度。
2. 查询失败若命中“unknown table / unknown column”类数据库错误，可触发一次受控 schema refresh，再提示模型重查 schema。
3. 落地 `SchemaRefreshCron` 调度，或至少在 datasource 更新后自动 refresh。
4. `DataSourceService.UpdateDataSource` 在 database datasource 的连接配置、白名单、字段黑名单变化后，应自动刷新 schema 或标记 schema stale。
5. `DataSourceService.DeleteDataSource` 删除 database datasource 后，应同步清理或失效 schema snapshot，避免 UI 或工具继续展示旧结构。

## 四、效率审查

### 4.1 已做得较好的点

#### 4.1.1 连接池复用已经存在

MySQL 和 PostgreSQL connector 都按 DSN 缓存 `*sql.DB`，并设置了 `MaxOpenConns=5`、`MaxIdleConns=2`、`ConnMaxLifetime=30min`。这比每次查询重新建连效率高很多。

#### 4.1.2 前端结构化 schema 载荷已有预算

`external_database_schema` 的 `Data["tables"]` 当前只保留前 8 张表、每表前 8 列、前 4 个索引和短注释。这能避免 schema 完整结构在 SSE / 前端渲染层重复放大。

#### 4.1.3 最终回答再综合阶段已做工具结果摘要

`finalize.go` 对 `external_database_schema` 和 `external_database_query` 有专门摘要逻辑：schema 只保留库名、schema、前若干表、join hints、sample queries；query 只保留列、行数、耗时和前 3 行样本。这个设计能降低最终回答二次综合时的上下文压力。

### 4.2 仍建议优化的效率问题

#### 4.2.1 P1：schema 工具重复读取 schema snapshot

现象：

`ExternalDatabaseSchemaTool.Execute` 先调用 `BuildPromptSchema`，而 `BuildPromptSchema` 内部会 `GetDatabaseSchema`；随后工具又调用一次 `GetDatabaseSchema` 来组装 `Data`。

影响：

1. 每次 schema 工具调用会重复读取、解析 snapshot。
2. 大 schema 下会增加数据库读取、JSON 反序列化和字符串拼接成本。
3. 多轮分析中，模型经常先看全库 schema，再按表补查 schema，这个重复成本会累积。

建议：

1. 提供 `BuildPromptSchemaFromSchema(schema, selectedTables)` 或让 `GetDatabaseSchema` 只调用一次。
2. 工具内部先拿 schema，再同时生成 prompt output 与 frontend data。
3. 若保留 service API，可以新增 `GetPromptSchema(ctx, kbID, selectedTables)` 返回 schema 与 prompt 两份结果，避免二次查询。

#### 4.2.2 P1：schema 输出缺少自适应预算，大库会撑大模型上下文

现象：

当前 `Output` 给模型的是完整 `BuildPromptSchema`。这对小库准确性好，但对几十张表、几百上千列的库，会在第一轮就占用大量上下文。

影响：

1. 首轮 schema 工具输出过大，降低后续查询结果和推理空间。
2. 模型在大量表字段中选择相关表时噪声增大，反而可能降低准确性。
3. 大 schema 在历史上下文里保留，会放大多轮对话成本。

建议：

采用自适应 schema 输出，不建议简单固定摘要：

1. 小库：继续输出完整 schema。
2. 中大型库：首轮输出表目录、表注释、主键、行数估计、少量代表列、真实外键/业务 join 规则。
3. 支持模型按 `tables` 参数二次请求指定表的完整字段。
4. 在 runtime prompt 中指导模型：先看 catalog，再按相关表精查 schema。
5. 增加“表名/字段名关键词匹配”的 schema narrowing 能力，帮助模型从自然语言问题定位候选表。

#### 4.2.3 P1：query 工具跳过统一截断，宽表或大文本字段会放大上下文

现象：

`ToolRegistry.shouldSkipOutputTruncation` 对 `external_database_query` 返回 `true`，查询结果完整进入下一轮模型上下文。`MaxRows` 默认兜底为 500，单元格没有文本长度预算。

影响：

1. 宽表 + 多行 + 长文本字段会快速撑大上下文。
2. 查询结果还会通过事件流传给前端，`Data.rows` 也包含完整结构化结果。
3. 一旦模型为了探索执行多次查询，历史上下文会迅速膨胀。

建议：

不要恢复粗暴截断，而是做结果感知预算：

1. 小结果完整返回。
2. 大结果按 `row_count * column_count * cell_length` 估算预算。
3. 对长文本单元格做单元格级截断，并标记 `cell_truncated=true`。
4. LLM 可见 `Output` 使用“完整统计 + 样本行 + 被截断说明”；前端 `Data` 可保留更多，但也应设置上限。
5. 对用户确实要求明细导出时，走单独导出/下载链路，而不是把全部明细塞进模型上下文。

#### 4.2.4 P1：外部数据库工具结果未纳入历史裁剪，跨轮对话可能复用旧数据

现象：

`observe.go` 的 `kbToolNames` 会裁剪历史中的 KB 检索工具结果，但当前未包含 `external_database_schema` 和 `external_database_query`。

影响：

1. 多轮对话中，旧 schema 和旧查询结果会继续进入模型上下文。
2. 如果数据库数据或 schema 发生变化，模型可能复用旧结果，影响准确性。
3. 大查询结果会在历史中持续占用 tokens，影响后续问题效率。

建议：

1. 将 `external_database_schema` 和 `external_database_query` 纳入历史裁剪。
2. 裁剪时不要只放“结果已省略”，建议保留紧凑摘要：库名、schema_hash、refreshed_at、SQL 摘要、行数、是否截断、前 1-3 行样本。
3. 对新一轮数据库问题，提示模型重新调用 schema/query，避免默认复用旧数据。

#### 4.2.5 P2：connector 内部 Query 二次创建 timeout context，成功路径未释放 cancel

现象：

`StructuredQueryService.ExecuteQuery` 已经基于 `validated.Timeout` 创建 `queryCtx`。MySQL / PostgreSQL connector 的 `Query` 内部又调用 `withDefaultTimeout` 创建第二层 context。成功返回 `rows` 时，内部 `cancel` 没有机会被调用；只有查询失败时会调用。

影响：

1. 每次成功查询都会留下一个等待超时自然结束的 timer/context。
2. 高并发查询下会产生不必要的资源占用。
3. timeout 逻辑重复，排查查询取消行为时更复杂。

建议：

1. connector `Query` 直接使用传入 ctx，超时由 StructuredQueryService 统一负责。
2. 若必须在 connector 层兜底超时，需要调整接口让 cancel 能绑定到 rows.Close，但当前返回 `*sql.Rows` 不方便封装。
3. 保留 Validate / DiscoverSchema 的 connector 内部短超时即可。

#### 4.2.6 P2：连接池缓存无淘汰，配置变更后旧 DSN 连接可能长期存在

现象：

MySQL / PostgreSQL connector 使用 DSN 字符串作为 `clients` map key 缓存 `*sql.DB`，但未看到淘汰或关闭机制。

影响：

1. DataSource 频繁修改密码、host、database 时，旧 DSN 对应连接池会留在内存中。
2. 多租户大量 Database KB 场景下，map 可能持续增长。

建议：

1. datasource 更新或删除时通知 connector 关闭旧 DSN。
2. 为 connector 增加 `Close()` 或 LRU/TTL 连接池管理。
3. 缓存 key 可以改为 datasource ID + config hash，方便生命周期管理和审计定位。

#### 4.2.7 P2：前端数据库工具结果类型和渲染能力偏薄

现象：

`frontend/src/types/tool-results.ts` 中 `DatabaseQueryData` 只声明了 `columns`、`rows`、`row_count`。后端 `external_database_query` 实际还返回 `column_definitions`、`truncated`、`duration_ms`、`executed_sql` 等字段。`ToolResultRenderer.vue` 也没有 `external_database_schema` 的专用结构化渲染分支，schema 工具只能走原始输出兜底。

影响：

1. 前端无法稳定展示查询是否被截断、耗时、执行 SQL、字段类型等信息。
2. 后续做 query 结果预算化后，前端如果没有截断提示，用户可能误以为看到的是完整结果。
3. schema 工具没有专用渲染，会让大 schema 输出在 UI 上可读性差，也不利于展示 `schema_hash`、`refreshed_at`、表/列省略数量。

建议：

1. 扩展 `DatabaseQueryData` 类型，补齐 `truncated`、`duration_ms`、`executed_sql`、`column_definitions`、`cell_truncated` 等字段。
2. 新增 `ExternalDatabaseSchemaData` 类型和 `ExternalDatabaseSchema.vue` 渲染组件。
3. 查询表格顶部展示行数、截断状态、耗时和 SQL 折叠区。
4. 对超长单元格提供 hover / 展开查看，默认表格仍保持轻量。

## 五、详细落地计划

### 5.1 总体策略

落地时不建议一次性大改全部数据库智能体链路。推荐分为六个阶段：

1. **先补测试基线**：先把当前缺口写成失败测试，避免后续重构误判。
2. **修 SQL Guard 准确性**：补全字段引用收集、alias 解析和聚合 LIMIT 规则。
3. **做输出预算**：schema 和 query 输出按规模自适应，保留准确性所需信息，同时控制上下文。
4. **治理历史上下文**：数据库工具结果按实时数据处理，历史中只留摘要。
5. **补前端展示与审计可见性**：让截断、耗时、SQL、schema 新旧程度对用户可见。
6. **补连接器生命周期**：解决 timeout 重复、连接池无法失效和 schema refresh 调度问题。

每个阶段都可以独立提交、独立测试。P0 阶段完成后即可显著提升准确性；P1 阶段完成后可明显改善大库、大结果场景效率。

### 5.3 阶段 1：SQL Guard 准确性修复

目标：让 SQL Guard 完整校验所有字段引用，并修正聚合查询 LIMIT 规则。

#### 5.3.1 新增 AST 字段引用收集器

建议新增文件：

- `internal/application/service/sql_reference_collector.go`
- `internal/application/service/sql_reference_collector_test.go`

建议核心结构：

```go
type sqlColumnReference struct {
	Table      string
	TableAlias string
	Column     string
	Clause     string
}

type sqlTableReference struct {
	Table string
	Alias string
}

type sqlReferenceSet struct {
	Tables  []sqlTableReference
	Columns []sqlColumnReference
	CTEs    map[string]struct{}
}
```

关键实现要求：

1. 使用 `pg_query.Parse` 得到 AST，不再依赖 `ParseSQL` 的 `SelectFields/WhereFields` 作为权限校验全集。
2. 遍历 FROM/JOIN 时建立 alias 到真实表的映射。
3. 遍历 SELECT、WHERE、JOIN ON、GROUP BY、HAVING、ORDER BY、WINDOW、CTE、子查询，收集所有 ColumnRef。
4. ColumnRef 解析规则：
   - `table.column`：按 table 或 alias 解析。
   - `schema.table.column`：忽略 schema 后按 table 解析，但 schema 应继续受安全校验限制。
   - `column`：单表时归属该表；多表时若只在一个表中存在可归属，否则返回 ambiguous。
   - `*`：交给现有 wildcard 逻辑处理。
5. CTE 名称不应被当作真实表，但 CTE 内部引用的基础表和字段必须校验。
6. 子查询内部按自己的作用域解析字段，不能拿外层 alias 误判。

#### 5.3.2 改造 `sqlGuard.Validate`

修改文件：

- `internal/application/service/sql_guard.go`

建议改动：

1. 保留 `utils.ValidateSQL` 做通用安全校验。
2. 新增 `collectSQLReferences(normalizedSQL, policy)`，返回表引用、字段引用、是否聚合、是否 group by、是否 wildcard。
3. 表校验从 `parseResult.TableNames + extractCTEBaseTables` 迁移到新的 reference set。
4. 字段校验从 `SelectFields + WhereFields` 迁移到 reference set 的全部字段。
5. 对敏感列、denylist、未知列统一按 `table.column` 判断。
6. `ValidatedSQL.SelectFields` 可继续保留 select 字段，用于审计和摘要；但权限判断不得再依赖它。

#### 5.3.3 修正 LIMIT 规则

修改文件：

- `internal/application/service/sql_guard.go`
- `internal/agent/tools/external_database_query.go`
- `config/prompt_templates/agent_system_prompt.yaml`

建议规则：

1. 原始明细查询：必须显式 LIMIT。
2. 无 GROUP BY 的全局聚合：可免 LIMIT，例如 `COUNT(*)`、`SUM(amount)`、`AVG(score)`。
3. GROUP BY 聚合：必须 LIMIT。
4. 窗口函数查询：必须 LIMIT。
5. DISTINCT 查询：必须 LIMIT，除非只返回全局聚合。

工具描述同步改为：

```text
Use LIMIT for detail, grouped, distinct, or window-function queries. Global aggregate queries that return a single row may omit LIMIT.
```

```bash
go test ./internal/databaseconnector/... ./internal/agent/tools -run 'ForeignKey|JoinHint|MySQL|Postgres|ExternalDatabaseSchema'
```

验收标准：

1. 有 FK 的测试 schema 能输出真实 join 关系。
2. 无 FK 场景仍可输出可能 join，但文案不把它当确定事实。
3. MySQL 常见分析 SQL 不再因简单方言函数被误拒。

## 七、风险与回滚策略

### 7.1 字段引用收集风险

风险：AST 收集器过严，导致原本可执行的合法 SQL 被拒绝。

回滚策略：

1. 保留旧 `ParseSQL` 路径为 fallback，仅在新收集器返回解析错误时临时使用。
2. 为 SQL Guard 增加配置开关，例如 `DATABASE_SQL_GUARD_STRICT_REFERENCES=true`。
3. 灰度期记录新旧判定差异，不立即拦截低风险差异。

### 7.2 输出预算风险

风险：预算过小导致模型缺少必要明细，出现结论不完整。

回滚策略：

1. 小结果完整输出规则必须保留。
2. 预算阈值配置化。
3. `external_database_query` 支持模型用更聚焦 SQL 重新查询，而不是一次查询返回全部。

### 7.3 历史摘要风险

风险：历史摘要过短导致多轮追问缺少上下文。

回滚策略：

1. 历史摘要至少保留 SQL 摘要、行数、字段名、前 1-3 行样本。
2. 对“继续分析刚才结果”类追问，允许模型重新执行上轮 SQL 或派生 SQL。
3. 保留 `RetainRetrievalHistory` 作为排查开关。

### 7.4 连接池失效风险

风险：关闭旧连接池时误关正在使用的连接。

回滚策略：

1. 使用 `database/sql` 的 `DB.Close()`，它会阻止新查询并等待已有连接自然归还。
2. Invalidate 只关闭旧 config hash 对应池，不影响新池。
3. 先在 datasource update/delete 后触发，不做后台 aggressive TTL 清理。

## 九、最终判断

当前数据库智能体已经具备可用的 MVP 基础，整体不是“方向错误”，而是进入了第二阶段优化：

1. 准确性上，要从“提示词要求模型守规则”升级为“SQL Guard 完整理解并校验所有字段引用”。
2. 效率上，要从“完整输出保证模型不丢信息”升级为“按库规模、结果规模、对话阶段自适应输出”。
3. 多轮对话上，要把数据库 schema 和查询结果视为实时数据，历史中只能保留摘要和重新查询提示，不能长期保留完整旧结果。
4. 生命周期上，要补齐 datasource 更新、schema refresh、连接池失效三者之间的联动，避免旧结构和旧连接长期存在。

建议优先顺序为：字段引用全量校验、聚合 LIMIT 规则修正、自适应 query/schema 输出、历史摘要化、schema 生命周期治理、连接池治理、方言和外键增强。前三项完成后，准确性和效率会有立竿见影的改善；后四项会增强长期运行稳定性和复杂业务库适配能力。
