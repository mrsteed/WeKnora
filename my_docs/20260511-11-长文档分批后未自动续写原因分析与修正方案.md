# 长文档分批后未自动续写原因分析与修正方案

## 1. 问题背景

用户请求如下：

```text
这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。
```

页面最终只展示了部分内容，并出现提示：

```text
输出被截断，结果不完整
文档版本 V1 部分完成
```

后端日志的关键终态如下：

```text
completion_status: partial
finish_reason: section_batch_limit
failure_reason: 
allow_complete: false
allow_indexing: false
agent_mode: true
content_len: 4150
agent_steps_count: 0
```

同时，日志中已经出现本地知识库片段：

```xml
<chunk rank="2" knowledge_base_id="4332b139-0b85-4059-a734-bce0f0180369" ...>
<knowledge_title>北海电厂二期智慧电厂项目方案介绍.docx</knowledge_title>
<content>建立标准化数据管控体系...</content>
</chunk>
```

因此，本次失败和前一类“长文档生成完全未查询本地知识库”的问题不同。当前链路已经使用了本地知识库，失败点发生在“知识库支撑长文档按批生成之后，没有继续把剩余章节自动生成完”。

## 2. 关键结论

本次不是模型自然截断，也不是本地知识库未命中，而是系统按设计在首轮只生成固定数量章节后主动结束本轮流式响应。

准确链路如下：

```text
POST /api/v1/agent-chat
-> AgentQA
-> shouldUseKnowledgeGroundedFullDocumentGenerationPath(req, agentConfig) = true
-> runKnowledgeGroundedFullDocumentGenerationPath
-> 检索本地知识库并生成大纲
-> dedicatedFullDocumentSectionLimitPerRun = 3
-> 只生成前 3 个章节
-> 剩余章节未生成完
-> completion_status = partial
-> finish_reason = section_batch_limit
-> document_generation_status = continuing
-> 注册 partial artifact
-> 前端未继续发起 auto_continue 请求
-> 页面显示“输出被截断，结果不完整”
```

也就是说，`section_batch_limit` 不是错误本身，它表示“当前批次已达到章节上限，文档还需要继续”。真正的问题是：后端已经表达了“continuing”，但前端没有默认接管续写；同时 UI 把这种可恢复的分批中间态显示成了失败式的“截断”。

## 3. 证据分析

### 3.1 当前请求已经进入知识库支撑长文档路径

`internal/application/service/session_agent_qa.go` 中的路由逻辑已经具备有效知识库范围判断：

```go
if shouldUseKnowledgeGroundedFullDocumentGenerationPath(req, agentConfig) {
    logger.Infof(ctx, "[FullDocument][Router] selected=knowledge_grounded_full_document kb_scope=%d", len(agentConfig.SearchTargets.GetAllKnowledgeBaseIDs()))
    return s.runKnowledgeGroundedFullDocumentGenerationPath(ctx, req, eventBus, summaryModel, agentConfig)
}
```

`shouldUseKnowledgeGroundedFullDocumentGenerationPath` 的关键条件是：

```go
if !hasEffectiveLocalKnowledgeScope(req, agentConfig) {
    return false
}
```

而本次日志中能看到 `<local_knowledge_context>` 和 `chunk rank`，说明请求已经通过智能体绑定知识库进入了知识库 grounded 生成链路。

### 3.2 `section_batch_limit` 是服务端主动批次边界

当前长文档批次上限定义如下：

```go
dedicatedFullDocumentSectionLimitPerRun = 3
```

`runKnowledgeGroundedFullDocumentGenerationPath` 在首轮只生成 `outline.Sections[:sectionLimit]`。当大纲章节数超过上限时，代码会主动设置：

```go
completionStatus = types.MessageCompletionStatusPartial
finishReason = "section_batch_limit"
documentGenerationStatus = types.ChatDocumentGenerationStatusContinuing
```

这与日志完全一致：

```text
completion_status: partial
finish_reason: section_batch_limit
allow_complete: false
allow_indexing: false
```

因此，4150 字左右停止不是因为模型输出失败，而是因为系统刻意把一次长文档请求拆成“先写 3 章，后续再续写”的多轮过程。

### 3.3 后端已经为续写准备了状态，但它不是自动执行器

知识库长文档路径会创建并更新 `ChatDocumentGenerationRun`：

```go
createKnowledgeGroundedGenerationRun(...)
updateKnowledgeGroundedGenerationRun(...)
```

完成事件中也会通过 `Extra` 带出：

```text
generation_run_id
effective_kb_ids
outline
local_knowledge_used
evidence_refs
evidence_queries
```

后续续写请求应携带：

```json
{
  "intent_hint": "continue_document",
  "base_artifact_id": "上一轮 artifact id",
  "document_output_mode": "delta_only",
  "auto_continue": true,
  "generation_run_id": "首轮 generation_run_id",
  "auto_continue_original_query": "原始完整文档需求"
}
```

前端已经有 `sendAutoContinueMessage` 逻辑来构造这类请求，但该逻辑只在 `autoContinueState.enabled` 为 true 时执行。

### 3.4 自动续写只由用户显式开关触发

`frontend/src/views/chat/index.vue` 中只有在用户开启“自动续写至完成”时才调用：

```js
if (autoContinueDocumentEnabled && !longDocumentTranslateEnabled) {
    startAutoContinueFlow(requestParams);
}
```

而 `maybeContinueDocumentAutomatically` 一开头就判断：

```js
if (!autoContinueState.value.enabled) {
    return;
}
```

本次用户请求语义是“请输出完整的技术方案”，但请求体中没有看到 `auto_continue`，页面也没有显示自动续写过程。因此首轮返回 `section_batch_limit` 后，前端没有继续发起下一轮，文档停在 V1 partial。

### 3.5 UI 将“计划内继续态”误展示为“截断错误”

前端 Agent 完成状态文案对 `completion_status=partial` 使用统一提示：

```text
输出被截断，结果不完整
```

但 `partial` 目前承载了多种含义：

```text
section_batch_limit: 当前批次写完，剩余章节待续写
section_generation_truncated: 模型流中断，当前章节可能不完整
section_generation_error: 章节生成报错
local_knowledge_not_found: 证据不足导致阻断
```

本次属于 `section_batch_limit + document_generation_status=continuing`，它不是失败，也不是模型截断，而是“生成任务未完成，等待续写”。统一显示“输出被截断”会误导用户，以为结果已经异常损坏。

## 4. 根因归纳

### 4.1 直接原因

首轮知识库长文档生成达到 `dedicatedFullDocumentSectionLimitPerRun=3` 的章节批次上限，后端按设计返回：

```text
partial + section_batch_limit + continuing
```

但前端没有自动进入续写流程，导致剩余章节没有继续生成。

### 4.2 产品语义原因

用户明确要求“输出完整的技术方案”，但系统把“完整文档生成”拆成了依赖用户额外开启的“自动续写至完成”开关。对于这种请求，完整性应是默认承诺，不应依赖用户发现并点击额外按钮。

### 4.3 协议语义原因

`completion_status=partial` 同时表示“真实异常截断”和“计划内分批待续写”，缺少更细的前端展示分流。后端虽然有 `document_generation_status=continuing` 和 `auto_continue_next` 能力，但前端默认不启用续写时不会消费这些字段。

### 4.4 体验原因

页面的“输出被截断，结果不完整”提示只看 `partial`，没有结合：

```text
finish_reason = section_batch_limit
document_generation_status = continuing
auto_continue_next = true
```

所以计划内中间态被展示成失败态。

## 5. 修正目标

修正后应满足以下目标：

1. 用户请求 `full_document` 且语义要求完整输出时，系统应自动生成到完成，不能停在首轮 partial。
2. `section_batch_limit` 应被视为“可续写中间态”，不能按模型截断错误展示。
3. 后端完成事件必须稳定带出 `document_generation_status`、`auto_continue_next`、`generation_run_id`、`chat_document_artifact`。
4. 前端续写请求必须沿用首轮 `agent_id`、`summary_model_id`、知识库范围、`generation_run_id` 和原始 query。
5. 只有真正的 `section_generation_truncated`、`section_generation_error`、`local_knowledge_not_found` 才显示异常或阻断提示。
6. 自动续写完成后，最终 artifact 应转为 `completed`，允许索引和导出完整文档。

## 6. 修正方案

### 6.1 前端默认接管完整文档自动续写

建议将自动续写从“手动开关能力”升级为 `full_document` 的默认执行策略。

触发条件：

```text
document_output_mode = full_document
且 agent_enabled = true 或普通长文档生成
且不是长文档翻译任务
且不是用户显式停止
```

在 `sendMsg` 构造请求后，如果满足上述条件，应默认调用 `startAutoContinueFlow(requestParams)`。

推荐策略：

```js
const shouldAutoCompleteFullDocument = documentOutputMode === 'full_document' && !longDocumentTranslateEnabled;
if ((autoContinueDocumentEnabled || shouldAutoCompleteFullDocument) && !longDocumentTranslateEnabled) {
    startAutoContinueFlow(requestParams);
}
```

保留“自动续写至完成”按钮，但它应变为显式状态提示或用户可关闭项，而不是完整文档任务能否完成的前置条件。

### 6.2 根据生成状态优化前端提示

在 `AgentStreamDisplay.vue` 或 `botmsg.vue` 中，不应只按 `completion_status=partial` 显示“输出被截断”。应增加分流：

```text
partial + document_generation_status=continuing + finish_reason=section_batch_limit
=> 已生成当前批次，正在继续生成剩余章节 / 可继续生成剩余章节

partial + finish_reason=section_generation_truncated
=> 输出被截断，当前章节可能不完整

partial + finish_reason=section_generation_error
=> 当前章节生成失败，可重试

partial + finish_reason=local_knowledge_not_found
=> 本地知识库证据不足，无法继续生成
```

这样用户看到的状态会从“失败”变成“还在继续”，符合真实系统行为。

### 6.3 后端完成事件直接补齐自动续写字段

当前 `emitFullDocumentCompletion` 只设置 `DocumentGenerationStatus`，`AutoContinueNext` 主要依赖 artifact 注册后由 `AgentStreamHandler` 再补充。

建议在 `emitFullDocumentCompletion` 内直接根据 `documentGenerationStatus` 设置：

```go
autoContinueNext := types.NormalizeChatDocumentGenerationStatus(documentGenerationStatus) == types.ChatDocumentGenerationStatusContinuing
```

并写入 `event.AgentCompleteData`：

```go
AutoContinueNext:   &autoContinueNext,
AutoContinueReason: chatDocumentAutoContinueReason(documentGenerationStatus),
```

这样即使 artifact 注册稍后完成或出现轻微失败，前端仍能从 complete payload 判断是否应该继续。

### 6.4 `section_batch_limit` 不应携带失败语义

本次日志里 `failure_reason` 为空，这是正确的，应保持。

需要明确约束：

```text
finish_reason=section_batch_limit 仅表示本轮批次上限，不表示失败。
completion_status=partial 是兼容旧协议的中间态。
document_generation_status=continuing 是更准确的任务状态。
```

前端和后端日志、埋点、文案都应优先使用 `document_generation_status` 判断文档任务是否完成，而不是只看 `completion_status`。

### 6.5 续写请求必须使用 generation run，而不是纯基线续写

对于知识库支撑长文档，后续自动续写必须进入：

```text
shouldUseKnowledgeGroundedDocumentContinuationPath
-> runKnowledgeGroundedDocumentContinuationRunPath
```

也就是必须携带：

```json
{
  "auto_continue": true,
  "generation_run_id": "...",
  "base_artifact_id": "...",
  "document_output_mode": "delta_only"
}
```

不能只靠普通“继续生成”提示词让模型自由续写，否则会丢失：

```text
首轮大纲
已完成章节列表
有效知识库范围
章节级检索策略
evidence refs
```

### 6.6 完成态收口

当 `remainingSections` 为空时，后端应输出完成标记或完成事件：

```text
completion_status = completed
finish_reason = stop
document_generation_status = completed
allow_complete = true
allow_indexing = true
```

前端收到 `document_generation_status=completed` 后应：

1. 停止自动续写。
2. 移除 `<!-- document_complete -->` 等内部标记。
3. 将最终 artifact 设为可导出基线。
4. 不再显示“部分完成”。

## 7. 建议改动清单

### 7.1 后端

涉及文件：

```text
internal/application/service/session_agent_qa.go
internal/handler/session/qa.go
internal/handler/session/agent_stream_handler.go
internal/application/service/chat_document_artifact.go
```

建议改动：

1. `emitFullDocumentCompletion` 直接设置 `AutoContinueNext` 和 `AutoContinueReason`。
2. `runKnowledgeGroundedFullDocumentGenerationPath` 保持 `section_batch_limit`，但确保 `Extra` 中稳定包含 `generation_run_id`、`outline`、`effective_kb_ids`。
3. `runKnowledgeGroundedDocumentContinuationRunPath` 每轮完成后准确更新 `CompletedSectionsJSON`，最后一轮无剩余章节时返回 completed。
4. artifact 注册失败时，不应吞掉 complete payload 的续写信号；前端至少可以展示“可继续生成”按钮。
5. 增加日志字段：`generation_run_id`、`completed_sections_count`、`remaining_sections_count`、`auto_continue_next`。

### 7.2 前端

涉及文件：

```text
frontend/src/views/chat/index.vue
frontend/src/views/chat/components/AgentStreamDisplay.vue
frontend/src/views/chat/components/botmsg.vue
frontend/src/i18n/locales/zh-CN.ts
frontend/src/i18n/locales/en-US.ts
```

建议改动：

1. `full_document` 默认启动 `startAutoContinueFlow`，用户停止后不再继续。
2. `maybeContinueDocumentAutomatically` 优先读取 `document_generation_status` 和 `auto_continue_next`。
3. 对 `section_batch_limit + continuing` 显示“正在继续生成剩余章节”或“已生成当前批次，等待继续”。
4. 只有真实中断和错误才显示“输出被截断，结果不完整”。
5. 自动续写过程中，用户消息已标记 `is_auto_continue=true`，建议继续保留，便于审计。

## 8. 测试矩阵

### 8.1 后端单测

1. `runKnowledgeGroundedFullDocumentGenerationPath` 大纲 6 章、批次上限 3：
   - 返回 `partial`。
   - `finish_reason=section_batch_limit`。
   - `document_generation_status=continuing`。
   - `auto_continue_next=true`。
   - `generation_run_id` 非空。
   - `KnowledgeRefs` 非空。

2. 自动续写第 2 轮：
   - 携带 `generation_run_id` 和 `base_artifact_id`。
   - 只生成剩余章节。
   - `CompletedSectionsJSON` 追加新章节。
   - 若仍有剩余章节，继续返回 `continuing`。

3. 最后一轮：
   - 返回 `completed`。
   - `document_generation_status=completed`。
   - `allow_complete=true`。
   - `allow_indexing=true`。
   - artifact 状态为 completed。

4. 真实模型流中断：
   - `finish_reason=section_generation_truncated`。
   - 前端不应自动无限续写同一坏章节，应提示可重试或人工检查。

### 8.2 前端测试

1. 用户发送完整文档请求但未手动点击自动续写按钮：
   - 首轮 complete payload 为 `section_batch_limit + continuing` 时，自动发起下一轮。

2. 用户点击“停止生成”：
   - 自动续写停止。
   - 后续 complete payload 不再触发续写。

3. `section_batch_limit + continuing`：
   - 不显示“输出被截断”。
   - 显示“正在自动续写”或“可继续生成剩余章节”。

4. `local_knowledge_not_found + blocked`：
   - 停止自动续写。
   - 提示本地知识库证据不足。

5. 刷新页面后恢复 partial artifact：
   - 如果 `document_generation_status=continuing`，显示可继续生成入口。
   - 不误判为最终完成。

## 9. 验收标准

1. 同样的投标技术方案请求，在用户不额外点击按钮的情况下，能连续生成直到 `document_generation_status=completed`。
2. 首轮达到章节批次上限时，页面不再显示“输出被截断，结果不完整”。
3. 自动续写请求能继续使用本地知识库，并保留 evidence refs。
4. 最终文档卡片显示“已完成”，版本链中最后一个版本可作为完整导出基线。
5. 日志中能看到完整链路：

```text
selected=knowledge_grounded_full_document
finish_reason=section_batch_limit document_generation_status=continuing auto_continue_next=true
auto_continue=true generation_run_id=...
selected=knowledge_grounded_document_continuation
...
document_generation_status=completed allow_complete=true allow_indexing=true
```

## 10. 优先级建议

### P0

1. `full_document` 默认开启自动续写到完成。
2. `section_batch_limit + continuing` 文案从“截断失败”改为“继续生成中”。
3. 后端 complete payload 直接携带 `auto_continue_next`，不依赖 artifact 注册后补齐。

### P1

1. 完善 generation run 的日志和观测指标。
2. 增加刷新页面后的“继续剩余章节”恢复入口。
3. 区分计划内 partial 与异常 partial 的前端样式。

### P2

1. 支持后端任务化生成完整文档，减少前端多轮 SSE 调度依赖。
2. 将章节批次上限做成配置项，按模型上下文长度和超时策略动态调整。
3. 为投标方案类模板增加章节完成度检查，避免最后一轮过早标记 completed。

## 11. 最终判断

本次失败的本质不是“没有查知识库”，也不是“模型生成能力不足”，而是长文档生成架构已经拆成分批续写后，完整文档请求缺少默认自动续写闭环。

`section_batch_limit` 应该触发下一轮，而不是成为用户看到的最终状态。修正重点是把 `full_document` 的产品承诺、后端 `document_generation_status=continuing` 协议、前端自动续写调度和 UI 文案统一起来。这样既保留分章节生成的稳定性，又能真正交付用户要求的完整投标技术方案。
