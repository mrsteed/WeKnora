# 长技术文档自动续写迭代审查与实现方案

## 1. 背景与需求

当前页面已经支持用户手工选择文档基线，并通过“请继续输出。”之类提示词触发下一轮续写。用户期望把这一步从手工操作优化为自动迭代：模型单轮输出结束后，系统自动选择最新文档版本作为基线，再以固定提示“以当前文档为基准，继续剩余内容输出”发起下一轮续写，直到整篇长技术文档生成完成后自动停止。

本次审查只分析当前代码是否满足需求，并给出技术实现方案，不直接修改业务代码。

## 2. 当前代码审查结论

结论：当前代码已经具备“文档版本基线、delta 续写、完整快照合成、完成后加载最终文档”的基础能力，但还不能满足“无需人工干预自动迭代生成完整长文档”的完整需求。

| 需求 | 当前满足情况 | 主要缺口 |
| --- | --- | --- |
| 单轮输出结束后自动选取最新文档基线并继续输出 | 部分满足。complete 事件返回 artifact 后，前端会把新 artifact 提升为 `selectedBaseArtifact` | 没有自动发起下一轮请求；仍依赖用户点击或输入“继续输出” |
| 自动续写时给出提示“以当前文档为基准，继续剩余内容输出” | 不满足 | 当前只有用户手动输入 query；没有自动续写提示词来源、展示方式和请求标记 |
| 完整文档输出完成后自动停止迭代 | 不满足 | 当前 `completion_status=completed` 只表示本轮回答完成，不表示整篇文档完成；缺少文档级完成判定 |
| 整体长文档体验自动闭环 | 部分满足 | 已有 artifact 版本链和最终快照加载，但缺少自动迭代状态机、循环上限、失败恢复和 UI 状态 |

## 3. 已具备的能力

### 3.1 前端能够识别手工续写意图

`frontend/src/views/chat/index.vue` 中 `inferDocumentContinuationIntent()` 已能识别“继续生成、接着写、续写、补全剩余、继续输出、请继续补齐”等表达，并映射为 `continue_document`。

同一文件中的 `inferDocumentOutputMode()` 会把 `continue_document` 和 `revise_document` 默认映射为 `delta_only`，避免每轮都输出完整文档。

### 3.2 前端会传递基线和输出模式

`frontend/src/views/chat/index.vue` 的 `sendMsg()` 会构造以下请求字段：

```json
{
  "intent_hint": "continue_document",
  "base_artifact_id": "...",
  "document_output_mode": "delta_only"
}
```

`frontend/src/api/chat/streame.ts` 会把这些字段写入 SSE POST body。

### 3.3 后端会基于 artifact 构造续写上下文

`internal/handler/session/chat_document_artifact.go` 的 `prepareDocumentRequest()` 会按 `base_artifact_id` 或当前 session 最新 artifact 加载基线，然后调用 `BuildQuotedContext()` 注入上下文。

`internal/application/service/chat_document_artifact.go` 的 `BuildQuotedContext()` 已经对续写场景输出 `document_continuation_context`，其中明确要求：

1. 不要从头重写上一份文档。
2. 从上一份文档末尾自然继续。
3. 默认只输出新增内容。
4. 直接输出文档正文。

这说明当前已经具备“手工触发续写时，基于上一版 delta 输出”的能力。

### 3.4 后端会把 delta 合成为完整快照

`internal/application/service/chat_document_artifact.go` 的 `RegisterFromAssistantMessage()` 会在消息完成后注册 `ChatDocumentArtifact`。

其中 `buildChatDocumentSnapshotResult()` 已支持：

1. `continue` + `delta_only`：调用 `mergeChatDocumentContinuation(base, delta)` 追加续写内容。
2. `revise` + `delta_only`：调用 `mergeChatDocumentRevisionDelta(base, delta)` 执行章节替换、追加或结构化 patch 合并。

同时，`shouldRegisterChatDocumentArtifact()` 对带基线的 `continue` 和 `revise` 会直接允许注册 artifact，不再受普通新文档 2000 字门槛限制。这解决了“短 delta 无法推进基线”的关键问题。

### 3.5 complete 后能够加载完整最终文档

`internal/handler/session/qa.go` 的 `buildFinalDocumentDelivery()` 会根据 artifact 快照大小决定：

1. 小于等于 `ChatDocumentArtifactInlineContinuationMaxChars` 时，直接通过 `final_document` 内联返回完整快照。
2. 超过阈值时，返回 `final_document_mode=fetch_artifact_snapshot` 和 `final_document_artifact_id`。

`frontend/src/views/chat/index.vue` 的 `hydrateFinalDocumentFromComplete()` 已经支持：

1. `inline_snapshot`：直接写入 `message.final_document_content`。
2. `fetch_artifact_snapshot`：调用 `getChatDocumentArtifact()` 拉取完整 `content_snapshot`。

这说明“完成后一次性加载完整文件”的基础协议已经存在。

## 4. 关键缺口

### 4.1 没有自动续写状态机

当前 `sendMsg()` 只在用户主动发送消息时调用。complete 事件处理逻辑虽然会通过 `promoteCompletedArtifactAsBase()` 自动把新 artifact 设为基线，但随后只是 `resetReplyState()`，不会继续调用 `startStream()` 或创建新请求。

因此当前流程是：

```text
用户发送长文档生成请求
  -> 模型输出一轮
  -> 后端生成 artifact Vn
  -> 前端自动选中 Vn 为基线
  -> 等待用户手工输入“请继续输出”
```

需求期望的流程是：

```text
用户发送长文档生成请求
  -> 模型输出一轮
  -> 后端生成 artifact Vn
  -> 系统自动选择 Vn
  -> 系统自动发起“以当前文档为基准，继续剩余内容输出”
  -> 重复，直到文档级完成
```

### 4.2 缺少文档级完成判定

现有 `completion_status` 定义在 `internal/types/message.go`，语义是当前 assistant message 的生成状态：`pending/completed/partial/failed/cancelled`。

这不能直接用于判断“整篇长技术文档是否已完整生成”。例如某轮模型正常 stop，`completion_status=completed`，但它可能只是完成了第 4 章，还没有输出第 5 章和附录。

因此如果只根据 `completion_status=completed` 停止，会过早停止；如果只要有 artifact 就继续，又会产生无限循环。

### 4.3 缺少自动续写请求标记和用户可见提示

需求要求自动续写时给出提示“以当前文档为基准，继续剩余内容输出”。当前请求结构 `CreateKnowledgeQARequest` 只有 `intent_hint/base_artifact_id/document_output_mode`，没有区分“用户主动发送”和“系统自动续写”的字段。

缺少该字段会带来三个问题：

1. 前端无法稳定展示“系统正在自动续写”的提示。
2. 后端无法记录自动迭代轮次、继承原始任务参数、执行防循环策略。
3. 历史消息中无法区分真实用户输入和系统代发续写提示。

### 4.4 超长基线存在 80000 字符内联限制

`ChatDocumentArtifactInlineContinuationMaxChars` 当前为 80000。`CanContinue()` 会阻止超过该阈值的 artifact 继续内联续写。

这对普通聊天上下文是必要保护，但对“生成完整长技术文档”的自动迭代场景会形成上限：文档接近或超过 80000 字符后，当前 inline baseline 机制无法继续。若目标文档可能超过该规模，需要引入“摘要 + 目录 + 尾部窗口”的续写上下文，而不是完全依赖整篇 `ContentSnapshot`。

### 4.5 长文档任务系统只覆盖翻译，不覆盖技术方案生成

`internal/application/service/long_document.go` 当前 `InferTaskKind()` 只识别全文翻译、导出 Markdown 等场景，`CreateTask()` 也只允许 `LongDocumentTaskKindTranslation`。这套 worker 能自动分批处理翻译任务，但不能直接承担“从知识库/提示生成一篇长技术方案”的自动续写。

## 5. 推荐总体方案

推荐在现有 chat document artifact 机制上扩展“自动续写控制层”，优先做前后端协同的轻量实现，后续再演进为后端任务化 worker。

### 5.1 新增自动续写请求字段

在 `CreateKnowledgeQARequest` 中新增字段：

```go
AutoContinue        bool   `json:"auto_continue,omitempty"`
AutoContinueRootID  string `json:"auto_continue_root_id,omitempty"`
AutoContinueRound   int    `json:"auto_continue_round,omitempty"`
AutoContinuePrompt  string `json:"auto_continue_prompt,omitempty"`
```

字段语义：

1. `auto_continue`：标记本轮由系统自动触发。
2. `auto_continue_root_id`：标记同一次长文档自动生成流程，便于日志、UI 和停止控制。
3. `auto_continue_round`：当前自动续写轮次，从 1 开始递增。
4. `auto_continue_prompt`：默认值为“以当前文档为基准，继续剩余内容输出”。

前端 `startStream()` 类型和 POST body 同步透传这些字段。

### 5.2 前端增加自动续写状态机

在 `frontend/src/views/chat/index.vue` 中新增状态：

```ts
const autoContinueState = ref({
  enabled: false,
  rootId: '',
  round: 0,
  maxRounds: 8,
  prompt: '以当前文档为基准，继续剩余内容输出',
  originalRequest: null,
  lastArtifactId: '',
  lastSnapshotCharCount: 0,
  stoppedReason: ''
});
```

触发方式建议分两类：

1. 显式触发：输入框增加“自动续写至完成”开关，用户打开后，本次长文档生成自动迭代。
2. 智能触发：当用户 prompt 包含“生成完整技术方案、输出完整长文档、整篇输出完毕”等关键词时，前端提示并默认启用。

为降低误触发，第一阶段建议只做显式开关。

### 5.3 complete 后自动发起下一轮

在 `hydrateFinalDocumentFromComplete()` 完成 artifact 加载后，调用 `maybeContinueDocumentAutomatically(message, payload)`。

伪代码：

```ts
const maybeContinueDocumentAutomatically = async (message, payload) => {
  if (!autoContinueState.value.enabled) return;
  if (loading.value || isReplying.value) return;

  const artifact = selectedBaseArtifact.value || message?.chat_document_artifact;
  if (!artifact?.id || artifact.can_inline_continue === false) {
    stopAutoContinue('当前文档过长，无法继续内联续写');
    return;
  }

  const decision = shouldContinueByDocumentState(message, artifact, payload);
  if (!decision.continue) {
    stopAutoContinue(decision.reason || '文档已完成');
    return;
  }

  if (autoContinueState.value.round >= autoContinueState.value.maxRounds) {
    stopAutoContinue('达到自动续写轮次上限');
    return;
  }

  autoContinueState.value.round += 1;
  await sendAutoContinueMessage(artifact);
};
```

`sendAutoContinueMessage()` 应复用上一轮知识库、模型、Agent、MCP、web search、memory 等请求参数，但覆盖文档相关字段：

```ts
{
  query: '以当前文档为基准，继续剩余内容输出',
  intent_hint: 'continue_document',
  base_artifact_id: artifact.id,
  document_output_mode: 'delta_only',
  auto_continue: true,
  auto_continue_root_id: autoContinueState.value.rootId,
  auto_continue_round: autoContinueState.value.round,
  auto_continue_prompt: autoContinueState.value.prompt
}
```

UI 上建议插入一条浅色系统提示消息，而不是伪装成用户手动输入：

```text
系统自动续写：以当前文档为基准，继续剩余内容输出
```

如果历史消息模型暂时没有 system 类型，也可以先使用 `role=user`，但增加 `is_auto_continue=true`，渲染成系统提示样式。

### 5.4 引入文档级完成标记

需要在 prompt、后端清洗和前端判定三处增加统一协议。

建议在续写 prompt 中追加：

```text
完成判定规则：
1. 如果你判断整篇文档已经完整输出，请在正文最后单独输出 <!-- document_complete -->。
2. 如果仍有剩余章节、表格、附录或总结未输出，不要输出该标记。
3. 除该标记外，不要输出解释性说明。
```

后端在 `RegisterFromAssistantMessage()` 前或 snapshot preparation 阶段检测并移除该标记：

```go
const ChatDocumentCompletionMarker = "<!-- document_complete -->"
```

新增派生字段：

```go
DocumentGenerationStatus string `json:"document_generation_status,omitempty" gorm:"-"`
```

建议状态值：

1. `continuing`：仍需继续。
2. `completed`：模型明确输出完成标记。
3. `blocked`：无法继续，例如基线过长、合并失败或轮次上限。
4. `needs_review`：疑似完成但缺少明确标记，需要用户确认。

complete payload 中增加：

```json
{
  "document_generation_status": "completed",
  "auto_continue_next": false,
  "auto_continue_reason": "document_complete_marker"
}
```

前端停止条件优先级：

1. `document_generation_status=completed`：停止并显示“文档已完成”。
2. `artifact.can_inline_continue=false`：停止并提示需要分章节或任务化处理。
3. 达到 `maxRounds`：停止并提示已达到自动续写上限。
4. 本轮快照增长低于阈值，例如小于 200 字，连续两次出现：停止为 `needs_review`。
5. 当前轮失败、取消或没有 artifact：停止并展示恢复操作。

### 5.5 自动续写 prompt 调整

`BuildQuotedContext()` 的续写 prompt 建议从“继续生成上一份文档”升级为“剩余内容续写”，并明确自动续写场景。

建议增加规则：

```text
你正在自动续写一篇长技术文档。

本轮用户提示：以当前文档为基准，继续剩余内容输出

续写规则：
1. 只输出上一份文档之后尚未完成的正文内容。
2. 不要重复已经完整输出的章节、表格或段落。
3. 优先根据已有目录、章节编号、上一段结尾和用户原始目标判断下一节内容。
4. 如果上一份文档末尾句子不完整，先补齐该句。
5. 如果整篇文档已经完成，在正文最后单独输出 <!-- document_complete -->。
6. 不要输出“我将继续”等解释性文字。
```

为了让模型知道“剩余内容”是什么，建议在首次生成长技术文档时要求输出稳定目录，并在 artifact 元数据中保存 `structure_info.heading_titles`。自动续写轮次中把最近 12 个标题、当前末尾窗口和原始用户需求一并注入上下文。

### 5.6 超长文档的上下文策略

当前超过 80000 字符无法继续内联。建议将 `BuildQuotedContext()` 扩展为两种模式：

1. `inline_full`：文档小于 80000 字符，沿用当前完整 snapshot 注入。
2. `outline_tail`：文档超过阈值时，注入目录、开头摘要、最近尾部窗口、原始用户目标，而不是直接失败。

建议 payload：

```xml
<document_continuation_context>
<document_outline>
...
</document_outline>

<document_tail>
最近 12000-16000 字
</document_tail>

<original_user_goal>
用户最初要求生成的长技术文档目标
</original_user_goal>
</document_continuation_context>
```

这样可以让自动续写突破当前内联阈值，但仍控制上下文长度。

## 7. 前端改造点

### 7.1 输入区增加开关

在 `frontend/src/components/Input-field.vue` 增加“自动续写至完成”开关，并通过 `send-msg` 事件传给 `frontend/src/views/chat/index.vue`。

建议默认关闭，避免普通问答被误判为自动长文任务。

### 7.2 请求参数扩展

修改 `frontend/src/api/chat/streame.ts` 的 `startStream()` 参数类型和 POST body，透传：

1. `auto_continue`
2. `auto_continue_root_id`
3. `auto_continue_round`
4. `auto_continue_prompt`

### 7.3 自动续写调度

在 `frontend/src/views/chat/index.vue`：

1. `sendMsg()` 首轮启动时初始化 `autoContinueState`。
2. complete 事件中，先调用 `hydrateFinalDocumentFromComplete()`，确保拿到最新 artifact snapshot。
3. 根据 complete payload 和 artifact 状态调用 `maybeContinueDocumentAutomatically()`。
4. 自动续写请求复用上一轮 `lastRequestMeta`，但覆盖 query、intent、base artifact 和 auto 字段。

注意：自动续写应串行执行，必须等上一轮 complete 并完成 artifact hydrate 后才能发下一轮，避免并发写出多个 sibling revision。

### 7.4 UI 状态

建议在基线 banner 下方增加自动续写状态：

```text
正在自动续写：第 3 轮，当前基线 V4
```

停止时显示：

```text
自动续写已完成：文档已生成完整版本 V8
```

如果停止原因是防护策略：

```text
自动续写已暂停：达到轮次上限，请检查完整文档后继续。
```

## 8. 防循环和异常恢复

必须加入以下保护：

1. `maxRounds`：默认 8，可在配置中调整。
2. `maxSnapshotChars`：默认沿用 `ChatDocumentArtifactSnapshotMaxChars=100000`，超过后自动停止或切换任务化模式。
3. `minGrowthChars`：当前 snapshot 比上一版增长小于 200 字时，记录一次低增长。
4. `maxLowGrowthRounds`：连续 2 轮低增长则停止为 `needs_review`。
5. `sameArtifactGuard`：complete 后 artifact id 未变化时不继续。
6. `can_inline_continue=false`：不继续，提示用户改为分章节或启用任务化生成。
7. 用户点击停止：清空 `autoContinueState.enabled`，并调用现有停止接口终止当前 SSE。

## 10. 分阶段落地计划

### 阶段一：前端自动串行续写闭环

目标：最小改造满足“无需手工输入继续输出”。

1. 增加自动续写开关和 `autoContinueState`。
2. complete 后自动以最新 artifact 发起下一轮。
3. 使用固定提示“以当前文档为基准，继续剩余内容输出”。
4. 增加轮次上限、增长检测、失败停止。

该阶段仍可先用启发式停止，例如用户关闭开关、达到上限、低增长停止。

### 阶段二：文档级完成协议

目标：满足“完整文档输出完成后自动停止”。

1. prompt 增加 `<!-- document_complete -->` 标记规则。
2. 后端清洗标记并返回 `document_generation_status`。
3. 前端按文档级状态停止自动续写。
4. 补齐历史回放和 Agent/非 Agent complete payload。

### 阶段三：超长上下文和任务化演进

目标：支持更长技术方案，不受 80000 字符 inline continuation 限制。

1. 引入 `outline_tail` 续写上下文模式。
2. 保存原始用户目标和文档目录结构。
3. 不把多轮生成迁移到后端异步任务，继续保持前端 SSE 串行自动迭代。

## 11. 最终判断

截至 2026-05-11，本次三个阶段已经完成实现并通过复核。当前代码具备 artifact 版本链、delta 续写、delta 合并完整快照、complete 后完整文档加载、自动把本轮 artifact 设为下一轮基线、前端 SSE 串行自动续写、文档级完成协议，以及超长文档 `outline_tail` 上下文续写能力。

本轮复核发现并修正了两处实现风险：

1. `can_inline_continue` 曾被复用为“是否可继续”的总开关，字段语义不准确。现已新增 `can_continue` 表示 artifact 是否还能继续续写，`can_inline_continue` 恢复为是否能使用完整快照内联上下文，前端自动续写判断改用 `can_continue`。
2. 超长 `outline_tail` 续写上下文曾只包含目录和尾部窗口，缺少首轮用户目标。现已通过 `auto_continue_original_query` 从前端自动续写请求透传原始需求，后端在上下文中注入 `<original_user_goal>`，让 10 万字以上文档继续生成时仍能围绕整体目标展开。

复核后的最终方案保持“前端 SSE 串行自动迭代”模式，不迁移到后端异步任务。对于 10 万字以上的长文档，系统通过 `outline_tail` 上下文和更高快照上限继续自动迭代，直到模型输出 `<!-- document_complete -->` 或触发防循环保护后停止。
