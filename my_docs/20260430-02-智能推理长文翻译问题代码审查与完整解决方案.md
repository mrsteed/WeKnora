# 智能推理长文翻译问题代码审查与完整解决方案

## 1. 审查结论

基于 my_docs/test/1.md 中给出的现象、请求日志和当前前后端实现，问题不是单一的“模型输出不稳定”，而是一个由协议建模、Agent 收尾逻辑、流式事件分发和前端完成态处理共同叠加出来的系统性缺陷。

当前链路对“长文翻译”这类超长输出任务存在以下核心问题：

1. 后端没有把“正常完成”“长度截断”“部分完成”“失败”“取消”区分成显式状态。
2. Agent 对 finish_reason=length 没有专门处理，导致被截断的输出仍可能被当作已完成结果收尾。
3. Agent 流式分发只把 final_answer_tool 的 answer chunk 当作答案流，其余普通 answer chunk 会落入 thought 流，答案/思考边界不稳定。
4. 前端收到 answer.done 或 complete 就直接把消息标记为已完成，没有 partial 概念。
5. 聊天历史入库没有质量门禁，半成品回答会被重新写入聊天历史知识库，污染后续检索。
6. 内置智能推理 Agent 的能力边界与“全文翻译并输出 markdown 文件”任务本身不匹配。

结论很明确：这不是简单调大 token 就能彻底解决的问题。即便把输出上限调大，当前状态建模、事件协议和入库策略不改，仍会继续出现“半成品被标记为完成”“思考文本混入正文”“脏回答被入库”的问题。

## 2. 高优先级审查发现

### 2.1 高危：后端无条件把消息标记为完成并入库

代码位置：

1. internal/handler/session/qa.go 中的 completeAssistantMessage
2. internal/application/service/message.go 中的 IndexMessageToKB

现状：

1. completeAssistantMessage 直接设置 assistantMessage.IsCompleted = true。
2. 同一函数会无条件异步调用 IndexMessageToKB。
3. IndexMessageToKB 只做 think 内容清理，不校验该回答是否完整、是否截断、是否允许入库。

影响：

1. 被截断的翻译结果会被当作完整交付结果。
2. 半成品回答会被写入聊天历史知识库。
3. 后续检索可能命中这条错误回答，造成二次污染。

这与日志中的现象完全一致：日志后半段“总 Chunk 数量: 1”对应的是聊天历史问答入库后的单条 passage，而不是原始 PDF 被重新解析成了 1 个 chunk。

### 2.2 高危：Agent 对 finish_reason=length 没有处理分支

代码位置：

1. internal/agent/observe.go 中的 analyzeResponse
2. internal/agent/engine.go 中的 runReActIteration
3. internal/agent/finalize.go 中的 handleMaxIterations

现状：

1. analyzeResponse 只显式处理 stop、content_filter 和 final_answer 工具调用。
2. finish_reason=length 没有被识别成“输出被截断”。
3. 达到最大轮次时，handleMaxIterations 会直接生成总结答案并将 state.IsComplete 置为 true。

影响：

1. 长文翻译在被长度截断后，系统无法进入“续写”或“部分完成”分支。
2. 后续可能进入重复回答检测、最大轮次兜底，或者以自然完成的形态结束。
3. 用户看到的是“看起来像做完了”的半成品，而不是明确的 partial 状态。

### 2.3 高危：Agent 普通 answer chunk 被错误分发到 thought 流

代码位置：

1. internal/agent/think.go 中的 streamThinkingToEventBus
2. internal/models/chat/remote_api.go 中的 tool-call 增量解析
3. internal/handler/session/agent_stream_handler.go 中的 handleComplete

现状：

1. streamThinkingToEventBus 只在 source=final_answer_tool 时把 answer chunk 作为最终答案事件发出。
2. 对普通的 ResponseTypeAnswer，没有单独的 answer 分支。
3. chunk.Content 非空时，会落入默认逻辑并作为 thought_chunk 发出。
4. handleComplete 又通过 data.FinalAnswer 回填内容，并在“此前没有答案事件”时补发 fallback answer。

影响：

1. 思考流和答案流边界被打乱。
2. 模型的整理性文本、收尾性文本、续写提示残留更容易混入用户可见正文。
3. 前端展示层只能被动做补偿，无法从协议上可靠区分“这是 thought 还是 answer”。

my_docs/test/1.md 中日志里的这组数据正好支持这个判断：

1. emitted_events=map[final_answer_chunk:8319 thought_chunk:245 tool_call_pending:1]
2. 随后又出现 Agent finished naturally: answer=1190 chars

这说明本次会话里既流出过 final_answer_tool 的答案块，又在最后一轮自然停止时把另一段 content 当成最终答案收尾。当前实现没有把这两类内容稳态隔离开。

### 2.4 中高危：前端把 answer.done 或 complete 直接当作完成

代码位置：

1. frontend/src/views/chat/index.vue 中的 handleAgentChunk
2. frontend/src/views/chat/components/AgentStreamDisplay.vue 中的 isConversationDone

现状：

1. answer 事件只要 data.done=true，就会把 message.is_completed=true。
2. complete 事件到达时，也会直接把 message.is_completed=true。
3. 当前前端没有 partial、truncated、failed、cancelled 等细分状态。

影响：

1. 即便后端将来能够识别 partial，前端当前也无法表达。
2. 用户看到的状态仍然是“已经完成”。
3. 交互层无法提示“结果仅为部分输出，未写入历史知识库”。

### 2.5 中危：当前 Agent 能力边界与任务目标不匹配

代码位置：config/builtin_agents.yaml

现状：

1. builtin-smart-reasoning 的 max_completion_tokens 只有 2048。
2. allowed_tools 中没有写 markdown 文件到工作区或对象存储的能力。
3. 该 Agent 被设计为多轮检索推理，不是长文批处理翻译器。

影响：

1. “逐字全文翻译 + 图表翻译 + 尽量保持原格式 + 输出 markdown 文件”本质是离线生成任务。
2. 现在却被压进在线 SSE 问答链路里一次性返回，天然容易触发长度上限。
3. 即使修好 partial/completed 状态，也只能止损，不能从产品能力上彻底闭环。

## 3. 根因归纳

### 3.1 第一层根因：完成态建模缺失

当前消息体系几乎只有 IsCompleted 这一个完成标识，缺少：

1. completion_status
2. finish_reason
3. is_partial
4. allow_indexing
5. failure_reason

因此系统只能把所有结束都折叠成“完成”。

### 3.2 第二层根因：流式协议把 answer 和 thought 混在一起

当前协议层无法保证：

1. 所有普通答案流都走 answer channel。
2. 所有思考流都只走 thought channel。
3. complete 只负责生命周期收尾，不参与正文补写。

因此前端只能从现象推断语义，无法从协议得到稳定真相。

### 3.3 第三层根因：长文翻译被错误塞进在线问答链路

在线问答适合：

1. 检索问答
2. 对比分析
3. 摘要提炼

不适合：

1. 超长全文翻译
2. 原格式 markdown 生成
3. 带图表、参考文献、分章节完整输出

这是任务类型和产品能力边界不匹配的问题。

## 4. 完整解决方案

## 4.1 P0：止损修复，先阻止错误完成与错误入库

目标：不再把截断/半成品结果当作完整交付，不再把它们写入聊天历史知识库。

### 方案 A：扩展完成元数据协议

建议修改：internal/event/event_data.go

为 AgentFinalAnswerData 和 AgentCompleteData 增加统一字段：

```go
CompletionStatus string `json:"completion_status,omitempty"`
FinishReason     string `json:"finish_reason,omitempty"`
IsPartial        bool   `json:"is_partial,omitempty"`
AllowIndexing    bool   `json:"allow_indexing,omitempty"`
AllowComplete    bool   `json:"allow_complete,omitempty"`
FailureReason    string `json:"failure_reason,omitempty"`
```

建议语义：

1. completed：完整完成。
2. partial：输出了部分结果，但未完整交付。
3. failed：执行失败。
4. cancelled：用户停止。

### 方案 B：completeAssistantMessage 必须按完成元数据执行

建议修改：internal/handler/session/qa.go

将 completeAssistantMessage 改为接收：

1. completionStatus
2. finishReason
3. allowIndexing
4. allowComplete

建议行为：

1. 只有 completionStatus=completed 且 allowComplete=true 时，才设置 IsCompleted=true。
2. partial、failed、cancelled 不应被伪装成 completed。
3. allowIndexing=false 时，绝对禁止调用 IndexMessageToKB。

### 方案 C：聊天历史入库增加硬门禁

建议修改：internal/application/service/message.go

让 IndexMessageToKB 显式接收 allowIndexing 或 completion metadata，不再依赖调用方默认正确。

建议入库前校验：

1. completion_status 必须为 completed。
2. finish_reason 不能是 length。
3. 内容不能为空。
4. 内容不能命中明显的计划/自述模式，例如“让我整理”“我将开始翻译”“由于内容太长”。
5. fallback 消息是否入库要单独配置，默认不入库。

这样即使上层漏传，也能在最后一层挡住脏数据。

## 4.2 P1：修复 Agent 对 length 截断的处理

目标：让 Agent 能正确识别长度截断，优先续写，续写失败时明确结束为 partial。

### 方案 D：analyzeResponse 增加 length 分支

建议修改：internal/agent/observe.go

新增 verdict 字段：

```go
type responseVerdict struct {
    isDone           bool
    finalAnswer      string
    emptyContent     bool
    isPartial        bool
    completionStatus string
    finishReason     string
    allowIndexing    bool
    allowComplete    bool
    step             types.AgentStep
}
```

新增判定规则：

1. stop + 无 tool calls：completed。
2. final_answer 工具：completed。
3. length + 无 tool calls：partial，默认 allowIndexing=false，allowComplete=false。
4. content_filter：failed。

### 方案 E：runReActIteration 遇到 partial(length) 时续写

建议修改：internal/agent/engine.go

当 verdict.finishReason=length 且 verdict.isPartial=true 时：

1. 把当前已生成正文保留在累计答案缓冲区中。
2. 追加一条 continuation message：

```text
Continue exactly from where you stopped. Do not repeat previous content. Keep the same output format.
```

3. 单独设置 maxLengthContinuationRounds，例如 3。
4. 若连续续写仍被截断，则以 partial 收尾，并明确告诉前端“这不是完整结果”。

## 4.3 P1：修正 answer/thought 流式分发协议

目标：从协议层稳定区分答案和思考，避免用 complete 阶段回填正文。

### 方案 F：普通 ResponseTypeAnswer 必须走 answer 事件

建议修改：internal/agent/think.go

当前逻辑中，只有 source=final_answer_tool 的 answer chunk 被发成 EventAgentFinalAnswer。建议改成：

1. 只要 chunk.ResponseType == ResponseTypeAnswer，就优先走 answer 事件。
2. source=final_answer_tool 仅作为 metadata 标记，而不是分流条件。
3. 只有 ResponseTypeThinking 才走 thought 事件。

这样可以消除“普通 answer 被当作 thought”的结构性偏差。

### 方案 G：complete 事件不再补正文，只做生命周期收尾

建议修改：internal/handler/session/agent_stream_handler.go

调整原则：

1. 正文只能来自 answer 事件。
2. complete 事件只携带状态、持续时间、步骤数、完成原因。
3. data.FinalAnswer 只作为数据库落库或异常兜底的后备值，不再无条件拼接到 assistantMessage.Content。
4. fallback answer 仅在明确没有任何 answer 事件且 completion_status=completed 时触发。

## 4.4 P1：前端增加 partial/truncated 状态

目标：让前端正确展示“部分完成”而不是误导性地显示“已完成”。

### 方案 H：前端会话状态改造

建议修改：frontend/src/views/chat/index.vue

新增消息状态字段：

1. completion_status
2. finish_reason
3. is_partial
4. allow_indexing

行为调整：

1. answer.done 不再直接等于消息已完成。
2. complete 事件必须读取 completion_status 后再决定是否置 is_completed。
3. partial 时展示“输出已截断，结果未完整生成”。
4. failed/cancelled 时展示对应状态和后续操作按钮。

### 方案 I：AgentStreamDisplay 用 completion_status 判定完成

建议修改：frontend/src/views/chat/components/AgentStreamDisplay.vue

当前 isConversationDone 只看：

1. stop
2. agent_complete
3. answer.done

建议改为：

1. completed：结束并显示已完成。
2. partial：结束但展示部分完成提示。
3. failed：结束并展示失败提示。
4. cancelled：结束并展示已停止。

这样前端才能和后端状态语义对齐。

## 4.5 P2：为长文翻译建立专门任务流

目标：从产品能力上彻底解决“长文全文翻译输出 markdown 文件”问题。

### 方案 J：新增长文翻译异步任务，而不是继续走普通 Agent QA

推荐流程：

1. 文档解析服务提取页面、段落、图片、表格、参考文献结构。
2. 按页或按 chunk 建立翻译任务。
3. 每个任务单元独立翻译并持久化中间结果。
4. 最终统一拼装为 markdown 文件。
5. 输出到对象存储、知识库附件或用户指定目录。

推荐能力边界：

1. Chat 对话只负责发起任务和查看进度。
2. 真正的全文翻译由后台任务执行。
3. 产物以文件形式交付，而不是一次性塞回 SSE 正文。

### 方案 K：短期兼容策略

如果暂时还不做异步任务流，至少应做两件事：

1. 为该类 prompt 单独路由到专用翻译 Agent，而不是 builtin-smart-reasoning。
2. 提升 max_completion_tokens，并强制使用分段续写策略。

但需要明确：这只能缓解，不能替代专门任务流。

## 5. 推荐改造顺序

### 第一阶段

1. 改 event_data.go，补 completion metadata。
2. 改 observe.go，处理 finish_reason=length。
3. 改 qa.go 和 message.go，阻止 partial/failed 内容入库。

### 第二阶段

1. 改 think.go，修正 answer/thought 分流。
2. 改 agent_stream_handler.go，禁止 complete 阶段补正文。
3. 改 frontend chat 流式状态机，支持 partial/truncated。

### 第三阶段

1. 设计长文翻译异步任务。
2. 提供 markdown 文件产物输出。
3. 在 UI 上新增任务进度、失败重试、下载文件入口。

## 6. 测试与验收方案

## 6.1 后端单测

建议新增或补强以下测试：

1. observe.go：finish_reason=length 返回 partial verdict。
2. engine.go：连续 length 时会续写，超过阈值后 partial 结束。
3. think.go：普通 ResponseTypeAnswer 会发 answer 事件，不再落到 thought。
4. qa.go：partial/failed/cancelled 时不会 IsCompleted=true。
5. message.go：allowIndexing=false 时不入库。

## 6.2 前端测试

建议新增：

1. answer.done 但 completion_status=partial 时，消息不展示“已完成”。
2. complete 事件带 partial 时，界面显示部分完成提示。
3. complete 事件带 completed 时，界面正常收尾。
4. 停止事件、失败事件、partial 事件三者状态互斥。

## 6.3 集成回归

建议准备以下场景：

1. 短问答：正常 completed，允许入库。
2. 长文翻译：首次 length，自动续写成功。
3. 长文翻译：多次 length，最终 partial，不入库。
4. final_answer 工具流式输出：正文与 thought 不混流。
5. 用户手动 stop：cancelled，不入库。

## 7. 最终建议

如果只允许做一轮最小修复，我建议优先落这三项：

1. 在 observe.go 中显式处理 finish_reason=length。
2. 在 qa.go 和 message.go 中禁止 partial/failed 结果标记 completed 或写入聊天历史知识库。
3. 在 think.go 中把普通 answer chunk 从 thought 流里拆出来。

如果允许做一轮完整修复，则应在上述基础上继续完成：

1. 前端 partial 状态改造。
2. Agent complete 协议改造。
3. 长文翻译异步任务流重构。

这三步做完后，系统才能真正稳定支持“长文全文翻译并输出 markdown 文件”这类任务，而不是继续依赖普通问答链路硬撑。
