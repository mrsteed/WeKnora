# 长文档生成未查询本地知识库原因分析与修正方案

## 1. 背景

本次审查对象是“智慧运行投标技术方案”完整文档生成流程。用户期望后续文档必须基于本地知识库生成，但实际运行日志显示，长文档生成阶段没有触发本地知识库检索，模型直接按照用户目标生成大纲和章节正文。

本次请求的关键参数如下：

```json
{
  "query": "这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。",
  "knowledge_base_ids": null,
  "knowledge_ids": null,
  "intent_hint": "normal",
  "document_output_mode": "full_document",
  "agent_enabled": true,
  "agent_id": "22a6c00a-9e10-4776-8465-aec4463d21e2",
  "web_search_enabled": true,
  "summary_model_id": "a0d12381-01c7-4243-8c48-6ce15a7d66a1"
}
```

从业务语义看，该请求不是普通闲聊或纯模型写作，而是“投标技术方案生成”。绑定了招投标专家智能体，且该智能体配置了本地知识库。因此，正确行为应是：先基于智能体绑定的本地知识库检索智慧运行、北海智慧电厂、数据湖、算力平台、设备运维智能体、标准化数据管控体系等相关内容，再以检索证据生成大纲和章节正文。

## 2. 关键结论

当前逻辑确实没有查询本地知识库，根因不是知识库工具不存在，而是 `full_document` 长文档生成被路由到了“专用长文档生成捷径”。该捷径直接调用模型生成大纲和章节，系统提示词明确写了 `Do not call tools`，并且生成消息中没有注入任何本地知识库检索结果。

准确链路如下：

```text
POST /api/v1/agent-chat
-> parseQARequest 得到 knowledge_base_ids=null、document_output_mode=full_document
-> resolveAgent 加载“招投标专家”智能体
-> buildAgentConfig 解析到智能体绑定的知识库 4332b139-0b85-4059-a734-bce0f0180369
-> buildSearchTargets 构造 1 个 search target
-> shouldUseDedicatedFullDocumentGenerationPath(req) 只检查 req.KnowledgeBaseIDs / req.KnowledgeIDs
-> 因请求体没有显式 knowledge_base_ids，函数返回 true
-> runDedicatedFullDocumentGenerationPath
-> Chat 生成大纲，ChatStream 分章节生成正文
-> 全程绕过 Agent Engine、knowledge_search、grep_chunks、wiki_search
```

也就是说，虽然智能体配置层面已经解析到了本地知识库，但专用长文档路径的进入条件只看“请求体是否显式带知识库 ID”，没有看“智能体实际生效的知识库配置”。这导致绑定知识库的智能体被误判为纯模型长文档写作。

## 3. 证据

### 3.1 请求体没有显式传知识库 ID

日志显示：

```text
qa.go:453[parseQARequest] | [AgentQA] Request: ...
"knowledge_base_ids":null,
"knowledge_ids":null,
"document_output_mode":"full_document",
"agent_id":"22a6c00a-9e10-4776-8465-aec4463d21e2"
```

这不是问题本身。因为智能体本来可以通过自身配置绑定知识库。后续日志也证明后端确实解析到了智能体绑定知识库。

### 3.2 智能体实际配置了本地知识库

日志显示：

```text
session_knowledge_qa.go:418[resolveKnowledgeBasesFromAgent] | KBSelectionMode=selected: using 1 configured knowledge bases
session_agent_qa.go:1396[buildAgentConfig] | Agent configured with 1 knowledge base(s): [4332b139-0b85-4059-a734-bce0f0180369]
session_knowledge_qa.go:528[buildSearchTargets] | Built 1 search targets: 1 full KB, 0 partial KB
session_agent_qa.go:1408[buildAgentConfig] | Agent search targets built: 1 targets
```

因此，后端不是没有知识库上下文，而是在后续路由判断中没有使用这个“有效知识库上下文”。

### 3.3 专用长文档路径明确禁止工具调用

`internal/application/service/session_agent_qa.go` 中的大纲生成提示词：

```go
systemPrompt.WriteString("You are planning a long-form markdown document. Return only a markdown outline with exactly one H1 title and a sequence of H2 section headings. ")
systemPrompt.WriteString("Do not write section body paragraphs. Do not call tools. Do not output JSON, code fences, commentary, or hidden reasoning. ")
```

章节生成提示词：

```go
systemPrompt.WriteString("You are writing one section of a long-form markdown document. Output only the section body for the requested section. ")
systemPrompt.WriteString("Do not repeat the document title or prior sections. Do not restart the document from the beginning. Do not call tools. Do not output hidden reasoning. ")
```

对应运行日志也显示，大纲生成请求是普通 Chat 调用：

```text
"content": "You are planning a long-form markdown document... Do not call tools..."
```

章节生成请求同样是普通 ChatStream 调用：

```text
"content": "You are writing one section of a long-form markdown document... Do not call tools..."
```

这些消息里只有 `User goal`、`Document title`、`Full outline`、`Completed content so far`、`Current section heading`，没有任何 `local_knowledge_context`、`retrieved information`、引用来源或知识库片段。

### 3.4 日志中没有知识库检索工具执行记录

如果进入 Agent Engine 并调用知识库工具，应出现类似日志：

```text
[Tool][KnowledgeSearch] Execute started
[Tool][KnowledgeSearch] Queries: ...
[Tool][KnowledgeSearch] Concurrent search completed: ... raw results
```

本次 `my_docs/test/10.md` 日志中未出现上述记录。相反，只出现了模型直接生成大纲和正文的请求。

## 4. 代码问题定位

### 4.1 `shouldUseDedicatedFullDocumentGenerationPath` 判断条件不完整

当前逻辑位于 `internal/application/service/session_agent_qa.go`：

```go
func shouldUseDedicatedFullDocumentGenerationPath(req *types.QARequest) bool {
    if req == nil || req.DocumentOutputMode != types.ChatDocumentOutputModeFull {
        return false
    }
    ...
    if len(req.KnowledgeBaseIDs) > 0 || len(req.KnowledgeIDs) > 0 || documentEditRequiresExternalRetrieval(req.Query) {
        return false
    }
    ...
    return true
}
```

问题在于：`req.KnowledgeBaseIDs` 和 `req.KnowledgeIDs` 只代表请求体显式传入的知识库，而不代表智能体最终生效的知识库。

在本次请求中：

```text
req.KnowledgeBaseIDs = nil
req.KnowledgeIDs = nil
agentConfig.KnowledgeBases = [4332b139-0b85-4059-a734-bce0f0180369]
agentConfig.SearchTargets = 1
```

但 `shouldUseDedicatedFullDocumentGenerationPath(req)` 没有接收 `agentConfig`，所以误判为“无知识库依赖的纯文档生成”。

### 4.2 专用长文档路径没有知识库预检索阶段

`runDedicatedFullDocumentGenerationPath` 当前只有两个阶段：

```text
1. buildDedicatedFullDocumentOutlineMessages -> chatModel.Chat 生成大纲
2. buildDedicatedFullDocumentSectionMessages -> chatModel.ChatStream 逐章生成正文
```

缺失阶段是：

```text
0. 基于有效知识库范围执行本地知识库检索
```

因此，即使把 `Do not call tools` 去掉也不够。因为这条路径不是 Agent 工具调用路径，普通 Chat/ChatStream 本身不会自动调用 `knowledge_search`。必须在服务端显式做本地知识库预检索，并把检索结果作为受控上下文注入大纲和章节生成提示词。

### 4.3 文档生成的“稳定性修复”和“知识库 grounded 生成”被混在一起

前一轮为了解决长文档中途重启，新增了专用长文档生成路径。该路径对稳定输出是有价值的：大纲、分章节、批次限制、partial artifact 都比普通 ReAct 更适合长文档。

但它把“长文档生成稳定性”建立在“跳过工具和检索”的前提上，导致投标方案这类知识库依赖型文档失去事实依据。

正确方向不是简单删除专用路径，而是把它拆成两类：

```text
纯模型长文档生成：无绑定 KB、无显式检索意图、无本地资料要求时才允许。
知识库支撑长文档生成：有智能体绑定 KB、请求显式 KB、@mention 文件或业务强制知识库时，先检索再生成。
```

## 5. 修正目标

修正后必须满足以下目标：

1. 只要智能体有效绑定了本地知识库，`full_document` 后续生成就不能跳过本地知识库。
2. 大纲生成必须参考本地知识库检索摘要，不能只基于用户一句目标自由发挥。
3. 每个章节生成前应基于章节标题和用户目标再次检索或复用相关证据包。
4. 如果本地知识库没有检索到足够内容，应明确提示“本地知识库证据不足”，不能直接编造完整投标方案。
5. 输出文档应保留可审计依据，至少在内部 completion metadata 或 answer context 中保存引用来源；如前端支持，可显示引用。
6. 继续生成时必须沿用同一份本地知识库证据策略，而不是续写阶段退回纯模型生成。

## 6. 当前文档复审结论

当前文档已经准确定位了“绑定知识库的智能体误走纯模型 `full_document` 专用路径”这一主因，但作为完整研发方案仍有明显缺口：

1. 只提出了“先检索再生成”的补丁思路，没有形成统一的长文档生成架构。
2. 没有把 `20260511-01` 的自动续写状态机、`20260511-02` 的文档级完成判定、`20260511-04` 的章节锚点续写、`20260511-06` 的轻量文档编辑路径、`20260511-09` 的 `final_answer` 收口和取消优先级纳入同一个闭环。
3. 没有定义“证据包”的生命周期：首轮大纲、章节生成、自动续写、基线修订、artifact 注册、导出审计之间如何复用与持久化。
4. 没有明确请求路由矩阵，容易出现两个相反问题：纯文档编辑误走知识库检索，知识库长文档生成又误走纯模型写作。
5. 没有给出数据模型、服务边界、事件协议、失败降级、质量门禁和测试矩阵，研发落地时仍会分散修补。

因此，本次最终方案应从“修一个 if 条件”升级为“知识库支撑长文档生成架构”。核心原则是：长文档生成可以分章节、可续写、可失败恢复，但任何业务型投标方案都必须先取得本地知识库证据，并且每个 artifact 版本都能说明自己引用了哪些本地知识。

## 7. 统一架构目标

### 7.1 原始需求还原

原始用户需求是：

```text
这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。
```

这类请求的业务要求有三层：

1. 输出对象是甲方投标技术方案，不是普通问答摘要。
2. 内容对象是智慧运行，必须覆盖业务部门、技术专家、实施技术方关注的完整方案。
3. 事实来源必须是本地知识库，不能用模型先验自由编写项目背景、产品能力、建设范围和技术细节。

### 7.2 架构验收目标

改造后的系统必须满足：

1. 有效知识库范围解析：请求显式 KB、@mention 文件、智能体绑定 KB、共享智能体 KB 都能统一解析为 `SearchTargets`。
2. 强制本地证据：只要有效 KB 范围非空，`full_document` 不能进入纯模型专用路径。
3. 服务端预检索：大纲和章节生成都由服务端检索证据包后注入模型，不依赖模型自行调用工具。
4. 分章节稳定生成：保留专用长文档路径的大纲、章节、批次、partial artifact 能力，避免普通 ReAct 多轮从头重启。
5. 自动续写闭环：首轮未完成时返回 `document_generation_status=continuing`，前端根据 `auto_continue_next=true` 自动续写；完成、阻塞、需复核时停止。
6. 章节锚点正确：指定章节补充必须走 `revise_document + document_patch`，不能被 `continue_document` 追加到文尾。
7. 可审计引用：生成结果、artifact、SSE complete payload 至少能追踪本轮使用的 KB、知识文档和 chunk。
8. 失败不污染：流尾异常、模型超时、用户停止、工具历史协议错误都不能把半截文档标记为 completed，也不能污染后续上下文。

## 8. 总体架构设计

### 8.1 分层结构

建议新增一条 `KnowledgeGroundedDocumentGeneration` 受控链路，位于 `AgentQA` 内部，但不复用通用 ReAct Agent 循环。

```text
API / 前端请求
  -> 请求解析与文档意图识别
  -> Agent 配置解析与有效 KB 范围解析
  -> 文档任务路由器 DocumentGenerationRouter
      -> DedicatedDocumentEditPath            // 基线修订、章节移动、章节补充
      -> KnowledgeGroundedFullDocumentPath    // 基于本地知识库的完整长文档生成
      -> PureFullDocumentPath                 // 无 KB、无本地资料要求的纯模型长文档
      -> GenericAgentPath                     // 普通问答、复杂工具推理、显式外部检索
  -> EvidenceRetriever / WikiEvidenceReader
  -> OutlinePlanner
  -> SectionPlanner
  -> SectionWriter
  -> ArtifactAssembler
  -> QualityGate
  -> SSE / Artifact / AutoContinue 状态输出
```

### 8.2 关键设计决策

1. 知识库型 `full_document` 不走通用 ReAct。

   ReAct 适合“查资料后回答”，不适合一次生成完整长文档。`20260511-09` 已证明普通 Agent 可能在 `final_answer` 流尾异常后进入下一轮并从头重启。因此 KB 长文档应走受控 pipeline，而不是让模型在 40 轮工具循环里自由决定何时结束。

2. 本地知识检索由服务端确定性执行。

   当前专用路径是普通 Chat/ChatStream，不具备自动工具调用能力。即使去掉 `Do not call tools`，模型也不会自动调用 `knowledge_search`。因此必须由服务端在大纲和每个章节前执行检索，把证据包注入 prompt。

3. 文档编辑与知识库生成必须分流。

   `20260511-07` 已证明纯基线编辑如果误走检索 Agent，会产生 `knowledge_search tool not found`。因此“修改上一版文档内部结构”默认走轻量编辑；只有用户明确写“结合知识库、搜索知识库、引用来源、查询数据库”等，才进入 KB-grounded 修订或通用 Agent。

4. 所有自动续写都必须围绕 artifact 快照推进。

   `20260511-01` 和 `20260511-02` 已经确认：自动续写不能靠聊天历史猜测进度，必须以上一版 artifact 为基线，使用 `document_generation_status` 和 `auto_continue_next` 决策是否继续。

## 9. 请求路由矩阵

### 9.1 路由优先级

`AgentQA` 在构建 `agentConfig` 和 `SearchTargets` 后执行路由，优先级如下：

| 优先级 | 条件 | 路径 | 原因 |
| --- | --- | --- | --- |
| 1 | `revise_document/delta_only/base_artifact_id`，且无明确外部检索需求 | `runDedicatedDocumentEditPath` | 基线内部编辑，避免大上下文 ReAct 超时 |
| 2 | `revise_document/delta_only/base_artifact_id`，且明确要求结合知识库 | `runKnowledgeGroundedDocumentRevisionPath` | 先检索证据，再生成 patch |
| 3 | `full_document` 且有效 KB 范围非空 | `runKnowledgeGroundedFullDocumentGenerationPath` | 投标方案、技术方案等必须 grounded |
| 4 | `full_document` 且无有效 KB、无本地资料要求 | `runDedicatedFullDocumentGenerationPath` | 纯模型长文档，仍使用分章节稳定路径 |
| 5 | 其他普通问题、复杂工具推理、显式 Web/DB/MCP | `GenericAgentPath` | 保留通用 Agent 能力 |

### 9.2 有效 KB 范围判定

新增统一函数：

```go
func hasEffectiveLocalKnowledgeScope(req *types.QARequest, agentConfig *types.AgentConfig) bool {
    if req != nil && (len(req.KnowledgeBaseIDs) > 0 || len(req.KnowledgeIDs) > 0) {
        return true
    }
    if agentConfig == nil {
        return false
    }
    return len(agentConfig.KnowledgeBases) > 0 ||
        len(agentConfig.KnowledgeIDs) > 0 ||
        len(agentConfig.SearchTargets) > 0
}
```

注意：不能只看 `req.KnowledgeBaseIDs`。本次真实故障就是请求体为 `knowledge_base_ids=null`，但智能体配置里已经解析出 `[4332b139-0b85-4059-a734-bce0f0180369]`。

### 9.3 外部检索需求判定

复用并扩展 `documentEditRequiresExternalRetrieval(query)`：

```text
明确进入外部/知识检索的关键词：结合知识库、检索知识库、引用来源、根据原文、查询数据库、联网、网络搜索、网页搜索、最新资料。
不应触发外部检索的情况：WebSearchEnabled=true 但用户只说“把 A 合并到 B”“继续补充某章节”“修改上一版”。
```

这保证 `20260511-07` 的修复不回退。

## 10. 核心服务设计

### 10.1 DocumentGenerationRouter

职责：在 `AgentQA` 内部完成一次性路由，不让同一个请求同时被多个路径处理。

伪代码：

```go
agentConfig, err := s.buildAgentConfig(ctx, req, tenantInfo, agentTenantID)
summaryModel := s.resolveSummaryModel(...)

switch {
case shouldUseDedicatedDocumentEditPath(req):
    return s.runDedicatedDocumentEditPath(ctx, req, eventBus, summaryModel, agentConfig)

case shouldUseKnowledgeGroundedDocumentRevisionPath(req, agentConfig):
    return s.runKnowledgeGroundedDocumentRevisionPath(ctx, req, eventBus, summaryModel, agentConfig)

case shouldUseKnowledgeGroundedFullDocumentGenerationPath(req, agentConfig):
    return s.runKnowledgeGroundedFullDocumentGenerationPath(ctx, req, eventBus, summaryModel, agentConfig)

case shouldUseDedicatedFullDocumentGenerationPath(req, agentConfig):
    return s.runDedicatedFullDocumentGenerationPath(ctx, req, eventBus, summaryModel, agentConfig)

default:
    return s.runGenericAgentPath(ctx, req, eventBus, summaryModel, agentConfig)
}
```

### 10.2 EvidenceRetriever

职责：把有效 KB 范围和 query plan 转换为可注入 prompt 的证据包。

```go
type LocalKnowledgeEvidence struct {
    Query           string
    KnowledgeBaseID string
    KnowledgeID     string
    ChunkID         string
    Title           string
    SourceName      string
    Score           float64
    Content         string
    EvidenceType    string // chunk, wiki_page, wiki_source_doc, database_row
}

type EvidencePack struct {
    ScopeKBIDs       []string
    Queries          []string
    Items            []LocalKnowledgeEvidence
    TotalChars       int
    MissingReason    string
    RetrievalMode    string // chunks, wiki, mixed
}
```

检索策略：

1. 对有 vector/keyword 能力的 KB，复用 `CHUNK_SEARCH -> CHUNK_RERANK -> CHUNK_MERGE -> FILTER_TOP_K`。
2. 对 wiki-only KB，不能静默跳过；必须调用 wiki 检索读取链路，或降级到 GenericAgentPath 让 `wiki_search/wiki_read_page` 执行。
3. 对 database KB，完整技术方案一般不应直接扫表；只有用户明确要求数据库事实时，走数据库工具或结构化查询路径。
4. 每个 evidence item 必须保留 `kb_id/knowledge_id/chunk_id/title/score`，用于引用审计。

### 10.3 QueryPlanner

职责：把用户目标、大纲标题、章节标题转成检索 query。

首轮大纲 query：

```text
原始用户目标
智慧运行 投标 技术方案
北海智慧电厂 数据湖 算力平台 智慧运行
设备运维智能体 标准化数据管控体系 语音交互
业务部门 技术专家 实施保障 运维服务
```

章节 query：

```text
原始用户目标 + 当前章节标题
文档标题 + 当前章节标题
当前章节标题中的核心业务名词
上一章/下一章标题 + 当前章节标题
```

自动续写 query：

```text
原始用户目标 + 未完成章节标题
当前 artifact 大纲 + 下一章节标题
上一版末尾窗口 + 下一章节标题
```

### 10.4 OutlinePlanner

职责：基于首轮 evidence 生成完整文档大纲，而不是模型自由规划。

输入：用户目标、有效 KB 范围、首轮 evidence、输出语言、文档类型。

输出：

```go
type GroundedDocumentOutline struct {
    Title    string
    Sections []GroundedDocumentSection
}

type GroundedDocumentSection struct {
    ID            string
    Heading       string
    Intent        string
    RequiredFacts []string
    EvidenceQuery []string
}
```

提示词原则：

```text
只能基于 local_knowledge_context 规划大纲。
不得创造未在本地知识库出现的项目名称、客户背景、产品能力、系统模块或技术指标。
如果证据不足，必须规划“本地知识库证据缺口”章节，而不是补写完整方案。
```

### 10.5 SectionWriter

职责：每个章节生成前重新检索章节证据，并只生成当前章节正文。

输入：大纲、当前章节、已完成内容摘要、章节 evidence、引用格式要求。

输出：章节正文、章节引用列表、章节状态。

章节提示词必须包含：

```text
当前任务是写当前章节，不要重复文档标题和已完成章节。
只能使用 local_knowledge_context 中的事实。
不得编造项目事实、产品能力、数字、组织名称、实施范围。
证据不足时，输出“本节本地知识库依据不足”并列出缺口，不要补成看似完整的技术方案。
```

### 10.6 ArtifactAssembler

职责：把标题、章节正文、引用信息合成为 artifact 快照。

生成模式：

1. 首轮 create：从空文档生成 `full_document` 的前 N 个章节，若未完成则 `document_generation_status=continuing`。
2. 自动续写 continue：基于上一版 artifact、outline 和下一章节继续生成，合并为新快照。
3. 章节修订 revise：输出 `<document_patch>`，由现有 patch merge 合并到目标章节。

必须保留现有 artifact 链路：

```text
parent_artifact_id
revision_no
content_snapshot
document_generation_status
quality_issues
can_continue
can_inline_continue
```

## 12. 状态机设计

### 12.1 生成运行状态

```text
planning
  -> writing
  -> continuing
  -> completed
  -> blocked
  -> needs_review
  -> failed
  -> cancelled
```

状态含义：

| 状态 | 含义 | 前端动作 |
| --- | --- | --- |
| `planning` | 正在检索证据并规划大纲 | 显示规划进度 |
| `writing` | 正在生成本轮章节 | 展示流式正文 |
| `continuing` | 本轮正常结束但文档未完成 | 可自动续写 |
| `completed` | 文档已完成 | 停止自动续写 |
| `blocked` | 证据不足、上下文过大或用户停止 | 停止并提示 |
| `needs_review` | 检测到重复头、章节回退、低增量等质量问题 | 停止并提示人工复核 |
| `failed` | 系统错误或模型失败且无可用输出 | 停止并保留上一版 |
| `cancelled` | 用户主动停止 | 取消态优先，禁止 completed 覆盖 |

### 12.2 自动续写决策

后端是唯一权威：

```text
document_generation_status == continuing -> auto_continue_next=true
completed / blocked / needs_review / failed / cancelled -> auto_continue_next=false
```

前端只执行后端返回的 `auto_continue_next`，不再自行猜测长文档是否完成。

### 12.3 完成判定

沿用并补强 `20260511-02` 的规则：

1. 显式 `<!-- document_complete -->`：completed。
2. 大纲全部章节完成：completed。
3. 输出为空或只有“已完成/无需继续”：completed。
4. 章节批次未写完：continuing。
5. 检索不到支撑下一章的本地证据：blocked。
6. 重复文档头、章节编号回退、低增量、结束章节后继续扩写：needs_review。
7. 流尾异常但已有可用输出：partial + continuing 或 needs_review，不能 completed。

## 13. 流式与失败收口设计

### 13.1 真实模型流优先

`20260511-08` 说明专用路径不能丢弃 reasoning 流。KB 长文档路径也应遵守：

1. `ResponseTypeThinking` 作为模型活动信号，避免误判 first content timeout。
2. 系统心跳使用 `AgentThoughtData{Replace:true, Synthetic:true}` 覆盖更新，不反复追加。
3. 只有模型完全无 thinking、无 answer、无 tool/result 时，才触发 first activity timeout。

### 13.2 final_answer 与流尾异常

`20260511-09` 的规则必须作为通用 Agent 兜底：

```text
FinalAnswerStreamed=true + stream_error_after_answer -> 终止 ReAct，partial 收口，不进入下一轮。
```

对于 KB 长文档专用路径，原则更简单：章节流尾异常时，保留已输出章节片段，标记当前 artifact 为 partial/continuing 或 needs_review，不调用通用 final answer synthesis 重新生成全文。

### 13.3 取消态优先级

已有 `messageCompletionStatusPriority()` 和 `shouldPreserveExistingAssistantCompletion()`，架构上必须坚持：

```text
cancelled > failed > partial > completed
```

用户 stop 后，后续 complete event 不能把同一 message 覆盖成 completed。

### 13.4 工具历史污染隔离

KB 长文档专用路径不写入通用 Agent tool history。通用 Agent 仍需保留 `20260511-05` 的协议清洗：任何历史 `tool` 消息进入 LLM 前必须满足 OpenAI 工具链协议；失败轮次的半截工具结果不能污染 Redis 上下文。

## 14. Prompt 与上下文设计

### 14.1 大纲规划 prompt

```text
你正在为投标技术方案规划 Markdown 大纲。
你只能使用 <local_knowledge_context> 中的本地知识库证据。
不要使用模型先验补充项目名称、客户背景、功能模块、技术指标或实施范围。
如果本地证据不足以支撑完整技术方案，必须规划“本地知识库证据缺口”章节。
只输出一个 H1 标题和 4-8 个 H2 章节标题。
```

### 14.2 章节生成 prompt

```text
你正在撰写长文档中的一个章节。
当前章节标题已经由系统输出，不要重复文档标题、不要重新开始整篇文档。
只能依据 <local_knowledge_context> 和 <completed_document_summary>。
需要面向业务部门、技术专家和实施方写清楚业务价值、技术架构、关键模块、集成边界、实施保障。
凡是本地证据没有支撑的事实，不得编造；请写“本节本地知识库依据不足”并列出缺口。
```

### 14.3 自动续写 prompt

```text
以当前 artifact 为基线，继续生成未完成章节。
只输出下一批章节正文，不要重复已完成章节。
继续使用本轮 generation_run 的本地知识库范围和大纲。
如果所有章节已完成，输出 <!-- document_complete -->。
```

### 14.4 章节修订 prompt

章节级补充、移动、合并继续使用 `<document_patch>`：

```xml
<document_patch>
  <append heading="智慧运行">
    ...新增内容...
  </append>
</document_patch>
```

如果用户说“继续补充智慧运行章节”，应识别为 `revise_document + append_to_section`，不是 `continue_document + append_to_tail`。

### 15.1 自动续写状态机

autoContinueState = {

### 15.2 引用展示

前端在文档卡片或导出面板展示：

```text
本轮使用本地知识库：智能 119 接处警系统项目资料 / 产品手册 / ...
引用来源：文档名、章节、chunk 或 Wiki 页面。
```

第一阶段可以只在调试信息和导出附录中展示，后续再做章节级引用脚注。

### 15.3 停止逻辑

前端收到以下状态必须停止自动续写：

```text

blocked
needs_review
failed
cancelled
auto_continue_next=false
```

不得因为普通 `completion_status=completed` 单独判断整篇文档完成；整篇文档只看 `document_generation_status`。

## 16. 质量门禁

### 16.1 生成前门禁

1. 有效 KB 范围为空：允许纯模型路径，但必须在日志注明 `local_knowledge_used=false`。
2. 有效 KB 范围非空但首轮 evidence 为空：阻断完整方案生成，返回 `local_knowledge_not_found`。
3. KB 是 wiki-only 且无法读取 Wiki：阻断或降级到 GenericAgentPath，不允许静默生成。

### 16.2 生成中门禁

1. 当前章节 evidence 为空：该章节写证据不足说明，或将 run 标记为 blocked。
2. 章节输出重复文档标题：停止本章，标记 `duplicate_document_head`。
3. 章节编号回退：标记 `section_number_reset`，停止自动续写。
4. 输出增量与 base 尾部重复率过高：标记 `low_novelty_delta`。

### 16.3 注册 artifact 门禁

注册 artifact 时增加质量判断：

```text
local_knowledge_used=false 且 hasEffectiveLocalKnowledgeScope=true -> missing_local_knowledge_context
evidence_refs 为空但生成完整方案 -> missing_evidence_refs
duplicate_document_head -> needs_review
section_number_reset -> needs_review
low_novelty_delta -> needs_review
terminal_section_tail -> completed 或 needs_review
```

需要新增质量 issue：

```go
ChatDocumentQualityIssueMissingLocalKnowledgeContext = "missing_local_knowledge_context"
ChatDocumentQualityIssueMissingEvidenceRefs = "missing_evidence_refs"
ChatDocumentQualityIssueLocalKnowledgeNotFound = "local_knowledge_not_found"
```

## 17. 落地计划

### 阶段一：止血路由

目标：绑定 KB 的 `full_document` 不再误走纯模型路径。

改动：

1. `shouldUseDedicatedFullDocumentGenerationPath(req, agentConfig)` 接收 `agentConfig`。
2. 新增 `hasEffectiveLocalKnowledgeScope()`。
3. 有效 KB 非空时，纯模型 dedicated full path 返回 false。
4. 增加路由日志：`[FullDocument][Router] selected=knowledge_grounded_full_document`。

### 阶段二：KB-grounded 长文档主链路

目标：完整投标方案先检索再分章节生成。

改动：

1. 新增 `runKnowledgeGroundedFullDocumentGenerationPath()`。
2. 新增 `EvidenceRetriever`、`QueryPlanner`、`OutlinePlanner`、`SectionWriter`。
3. 大纲和章节 prompt 注入 `<local_knowledge_context>`。
4. evidence 为空时返回 `blocked/local_knowledge_not_found`。
5. complete payload 返回 `local_knowledge_used/evidence_refs/document_generation_status/auto_continue_next`。

### 阶段三：自动续写与 run 持久化

目标：跨轮次稳定延续同一大纲和 KB 范围。

改动：

1. 新增 `chat_document_generation_runs`。
2. 自动续写请求带 `generation_run_id`。
3. 后端按 run 找下一批未完成章节。
4. 前端只依据 `auto_continue_next` 触发下一轮。

### 阶段四：引用审计

目标：每个 artifact 能追踪本地知识来源。

改动：

1. 新增 `chat_document_evidence_refs`。
2. artifact 注册时批量写入 evidence refs。
3. artifact API 返回引用摘要。
4. 导出文档支持“本地知识库引用附录”。

### 阶段五：质量与故障收口

目标：杜绝重复开头、无限续写、取消被覆盖和历史工具污染。

改动：

1. 接入 `FinalAnswerStreamed + stream_error_after_answer` partial 收口。
2. 坚持 `cancelled > failed > partial > completed`。
3. 专用长文档路径不调用通用 final synthesis 重写全文。
4. 加强 Agent 历史工具消息协议清洗。
5. 增加重复标题、章节回退、低增量、证据缺失质量门禁。

## 18. 回归测试矩阵

| 场景 | 输入 | 预期 |
| --- | --- | --- |
| 智能体绑定 KB，完整技术方案 | `knowledge_base_ids=null`，agent 配置 KB | 进入 `knowledge_grounded_full_document` |
| 请求显式 KB，完整技术方案 | `knowledge_base_ids=[kb]` | 进入 KB-grounded 路径 |
| 无 KB 纯模型长文档 | 无 agent KB、无显式 KB | 进入 pure dedicated full path |
| 首轮 evidence 为空 | KB 存在但无检索结果 | blocked，不能输出完整方案 |
| 大纲生成 | evidence 含“数据湖、算力平台” | prompt 包含 evidence，大纲覆盖相关章节 |
| 章节生成 | 当前章节“智慧运行总体架构” | 章节检索 query 包含当前标题 |
| 自动续写 | 首轮 section_batch_limit | 返回 continuing + auto_continue_next=true |
| 文档完成 | 大纲章节全部完成或 marker | completed + auto_continue_next=false |
| 章节补充 | “继续补充智慧运行章节” | revise + append_to_section，不追加到文尾 |
| 基线移动 | “把 2.5.5 后续合并到第二章” | dedicated edit path，输出 patch |
| WebSearch 开关打开但纯编辑 | `web_search_enabled=true` | 不阻断 dedicated edit path |
| final_answer 流尾异常 | `FinalAnswerStreamed=true` | partial 收口，不进入下一轮 |
| 用户 stop | stop 后又收到 complete | 保持 cancelled，不被 completed 覆盖 |
| 重复文档头 | delta 再次出现 H1 | needs_review，停止自动续写 |
| 工具历史污染 | Redis 有孤立 tool | sanitize 后不发送非法 tool 消息 |

## 19. 验收标准

用原始投标方案请求验收：

```text
这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。
```

必须看到：

1. 路由日志显示进入 `knowledge_grounded_full_document`。
2. 有效 KB 范围包含智能体绑定知识库 `4332b139-0b85-4059-a734-bce0f0180369`。
3. 首轮大纲前出现本地知识库检索日志，且 evidence 数量大于 0。
4. 每个章节生成前都有章节 query 和章节 evidence。
5. 大纲和正文 prompt 中包含 `<local_knowledge_context>`。
6. 输出内容中的项目事实能在 evidence refs 中追溯。
7. evidence 为空时不会输出完整投标方案。
8. 长文档分批生成时 artifact 状态为 continuing，完成后为 completed。
9. 自动续写不会重复标题、不会章节编号回退、不会无限循环。
10. 停止、超时、流尾异常都以 partial/blocked/cancelled 正确收口，不污染下一轮。

## 20. 最终结论

当前问题不能只修 `shouldUseDedicatedFullDocumentGenerationPath` 一个判断。真正需要落地的是一条“知识库支撑长文档生成”受控架构：先解析有效 KB 范围，再由服务端检索证据包，随后按大纲和章节分批生成，最终通过 artifact 状态机、自动续写、质量门禁和引用审计形成闭环。

该架构同时吸收了 `20260511-*` 系列问题的经验：普通 ReAct 会导致长文档从头重启，纯文档编辑误走检索会失败，章节补充误判为文尾续写会错位，流尾异常和取消竞态会污染状态，自动续写缺少文档级完成判定会循环。因此，KB 长文档生成必须从通用 Agent 中拆出受控 pipeline，并把“本地知识库证据”作为贯穿规划、章节生成、续写、artifact 注册和导出的第一等数据。
