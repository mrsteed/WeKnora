# 长文档规划展示与正文排版优化修正方案

## 1. 问题背景

本次长文档生成已经能完成 `# 北海电厂二期智慧电厂项目技术方案` 的完整生成，但页面侧仍暴露两个体验问题：

1. 规划过程没有被页面直观看到。用户期望看到完整大纲：

   ```markdown
   # 北海电厂二期智慧电厂项目技术方案
   ## 项目背景与建设目标
   ## 总体技术架构
   ## 数据湖与基础算力平台
   ## 智慧运行系统
   ## 智慧安防系统
   ## 智能安全监控应急中心
   ## AR眼镜智能作业
   ## 实施与运维保障
   ```

   但页面只显示“已完成 1 个步骤，耗时 8m58s”，规划内容被折叠或根本没有以完整 Markdown 大纲事件输出。

2. 正文排版不够美观。截图中“数据湖与基础算力平台”章节出现 `###3.1全域数据湖建设...`，Markdown 标题缺少空格，且标题与正文粘连，导致标题不能按 H3 样式渲染，也影响投标技术方案的正式交付质感。

## 2. 根因分析

### 2.1 规划大纲只被后台解析，没有作为完整大纲展示事件稳定输出

后端 `chatFullDocumentOutlineWithProgress` 已经把大纲生成从普通 `Chat` 改为 `ChatStream`，并在流式过程中调用 `emitOutlineProgress` 识别标题和章节数量。相关逻辑在 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L805-L915)。

当前进度事件内容主要是：

- `正在生成大纲：已识别标题“...”`
- `正在生成大纲：已识别 N 个章节。`
- `正在检索第 N/M 章...`
- `第 N/M 章...已完成...`

这些是进度文本，不是完整大纲 Markdown。前端 `AgentStreamDisplay.vue` 虽然已有 `extractPlanningOutlineFromText` 和 `planning-outline-preview`，但它只能从 thinking 内容中解析 `# ...` / `## ...`，见 [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue#L840-L915)。如果后端没有发送完整大纲文本，前端就没有可解析的内容。

### 2.2 后端 synthetic thought 使用同一个 event id 且 Replace=true，规划内容容易被后续进度覆盖

`fullDocumentProgressReporter.UpdateStage` 使用同一个 `progressEventID`，并设置 `Replace: true`、`Synthetic: true`、`Stage: planning/retrieving/generating/finalizing`，见 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L760-L779)。这适合展示“当前正在做什么”，但不适合持久展示“已经完成的大纲”。

即使后端在某一刻把完整大纲放到 `UpdateStage("planning", outlineMarkdown)`，后续 `retrieving/generating/finalizing` 也会以同一个事件替换它。最终页面只剩最后一条“第 8/8 章已完成，正在收尾完整文档”之类的进度摘要。

### 2.3 前端把中间步骤默认折叠，规划信息被藏起来

前端在 [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue#L1005-L1248) 中定义 `showIntermediateSteps = false`，并在 `shouldShowCollapsedSteps` 为真时只展示折叠树根。答案开始生成或会话完成后，中间 thinking/tool_call 事件被放进折叠树，主区域只显示 answer 事件，相关逻辑在 [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue#L1488-L1545)。

这解释了截图中的现象：用户看到“已完成 1 个步骤”，但规划大纲不在首屏显式可见。

### 2.4 章节 Prompt 对排版约束不足

当前知识库长文档章节 Prompt 在 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L2391-L2435)，核心约束是：

```text
Write the body for this section only. You may use lower-level markdown headings such as ### within this section when necessary.
```

它只允许使用 `###`，但没有明确要求：

- `###` 后必须有一个空格；
- 标题必须独占一行；
- 标题后必须空一行再写正文；
- 正文应按“短段落 + 要点 + 小标题”组织；
- 不得输出超长整段；
- 不得把标题与正文粘连在同一行。

因此模型容易生成 `###3.1全域数据湖建设北海电厂...` 这样的非标准 Markdown。

## 3. 修正目标

### 3.1 页面展示目标

长文档生成期间，页面必须稳定显示规划大纲，而不是只显示步骤摘要。推荐展示形态：

- 在 Agent 中间步骤折叠树上方或答案上方显示“文档规划”卡片；
- 卡片展示 H1 标题和所有 H2 章节；
- 答案开始生成后仍保留卡片，不被折叠树隐藏；
- 会话完成后，历史记录重新加载仍能展示该大纲；
- 不把规划大纲混入最终正文，避免正文重复。

### 3.2 正文排版目标

正文必须符合 Markdown 和投标文档阅读习惯：

- H2 章节由后端按大纲统一发出；章节正文不得重复 H2；
- 章节内部允许使用 H3/H4，但必须为 `### 3.1 全域数据湖建设` 这种格式；
- 标题必须独占一行，标题后空一行再写正文；
- 正文段落不宜过长，复杂能力用项目符号或编号拆分；
- 适合甲方/业务部门/技术专家阅读，表达要正式、结构化、可扫描；
- 最终 artifact、页面渲染、复制导出都不应出现 `###3.1...` 这类标题。

## 4. 后端修正方案

### 4.1 新增独立大纲展示事件，不复用进度替换事件

在 `fullDocumentProgressReporter` 中新增独立方法，例如：

```go
func (r *fullDocumentProgressReporter) PublishOutline(outline dedicatedFullDocumentOutline) {
    outlineMarkdown := formatFullDocumentOutlineMarkdown(outline)
    eventID := generateEventID("document-outline")
    _ = r.eventBus.Emit(r.ctx, event.Event{
        ID:        eventID,
        Type:      event.EventAgentThought,
        SessionID: r.sessionID,
        Data: event.AgentThoughtData{
            Content:   outlineMarkdown,
            Iteration: 0,
            Done:      true,
            Replace:   false,
            Synthetic: true,
            Stage:     "planning",
        },
    })
}
```

关键点：

- 使用新的 event id，不和 `document-outline-progress` 共用；
- `Replace=false`，防止被后续检索/生成进度覆盖；
- `Done=true`，表示大纲卡片是稳定内容，不再继续流式追加；
- `Content` 必须是纯大纲 Markdown，第一行 H1，后续 H2，不加“已完成大纲如下”等非标题文本，这样前端现有 `extractPlanningOutlineFromText` 能直接解析。

### 4.2 在大纲校验通过后发布大纲

在两个首轮完整文档路径中调用：

- `runKnowledgeGroundedFullDocumentGenerationPath`：大纲生成并通过 `generateValidatedFullDocumentOutline` 后、创建 generation run 前发布；
- `runDedicatedFullDocumentGenerationPath`：大纲生成并通过校验后、进入章节循环前发布。

示例位置：

```go
outline, err := generateValidatedFullDocumentOutline(...)
if err != nil { ... }
progress.PublishOutline(outline)
```

这样用户在章节正文开始前就能看到完整规划。

### 4.3 将大纲写入完成事件 Extra，支持历史回放

当前知识库路径已经在 `buildKnowledgeGroundedGenerationRunExtra` 中带有 outline 信息，建议统一保证 `emitFullDocumentCompletion` 的 `Extra` 中包含：

```json
{
  "outline": {
    "title": "北海电厂二期智慧电厂项目技术方案",
    "sections": ["项目背景与建设目标", "总体技术架构", "数据湖与基础算力平台", "智慧运行系统", "智慧安防系统", "智能安全监控应急中心", "AR眼镜智能作业", "实施与运维保障"]
  },
  "document_generation_status": "completed"
}
```

前端历史加载时，如果 stream 中没有大纲 thought，也可以从 `agent_complete.extra.outline` 或 artifact 元数据中恢复展示。

### 4.4 增强章节 Prompt 的排版约束

建议抽取共享函数，供 `buildDedicatedFullDocumentSectionMessages` 和 `buildKnowledgeGroundedFullDocumentSectionMessages` 复用：

```go
func fullDocumentSectionMarkdownStyleInstructions() string {
    return strings.Join([]string{
        "Markdown layout requirements:",
        "1. Use polished proposal-style writing suitable for customer-facing technical bids.",
        "2. Do not repeat the current H2 section heading; it has already been emitted.",
        "3. Use H3/H4 headings only when they improve readability.",
        "4. Every markdown heading must be on its own line and must contain exactly one space after heading markers, for example: ### 3.1 全域数据湖建设.",
        "5. Never write malformed headings such as ###3.1标题, ### 标题正文粘连, or headings and body text on the same line.",
        "6. Add one blank line after each heading before body paragraphs.",
        "7. Keep paragraphs concise; prefer 2-4 sentences per paragraph.",
        "8. Use bullet lists for capability groups, implementation steps, constraints, and value points.",
        "9. Use bold lead-in labels such as **建设内容：** or **技术价值：** when it helps scanning.",
        "10. Do not output HTML, code fences, JSON, hidden reasoning, or raw prompt text.",
    }, " ")
}
```

并在章节 system prompt 和 user prompt 末尾都补充一次，降低模型忽略概率：

```go
systemPrompt.WriteString(fullDocumentSectionMarkdownStyleInstructions())
userContent.WriteString("\n\n")
userContent.WriteString(fullDocumentSectionMarkdownStyleInstructions())
```

### 4.5 增加标题格式轻量校验与修复

仅靠 Prompt 不能完全保证格式，因此建议增加轻量校验：

```go
var malformedMarkdownHeadingRE = regexp.MustCompile(`(?m)^#{1,6}[^#\s].*$`)
var overlongMarkdownHeadingRE = regexp.MustCompile(`(?m)^#{1,6}\s+.{80,}$`)
```

处理策略分两层：

1. **自动修复可确定问题**：`###3.1 全域数据湖建设` 修复为 `### 3.1 全域数据湖建设`；`####能力说明` 修复为 `#### 能力说明`。
2. **质量标记不可确定问题**：标题行超过 80 字、疑似标题正文粘连时，记录 `quality_issue=malformed_heading_or_body_joined`，必要时将 artifact 标记为 `needs_review`，避免正式导出时误认为质量完全达标。

为避免流式显示和最终 artifact 不一致，建议同时做：

- 后端最终 `finalAnswer` 入库前执行一次 `normalizeFullDocumentMarkdownForArtifact`；
- 前端 Markdown 渲染前执行同样的标题空格修复，保证流式阶段也能尽量正常显示。

## 5. 前端修正方案

### 5.1 新增独立“文档规划”卡片

在 [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue) 中新增计算属性：

```ts
const visiblePlanningOutline = computed(() => {
  // 1. 优先从 thinking event 中解析完整大纲
  // 2. 其次从 agent_complete.extra.outline 恢复
  // 3. 最后从 session.chat_document_artifact 或 final_document_content 的 H1/H2 中恢复
});
```

模板中将该卡片放在折叠树之前：

```vue
<PlanningOutlineCard
  v-if="visiblePlanningOutline"
  :outline="visiblePlanningOutline"
  class="full-document-planning-card"
/>
```

卡片内容：

- 标题：`文档规划`；
- 状态：规划中/已完成；
- H1 标题；
- H2 章节列表；
- 章节数量，例如 `8 个章节`。

这比把大纲藏在“思考”折叠卡里更符合长文档任务的用户预期。

### 5.2 避免重复展示同一个大纲

当前 `intermediateEvents` 会展示 thinking 事件。新增大纲卡片后，需要避免同一个 outline 既出现在规划卡片，又出现在折叠树里。

建议新增 `hiddenPlanningOutlineEventIds`：

```ts
const hiddenPlanningOutlineEventIds = computed(() => {
  const hidden = new Set<string>();
  for (const event of eventStream.value || []) {
    if (event.type === 'thinking' && hasPlanningOutline(event) && event.synthetic) {
      hidden.add(event.event_id);
    }
  }
  return hidden;
});
```

然后在 `intermediateEvents` 过滤掉这些大纲专用事件。普通阶段进度仍保留在折叠树中。

### 5.3 历史回放也显示大纲

如果历史消息的 `agentEventStream` 没有完整 thinking 事件，前端应从以下字段恢复：

1. `agent_complete.extra.outline`；
2. `session.chat_document_artifact.structure_info.heading_titles`；
3. `session.final_document_content` 中解析 H1/H2。

优先级以“结构化 outline”为准，避免从正文中误提取 H3/H4。

### 5.4 渲染前修复 Markdown 标题空格

当前 `preprocessMarkdown` 只处理图片和引用标签，没有修复 `###3.1`。建议新增：

```ts
const normalizeMarkdownHeadings = (content: string): string => {
  return content
    .replace(/\r\n?/g, '\n')
    .replace(/(^|\n)(#{1,6})(?![#\s])([^\n]+)/g, '$1$2 $3')
    .replace(/([^\n])\n?(#{1,6}\s+)/g, '$1\n\n$2')
    .replace(/(#{1,6}\s+[^\n]+)\n(?!\n)/g, '$1\n\n');
};
```

并在 `renderMarkdownContent` 或 `preprocessMarkdown` 的入口调用。这样即使模型偶发输出 `###3.1全域数据湖建设`，页面也能至少渲染成 H3，而不是原样文本。

注意：前端修复只能解决显示问题，不能替代后端 artifact 内容质量修复。正式复制、导出、复用上一版文档时，仍应以后端归一化后的 snapshot 为准。

## 6. 推荐排版 Prompt 模板

建议章节 Prompt 追加如下中文版本，便于中文投标方案稳定输出：

```text
## 排版与行文要求
- 本节正文用于提交甲方的投标技术方案，请使用正式、清晰、专业的技术方案文风。
- 当前 H2 章节标题已经由系统输出，请不要重复输出 H1/H2 标题。
- 如需拆分小节，只能使用 H3/H4 标题；标题必须独占一行，且 `###` 或 `####` 后必须有一个空格。
- 正确示例：`### 3.1 全域数据湖建设`。错误示例：`###3.1全域数据湖建设`、`### 3.1 全域数据湖建设正文内容...`。
- 每个标题后必须空一行再写正文。
- 每个自然段控制在 2-4 句，避免大段堆叠。
- 对平台能力、建设内容、技术路径、业务价值、保障措施等内容，优先使用项目符号拆分。
- 可使用加粗标签提升可读性，例如：`**建设内容：**`、`**技术价值：**`、`**实施要点：**`。
- 不要输出 HTML、代码块、JSON、提示词说明或隐藏推理。
```

针对“数据湖与基础算力平台”这类内容，期望输出示例：

```markdown
### 3.1 全域数据湖建设

围绕北海电厂二期生产、设备、安防、经营等多源数据，建设统一的数据接入与治理体系，形成覆盖采集、清洗、存储、服务和共享的数据底座。

- **数据接入：** 对接 DCS、SIS、NCS、视频监控、门禁、消防、环境传感器等系统，补齐 SIS 系统外的数据缺口。
- **数据治理：** 建立标准化数据管控体系，统一数据开发规范、质量治理流程、标签管理和命名规则。
- **数据服务：** 通过服务目录、API 网关和消息机制，为智慧运行、智慧安防、应急中心等应用提供标准接口。

### 3.2 弹性基础算力平台

面向 AI 模型训练、实时推理和大数据分析任务，构建可弹性调度的基础算力资源池，支撑智慧电厂各类智能化应用持续运行。
```

## 7. 实施步骤

### 阶段一：后端大纲事件与 Prompt 修复

1. 新增 `formatFullDocumentOutlineMarkdown(outline)`，统一输出 H1/H2 大纲。
2. 新增 `PublishOutline(outline)`，使用独立 event id 输出完整大纲 thinking 事件。
3. 在知识库完整文档路径和纯模型完整文档路径的大纲校验后调用 `PublishOutline`。
4. 抽取 `fullDocumentSectionMarkdownStyleInstructions()`，同时注入 dedicated 与 knowledge-grounded 两条章节 Prompt。
5. 增加 `normalizeFullDocumentMarkdownForArtifact` 与 malformed heading 检测，保证最终 snapshot 不保留明显错误标题。

### 阶段二：前端规划卡片与渲染修复

1. 新增 `PlanningOutlineCard.vue`，或在 `AgentStreamDisplay.vue` 中先实现轻量内联卡片。
2. 新增 `visiblePlanningOutline` 计算属性，优先从 thinking 事件解析，fallback 到 complete extra / artifact / final document。
3. 将规划卡片放在折叠树和答案之前，长文档生成期间保持可见。
4. 新增 `hiddenPlanningOutlineEventIds`，避免大纲 thinking 事件在折叠树中重复显示。
5. 在 Markdown 渲染入口增加 `normalizeMarkdownHeadings`，修复 `###3.1` 不渲染问题。

### 阶段三：质量校验与回归测试

1. 后端单测：
   - `formatFullDocumentOutlineMarkdown` 输出严格 H1/H2；
   - `PublishOutline` 事件 `Replace=false`、`Done=true`、`Stage=planning`；
   - 章节 Prompt 包含标题格式、空行、段落、列表要求；
   - `normalizeFullDocumentMarkdownForArtifact` 能修复 `###3.1标题`。
2. 前端单测或组件测试：
   - answer 开始后规划卡片仍可见；
   - 折叠树默认收起时规划卡片不消失；
   - `###3.1全域数据湖建设` 能渲染为 H3；
   - 历史消息仅有 `agent_complete.extra.outline` 时仍能显示规划卡片。
3. 手工验证：
   - 重新执行投标技术方案生成；
   - 截图确认规划卡片展示完整 8 个章节；
   - 截图确认“数据湖与基础算力平台”下 H3 标题正确渲染；
   - 导出/复制内容中不再出现 `###3.1`。

## 8. 验收标准

1. 在长文档生成的规划阶段，页面能直接看到 H1 和全部 H2 大纲。
2. 当答案开始生成、折叠树出现后，大纲仍在页面上方可见。
3. 会话完成后刷新历史，大纲仍能恢复展示。
4. “数据湖与基础算力平台”等章节不再出现标题与正文粘连。
5. 页面渲染、artifact snapshot、复制导出内容均不出现 `###3.1`、`####标题` 等非标准标题。
6. 章节正文具有更好的可读性：小标题、短段落、加粗标签、列表化能力说明齐备。
7. 原有长文档完成状态、artifact 注册、自动续写判断不回退。

## 9. 风险与兼容性

1. **大纲重复展示风险**：新增大纲卡片后，需要隐藏对应 thinking 事件，避免卡片和折叠树重复。
2. **历史数据兼容风险**：旧消息可能没有 `extra.outline`，因此必须保留从最终正文解析 H1/H2 的 fallback。
3. **标题自动修复风险**：正则只能修复明确的 Markdown 标题空格问题；标题正文粘连仍主要依赖 Prompt 和质量检测，不能盲目截断正文。
4. **流式体验风险**：如果后端对章节内容做完整缓冲后再修复，会牺牲流式体验。推荐短期采用“Prompt 约束 + 前端渲染修复 + 最终 snapshot 修复”的组合，后续再评估章节级缓冲校验。
5. **样式回归风险**：新增规划卡片需要在桌面和移动端验证，避免挤占正文空间或与折叠树样式冲突。

## 10. 推荐落地优先级

| 优先级 | 修正项 | 原因 |
| --- | --- | --- |
| P0 | 后端独立发布完整大纲事件 | 没有完整大纲事件，前端无法稳定展示规划内容 |
| P0 | 前端新增常驻规划卡片 | 解决用户截图中只看到“已完成 1 个步骤”的核心问题 |
| P0 | 章节 Prompt 增加强排版约束 | 从源头减少 `###3.1` 和大段堆叠 |
| P1 | 前端 Markdown 渲染前修复标题空格 | 兜底解决存量/流式阶段标题不显示 |
| P1 | 后端 artifact snapshot 标题归一化和质量标记 | 保证复制、导出、后续续写基线质量 |
| P2 | 章节级格式校验失败后自动重写 | 提升正式交付质量，但成本和延迟更高 |

综合建议：先按 P0 + P1 落地，能够快速解决本次两个用户可见问题；P2 作为后续质量增强，避免一次改动过大影响长文档生成稳定性。
