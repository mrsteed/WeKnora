# 北海电厂技术方案长文档模式未触发审查与解决方案

## 一、结论

本次请求“请输出北海电厂的技术方案”没有触发长文档模式，直接原因是：后端统一路由模型在固定 4 秒超时时间内未返回，随后 `fallbackChatRouteDecision` 兜底为普通 `agent_qa`，且兜底结果明确 `use_long_document=false`、`need_artifact=false`，所以最终进入普通 Agent 工具调用流程。

结合 `my_docs/20260513-04-统一模型路由触发条件审查与修正方案.md` 的架构结论，本问题不应通过“前端补长文档正则”解决。前端应退出自然语言意图判定，只传用户 prompt 与显式结构化信号；是否进入普通问答、普通 Agent、短文档、长文档或文档续写，应由后端统一路由负责。

本次修复建议收敛为三点：

1. 前端删除或停用基于 query 文本的长文档自然语言判定。
2. 后端意图识别模型调用不应使用独立硬编码 4 秒超时，应按通用大模型调用超时时间执行。
3. 当模型路由失败、超时或返回非法 JSON 时，后端启用严格受保护的正则兜底；仅对“生成动作 + 正式长文档类型”的高置信请求升级到长文档。

## 三、结合统一路由方案后的代码审查结论

### 1. 前端应删除或停用长文档自然语言判定职责

`20260513-04` 已明确：前端应只上传用户 prompt 和显式结构化交互信号，不应根据 query 文本推断 `continue_document`、`revise_document`、`full_document`。

因此，前端侧应删除或停用基于 query 文本的长文档判定逻辑，尤其是把“技术方案”“完整方案”等自然语言关键词映射为 `document_output_mode=full_document` 的逻辑。

基于这个原则，本文件不再建议修改 `frontend/src/views/chat/index.vue` 的 `inferDocumentOutputMode` 来补充“技术方案”正则。这样做虽然能让本次样例命中，但会继续扩大前端自然语言规则，形成与后端模型路由、后端正则兜底并行维护的第三套路由逻辑，后续仍会出现分歧。

前端允许保留和传递的是显式信号，例如：

- 用户点击了“继续上一版”按钮。
- 用户显式选择了某个可续写 artifact。
- 用户显式打开未来的“长文档模式”开关。
- 用户选择了知识库、文件、Agent、MCP、附件和图片。

但“请输出北海电厂的技术方案”这类自然语言意图，应由后端统一判定。

### 2. 当前后端路由模型超时时间过短且硬编码

当前 `internal/application/service/chat_route_service.go` 中存在独立固定值：

```go
const chatRouteDecisionTimeout = 4 * time.Second
```

日志显示 deepseek-v4-pro 的非流式 JSON 路由调用在 4 秒内未完成，导致路由模型没有机会返回 `knowledge_grounded_full_document`。

仓库已有通用大模型调用超时配置：

- `internal/config/config.go` 中 `AgentConfig.LLMCallTimeout`
- 环境变量覆盖：`WEKNORA_AGENT_LLM_TIMEOUT`
- 注释说明默认用于单次 LLM 调用，默认值通常为 120 秒，可按环境覆盖

因此，路由模型也应纳入通用 LLM 调用超时体系，而不是维护一个过短的独立 4 秒常量。

### 3. 当前失败兜底只会回到普通 AgentQA

`chat_route_service.go` 中 `fallbackChatRouteDecision` 当前逻辑是：

```go
kind := types.ChatRouteNormalQA
useAgent := false
if input.AgentModeEnabledByConfig && strings.TrimSpace(input.EndpointMode) == "agent_qa" {
    kind = types.ChatRouteAgentQA
    useAgent = true
}
...
UseLongDocument: false,
NeedArtifact:    false,
```

这对普通问答是安全的，但对本次“输出技术方案”这类显式生成正式文档的请求不够完整。模型路由失败时，后端没有第二层本地语义兜底，导致本应进入知识增强长文档的请求被降级成普通 AgentQA。

### 4. 后端已有显式长文档意图规则，但使用位置太晚

`internal/application/service/session_agent_qa.go` 中已有显式长文档意图识别：

```go
var explicitFullDocumentIntentRE = regexp.MustCompile(`(?i)((输出|生成|撰写|编写|写(?:一份|一篇)?|整理|形成|给我|提供).{0,40}(完整(?:版)?|全文|文档|方案|报告|技术方案|设计方案|标书|投标方案|实施方案|markdown))|((完整(?:版)?|全文).{0,20}(文档|方案|报告|技术方案|设计方案|标书|markdown))`)
```

该正则可以覆盖“请输出北海电厂的技术方案”。问题在于它只在 `DocumentOutputMode == full_document` 之后才参与服务层专用路径判断：

- `shouldUseKnowledgeGroundedFullDocumentGenerationPath`
- `shouldUseDedicatedFullDocumentGenerationPath`

本次请求没有 `document_output_mode=full_document`，所以服务层正则没有机会发挥兜底作用。正确位置应前移到统一路由失败兜底阶段。

### 5. 如果路由判为知识增强长文档，本次请求满足执行条件

`applyDocumentRouteDecision` 对 `knowledge_grounded_full_document` 的基本条件是：

- query 非空
- 非自动续写
- 没有 base artifact
- 没有附件和图片
- 存在知识库作用域

本次日志中这些条件均满足：请求有明确 query，有 `knowledge_base_ids` 和 effective agent KB，没有图片和附件，也不是续写。因此如果模型路由正常返回，或失败兜底能识别为 `knowledge_grounded_full_document`，后端会设置：

```go
reqCtx.documentOutputMode = types.ChatDocumentOutputModeFull
```

随后 `session_agent_qa.go` 可以进入知识增强长文档生成。

## 四、根因归纳

根因不是前端少写了某个关键词，而是后端统一路由链路缺少稳定的失败兜底：

1. 前端不应根据自然语言判定长文档，本次请求也不应通过补前端正则解决。
2. 后端意图识别模型调用使用固定 4 秒超时，明显短于通用大模型调用超时，导致远程模型稍慢时直接失败。
3. 模型失败后，兜底策略只保守回到普通 AgentQA，没有对“输出/生成/撰写 + 技术方案/实施方案/投标方案/标书/完整报告”等高置信长文档请求启用后端正则兜底。
4. 已有长文档正则位于服务层路径判断之后，无法在 `document_output_mode` 缺失时参与路由。

因此，这类业务上典型的“输出技术方案/实施方案/投标方案”请求，在模型路由超时时，会被误走普通 AgentQA。

## 五、推荐解决方案

### 方案 A：删除或停用前端长文档自然语言判定

本次文档修正后，前端不再承担长文档自然语言判定职责，也不再建议补充如下逻辑：

```js
/(输出|生成|撰写).{0,40}(技术方案|实施方案|投标方案)/
```

前端侧建议与 `20260513-04` 保持一致：

1. 删除或停用“技术方案”等自然语言关键词判定。
2. 删除或停用通过 query 文本推断 `document_output_mode=full_document` 的逻辑。
3. 只传用户原始 `query`、知识库/文件选择、Agent 开关、附件/图片，以及显式 UI 操作产生的结构化信号。
4. 如果未来产品提供“长文档模式”按钮，可传 `requested_route=full_document` 或类似显式字段；这属于用户显式交互信号，不属于前端自然语言判定。

### 方案 B：路由模型超时时间改用通用大模型调用超时

建议重构 `ChatRouteService`，让路由模型调用使用通用大模型调用超时，而不是固定 4 秒。

推荐做法：

1. `chatRouteService` 注入全局配置或一个 timeout resolver。
2. 优先使用 `cfg.Agent.LLMCallTimeout`。
3. 允许通过 `WEKNORA_AGENT_LLM_TIMEOUT` 统一覆盖。
4. 如果配置缺失，再回退到一个合理默认值，例如当前 Agent LLM 默认值，而不是 4 秒。

示意代码：

```go
type chatRouteService struct {
    modelService interfaces.ModelService
    llmCallTimeoutSeconds int
}

func (s *chatRouteService) routeDecisionTimeout() time.Duration {
    if s.llmCallTimeoutSeconds > 0 {
        return time.Duration(s.llmCallTimeoutSeconds) * time.Second
    }
    return defaultAgentLLMCallTimeout()
}

func (s *chatRouteService) Decide(ctx context.Context, input types.ChatRouteInput) (*types.ChatRouteDecision, error) {
    routeCtx, cancel := context.WithTimeout(ctx, s.routeDecisionTimeout())
    defer cancel()
    ...
}
```

这样路由模型与普通 Agent LLM 调用采用同一套超时策略，避免“普通问答可等 120 秒，但路由只等 4 秒”的不一致。

注意：路由模型使用通用超时不意味着用户一定要等待完整超时。可以在后续优化中增加异步观测或快速取消策略，但底层不应再用一个过短硬编码值导致确定性误路由。

### 方案 C：模型路由失败时启用后端正则兜底

当路由模型发生以下情况时，后端应进入正则兜底：

- 模型请求超时
- 模型加载失败
- 模型返回非法 JSON
- 模型返回 unknown kind
- 模型服务不可用

兜底规则必须只在后端执行，并且必须带保护条件。建议新增：

```go
var explicitRouteFullDocumentIntentRE = regexp.MustCompile(`(?i)((输出|生成|撰写|编写|写(?:一份|一篇)?|整理|形成|给我|提供).{0,40}(完整(?:版)?|全文|文档|方案|报告|技术方案|设计方案|标书|投标方案|实施方案|markdown))|((完整(?:版)?|全文).{0,20}(文档|方案|报告|技术方案|设计方案|标书|markdown))`)

func fallbackChatRouteDecision(input types.ChatRouteInput, reason string) *types.ChatRouteDecision {
    if decision := regexFallbackChatRouteDecision(input, reason); decision != nil {
        return decision
    }
    return conservativeChatRouteDecision(input, reason)
}
```

其中 `regexFallbackChatRouteDecision` 必须先做硬约束校验：

```go
func regexFallbackChatRouteDecision(input types.ChatRouteInput, reason string) *types.ChatRouteDecision {
    if !canRegexFallbackToFullDocument(input) {
        return nil
    }

    kind := types.ChatRouteFullDocument
    useKnowledge := false
    if input.HasSelectedKnowledge || input.HasEffectiveAgentKB {
        kind = types.ChatRouteKnowledgeGroundedFullDoc
        useKnowledge = true
    }

    return &types.ChatRouteDecision{
        Kind:            kind,
        Intent:          types.ChatDocumentIntentNormal,
        Operation:       types.ChatDocumentOperationCreate,
        OutputMode:      types.ChatDocumentOutputModeFull,
        UseAgent:        strings.TrimSpace(input.EndpointMode) == "agent_qa" || input.AgentModeEnabledByConfig,
        UseKnowledge:    useKnowledge,
        UseLongDocument: true,
        NeedArtifact:    true,
        Confidence:      0.65,
        Reason:          reason + "; regex_full_document_fallback",
    }
}

func canRegexFallbackToFullDocument(input types.ChatRouteInput) bool {
    if strings.TrimSpace(input.Query) == "" {
        return false
    }
    if input.AutoContinue || strings.TrimSpace(input.ExplicitBaseArtifactID) != "" {
        return false
    }
    if input.HasAttachments || input.HasImages {
        return false
    }
    return explicitRouteFullDocumentIntentRE.MatchString(input.Query)
}
```

这里的正则兜底不是主判定器。正常情况下仍由大模型结构化路由判断；正则只在模型路由失败时启用，用于兜住“请输出/生成/撰写技术方案、实施方案、投标方案、标书、完整报告”等高置信长文档请求。

### 方案 D：保留代码硬约束，防止失败兜底误升级

即使正则命中，也不能跳过现有安全边界：

1. `AutoContinue=true` 时不能新建完整长文档。
2. 存在 `BaseArtifactID` 时优先进入文档编辑/续写语义，不能当成新建长文档。
3. 有附件或图片时，如果专用长文档路径不支持，应回退 generic Agent 或普通 QA。
4. 没有知识库作用域时只能选择 `full_document`，不能选择 `knowledge_grounded_full_document`。
5. 有知识库作用域时优先选择 `knowledge_grounded_full_document`，避免绕过本地证据生成泛化方案。
6. 正则只匹配“生成动作 + 文档类型”，不能把“这个方案有哪些风险”“比较两个方案差异”“方案包含哪些字段”这类问答升级成长文档。

### 方案 E：增强日志可观测性

建议补充三类日志：

1. 路由模型耗时和超时来源：

```text
[ChatRouter] route_model_start model=... timeout_seconds=120 timeout_source=agent_llm_call_timeout
```

2. 正则兜底命中：

```text
[ChatRouter] regex fallback selected kind=knowledge_grounded_full_document reason=route_model_request_failed;regex_full_document_fallback
```

3. 路由未应用原因：

```text
[ChatRouter] document route not applied kind=agent_qa reason=not_document_route
```

或在 `canApplyFullDocumentRouteDecision` 拆出返回原因：

```text
blocked_by=has_attachments | blocked_by=auto_continue | blocked_by=missing_knowledge_scope
```

1. P0：删除或停用前端长文档自然语言判定，不再新增前端正则；前端只保留显式结构化信号。

## 八、风险与边界

1. 正则兜底只能在模型路由失败时启用，不能取代模型路由成为主判定器。
2. 不能把所有包含“方案”的问题都升级成长文档。例如“这个方案有哪些风险？”“比较两个方案差异”更像问答。
3. 正则必须要求“输出/生成/撰写/编写/整理/形成/提供”等生成动作，并匹配正式文档类型。
4. 对附件、图片、续写、修订场景要保持保护条件，避免误把编辑请求当成新建长文档。
5. 有知识库作用域时应优先选择 `knowledge_grounded_full_document`，否则会绕过本地知识证据，生成泛化方案。
6. 路由模型使用通用 LLM 超时后，首轮路由最坏等待时间可能变长，需要通过日志观察耗时，并可后续引入更轻量模型或异步灰度策略优化体验。

## 九、最终建议

本问题建议按“前端退出自然语言判定 + 后端统一模型路由 + 通用 LLM 超时 + 失败正则兜底”的组合方案处理。

对本次“请输出北海电厂的技术方案”而言，最关键的修复不是让前端识别“技术方案”，而是让后端统一路由具备稳定执行能力：模型路由按通用大模型超时时间等待；如果模型仍然失败，则由后端正则兜底识别“输出/生成/撰写 + 技术方案/实施方案/投标方案/标书/完整报告”等高置信长文档请求，在通过附件、图片、续写、artifact、知识库作用域等硬约束校验后，选择 `knowledge_grounded_full_document` 或 `full_document`，并确保进入长文档生成链路。
