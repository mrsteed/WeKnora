# Qwen3-VL-8B Q4 基于 llama.cpp 的 Docker 部署方案

## 1. 方案结论

结合当前 WeKnora 架构与“系统已有 llama.cpp”这一前提，Qwen3-VL-8B Q4 最合适的部署方式不是再单独引入一套新的推理栈，而是：

1. 继续保留 MinerU 作为第一阶段文档解析引擎。
2. 使用现有 llama.cpp 体系中的 `llama-server` 单独拉起一个视觉模型服务实例。
3. 让 WeKnora 以 `VLLM` 远程模型方式，通过 OpenAI 兼容接口调用这个 Qwen3-VL-8B Q4 服务。

这是当前改造最小、与现有代码最一致、也最容易 Docker 化落地的方案。

原因很明确：

1. WeKnora 当前远程 VLM 走的是 OpenAI 兼容 Chat Completions 接口。
2. `internal/models/vlm/remote_api.go` 已经表明，WeKnora 会把图片编码成 `data:image/...;base64,...`，并以 `image_url` 多段消息发送到远程接口。
3. llama.cpp 的 `llama-server` 已经提供 OpenAI 兼容 `/v1/chat/completions` 接口，并明确支持多模态图片输入。
4. llama.cpp 当前代码已经包含 `QWEN3VL` / `PROJECTOR_TYPE_QWEN3VL`，说明底层多模态路径已经覆盖到 Qwen3-VL 架构。

因此，从当前系统架构出发，推荐的主方案是：

“MinerU 做首轮解析，Qwen3-VL-8B Q4 通过 llama.cpp 做第二阶段图像 OCR 与 Caption 增强。”

## 2. 当前系统里的推荐角色分工

结合前一份流程审查文档，建议把职责边界固定成下面这样：

### 2.1 第一阶段：MinerU

MinerU 继续负责：

1. PDF / DOCX / PPTX / 图片等文件的首轮结构化解析。
2. 输出 Markdown。
3. 抽取图片并返回 `ImageRefs`。

### 2.2 第二阶段：Qwen3-VL-8B Q4

Qwen3-VL-8B Q4 通过 llama.cpp 负责：

1. 对解析后抽取出的图片执行 OCR。
2. 对同一张图片生成 Caption。
3. 由 WeKnora 写入图片 OCR chunk 与图片 Caption chunk。

### 2.3 为什么推荐这种分层方案

因为它完全顺着当前代码路径走：

1. `knowledge.go` 负责导入、选引擎、切块、入队多模态任务。
2. `image_multimodal.go` 负责取 VLM 模型并调用远程视觉接口。
3. `vlm/remote_api.go` 负责发 OpenAI 兼容请求。

所以这个方案不要求你新增解析引擎，也不要求你重写 DocReader 或 MinerU 适配器。

## 3. 为什么优先选 llama.cpp，而不是再起一套 vLLM

在你当前环境里，优先选 llama.cpp 有四个直接好处：

1. 现有系统已经有 llama.cpp，运维链路更短。
2. GGUF + Q4 量化模型与 llama.cpp 天然匹配。
3. `llama-server` 直接暴露 OpenAI 兼容接口，刚好能被 WeKnora 复用。
4. Docker 镜像官方已经提供 `server`、`server-cuda`、`server-vulkan`、`server-rocm` 等形态，不需要自己从零构建推理框架。

如果你当前目标是“尽快让 Qwen3-VL-8B Q4 在 WeKnora 里可用”，那么 llama.cpp 是比再引入 vLLM 更低风险的路径。

## 4. 推荐部署模式

## 4.1 推荐模式：为视觉模型单独起一个 llama-server 容器

虽然系统里已经有 llama.cpp，但仍然建议不要把 Qwen3-VL-8B Q4 直接和现有文本模型实例混跑在同一个服务槽位里，而是单独拉一个容器，例如：

- `qwen3-vl-llamacpp`

这样做的原因：

1. 视觉模型通常需要单独的 `mmproj`。
2. 图片请求会显著拉高上下文与显存压力。
3. OCR / Caption 任务和普通文本问答混跑时，更容易互相影响吞吐。
4. WeKnora 中的 VLM 模型名最好固定映射到一个稳定 alias，单独实例更清晰。

因此，推荐模式不是“替换现有 llama.cpp”，而是“在现有 llama.cpp 体系里新增一个专门服务 Qwen3-VL 的 Docker 实例”。

## 4.2 备选模式：复用单一 llama.cpp 多模型服务

llama.cpp 也支持多模型或路由式服务，但对当前 WeKnora 来说，这不是第一选择。

原因是当前 WeKnora 在调用 VLM 时，实际发送的 `model` 字段就是你在模型中心里配置的模型名。因此如果你走多模型路由：

1. 需要额外管理 alias 和路由配置。
2. 需要确保视觉模型实例始终能正确加载 mmproj。
3. 运维排障会比“单模型单容器”更复杂。

所以除非你已经有成熟的 llama.cpp 路由层，否则不建议把 Qwen3-VL-8B Q4 的首版上线放在这个模式上。

## 5. Docker 部署前提

在真正部署前，需要准备以下内容：

1. 一份 Qwen3-VL-8B 的 GGUF 量化权重，建议 Q4 系列。
2. 与之匹配的 `mmproj` 文件，或使用支持自动下载 `mmproj` 的 Hugging Face 仓库。
3. 单独的模型目录，例如：
   - `/data/models/qwen3-vl-8b-q4/`
4. 一张适合推理的 GPU。

### 5.1 资源建议

Qwen3-VL-8B Q4 的“Q4”只降低了文本主权重体积，不代表视觉场景就会非常轻。

实际资源压力还来自：

1. 视觉 projector
2. 图片 token 展开
3. KV cache
4. 并发请求数

建议按下面的经验值规划：

1. 单卡 16GB：可做低并发验证环境。
2. 单卡 24GB 及以上：更适合生产低到中并发。
3. 如果完全 CPU 跑 8B VL，能够跑通，但不建议用于生产 OCR 链路。

## 6. 推荐的 Docker Compose 方案

下面给出一个最适合当前 WeKnora 架构的推荐 Compose 方案。

假设：

1. 你已经有可用的 GGUF 与 `mmproj` 文件。
2. WeKnora 主服务和这个视觉服务处在同一个 Docker 网络。
3. 你准备把 WeKnora 的 VLM BaseURL 指向这个容器。

```yaml
services:
  qwen3-vl-llamacpp:
    image: ghcr.io/ggml-org/llama.cpp:server-cuda
    container_name: qwen3-vl-llamacpp
    restart: unless-stopped
    gpus: all
    ports:
      - "18080:8080"
    volumes:
      - ./models/qwen3-vl-8b-q4:/models
    environment:
      LLAMA_ARG_MODEL: /models/Qwen3-VL-8B-Instruct-Q4_K_M.gguf
      LLAMA_ARG_MMPROJ: /models/mmproj-Qwen3-VL-8B-Instruct-f16.gguf
      LLAMA_ARG_ALIAS: qwen3-vl-8b-q4
      LLAMA_ARG_HOST: 0.0.0.0
      LLAMA_ARG_PORT: "8080"
      LLAMA_ARG_CTX_SIZE: "8192"
      LLAMA_ARG_N_GPU_LAYERS: "99"
      LLAMA_ARG_THREADS: "8"
      LLAMA_ARG_N_PARALLEL: "2"
      LLAMA_ARG_CONT_BATCHING: "true"
      LLAMA_ARG_MMPROJ_OFFLOAD: "true"
      LLAMA_ARG_IMAGE_MIN_TOKENS: "256"
      LLAMA_ARG_IMAGE_MAX_TOKENS: "4096"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
    networks:
      - WeKnora-network

networks:
  WeKnora-network:
    external: true
```

### 6.1 关键参数说明

1. `LLAMA_ARG_MODEL`
   - 指向主 GGUF 权重文件。
2. `LLAMA_ARG_MMPROJ`
   - 指向视觉 projector。
   - 对本地文件加载模式，这是关键参数。
3. `LLAMA_ARG_ALIAS`
   - 强烈建议显式设置。
   - 这个值建议与 WeKnora 里配置的模型名完全一致。
4. `LLAMA_ARG_CTX_SIZE`
   - 视觉任务上下文建议不要太小。
5. `LLAMA_ARG_N_GPU_LAYERS`
   - 若显存足够，可尽量高；`99` 常用于尽可能全量 offload。
6. `LLAMA_ARG_IMAGE_MIN_TOKENS` / `LLAMA_ARG_IMAGE_MAX_TOKENS`
   - 控制动态分辨率图像 token 范围。
   - 对 OCR 质量与显存消耗都会有影响。

## 7. 如果你想用自动下载模式

如果你拿到的是支持 llama.cpp `-hf` 自动下载的 Hugging Face GGUF 仓库，也可以用自动拉取模式，而不手动挂本地模型文件。

这种方式的优点是：

1. 部署更简单。
2. 若仓库同时包含 mmproj，llama.cpp 可以自动处理。

示例：

```yaml
services:
  qwen3-vl-llamacpp:
    image: ghcr.io/ggml-org/llama.cpp:server-cuda
    container_name: qwen3-vl-llamacpp
    restart: unless-stopped
    gpus: all
    ports:
      - "18080:8080"
    environment:
      LLAMA_ARG_HF_REPO: <支持自动下载的GGUF仓库>:<量化文件名或别名>
      LLAMA_ARG_ALIAS: qwen3-vl-8b-q4
      LLAMA_ARG_HOST: 0.0.0.0
      LLAMA_ARG_PORT: "8080"
      LLAMA_ARG_CTX_SIZE: "8192"
      LLAMA_ARG_N_GPU_LAYERS: "99"
    volumes:
      - ./llama-cache:/root/.cache/llama.cpp
    networks:
      - WeKnora-network
```

但需要注意：

1. 上面的 `LLAMA_ARG_HF_REPO` 只是占位示例，不是已经替你验证过的固定仓库名。
2. 具体 GGUF 仓库可用性会变化。
3. 并不是每个仓库都稳定提供 mmproj。
4. 首次拉取耗时会比较长。

所以如果你是生产环境，仍然更推荐本地挂载模型文件，而不是依赖自动下载模式。

## 8. 与 WeKnora 的接入方式

这是整个方案最关键的一步。

当前 WeKnora 的远程 VLM 调用路径是：

1. 在模型中心创建 `VLLM` 模型。
2. 通过 `GetVLMModel()` 构造远程视觉模型对象。
3. 若 `InterfaceType` 为空且来源是远程，默认按 `openai` 处理。
4. `remote_api.go` 用 OpenAI 兼容 chat completions 发送图片与文本。

因此，你在 WeKnora 中应该这样配置模型：

### 8.1 模型中心推荐配置

1. 模型类型：`VLLM`
2. 模型来源：远程模型
3. 模型名称：`qwen3-vl-8b-q4`
4. Base URL：
   - 同 Compose 网络内：`http://qwen3-vl-llamacpp:8080/v1`
   - 若跨主机：`http://<宿主机IP>:18080/v1`
5. API Key：可留空；若前端表单强制要求，可填占位值 `sk-local`
6. Interface Type：`openai`
7. `supports_vision`：应设为 `true`

### 8.2 为什么模型名称必须特别注意

当前 WeKnora 在构造 VLM 请求时，发送到远程接口的 `model` 字段就是模型中心里保存的 `model.Name`。

因此：

1. 如果 llama.cpp 使用 `--alias qwen3-vl-8b-q4`
2. 那么 WeKnora 里这个 VLM 模型的名称也应写成 `qwen3-vl-8b-q4`

否则会出现：

- 远程接口是通的
- 但 `model` 字段不匹配
- 请求返回 model not found 或找不到对应实例

## 9. 与当前“问文档”链路的关系

把这个服务接入之后，当前链路会变成：

1. 文档导入。
2. 按知识库规则决定是否走 MinerU。
3. MinerU 输出 Markdown 与图片。
4. WeKnora 存储图片。
5. WeKnora 的 `image:multimodal` 任务调用 Qwen3-VL-8B Q4 的 llama.cpp 服务。
6. llama.cpp 返回 OCR / Caption 文本。
7. WeKnora 写入图片 OCR chunk 和图片 Caption chunk。

这意味着它与当前架构是完全对齐的：

- 不是把 Qwen3-VL 塞进 parser engine 层。
- 而是把它作为知识库 VLM 后处理层。

## 10. 是否要关闭 MinerU OCR

这是部署时必须一起决定的策略点。

如果你现在把 Qwen3-VL-8B Q4 接进来，而 MinerU 仍然保留 OCR，那么就可能出现：

1. MinerU 首轮 OCR
2. Qwen3-VL 第二轮 OCR

这会导致：

1. 时延更高。
2. GPU 推理成本更高。
3. 结果可能重复。

### 10.1 推荐策略

如果你的目标是“Qwen3-VL 成为主要图片 OCR 增强模型”，建议优先尝试：

```json
{
  "mineru_enable_ocr": false,
  "mineru_enable_table": true,
  "mineru_enable_formula": true
}
```

这样可把职责拆成：

1. MinerU 负责结构、表格、公式、版面抽取。
2. Qwen3-VL 负责图片 OCR 与描述。

这是当前架构下更清晰的责任边界。

## 11. 上线前验证步骤

正式接入 WeKnora 前，建议按下面顺序做验证。

### 11.1 验证 llama.cpp 服务本身

先检查模型是否加载成功：

```bash
curl http://localhost:18080/health
curl http://localhost:18080/v1/models
```

你应该能看到：

1. 服务健康。
2. 返回的模型 id 或 alias 包含 `qwen3-vl-8b-q4`。

### 11.2 验证多模态请求

再直接测试 OpenAI 兼容多模态接口：

```bash
curl -X POST http://localhost:18080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3-vl-8b-q4",
    "messages": [
      {
        "role": "user",
        "content": [
          {"type": "text", "text": "请提取图片中的文字，并用中文概括图片内容"},
          {"type": "image_url", "image_url": {"url": "https://example.com/test.png"}}
        ]
      }
    ],
    "max_tokens": 800
  }'
```

如果这个请求可以返回正常内容，那么对 WeKnora 来说接口已经满足接入条件。

### 11.3 验证 WeKnora 接入

最后在 WeKnora 中：

1. 新建或修改一个 `VLLM` 模型。
2. 绑定到测试知识库的多模态配置。
3. 上传以下四类样本：
   - 扫描 PDF
   - 图文混排 DOCX
   - 复杂表格截图
   - 普通图片
4. 观察是否生成：
   - 图片 OCR chunk
   - 图片 Caption chunk

## 12. 推荐的上线方案

如果你现在要落生产，我建议分两步：

### 第一步：先单容器灰度

1. 用单独的 `qwen3-vl-llamacpp` 容器上线。
2. 只绑定一个测试知识库。
3. 优先验证 OCR 质量、Caption 质量、单页耗时和显存占用。

### 第二步：再扩大范围

1. 根据结果决定是否关闭 MinerU OCR。
2. 再扩到更多知识库。
3. 若并发增加，再考虑单独扩容视觉推理节点。

## 13. 最终建议

对你当前的系统状态，最终建议很明确：

1. 不要把 Qwen3-VL-8B Q4 当成新的 parser engine 来改造。
2. 直接利用现有 llama.cpp，用 Docker 单独起一个 `llama-server` 视觉实例。
3. 让 WeKnora 以远程 `VLLM` + `openai` 接口方式接入。
4. 若要减少重复 OCR，优先考虑关闭 MinerU OCR，仅保留 MinerU 的结构化抽取能力。

在当前仓库架构下，这就是最省改造、最符合现有链路、也最容易上线的 Qwen3-VL-8B Q4 部署方案。
