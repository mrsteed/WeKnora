## 3. 同步策略与裁决原则

### 3.1 本轮明确采用 merge

本轮主路径采用：

```bash
git merge upstream/main
```

而不是：

```bash
git rebase upstream/main
```

原因如下：

1. 当前差异规模为 `63 / 158`，已经超出逐提交 `rebase` 的低成本窗口。
2. 本地有过一次 2026-06-08 的上游同步整合历史，继续使用 `merge` 更符合当前分支演进形态。
3. 冲突会集中在权限、共享、长文档、数据库和前端交互等高耦合链路，集中式合并更利于统一裁决。
4. `merge` 更适合作为一次可审计的整合事件保留在历史中。

### 3.2 核心裁决原则

本轮同步以“本地功能优先、上游修复补入”为总原则，具体规则如下：

1. 本地产品行为、本地业务流程、本地权限模型冲突时，以本地为准。
2. 上游纯新增能力、公共修复、安全修复、依赖升级和兼容性增强，只要不破坏本地语义，应吸收。
3. 上游安全修复不能直接丢弃；若与本地逻辑冲突，必须以本地代码为基底手动补入安全逻辑。
4. 对配置、依赖、国际化、版本文件等公共文件，必须做显式合并，不能整文件覆盖。

### 3.3 权限冲突的特别原则

本轮虽然不单独做完整 RBAC 重构，但权限冲突处理必须遵循下列原则：

1. Tenant RBAC 相关公共结构可以吸收，但不能直接替换本地 Organization 共享空间语义。
2. 本地组织成员、共享关系、知识库归属和智能体归属控制必须保留。
3. 任何会导致历史成员、共享关系、API Key 或跨租户管理路径直接进入 `403` 的改动，都必须先有兼容层。
4. 若上游默认要求强制启用鉴权，必须在角色回填和回归验证后再启用，不能在同步时直接开启。

### 3.4 禁止事项

执行过程中禁止以下做法：

1. 直接在 `dev` 上执行合并并提交。
2. 直接对全量冲突文件批量使用 `git checkout --theirs` 或 `git checkout --ours`。
3. 在未建立备份锚点前开始同步。
4. 在未完成验证前直接更新 `origin/dev`。
5. 在未核对远端配置的情况下执行 `git push`。
6. 默认使用 `git push --force-with-lease` 重写 `dev` 历史。

## 4. 执行前检查清单

正式同步前，必须按以下顺序完成检查。

### 4.1 校验仓库状态

```bash
git status --short --branch
git remote -v
git branch -vv
git config --get branch.dev.remote
git config --get branch.dev.merge
git config --get remote.pushDefault
```

通过标准：

1. 当前位于仓库根目录。
2. `dev` 存在且跟踪 `origin/dev`。
3. `origin` 指向本地仓库远端。
4. `upstream` 已配置为 `Tencent/WeKnora`。
5. `remote.pushDefault=origin`。

### 4.2 修正远端配置

若 `upstream` 尚未配置，执行：

```bash
git remote add upstream https://github.com/Tencent/WeKnora.git
git config remote.pushDefault origin
git branch --set-upstream-to=origin/dev dev
```

若 `upstream` 已存在但 URL 不正确，执行：

```bash
git remote set-url upstream https://github.com/Tencent/WeKnora.git
git config remote.pushDefault origin
git branch --set-upstream-to=origin/dev dev
```

执行后再复核：

```bash
git remote -v
git branch -vv
git config --get remote.pushDefault
```

## 5. 正式同步步骤

### 5.1 创建临时同步分支

```bash
git checkout -b sync/dev-20260629-main dev
```

要求：

1. 临时分支必须从当前最新 `dev` 切出。
2. 后续所有冲突解决、验证、补丁修复都只在该分支进行。

### 5.2 执行合并

```bash
git merge upstream/main --no-ff --no-edit
```

可能出现三种结果：

1. 无冲突直接合并成功。
2. 存在冲突，进入手工裁决阶段。
3. 自动合并成功但仍需后续人工补丁修复和验证。

### 5.3 记录冲突清单

进入冲突状态后，先不要急着修改代码，先记录全量冲突文件：

```bash
git diff --name-only --diff-filter=U
git status --short
```

建议同时按主题做一次快速归类：

```bash
git diff --name-only --diff-filter=U | rg "organization|rbac|agent|share|knowledge|docreader|duckdb|export|timeline|mcp|opensearch|StorageEngine|zh-CN|go.mod|go.sum|package.json|pnpm-lock"
```

归类目的不是拖延处理，而是先判断冲突是否集中在以下高风险域：

1. 权限与组织。
2. 知识库与共享。
3. 智能体、MCP 与数据库链路。
4. DuckDB / OpenSearch / 检索引擎。
5. 前端设置页、分享页和国际化。

## 6. 冲突处理执行规则

### 6.1 总体处理顺序

建议严格按下列顺序解决冲突，避免前面修完、后面又被依赖改写：

1. 依赖与构建文件。
2. 后端基础设施与中间件。
3. 权限、组织、知识库、智能体主链路。
4. 配置模板与默认值。
5. 前端页面、状态管理、国际化文案。
6. 文档、版本号、示例文件。

### 6.2 文件类型裁决矩阵

| 文件类型 | 处理策略 |
|---|---|
| 本地业务逻辑文件 | 以本地实现为基底，补入上游新增字段、参数、守卫和修复 |
| 上游纯新增文件 | 直接吸收上游 |
| 安全修复相关文件 | 不能直接丢弃上游改动，需在本地逻辑中手工融合 |
| `go.mod` / `go.sum` | 保留双方必要依赖，整理依赖后再验证 |
| `package.json` / 锁文件 | 合并依赖版本与脚本，禁止无理由回退上游升级 |
| 国际化文案文件 | 合并双方 key，不能整文件覆盖 |
| 配置样例文件 | 保留本地定制项，同时补入上游新配置项 |
| `CHANGELOG.md` / `VERSION` | 优先采用上游版本信息，同时确认本地是否需要保留附加说明 |

### 6.3 单文件冲突处理建议流程

对每个冲突文件，统一按以下流程处理：

1. 先识别该文件承载的是“本地业务语义”还是“上游公共修复”。
2. 若上游只是结构重排、参数重命名或控制流调整，则把本地行为迁移到上游新结构上。
3. 若上游引入新 guard、新字段或新配置项，先接入结构，再补回本地行为。
4. 处理完单文件后立即 `git add`，不要堆积大量未暂存冲突。

操作模板：

```bash
# 编辑并解决冲突
git add <resolved-file>

# 查看剩余冲突
git diff --name-only --diff-filter=U
```

### 6.4 本地优先的重点模块

以下模块出现冲突时，应优先保护本地语义：

1. 长文档输出链路。
2. 结构化数据库、DuckDB、数据库知识库接入链路。
3. 对话导出链路。
4. Organization、共享空间、成员、权限、跨租户协作链路。
5. 智能体分享、共享 Agent 访问、分享设置页。
6. 管理员增强能力和本地定制路由。
7. 设置页、知识库列表、登录页等已定制前端页面。

### 6.5 上游必须补入的重点模块

以下模块即便与本地产生冲突，也必须做“基于本地的手工补入”：

1. SSRF 和外部 HTTP 访问安全限制。
2. 租户验证与权限边界补丁。
3. DuckDB 启动修复和检索基础设施修复。
4. 前端集成入口、引导和通用交互优化。
5. 上游版本推进涉及的依赖与公共配置变更。

## 7. 模块级处理建议

### 7.1 依赖与构建文件

优先处理：

1. `go.mod`
2. `go.sum`
3. 前端依赖文件和锁文件

处理要求：

1. 保留上游新增依赖和版本推进。
2. 保留本地仍在使用的依赖。
3. 避免因手工删除导致后续编译阶段才暴露缺失包问题。

建议命令：

```bash
git diff --name-only --diff-filter=U
go mod tidy
cd frontend && pnpm install --frozen-lockfile
```

### 7.2 后端基础设施与配置

优先关注：

1. `internal/container`
2. `internal/router`
3. `config`
4. `migrations`

处理原则：

1. 先恢复所有必要的 provider、路由注册和配置结构。
2. RBAC 路由与 Organization 路由并存，不能互相覆盖。
3. 迁移编号必须唯一，不能与本地迁移冲突。

建议检查命令：

```bash
rg -n "Provide\(|Register|New.*Service|enable_rbac|registration_mode|retention_days" internal config
ls migrations/versioned | sort | sed -n '1,160p'
```

### 7.3 权限、Organization 与共享链路

优先关注：

1. `internal/application/service/organization.go`
2. `internal/application/service/kbshare.go`
3. `internal/application/service/agent_share.go`
4. `internal/handler/knowledge.go`
5. `internal/middleware/auth.go`
6. `internal/middleware/rbac.go`
7. `internal/middleware/kb_access.go`
8. `internal/router/rbac.go`

处理原则：

1. 保留本地 Organization 共享空间和成员管理逻辑。
2. 保留知识库共享权限计算和共享 Agent 只读访问路径。
3. 吸收上游守卫结构和安全校验，但不要让既有共享路径被直接切断。
4. API Key 和跨租户切换兼容路径不能回退。

建议复核命令：

```bash
rg -n "enable_rbac|tenant_members|RequireRole|RequireOwnershipOrRole|creator_id|organization_members|kb_shares|agent_shares|CanAccessAllTenants|EnableCrossTenantAccess" internal config migrations frontend
```

### 7.4 知识库、检索、DuckDB 和数据库能力

优先关注：

1. 文档解析与时间线链路。
2. 检索引擎、OpenSearch、Embedding 安全相关改动。
3. DuckDB 和数据库查询知识库链路。

处理原则：

1. 吸收上游检索和安全修复。
2. 保留本地数据库知识库和长文档业务语义。
3. 保证本地数据源类型、查询流程和结果展示不回退。

### 7.5 智能体、MCP 与分享页

优先关注：

1. MCP 传输支持和智能体调用链。
2. 智能体共享、共享 Agent 配置和分享页细节。

处理原则：

1. 吸收上游 MCP 公共能力。
2. 保留本地共享 Agent 访问控制和本地页面细节。

### 7.6 前端页面与国际化

优先关注：

1. 设置页。
2. 集成入口和卡片交互。
3. 知识库列表、登录页、组织页、分享页。
4. `zh-CN` 等国际化文案文件。

处理原则：

1. 上游公共交互可吸收。
2. 本地定制页面行为和展示不能回退。
3. 国际化 key 只能增量合并，不能整文件覆盖。

5. 智能体共享、共享 Agent 停用和恢复仍可用。
6. API Key 调用路径仍可访问所属租户资源。
7. 跨租户切换能力仍受本地开关和目标租户校验控制。
8. 上游前端集成入口和通用交互改进已进入本地页面。
9. 上游安全修复，特别是 Embedding 侧 SSRF 防护逻辑，已实际进入本地代码路径。

### 8.4 验证失败时的处理原则

如果验证失败，按以下顺序处理：

1. 先修复当前合并直接导致的编译和测试错误。
2. 再修复由于上游结构调整导致的本地调用链断裂。
3. 对环境依赖型失败，明确记录是外部依赖缺失还是代码回归。

不允许在未完成最小验证的情况下直接推进到 `dev`。

## 9. 收口与提交流程

### 9.1 推送同步分支供复核

验证通过后，先推送临时同步分支：

```bash
git push -u origin sync/dev-20260629-main
```

如果最终走的是重建路径，则推送：

```bash
git push -u origin sync/dev-20260629-rebuild
```

### 9.2 受控更新 `dev`

原则上不直接重写 `dev` 历史，优先采用可快进更新：

```bash
git checkout dev
git pull --ff-only origin dev
git merge --ff-only sync/dev-20260629-main
git push origin dev
```

如果同步期间 `origin/dev` 又有新提交，则先将同步分支与最新 `origin/dev` 再次对齐并重新验证，然后再更新 `dev`。

### 9.3 仅在明确审批后才允许强推

只有在团队明确确认需要重写 `dev` 历史时，才考虑：

```bash
git push --force-with-lease origin HEAD:dev
```

本轮默认不采用此路径。

## 10. 风险与回滚

| 风险 | 表现 | 预防措施 | 回滚措施 |
|---|---|---|---|
| 未先补齐 `upstream` | 同步命令误用 `FETCH_HEAD` 或无法复现 | 正式同步前先固定远端配置 | 回到 `dev`，修正远端后重做 |
| 工作区脏状态直接合并 | 把本地未固化修改混入同步结果 | 同步前先提交或 stash | 重置到 `sync-dev-20260629-pre` 或 backup 分支 |
| 安全修复遗漏 | 功能保住但留下已知风险 | 冲突时优先补入上游安全逻辑 | 回到同步分支修补并重测 |
| 权限链路被上游覆盖 | 组织成员、共享、API Key、跨租户能力失效 | 权限模块单独复核与回归 | 回退同步分支或恢复本地权限链路 |
| 前端公共更新被本地整文件覆盖 | 丢失上游交互与版本修复 | 前端文件逐块合并而不是整文件覆盖 | 对比上游重新补丁合并 |
| 直接更新 `dev` 后发现回归 | 影响后续开发基线 | 先推同步分支复核再入 `dev` | 使用 backup 分支或 tag 快速回退 |

## 11. 最小可交付版本

如果一次性完整吸收风险过高，可以先交付最小版本：

1. 先完成远端修正、工作区冻结和同步分支建立。
2. 吸收上游安全修复、公共依赖更新和前端公共交互改进。
3. 保持本地 Organization、共享空间、长文档、数据库和 API Key 行为不变。
4. 完成后端最小测试和前端构建验证。
5. 先推送同步分支供复核，再决定是否继续吸收更深层 RBAC 或结构优化。

这个最小版本的目标不是一次性统一所有结构，而是在不破坏现有业务的前提下，先把必要的上游修复安全落地。

## 12. 最终结论

本轮 2026-06-29 上游同步的关键不是单纯“把代码合进来”，而是按当前真实仓库状态做一次受控整合：

1. 先补齐 `upstream`，并把默认推送目标固定为 `origin`。
2. 先清理工作区并建立备份锚点，再开始同步。
3. 以 `merge upstream/main` 为主路径，在同步分支上集中处理冲突。
4. 冲突处理时吸收上游安全修复和公共能力，但始终以本地业务语义、产品行为和权限设计为准。
5. 只有在后端、前端和关键业务链路验证通过后，才允许把结果推进到本仓库 `dev` 分支。
