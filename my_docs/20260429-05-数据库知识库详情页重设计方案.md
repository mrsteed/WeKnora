# 数据库知识库详情页重设计方案

## 1. 背景与问题

当前知识库详情页由统一页面承载，但页面结构长期围绕文档型知识库设计，核心内容包括：文档上传、标签分类、文档列表、卡片/列表切换以及文档解析相关提示。数据库型知识库虽然已经完成创建、连接测试、数据源维护与 Schema 刷新等能力，但详情页仍沿用文档型布局，导致以下问题：

1. 信息架构错误。数据库型知识库的核心对象是数据源连接、Schema 快照与查询审计，不是文档文件。
2. 操作路径混乱。用户进入数据库型知识库后，首先看到的是上传文件与文档列表，无法直接找到数据库连接和结构信息。
3. 类型识别不足。数据库型知识库与文档型知识库在列表和详情页上缺少足够显式的类型区分，容易误导用户。
4. 能力闭环未完成。后端与设置页已支持数据库能力，但详情页没有形成以数据库运维与查询安全为中心的闭环体验。

## 2. 设计目标

本次重设计聚焦数据库型知识库的产品语义，目标如下：

1. 在不新增路由的前提下，完成数据库型知识库与文档型知识库的页面分流。
2. 让数据库型知识库详情页以“连接、Schema、审计”为主轴组织内容。
3. 去除数据库型知识库中的文档上传、标签筛选、文档列表等无关模块。
4. 在列表页和详情页都增加显式类型标识，确保数据库型知识库能够被快速识别。
5. 最大化复用现有数据源管理、Schema 查看、查询审计等已有能力，控制改动范围。

## 3. 设计原则

1. 单页分流。继续复用现有路由与详情页容器，在页面内部按知识库类型进行分支渲染。
2. 以高频任务为中心。数据库型知识库详情页优先服务以下任务：查看当前连接、刷新 Schema、查看结构、查看查询审计。
3. 与数据库能力一致。页面内容必须反映数据库型知识库的只读查询、结构快照、审计可追踪等特点。
4. 渐进式演进。优先完成最小可用闭环，不引入额外复杂路由与全新状态管理。

## 4. 页面信息架构

数据库型知识库详情页建议拆分为四个信息层次：

### 4.1 头部识别区

职责：明确当前知识库是数据库类型，并提供与文档型不同的导航语义。

内容包括：

1. 知识库名称。
2. 显式类型标签“数据库”。
3. 面包屑末级由“文档”改为“数据库详情”。
4. 页面副标题从文档上传语义改为“围绕连接、Schema 快照与查询审计管理数据库知识库”。

### 4.2 数据库概览区

职责：在进入页面后，第一屏直接展示数据库型知识库的关键信息与当前状态。

建议展示：

1. 当前数据源连接状态。
2. Schema 快照概况，包括表数量、视图数量、字段数量。
3. 查询约束，包括行数限制、查询超时等只读保护信息。
4. 查询审计概况，包括审计数量与最近执行时间。

### 4.3 场景说明与使用路径区

职责：告诉用户数据库型知识库与文档型知识库的使用方式差异，降低误解成本。

建议内容：

1. 该类型不再展示文档上传、标签和文件列表。
2. 页面围绕外部数据库连接与结构快照组织。
3. 推荐使用路径为：维护连接 → 刷新 Schema → 查看结构/审计。

### 4.4 数据源管理区

职责：承接实际管理动作，形成闭环。

复用现有 DataSourceSettings 能力，承接以下动作：

1. 查看并维护数据源配置。
2. 刷新 Schema 快照。
3. 查看数据库结构。
4. 查看查询审计。

## 5. 与现有实现的映射关系

### 5.1 页面承载

继续使用现有详情页路由：

1. frontend/src/router/index.ts 中的 knowledgeBaseDetail。
2. frontend/src/views/knowledge/KnowledgeBase.vue 作为统一承载页面。

### 5.2 类型分流

在 KnowledgeBase.vue 内增加数据库类型判断：

1. document 类型继续沿用文档详情页逻辑。
2. faq 类型继续沿用 FAQ 管理逻辑。
3. database 类型切换为数据库概览页，不再加载文档列表。

### 5.3 复用的数据源能力

现有组件和 API 已经能够支撑数据库详情页：

1. frontend/src/views/knowledge/settings/DataSourceSettings.vue
2. frontend/src/views/knowledge/settings/DatabaseSchemaDialog.vue
3. frontend/src/views/knowledge/settings/DatabaseQueryAuditDialog.vue
4. frontend/src/api/datasource/index.ts 中的 listDataSources、getDatabaseSchema、listDatabaseQueryAudits、refreshDataSourceSchema

### 5.4 新增数据库概览组件

建议新增独立组件承载概览内容，避免将 KnowledgeBase.vue 继续堆大：

1. frontend/src/views/knowledge/components/DatabaseKnowledgeOverview.vue

该组件负责：

1. 拉取数据库概览信息。
2. 组织首屏卡片与说明区。
3. 内嵌 DataSourceSettings 形成闭环。

## 6. 类型区分方案

### 6.1 详情页区分

详情页通过以下方式明确当前为数据库型知识库：

1. 标题旁增加“数据库”类型标签。
2. breadcrumb 改为“数据库详情”。
3. 副标题改为数据库语义。
4. 主内容直接进入数据库概览，而不是文档列表。

### 6.2 列表页区分

知识库列表页在卡片标题区域增加显式类型标签，至少区分：

1. 文档
2. FAQ
3. 数据库

这样用户在进入详情页前即可识别知识库类型。

## 7. 交互流程

数据库型知识库的推荐交互路径如下：

1. 用户从知识库列表看到“数据库”类型标签并进入详情页。
2. 详情页头部再次确认当前类型为数据库。
3. 用户在概览区快速确认连接状态、Schema 快照与查询约束。
4. 用户进入数据源管理区执行刷新 Schema、查看结构、查看审计等动作。
5. 当数据库结构变化后，用户重新刷新 Schema，详情页同步体现新的快照状态。

## 8. 本次落地范围

本次改造建议纳入以下内容：

1. KnowledgeBase.vue 增加数据库类型分流。
2. 新增 DatabaseKnowledgeOverview.vue。
3. 详情页头部增加显式类型标签与数据库专属副标题。
4. 数据库型知识库不再触发文档列表加载。
5. 列表页卡片增加数据库类型标签。
6. 补充中英文文案。

## 9. 后续可继续增强的方向

本次优先做最小闭环，后续可按价值继续增强：

1. 在概览区增加最近 Schema 刷新结果摘要，例如新增表数、变更字段数。
2. 增加数据库账号风险提示，例如是否为只读账号、是否启用白名单。
3. 增加数据库知识库健康分数，例如连接状态、Schema 新鲜度、审计异常数。
4. 当无数据源时，提供更明确的空状态引导和一步式配置入口。

## 10. 预期结果

改造完成后，数据库型知识库详情页应形成完整闭环：

1. 用户能一眼识别当前知识库类型。
2. 页面第一屏展示数据库相关的核心状态。
3. 用户不再被文档型操作误导。
4. 数据源、Schema、审计形成连续的使用路径。
5. 数据库型知识库与文档型知识库在认知与交互上彻底分离。
