# 智能推理 Agent 长文翻译截断修复方案

## 1. 问题背景

当前使用内置“智能推理”Agent对知识库中的长文档执行“全文翻译为中文，并输出为带格式 markdown 文件”时，结果会在正文中途停止，随后系统仍将该半成品回答标记为已完成，并异步写入聊天历史知识库。

从现象上看，用户会误以为：

1. 文档解析或知识库检索只返回了部分 chunk。
2. Agent 已完整完成任务，只是翻译结果异常混入英文规划文本。
3. 后端在知识处理阶段把原始 PDF 错误地覆盖成了 1 个 chunk。

实际情况并非如此。当前日志中的“总 Chunk 数量: 1”对应的是“聊天问答入库后的单条 passage”，并不是原始 PDF 的解析结果。

## 2. 直接证据

### 2.1 聊天回答会被重新写入知识库

当前聊天历史入库逻辑会把问答对拼装为单条文本后入库：

- 文件：internal/application/service/message.go
- 函数：IndexMessageToKB
- 当前行为：将内容组装为 `[Session: ...]\nQ: ...\nA: ...`，然后调用 `CreateKnowledgeFromPassage`。

这意味着一旦 assistant 的回答是半成品，半成品也会被作为知识内容重新进入知识库。

### 2.2 Agent 收尾逻辑无条件将消息标记为完成

当前对话收尾逻辑如下：

- 文件：internal/handler/session/qa.go
- 函数：completeAssistantMessage
- 当前行为：直接设置 `assistantMessage.IsCompleted = true`，随后异步调用 `IndexMessageToKB`。

这里没有区分：

1. 正常完成。
2. 因输出长度截断而停止。
3. 因达到最大轮次而退化生成总结答案。
4. 因请求取消或上下文超时而提前结束。

### 2.3 智能推理 Agent 默认输出额度偏小

当前内置智能推理 Agent 配置如下：

- 文件：config/builtin_agents.yaml
- Agent：builtin-smart-reasoning
- 当前配置：`max_completion_tokens: 2048`

对于“全文翻译 164 个 chunk 并输出完整 markdown 文件”这一类长文任务，2048 的完成 token 上限明显不足。

### 2.4 Agent 未对 `finish_reason=length` 做专门处理

当前 Agent 的终止判定主要关注：

1. `stop`
2. `content_filter`
3. `final_answer` 工具调用

对应代码：

- 文件：internal/agent/observe.go
- 函数：analyzeResponse

这里没有把 `finish_reason=length` 当作“输出被截断，需要续写或标记为部分完成”的分支处理。

### 2.5 当前 Agent 不具备工作区落盘能力

当前内置智能推理 Agent 的工具集为：

- thinking
- todo_write
- knowledge_search
- grep_chunks
- list_knowledge_chunks
- query_knowledge_graph
- get_document_info

该工具集不包含“写 markdown 文件到工作区”的能力，因此它只能尝试把整份译文一次性放进聊天回复，天然容易触发长度上限。

## 3. 根因归纳

本问题不是单点缺陷，而是三层问题叠加：

### 3.1 能力模型与任务类型不匹配

“智能推理”Agent适合多轮检索、归纳、问答，不适合一次性输出超长文档，更不适合承担“读完整篇、逐段翻译、最终写成 markdown 文件”的批处理任务。

### 3.2 收尾状态建模缺失

当前消息层只有 `IsCompleted`，没有：

1. 完成原因。
2. 是否部分完成。
3. LLM 最终 `finish_reason`。
4. 是否允许入库。

因此系统只能把所有结束都归类为“已完成”。

### 3.3 聊天历史入库缺少质量门禁

当前 `IndexMessageToKB` 只做了 `<think>` 清理，没有判断回答是否：

1. 被截断。
2. 明显是计划/自述文本。
3. 并未真正完成用户要求。

所以半成品回答会污染聊天历史知识库，进一步干扰后续检索。

## 4. 修复目标

本次修复建议分三层推进，目标如下：

1. 阻止“截断/半成品回答”被错误标记为完整完成并写入聊天历史知识库。
2. 让 Agent 能识别 `finish_reason=length`，并具备续写或明确失败返回的能力。
3. 为“超长文档翻译生成 markdown 文件”提供专门任务流，而不是继续挤压在普通问答链路中。

## 5. 推荐实施顺序

### 5.1 P0：止损修复，优先上线

目标：先消除错误入库和错误完成状态，避免继续产生脏数据。

### 5.2 P1：状态建模增强

目标：把“完成”“部分完成”“截断”“超时”“取消”从实现层提升为显式状态。

### 5.3 P2：新增长文翻译任务流

目标：为全文翻译输出 markdown 文件提供正确的产品能力边界。

## 6. 具体改动点

## 6.1 P0 止损修复

### 改动点 A：为 Agent 完成事件补充完成元数据

文件：internal/event/event_data.go

建议修改：扩展 `AgentCompleteData`，新增以下字段。

```go
CompletionStatus string `json:"completion_status,omitempty"`
FinishReason     string `json:"finish_reason,omitempty"`
IsPartial        bool   `json:"is_partial,omitempty"`
AllowIndexing    bool   `json:"allow_indexing,omitempty"`
```

建议语义：

1. `completion_status=completed` 表示任务正常完成。
2. `completion_status=partial` 表示拿到部分结果，但不能视为完整交付。
3. `completion_status=failed` 表示未能生成可交付结果。
4. `AllowIndexing=false` 时，后续不得进入聊天历史知识库。

### 改动点 B：在 Agent 执行结果中区分 `length` 截断

文件：internal/agent/observe.go

建议修改函数：`analyzeResponse`

建议新增逻辑：

1. 当 `response.FinishReason == "length"` 且 `len(response.ToolCalls) == 0` 时，不应视为正常完成。
2. 若 `response.Content` 非空，则将本轮结果标记为 `partial`。
3. 不直接发出“完成且可入库”的结论，而是返回一种“需要续写或降级处理”的 verdict。

建议新增 verdict 字段：

```go
type responseVerdict struct {
	isDone           bool
	finalAnswer      string
	emptyContent     bool
	isPartial        bool
	completionStatus string
	finishReason     string
	allowIndexing    bool
	step             types.AgentStep
}
```

建议判定规则：

1. `stop + 无 tool calls` 才视为正常完成。
2. `final_answer` 工具调用视为正常完成。
3. `length` 视为“部分完成”，默认不允许入库。
4. `content_filter` 视为失败，不允许入库。

### 改动点 C：Agent 循环遇到截断时进行续写，而不是直接结束

文件：internal/agent/engine.go

建议修改函数：`runReActIteration`

当前问题：`analyzeResponse` 不处理 `length`，后续流程容易落到重复响应检测、最大轮次兜底，或者在上层被当成完成消息处理。

建议增加分支：

1. 当 verdict 为 `isPartial=true` 且 `finishReason=length` 时，向 `messages` 追加一条 continuation 提示。
2. continuation 提示内容建议为：

```text
Continue exactly from where you stopped. Do not repeat previous content. Keep the same output format.
```

3. 同时保留已生成的正文，让下一轮只做续写。
4. 若连续 N 轮仍然触发 `length`，则结束为 `partial` 状态，而不是伪装为 completed。

建议新增常量：

- 文件：internal/agent/const.go
- 建议新增：`maxLengthContinuationRounds = 3`

### 改动点 D：完成消息写库前必须看完成元数据

文件：internal/handler/session/qa.go

建议修改函数：`completeAssistantMessage`

当前签名：

```go
func (h *Handler) completeAssistantMessage(ctx context.Context, assistantMessage *types.Message, userQuery string)
```

建议改为：

```go
func (h *Handler) completeAssistantMessage(
	ctx context.Context,
	assistantMessage *types.Message,
	userQuery string,
	completionStatus string,
	finishReason string,
	allowIndexing bool,
)
```

建议行为：

1. 仅当 `completionStatus == "completed"` 时设置 `IsCompleted = true`。
2. 当 `completionStatus == "partial"` 时，保留消息内容，但不要视为完成交付。
3. 当 `allowIndexing == false` 时，不调用 `IndexMessageToKB`。
4. 对 `finishReason=length` 追加明确日志，方便定位长文任务被截断。

### 改动点 E：聊天历史入库增加硬性门禁

文件：internal/application/service/message.go

建议修改函数：`IndexMessageToKB`

建议方式二选一。

方式 1：扩展函数签名，显式传入是否允许入库。

```go
func (s *messageService) IndexMessageToKB(
	ctx context.Context,
	userQuery string,
	assistantAnswer string,
	messageID string,
	sessionID string,
	allowIndexing bool,
)
```

方式 2：传入一个结构体参数，包含：

1. `CompletionStatus`
2. `FinishReason`
3. `AllowIndexing`

推荐方式 2，便于后续扩展。

建议增加以下门禁：

1. `AllowIndexing == false` 直接返回。
2. `CompletionStatus != completed` 直接返回。
3. `FinishReason == length` 直接返回。
4. assistantAnswer 命中明显计划性模板文本时直接返回，例如：
   - `Let me organize`
   - `This is going to be a very long response`
   - `Now I need to produce`

第 4 条属于保底型脏数据拦截，不能代替状态修复，但可以降低线上污染面。

### 改动点 F：调整智能推理 Agent 默认完成额度

文件：config/builtin_agents.yaml

建议修改项：`builtin-smart-reasoning.config.max_completion_tokens`

建议值：

1. 最低先提到 4096。
2. 若所接模型支持稳定更长输出，可提到 8192。

说明：

这只是缓解，不是根治。它可以减少普通较长回答的截断概率，但不能解决“整篇文档翻译一次性输出”的架构不匹配问题。

## 6.2 P1 状态建模增强

### 改动点 G：为消息表增加显式完成状态字段

当前 `messages` 表只有 `is_completed`，不够表达本问题。

建议新增字段：

1. `completion_status varchar(32)`
2. `finish_reason varchar(32)`

建议取值：

1. `pending`
2. `completed`
3. `partial`
4. `failed`
5. `cancelled`

涉及文件：

1. internal/types/message.go
2. migrations/versioned/000044_add_message_completion_status.up.sql
3. migrations/versioned/000044_add_message_completion_status.down.sql
4. migrations/mysql/00-init-db.sql
5. migrations/sqlite/000000_init.up.sql
6. migrations/paradedb/00-init-db.sql

建议说明：

1. `IsCompleted` 可保留用于兼容前端旧逻辑。
2. 新逻辑以 `completion_status` 为准。
3. 前端展示时，`partial` 可显示为“结果不完整，请继续生成”。

### 改动点 H：Agent 完成事件下钻到消息持久化层

涉及文件：

1. internal/agent/finalize.go
2. internal/handler/session/agent_stream_handler.go
3. internal/handler/session/qa.go

建议修改：

1. `emitCompletionEvent` 输出完整 completion metadata。
2. `handleComplete` 和 `completeAssistantMessage` 使用该 metadata 更新消息状态。
3. 不再由上层凭空假设“收尾即完成”。

## 6.3 P2 新增长文翻译任务流

这是根治方案，建议单独排期。

### 改动点 I：不要再让普通问答链路承担整本翻译输出

建议新增“长文翻译任务”模式，核心流程如下：

1. 读取指定 knowledge/document 的 chunk 列表。
2. 按批次翻译，例如每 5 到 10 个 chunk 一批。
3. 每批翻译结果先落为中间稿。
4. 最终汇总成 markdown 文件。
5. 任务完成后返回文件路径或下载链接，而不是把全文塞进单次聊天回复。

### 改动点 J：新增专用 Agent 或任务服务

建议有两种实现路线。

路线 1：新增专用后端任务服务，推荐。

涉及目录建议：

1. internal/application/service 下新增 document_translation.go
2. internal/handler 下新增文档翻译任务接口
3. 内部异步任务队列新增 translation/export task

路线 2：新增专用内置 Agent，作为过渡方案。

涉及文件：config/builtin_agents.yaml

建议新增：`builtin-document-translator`

但需要前提：必须补齐“写文件/导出 markdown”的工具能力，否则仍然只能输出聊天文本。

### 改动点 K：输出文件不要走聊天历史入库链路

当任务目标是“生成文档文件”时，结果产物应进入：

1. 工作区文件。
2. 导出文件表。
3. Wiki 页面。

不应再作为普通聊天回答回灌到聊天历史知识库，否则后续仍会重复污染。

## 7. 建议的最小可上线版本

若希望最快止血，建议先做以下最小变更集：

1. 在 internal/agent/observe.go 增加 `finish_reason=length` 的 partial 判定。
2. 在 internal/event/event_data.go 为 `AgentCompleteData` 增加 `CompletionStatus`、`FinishReason`、`AllowIndexing`。
3. 在 internal/agent/finalize.go 透传这些字段。
4. 在 internal/handler/session/qa.go 中只有 `completed` 才设置 `IsCompleted=true` 并触发 `IndexMessageToKB`。
5. 在 internal/application/service/message.go 中增加入库门禁，禁止 partial/length 结果入库。
6. 在 config/builtin_agents.yaml 把智能推理的 `max_completion_tokens` 调高到 4096 或 8192。

该版本可以先解决两个最严重问题：

1. 半成品回答被错当成完成结果。
2. 半成品回答污染聊天历史知识库。

## 8. 验证方案

### 8.1 单元测试

建议补充以下测试。

涉及文件：

1. internal/agent/engine_test.go
2. internal/handler/session 相关测试文件
3. internal/application/service/message 相关测试文件

建议用例：

1. `finish_reason=stop` 时，消息状态为 completed，允许入库。
2. `finish_reason=length` 时，消息状态为 partial，不允许入库。
3. `content_filter` 时，消息状态为 failed，不允许入库。
4. 当回答中出现规划性模板文本时，即使上层漏传状态，也会被 `IndexMessageToKB` 拦截。

### 8.2 集成验证

建议验证场景：

1. 普通短回答不受影响。
2. 普通中长回答在提高 token 上限后仍能正常完成。
3. 用长文翻译提示词强制触发 `length`，确认消息不被标记为 completed。
4. 查看知识处理日志，确认不再出现将半成品 Q/A 作为单条知识入库的情况。

### 8.3 回归关注点

重点关注：

1. SSE 流程是否还能正常收尾。
2. 前端是否依赖 `IsCompleted=true` 才结束显示。
3. 旧会话兼容是否受影响。
4. 聊天历史知识库召回率是否因更严格门禁而下降。

## 9. 风险与取舍

### 9.1 仅提高 token 上限的风险

如果只调大 `max_completion_tokens`，短期可能改善体验，但：

1. 不能保证完整输出。
2. 不能阻止半成品入库。
3. 不能提供 markdown 文件产物。

因此不能把它视为完整修复。

### 9.2 引入消息状态字段的成本

新增 `completion_status` 和 `finish_reason` 需要迁移与前后端协同，但收益明显：

1. 状态表达更准确。
2. 后续所有异步生成任务都可复用。
3. 能显著减少“误完成”和“脏数据入库”问题。

### 9.3 长文翻译专用任务流的收益

这是唯一能真正满足“全文翻译并输出 markdown 文件”的方案。虽然实现成本高于简单调参，但能把产品能力边界从“聊天回答”提升到“文档加工任务”。

## 10. 最终建议

建议按以下顺序落地：

1. 本周内先完成 P0，阻止 partial 结果被标记完成和入库。
2. 下个迭代完成 P1，为消息增加 `completion_status` 和 `finish_reason`。
3. 单独立项完成 P2，提供专门的长文翻译任务流和 markdown 文件输出能力。

如果只允许做一次改动，优先级最高的是：

1. `internal/agent/observe.go` 的 `length` 判定。
2. `internal/handler/session/qa.go` 的完成/入库门禁。
3. `internal/application/service/message.go` 的入库兜底拦截。

这三处一起改，才能真正把这次问题从根上压住。
