# 文档编辑长时间无输出与基线合并展示问题解决方案

## 1. 背景

用户在已选择基线文档 `北海电厂二期智慧电厂项目技术方案 V4` 的情况下执行：

```text
把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

出现两个问题：

1. 执行后长时间没有用户可见输出。
2. 等待较久后虽然有输出，但没有按“基于基线文档合并后展示修正版完整文档”的预期呈现。

本次仅审查当前代码与日志，先输出原因分析和解决方案，不修改代码。

## 2. 日志证据

### 2.1 请求确实是基线文档编辑请求

`my_docs/test/6.md` 与 `my_docs/test/7.md` 中请求参数一致：

```json
{
  "query": "把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。",
  "intent_hint": "revise_document",
  "base_artifact_id": "14a55959-399a-4510-9c0d-9f43e438e7dd",
  "document_output_mode": "delta_only",
  "agent_enabled": true,
  "web_search_enabled": true
}
```

这说明用户不是要求重新检索知识库生成新文档，而是要求基于已有 artifact 做结构调整。

### 2.2 已进入专用文档编辑路径

日志中的 LLM 请求 system prompt 为：

```text
You are a dedicated document editor. Complete the task in a single pass using only the provided user goal and document editing context. Do not call tools.
```

user prompt 中包含：

```text
- artifact_id: 14a55959-399a-4510-9c0d-9f43e438e7dd
- revision_no: 4
- snapshot_char_count: 18607
- target_heading: 第二章
- source_heading: 2.5.5 火电设备运维智能体——技术实现
- document_edit_operation: move_after_heading_to_section
- document_merge_mode: append_to_section
```

说明阶段三的专用文档编辑路径已经被触发，且上下文中确实包含基线文档和源/目标章节信息。

### 2.3 用户可见首字延迟接近 94 秒

关键时间线：

```text
14:16:38.182  发起 LLM ChatStream
14:16:38.712  First reasoning_content，elapsed_ms=368
14:18:17.794  First delta.Content，elapsed_ms=93923
14:19:13.280  Natural-stop，elapsed_ms=145727
```

模型很快开始内部推理，但直到约 94 秒后才输出第一个用户可见正文字符 `<`。前端只能看到等待状态，因此表现为“很长时间没有输出”。

### 2.4 assistant 被过早持久化为空完成态

更严重的是，在 LLM 还未输出正文前，后端已经把 assistant 消息标记为 completed：

```text
14:16:38.343 Preparing assistant completion persistence ... completion_status: completed ... content_len: 0
14:16:38.348 Persisted assistant completion state ... is_completed: true ... content_len: 0
```

紧接着尝试按该 assistant message 查询 artifact：

```text
SELECT * FROM chat_document_artifacts WHERE source_message_id = 'c13ea7db-fb16-437c-8fa3-25616fee2c9f'
record not found
```

日志中没有看到 14:19 模型真正结束后再次注册 artifact 的记录，例如：

```text
Registered chat document artifact ...
```

因此最终内容没有进入“消息持久化 -> artifact 注册 -> 基线合并快照 -> complete payload -> 前端展示”的闭环。

## 3. 当前代码链路审查

### 3.1 专用文档编辑路径异步返回过早

当前 `internal/application/service/session_agent_qa.go` 中：

```go
func (s *sessionService) runDedicatedDocumentEditPath(...) error {
    responseChan, err := chatModel.ChatStream(ctx, messages, opt)
    ...
    go s.consumeDedicatedDocumentEditStream(context.WithoutCancel(ctx), req, eventBus, responseChan)
    return nil
}
```

该函数启动后台 goroutine 后立即返回 `nil`。

但 `internal/handler/session/qa.go` 的 `executeQA()` 在 Agent 模式下有 defer：

```go
defer func() {
    if mode == qaModeAgent {
        h.completeAssistantMessage(..., agentCompletionOptions)
        logger.Infof(..., "Agent QA service completed")
    }
}()
```

于是执行顺序变成：

1. `AgentQA()` 发现是文档编辑请求。
2. `runDedicatedDocumentEditPath()` 启动后台流消费 goroutine。
3. `runDedicatedDocumentEditPath()` 立即返回 `nil`。
4. `executeQA()` 的 Agent defer 立刻执行。
5. `completeAssistantMessage()` 用当前空内容持久化 assistant。
6. artifact 注册看到空内容，无法创建修正版 artifact。
7. 两分钟后 LLM 真正输出 patch，但外层完成态已经写错。

这就是“等很久有输出，但没有基于基线文档输出修正后文档”的主因。

### 3.2 专用路径没有首字超时或进度事件

`consumeDedicatedDocumentEditStream()` 只在收到 `ResponseTypeAnswer` 时发 `EventAgentFinalAnswer`：

```go
if response.ResponseType != types.ResponseTypeAnswer {
    continue
}
finalContent.WriteString(response.Content)
eventBus.Emit(EventAgentFinalAnswer, ...)
```

对于模型的 `reasoning_content`，当前没有用户可见进度事件；对于“长时间无正文 token”的情况，也没有 first-content timeout、降级策略或“正在分析基线文档并生成补丁”的心跳事件。

日志显示 prompt tokens 为 10451，completion tokens 为 4721。虽然基线快照只有 18607 字符，但一次性把完整文档、结构化目标和 patch 规则交给深度推理模型，仍可能触发很长的内部推理期。

### 3.3 当前前端仍存在完整文档主展示缺口

当前 `frontend/src/views/chat/components/botmsg.vue` 中主正文渲染仍使用：

```js
const renderedHTML = computed(() => {
    const text = props.content || props.session?.content || '';
    ...
});
```

而完整文档快照只进入：

```js
const artifactDocumentContent = computed(() => {
    const content = props.session?.final_document_content || props.session?.chat_document_artifact?.content_snapshot || '';
    return typeof content === 'string' ? content.trim() : '';
});
```

也就是说，即使后端成功注册 artifact 并返回完整快照，普通消息主区域仍可能只显示 `session.content` 中的 delta/patch，而不是完整修正版文档。

当前 `AgentStreamDisplay.vue` 中已经残留了一部分 artifact 快照优先逻辑，但它依赖 `session.final_document_content` 或 `chat_document_artifact.content_snapshot` 已经存在。后端没有成功注册 artifact 时，这条前端兜底不会生效。

## 4. 根因结论

本次失败不是“没找到基线文档”，也不是“专用路径完全没走”。真正根因是三段链路没有闭合。

### 4.1 P0 根因：专用文档编辑流与 Agent 外层完成态竞态

专用文档编辑路径把真正的 LLM 流消费放到后台 goroutine，然后立即返回。Agent 外层把“服务函数返回”误当作“回答已经完成”，提前持久化空 assistant 并尝试注册 artifact。

这是最核心问题。

### 4.2 P1 根因：首字输出依赖模型自然产出正文

模型 368ms 就产生 reasoning，但用户可见正文 93.9s 后才出现。系统没有专用进度事件、首字超时检测、快速结构化 patch 兜底，因此用户看到的是长时间空白等待。

### 4.3 P1 根因：完整文档展示依赖 artifact 成功注册和前端快照优先渲染

当前前端普通消息主区域仍优先展示 `session.content`。如果 `session.content` 是 patch 或尾部片段，而完整文档只在 artifact 快照里，用户默认看到的仍不是完整修正版。

## 5. 解决方案

### 5.1 后端 P0：专用文档编辑路径必须同步生命周期

目标：`AgentQA()` 只有在专用文档编辑流真正完成、失败或超时后才返回。

建议把 `runDedicatedDocumentEditPath()` 改为阻塞等待消费完成：

1. 不再直接 `go consumeDedicatedDocumentEditStream(...)` 后返回。
2. `consumeDedicatedDocumentEditStream()` 返回结构化结果，例如：
   - `FinalAnswer`
   - `CompletionStatus`
   - `FinishReason`
   - `FailureReason`
   - `TotalDurationMs`
3. `runDedicatedDocumentEditPath()` 在当前 goroutine 内消费 stream，直到 done/error/context done。
4. 完成事件仍通过 event bus 发出，保持前端 SSE 协议不变。
5. `AgentQA()` 返回时，`agentCompletionOptions` 已经被 `EventAgentComplete` handler 更新为最终内容。
6. `executeQA()` 的 Agent defer 再执行 `completeAssistantMessage()` 时，写入的是最终 patch，而不是空内容。

验收标准：

```text
日志顺序必须变为：
First delta.Content / Natural-stop
EventAgentComplete received
Preparing assistant completion persistence ... content_len > 0
Registered chat document artifact ... revision_no = base.revision_no + 1
complete payload includes chat_document_artifact and final_document/final_document_artifact_id
```

### 5.2 后端 P0：避免重复完成和空完成覆盖

即使改为同步消费，也建议在 handler 层加防线：

1. Agent 模式下如果收到 `EventAgentComplete` 并已完成持久化，defer 不应再次用旧 options 覆盖。
2. 增加 `agentCompletionHandled` 标志，或复用类似普通模式的 `completionHandled`。
3. 对文档编辑请求，如果 `FinalAnswer == ""` 且 `AllowComplete=true`，禁止标记 completed，可改为 failed/partial，并禁止 artifact 注册。
4. `completeAssistantMessage()` 对文档 artifact 注册前增加空内容保护日志，避免“completed + content_len=0 + allow_indexing=true”。

建议新增测试：

```text
TestExecuteQA_DedicatedDocumentEditDoesNotPersistEmptyCompletionBeforeStreamDone
TestCompleteAssistantMessage_DocumentEditRejectsEmptyCompletedArtifact
TestAgentCompletionEvent_IsPersistedOnceAndNotOverwrittenByDefer
```

### 5.3 后端 P0：专用路径完成后必须注册基线合并 artifact

专用文档编辑输出的是 `<document_patch>`，不是完整文档。因此必须确认完成时调用：

```go
RegisterFromAssistantMessage(ctx, assistantMessage, RegisterChatDocumentArtifactOptions{
    Intent: revise_document,
    Operation: revise,
    OutputMode: delta_only,
    BaseArtifact: baseArtifact,
    TargetHeading: 第二章,
    MergeMode: append_to_section,
})
```

`chat_document_artifact.go` 中已有能力：

1. `buildChatDocumentSnapshotResult()` 对 revise + delta 调用 `mergeChatDocumentRevisionDelta()`。
2. `mergeChatDocumentRevisionDelta()` 能解析 `<document_patch>`。
3. `<append heading="第二章">...</append>` 会被合并回基线文档的第二章。

因此修复重点不是重写 merge，而是保证最终 patch 在正确时机进入注册流程。

### 5.4 后端 P1：减少首字等待

针对 94 秒无用户可见输出，建议分两层治理。

第一层：可见进度治理。

1. 专用路径开始时立即发一个轻量事件：`document_edit_started` 或 answer metadata，前端显示“正在分析基线文档并生成修订补丁”。
2. 如果 5-10 秒内没有 `ResponseTypeAnswer`，发心跳进度：`document_edit_waiting_for_model`。
3. 如果超过可配置阈值仍无首字，例如 30 秒，标记 `first_content_delay_ms` 并写日志。

第二层：模型负载治理。

1. 对明确的“章节移动/合并”请求，优先走确定性 patch 生成，不让 LLM 阅读全量 18k 文档后自行决定。
2. `BuildQuotedContext()` 已能构造 source/destination 双锚点上下文，应进一步确保 prompt 只包含：文档目录、源章节、目标章节、相邻兄弟节点，而不是完整文档全文。
3. 对 `move_after_heading_to_section` 场景，可以由后端直接生成基础 patch：把 source section 或 source heading 后续内容 append 到目标章节，并可让 LLM 只做润色，而不是让 LLM 从全量文档中推断移动范围。
4. 为 dedicated path 单独设置更小的 `MaxCompletionTokens` 与 first-token timeout，避免模型长时间输出大段非必要内容。

建议新增测试：

```text
TestBuildQuotedContext_MoveSectionUsesFocusedDualAnchorPayload
TestDedicatedDocumentEdit_EmitsProgressBeforeFirstContentTimeout
TestDedicatedDocumentEdit_FirstContentTimeoutFallsBackToDeterministicPatch
```

### 5.5 前端 P1：完整 artifact 快照优先作为文档消息主内容

后端修复后，complete payload 应包含：

```json
{
  "chat_document_artifact": { ... },
  "final_document_mode": "inline_snapshot" 或 "fetch_artifact_snapshot",
  "final_document": "...",
  "final_document_artifact_id": "..."
}
```

前端已有 `hydrateFinalDocumentFromComplete()` 会在 fetch 模式下拉取 artifact snapshot。但普通消息主正文仍应统一改为：

```text
displayContent = final_document_content || chat_document_artifact.content_snapshot || session.content
```

影响范围：

1. `botmsg.vue` 的 `renderedHTML`、`hasActualContent`、复制、导出、添加到知识库应使用 `displayContent`。
2. `AgentStreamDisplay.vue` 应保留并完善当前 artifact 快照优先逻辑。
3. 文档版本卡片继续保留“查看完整文档 / 查看版本链 / 取消基线”等操作，但不再作为完整文档的唯一入口。

验收标准：

```text
文档编辑完成后，聊天主区域直接显示合并后的完整 V5 文档。
文档版本卡显示 V5 可继续。
当前基线自动切换到 V5。
再次追问时 base_artifact_id 使用 V5 artifact id。
```

### 5.6 可观测性 P1：补齐关键日志

建议增加以下日志点：

1. `document_edit_path_selected`：记录 session_id、message_id、base_artifact_id、output_mode、quoted_context_len、snapshot_mode。
2. `document_edit_stream_started`：记录 model、prompt_tokens 估算、max_completion_tokens。
3. `document_edit_first_content`：记录 first_content_delay_ms。
4. `document_edit_complete`：记录 final_answer_len、duration_ms、finish_reason。
5. `document_edit_artifact_registered`：记录 artifact_id、revision_no、snapshot_char_count、quality_issues。
6. `document_edit_empty_completion_blocked`：发现文档编辑空完成态时报警。

这些日志可以直接避免本次排查中“到底是没生成、没合并、没注册还是没展示”的不确定性。

## 6. 建议实施顺序

### 阶段一：修复生命周期竞态

1. 改造 `runDedicatedDocumentEditPath()`，专用流消费完成前不返回。
2. `executeQA()` Agent defer 增加 completion handled 防重。
3. 文档编辑空 completed 保护。
4. 补 `content_len > 0`、artifact 注册成功的回归测试。

### 阶段二：修复完整文档展示

1. 确保 complete payload 携带 artifact 元数据和 final document delivery 字段。
2. 前端主正文优先展示 artifact snapshot。
3. 验证 inline_snapshot 与 fetch_artifact_snapshot 两种路径。

### 阶段三：治理长时间无输出

1. 增加专用文档编辑进度事件与首字延迟日志。
2. 对章节移动/合并请求引入确定性 patch 或 focused context。
3. 增加 first-content timeout 降级策略。

## 8. 最终结论

当前问题的核心不是“没有基线文档”，而是专用文档编辑流的生命周期与 Agent 外层完成持久化发生竞态：后端在模型输出前就把 assistant 写成空 completed，导致 artifact 没有注册，基线合并快照没有产生，前端自然无法展示修正后的完整文档。

同时，模型首个正文 token 延迟约 94 秒，系统没有进度事件和首字超时治理，所以用户体验上表现为长时间无输出。

优先修复顺序应为：先让专用文档编辑路径同步等待流完成并注册 artifact，再修复前端完整快照主展示，最后治理首字等待和确定性 patch 生成。
