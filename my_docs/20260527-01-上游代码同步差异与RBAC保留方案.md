# 20260527 上游代码同步差异与 RBAC 保留方案

## 1. 文档目标

本文档基于 `my_docs/20260429-04-通用上游同步操作手册.md`，用于指导本仓库从 `Tencent/WeKnora` 上游主线同步代码时的差异判断、冲突处理和最终提交。

本轮同步必须同时满足以下目标：

1. 吸收 `upstream/main` 的最新代码、修复和公共能力。
2. 逻辑冲突、产品行为冲突、权限模型冲突统一以本地代码和本地设计为准。
3. 对租户 RBAC 相关冲突采取特别保护策略：目标是统一到上游 RBAC 的角色与守卫体系，但必须保留本地已经落地的 Organization 共享空间、组织人员管理、知识库和智能体归属控制能力。
4. 同步完成、验证通过后，整体结果提交到本仓库 `dev` 分支。

---

## 2. 当前同步基线

### 2.1 远端约定

当前仓库远端符合通用手册约定：

```bash
origin   https://github.com/mrsteed/WeKnora.git
upstream https://github.com/Tencent/WeKnora.git
```

本地开发分支为：

```bash
dev
```

`dev` 应跟踪 `origin/dev`，不能直接跟踪 `upstream/main`。

### 2.2 本轮差异规模

在 2026-05-27 拉取远端后，本地 `dev` 与 `upstream/main` 的差异统计为：

```bash
git rev-list --left-right --count dev...upstream/main
# 48 225
```

含义：

1. 本地 `dev` 相对上游仍有 48 个本地提交需要保留。
2. 上游 `main` 相对本地有 225 个提交需要评估吸收。
3. 本轮同步不是小范围补丁合并，权限、系统设置、模型、检索、前端交互等模块都可能出现冲突。

上游最新主线包含如下近期提交方向：

1. 系统审计与系统管理员设置改造。
2. Agent 意图提示词、内置模型、OpenSearch、Doris、Yuque 等功能修复和增强。
3. 租户 RBAC 相关文档、迁移、后端守卫和前端表现。

## 3. 总体裁决原则

### 3.1 默认同步方式

按通用手册优先采用以下方式：

```bash
git checkout dev
git pull --ff-only origin dev
git branch backup/dev-$(date +%Y%m%d)-before-upstream-sync
git tag sync-dev-$(date +%Y%m%d)-pre
git fetch origin --prune
git fetch upstream --prune
git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main
```

如果 `rebase` 冲突不可控，再改用“上游新基线 + 本地主题回放”的兜底方案。

### 3.2 冲突裁决

出现逻辑冲突时，执行以下顺序：

1. 先判断冲突是否影响本地已上线或已验证功能。
2. 若影响本地已上线功能，则以本地行为为准。
3. 在保留本地行为的前提下，吸收上游非冲突的安全修复、兼容性修复、测试补充和结构优化。
4. 不允许用上游默认实现直接覆盖本地 Organization、租户、权限、知识库、Agent、长文档、导出和数据分析等定制能力。

### 3.3 RBAC 特别原则

租户 RBAC 是本轮最高风险冲突域。修正后的处理规则如下：

1. 可以统一到上游 Tenant RBAC，但统一目标必须覆盖本地现有能力，而不是用上游默认矩阵删减本地 Organization 能力。
2. 新统一角色采用 `owner` / `admin` / `contributor` / `viewer`；本地现有 `editor` 作为 `contributor` 的兼容别名逐步迁移。
3. 本地 `organizations.owner_id` 当前已经表达 Owner 语义，但不是成员角色；同步时应把它角色化为 `owner`，并保留“创建者不可移除、不可降级、仅 owner 可删除空间”的保护。
4. 上游 RBAC 的 `tenant_members`、`creator_id`、审计日志、守卫和前端角色状态可以接入，但必须通过兼容层保留本地组织人员管理、KB/Agent 共享、共享 Agent 停用和跨租户协作路径。
5. 如果上游代码要求默认启用 `tenant.enable_rbac=true`，本地需要先完成角色回填、别名兼容和回归验证，再进入强制鉴权；不能在未迁移数据时直接让线上系统进入 403 状态。

## 4. 本地权限设计基线与统一方向

本地当前权限体系以“租户隔离 + Organization 共享空间”为主，尚未完整落入上游 Tenant RBAC 的 `tenant_members` 角色矩阵。但从职责边界看，本地已有能力可以统一到上游 RBAC：把租户内权限交给 `owner` / `admin` / `contributor` / `viewer`，把跨租户协作继续交给 Organization 共享空间。

### 4.1 本地主模型

本地已实现模型如下：

1. 所有基础业务数据通过 `tenant_id` 做租户隔离。
2. 跨租户协作通过 Organization 共享空间完成。
3. 共享空间成员角色当前为 `admin` / `editor` / `viewer`，其中 `editor` 应统一为上游语义中的 `contributor`。
4. 知识库通过 `kb_shares` 共享到 Organization。
5. Agent 通过 `agent_shares` 共享到 Organization。
6. Agent 可通过 `tenant_disabled_shared_agents` 在目标租户侧做个人化停用。
7. 跨租户管理能力仍沿用 `users.can_access_all_tenants` + `tenant.enable_cross_tenant_access` + `X-Tenant-ID`。

核心代码落点：

1. `internal/types/organization.go`：Organization、成员角色、KB/Agent 共享数据结构。
2. `internal/application/service/organization.go`：组织创建、加入、成员角色、审批和邀请码逻辑。
3. `internal/application/service/kbshare.go`：知识库共享、共享权限更新、共享访问权限计算。
4. `internal/application/service/agent_share.go`：Agent 共享、只读共享、共享 Agent 访问关联知识库。
5. `internal/handler/knowledge.go`：知识库访问时按本租户、KB 共享、Agent 共享三条路径判断。
6. `internal/middleware/auth.go`：JWT、API Key、跨租户切换认证。

### 4.2 本地同租户权限语义

当前本地实现中，用户访问本租户知识库时，只要 `kb.tenant_id == 当前 tenant_id`，后端会按本租户资源处理，知识库访问函数返回 `OrgRoleAdmin` 等价权限。

这意味着本地目前没有上游那种 `tenant_members(user_id, tenant_id, role)` 级别的租户内成员矩阵，也没有在所有租户内写操作上强制区分 viewer / contributor / admin / owner。

同步时可以引入上游同租户 RBAC，但必须先完成三件事：

1. 把租户创建者、Organization 创建者或当前 `owner_id` 回填为 `owner`。
2. 把现有 `editor` 统一为 `contributor`，并在 API、前端和数据库层兼容历史 `editor` 值。
3. 把 KB / Agent 的创建者和共享发起人语义迁移到 `creator_id` / `created_by` 体系，避免 Contributor 失去对自有资源的管理能力。

未完成以上迁移前，不能把上游 `RequireRole` / `RequireOwnershipOrRole` 直接挂到本地既有路由上。

### 4.3 本地跨租户共享语义

本地跨租户知识库访问遵循以下逻辑：

1. 如果知识库属于当前租户，则按本租户资源访问。
2. 如果知识库不属于当前租户，则检查用户是否通过某个 Organization 获得共享访问。
3. 有共享关系时，最终权限取“共享权限”和“用户在空间内角色”的较低权限。
4. 如果没有直接 KB 共享，但用户拥有可访问该 KB 的共享 Agent，则允许只读路径访问。

这套规则是本地共享空间功能的核心，必须保留。

### 4.4 本地数据模型边界

本地已落地或文档确认的数据模型包括：

| 本地对象 | 作用 |
|---|---|
| `organizations` | 共享空间主体；`owner_id` 是本地 Owner 语义来源 |
| `organization_members` | 共享空间成员与空间角色；当前保存 `admin` / `editor` / `viewer` |
| `kb_shares` | 知识库共享关系 |
| `agent_shares` | Agent 共享关系 |
| `tenant_disabled_shared_agents` | 租户侧隐藏共享 Agent 偏好 |
| `users.can_access_all_tenants` | 跨租户超管能力 |
| `custom_agents.created_by` | Agent 创建者字段，已存在 |

本地当前没有以下上游 RBAC 主表或主字段作为运行前提，但这些是统一到上游 RBAC 时需要补齐的对象：

1. `tenant_members`。
2. `audit_logs` 的 RBAC 事件体系。
3. `knowledge_bases.creator_id`。
4. `custom_agents.runnable_by_viewer`。
5. `internal/middleware/rbac.go` 风格的全局角色守卫。
6. `RequireRole` / `RequireOwnershipOrRole` 类租户 RBAC 中间件。

统一策略不是永久排斥这些对象，而是在数据回填、角色映射和兼容 API 完成后再接入。

## 5. 上游 Tenant RBAC 设计摘要

上游文档 `docs/RBAC说明.md` 说明，Tenant RBAC 已随上游 #1303 发布，并由 `tenant.enable_rbac` 控制，默认强制鉴权。

### 5.1 上游角色矩阵

上游每个租户成员在 `tenant_members` 中拥有且仅拥有一个角色：

| 角色 | 标识 | 能力 |
|---|---|---|
| 只读 | `viewer` | 仅查阅、提问，不可发起变更 |
| 贡献者 | `contributor` | 可上传文档、维护自己创建的 KB / Agent |
| 管理员 | `admin` | 可变更租户内任意资源，管理成员和共享基础设施 |
| Owner | `owner` | Admin 全部权限 + 删除租户；每个租户有且只有一个 |

角色层级为：

```text
viewer < contributor < admin < owner
```

### 5.2 上游资源归属模型

上游通过迁移 `000043_tenant_rbac` 给关键资源加入归属字段：

1. `knowledge_bases.creator_id`：老数据回填为租户 Owner，空值表示租户共有资源，仅 Admin+ 可变更。
2. `custom_agents.creator_id`：Agent 创建者。
3. `custom_agents.runnable_by_viewer`：控制 Viewer 是否可在对话中运行该 Agent。

上游由此形成两类守卫：

1. 角色守卫：`Viewer()` / `Contributor()` / `Admin()` / `Owner()`。
2. 归属守卫：`OwnedKBOrAdmin()` / `OwnedAgentOrAdmin()` / `OwnedChunkKBOrAdmin()` 等。

目标语义是：Contributor 在自己的资源里接近 Owner，在别人的资源里接近 Viewer。

### 5.3 上游共享空间关系

上游认为 Tenant RBAC 与 Organization 共享空间是正交关系：

1. Tenant RBAC 解决同一租户内能做什么。
2. Organization 解决跨租户把 KB / Agent 共享给谁。
3. 跨租户写操作需要同时满足共享权限、空间角色和源租户 RBAC。

上游关键判定顺序为：

```text
KB 属于当前租户？
  是：进入租户 RBAC，按角色 + creator_id 判定
  否：检查是否通过共享空间共享给当前用户
      是：按空间角色与共享权限判定
      否：403 / 404
```

### 5.4 上游配置与审计

上游新增配置包括：

```yaml
tenant:
  enable_rbac: true
  enable_cross_tenant_access: false

auth:
  registration_mode: self_serve

audit:
  retention_days: 90
```

对应环境变量包括：

| 环境变量 | YAML 路径 |

| `WEKNORA_AUDIT_RETENTION_DAYS` | `audit.retention_days` |

### 5.5 上游相关文件

上游 `upstream/main` 中与 RBAC 直接相关的文件包括：

2. `internal/middleware/rbac.go`。
3. `internal/middleware/kb_access.go`。

## 6. 本地与上游 RBAC 差异对照

| 维度 | 本地实现 | 上游 RBAC | 同步裁决 |
|---|---|---|---|
| 权限主模型 | 租户隔离 + Organization 共享空间 | Tenant RBAC + Organization 正交校验 | 可以统一为“Tenant RBAC 管租户内，Organization 管跨租户共享” |
| Owner | `organizations.owner_id`，不是成员角色 | `tenant_members.role = owner`，且每租户唯一 | 增加 `owner` 角色，并把现有 `owner_id` 语义回填为 owner |
| 管理员 | `admin` 管理空间设置、成员、共享关系 | `admin` 管理租户成员、基础设施、任意资源 | 可统一，保留空间管理员能力，租户 Admin 不自动等于所有共享空间 Owner |
| 贡献者 / 编辑者 | `editor` 可共享 KB/Agent，可编辑有写权限的共享 KB | `contributor` 可维护自己创建的 KB/Agent，他人资源等同 Viewer | 语义高度接近但不完全相同；统一命名为 `contributor`，历史 `editor` 作为别名兼容 |
| 只读 | `viewer` 只能查看和检索共享资源 | `viewer` 只读、提问，不可变更 | 可直接统一 |
| 同租户 KB 写权限 | 租户匹配即按本地租户资源处理 | 角色 + `creator_id` 判定 | 逐步统一到上游；先回填 `creator_id`，再启用强制守卫 |
| 跨租户 KB 写权限 | 共享权限与空间角色取较低权限 | 共享空间权限 + 源租户 RBAC 同时满足 | 可以统一，但要保留本地“共享权限与成员角色取较低权限”的计算，再叠加源租户资源归属 |
| API Key | API Key 解析租户，找不到用户时构造系统虚拟用户 | API Key 在所属租户内固定 Admin，删除租户仍需 Owner | 保留本地虚拟用户兼容，并映射为所属租户 Admin |
| 跨租户超管 | `CanAccessAllTenants` + `EnableCrossTenantAccess` + `X-Tenant-ID` | 同样支持，且等同目标租户 Admin | 可统一，保留开关和目标租户存在性校验 |
| 资源归属字段 | `custom_agents.created_by` 已存在；KB 无 `creator_id` | KB / Agent 都使用 `creator_id` | 需要补齐 KB `creator_id`，Agent 继续沿用或迁移兼容 `created_by` |
| 审计日志 | 目前无上游 RBAC 审计主流程 | `audit_logs` + RBAC 事件 + 保留期清理 | 可以吸收，并补充本地 Organization 成员与共享事件 |
| 前端权限 | 主要围绕共享空间、菜单、资源列表控制 | `authStore.currentTenantRole` + `hasRole` + owner 判断 | 可以统一到四级角色展示，同时保留 `is_owner` 兼容字段 |
| 迁移策略 | 已有本地组织共享迁移 | 新增 `000043` / `000044` / `000045` | 可采用上游迁移，但必须增加本地 `editor -> contributor` 和 `owner_id -> owner` 回填 |

## 7. 统一到上游 RBAC 的同步措施

### 7.1 Owner 角色补齐

当前本地已经通过 `organizations.owner_id` 实现了 Owner 语义：

1. 空间创建者不能被移出空间。
2. 空间创建者不能被降级。
3. 删除空间仅允许创建者执行。
4. 前端通过 `is_owner` 展示创建者身份。

同步上游 RBAC 时应把这套字段语义角色化：

1. `organization_members.role` 增加 `owner`，或在兼容期继续保存 `admin`，但响应层把 `owner_id == current_user_id` 映射为 `owner`。
2. `tenant_members` 中每个租户必须有且只有一个 `owner`。
3. `owner` 拥有 `admin` 的全部能力，并额外拥有删除租户、删除空间、移交 Owner 等高危能力。
4. 历史数据回填时，优先把租户创建者、Organization `owner_id`、最早活跃管理员回填为 `owner`；无法唯一判断时进入人工核对清单。

### 7.2 contributor 与 editor 的统一判断

结论：本地 `editor` 与上游 `contributor` 在“贡献者”这个产品含义上基本一致，可以统一命名为 `contributor`，但需要保留兼容层。

一致点：

1. 都高于 `viewer`，可以发起内容型变更。
2. 都低于 `admin`，不能管理成员、空间设置、系统基础设施等管理面能力。
3. 本地 `editor` 可以共享 KB / Agent 到空间，上游 `contributor` 可以维护自己创建的 KB / Agent，本质都是“可贡献内容但不是管理员”。

差异点：

1. 本地 `editor` 是 Organization 空间角色，作用范围是某个共享空间。
2. 上游 `contributor` 是 Tenant RBAC 租户角色，作用范围是源租户内的自有资源。
3. 本地 `editor` 对共享 KB 的写能力来自“共享权限 + 空间角色”，上游 `contributor` 对自有 KB 的写能力来自 `creator_id`。

统一措施：

1. 对外展示、接口新值和文档统一使用 `contributor`。
2. 数据库、旧接口和前端旧判断在兼容期同时接受 `editor` 与 `contributor`。
3. 权限等级统一为 `viewer < contributor(editor) < admin < owner`。
4. 对 KB 共享权限，历史 `editor` 值迁移为 `contributor`，但权限含义仍是“可写共享”。
5. 前端国际化、角色筛选、成员管理下拉框中显示“贡献者”，不再新增“编辑者”作为独立角色。

### 7.3 可统一到上游 RBAC 的目标架构

建议统一后的权限架构如下：

1. Tenant RBAC 负责租户内成员与资源归属：`tenant_members.role` + `creator_id`。
2. Organization 负责跨租户协作：`organization_members.role` + `kb_shares` + `agent_shares`。
3. Organization 角色集合也统一为 `owner` / `admin` / `contributor` / `viewer`，其中 `owner` 来自 `organizations.owner_id`，`contributor` 兼容历史 `editor`。
4. 对一次 KB 写操作，先判断是否当前租户资源；当前租户资源走 Tenant RBAC 的角色与归属守卫，跨租户资源走 Organization 共享权限，再叠加源资源归属约束。
5. Agent 共享继续只读；Agent 本体编辑和删除走 `created_by` / `creator_id` 或 Admin+。
6. API Key 在所属租户内映射为 Admin，但不能绕过 Organization 共享空间角色，也不能替代 Owner 删除租户或空间。

### 7.4 必须保留的本地能力

统一上游 RBAC 后，以下能力必须保持行为不变：

1. 用户可创建、加入、搜索 Organization 共享空间。
2. Organization 创建者自动成为 Owner，且创建者保护逻辑不变。
3. 空间成员管理、邀请码、审批、成员上限、角色升级申请继续可用。
4. 只有空间 `owner` / `admin` / `contributor` 可将本租户 KB 或 Agent 共享到空间。
5. KB 共享继续支持只读和可写权限，并与成员角色取较低权限。
6. Agent 共享继续保持只读共享语义。
7. 租户侧可停用共享 Agent，不影响共享关系本身。
8. 通过共享 Agent 间接访问 KB 的只读路径必须保留。
9. `users.can_access_all_tenants` 的跨租户切换兼容能力必须保留。
10. API Key 找不到真实用户时构造系统虚拟用户的兼容逻辑必须保留。

### 7.5 不应直接落地的上游内容

以下上游内容不能在本地未完成迁移前直接强制启用：

1. `tenant.enable_rbac` 默认强制开启。
2. `tenant_members` 作为所有租户内操作的强制前置校验。
3. `knowledge_bases.creator_id` 作为本地 KB 写权限唯一依据。
4. `RequireRole` / `RequireOwnershipOrRole` 对既有接口的全量接管。
5. 未完成 `creator_id` 与共享权限兼容迁移前，源租户 RBAC 对跨租户共享写操作的额外阻断。
6. 上游前端 `currentTenantRole` 对本地 Organization 角色状态的直接替换。
7. 上游 RBAC 成员管理接口对本地 Organization 成员管理接口的替换。

### 7.6 可选择吸收的上游内容

以下内容可以在保留本地权限语义时吸收：

1. `tenant_members` 角色矩阵和 Owner 唯一性约束。
2. `knowledge_bases.creator_id` 与 `custom_agents.creator_id` 的资源归属模型。
3. RBAC 审计日志、拒绝日志去重和保留期清理机制。
4. 跨租户超管访问的安全校验和测试补充。
5. 权限相关单元测试的组织方式。
6. 前端权限按钮防闪烁、加载态、角色徽章等交互细节。

## 8. 可落地同步计划

本节把前面的裁决原则拆成可执行阶段。实际同步时不建议一次性把所有冲突揉成一个巨大提交，而应按“先上游基线、再权限模型、再业务兼容、最后前端和验证”的顺序推进。

### 8.1 阶段总览

| 阶段 | 目标 | 主要产物 | 进入下一阶段条件 |
|---|---|---|---|
| P0 准备与冻结 | 固化当前 `dev`、建立回退点、确认差异范围 | backup 分支、pre tag、差异清单 | 工作区 clean，远端拉取完成 |
| P1 上游基线同步 | 把 `upstream/main` 引入同步分支 | `sync/dev-YYYYMMDD-upstream` 分支 | rebase 或 rebuild 路径确定 |
| P2 编译级冲突收敛 | 解决依赖、类型、路由、DI、迁移编号等硬冲突 | 可编译的后端和前端结构 | `go test` 至少能开始编译，前端依赖能安装 |
| P3 RBAC 数据模型统一 | 接入 `owner/admin/contributor/viewer`、`tenant_members`、`creator_id`、审计表 | 本地化迁移脚本和回填脚本 | 迁移在测试库可重复执行，历史数据不丢失 |
| P4 后端权限链路统一 | 接入 RBAC 守卫，同时保留 Organization、KB/Agent 共享和 API Key 兼容 | 后端权限实现与测试 | 组织、知识库、智能体核心回归通过 |
| P5 前端角色与页面统一 | 前端显示 Owner / Admin / Contributor / Viewer，兼容旧 `editor` | 前端状态、页面和国际化修订 | `pnpm build` 通过，关键页面可操作 |
| P6 全量验证与提交 | 执行测试、整理同步说明、提交并推送 | 验证记录、最终提交、同步分支 | 验证通过或环境依赖已记录 |

### 8.2 P0 准备与冻结

执行动作：

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch
git branch backup/dev-$(date +%Y%m%d)-before-rbac-sync
git tag sync-dev-$(date +%Y%m%d)-rbac-pre
git fetch origin --prune
git fetch upstream --prune
git rev-list --left-right --count dev...upstream/main
git log --oneline dev..upstream/main -n 30
git log --oneline upstream/main..dev -n 30
```

输出要求：

1. 记录本地相对上游的提交数量和上游相对本地的提交数量。
2. 明确是否存在未提交工作区改动；如果存在，先提交或 stash。
3. 确认 `origin` 指向 `mrsteed/WeKnora`，`upstream` 指向 `Tencent/WeKnora`。
4. 把上游 RBAC 相关文件清单单独记录，包括 `internal/middleware/rbac.go`、`internal/middleware/kb_access.go`、`internal/router/rbac.go`、`internal/handler/rbac_lookups.go`、`000043`、`000044`、`000045` 迁移。

### 8.3 P1 上游基线同步

首选 rebase 路径：

```bash
git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main
```

如果 rebase 在同一批文件上反复冲突，切换 rebuild 路径：

```bash
git rebase --abort
git checkout -b sync/dev-$(date +%Y%m%d)-rebuild upstream/main
```

rebuild 路径下按主题回放本地能力，不按时间顺序机械 cherry-pick：

1. 认证、租户和 API Key 兼容。
2. Organization 共享空间和组织人员管理。
3. KB/Agent 共享、共享 Agent 停用和资源计数。
4. 长文档、导出、数据分析等本地业务能力。
5. 前端页面、菜单、空间切换和本地交互。

阶段门禁：

1. 同步分支已经建立。
2. 冲突文件被归类到“编译级”“RBAC 数据模型”“后端权限链路”“前端页面”四组。
3. 每组冲突都有负责人或处理顺序，避免多人同时改同一核心文件。

### 8.4 P2 编译级冲突收敛

优先处理不会改变业务语义的冲突：

1. `go.mod` / `go.sum`：保留上游新增依赖，例如 OpenSearch、系统审计等依赖；同时保留本地仍在使用的依赖。
2. `internal/container`：保留上游新增 provider，同时确认本地 Organization、KBShare、AgentShare、长文档等 provider 未丢失。
3. `internal/router`：先恢复所有路由注册，RBAC 路由与 Organization 路由并存，不互相覆盖。
4. `config`：合并 `tenant.enable_rbac`、`auth.registration_mode`、`audit.retention_days` 等上游配置，同时保留本地已有配置项。
5. `migrations/versioned`：检查迁移编号唯一性，避免与本地已有 `000043` 等编号冲突。

建议检查命令：

```bash
rg -n "New.*Service|Provide\(|Register|tenant.enable_rbac|registration_mode|retention_days" internal config
ls migrations/versioned | sort | sed -n '1,120p'
```

阶段门禁：

1. Go 包能进入编译阶段，不再卡在明显缺失类型、重复函数、重复路由注册。
2. 前端依赖文件没有未解决冲突标记。
3. 迁移编号没有重复。

### 8.5 P3 RBAC 数据模型统一

本阶段是同步成败的核心。目标不是简单照搬上游迁移，而是把上游 RBAC 数据模型与本地 Organization 模型打通。

数据迁移落点：

1. 引入或保留 `tenant_members(user_id, tenant_id, role)`，角色集合为 `owner` / `admin` / `contributor` / `viewer`。
2. `organization_members.role` 进入兼容期：新值使用 `owner` / `admin` / `contributor` / `viewer`，历史 `editor` 继续可读可判权。
3. `organizations.owner_id` 继续保留，作为空间 Owner 的权威来源；响应层和权限层把 `owner_id` 映射为 `owner`。
4. `knowledge_bases.creator_id` 需要补齐；无法确认创建者的历史 KB 标记为租户共有资源，仅 Admin+ / Owner 可改。
5. `custom_agents.created_by` 已存在，优先沿用；如上游使用 `creator_id`，需做字段映射或兼容读取。
6. `audit_logs` 可引入，但要同时覆盖上游 RBAC 事件和本地 Organization 成员、共享关系事件。

回填规则：

| 对象 | 回填策略 |
|---|---|
| 租户 Owner | 优先租户创建者；缺失时选最早活跃管理员；仍不确定时进入人工核对 |
| Organization Owner | 使用 `organizations.owner_id`，并确保该用户在 `organization_members` 中具备 owner/admin 级别记录 |
| `editor` 角色 | 迁移为 `contributor`；兼容期 `editor` 与 `contributor` 等价 |
| KB 创建者 | 能从历史字段、上传记录或操作记录推断时回填；不能推断则设为租户共有 |
| Agent 创建者 | 沿用 `custom_agents.created_by`；空值按租户 Owner 或共有资源处理 |

建议新增或修订迁移时遵循：

1. 不复用已存在迁移编号。
2. `up.sql` 必须幂等处理历史空值和历史 `editor`。
3. `down.sql` 不得误删本地已有 Organization、KB/Agent 共享表。
4. 强制鉴权前先支持观察模式，避免迁移后大量 403。

阶段门禁：

1. 测试库迁移可重复执行。
2. 每个租户至少一个 Owner。
3. 历史 `editor` 用户仍能执行原本可执行的共享和写操作。
4. 历史 Organization、KB 共享、Agent 共享数据仍可查询。

### 8.6 P4 后端权限链路统一

后端改造按“兼容接口优先，守卫后置启用”的顺序执行。

接口与类型：

1. `internal/types/organization.go`：增加 `owner`、`contributor` 角色常量；`IsValid` 接受 `editor` 兼容值；`HasPermission` 将 `editor` 与 `contributor` 视为同级。
2. `internal/types/tenant.go` 或 RBAC 类型文件：统一租户角色枚举，避免 Organization 和 Tenant 各自定义不可比较的角色层级。
3. 响应结构保留 `is_owner`，同时新增或统一返回 `my_role = owner`。
4. 请求结构在兼容期接受 `editor`，服务层内部归一化为 `contributor`。

服务层：

1. `organizationService.CreateOrganization`：创建者写入 Owner 语义，而不是仅作为 `admin`。
2. `UpdateMemberRole`：禁止降级 Owner；Admin 不可修改 Owner；支持移交 Owner 时单独接口或明确流程。
3. `kbShareService.ShareKnowledgeBase`：共享发起条件改为 `contributor+`，兼容历史 `editor`。
4. `agentShareService.ShareAgent`：共享发起条件改为 `contributor+`，Agent 共享仍只读。
5. `KnowledgeHandler` / `kb_access`：当前租户资源走 Tenant RBAC，跨租户资源走 Organization 共享判定，再叠加源资源归属约束。
6. API Key：所属租户内映射为 Admin；不能绕过 Owner 专属能力；不能绕过 Organization 空间角色。

中间件与路由：

1. `internal/middleware/rbac.go` 可以接入，但先以观察模式或局部路由启用。
2. `internal/middleware/kb_access.go` 可作为统一入口，但必须保留本地共享 Agent 间接访问 KB 的只读路径。
3. `internal/router/rbac.go` 引入租户成员管理接口，但不替换 Organization 成员管理接口。
4. RBAC 拒绝日志写入审计表时增加去重，避免高频探测刷表。

阶段门禁：

1. Owner、Admin、Contributor、Viewer 四类角色在服务层测试中都有覆盖。
2. `editor` 兼容测试通过。
3. KB/Agent 分享、取消分享、更新分享权限、共享 Agent 停用均通过。
4. API Key 和跨租户超管回归通过。

### 8.7 P5 前端角色与页面统一

前端目标是让用户看到统一角色，但不破坏当前组织人员管理体验。

改造范围：

1. `frontend/src/api/organization/index.ts`：角色类型增加 `owner` / `contributor`，兼容 `editor`。
2. `frontend/src/stores/auth.ts`：增加租户角色状态；如果接入上游 `currentTenantRole`，仍保留当前 Organization 角色状态。
3. `OrganizationList.vue`：`is_owner` 与 `my_role=owner` 兼容显示；创建的空间仍可单独筛选。
4. `OrganizationSettingsModal.vue`：成员管理角色下拉改为 Owner / Admin / Contributor / Viewer；Owner 不允许被普通 Admin 降级或移除。
5. `ShareKnowledgeBaseDialog.vue`、`KBShareSettings.vue`、`AgentShareSettings.vue`：允许 `owner` / `admin` / `contributor` 发起共享，兼容历史 `editor`。
6. i18n 文案：`editor` 不再作为新展示名称，统一展示为“贡献者”；历史数据如返回 `editor`，前端也显示“贡献者”。

阶段门禁：

1. 组织人员管理页展示 Owner、Admin、Contributor、Viewer 正确。
2. Owner 不能被移除、不能被降级。
3. Contributor 能共享 KB / Agent，Viewer 不能共享。
4. 历史 `editor` 数据展示为“贡献者”，并仍有共享权限。
5. `pnpm build` 通过。

### 8.8 P6 验证、收口与提交

建议按以下顺序验证：

```bash
go test ./internal/types/...
go test ./internal/application/service/...
go test ./internal/handler/...
go test ./internal/middleware/...
go test ./internal/container/...
```

如果存在外部依赖失败，需要记录是环境问题还是同步引入问题。例如部分 service 测试可能依赖 Elasticsearch 等外部服务，不能简单等同于同步失败。

前端验证：

```bash
cd frontend
pnpm install --frozen-lockfile
pnpm build
```

手工回归清单：

1. 新用户创建租户后成为租户 Owner。
2. 用户创建 Organization 后成为空间 Owner。
3. Owner 不可被移除、不可被降级。
4. Admin 可管理成员，但不可删除空间 Owner。
5. Contributor 可创建和维护自己的 KB / Agent。
6. Contributor 可把本租户 KB / Agent 共享到所在空间。
7. Viewer 只能读取和提问，不能发起共享和写操作。
8. 历史 `editor` 成员升级后显示为 Contributor，能力不回退。
9. 共享 KB 的最终权限仍取共享权限与成员角色的较低者。
10. 共享 Agent 仍只读，且停用/恢复只影响当前租户视图。
11. API Key 所属租户内可用，但不能执行 Owner 专属删除能力。
12. `CanAccessAllTenants` + `X-Tenant-ID` 跨租户切换仍受开关控制。

提交建议按主题拆分：

```bash
git add migrations internal/types internal/application/repository
git commit -m "feat(rbac): add tenant role data model migration"

git add internal/application/service internal/handler internal/middleware internal/router internal/container config
git commit -m "feat(rbac): unify tenant roles with organization permissions"

git add frontend
git commit -m "feat(frontend): align organization roles with tenant rbac"

git add docs my_docs
git commit -m "docs: document rbac upstream sync rollout"
```

如果同步过程通过 rebase 得到线性历史，也可以在最终收口时保留较少提交；但每个提交的职责必须能说明清楚，不能把迁移、后端守卫、前端页面和文档全部混在一个无法审查的大提交里。

最终推送：

```bash
git push -u origin sync/dev-$(date +%Y%m%d)-upstream
# 复核通过后
git push --force-with-lease origin HEAD:dev
```

### 8.9 风险与回滚

| 风险 | 表现 | 预防措施 | 回滚措施 |
|---|---|---|---|
| Owner 回填错误 | 用户无法管理自己的租户或空间 | 生成 Owner 回填审计清单，人工确认异常租户 | 临时关闭强制 RBAC，修正 `tenant_members` / `organization_members` |
| `editor` 兼容遗漏 | 历史编辑者失去共享或写权限 | 服务层统一归一化角色，前端和后端同时接受 `editor` | 回滚角色迁移或补充兼容判断 |
| `creator_id` 缺失 | Contributor 无法维护历史 KB / Agent | 迁移前统计空值，无法判断的设为租户共有 | 关闭归属强制守卫，补充回填 |
| Organization 被 RBAC 替代 | 组织人员管理、共享空间页面不可用 | 路由和服务并存，不用租户成员接口替换组织成员接口 | 回退相关路由注册和前端入口 |
| API Key 行为变化 | 脚本调用 403 | API Key 映射为所属租户 Admin，Owner 专属能力单独限制 | 临时切回兼容认证逻辑 |
| 审计表写入过量 | 权限拒绝导致数据库压力上升 | 使用 1 分钟滑动窗口去重 | 关闭拒绝事件写入或调低审计级别 |
| 迁移回滚误删共享表 | 回滚后共享关系丢失 | `down.sql` 禁止删除本地已有核心表 | 使用 backup 分支或数据库备份恢复 |

### 8.10 最小可交付版本

如果一次性完整统一风险过高，可以先交付最小版本：

1. 引入四级角色枚举和兼容层：`owner` / `admin` / `contributor` / `viewer`，历史 `editor` 等同 `contributor`。
2. Organization 创建者在响应和前端显示为 Owner，但数据库可暂时继续用 `owner_id` + `admin` 成员记录。
3. KB/Agent 共享判断改为 `contributor+`，兼容 `editor`。
4. 引入 `tenant_members` 和审计表，但 `tenant.enable_rbac=false` 观察运行。
5. 前端完成角色展示统一和 Owner 保护展示。
6. 完成上述回归后，再在下一阶段启用 `creator_id` 强制守卫和 `tenant.enable_rbac=true`。

这个最小版本可以先解决角色命名、Owner 显示和历史 editor 兼容问题，同时避免一次性强制上游 RBAC 导致现有组织人员管理和共享能力大面积回退。

## 9. 推荐同步执行流程

### 9.1 同步前备份

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch
git branch backup/dev-$(date +%Y%m%d)-before-upstream-sync
git tag sync-dev-$(date +%Y%m%d)-pre
```

检查点：

1. 工作区必须 clean。
2. `dev` 必须对齐 `origin/dev`。
3. 备份分支和 tag 必须创建成功。

### 9.2 创建同步分支并 rebase

```bash
git fetch origin --prune
git fetch upstream --prune
git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main
```

冲突处理时按以下优先级分批解决：

1. 先解决构建级冲突：依赖、导入、类型签名、路由注册、DI 注册。
2. 再解决权限模型冲突：Tenant RBAC、Organization、KB/Agent 共享、Auth。
3. 最后解决前端行为冲突：菜单、空间切换、按钮权限、共享弹窗。

### 9.3 RBAC 冲突专门复核

每次解决 RBAC 相关冲突后，必须执行人工复核：

1. 搜索 `tenant_members` 是否完成 Owner 唯一性、Admin、Contributor、Viewer 的回填策略。
2. 搜索 `organization_members.role` 是否同时兼容 `owner`、`contributor` 和历史 `editor`。
3. 搜索 `enable_rbac` 是否在迁移完成前默认导致强制拦截。
4. 搜索 `RequireRole`、`RequireOwnershipOrRole` 是否挂到了尚未完成 `creator_id` 回填的既有路由。
5. 搜索 `creator_id` 是否改变本地 KB 列表、上传、删除、共享行为。
6. 检查 `kb_shares`、`agent_shares`、`organization_members` 的读写路径是否仍然可用。
7. 检查 API Key 调用是否仍然能走本地兼容路径，并映射为所属租户 Admin。

建议命令：

```bash
rg -n "enable_rbac|tenant_members|RequireRole|RequireOwnershipOrRole|creator_id|runnable_by_viewer|audit_logs" internal frontend config migrations
rg -n "OrgRoleEditor|editor|contributor|owner|is_owner|organization_members|kb_shares|agent_shares|tenant_disabled_shared_agents|CanAccessAllTenants|EnableCrossTenantAccess" internal frontend docs migrations
```

### 9.4 不可控时的兜底方案

如果 Tenant RBAC 与本地 Organization 体系冲突过多，立即中止 rebase：

```bash
git rebase --abort
git checkout -b sync/dev-$(date +%Y%m%d)-rebuild upstream/main
```

随后按主题回放本地能力：

1. 基础认证与租户兼容。
2. Organization 共享空间。
3. KB/Agent 共享与共享 Agent 停用。
4. 长文档、导出、数据分析等本地定制功能。
5. 前端共享空间和本地页面行为。

该路径下，RBAC 相关上游代码仍然按“先保留本地能力，再统一角色与守卫”的原则处理。重建分支不是放弃上游 RBAC，而是降低一次性 rebase 的冲突复杂度。

## 12. 最终结论

本轮上游同步的关键不是简单解决 Git 冲突，而是把上游 Tenant RBAC 与本地 Organization 权限体系统一起来，同时不丢失本地已经落地的组织人员管理、知识库共享、智能体共享和资源归属控制。

最终裁决如下：

1. 普通代码冲突：优先吸收上游结构和修复，再恢复本地业务语义。
2. 功能逻辑冲突：以本地代码和本地产品行为为准。
3. RBAC 冲突：可以统一到上游 `owner` / `admin` / `contributor` / `viewer` 模型。
4. Owner：本地 `owner_id` 需要角色化为 `owner`，并保留不可移除、不可降级和删除空间保护。
5. contributor/editor：本地 `editor` 与上游 `contributor` 在产品职责上基本一致，应统一为 `contributor`，同时兼容历史 `editor`。
6. Organization：继续作为跨租户协作通道存在，不被 Tenant RBAC 成员管理替代。
7. 同步结果：必须通过后端、前端、数据库迁移和共享空间回归验证后，才能整体提交到 `dev` 分支。
