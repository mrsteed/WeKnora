# 20260527 上游代码同步差异与 RBAC 保留方案

## 1. 文档目标

本文档基于 `my_docs/20260429-04-通用上游同步操作手册.md`，用于指导本仓库从 `Tencent/WeKnora` 上游主线同步代码时的差异判断、冲突处理和最终提交。

本轮同步必须同时满足以下目标：

1. 吸收 `upstream/main` 的最新代码、修复和公共能力。
2. 逻辑冲突、产品行为冲突、权限模型冲突统一以本地代码和本地设计为准。
3. 对租户 RBAC 相关冲突采取特别保护策略：上游 RBAC 可作为参考和可选能力来源，但不得替换本地已经落地的 Organization 共享空间权限体系。
4. 同步完成、验证通过后，整体结果提交到本仓库 `dev` 分支。

---

## 2. 当前同步基线

### 2.1 远端约定

当前仓库远端符合通用手册约定：

```bash
origin   https://github.com/mrsteed/WeKnora.git
upstream https://github.com/Tencent/WeKnora.git
```

本地开发分支为：

```bash
dev
```

`dev` 应跟踪 `origin/dev`，不能直接跟踪 `upstream/main`。

### 2.2 本轮差异规模

在 2026-05-27 拉取远端后，本地 `dev` 与 `upstream/main` 的差异统计为：

```bash
git rev-list --left-right --count dev...upstream/main
# 48 225
```

含义：

1. 本地 `dev` 相对上游仍有 48 个本地提交需要保留。
2. 上游 `main` 相对本地有 225 个提交需要评估吸收。
3. 本轮同步不是小范围补丁合并，权限、系统设置、模型、检索、前端交互等模块都可能出现冲突。

上游最新主线包含如下近期提交方向：

1. 系统审计与系统管理员设置改造。
2. Agent 意图提示词、内置模型、OpenSearch、Doris、Yuque 等功能修复和增强。
3. 租户 RBAC 相关文档、迁移、后端守卫和前端表现。

## 3. 总体裁决原则

### 3.1 默认同步方式

按通用手册优先采用以下方式：

```bash
git checkout dev
git pull --ff-only origin dev
git branch backup/dev-$(date +%Y%m%d)-before-upstream-sync
git tag sync-dev-$(date +%Y%m%d)-pre
git fetch origin --prune
git fetch upstream --prune
git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main
```

如果 `rebase` 冲突不可控，再改用“上游新基线 + 本地主题回放”的兜底方案。

### 3.2 冲突裁决

出现逻辑冲突时，执行以下顺序：

1. 先判断冲突是否影响本地已上线或已验证功能。
2. 若影响本地已上线功能，则以本地行为为准。
3. 在保留本地行为的前提下，吸收上游非冲突的安全修复、兼容性修复、测试补充和结构优化。
4. 不允许用上游默认实现直接覆盖本地 Organization、租户、权限、知识库、Agent、长文档、导出和数据分析等定制能力。

### 3.3 RBAC 特别原则

租户 RBAC 是本轮最高风险冲突域。处理规则如下：

1. 上游 Tenant RBAC 不作为本地权限主模型直接落地。
2. 本地已实现的 Organization 共享空间模型继续作为权限和跨租户协作主模型。
3. 上游 RBAC 中可复用的审计、配置、测试、前端提示等能力，只能在不改变本地权限语义的前提下选择性迁移。
4. 如果上游代码要求默认启用 `tenant.enable_rbac=true`，本地必须改为显式兼容策略，不能让同步后系统默认进入强制租户 RBAC。

## 4. 本地权限设计基线

本地当前权限体系不是上游的 Tenant RBAC，而是“租户隔离 + Organization 共享空间”。这一点在 `docs/权限管理系统技术开发文档.md` 中已有明确说明。

### 4.1 本地主模型

本地已实现模型如下：

1. 所有基础业务数据通过 `tenant_id` 做租户隔离。
2. 跨租户协作通过 Organization 共享空间完成。
3. 共享空间成员角色为 `admin` / `editor` / `viewer`。
4. 知识库通过 `kb_shares` 共享到 Organization。
5. Agent 通过 `agent_shares` 共享到 Organization。
6. Agent 可通过 `tenant_disabled_shared_agents` 在目标租户侧做个人化停用。
7. 跨租户管理能力仍沿用 `users.can_access_all_tenants` + `tenant.enable_cross_tenant_access` + `X-Tenant-ID`。

核心代码落点：

1. `internal/types/organization.go`：Organization、成员角色、KB/Agent 共享数据结构。
2. `internal/application/service/organization.go`：组织创建、加入、成员角色、审批和邀请码逻辑。
3. `internal/application/service/kbshare.go`：知识库共享、共享权限更新、共享访问权限计算。
4. `internal/application/service/agent_share.go`：Agent 共享、只读共享、共享 Agent 访问关联知识库。
5. `internal/handler/knowledge.go`：知识库访问时按本租户、KB 共享、Agent 共享三条路径判断。
6. `internal/middleware/auth.go`：JWT、API Key、跨租户切换认证。

### 4.2 本地同租户权限语义

当前本地实现中，用户访问本租户知识库时，只要 `kb.tenant_id == 当前 tenant_id`，后端会按本租户资源处理，知识库访问函数返回 `OrgRoleAdmin` 等价权限。

这意味着本地目前没有上游那种 `tenant_members(user_id, tenant_id, role)` 级别的租户内成员矩阵，也没有在所有租户内写操作上强制区分 viewer / contributor / admin / owner。

同步时不能把上游同租户 RBAC 强制接入本地所有接口，否则会直接改变现有用户、API Key、前端入口和组织共享行为。

### 4.3 本地跨租户共享语义

本地跨租户知识库访问遵循以下逻辑：

1. 如果知识库属于当前租户，则按本租户资源访问。
2. 如果知识库不属于当前租户，则检查用户是否通过某个 Organization 获得共享访问。
3. 有共享关系时，最终权限取“共享权限”和“用户在空间内角色”的较低权限。
4. 如果没有直接 KB 共享，但用户拥有可访问该 KB 的共享 Agent，则允许只读路径访问。

这套规则是本地共享空间功能的核心，必须保留。

### 4.4 本地数据模型边界

本地已落地或文档确认的数据模型包括：

| 本地对象 | 作用 |
|---|---|
| `organizations` | 共享空间主体 |
| `organization_members` | 共享空间成员与空间角色 |
| `kb_shares` | 知识库共享关系 |
| `agent_shares` | Agent 共享关系 |
| `tenant_disabled_shared_agents` | 租户侧隐藏共享 Agent 偏好 |
| `users.can_access_all_tenants` | 跨租户超管能力 |
| `custom_agents.created_by` | Agent 创建者字段，已存在 |

本地当前没有以下上游 RBAC 主表或主字段作为运行前提：

1. `tenant_members`。
2. `audit_logs` 的 RBAC 事件体系。
3. `knowledge_bases.creator_id`。
4. `custom_agents.runnable_by_viewer`。
5. `internal/middleware/rbac.go` 风格的全局角色守卫。
6. `RequireRole` / `RequireOwnershipOrRole` 类租户 RBAC 中间件。

## 5. 上游 Tenant RBAC 设计摘要

上游文档 `docs/RBAC说明.md` 说明，Tenant RBAC 已随上游 #1303 发布，并由 `tenant.enable_rbac` 控制，默认强制鉴权。

### 5.1 上游角色矩阵

上游每个租户成员在 `tenant_members` 中拥有且仅拥有一个角色：

| 角色 | 标识 | 能力 |
|---|---|---|
| 只读 | `viewer` | 仅查阅、提问，不可发起变更 |
| 贡献者 | `contributor` | 可上传文档、维护自己创建的 KB / Agent |
| 管理员 | `admin` | 可变更租户内任意资源，管理成员和共享基础设施 |
| Owner | `owner` | Admin 全部权限 + 删除租户；每个租户有且只有一个 |

角色层级为：

```text
viewer < contributor < admin < owner
```

### 5.2 上游资源归属模型

上游通过迁移 `000043_tenant_rbac` 给关键资源加入归属字段：

1. `knowledge_bases.creator_id`：老数据回填为租户 Owner，空值表示租户共有资源，仅 Admin+ 可变更。
2. `custom_agents.creator_id`：Agent 创建者。
3. `custom_agents.runnable_by_viewer`：控制 Viewer 是否可在对话中运行该 Agent。

上游由此形成两类守卫：

1. 角色守卫：`Viewer()` / `Contributor()` / `Admin()` / `Owner()`。
2. 归属守卫：`OwnedKBOrAdmin()` / `OwnedAgentOrAdmin()` / `OwnedChunkKBOrAdmin()` 等。

目标语义是：Contributor 在自己的资源里接近 Owner，在别人的资源里接近 Viewer。

### 5.3 上游共享空间关系

上游认为 Tenant RBAC 与 Organization 共享空间是正交关系：

1. Tenant RBAC 解决同一租户内能做什么。
2. Organization 解决跨租户把 KB / Agent 共享给谁。
3. 跨租户写操作需要同时满足共享权限、空间角色和源租户 RBAC。

上游关键判定顺序为：

```text
KB 属于当前租户？
  是：进入租户 RBAC，按角色 + creator_id 判定
  否：检查是否通过共享空间共享给当前用户
      是：按空间角色与共享权限判定
      否：403 / 404
```

### 5.4 上游配置与审计

上游新增配置包括：

```yaml
tenant:
  enable_rbac: true
  enable_cross_tenant_access: false

auth:
  registration_mode: self_serve

audit:
  retention_days: 90
```

对应环境变量包括：

| 环境变量 | YAML 路径 |
|---|---|
| `WEKNORA_TENANT_ENABLE_RBAC` | `tenant.enable_rbac` |
| `WEKNORA_AUDIT_RETENTION_DAYS` | `audit.retention_days` |

上游还新增 `audit_logs`，记录 `rbac.member_added`、`rbac.member_removed`、`rbac.member_role_changed`、`rbac.member_left`、`rbac.access_denied` 等事件，并带有保留期清理任务。

### 5.5 上游相关文件

上游 `upstream/main` 中与 RBAC 直接相关的文件包括：

1. `docs/RBAC说明.md`。
2. `internal/middleware/rbac.go`。
3. `internal/middleware/kb_access.go`。
4. `internal/router/rbac.go`。
5. `internal/handler/rbac_lookups.go`。
6. `migrations/versioned/000043_tenant_rbac.up.sql` / `.down.sql`。
7. `migrations/versioned/000044_audit_log.up.sql` / `.down.sql`。
8. `migrations/versioned/000045_org_tenant_members.up.sql` / `.down.sql`。
9. `internal/middleware/rbac_test.go`、`internal/middleware/kb_access_test.go`、`internal/middleware/rbac_audit_test.go`、`internal/handler/rbac_lookups_test.go`。

## 6. 本地与上游 RBAC 差异对照

| 维度 | 本地实现 | 上游 RBAC | 同步裁决 |
|---|---|---|---|
| 权限主模型 | 租户隔离 + Organization 共享空间 | Tenant RBAC + Organization 正交校验 | 保留本地主模型 |
| 租户内角色 | 无 `tenant_members` 强制矩阵 | viewer / contributor / admin / owner | 不直接启用上游矩阵 |
| 组织角色 | admin / editor / viewer | 共享空间仍有角色，但受源租户 RBAC 叠加限制 | 保留本地 Organization 角色语义 |
| 同租户 KB 写权限 | 租户匹配即按本地租户资源处理 | 角色 + `creator_id` 判定 | 保留本地现状，不强制回归上游 |
| 跨租户 KB 写权限 | 共享权限与空间角色取较低权限 | 还要叠加源租户 RBAC | 保留本地共享路径，不增加源租户 RBAC 阻断 |
| API Key | API Key 解析租户，找不到用户时构造系统虚拟用户 | API Key 在所属租户内固定 Admin，删除租户仍需 Owner | 保留本地 API Key 兼容逻辑 |
| 跨租户超管 | `CanAccessAllTenants` + `EnableCrossTenantAccess` + `X-Tenant-ID` | 同样支持，且等同目标租户 Admin | 保留本地实现，可吸收安全校验 |
| 资源归属字段 | `custom_agents.created_by` 已存在；KB 无 `creator_id` | KB / Agent 都使用 `creator_id` | 不自动把上游 `creator_id` 作为本地主模型 |
| 审计日志 | 目前无上游 RBAC 审计主流程 | `audit_logs` + RBAC 事件 + 保留期清理 | 可作为后续增强，不作为本次强制前置 |
| 前端权限 | 主要围绕共享空间、菜单、资源列表控制 | `authStore.currentTenantRole` + `hasRole` + owner 判断 | 不替换本地前端权限状态 |
| 迁移策略 | 已有本地组织共享迁移 | 新增 `000043` / `000044` / `000045` | 不无脑执行语义改造，需本地化评估 |

## 7. RBAC 同步措施

### 7.1 必须保留的本地能力

同步时以下能力必须保持行为不变：

1. 用户可创建、加入、搜索 Organization 共享空间。
2. Organization 创建者自动成为空间管理员，且创建者保护逻辑不变。
3. 空间角色继续使用 `admin` / `editor` / `viewer`。
4. 只有空间 `admin` / `editor` 可将本租户 KB 或 Agent 共享到空间。
5. KB 共享继续支持 `viewer` / `editor` / `admin` 权限，并与成员角色取较低权限。
6. Agent 共享继续保持只读共享语义。
7. 租户侧可停用共享 Agent，不影响共享关系本身。
8. 通过共享 Agent 间接访问 KB 的只读路径必须保留。
9. `users.can_access_all_tenants` 的跨租户切换兼容能力必须保留。
10. API Key 找不到真实用户时构造系统虚拟用户的兼容逻辑必须保留。

### 7.2 不应直接落地的上游内容

以下上游内容不能在本地未完成产品确认前直接启用：

1. `tenant.enable_rbac` 默认强制开启。
2. `tenant_members` 作为所有租户内操作的强制前置校验。
3. `knowledge_bases.creator_id` 作为本地 KB 写权限唯一依据。
4. `RequireRole` / `RequireOwnershipOrRole` 对既有接口的全量接管。
5. 源租户 RBAC 对跨租户共享写操作的额外阻断。
6. 上游前端 `currentTenantRole` 对本地菜单和按钮展示的直接替换。
7. 上游 RBAC 成员管理接口对本地 Organization 成员管理接口的替换。

### 7.3 可选择吸收的上游内容

以下内容可以在不改变本地权限语义时选择性吸收：

1. 审计日志的基础表结构和保留期清理机制，但事件名和调用点需本地化。
2. RBAC 拒绝日志去重思路，用于本地 Organization 权限拒绝审计。
3. 跨租户超管访问的安全校验和测试补充。
4. 权限相关单元测试的组织方式。
5. 前端权限按钮防闪烁、加载态、角色徽章等交互细节。
6. 文档中对 Tenant RBAC 与 Organization 正交关系的解释，但要标明本地当前未强制 Tenant RBAC。

### 7.4 文件级冲突处理建议

| 文件或模块 | 处理建议 |
|---|---|
| `internal/types/organization.go` | 保留本地字段、树形组织字段、共享结构和 `HasPermission` 语义 |
| `internal/application/service/organization.go` | 保留本地邀请码、审批、成员上限、搜索加入、升级申请逻辑 |
| `internal/application/service/kbshare.go` | 保留本地共享权限计算，不叠加源租户 RBAC 强制阻断 |
| `internal/application/service/agent_share.go` | 保留 Agent 只读共享、配置完整性校验和停用逻辑 |
| `internal/handler/knowledge.go` | 保留本租户、KB 共享、Agent 共享三路径访问判定 |
| `internal/middleware/auth.go` | 保留本地 JWT / API Key / 系统虚拟用户兼容逻辑 |
| `internal/middleware/rbac.go` | 不作为全局默认守卫接入；如保留，应改为可选或待本地化模块 |
| `internal/middleware/kb_access.go` | 可参考其结构，但不能替换本地知识库共享判定 |
| `internal/router/rbac.go` | 不替换本地 Organization 路由；如引入需放在明确隔离的可选接口下 |
| `internal/handler/rbac_lookups.go` | 可参考查询能力，不作为本地成员权限真源 |
| `migrations/versioned/000043_*` | 不能无脑把 `creator_id` 作为本地 KB 权限依据；如保留迁移，需确认默认关闭 enforcement |
| `migrations/versioned/000044_*` | 可评估审计表，但事件写入点要本地化 |
| `migrations/versioned/000045_*` | 需确认是否与本地 `organization_members` 语义冲突，不能破坏现有成员数据 |
| `frontend/src/stores/auth.ts` | 不直接替换本地认证状态；如吸收角色字段，必须兼容 Organization 当前状态 |
| Organization 前端页面 | 保留本地空间列表、共享设置、Agent 停用和资源数量展示 |

## 8. 推荐同步执行流程

### 8.1 同步前备份

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch
git branch backup/dev-$(date +%Y%m%d)-before-upstream-sync
git tag sync-dev-$(date +%Y%m%d)-pre
```

检查点：

1. 工作区必须 clean。
2. `dev` 必须对齐 `origin/dev`。
3. 备份分支和 tag 必须创建成功。

### 8.2 创建同步分支并 rebase

```bash
git fetch origin --prune
git fetch upstream --prune
git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main
```

冲突处理时按以下优先级分批解决：

1. 先解决构建级冲突：依赖、导入、类型签名、路由注册、DI 注册。
2. 再解决权限模型冲突：Tenant RBAC、Organization、KB/Agent 共享、Auth。
3. 最后解决前端行为冲突：菜单、空间切换、按钮权限、共享弹窗。

### 8.3 RBAC 冲突专门复核

每次解决 RBAC 相关冲突后，必须执行人工复核：

1. 搜索 `tenant_members` 是否被接入现有 Organization 主流程。
2. 搜索 `enable_rbac` 是否默认导致强制拦截。
3. 搜索 `RequireRole`、`RequireOwnershipOrRole` 是否挂到了本地已有路由。
4. 搜索 `creator_id` 是否改变本地 KB 列表、上传、删除、共享行为。
5. 检查 `kb_shares`、`agent_shares`、`organization_members` 的读写路径是否仍然可用。
6. 检查 API Key 调用是否仍然能走本地兼容路径。

建议命令：

```bash
rg -n "enable_rbac|tenant_members|RequireRole|RequireOwnershipOrRole|creator_id|runnable_by_viewer|audit_logs" internal frontend config migrations
rg -n "organization_members|kb_shares|agent_shares|tenant_disabled_shared_agents|CanAccessAllTenants|EnableCrossTenantAccess" internal frontend docs migrations
```

### 8.4 不可控时的兜底方案

如果 Tenant RBAC 与本地 Organization 体系冲突过多，立即中止 rebase：

```bash
git rebase --abort
```

## 11. 最终结论

本轮上游同步的关键不是简单解决 Git 冲突，而是避免上游 Tenant RBAC 在同步过程中替换本地已经落地的权限架构。

最终裁决如下：

1. 普通代码冲突：优先吸收上游结构和修复，再恢复本地业务语义。
2. 功能逻辑冲突：以本地代码和本地产品行为为准。
3. RBAC 冲突：保留本地“租户隔离 + Organization 共享空间”主模型。
4. 上游 Tenant RBAC：作为参考和可选增强来源，不作为本次同步的默认强制权限模型。
5. 同步结果：必须通过后端、前端、数据库迁移和共享空间回归验证后，才能整体提交到 `dev` 分支。
