# 知识库内部文档列表视图改造设计方案

## 1. 背景

当前知识库内部文档页采用卡片视图展示文档条目，入口位于 frontend/src/views/knowledge/KnowledgeBase.vue，数据加载逻辑由 frontend/src/hooks/useKnowledgeBase.ts 驱动。现有方案适合浏览摘要，但在以下场景存在不足：

1. 当单个知识库内文档数量较多时，卡片视图的信息密度偏低，用户很难快速对比文件类型、更新时间、分类、解析状态等字段。
2. 用户无法在同一屏内直接查看“源文件”相关信息，只能依赖卡片摘要与详情抽屉间接判断。
3. 虽然前端已经存在原文件下载能力，接口为 GET /api/v1/knowledge/{id}/download，但知识库内部列表页没有把该能力直接暴露在主列表交互中。

基于以上问题，需要为“知识库内部文档列表页面”增加可切换的列表视图，并在列表视图中直接暴露源文件信息与下载入口。

## 2. 目标

### 2.1 业务目标

1. 在知识库内部文档页支持“卡片视图 / 列表视图”切换。
2. 列表视图保留现有筛选、搜索、滚动加载、权限控制能力。
3. 列表视图中能够直接看到文件信息与源文件信息，并支持源文件下载。

### 2.2 交互目标

1. 用户切换视图后无需离开当前页面或重新理解操作模型。
2. 卡片视图继续保留，作为默认浏览模式。
3. 列表视图优先服务“检索、比对、批量扫描”场景。

### 2.3 技术目标

1. 尽量复用现有知识文件分页接口 GET /api/v1/knowledge-bases/{id}/knowledge。
2. 尽量复用现有原文件下载接口 GET /api/v1/knowledge/{id}/download。
3. 变更聚焦前端页面层，不在第一阶段引入新的后端接口。

## 3. 现状分析

## 3.1 页面与数据入口

当前文档页主实现位于以下位置：

1. frontend/src/views/knowledge/KnowledgeBase.vue
2. frontend/src/hooks/useKnowledgeBase.ts
3. frontend/src/api/knowledge-base/index.ts

现有页面已经具备以下基础能力：

1. 文档搜索：keyword
2. 文件类型筛选：file_type
3. 分类筛选：tag_id
4. 分页与滚动加载：page、page_size
5. 文档详情查看
6. 文档重建、移动、删除
7. 原文件下载：downKnowledgeDetails(id)

## 3.2 当前列表返回字段

后端 Knowledge 类型已包含以下可直接用于列表视图的字段：

1. id
2. title
3. description
4. source
5. channel
6. parse_status
7. summary_status
8. file_name
9. file_type
10. file_size
11. created_at
12. updated_at
13. error_message
14. tag_id

这意味着列表视图所需的大部分字段已经可用，不需要额外新增查询接口。

## 3.3 当前问题归纳

1. 文档页仅有卡片视图，无法在高密度模式下查看多字段。
2. 原文件下载能力没有在主列表显式呈现。
3. 文件相关字段虽然存在，但主要被卡片标题和悬浮气泡消费，未形成结构化表格。
4. 用户需要在“看摘要”和“操作文件”之间频繁切换上下文。

## 4. 设计范围

本次设计只覆盖“知识库内部文档页”的 documents 视图，不覆盖以下区域：

1. FAQ 管理页
2. Wiki 浏览页
3. 知识库外层列表页
4. 文档详情抽屉的内容结构

## 5. 设计原则

1. 默认不改变现有卡片视图行为，降低回归风险。
2. 列表视图与卡片视图共享同一份数据源，避免双份状态分叉。
3. 下载行为优先采用显式按钮，避免用户误触整行。
4. 对无源文件条目保持可见但降级展示，避免列表信息不一致。
5. 第一阶段先实现高价值基础列，不引入复杂批量操作。

## 6. 交互方案

## 6.1 视图切换入口

在文档页现有顶部工具栏中，搜索框和文件类型筛选器右侧新增“视图切换控件”。

建议方案：

1. 使用双态切换按钮或 Segmented Control。
2. 提供两个选项：卡片视图、列表视图。
3. 默认值为卡片视图。
4. 切换后立即在当前页生效，不重新请求接口。

建议文案：

1. 卡片视图
2. 列表视图

建议图标：

1. card-view 使用 app 或 dashboard 类图标
2. list-view 使用 bulletpoint 或 table 类图标

## 6.2 视图状态持久化

建议分两级处理：

1. 第一阶段：仅当前页面会话内记忆，使用 ref 或本地 store。
2. 第二阶段：写入 localStorage，按知识库页维度记住用户上次选择。

若本次实现周期有限，优先做第一阶段；设计上预留持久化扩展点。

## 6.3 卡片视图保持不变

卡片视图继续使用当前实现，不调整已有卡片 DOM 结构和操作逻辑。这样可以确保：

1. 现有用户习惯不受影响。
2. 现有 hover 气泡、更多菜单、详情抽屉全部可复用。
3. 新增功能的变更面主要集中在列表视图新增区域。

## 6.4 列表视图布局

列表视图采用高密度表格布局，建议使用前端已有表格组件体系实现。

推荐列定义如下：

1. 文档名称
2. 源文件
3. 文件类型
4. 文件大小
5. 分类
6. 解析状态
7. 更新时间
8. 操作

### 6.4.1 文档名称列

展示内容：

1. 优先显示 file_name
2. 若无 file_name，则回退 title
3. 副标题可展示 description 首行摘要或 source

交互：

1. 点击名称进入现有详情抽屉
2. 解析失败时在名称旁显示失败状态色标

### 6.4.2 源文件列

该列是本次需求的核心列，建议展示为“源文件信息 + 下载入口”的组合结构。

展示规则：

1. 文件导入类型：显示原始文件名，支持下载。
2. URL 导入类型：显示来源 URL，不显示下载按钮，展示“网页来源”状态。
3. 手动创建类型：显示“手动创建”，不显示下载按钮。

推荐单元格结构：

1. 第一行：源文件名称或来源标识
2. 第二行：一个轻量标签，标识 文件导入 / 网页导入 / 手动创建
3. 右侧：下载按钮，仅在可下载时显示

### 6.4.3 文件类型列

直接展示 file_type，保持大写风格，与现有卡片保持一致。

示例：

1. PDF
2. DOCX
3. XLSX
4. MANUAL
5. URL

### 6.4.4 文件大小列

直接基于 file_size 做格式化展示，复用页面现有 formatFileSize 能力。

展示规则：

1. 有值时展示格式化结果，如 2.4 MB
2. 无文件大小时展示 --

### 6.4.5 分类列

复用现有 getTagName(item.tag_id) 逻辑。

展示规则：

1. 有分类显示标签名
2. 无分类显示“未分类”或现有默认值

### 6.4.6 解析状态列

复用当前卡片视图里的状态语义，但改成表格状态标签。

状态映射：

1. pending / processing：解析中
2. failed：解析失败
3. draft：草稿
4. completed 且 summary_status 为 pending / processing：生成摘要中
5. completed：已完成

若失败，建议支持 hover 查看 error_message。

### 6.4.7 更新时间列

复用当前 formatDocTime(item.updated_at) 格式。

### 6.4.8 操作列

建议提供以下操作：

1. 查看
2. 下载源文件
3. 更多

说明：

1. 查看：打开现有详情抽屉
2. 下载源文件：仅文件导入类型显示
3. 更多：复用现有重建、移动、删除、编辑文档等逻辑

## 7. 源文件下载设计

## 7.1 接口复用

直接复用现有前端 API：

1. frontend/src/api/knowledge-base/index.ts 中的 downKnowledgeDetails(id)

该方法已经对应后端接口：

1. GET /api/v1/knowledge/{id}/download

因此本次不新增后端下载接口。

## 7.2 可下载判定

建议前端增加一个统一判定函数，例如：

1. 当 item.type 为 manual 时，不可下载
2. 当 item.source 为 manual 时，不可下载
3. 当 item.file_name 为空且无 file_path 语义时，不可下载
4. 其余文件导入类型默认可下载

若当前实际数据里 URL 导入条目也保存了 file_name，需要结合 item.type 或 source 做更稳妥判定，避免把 URL 误识别为本地文件。

## 7.3 下载交互

点击下载后行为：

1. 直接触发浏览器下载
2. 按钮进入短暂 loading 或 disabled 状态，避免重复点击
3. 下载失败时统一使用 MessagePlugin 提示

建议提示文案：

1. 下载中
2. 下载失败，请稍后重试
3. 当前条目不支持原文件下载

## 8. 页面结构建议

## 8.1 状态拆分

在 KnowledgeBase.vue 中新增视图状态：

1. documentViewMode: 'card' | 'list'

同时抽离一个列表视图专用的 columns 配置与辅助方法，避免把表格逻辑全部塞进模板。

## 8.2 组件拆分建议

推荐采用“保守拆分”方案。

第一阶段可选两种实现路径：

### 方案 A：直接在 KnowledgeBase.vue 内增量实现

优点：

1. 改动集中
2. 便于复用现有局部方法
3. 上线速度快

缺点：

1. 主文件会继续膨胀

### 方案 B：新增列表子组件

建议子组件名：

1. frontend/src/views/knowledge/components/KnowledgeDocumentTable.vue

优点：

1. 主页面职责更清晰
2. 卡片与列表视图边界清楚
3. 便于后续扩展批量操作、排序、列设置

缺点：

1. 需要梳理 props 和事件

本次从长期维护性考虑，建议采用方案 B。

## 8.3 推荐组件职责

### KnowledgeBase.vue

负责：

1. 页面级状态
2. 搜索、筛选、分页加载
3. 详情抽屉
4. 上传、删除、重建、移动等业务动作
5. 视图切换

### KnowledgeDocumentTable.vue

负责：

1. 列表视图渲染
2. 列定义
3. 下载按钮交互
4. 行内查看和更多菜单触发

## 9. 数据与字段设计

## 9.1 前端类型补充建议

当前 KnowledgeCard 类型较偏向卡片语义，建议新增更中性的文档项类型，避免继续以 card 命名承载表格。

建议定义：

1. KnowledgeDocumentItem

字段建议至少包含：

1. id
2. title
3. file_name
4. original_file_name
5. file_type
6. file_size
7. source
8. type
9. parse_status
10. summary_status
11. description
12. updated_at
13. created_at
14. tag_id
15. error_message
16. channel

## 9.2 派生字段建议

列表渲染时可计算下列派生字段：

1. display_name
2. source_display_text
3. source_kind
4. formatted_file_size
5. status_label
6. status_theme

## 10. 与现有能力的兼容性

## 10.1 搜索与筛选

列表视图与卡片视图共享 cardList 数据源，因此：

1. 搜索关键词继续生效
2. 文件类型筛选继续生效
3. 分类筛选继续生效

无需新增接口参数。

## 10.2 无限滚动

当前页面采用滚动容器 + 分页追加的方式加载数据。

列表视图建议继续沿用该模式：

1. 卡片视图和列表视图共用同一个滚动容器
2. 切换视图不重置分页
3. 列表视图下滚动到底部时继续触发 append

## 10.3 权限控制

列表视图中操作按钮沿用现有权限逻辑：

1. 查看：有读权限即可
2. 下载：有读权限即可
3. 编辑、重建、移动、删除：沿用 canEdit / canManage

## 10.4 文档详情

点击列表行或“查看”按钮时，继续复用 openCardDetails(item) 与 getCardDetails(item) 逻辑，不额外新增详情接口。

## 11. 实施方案

## 11.1 第一阶段

目标：完成最小可用版本。

实施内容：

1. 增加视图切换控件
2. 增加列表视图组件
3. 复用现有数据源渲染表格
4. 增加源文件列
5. 增加下载按钮
6. 保持卡片视图原样

## 11.2 第二阶段

可选增强项：

1. 记忆用户上次视图选择
2. 列宽拖拽与表头固定
3. 批量下载或批量删除
4. 列显隐配置
5. 表头排序

## 12. 风险与对策

## 12.1 主页面复杂度继续升高

风险：

1. KnowledgeBase.vue 已较大，继续堆叠可能降低可维护性。

对策：

1. 将列表视图独立为子组件。

## 12.2 数据字段语义不完全统一

风险：

1. manual、url、file 三类知识条目的 source / file_name 语义可能不完全一致。

对策：

1. 在前端统一封装 source_kind 判定逻辑。
2. 对不可下载场景提供明确禁用态和文案。

## 12.3 下载按钮误导

风险：

1. 若 URL 导入或手动创建条目展示了下载按钮，会造成用户误解。

对策：

1. 将下载按钮完全建立在 can_download_source 判定之上。
2. 对不可下载场景只展示来源类型，不展示伪按钮。

## 12.4 无限滚动与表格渲染联动

风险：

1. 表格容器切换后可能影响现有 scroll 监听。

对策：

1. 保持滚动容器节点不变，只替换内部内容区域。
2. 先复用当前 knowledgeScroll 和 handleScroll 机制。

## 13. 验收标准

满足以下条件即视为本次需求完成：

1. 在知识库内部文档页可切换卡片视图和列表视图。
2. 默认进入页面仍展示卡片视图。
3. 切换到列表视图后，能看到文档名称、源文件、文件类型、文件大小、分类、状态、更新时间和操作。
4. 文件导入类型文档可以直接下载原文件。
5. URL 导入和手动创建类型不会错误展示下载能力。
6. 搜索、文件类型筛选、分类筛选在两种视图下行为一致。
7. 详情查看、重建、移动、删除等操作在列表视图下仍可触发。
8. 切换视图不会导致重复请求或列表状态错乱。

## 14. 推荐实施顺序

1. 抽象 KnowledgeDocumentItem 与派生字段映射。
2. 新增视图切换状态和工具栏 UI。
3. 新增列表视图组件骨架。
4. 接入源文件列与下载按钮。
5. 对接现有详情、更多菜单和权限逻辑。
6. 联调滚动加载、搜索筛选和空态。
7. 补充 i18n 文案与交互测试。

## 15. 结论

本需求适合以“前端增量改造”为主完成。现有知识文件分页接口、知识详情接口和原文件下载接口已经具备实现基础，不需要先扩展后端。推荐在保留卡片视图的前提下，为知识库内部文档页新增列表视图组件，并在列表视图中引入“源文件 + 下载”作为核心增强项，以提高高密度信息浏览和文件操作效率。
