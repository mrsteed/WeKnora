# 思考模式结束定义与状态收口问题前后端复审文档

## 1. 复审目标

本次复审不再聚焦“为什么首轮会提前结束”这一单一现象，而是专门梳理智能推理 / 思考模式下，前后端到底各自如何定义“结束”，以及这些定义为什么会互相冲突，最终造成两类相反问题：

1. **没有正常结束，但是流程结束了。**
2. **流程已经结束，但是最终问答状态没有进入结束流程。**

这两个问题表面相反，本质上都来自同一个根因：

**当前系统中至少存在 4 套“结束”语义，但没有被收敛成唯一权威协议。**

## 2. 结论先行

### 2.1 当前系统实际上有 4 套结束定义

在智能推理模式下，当前前后端至少同时存在以下 4 个层面的“结束”：

1. **模型轮次结束**：单轮 LLM stream 是否结束，由 `finish_reason`、`tool_calls`、`chunk.Done` 决定。
2. **Agent 业务结束**：整个 Agent 任务是否结束，由 `EventAgentComplete` / `ResponseTypeComplete` 决定。
3. **传输层结束**：SSE 是否停止推送，由 StreamManager 中是否出现 `complete` 事件决定。
4. **前端界面结束**：UI 是否停止 loading / replying，由 `resetReplyState()`、`completion_status`、`answer.done`、`stop` 等多个条件混合决定。

这 4 套定义现在没有完全对齐。

### 2.2 当前最核心的协议错误

当前智能推理模式存在两个致命的协议混用：

1. **把 `answer.done` 混用成 Agent 终态信号。**
2. **把 `complete` 缺失时的 `answer.done` 继续兜底推断成 completed。**

这两个问题分别对应两类线上异常：

1. `answer.done` 过早触发前端或恢复逻辑收尾，就会出现“没正常结束，但流程结束了”。
2. 如果后端真正已经走完 defer 落库或消息已终态，但 continue-stream / 历史修复没有拿到 `complete`，又会出现“流程结束了，但状态没走完整收尾”。

### 2.3 正确的唯一协议应该是什么

针对智能推理 / 思考模式，应该强制收敛成以下唯一终态协议：

1. **`answer` 事件只表示正文流，不表示任务正式结束。**
2. **`answer.done=true` 只表示当前 answer 流分段结束，最多表示“finalizing”，不能直接代表 completed。**
3. **`complete` 事件是唯一权威业务终态。**
4. **`stop` 事件是唯一权威取消终态。**
5. **continue-stream、消息修复、历史重建、前端 UI 收尾，都必须只认 `complete/stop` 为 Agent 终态来源。**

只要这条协议没有完全落地，系统就会继续在不同路径下出现“提前结束”与“状态漏收口”两类问题。

## 4. 现状梳理：后端如何定义结束

### 4.1 EventBus 语义：Agent 的正式完成事件是 `EventAgentComplete`

`internal/event/event.go` 中定义了：

1. `EventAgentFinalAnswer`
2. `EventAgentComplete`
3. `EventStop`

其中从职责上看：

1. `EventAgentFinalAnswer` 用于正文内容流。
2. `EventAgentComplete` 用于整个 Agent 生命周期完成。
3. `EventStop` 用于用户取消或停止。

这说明后端事件模型本身已经有“正文流”和“任务完成”分层，问题不在事件类型缺失，而在后续实现没有始终坚持这层分离。

### 4.2 AgentStreamHandler：`handleComplete` 才是 Agent 终态权威出口

`internal/handler/session/agent_stream_handler.go` 的 `handleComplete` 明确承担了以下职责：

1. 将 `EventAgentComplete` 中的 `CompletionStatus`、`FinishReason`、`FailureReason` 写回 `assistantMessage`。
2. 根据 `FinalAnswer` 与已流出的 `finalAnswer` 决定 authoritativeAnswer。
3. 回填 `AgentSteps`、`AgentDurationMs`。
4. 最终向 StreamManager 追加一条 `ResponseTypeComplete` 事件。

从这个实现看，后端真实设计意图非常清楚：

**Agent 的正式结束，应该以 `ResponseTypeComplete` 为准。**

### 4.3 `handleFinalAnswer` 只是正文流，不是正式完成流

同一个文件中的 `handleFinalAnswer` 做的是：

1. 累积 `h.finalAnswer += data.Content`
2. 将事件转成 `ResponseTypeAnswer`
3. 带上 `completion_status / finish_reason / failure_reason` 等 metadata

这里最容易误导人的点是：

1. `answer` 事件上也带了 `completion_status`。
2. `answer` 事件也可能带 `Done=true`。

但从后端职责分工上，它仍然只是“正文流事件”，并没有承担“整个 Agent 生命周期收尾”的职责。

### 4.4 QA 请求生命周期：Agent 落库真正依赖 `EventAgentComplete`

`internal/handler/session/qa.go` 中，Agent 模式与普通模式是分开的。

普通模式：

1. 由 `EventAgentFinalAnswer` 的 `data.Done` 触发 `completeAssistantMessage()`。

Agent 模式：

1. 先监听 `EventAgentComplete`，把其内容写入 `agentCompletionOptions`。
2. 然后在 goroutine 的 defer 中统一调用 `completeAssistantMessage()` 落库。

这意味着：

**Agent 模式下，数据库最终状态并不是由 final_answer 正文事件决定，而是由 complete 事件决定。**

这是本次复审最重要的一条事实。

### 4.5 SSE 传输层：ContinueStream 只认 `complete` 事件结束轮询

`internal/handler/session/stream.go` 的 `ContinueStream` 逻辑表明：

1. 先从 StreamManager 重放已有事件。
2. 如果发现存在 `evt.Type == "complete"`，则直接判定 `streamCompleted=true`。
3. 如果没有 `complete`，就继续轮询。
4. 轮询过程中只有看到新的 `complete` 事件，才发送最后的 completion event 并退出。

所以从传输层看：

**SSE 是否结束，当前也是以 `complete` 事件为准，而不是 `answer.done`。**

### 4.6 后端第一条断层：`emitTerminalStreamState` 仍然回放为 `answer.done`

`stream.go` 中的 `emitTerminalStreamState()` 在消息已经 terminal 时，返回的是：

1. `ResponseTypeAnswer`
2. `Done=true`
3. `Data.completion_status / finish_reason / failure_reason`

而不是 `ResponseTypeComplete`。

这会产生协议不一致：

1. 实时流路径中，后端要求 `complete` 才代表 Agent 正式结束。
2. 但 continue-stream 的终态回放路径中，后端又退回成 `answer.done` 作为终态表达。

结果是：

1. 前端要么被迫继续兼容 `answer.done` 作为终态。
2. 要么会在恢复路径与实时路径上出现不同结束行为。

### 4.7 后端第二条断层：消息修复逻辑把 `answer.done` 兜底推断为 completed

`internal/handler/message.go` 的 `reconcilePendingAssistantMessage()` 中存在这段兜底逻辑：

1. 先扫描 events。
2. 若存在 `ResponseTypeComplete`，则按其 metadata 正式修复消息状态。
3. 但如果没有 `complete`，只要出现过 `answer.done`，就会：
   - `terminalDetected = true`
   - `CompletionStatus = completed`
   - `IsCompleted = true`
   - 默认 `FinishReason = stop`

这条逻辑对于普通问答是历史兼容兜底，但对 Agent 模式是危险的，因为它把：

**“正文流结束”错误升级成了“业务完成”。**

这正好会制造第一类异常：

1. 后端没有真正走到 complete。
2. 但历史修复把消息补成 completed。
3. 刷新页面后用户看到的是“流程结束了”。

## 5. 现状梳理：前端如何定义结束

### 5.1 普通问答路径：`answer.done` 就是终态

`frontend/src/views/chat/index.vue` 的普通非 Agent 流处理逻辑是：

1. 正文通过 `answer` chunk 累加到 `fullContent`。
2. 一旦 `data.done` 为真：
   - 直接 `syncMessageCompletionState(...)`
   - `isReplying = false`
   - `currentAssistantMessageId = ''`

这对普通问答是合理的，因为普通问答没有独立的 `complete` 业务事件。

### 5.2 Agent 路径：当前前端同时接受两类终态触发器

在 Agent 处理分支里，前端目前接受两类结束信号：

#### 第一类：`answer.done + terminal completion_status`

当前逻辑是：

1. 收到 `answer` 事件后，更新 `answerEvent.content`。
2. 若 `data.done && isTerminalCompletionStatus(answerCompletionStatus)`：
   - `syncMessageCompletionState(message, data.data || {})`
   - `upsertAgentCompleteEvent(...)`
   - `resetReplyState()`

也就是说，当前 Agent 前端并不是只认 `complete`，它在 `answer.done` 时就可能先本地收尾。

#### 第二类：`complete`

收到 `complete` 事件时，前端又会：

1. 再次 `syncMessageCompletionState(message, data.data || {})`
2. 如有必要，用 `data.data.final_answer` 覆盖 `message.content`
3. `resetReplyState()`
4. `upsertAgentCompleteEvent(message, data.data)`

这说明前端目前对 Agent 结束采用的是“双终态入口”。

### 5.3 前端第一条断层：Agent 结束并没有被强制绑定到 `complete`

从前端实现看，当前实际规则是：

1. 如果 `answer.done + terminal completion_status`，可以结束。
2. 如果 `complete` 来了，也可以结束。

这会导致：

1. `answer.done` 先到了，UI 提前结束。
2. 真正的 `complete` 稍后到来，再次收尾。
3. 两者元数据不一致时，会产生内容覆盖、状态覆盖、时序不稳定等问题。

### 5.4 前端第二条断层：stream close 时会把“有答案内容”自动补成 partial

`finalizeActiveAssistantOnStreamClose()` 中，当前只要满足：

1. 有答案内容，或者
2. Agent 流里存在 `agent_complete / stop / answer.done / answer.content`

且消息还不是 terminal，就会自动把消息补成：

1. `completion_status = partial`
2. `finish_reason = stream_closed`

这条逻辑本意是防止 UI 卡住，但在 Agent 模式下仍然不够严格，因为它把：

1. “出现过 answer 内容”
2. “收到过真正的 agent_complete”
3. “只有 answer.done”

混在同一组判断里。

它虽然比“直接 completed”更安全，但仍然说明前端对 Agent 终态来源没有做到完全单一。

### 5.5 AgentStreamDisplay 其实已经更接近正确协议

`frontend/src/views/chat/components/AgentStreamDisplay.vue` 中的 `completionStatus` 计算规则更清晰：

1. 优先读 `props.session.completion_status`
2. 否则看 `stop`
3. 否则看 `agent_complete`
4. 不再直接从 `answer.done` 推导 completed

也就是说：

**AgentStreamDisplay 本身已经趋向“complete/stop 才是终态来源”的设计。**

问题主要出在：

1. `index.vue` 的主流式处理还没完全收敛到这套协议。
2. continue-stream 和消息修复仍在兜底认 `answer.done`。

## 6. 两类异常为何会同时存在

### 6.1 异常 A：没有正常结束，但是流程结束了

这类问题的典型路径是：

1. 模型或流输出了 `answer.done`。
2. 但后端其实没有合法走到 `complete`。
3. 前端 `index.vue` 因为 `answer.done + terminal completion_status` 提前调用 `resetReplyState()`。
4. 或者消息修复逻辑因为 `hasAnswerDone` 把消息补成 completed。
5. 用户看到的是：
   - loading 消失
   - 流程看似结束
   - 但这不是一个真正完成的 Agent 终态

这就是“没有正常结束，但是流程结束了”。

### 6.2 异常 B：流程已经结束，但是最终问答状态没有进入结束流程

这类问题的典型路径是：

1. 后端真实已经走完了业务流程。
2. 但某些恢复路径、历史路径、SSE 丢事件路径没有拿到 `complete`。
3. continue-stream 用 `answer.done` 回放终态，而不是正式 `complete`。
4. 前端或历史修复侧拿不到完整 `agent_steps / agent_duration_ms / final_answer authoritative payload`。
5. 结果表现为：
   - 流程已经跑完
   - 但消息状态、Agent 步骤、耗时、最终答案权威内容没有完整进入正式结束流程

这就是“流程已经结束，但是最终问答状态没有进入结束流程”。

### 6.3 两类问题的共同根因

这两类问题不是两个独立 bug，而是同一问题的两个方向：

**Agent 的“正文流结束”和“业务完成结束”没有被强制区分。**

## 7. 正确的结束协议应该如何定义

### 7.1 应严格区分 4 个阶段

建议把思考模式的结束协议明确拆成 4 层：

#### 第一层：LLM chunk 级结束

定义：

1. `thinking.done`
2. `answer.done`
3. `tool_result` 返回完成

语义：

1. 只表示某个子流或某个 chunk 序列结束。
2. 不表示 Agent 整体任务结束。

#### 第二层：Agent 轮次结束

定义：

1. 一轮 LLM 调用返回。
2. `finish_reason` 已知。
3. 当前轮 tool calls 已收齐。

语义：

1. 只表示一次 ReAct 轮次结束。
2. 不表示整个问题结束。

#### 第三层：Agent 业务结束

定义：

1. 产生 `EventAgentComplete`
2. StreamManager 中出现 `ResponseTypeComplete`

语义：

1. 这是唯一的正式完成态。
2. 所有最终状态字段都应以这里为准：
   - `completion_status`
   - `finish_reason`
   - `failure_reason`
   - `final_answer`
   - `agent_steps`
   - `agent_duration_ms`

#### 第四层：用户取消结束

定义：

1. `ResponseTypeStop`

语义：

1. 这是唯一的取消终态。
2. 不应再等待 complete。

### 7.2 明确规定：Agent 模式下谁可以触发 UI 收尾

Agent 模式下，只允许以下两类事件触发前端 `resetReplyState()`：

1. `complete`
2. `stop`

其它信号一律不能触发正式收尾，包括：

1. `answer.done`
2. `thinking.done`
3. `tool_result`

### 7.3 明确规定：Agent 模式下谁可以触发消息正式终态

Agent 模式下，只允许以下两类来源驱动消息终态落库 / 修复：

1. `ResponseTypeComplete`
2. `ResponseTypeStop`

如果缺失这两类事件，则只能：

1. 推断为 `partial` / `failed`
2. 但不能把 `answer.done` 直接推断成 `completed`

### 7.4 明确规定：continue-stream 的终态回放必须与实时流同协议

如果消息已经 terminal，则 continue-stream 不应再只发：

1. `ResponseTypeAnswer`
2. `Done=true`

而应返回与实时流一致的终态协议，至少要包含：

1. 一条权威 `complete` 或 `stop` 表达
2. 完整的 `completion_status / finish_reason / failure_reason / final_answer`
3. Agent 所需的 `agent_steps / agent_duration_ms`

否则恢复路径永远与实时路径不一致。

## 8. 彻底解决这两类问题的实施原则

### 8.1 原则一：Agent 模式下，`complete/stop` 是唯一终态协议

需要全链路统一：

1. 实时流
2. continue-stream
3. 历史修复
4. 前端本地状态机
5. AgentStreamDisplay

都只能认 `complete/stop`。

### 8.2 原则二：`answer.done` 降级为“正文流结束”信号

它只能表达：

1. 当前 answer chunk 序列暂时结束
2. UI 可以从“正在输出”切到“正在收尾”

但绝不能直接表达：

1. completed
2. partial
3. cancelled
4. failed

这些终态必须来自 `complete/stop` 或明确的异常出口。

### 8.3 原则三：没有 `complete` 的 Agent 流只能进入受控降级，不能直接 completed

如果 Agent 最终没有 `complete`，则只允许两种兜底：

1. `stream_unavailable` -> `failed` 或 `partial`
2. `stream_closed` -> `partial`

但绝不能因为 `answer.done` 存在就自动 completed。

### 8.4 原则四：前端界面结束与消息正式结束必须由同一终态源驱动

现在的问题之一就是：

1. UI 收尾可能由 `answer.done` 提前触发。
2. 数据收尾却依赖 `complete`。

未来必须统一为：

1. `complete/stop` 同时驱动 UI 收尾与消息终态同步。

## 9. 建议修正点清单

本次文档不做代码修改，但为后续落地给出明确修正点。

### 9.1 后端

1. `internal/handler/session/stream.go`
   - `emitTerminalStreamState()` 不应再只回放 `ResponseTypeAnswer done=true`。
   - Agent 消息的 terminal replay 应与实时协议对齐。
2. `internal/handler/message.go`
   - `reconcilePendingAssistantMessage()` 中，Agent 模式不得再用 `hasAnswerDone` 兜底推断 completed。
3. `internal/handler/session/agent_stream_handler.go`
   - 继续保持 `ResponseTypeComplete` 为权威终态出口。
4. `internal/handler/session/qa.go`
   - 保持 Agent 最终落库只由 `EventAgentComplete` 驱动；不要回退到 `final_answer done` 驱动。

### 9.2 前端

1. `frontend/src/views/chat/index.vue`
   - Agent 分支里，`answer.done + terminal completion_status` 不应直接 `resetReplyState()`。
   - `resetReplyState()` 只应由 `complete` 或 `stop` 触发。
2. `frontend/src/views/chat/index.vue`
   - `finalizeActiveAssistantOnStreamClose()` 要区分：
     - 只有正文内容
     - 有 `answer.done`
     - 有 `agent_complete`
   - 不能把这三种情况混成同一组 Agent 终态信号。
3. `frontend/src/views/chat/components/AgentStreamDisplay.vue`
   - 保持现有方向：只认 `agent_complete/stop`。

## 10. 建议测试矩阵

### 10.1 实时流测试

1. `answer.done` 已到，但 `complete` 未到：
   - UI 不应正式结束。
   - 消息状态不应 completed。
2. `complete` 到达：
   - UI 正式结束。
   - completion_status 进入终态。

### 10.2 continue-stream 测试

1. 消息已 terminal：
   - 回放应提供权威终态语义。
   - 不应只返回 `answer.done`。
2. 消息无 `complete` 但有 answer 内容：
   - 应恢复为 partial/failed，而不是 completed。

### 10.3 历史加载修复测试

1. StreamManager 中只有 `answer.done`：
   - Agent 消息不能被补成 completed。
2. StreamManager 中有 `complete`：
   - AgentSteps / AgentDuration / FinalAnswer 都能回填。

### 10.4 前端状态测试

1. Agent 模式接收 `answer.done`：
   - loading 可进入 finalizing，但不能 resetReplyState。
2. Agent 模式接收 `complete`：
   - 才允许 resetReplyState。

## 11. 最终判断

本次复审后的最终判断是：

**思考模式当前并不是单纯“结束判定写错了”，而是前后端对结束定义使用了两套半重叠协议。**

后端真实想表达的是：

1. `answer` 是正文流
2. `complete` 是任务终态

但前端主流处理、continue-stream 终态回放、消息修复兜底，仍然在部分路径上把：

1. `answer.done`

当成了终态替身。

这就是为什么系统会同时出现：

1. 没有正常结束，但流程结束了。
2. 流程已经结束，但最终问答状态没进入完整结束流程。

彻底解决这两个问题，不是再加一个 if，而是必须把 Agent 模式的终态协议统一成一句话：

**在思考模式下，只有 `complete` 与 `stop` 能定义结束；`answer.done` 永远不能单独定义结束。**
