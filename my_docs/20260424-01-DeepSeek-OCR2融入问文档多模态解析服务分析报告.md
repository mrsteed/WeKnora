# DeepSeek-OCR-2 融入“问文档”多模态解析服务分析报告

## 1. 结论摘要

结论先行：

1. 当前仓库里，DeepSeek-OCR-2 可以接入 WeKnora，但更适合接入为“VLLM 视觉模型”，用于知识库多模态后处理链路，而不是直接作为“文档解析引擎”出现在解析引擎下拉框里。
2. 当前“解析引擎”和“VLLM 视觉模型”是两套独立架构：
   - 解析引擎负责把文件转成 Markdown、图片引用、元数据。
   - VLLM 视觉模型负责对图片做 OCR / Caption / 多模态理解。
3. 知识库设置页面里看不到“VLLM 解析引擎”，不是前端漏了，而是后端本来就没有把 VLLM 注册成 parser engine。
4. 如果你的目标是“让问文档在导入扫描 PDF、图片、图文文档时用 DeepSeek-OCR-2 参与识别”，答案是可以，但当前最现实的接入点是知识库 VLLM 模型链路。
5. 如果你的目标是“让 DeepSeek-OCR-2 成为解析引擎列表中的一个独立引擎”，当前代码还不支持，需要补一层新的 parser engine 适配，并解决租户级解析引擎配置和知识库级 VLM 配置之间的作用域断层。

## 2. 当前后端架构拆解

### 2.1 文档解析引擎链路

知识库“解析引擎”不是模型中心自动生成的，而是后端固定注册出来的。

关键实现：

- [internal/infrastructure/docparser/engine_registry.go](../internal/infrastructure/docparser/engine_registry.go)
- [internal/handler/system.go](../internal/handler/system.go)
- [frontend/src/views/settings/ParserEngineSettings.vue](../frontend/src/views/settings/ParserEngineSettings.vue)

当前 Go 侧本地注册的解析引擎只有：

1. `builtin`
2. `simple`
3. `weknoracloud`
4. `mineru`
5. `mineru_cloud`

其中：

- `builtin` / `simple` / `weknoracloud` / `mineru` / `mineru_cloud` 是 Go 侧显式注册的。
- `markitdown` 是 docreader 远端服务通过 `ListEngines` 返回后，被 Go 侧合并进来的。
- VLLM 并没有在这条链路里注册，因此前端解析引擎设置页不会显示它。

这和截图表现完全一致，属于架构设计结果，不是界面缺项。

### 2.2 “问文档”导入时的真实执行流程

知识库文件导入时，核心流程在：

- [internal/application/service/knowledge.go](../internal/application/service/knowledge.go)

核心步骤是：

1. 根据知识库分块规则解析 `parserEngine`。
2. 调用 `convert()` 组织 `ReadRequest`。
3. 通过 docreader / Go 内置 reader 把文件转换成 Markdown、图片引用、音频数据等。
4. 再做 chunk、向量化、索引。
5. 如果启用了多模态，再把图片送到异步任务 `image:multimodal` 做 OCR + Caption。

也就是说，“问文档”的多模态并不是完全发生在解析器里，而是分成两段：

1. 文档转换阶段：把图片从文档里提出来。
2. 多模态后处理阶段：对提取出来的图片再做 OCR / Caption。

这正是 DeepSeek-OCR-2 最容易融入的位置。

## 3. VLLM 视觉模型在项目里的实际使用场景

### 3.1 知识库多模态后处理

这是当前和“问文档”最相关、也是最直接的 VLLM 使用场景。

关键实现：

- [internal/application/service/image_multimodal.go](../internal/application/service/image_multimodal.go)
- [internal/application/service/knowledge.go](../internal/application/service/knowledge.go)
- [internal/application/service/model.go](../internal/application/service/model.go)
- [internal/models/vlm/vlm.go](../internal/models/vlm/vlm.go)

实际行为：

1. 文档解析后，如果抽取到了图片，`knowledgeService` 会入队 `image:multimodal` 任务。
2. `ImageMultimodalService` 读取图片后，会调用知识库配置的 VLM 模型两次：
   - 一次做 OCR 文本抽取。
   - 一次做图片 Caption 描述。
3. OCR 结果和 Caption 结果会各自生成子 chunk，再参与检索。

这意味着：

- 如果把 DeepSeek-OCR-2 配置成知识库使用的 VLLM 模型，它已经可以实际参与“问文档”的图片 OCR 抽取。
- 但它会被同一个服务同时拿去做 Caption。DeepSeek-OCR-2 如果偏 OCR 专用，Caption 质量可能不如通用视觉大模型。

### 3.2 扫描 PDF 页面图像的 OCR 补偿

关键实现：

- [docreader/parser/pdf_parser.py](../docreader/parser/pdf_parser.py)
- [internal/application/service/image_multimodal.go](../internal/application/service/image_multimodal.go)

当前 PDF 解析逻辑里，扫描版 PDF 在文本提取失败时，会退化成“每页转图片”。随后这些图片会进入知识库的异步多模态任务，交由 VLM 做 OCR。

因此，DeepSeek-OCR-2 对扫描版 PDF 是有现实价值的，而且这个价值比普通图片 Caption 更明确。

### 3.3 图片知识入库

对于单独上传的 JPG、PNG、WebP 等图片，系统会先保留图片引用，然后在异步任务里追加 OCR 文本和 Caption chunk。

这条链路同样可直接使用 DeepSeek-OCR-2。

### 3.4 初始化页面的多模态测试

关键实现：

- [internal/handler/initialization.go](../internal/handler/initialization.go)

系统提供了 `/initialization/multimodal/test` 接口，上传图片并测试 VLM 模型连通性与处理能力。这个接口可以直接用来验证 DeepSeek-OCR-2 是否能作为 VLLM 模型接入。

### 3.5 会话与 Agent 图像理解

仓库里 VLLM 还用于：

- 会话图片上传分析
- Agent 工具结果中的图片描述
- 通用多模态问答

相关实现可见：

- [internal/handler/session/image_upload.go](../internal/handler/session/image_upload.go)
- [internal/application/service/agent_service.go](../internal/application/service/agent_service.go)
- [internal/agent/engine.go](../internal/agent/engine.go)

所以 VLLM 在项目里不是“只给知识库用”，而是一个跨知识库、会话、Agent 的视觉模型能力层。

## 4. 为什么知识库解析引擎里没有 VLLM 引擎

根因不是前端没做，而是后端建模方式不同。

### 4.1 Parser Engine 是“文档转换器”

当前 parser engine 的职责是：

1. 接收文件或 URL。
2. 输出 Markdown、图片引用、元数据。
3. 可根据租户解析引擎配置覆盖 MinerU 参数等。

这条链路的配置入口是租户级 `ParserEngineConfig`，实现位于：

- [internal/types/tenant.go](../internal/types/tenant.go)

目前它只包含：

- MinerU endpoint / API key
- MinerU model / OCR / table / formula / language
- MinerU Cloud 对应参数

没有任何 VLM 基础地址、模型名、API Key 字段。

### 4.2 VLLM 是“视觉模型能力”

VLLM 的配置入口是知识库级 `VLMConfig`，实现位于：

- [internal/types/knowledgebase.go](../internal/types/knowledgebase.go)

这套配置用于：

- 选择知识库多模态模型
- 通过 `modelService.GetVLMModel()` 实例化视觉模型
- 驱动异步 OCR + Caption 任务

所以系统里实际存在两个不同作用域：

1. 租户级 parser engine config
2. 知识库级 VLM model config

这两个域当前没有打通。

### 4.3 代码里存在“本来想打通但没真正接上”的痕迹

`knowledgeService` 中有一个辅助方法：

- [internal/application/service/knowledge.go](../internal/application/service/knowledge.go)

该方法名为 `getVLMConfig()`，会把知识库 VLM 配置转换成 `DocParserVLMConfig`。但在当前导入主链路里，`convert()` 实际只把 `ParserEngineOverrides` 传给 reader，没有把 KB 的 VLM 配置真正传入 docreader。

这说明当前版本里，作者有“把 KB VLM 能力继续下放到 docparser”的意图，但还没有完成接线。

## 5. docreader 里有没有现成的 VLM OCR 能力

答案分成两层。

### 5.1 代码层面：有残留能力

相关实现：

- [docreader/ocr/vlm.py](../docreader/ocr/vlm.py)
- [docreader/ocr/__init__.py](../docreader/ocr/__init__.py)

从文件上看，docreader 里有：

1. `VLMOCRBackend`
2. `PaddleOCRBackend`
3. `OCREngine.get_instance("vlm")`

单看这些代码，DeepSeek-OCR-2 这种 OpenAI 兼容视觉模型，本来是能被包装成 docreader OCR 后端的。

### 5.2 但当前主链路里，这套能力基本没有真正接上

更关键的实现是：

- [docreader/parser/base_parser.py](../docreader/parser/base_parser.py)

这里明确写了：

- 不做 chunking
- 不做 OCR
- 不做 VLM caption
- 这些都交给 Go App 处理

而且当前仓库里没有看到 parser 主链路对 `OCREngine` 的实际调用。

这说明当前版本的 docreader 已经被“轻量化”为文档转 Markdown + 图片提取服务，OCR 主职责被迁移到 Go 的多模态异步任务里了。

### 5.3 现状还有配置不一致问题

还有两个明显不一致点：

1. [docreader/README.md](../docreader/README.md) 里写的是 `OCR_BACKEND=api`，但代码里 `OCREngine` 实际识别的是 `vlm`。
2. [docreader/config.py](../docreader/config.py) 当前展示的配置结构没有 `ocr_api_base_url`、`ocr_api_key`、`ocr_model` 这些字段，但 [docreader/ocr/vlm.py](../docreader/ocr/vlm.py) 又直接访问这些配置。

这说明“docreader 内置 VLM OCR 后端”在当前分支更像是遗留或半完成能力，不能视为已经稳定可用。

因此，不能把“仓库里有 VLM OCR 文件”直接等同于“知识库解析引擎已经支持 DeepSeek-OCR-2”。

## 6. DeepSeek-OCR-2 能否用到当前项目里

## 6.1 可以，用法一：作为知识库 VLLM 模型接入

这是当前最稳妥、改造最少的方案。

接入方式：

1. 在模型管理中新增一个 `VLLM` 类型模型。
2. 如果 DeepSeek-OCR-2 暴露的是 OpenAI 兼容接口，则使用 Remote API 方式接入。
3. 在知识库高级设置 / 多模态配置中，把该模型设为 VLM 模型。
4. 启用知识库多模态。

这样系统会在这些场景下调用 DeepSeek-OCR-2：

1. 图片文件入库后的 OCR。
2. 扫描 PDF 页面转图后的 OCR。
3. 文档中抽取出来的图片 OCR。
4. 同时还会尝试做图片 Caption。

优点：

1. 不需要新增 parser engine。
2. 不需要改 parser engine 设置页。
3. 已经符合现有“问文档”后处理架构。

限制：

1. 它不会出现在“解析引擎”下拉框里。
2. OCR 与 Caption 共用一个 VLM 模型，DeepSeek-OCR-2 可能不适合作 Caption。
3. 文档首轮转换仍然依赖 builtin / markitdown / MinerU 等引擎提图。

## 6.2 可以，但需要改造：作为独立 parser engine 接入

如果你的目标是“把 DeepSeek-OCR-2 做成知识库解析引擎列表里的一个选项”，当前需要做这些事：

1. 在 Go 侧新增 `EngineRegistration`，例如 `deepseek_ocr` 或更通用的 `vlm_ocr`。
2. 在 docreader 侧新增真正会调用 OCR/VLM 的 parser 或 reader，而不是只提图。
3. 给 `ParserEngineConfig` 增加模型地址、模型名、API Key、接口类型等字段。
4. 在前端解析引擎设置页新增该引擎配置项。
5. 解决“租户级 parser engine config”和“知识库级 VLMConfig”如何复用的问题。

这是可行的，但不是简单配置能完成的工作。

## 6.3 不建议直接替换为全局唯一 VLM 模型

原因是当前 `ImageMultimodalService` 对同一个 VLM 同时做：

1. OCR 文本抽取
2. 图片内容概述 Caption

DeepSeek-OCR-2 如果是偏文档 OCR 专用模型：

- 对 OCR 可能更强。
- 对自然图像理解、开放式 Caption、工具图像描述可能未必最佳。

因此更合理的策略往往是：

1. 文档 OCR 场景用 DeepSeek-OCR-2。
2. 通用图像理解场景保留通用视觉模型。

但当前仓库没有把“OCR 模型”和“Caption 模型”拆开配置，所以如果直接接入现有知识库 VLLM 配置，只能一模两用。

## 7. 面向当前仓库的可行方案建议

### 方案 A：最小改造，优先验证业务价值

适用目标：先验证 DeepSeek-OCR-2 对扫描 PDF / 图片知识入库是否明显提升。

建议做法：

1. 将 DeepSeek-OCR-2 作为 `VLLM` 模型接入模型管理。
2. 在一个测试知识库中启用多模态，并选择该 VLLM 模型。
3. 用以下样本验证：
   - 扫描 PDF
   - 图文混排 DOCX
   - 单页图片
4. 对比 OCR 子 chunk 的内容质量、召回率、问答准确率。

这是当前仓库改动最少、风险最低的路径。

### 方案 B：中等改造，拆分 OCR 与 Caption 模型

适用目标：既要保留 DeepSeek-OCR-2 的 OCR 能力，又不想牺牲 Caption / 图像理解质量。

建议做法：

1. 扩展知识库配置，新增：
   - OCR 专用 VLM 模型
   - Caption 专用 VLM 模型
2. 修改 [internal/application/service/image_multimodal.go](../internal/application/service/image_multimodal.go)，分别实例化两个模型。
3. OCR 使用 DeepSeek-OCR-2，Caption 使用通用视觉模型。

这是我认为最符合业务语义的中期方案。

### 方案 C：重构 parser engine，把 DeepSeek-OCR-2 做成解析引擎

适用目标：希望在知识库“解析引擎”页中直接选 DeepSeek-OCR-2，并让它在文档转换阶段就完成 OCR。

建议做法：

1. 新增 `vlm_ocr` 类 parser engine。
2. 明确配置作用域：
   - 方案 C1：租户级配置
   - 方案 C2：知识库级模型引用
3. 修改 docreader 主链路，让 parser 真正调用 OCR/VLM。
4. 再决定是否保留后续异步 OCR + Caption 任务，避免重复 OCR。

这是工程量最大的方案，但如果你希望“解析引擎列表里就能显式选择 DeepSeek-OCR-2”，只能走这条路。

## 8. 最终判断

基于当前代码架构，最终判断如下：

1. DeepSeek-OCR-2 能融入“问文档”的多模态服务。
2. 当前最自然的融入点不是 parser engine，而是知识库的 VLLM 模型链路。
3. 当前解析引擎列表里没有 VLLM，是因为系统把 VLLM 建模成“视觉能力层”，而不是“文档解析引擎层”。
4. 如果只想让扫描 PDF / 图片知识入库 OCR 变强，直接把 DeepSeek-OCR-2 配成知识库 VLLM 模型即可开始验证。
5. 如果想让它成为解析引擎页的正式选项，需要补一轮后端架构改造，并处理配置作用域和重复 OCR 的问题。

## 9. 建议的下一步

建议按以下顺序推进：

1. 先做方案 A，用一个测试知识库验证 DeepSeek-OCR-2 对扫描 PDF 和图片 OCR 的真实收益。
2. 如果 OCR 收益明显，再决定是否进入方案 B，把 OCR 与 Caption 拆模。
3. 只有在产品上明确要求“解析引擎页必须能直接选 OCR 模型”时，再投入方案 C 的 parser engine 改造。
