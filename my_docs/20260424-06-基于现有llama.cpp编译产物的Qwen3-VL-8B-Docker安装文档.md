# 基于现有 llama.cpp 编译产物的 Qwen3-VL-8B Docker 安装文档

## 1. 本文结论

你当前环境已经具备两个前提：

1. 本机已有源码目录：`/home/xmkp/workspace/llama.cpp`
2. 本机已完成编译，且实际存在可执行文件：`/home/xmkp/workspace/llama.cpp/build/bin/llama-server`

所以现在并不是“能不能跑”的问题，而是“采用哪种运行方式更适合后续交付”。

对你当前这个 WeKnora 场景，我的明确建议是：

1. 开发验证阶段，可以直接使用本机已编译的 `llama-server` 快速试跑。
2. 真正进入持续部署、灰度、回滚、维护阶段，优先采用 Docker 服务方式。
3. 如果只能二选一，当前应选 Docker 服务启动，而不是长期依赖宿主机手工执行本地二进制。

原因很直接：

1. 你已经明确更倾向 Docker，这与当前运维目标一致。
2. WeKnora 接的是远程 OpenAI 兼容接口，天然适合接一个稳定容器服务。
3. Docker 更方便固化启动参数、挂载模型目录、做重启策略与后续迁移。
4. 对多模态模型来说，`model`、`mmproj`、显卡参数、上下文参数都需要长期保持一致，容器化比手工命令更稳。

因此，本文给出的主线是：

1. 用本地已编译 llama.cpp 做一次最小验证。
2. 用 Docker 方式落正式服务。
3. 再把 WeKnora 的 VLLM 模型指向这个 Docker 服务。

## 2. 两种启动方式怎么选

## 2.1 方式一：直接使用本地编译好的 llama.cpp 启动

适合场景：

1. 你要先确认模型文件、`mmproj`、显卡和参数是否能跑通。
2. 你希望最快看到 `llama-server` 是否能成功加载 Qwen3-VL-8B。

优点：

1. 复用你已经编译好的环境。
2. 不需要先处理 Docker 镜像、容器网络和卷挂载。
3. 适合首次排查模型兼容性。

缺点：

1. 参数容易散落在 shell 历史里。
2. 不利于后续团队复现。
3. 不适合作为长期服务交付方式。
4. 宿主机升级、库冲突、环境漂移都会影响稳定性。

## 2.2 方式二：采用 Docker 服务启动

适合场景：

1. 你已经确定要长期供 WeKnora 调用。
2. 你希望后续部署、维护、迁移都更标准化。
3. 你希望把模型服务和宿主机环境隔离开。

优点：

1. 启动命令可固化。
2. 模型目录、缓存目录、日志目录可标准化挂载。
3. 更容易对接 compose、systemd、CI/CD、监控和回滚。
4. 和 WeKnora 当前“远程 VLM 服务”架构最匹配。

缺点：

1. 第一次准备卷挂载和网络会多一步。
2. 如果你坚持复用本机已编译二进制，仍需要额外做镜像封装。

## 2.3 最终选择建议

对你当前需求，我建议这样执行：

1. 用本机 `llama-server` 跑一次最小自测。
2. 自测通过后，不再长期使用本地命令方式。
3. 正式接 WeKnora 时，切到 Docker 服务方式。

一句话总结就是：

“本地编译版用于验通，Docker 版用于交付。”

## 5. 第一步，先用本地 llama.cpp 验证模型能跑

这一步的目标不是长期服务化，而是确认下面四件事：

1. GGUF 文件可加载。
2. `mmproj` 文件匹配。
3. CUDA / GPU 路径正常。
4. OpenAI 兼容接口能正常返回。

建议直接执行：

```bash
/home/xmkp/workspace/llama.cpp/build/bin/llama-server \
  --model /home/xmkp/workspace/deploy/models/qwen3-vl-8b-q4/Qwen3-VL-8B-Instruct-Q4_K_M.gguf \
  --mmproj /home/xmkp/workspace/deploy/models/qwen3-vl-8b-q4/mmproj-Qwen3-VL-8B-Instruct-f16.gguf \
  --alias qwen3-vl-8b-q4 \
  --host 0.0.0.0 \
  --port 18080 \
  --ctx-size 8192 \
  --n-gpu-layers 99 \
  --threads 8 \
  --parallel 2 \
  --cont-batching \
  --mmproj-offload \
  --image-min-pixels 262144 \
  --image-max-pixels 4194304
```

说明：

1. `--alias qwen3-vl-8b-q4`
   - 后续 WeKnora 里建议使用同名模型名。
2. `--ctx-size 8192`
   - 先用相对稳妥的上下文值。
3. `--n-gpu-layers 99`
   - 表示尽可能多地 offload 到 GPU。
4. `--mmproj-offload`
   - 视觉 projector 也尽量 offload。
5. `--image-min-pixels` / `--image-max-pixels`
   - 用像素范围限制图像展开规模，避免一上来把显存打满。

如果你的当前 llama.cpp 版本参数名和这里略有出入，以你本地二进制实际支持的参数为准。可以先执行：

```bash
/home/xmkp/workspace/llama.cpp/build/bin/llama-server --help
```

## 8. 路线 A，直接使用官方 Docker 镜像

先准备工作目录：

```bash
mkdir -p /home/xmkp/workspace/deploy/qwen3-vl-llamacpp
cd /home/xmkp/workspace/deploy/qwen3-vl-llamacpp
```

建议同时准备日志和缓存目录，便于后续排障与升级：

```bash
mkdir -p /home/xmkp/workspace/deploy/logs/qwen3-vl-llamacpp
mkdir -p /home/xmkp/workspace/deploy/cache/llama.cpp
```

创建 compose 文件：

```yaml
services:
  qwen3-vl-llamacpp:
    image: ghcr.io/ggml-org/llama.cpp:server-cuda
    container_name: qwen3-vl-llamacpp
    restart: unless-stopped
    gpus: all
    ports:
      - "18080:8080"
    volumes:
      - /home/xmkp/workspace/deploy/models/qwen3-vl-8b-q4:/models:ro
      - /home/xmkp/workspace/deploy/cache/llama.cpp:/root/.cache/llama.cpp
      - /home/xmkp/workspace/deploy/logs/qwen3-vl-llamacpp:/logs
    environment:
      LLAMA_ARG_MODEL: /models/Qwen3-VL-8B-Instruct-Q4_K_M.gguf
      LLAMA_ARG_MMPROJ: /models/mmproj-Qwen3-VL-8B-Instruct-f16.gguf
      LLAMA_ARG_ALIAS: qwen3-vl-8b-q4
      LLAMA_ARG_HOST: 0.0.0.0
      LLAMA_ARG_PORT: "8080"
      LLAMA_ARG_CTX_SIZE: "8192"
      LLAMA_ARG_N_GPU_LAYERS: "99"
      LLAMA_ARG_THREADS: "8"
      LLAMA_ARG_N_PARALLEL: "2"
      LLAMA_ARG_CONT_BATCHING: "true"
      LLAMA_ARG_MMPROJ_OFFLOAD: "true"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
```

保存为：

```text
/home/xmkp/workspace/deploy/qwen3-vl-llamacpp/docker-compose.yml
```

启动命令：

```bash
cd /home/xmkp/workspace/deploy/qwen3-vl-llamacpp
docker compose up -d
```

查看日志：

```bash
docker compose logs -f qwen3-vl-llamacpp
```

验证接口：

```bash
curl http://127.0.0.1:18080/health
curl http://127.0.0.1:18080/v1/models
```
