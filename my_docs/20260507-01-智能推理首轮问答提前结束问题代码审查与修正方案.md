# 智能推理首轮问答提前结束问题代码审查与修正方案

## 1. 问题概述

本次问题发生在智能推理模式的第一次问答中。用户问题为“强风情况下火灾应如何加强首批力量调派？”，模型在明显尚未完成检索与精读流程的情况下，直接以 `finish_reason="stop"` 收尾，后端将该轮结果判定为 `completed`，随后 SSE 流发送 complete 事件，前端展示为“已完成”。

从日志可以看到以下关键信号：

1. 模型输出内容仍然是“我准备继续搜索/阅读哪些文档”的过程性文字，而不是面向用户的最终答案。
2. 当轮没有任何 tool call，`tool_calls=0`。
3. 后端仍然记录：`Agent finished naturally`、`round_final_answer`、`execute_complete complete=true`。
4. 整个执行只跑了 1 轮，但前端已经收到完成态。

这说明问题不是“前端误关流”这么简单，而是后端把一次明显未满足智能推理协议的普通文本 stop 响应，错误地当成了正式完成结果。

## 2. 结论摘要

本问题不是单点缺陷，而是 4 个问题叠加后的结果：

1. **完成态判定过宽**：只要 `finish_reason=stop` 且没有 tool call，就直接视为自然完成。
2. **流式输出通道设计过宽**：思考阶段产生的普通 answer chunk，会立刻走最终答案事件通道，导致中间草稿提前展示给用户。
3. **提示词约束互相冲突**：系统提示词一边要求必须调用 `final_answer`，另一边又允许“对用户可见的 thinking/reasoning”，会诱导模型把中间推理直接写成普通文本。
4. **错误完成态会继续落库和索引**：一旦误判为 completed，会污染历史消息、聊天索引和后续续聊上下文。

因此，修复不能只改一个 if 分支，而需要同时改：

1. Agent 完成态状态机。
2. think 阶段的流式事件出口。
3. 智能推理提示词。
4. 落库/索引保护与回归测试。

## 3. 关键代码审查发现

### 3.1 P0：`stop + 无 tool_call` 被无条件判定为完成

涉及文件：

1. `internal/agent/observe.go`
2. `internal/agent/engine.go`

当前 `analyzeResponse` 中存在如下行为：

1. 当 `response.FinishReason == "stop"` 且 `len(response.ToolCalls) == 0` 时，直接进入“自然完成”分支。
2. 该分支会：
   - 把 `response.Content` 当成最终答案。
   - 发出 `EventAgentFinalAnswer`。
   - 设置 `completion_status=completed`。
   - 设置 `allow_indexing=true`、`allow_complete=true`。
3. `engine.executeLoop` 随后把该 verdict 写入 `state`，再由 `emitCompletionEvent` 发出 complete 事件。

这与智能推理模式的协议是直接冲突的。当前系统提示词多处都写明：**必须以 `final_answer` 工具结束，不能以普通 assistant 文本结束。**

也就是说：

1. 在智能推理模式下，`stop + 无 tool_call` 不能直接等价于“答案已经完成”。
2. 它最多只能表示“模型停止了”，不能表示“业务上完成了”。

这正是本次线上问题的直接根因。

### 3.2 P0：think 阶段把普通 answer chunk 直接当成最终答案流给前端

涉及文件：

1. `internal/agent/think.go`
2. `internal/handler/session/agent_stream_handler.go`
3. `frontend/src/views/chat/index.vue`

当前 `streamThinkingToEventBus` 的行为是：

1. 只要 LLM 流里出现 `ResponseTypeAnswer`，就立刻发出 `EventAgentFinalAnswer`。
2. 这条通道在前端会被当成“最终答案内容”持续累加到 `message.content`。
3. 前端并不知道这段 answer chunk 是“正式 final_answer 输出”，还是“模型在 think 阶段误吐出的普通文本草稿”。

因此，只要模型在工具调用前输出了任何可见文本，这些文本就会提前进入答案面板。

本次日志中的“搜索结果显示了很多与调派相关的文档……那么，我再调用……”就是典型的中间推理草稿，但它已经被系统按最终答案通道展示了。

这里的设计问题比单纯的完成态误判更底层：

1. 即使后续不立即 complete，用户也已经看到了错误的“答案正文”。
2. 一旦后续又走到了自然 stop 分支，就会被彻底固化为 completed。

### 3.3 P0：系统提示词存在“必须 final_answer”与“可见 reasoning”双重信号冲突

涉及文件：

1. `config/prompt_templates/agent_system_prompt.yaml`

当前提示词同时包含两类信号：

1. 强约束：
   - “You MUST always end by calling final_answer”
   - “NEVER end your turn without calling it”
   - “never reply with plain assistant text”
2. 弱化约束：
   - “In ALL outputs visible to users (including your thinking/reasoning)…”

第二类表述会给模型一个错误暗示：

1. 它会认为“thinking/reasoning 是可以直接对用户可见输出的”。
2. 在不稳定的工具调用模型上，这很容易演化成普通正文式的 chain-of-thought。

这就是为什么本次异常输出不是空白，也不是纯胡答，而是一段非常像“正在执行计划”的显式过程文本。

严格说，当前提示词没有把下面这条规则表达清楚：

**中间推理不得以普通 assistant 正文输出；如果需要显式思考，只能走 thinking 工具或 provider 的隐藏 reasoning 通道。**

### 3.4 P1：错误完成态会污染消息落库和聊天索引

涉及文件：

1. `internal/handler/session/agent_stream_handler.go`
2. `internal/handler/session/qa.go`
3. `internal/handler/message.go`

当前完成链路一旦收到 completed：

1. `assistantMessage.Content` 会被写成这段中间推理文本。
2. `IsCompleted` 会被置为 true。
3. `completeAssistantMessage` 会在 `AllowIndexing=true` 时把这条问答索引进聊天历史知识库。

这会带来 3 个副作用：

1. 历史消息列表里用户会看到一条“已完成但实际没答完”的消息。
2. 聊天索引里会写入错误答案，影响后续历史问答召回。
3. 如果后续多轮上下文引用了该消息，模型会把这段中间推理当成上一轮正式结论。

所以这个问题不能只停留在“显示不对”，它已经影响到数据正确性。

### 3.5 P1：缺少“协议违约”状态分支与重试机制

涉及文件：

1. `internal/agent/engine.go`
2. `internal/agent/observe.go`

当前代码只对两类异常做了特殊处理：

1. `finish_reason=length`
2. `empty stop response`

但对下面这种情况没有任何专门逻辑：

1. 非空内容。
2. 无 tool call。
3. 明显仍是中间推理或执行计划。
4. 违反了“必须 final_answer 收尾”的协议。

这种情况本应进入“协议违约重试”或“受控失败”分支，而不是 completed。

### 3.6 P2：Agent 运行态没有承接 `max_completion_tokens`

涉及文件：

1. `internal/types/custom_agent.go`
2. `internal/types/agent.go`
3. `internal/application/service/session_agent_qa.go`
4. `internal/agent/think.go`

当前 `CustomAgentConfig` 有 `max_completion_tokens` 字段，但 `AgentConfig` 运行态没有对应字段，think 阶段也没有把该值传入 `ChatOptions`。

这不是本次“首轮提前结束”的直接根因，因为日志里的结束原因是 `stop` 而不是 `length`。但这是一个真实缺口：

1. 智能体运行时的长度控制配置实际上没有生效。
2. 后续在强制 tool 协议后，如果模型仍然容易超长，这个参数缺失会降低可调节性。

建议本次顺手补齐，避免再次出现“配置看起来有，运行时其实没用”的问题。

## 4. 根因链路复盘

本次异常的完整链路可以归纳为：

1. 用户发起第一次智能推理问答。
2. 系统提示词要求必须 `final_answer`，但同时又允许“可见 reasoning”。
3. 模型输出了一段过程性文本，没有调用任何工具，也没有调用 `final_answer`。
4. `streamThinkingToEventBus` 把这段普通 answer chunk 当成最终答案流发给前端。
5. `analyzeResponse` 看到 `finish_reason=stop && tool_calls=0`，直接判定为 natural stop completed。
6. `emitCompletionEvent` 发 complete。
7. 前端收到 complete 后收尾，界面展示“已完成”。
8. 消息落库时被标记为 completed，且允许索引。

所以本问题的准确描述不是“执行中断了”，而是：

**系统把一次不符合智能推理协议的中间态输出，错误地升级成了最终完成态。**

## 5. 修正方案

### 5.1 第一优先级：收紧完成态状态机

目标：让“模型停了”与“任务完成了”彻底解耦。

建议修改点：

1. 在 `internal/agent/observe.go` 为智能推理模式新增“协议违约”判定。
2. 当满足以下条件时，不得直接 completed：
   - `finish_reason == "stop"`
   - `len(tool_calls) == 0`
   - 当前工具集中包含 `final_answer`
3. 此时进入新的分支，例如：
   - `completion_status = pending_retry`（内部态，不必对外持久化）
   - 或直接在 engine 中走“协议违约重试”流程。

推荐执行策略：

1. 第 1 次出现违约时，不对用户暴露正文，不落 completed。
2. 将该轮文本只作为 `AgentStep.Thought/ReasoningContent` 保存。
3. 自动追加一条 system/user 级纠偏消息，例如：
   - “Do not continue with plain assistant text. Call a tool next. If you are done, call final_answer.”
4. 重新发起下一轮。
5. 若连续 2 次仍然违约：
   - 如果已有充分 tool 结果，则走 `streamFinalAnswerToEventBus` 做一次受控总结。
   - 如果没有 tool 结果，则标记 `failed` 或 `partial`，`finish_reason=protocol_violation`，并返回简短 fallback。

关键要求：

1. `protocol_violation` 不允许 `AllowIndexing=true`。
2. `protocol_violation` 不允许 `AllowComplete=true`。
3. 不得把中间推理正文写入 completed 消息内容。

### 5.2 第一优先级：禁止 think 阶段的普通 answer chunk 直接进入最终答案通道

目标：消除“中间草稿先展示给用户”的问题。

建议修改点：

1. 修改 `internal/agent/think.go`。
2. 当前逻辑中，所有 `ResponseTypeAnswer` 都会发成 `EventAgentFinalAnswer`，需要改为分流。

建议的新规则：

1. 只有两类内容可以直接走用户可见的 final answer 事件通道：
   - `source == "final_answer_tool"` 的 chunk。
   - 已经被后端状态机确认是合法 terminal answer 的内容。
2. 其它普通 answer chunk：
   - 仅缓存在 `llmResult.Content` 中供后续判定。
   - 不立即推给前端正文区域。

这样做的好处：

1. 即使模型误吐了一段普通文本，也不会立刻污染用户界面。
2. 最终是否可见，由 `analyzeResponse` 或补救分支统一裁决。

这是本次修复中最关键的流控改造之一。

### 5.3 第一优先级：重写智能推理提示词中的“可见 reasoning”规则

目标：避免系统提示词继续鼓励模型输出明文思维链。

建议修改点：

1. 修改 `config/prompt_templates/agent_system_prompt.yaml`。
2. 删除或改写以下表述：
   - `including your thinking/reasoning`
   - `including thinking/reasoning blocks`

建议替换为：

1. “Intermediary reasoning must not be emitted as plain assistant text.”
2. “If the thinking tool is available, use it for explicit reasoning; otherwise keep reasoning hidden.”
3. “Only final user-facing answer may be emitted through final_answer.”

还应新增一条硬规则：

1. `Never output planning steps, internal search strategy, or protocol narration as plain assistant text.`

这样可以从源头降低模型产生“我接下来准备搜索什么”的概率。

### 5.4 第二优先级：启用 `tool_choice=required` 作为协议加固

目标：减少模型直接 plain stop 的概率。

现状：

1. `ChatOptions` 和底层 OpenAI 兼容请求已经支持 `tool_choice="required"`。
2. 但智能推理 think 阶段当前没有使用。

建议：

1. 在智能推理模式、且工具列表非空时，为 think 调用设置 `ToolChoice="required"`。
2. 对明确不支持该参数的 provider 保留现有兼容逻辑，例如 DeepSeek 继续跳过。
3. 对跳过 `tool_choice` 的 provider，仍然依赖 5.1 的协议违约兜底。

注意：

1. `required` 只能保证“本轮必须有工具调用”，不能保证最后一定是 `final_answer`。
2. 因此它只能是“减灾措施”，不能替代状态机修复。

### 5.5 第二优先级：落库与索引保护

目标：防止错误中间态进入正式数据面。

建议修改点：

1. `internal/handler/session/agent_stream_handler.go`
2. `internal/handler/session/qa.go`
3. `internal/handler/message.go`

建议规则：

1. 只有 `completion_status=completed` 且 `finish_reason` 为合法终态时，才允许 `AllowIndexing=true`。
2. 对 `protocol_violation`、`cancelled`、`partial`、`failed`：
   - 不索引。
   - 不把 `is_completed` 置为 true。
3. 如果消息内容仅来自未验证的普通 answer chunk，不应覆盖数据库中的正式 `Content`。

可选增强：

1. 在前端增加对 `finish_reason=protocol_violation` 的友好提示，例如“模型未按智能推理协议返回最终答案，系统已终止本轮并保留中间步骤”。

### 5.6 第二优先级：补齐 `max_completion_tokens` 运行态承接

建议修改点：

1. 在 `internal/types/agent.go` 的 `AgentConfig` 增加 `MaxCompletionTokens`。
2. 在 `internal/application/service/session_agent_qa.go` 的 `buildAgentConfig` 中承接 `customAgent.Config.MaxCompletionTokens`。
3. 在 `internal/agent/think.go` 和 `internal/agent/finalize.go` 的 `ChatOptions` 中传入该值。

这项修正不是为了解决本次 stop 误完成，而是为了让智能体运行态参数真正完整可控。

## 7. 建议测试方案

### 7.1 后端单元测试

至少补充以下测试：

1. `stop + no tool_calls + non-empty content + final_answer available` 不应 completed。
2. 上述场景应触发协议违约重试。
3. 连续协议违约达到上限后，应返回 failed/partial，且 `AllowIndexing=false`。
4. `final_answer_tool` 的 answer chunk 可以正常流到前端。
5. 普通 think 阶段 answer chunk 不应直接作为最终答案事件发出。
6. `protocol_violation` 消息不得进入 completed 索引路径。

### 7.2 集成测试

构造一个 mock chat model，按以下脚本返回：

1. 第 1 轮：返回一段“我先去搜索文档”的正文，`finish_reason=stop`，无 tool call。
2. 预期系统不 complete，而是继续下一轮或进入受控失败。
3. 第 2 轮：返回合法 `final_answer` tool call。
4. 预期最终 only final_answer 内容进入 `message.content`。

### 7.3 前端联调验证

验证以下场景：

1. 中间步骤仍然展示在 Agent 步骤区。
2. 未经验证的普通 answer chunk 不再提前污染正文区域。
3. complete 只在合法终态出现。
4. `protocol_violation` 不显示为“已完成”。

## 8. 建议新增观测指标

建议增加以下日志或指标：

1. `agent_protocol_violation_total`
2. `agent_stop_without_tool_total`
3. `agent_retry_missing_final_answer_total`
4. `agent_plain_answer_chunk_suppressed_total`
5. `agent_protocol_violation_by_model`

这样可以快速区分：

1. 是某一类模型更容易 plain stop。
2. 还是提示词/运行时策略存在系统性问题。

## 9. 推荐实施顺序

### 阶段一：先止血

1. 改 `observe.go`，禁止 `stop + 无 tool_call` 直接 completed。
2. 改 `think.go`，禁止 think 阶段普通 answer chunk 直出到最终答案通道。
3. 增加 `protocol_violation` 失败保护，禁止索引。

### 阶段二：再加固

1. 改系统提示词，删除“可见 reasoning”误导表述。
2. 在支持的 provider 上启用 `tool_choice=required`。
3. 补齐 `max_completion_tokens` 运行态传递。

### 阶段三：补测试与观测

1. 新增单元测试和集成测试。
2. 增加线上协议违约指标。
3. 联调前端状态展示。

## 10. 最终判断

这次问题的本质不是模型单次偶发失误，而是后端协议执行层把“普通文本 stop”误当成了“智能推理已完成”。

如果不修复状态机和流式输出边界，即使换模型、调温度、改提示词，也仍然会重复发生，只是触发概率不同。

因此本次修复必须以 **后端协议收口** 为主，以 **提示词去歧义** 和 **数据面保护** 为辅。按照本文方案改完后，智能推理首轮即使遇到模型异常输出，也只会表现为“自动纠偏重试”或“受控失败”，不会再被错误展示成一次已经完成的正式答案。
