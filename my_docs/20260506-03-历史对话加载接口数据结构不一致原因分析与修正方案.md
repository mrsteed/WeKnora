# 历史对话加载接口数据结构不一致原因分析与修正方案

## 1. 问题背景

本次问题中的 A 接口与 B 接口，用户已明确说明：

1. 二者都属于“智能推理 agent”模式。
2. 本次需要解释的是，同样是智能推理历史消息，为什么返回结构会不一致。
3. 本次文档整改目标已经调整，不再要求补救老数据，也不再要求围绕历史数据兼容、回填、降级展示做方案。
4. 本次只聚焦一个目标：后续新产生的智能推理消息，必须稳定保留 `agent_steps`，不能再出现“答案有了，但轨迹丢了”的情况。

因此，这份文档不再以“如何修复旧历史数据”为目标，而是以“如何从代码链路上堵住新数据丢轨迹的问题”为目标，重新梳理根因和整改过程。

## 2. 结论摘要

本次问题的核心结论如下：

1. A 和 B 都是智能推理 agent 产生的消息，不存在“一个是普通问答，一个是智能体问答”的模式差异。
2. 当前前端的智能推理富展示强依赖 `agent_steps`，没有该字段时就会退化成普通文本显示。
3. `agent_steps` 的权威来源不是前端重建，也不是历史接口临时补算，而是 Agent 完成事件中的 `state.RoundSteps`。
4. 当前新数据是否能把 `agent_steps` 正确写入消息表，依赖“完成事件写回内存消息”与“最终持久化 assistantMessage”这两步都成功。
5. 现有链路中，`agent_steps` 没有被当作“必须成功持久化的完成结果”来约束和校验，因此存在“完成状态已写库，但轨迹未落库”的窗口。
6. 当前流式 complete 事件也没有把完整 `agent_steps` 放进事件数据，因此一旦消息表丢了轨迹，后面就缺少可追溯的数据来源。

所以，本次整改的重点不是历史兼容，而是把新数据链路改成“只要智能推理完成，`agent_steps` 就必须随消息一起落库，并且整个过程可观测、可校验、可测试”。

## 3. 关键代码证据

### 3.1 智能推理步骤的源头是 Agent 完成事件中的 `AgentSteps`

在智能体引擎执行完成时，系统会把当前轮次收集到的步骤轨迹放入完成事件。

关键链路如下：

1. `internal/agent/engine.go` 持续把每轮推理、工具调用、工具结果收集到 `state.RoundSteps`。
2. 完成时，`internal/agent/finalize.go` 发出 `EventAgentComplete`。
3. `internal/event/event_data.go` 中的 `AgentCompleteData` 定义了 `AgentSteps interface{} 'json:"agent_steps,omitempty"'`。

这说明：

1. 智能推理轨迹在执行阶段就已经产生。
2. `agent_steps` 本质上是执行结果的一部分，不是历史接口附加生成的可选字段。

### 3.2 `agent_steps` 是在流式完成处理阶段写回 assistantMessage 的

在 `internal/handler/session/agent_stream_handler.go` 的 `handleComplete(...)` 中，系统会读取完成事件里的 `data.AgentSteps`，并写回当前内存中的 assistant message。

关键逻辑可以概括为：

1. 收到 complete 事件。
2. 读取 `data.AgentSteps`。
3. 若能够断言成 `[]types.AgentStep`，则执行 `h.assistantMessage.AgentSteps = steps`。

这说明：

1. `agent_steps` 是否进入最终消息对象，不是在创建 assistant message 时决定的。
2. 它依赖 complete 事件到达后再回填。
3. 这天然意味着，完成事件处理和最终持久化之间如果缺乏强约束，就会出现丢轨迹窗口。

### 3.3 智能推理消息最终是否写库，依赖 `completeAssistantMessage(...)`

在 `internal/handler/session/qa.go` 中，智能推理执行完后，会调用 `completeAssistantMessage(...)` 持久化 assistant message。

这里最关键的事实是：

1. 持久化的是当时内存中的 `assistantMessage` 整体对象。
2. 如果在进入 `completeAssistantMessage(...)` 前，`assistantMessage.AgentSteps` 还没有被正确回填，那么数据库里就只会有终态字段和答案文本，不会有轨迹。

也就是说，当前真正的风险点不在“数据库字段有没有”，而在“进入最终 UpdateMessage 时，内存对象是否已经完整”。

### 3.4 `completeAssistantMessage(...)` 当前更关注完成态，不够强调轨迹完整性

从现有职责看，`completeAssistantMessage(...)` 的核心目的是：

1. 写入 `completion_status`。
2. 写入 `finish_reason`。
3. 写入 `failure_reason`。
4. 更新完成态消息。

但对于智能推理模式，它目前缺少一个明确约束：

1. 在允许消息进入 completed 之前，是否已经确认 `agent_steps` 同步完成。
2. 如果 `agent_steps` 为空，是否应该直接告警或阻止静默完成。

因此，当前代码的“完成”更像是“状态完成”，而不是“状态与轨迹一起完成”。

### 3.5 当前流式 complete 事件没有保存完整 `agent_steps`

在 `internal/handler/session/agent_stream_handler.go` 中，追加到流里的 `ResponseTypeComplete` 事件数据主要包括：

1. `final_answer`
2. `completion_status`
3. `finish_reason`
4. `failure_reason`
5. `total_steps`
6. `total_duration_ms`

但没有把完整 `agent_steps` 一起放入事件数据。

这说明：

1. 目前流只保存了“完成摘要”，没有保存“完整轨迹快照”。
2. 即使本次整改不处理老数据，新的链路里也仍然缺少一份对 `agent_steps` 的二次冗余存储。
3. 如果未来再次发生消息落库异常，排查和修复空间会很有限。

因此，`complete` 事件缺少 `agent_steps` 也是新数据链路中需要补齐的一环。

### 3.6 前端展示依赖 `agent_steps`，因此后端必须把它当成主数据而不是附属数据

在 `frontend/src/views/chat/index.vue` 中，历史消息的智能推理渲染只有在 `agent_steps` 非空时才会进入 `reconstructEventStreamFromSteps(...)`。

这说明：

1. `agent_steps` 不是一个可有可无的增强字段。
2. 它实际上决定了这条消息在前端是“智能推理消息”还是“普通文本消息”。
3. 后端如果继续把它视为附属信息，就会继续造成展示结构漂移。

因此，从产品效果上看，`agent_steps` 应被视为智能推理消息的主字段之一。

## 4. A 与 B 接口差异的真实原因

### 4.1 A 接口属于“完成了答案，但没有保住轨迹”的智能推理消息

A 接口中的 assistant 消息有以下特征：

1. `content` 存在。
2. `completion_status = completed`。
3. `finish_reason = stop`。
4. 但没有 `agent_steps`。

这说明：

1. 这条消息不是普通问答。
2. 它确实已经走完了智能推理执行流程。
3. 但在最终落库时，轨迹没有和终态一起被稳定保存下来。

### 4.2 B 接口属于“答案和轨迹都完整保留下来”的智能推理消息

B 接口中的 assistant 消息有以下特征：

1. `agent_steps` 非空。
2. 步骤中包含 `iteration`、`thought`、`tool_calls`、`result` 等结构。
3. 前端可以基于这些步骤完整重建智能推理展示。

这说明：

1. B 的差异不是协议不同。
2. 而是它在完成链路里，步骤轨迹和消息终态一起成功落库了。

### 4.3 两者差异的本质是“新消息落库完整性不稳定”

因此，A 和 B 的差异本质上不是接口层 schema 设计不一致，而是智能推理消息在完成落库阶段的完整性不稳定：

1. 有的消息只落了答案和状态。
2. 有的消息答案和轨迹都落了。

这正是本次整改要解决的核心问题。

## 5. 根因判定

### 5.1 第一根因：`agent_steps` 回填发生在完成事件阶段，天然存在时序窗口

当前链路中，assistant message 是先创建、后补轨迹、再最终持久化。

只要以下任一条件不满足，就可能出现“完成了但没轨迹”：

1. complete 事件未按预期进入处理器。
2. `data.AgentSteps` 未成功转换成 `[]types.AgentStep`。
3. `assistantMessage.AgentSteps` 尚未更新时，就进入了最终持久化。
4. 最终 UpdateMessage 成功时，传入对象仍然是未补轨迹版本。

这说明当前链路对时序依赖过强，但缺少显式的完整性校验。

### 5.2 第二根因：完成态持久化没有把“轨迹完整”作为前置条件

从职责设计上看，当前代码更强调：

1. 这条消息是否可以标记为 completed。
2. 最终答案是否已写入。

但对智能推理模式而言，真正应该被视为“完成”的应当是：

1. 终态字段已写入。
2. 最终答案已写入。
3. `agent_steps` 已写入。

当前缺少第 3 条约束，因此产生了“伪完成”消息，即状态上是 completed，但数据上不是完整的智能推理消息。

### 5.3 第三根因：缺少对 `agent_steps` 的冗余记录与失败可观测性

当前系统对 `agent_steps` 存在两个明显薄弱点：

1. complete 事件里没有保留完整 `agent_steps`。
2. 完成落库后没有对“轨迹是否真的写入”做强日志和强校验。

这会带来两个后果：

1. 一旦丢失，排查时缺少第二证据源。
2. 即使丢失了，也可能在运行期没有任何明显告警。

所以本次整改必须同时补齐“数据冗余”和“过程可观测”。

## 6. 本次整改范围与非范围

### 6.1 本次整改范围

本次只处理新数据链路，目标如下：

1. 后续新产生的智能推理 assistant 消息，必须稳定带上 `agent_steps` 落库。
2. 智能推理 complete 事件中要同步保留完整 `agent_steps`，作为链路冗余。
3. 完成落库阶段要增加完整性校验和日志，避免静默丢轨迹。
4. 补齐单元测试，确保后续代码改动不会再把该问题引回。

### 6.2 本次明确不处理的内容

以下内容本次不纳入整改范围：

1. 不处理老历史数据的兼容补偿。
2. 不做离线回填脚本。
3. 不为历史缺轨迹消息增加前端降级展示方案。
4. 不修改 `LoadMessages` 去尝试补旧消息的 `agent_steps`。

这样做的原因是：

1. 本次目标已经明确收缩为“止住新增问题”。
2. 旧数据缺少可靠原始轨迹来源，兼容方案会显著扩大改动面。
3. 当前最优先的是把写链路改对，避免继续产生新的脏数据。

## 7. 整改目标

本次整改目标只保留三条：

1. 智能推理消息的“完成”必须升级为“终态、答案、轨迹三者一起完成”。
2. 新数据链路里，`agent_steps` 必须同时进入消息表和 complete 事件数据。
3. 整个写链路要具备明确的日志、校验和测试，能发现、定位、阻断再次丢轨迹。

对应到工程目标，就是：

1. 不再新增 A 这种“completed 但没有 `agent_steps`”的智能推理消息。
2. 以后出现问题时，能够从日志和 complete 事件里快速定位是“没生成”“没回填”还是“没落库”。

## 8. 详细整改过程

## 8.1 第一步：梳理并固定“智能推理完成”的标准

### 8.1.1 明确完成态的业务定义

整改前，需要先把“什么叫智能推理消息完成”写清楚，否则后续代码仍会各自理解。

建议统一定义为：

1. 收到 Agent complete 事件。
2. `assistantMessage.Content` 已经具备最终回答。
3. `assistantMessage.AgentSteps` 已经写回。
4. `completion_status`、`finish_reason`、`failure_reason` 已经确定。
5. 上述内容被一次性持久化成功。

只有同时满足这 5 条，才能把消息视为“真正 completed”。

### 8.1.2 落到代码层的约束

这一定义需要落实为代码检查点：

1. `handleComplete(...)` 负责把 complete 事件里的 `AgentSteps` 写回内存消息。
2. `completeAssistantMessage(...)` 在智能推理模式下负责校验 `AgentSteps` 是否已就绪。
3. 若 `AllowComplete = true` 且消息是 agent 模式，但 `AgentSteps` 为空，需要输出告警日志。
4. 如果团队允许更强约束，可直接拒绝将该消息写成 completed，而是改为 failed 或 partial；如果暂不改状态，至少必须高亮告警。

本次文档推荐的最小落地方案是：

1. 不强改状态机。
2. 先增加强日志与完整性校验。
3. 再逐步评估是否要把“缺轨迹禁止 completed”做成硬约束。

## 8.2 第二步：补齐 complete 事件的数据冗余

### 8.2.1 为什么必须补 complete 事件

虽然本次不处理老数据，但 complete 事件里仍应补齐 `agent_steps`，原因有三点：

1. 它是当前链路最自然的第二份事实来源。
2. 能帮助后续排查“回填成功但落库失败”还是“complete 事件本身就没带步骤”。
3. 未来如果还要做修复或审计，也有可用依据。

### 8.2.2 具体改法

在 `internal/handler/session/agent_stream_handler.go` 追加 `ResponseTypeComplete` 事件时，除现有字段外，再补充：

1. `agent_steps`
2. `agent_duration_ms`（若当前上下文已有）
3. 其他已确定属于智能推理最终快照的字段

建议整理为统一结构，而不是在多处零散拼装，避免未来再次漏字段。

### 8.2.3 改完后的效果

补齐后，complete 事件至少可以承担两类职责：

1. 前端流式消费的终态摘要。
2. 后端排查与调试用的完整完成快照。

这一步本身不是为了修老数据，而是为了提升新链路的健壮性和可观测性。

## 8.3 第三步：收紧 `handleComplete(...)` 到 `completeAssistantMessage(...)` 之间的交接

### 8.3.1 当前风险点

当前代码的风险不是单点故障，而是交接不够显式：

1. `handleComplete(...)` 回填轨迹。
2. 之后由上层流程进入 `completeAssistantMessage(...)`。
3. 但中间没有显式证明“这次持久化对象就是已经带轨迹的对象”。

### 8.3.2 整改要求

建议在这段交接处增加明确校验和日志：

1. `handleComplete(...)` 成功写入 `AgentSteps` 后，打印包含 message ID、request ID、step 数量的 info 日志。
2. 进入 `completeAssistantMessage(...)` 前，再打印一次 assistant message 当前 `AgentSteps` 长度。
3. 如果两处日志出现不一致，就能快速判断轨迹是在“事件处理阶段”还是“完成持久化阶段”丢失。

### 8.3.3 推荐日志内容

建议日志至少包括：

1. `session_id`
2. `message_id`
3. `request_id`
4. `agent_steps_count`
5. `completion_status`
6. `finish_reason`
7. `update_success / update_error`

这样排查时不需要重新跟堆栈，可以直接通过日志串联完整链路。

## 8.4 第四步：把 `completeAssistantMessage(...)` 改成“先校验，再落库”

### 8.4.1 当前职责需要增强

`completeAssistantMessage(...)` 当前更偏向“状态更新器”，整改后应提升为“智能推理完成落库的最后守门人”。

### 8.4.2 详细整改动作

建议在该函数内新增如下过程：

1. 复制当前 `assistantMessage` 为待持久化对象，避免边更新边污染调用侧状态。
2. 写入 `UpdatedAt`、`CompletionStatus`、`FinishReason`、`FailureReason`、`IsCompleted`。
3. 如果当前消息属于智能推理模式，则增加 `agent_steps` 完整性检查。
4. 记录本次持久化前的 `agent_steps_count`。
5. 调用 `UpdateMessage(...)`。
6. 若更新失败，输出 error 日志并返回 false。
7. 若更新成功，再输出一次 info 日志，明确本次 completed 消息的 `agent_steps_count`。

### 8.4.3 为什么这一层必须做

因为前面的 complete 事件处理即使做对了，如果最后落库阶段不再检查一次，问题仍可能在这里静默溜过去。

所以，`completeAssistantMessage(...)` 必须成为最终一致性检查点。

## 8.5 第五步：统一“智能推理消息”的判定口径

### 8.5.1 为什么需要判定口径

完整性校验只应对智能推理消息生效，不能误伤普通 assistant 消息。

因此需要在代码中统一判断：

1. 这条 assistant 消息是不是 agent 模式消息。
2. 哪些链路需要执行 `agent_steps` 完整性校验。

### 8.5.2 推荐做法

建议不要在多个函数里各自凭经验判断，而是统一通过明确条件识别，例如：

1. 当前请求模式是 agent。
2. 当前处理器是 agent stream handler。
3. 或消息创建时就带有明确的 agent 标识上下文。

如果工程里还没有统一的 agent 消息判定函数，建议在这次整改里顺手补一个，避免未来又出现某处忘记校验的问题。

## 8.6 第六步：增加链路级测试，防止回归

### 8.6.1 需要覆盖的最小测试面

本次虽然只改文档，但从整改过程上，必须明确要补哪些测试。建议最少包含以下四类：

1. `handleComplete(...)` 收到带 `agent_steps` 的 complete 事件后，能够正确写回 `assistantMessage.AgentSteps`。
2. `completeAssistantMessage(...)` 在持久化成功时，能够保留已有 `agent_steps`，不会被后续状态更新覆盖掉。
3. complete 事件写入流时，事件数据中包含完整 `agent_steps`。
4. 当智能推理消息准备落库但 `agent_steps` 为空时，至少能打出预期日志或返回明确结果。

### 8.6.2 推荐测试层次

建议测试分两层：

1. 单元测试：针对 `agent_stream_handler.go`、`qa.go` 的局部行为。
2. 轻量集成测试：模拟一条 agent 消息从 complete 事件进入，到最终 UpdateMessage 的完整链路。

如果只做单元测试，容易漏掉“对象交接阶段”的问题；因此推荐至少有一条跨函数链路测试。

## 8.7 第七步：增加上线前人工校验步骤

### 8.7.1 为什么不能只靠测试

这个问题高度依赖真实的流式完成链路，单测只能覆盖主要逻辑，仍建议在联调环境做一次人工验证。

### 8.7.2 详细联调步骤

建议联调时按以下步骤执行：

1. 新建一条智能推理会话。
2. 触发一次明确会经过 `thinking + tool_call + final_answer` 的 agent 问答。
3. 等待消息完成。
4. 直接查看消息表中该条 assistant 记录，确认：
   - `completion_status = completed`
   - `finish_reason` 正常
