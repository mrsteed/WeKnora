# 默认调派规则问答被收尾阶段误判为长文档分析

## 1. 问题结论

本次第二个问题“默认调派规则是什么”并不是在请求入口阶段直接走进了 full_document 或 knowledge_grounded_full_document 的长文档执行链路。

但是，它在回答完成后的收尾持久化阶段，被后端重新识别并注册成了 chat document artifact，同时被标记为 long_document，document_generation_status 也被写成了 continuing。前端随后依据这个 artifact 和正文标题结构，展示了“文档规划”卡片。

因此，这个问题的准确结论是：

1. 执行阶段：没有进入长文档执行链路，实际执行的是普通 AgentQA 检索问答。
2. 收尾阶段：进入了长文档持久化/展示链路，并最终在页面上表现为长文档流程。

从用户体验和系统结果看，这次请求已经属于“误入长文档流程”，只是误入点不在入口路由，而在回答完成后的 artifact 注册与索引阶段。

## 3. 根因分析

## 3.1 根因一：artifact 注册条件过宽，只看“长且结构化”，不看“是否明确是文档模式”

当前 artifact 注册的核心条件在 shouldRegisterChatDocumentArtifact：

- [internal/application/service/chat_document_artifact.go](../internal/application/service/chat_document_artifact.go#L1020)

这段逻辑的关键问题是：

1. 只要正文长度超过 2000 字。
2. 且 Markdown 标题数达到 2 个以上。

就会注册 chat document artifact。

它没有强依赖以下任何一个“权威文档信号”：

1. OutputMode 必须是 full_document。
2. Intent 必须是 continue_document / revise_document / regenerate_document。
3. 必须存在 BaseArtifact。
4. 必须存在 GenerationRunID。
5. 必须来自真正的长文档执行路径。

因此，只要普通问答回答得足够长、并且用了 `#` / `##` 结构化排版，就会被误当成文档产物。

第二问恰好完全满足这个误判条件。

## 3.2 根因二：document_generation_status 在未显式设置时被归一化成了 continuing

当前类型层的 NormalizeChatDocumentGenerationStatus 在传入空字符串时，会默认返回 continuing：

- [internal/types/chat_document_artifact.go](../internal/types/chat_document_artifact.go#L151)

在 RegisterFromAssistantMessage 中，代码先做了：

1. `documentGenerationStatus := NormalizeChatDocumentGenerationStatus(options.DocumentGenerationStatus)`
2. 即使 options.DocumentGenerationStatus 原本为空，也会变成 continuing。

对应位置：

- [internal/application/service/chat_document_artifact.go](../internal/application/service/chat_document_artifact.go#L627)

这会带来两个后果：

1. 普通回答一旦被注册成 artifact，就天然携带 continuing。
2. 后面的索引与 UI 判断会误以为它是一个仍在继续的长文档过程。

所以这不是简单的展示问题，而是状态默认值本身就把“未设置”错误地折叠成了“继续生成中”。

## 3.3 根因三：索引侧把“artifact 已存在”或“status 非空”继续放大成 long_document

buildMessageIndexOptions 中有如下逻辑：

- [internal/handler/session/qa.go](../internal/handler/session/qa.go#L215)

只要满足以下任一条件，就会把消息索引成 long_document：

1. artifactOptions.OutputMode == full_document
2. documentGenerationStatus != ""
3. len(documentSections) > 0

由于前一个阶段已经把空状态归一成了 continuing，而且又注册了 artifact，于是这里自然就把第二问写成了：

1. TaskKind = long_document
2. DocumentGenerationStatus = continuing

这一步把误判从“artifact 层”进一步扩散到了“索引层”和“后续会话复用层”。

## 4. 为什么这是一个真正的问题

这个问题的影响不只是“界面样式不对”，而是会污染后续系统行为：

1. UI 会展示“文档规划”“章节数”“继续生成”等长文档语义。
2. 聊天历史知识库会把这条普通问答按 long_document 入库。
3. 后续续写、修订、基线选择、artifact 版本链等逻辑可能基于这个误判继续扩散。
4. 用户会误以为系统把一个普通问答升级成了文档生成任务。

所以该问题本质上是“普通问答在 completion 阶段被错误晋升为长文档任务”。

## 5. 修正建议

## 5.1 后端 P0：artifact 注册必须增加“权威文档信号”门槛

最优先修正点是：不要再让 shouldRegisterChatDocumentArtifact 仅凭正文长度和标题结构注册 artifact。

建议改为必须满足以下条件之一，才允许注册 chat document artifact：

1. `OutputMode == full_document`
2. `Intent` 属于 `continue_document / revise_document / regenerate_document`
3. `BaseArtifact != nil`
4. `GenerationRunID != ""`
5. completion extra 中带有明确的长文档运行元数据，例如 outline、document_sections、authoritative document_generation_status

反过来，如果当前请求是普通问答：

1. Intent=normal
2. OutputMode 为空
3. 没有 BaseArtifact
4. 没有 GenerationRunID

那么无论回答多长、标题多规范，都不应该注册 chat document artifact。

这应该作为主修复。

## 5.2 后端 P0：未显式设置的 document_generation_status 不能默认变成 continuing

建议把 document_generation_status 的“空值”与“continuing”分开处理。

可选修法有两种：

1. 修改 NormalizeChatDocumentGenerationStatus，对空字符串返回空，而不是 returning continuing。
2. 保留 Normalize 现状，但只在确认当前请求属于文档模式后才调用 Normalize。

推荐第二种更稳妥：

1. 文档模式内部仍使用统一的状态归一化。
2. 普通问答路径保持 status 为空，不制造伪 continuing。

## 5.3 后端 P0：索引侧不要仅凭 artifact/status 非空就写成 long_document

buildMessageIndexOptions 目前的判定条件过于宽松。

建议改为：只有在“权威文档信号”为真时，才设置：

1. `TaskKind = long_document`
2. `DocumentGenerationStatus = ...`
3. `ArtifactID = ...`

否则：

1. 即使存在某个宽松注册出的 artifact，也不要把消息索引成 long_document。
2. 普通回答应继续按普通 Q/A 写入历史知识库。

## 5.4 前端 P1：规划卡片展示应再加一层模式保护

前端不是主根因，但可以补一道保险。

建议 AgentStreamDisplay 的 visiblePlanningOutline 展示增加限制：

1. 只有当当前消息带有明确 document_generation_status，且该状态来自权威文档模式。
2. 或者当前消息存在真实 long document task / authoritative artifact 标记。

才展示顶部“文档规划”卡片。

不要仅凭 artifact snapshot 中存在 `#` / `##` 标题，就把普通长回答解析成规划视图。

否则即使后端偶发混入结构化回答，前端也会进一步放大误判。

## 6. 最终判定

针对“默认调派规则是什么”这一问，本次问题的最终判定如下：

1. 它没有在入口路由阶段进入 long-document 执行链。
2. 它在 completion 收尾阶段被错误注册为 chat document artifact。
3. 它随后被索引成 long_document，并带上 continuing 状态。
4. 前端据此展示了“文档规划”卡片。

所以，这不是“单纯前端误显示”，也不是“入口路由误判 full_document”，而是：

普通问答在回答完成后，被后端 artifact 注册与索引逻辑错误晋升为长文档结果。

这也是本次应优先修复的真实根因。
