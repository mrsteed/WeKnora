# WeKnora 工程中 Go 测试目录组织建议

## 1. 结论

对于当前 WeKnora 代码库，**不建议为 Go 的 `*_test.go` 文件单独建立一个统一的专属测试目录**，应继续以**测试文件与被测包就近放置**为主。

原因不是泛泛的“Go 社区习惯如此”，而是当前工程已经形成了较稳定且与 Go 工具链高度一致的测试组织方式：

1. 仓库内现有大量测试文件已经分散放在各自包目录下，而不是集中放在顶层 `tests/` 或 `test/`。
2. 顶层 [Makefile](../Makefile#L94-L95) 的测试入口直接使用 `go test -v ./...`，这天然假设测试文件跟随包结构参与发现与编译。
3. 现有测试中存在直接访问包内未导出类型或方法的情况，例如 [internal/application/service/long_document_test.go](../internal/application/service/long_document_test.go#L1) 与 [internal/handler/session/long_document_task_test.go](../internal/handler/session/long_document_task_test.go#L1) 均采用与被测代码相同的 package 声明；其中前者还直接校验未导出实现细节 `planBatches`。
4. 仓库当前没有形成 Go 级别的集中测试目录先例。现有 [tests](../tests) 目录下也没有发现 `*_test.go` 文件，说明它并不是现有 Go 单元测试的承载位置。

因此，从工程一致性、维护成本、Go 原生工具链兼容性三个维度看，当前最优策略不是“集中收拢测试文件”，而是“维持包内测试为主，按测试层级补充少量专用目录”。

## 2. 先纠正一个命名点

在 Go 工程里，真正被 `go test` 自动识别的是 **`*_test.go`**，而不是其他语言里常见的 `*.test.go` 语义化变体。

也就是说，若讨论“Go 是否应该有专属测试目录”，准确对象应是：

- `*_test.go` 是否要统一迁移到一个测试目录；
- 还是继续与源代码同目录放置。

对当前仓库而言，答案是后者更合适。

## 3. 为什么当前工程不适合统一测试目录

## 3.1 Go 的包编译模型决定了“就近测试”成本更低

Go 测试不是简单的文件扫描，而是按 package 编译与执行。`*_test.go` 跟随包目录放置时，有几个直接收益：

1. `go test ./...` 可以自然递归所有包，不需要额外维护测试映射。
2. 测试可直接复用同目录包内辅助函数、测试夹具和 `testdata` 资源。
3. 对未导出符号的单元测试更自然，尤其适合验证边界条件、内部状态变换、分批算法、解析逻辑等。
4. 代码重构时，测试与实现一起移动，路径漂移最小。

WeKnora 当前的测试分布正是沿着这些收益展开的，已经与 Go 语言机制对齐。

## 3.2 当前仓库的测试文件分布已经证明“包内测试”是主模式

快速检查仓库可见：

- 测试文件主要集中在 `internal/application/service`、`internal/agent/tools`、`internal/types`、`internal/utils` 等业务或基础能力包内。
- 没有发现 `tests/` 或 `test/` 下承载 Go `*_test.go` 的现成结构。
- 当前测试入口仍是全仓库级的包递归执行，而不是先收集某个独立测试目录再做装配。

这说明当前项目不是“测试目录规划缺失”，而是已经明确选择了 **package-local test** 模式。

如果此时强行引入统一测试目录，会带来结构反转：

1. 原本按包自然归属的测试被迁移到外部目录，代码定位反而变慢。
2. 访问未导出实现会变得困难，往往迫使代码为“方便测试”而错误扩大导出面。
3. 测试辅助资源与相对路径引用更容易碎裂。
4. 包移动或重构时，需要同步维护集中目录中的镜像路径，变更面扩大。

## 3.3 当前仓库已有测试明显依赖包内可见性

例如 [internal/application/service/long_document_test.go](../internal/application/service/long_document_test.go#L1) 使用 `package service`，并直接验证内部服务对象与未导出方法的行为。这类测试如果被抽到独立目录，通常会面临三种不理想选择：

1. 改成只测导出接口，导致内部关键分支覆盖下降。
2. 人为导出原本不该暴露的函数或类型，污染设计边界。
3. 通过额外测试钩子、构造器或中转封装绕回来，增加工程噪音。

对于当前这样的服务层、handler 层、工具层密集工程，这种代价不值得。

## 4. 什么情况下可以有“专属测试目录”

不建议统一测试目录，不等于任何情况下都不能建测试专属目录。更专业的做法是：**把专属测试目录限制在跨包、跨模块、系统级验证场景**，而不是替代单元测试主布局。

当前仓库若未来要新增专属测试目录，建议仅用于以下场景：

### 4.1 端到端测试

例如：

- 从 HTTP API 入口触发完整流程；
- 联动数据库、对象存储、向量库、异步任务系统；
- 验证长文档生成、知识库导入、外部数据源同步等真实链路。

这类测试更像系统验证，而不是包内单元测试。适合放在例如：

- `integration/`
- `e2e/`
- `test/e2e/`

目录中，并通过独立命令、环境变量或 CI Job 运行。

### 4.2 测试夹具与共享测试资产

如果未来出现大量公共测试资产，而这些资产不适合放在某一个包目录中，可考虑集中存放：

- 通用 JSON/Markdown 样例
- 多模块共用的黄金文件
- 启动测试环境的脚本
- 大体积集成测试数据

但这里集中的是 **测试资源**，不一定是 Go `*_test.go` 文件本身。

### 4.3 黑盒契约测试

当目标是严格从外部接口视角验证行为，而故意不依赖包内实现时，可以建立少量独立测试目录，例如：

- API contract tests
- CLI behavior tests
- MCP/service integration tests

这类测试应被明确标识为“黑盒层”，而不是取代包内单测。

## 5. 对 WeKnora 更合适的推荐组织方式

建议采用分层组织，而不是“所有 Go 测试统一迁到一个目录”：

### 5.1 第一层：单元测试继续与包同目录放置

适用对象：

- service、repository、handler、utils、types、tools 等包内逻辑；
- 需要访问未导出符号或紧贴实现演化的测试。

执行方式：

- 继续保留 `go test ./...`
- 新增测试时优先放在对应实现文件旁边

这是当前仓库最符合成本收益比的默认策略。

### 5.2 第二层：为集成测试单独设目录，但不要混入单元测试

如果后续需要系统级验证，建议新增明确语义的目录，例如：

- `integration/`
- `test/integration/`
- `e2e/`

并配套：

- 独立运行命令，例如 `make test-integration`
- 明确依赖的外部服务
- 与 CI 分阶段执行

这样既保留 Go 原生单测体验，也避免把重型测试塞进默认快速反馈回路。

### 5.3 第三层：按需集中测试资源，而不是集中测试代码

对于复用率高的数据样本、Golden Files、协议样例、模拟响应，可以按需要建立：

- 包级 `testdata/`
- 仓库级 `testdata/`
- 集成测试专用 fixtures 目录

这比“统一搬迁所有 `*_test.go`”更实用，也更符合 Go 项目常见工程实践。

## 6. 不建议采取的方案

### 6.1 不建议把所有 `*_test.go` 统一迁入 `tests/`

这是当前最不推荐的方案。主要问题：

1. 破坏 Go 按包测试的天然组织。
2. 增加测试对导出 API 的不必要依赖。
3. 提高重构时路径同步和依赖维护成本。
4. 让测试与被测代码脱节，降低局部维护效率。

### 6.2 不建议为了集中测试而扩大导出面

如果迁移测试目录的代价是把内部函数、内部状态、内部构造细节公开出来，那么这是典型的测试反向塑造生产 API，会弱化封装边界。

### 6.3 不建议混淆“集成测试目录”和“所有测试目录”

工程上可以有集成测试目录，但这不等于所有测试都应该搬进去。二者语义不同：

- 包内 `*_test.go`：快速、局部、贴近实现。
- 独立测试目录：跨模块、重环境、偏黑盒。

应该并存，而不是互相替代。

## 7. 最终建议

针对当前 WeKnora 工程，建议如下：

1. **保持现状**：Go 单元测试继续与被测包同目录放置，不建立统一的 `*_test.go` 专属目录。
2. **补充分层**：如后续新增端到端或跨模块测试，可建立 `integration/` 或 `test/integration/`，但只承载系统级测试。
3. **统一约定**：把“单测就近、集成测试独立、测试资源按需集中”的规则写入开发文档或贡献指南。
4. **避免误导**：在团队内部统一使用 Go 正确术语 `*_test.go`，不要把其他语言的 `*.test.go` 命名习惯直接套用到 Go。

一句话总结：

**当前仓库不应该为 Go 的 `*_test.go` 建立统一专属测试目录；更合理的工程做法是保留包内测试为主，仅为集成/端到端测试增加少量语义明确的独立目录。**
