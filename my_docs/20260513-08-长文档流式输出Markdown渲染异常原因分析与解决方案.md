# 长文档流式输出 Markdown 渲染异常原因分析与解决方案

## 1. 问题结论

本次 `my_docs/test/3.mp4` 暴露的问题，不是上一版文档中重点分析的“完成态 artifact 内容覆盖 final_answer”单一问题。当前代码里，Agent 展示组件已经基本改成 `answer.content -> complete.final_answer -> artifact` 的优先级，完成态 artifact 也只在 `documentGenerationStatus === completed` 且没有质量问题时作为兜底答案使用。

视频中的主要异常是：长文档仍处于“生成中”时，正文里的合法 Markdown 加粗标记被原样展示，例如：

```markdown
**建设内容：**搭建覆盖全厂生产
**功能边界：**覆盖实时监测
**核心价值：**实现隐患早识别
```

这些内容本应渲染为加粗标签，但视频里直接显示 `**`。结合当前代码和最小复现实验，根因更准确地说是：**流式输出阶段直接把模型原始 chunk 交给前端 `marked` 渲染，而 `marked@17.0.5` 对“中文冒号加粗标签后紧跟正文”的写法会解析失败，导致 `**...**` 原样进入 HTML**。后端已有 Markdown 规范化逻辑能在最终文档阶段补空格，但流式中间态没有使用同一套规范化结果，所以用户在生成过程中仍会看到排版异常。

## 2. 视频现象证据

`my_docs/test/3.mp4` 关键帧显示：

- 右侧回答区域底部仍显示绿色状态点和“生成中”，说明这是 Agent/full_document 的同步流式正文展示，而不是异步 `LongDocumentTaskCard` 任务卡片。
- 正文中多处出现 `**建设内容：**`、`**功能边界：**`、`**核心价值：**` 等原始 Markdown 标记。
- 这些标记本身并非非法 Markdown 语义，问题不是“应该删除星号”，而是前端渲染链路没有把它稳定转换为 `<strong>`。

## 3. 当前代码链路

### 3.1 后端流式输出链路

长文档生成路径会通过 `EventAgentFinalAnswer` 事件持续发送 answer chunk：

```go
// internal/application/service/session_agent_qa.go
func emitDedicatedFullDocumentAnswerChunk(...) error {
    return eventBus.Emit(ctx, event.Event{
        Type: event.EventAgentFinalAnswer,
        Data: dedicatedDocumentEditEventData(content, done, ...),
    })
}
```

在章节生成过程中，代码先把模型返回的原始片段直接发给前端：

```go
rawSectionContent.WriteString(content)
emitDedicatedFullDocumentAnswerChunk(..., content, false, ...)
```

章节完成后才会进入质量闸：

```go
normalizedSectionContent, sectionSignals, qualityOK := applyGeneratedSectionMarkdownQualityGate(..., rawSectionContent.String())
```

因此，**生成中的页面看到的是 raw chunk，而不是最终 normalized section**。

### 3.2 后端最终 Markdown 质量闸

`internal/application/service/full_document_markdown_quality.go` 已经包含针对本问题的规范化规则：

```go
markdownQualityBoldLabelTightRE = regexp.MustCompile(`(\*\*[^*\n]+[：:]\*\*)([^\s\n])`)
...
line = markdownQualityBoldLabelTightRE.ReplaceAllString(line, `$1 $2`)
```

这条规则会把：

```markdown
**建设内容：**搭建覆盖全厂生产
```

修正为：

```markdown
**建设内容：** 搭建覆盖全厂生产
```

最终文档阶段这类内容有机会被修复，但流式阶段没有同步应用，所以视频里仍能看到原始星号。

### 3.3 前端 Agent 展示链路

`frontend/src/views/chat/index.vue` 中 `handleAgentChunk` 收到 `answer` 事件后，会累加到 `message.content` 和 answer event：

```ts
if (data.content) {
    message.content = (message.content || '') + data.content;
}
...
answerEvent.content = message.content;
```

`frontend/src/views/chat/components/AgentStreamDisplay.vue` 再通过：

```vue
<div v-html="renderAnswerContent(getAnswerDisplayContent(event))"></div>
```

将 answer 内容交给 `renderMarkdownContent`，最终调用 `marked.parse` 渲染。

表面上这条链路已经走了 Markdown 渲染，但关键问题是：**传给 `marked` 的流式内容仍保留了 `**标签：**正文` 的紧贴格式，而当前 `marked` 版本无法正确解析这种中文加粗标签场景**。

## 4. 最小复现结果

在 `frontend` 目录使用当前依赖 `marked@17.0.5` 做最小验证，结果如下：

```markdown
输入：• **建设内容：**搭建覆盖全厂生产
输出：<p>• **建设内容：**搭建覆盖全厂生产</p>

输入：- **建设内容：**搭建覆盖全厂生产
输出：<ul><li>**建设内容：**搭建覆盖全厂生产</li></ul>

输入：**建设内容：**
输出：<p><strong>建设内容：</strong></p>
```

这说明异常不是 `v-html` 或 DOMPurify 把 `<strong>` 清掉，也不是 `sanitizeForDisplay` 主动删除了 Markdown 能力，而是 `marked` 对“加粗闭合符后立刻跟随中文正文”的强调语法识别失败。

换句话说，下面的紧贴格式风险较高：

```markdown
- **建设内容：**搭建覆盖全厂生产
```

而下面的格式可以稳定渲染：

```markdown
- **建设内容：** 搭建覆盖全厂生产
```

## 5. 为什么上一版修复后仍不理想

上一版方案主要处理的是：

- 完成态 answer、final_answer、artifact 多正文来源冲突。
- artifact-first 导致正文覆盖或回退错误。
- 后端最终文档 Markdown 质量闸不完整。
- 标题、章节、内部提示泄漏等完成态质量问题。

当前代码已经体现了一部分修正：

- `AgentStreamDisplay.vue` 的 `getActualContent` 已优先使用 `answerEvent.content`，其次是 `complete.final_answer`，最后才在严格条件下使用 artifact。
- `canUseArtifactDocumentAsAnswer` 要求 artifact 已完成且没有质量问题。
- `full_document_markdown_quality.go` 已包含 `markdownQualityBoldLabelTightRE`，可修正最终文档里的 `**标签：**正文`。

但视频对应的是“生成中”的流式正文展示。这个阶段有两个缺口：

1. 模型 raw chunk 在章节完成前已经发给前端，尚未经过 `normalizeGeneratedMarkdown`。
2. 前端 `normalizeMarkdownHeadings` 只处理标题粘连，不处理 `**标签：**正文` 这类 inline Markdown 兼容性问题。

因此，即使最终完成态能被规范化，用户在长文档生成过程中仍会看到明显排版异常。

## 6. 根因分层

### 6.1 直接根因

`marked@17.0.5` 对 `**中文标签：**中文正文` 的紧贴写法解析不稳定，尤其出现在列表项、项目符号或段落内时，会保留原始星号。

### 6.2 触发条件

长文档模型经常生成投标方案式条目：

```markdown
• **建设内容：**搭建...
• **功能边界：**覆盖...
• **核心价值：**实现...
```

这类业务文本天然高频使用“加粗标签 + 冒号 + 正文”的格式，而且模型通常不会在 `**` 后主动补空格。

### 6.3 架构原因

后端的 Markdown 质量闸在章节结束或最终文档阶段生效，但前端流式展示的是每个 raw chunk 的累积结果。生成中页面和最终 artifact 使用的不是同一份规范化内容。

### 6.4 体验原因

前端没有在流式渲染前执行与后端一致的 lightweight Markdown 规范化，也没有为高频业务格式建立 fixture 测试，所以问题在最终内容修复后仍然会在“生成中”可见。

## 7. 解决方案

### 7.1 前端增加 streaming-safe Markdown 规范化

在 `AgentStreamDisplay.vue` 的 `renderMarkdownContent` 进入 `marked.parse` 前，增加一个仅面向展示的轻量规范化函数，例如：

```ts
const normalizeStreamingMarkdownForRender = (content: string): string => {
  if (!content) return content;

  return content
    .split('\n')
    .map((line) => {
      const trimmed = line.trimStart();
      if (trimmed.startsWith('```')) return line;
      return line.replace(/(\*\*[^*\n]+[：:]\*\*)([^\s\n])/g, '$1 $2');
    })
    .join('\n');
};
```

接入位置建议放在 `sanitizeForDisplay` 之后、`marked.parse` 之前：

```ts
let sanitized = sanitizeForDisplay(preservedWithWiki);
sanitized = normalizeStreamingMarkdownForRender(sanitized);
```

这样不会改变后端原文存储，也不会影响 SSE 协议，只用于前端渲染兼容。

注意事项：

- 必须跳过 fenced code block，避免修改代码示例。
- 只处理 `**...：**正文` 和 `**...:**正文`，不要泛化到所有星号场景。
- 不要删除星号，正确做法是补一个空格，让 Markdown parser 可以识别 `<strong>`。

### 7.2 后端流式 chunk 发出前做轻量规范化

前端修复能快速改善体验，但更稳妥的做法是在后端发送长文档 answer chunk 前，也做与最终质量闸一致的低风险规范化。

推荐在 `emitDedicatedFullDocumentAnswerChunk` 调用前，针对“展示性 chunk”增加 lightweight normalizer：

```go
func normalizeStreamingMarkdownChunkForDisplay(content string) string {
    // 只处理不会改变语义的 Markdown 间距问题：
    // 1. **标签：**正文 -> **标签：** 正文
    // 2. -正文 -> - 正文
    // 3. #标题 -> # 标题
    // 跳过代码围栏内文本。
}
```

后端方案的收益是：

- Web、桌面端、小程序或未来其它客户端都能收到更稳定的 Markdown。
- 流式 answer 和最终 final_answer 的格式更一致。
- 避免不同前端各自补丁导致表现不一致。

但后端流式 chunk 有一个难点：chunk 可能切在 `**建设内容：**` 中间。为了避免误改半截 token，建议采用“章节级缓冲 + 行级 flush”策略，而不是逐 token 直接 regex：

- 后端维护当前未完成行 buffer。
- 只有遇到换行或章节结束时，才对完整行执行 lightweight normalizer。
- 对仍未闭合的最后一行，继续原样缓存，下一次 chunk 补齐后再发送。

如果短期不想改后端流式协议，可以先落地前端规范化；后端作为第二阶段收口。

### 7.3 统一生成中与完成态的 Markdown 标准

当前最终文档已经有 `normalizeGeneratedMarkdown`，但生成中展示没有复用同一标准。建议抽象一份“Markdown 展示规范”并明确适用范围：

- 后端最终内容：继续使用 `normalizeGeneratedMarkdown` 和 `applyGeneratedDocumentMarkdownQualityGate`。
- 后端流式展示：只使用不会改变语义的 lightweight normalizer。
- 前端渲染兜底：使用同名规则做 display-only normalize，保证旧服务端或缓存场景也能正确展示。
- 测试 fixture：同一批 Markdown 样例必须覆盖 streaming 和 completed 两种状态。

### 7.4 补齐前端回归测试

需要为 `AgentStreamDisplay.vue` 或其 Markdown 渲染工具增加 fixture。至少覆盖：

```markdown
- **建设内容：**搭建覆盖全厂生产
- **功能边界：**覆盖实时监测、智能预警
• **核心价值：**实现隐患早识别
**实施路径：**先建设平台，再接入业务系统
```

断言结果应包含：

```html
<strong>建设内容：</strong>
<strong>功能边界：</strong>
<strong>核心价值：</strong>
<strong>实施路径：</strong>
```

同时增加负例：

- fenced code block 内的 `**建设内容：**搭建` 不应被改写。
- 普通数学表达式、URL、图片链接不应被破坏。
- 已经有空格的 `**建设内容：** 搭建` 不应重复插入空格。

### 7.5 检查构建与缓存

由于当前源码中 `AgentStreamDisplay.vue` 已经走 `v-html + marked`，如果修复后仍看到原始星号，需要检查运行环境是否加载了最新前端构建：

- 确认容器内前端静态资源已经重新构建并替换。
- 清理浏览器缓存或确认资源 hash 已变化。
- 检查 nginx/CDN 是否仍在返回旧的 `assets/*.js`。
- 在浏览器开发者工具里搜索 `canUseArtifactDocumentAsAnswer` 或新增 normalizer 函数名，确认运行时代码与源码一致。

## 8. 推荐落地顺序

### 第一阶段：前端快速止血

在 `AgentStreamDisplay.vue` 增加 `normalizeStreamingMarkdownForRender`，并在 `renderMarkdownContent` 进入 `marked.parse` 前调用。这个改动范围小，对用户可见问题改善最快。

### 第二阶段：后端流式输出收口

在长文档 answer chunk 发送前，基于完整行进行 lightweight Markdown normalize，保证各端一致。不要直接对任意 token chunk 做全局替换，避免跨 chunk 半截语法被误处理。

### 第三阶段：完成态质量闸复核

保留并扩展 `full_document_markdown_quality.go` 的现有规则，确认以下最终内容都能被规范化并通过校验：

- 标题缺空格：`##标题`
- 列表缺空格：`-正文`
- 加粗标签后紧贴正文：`**建设内容：**正文`
- 内部 prompt 标签泄漏
- 非规划章节或过深标题

### 第四阶段：端到端验证

使用与视频相同的长文档提示词重新生成，分别检查：

- 生成中阶段不再出现可见的 `**建设内容：**` 原始星号。
- 完成后 answer 与 artifact 内容一致。
- 自动续写过程中新增章节同样保持加粗渲染。
- 历史会话重新打开后仍然显示 `<strong>`，不是原始 Markdown。

## 9. 验证清单

修复完成后建议按以下清单验收：

- [ ] `- **建设内容：**搭建...` 渲染为列表项加粗标签。
- [ ] `• **核心价值：**实现...` 渲染为加粗标签，或至少不显示原始 `**`。
- [ ] 生成中、完成态、历史会话三种状态表现一致。
- [ ] `LongDocumentTaskCard` 异步任务卡片不受影响。
- [ ] fenced code block 内 Markdown 样例不被改写。
- [ ] 图片、知识引用、wiki 链接、KaTeX 渲染不受影响。
- [ ] `completion_status=partial/failed/cancelled` 时，已生成内容仍可正常 Markdown 渲染。
- [ ] 前端重新构建部署后，浏览器加载的是最新静态资源。

## 10. 最终判断

本次视频问题的核心不是“长文档最终内容没有被保存为 artifact”，也不是“Markdown 星号应该被清洗掉”。真正的问题是：**长文档流式展示阶段缺少与最终质量闸一致的 Markdown 兼容性规范化，而当前 `marked` 对中文加粗标签紧贴正文的语法解析会失败**。

优先在前端渲染前补一个 display-only normalizer，可以快速消除用户在生成中看到的原始星号；随后在后端流式 chunk 层按完整行补齐轻量规范化，才能让长文档在生成中、完成后、历史回放和多端展示中保持一致。
