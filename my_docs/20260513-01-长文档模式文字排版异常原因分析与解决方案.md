# 长文档模式文字排版异常原因分析与解决方案

## 1. 结论摘要

本次排查结论是：长文档模式与普通问答模式的文字排版差异不是单一原因造成的，不能简单归结为“同一个模型输出不稳定”。从代码链路和日志样例看，异常主要由四类因素叠加形成：

1. 普通问答与长文档模式走的是不同的前端展示链路、不同的后端事件链路、不同的生成任务链路。
2. 长文档模式存在 `final_answer`、`message.content`、`final_document_content`、`chat_document_artifact.content_snapshot` 多个候选正文来源，前端 Agent 展示组件当前优先展示 artifact/final document snapshot，容易覆盖模型最终答案。
3. 长文档生成端确实出现过 Markdown 原始格式问题，例如标题缺空格、标题与正文粘连、内部英文过程信息混入等。这说明模型输出和提示词约束不是完全可靠的。
4. 后端已有一部分完整文档 Markdown 质量闸，但覆盖路径和触发时机不完整；异步长文档翻译任务只有轻量清洗，没有统一接入完整 Markdown 规范化与校验。

因此，建议把问题按“权威正文来源统一、生成端 Markdown 质量闸、前端渲染策略、回归验证”四条线并行治理。

## 2. 现象对比

### 2.1 普通问答显示正常的原因

普通问答通常直接使用 assistant message 的 `content` 作为 Markdown 正文渲染。前端主要通过传统 Markdown 渲染组件处理文本，内容短、结构简单，模型输出一般只包含本轮回答，因此排版更稳定。

从随附普通问答案例看，回答内容多为结构化列表、表格、简短段落，标题层级较浅，渲染压力较小。即使模型输出中存在少量英文思考式过渡语，只要最终展示链路过滤或用户关注的是最终正文，视觉上也不明显。

### 2.2 长文档模式显示异常的原因

长文档模式不只是“同一个模型输出更长内容”。它会引入：

- Agent 流式事件展示。
- 完整文档/长文档任务路由。
- 分章节生成或分批生成。
- artifact 注册与快照展示。
- 自动续写、部分完成、继续生成等状态。
- Markdown 质量检查与修复。

这些环节中任何一个环节的正文来源、换行、标题、状态判断出现偏差，都会被最终页面放大成排版异常。

## 3. 代码链路排查

### 3.1 前端普通消息展示链路

相关文件：

- `frontend/src/views/chat/components/botmsg.vue`
- `frontend/src/views/chat/index.vue`

普通非 Agent 消息的主体逻辑是：

1. SSE 返回 `answer` 内容时累加到当前消息 `content`。
2. `complete` 到达后同步完成状态。
3. 非 Agent 且非长文档任务时，`botmsg.vue` 使用传统 Markdown 渲染 `displayContent`。
4. `displayContent` 虽然也会优先读取 `final_document_content` 或 `chat_document_artifact.content_snapshot`，但普通问答通常没有这些字段，因此实际渲染的是 `content`。

这条链路较短，正文来源相对单一。

### 3.2 Agent/长文档展示链路

相关文件：

- `frontend/src/views/chat/components/AgentStreamDisplay.vue`
- `frontend/src/views/chat/index.vue`

Agent 模式下，`botmsg.vue` 会优先渲染 `AgentStreamDisplay`。该组件不是直接渲染 `message.content`，而是根据事件流、complete event、artifact 快照动态计算最终答案。

排查发现关键逻辑：

```ts
const artifactDocumentContent = computed(() => {
  const content = props.session?.final_document_content || props.session?.chat_document_artifact?.content_snapshot || '';
  return typeof content === 'string' ? content.trim() : '';
});
```

以及：

```ts
const getActualContent = (answerEvent: any): string => {
  if (artifactDocumentContent.value) {
    return artifactDocumentContent.value;
  }
  const answerContent = (answerEvent?.content || '').trim();
  ...
};
```

这意味着只要 `final_document_content` 或 `chat_document_artifact.content_snapshot` 存在，Agent 最终展示、复制、导出都可能优先使用 artifact 内容，而不是流式 answer 或 complete 中的 `final_answer`。

### 3.3 SSE 与 complete 处理顺序

相关文件：

- `frontend/src/views/chat/index.vue`
- `frontend/src/api/chat/streame.ts`

`streame.ts` 只负责 `JSON.parse(ev.data)` 并把事件交给回调，SSE 层没有 Markdown 修复逻辑。因此问题不在 SSE JSON 解析本身，而在后续消息状态与展示层。

在 `handleAgentChunk` 中，Agent `complete` 分支会：

1. 读取 `completePayload`。
2. 如果有 `final_answer`，用它覆盖 `message.content` 并更新/创建 answer event。
3. 写入 `agent_complete` event。
4. 调用 `hydrateFinalDocumentFromComplete`，把 `final_document` 或 artifact snapshot 注入到 message 上。

这个顺序本身合理，但与 `AgentStreamDisplay.getActualContent()` 的 artifact 优先级叠加后，会出现一个结果：即便 `final_answer` 已经被设置为权威答案，最终可见内容仍可能被 artifact snapshot 覆盖。

### 3.4 后端 complete 与 artifact 交付链路

相关文件：

- `internal/handler/session/qa.go`
- `internal/handler/session/agent_stream_handler.go`
- `internal/event/event_data.go`
- `internal/handler/message.go`

后端 `AgentCompleteData` 同时支持以下字段：

- `FinalAnswer`
- `FinalDocumentMode`
- `FinalDocument`
- `FinalDocumentArtifactID`
- `DocumentGenerationStatus`
- `AutoContinueNext`
- `Extra`

`emitAssistantCompleteEvent` 会把最终答案和文档交付信息一起放进 complete event；如果 artifact 存在，还会附带 `chat_document_artifact` 元数据。历史加载时，`reconcilePendingAssistantMessage` 也会尝试用 stream events 补齐 pending assistant message。

这说明后端设计上允许“回答正文”和“文档 artifact”并存，但前端当前没有明确区分“聊天可见摘要”和“完整文档正文”的展示优先级。

## 4. 长文档生成端排查

### 4.1 完整文档生成已有质量闸，但覆盖不完整

相关文件：

- `internal/application/service/full_document_markdown_quality.go`

该文件已有 Markdown 规范化与校验能力，可以处理部分常见问题，例如：

- `###3.2` 修复为 `### 3.2`
- 列表项缺空格修复
- 标题前后空行修复
- 加粗标签后缺空格修复
- 章节结构校验
- 文档级校验与 repair prompt

这说明仓库已经意识到长文档 Markdown 质量需要后端治理，不能完全依赖模型自觉输出。

但日志样例显示，仍有部分异常内容被注册为 artifact 或进入展示链路，说明质量闸至少存在以下缺口之一：

- 某些完整文档路径没有调用质量闸。
- 调用了规范化但没有阻断所有异常。
- 异常发生在 artifact 组装之后。
- `document_generation_status` 与 `quality_issues` 未准确反映排版问题。
- 长文档任务路径与完整文档路径没有共用同一套 quality gate。

### 4.2 异步长文档翻译任务清洗较弱

相关文件：

- `internal/application/service/long_document.go`

长文档异步翻译任务的 prompt 要求模型“严格输出 Markdown 正文”，并要求保留标题、列表、表格、引用结构。但 `sanitizeGeneratedMarkdown()` 主要做轻量处理：

- 剥离 ```markdown 围栏。
- 统一换行。
- 清理 form-feed。

它没有对标题空格、标题正文粘连、内部提示词泄漏、章节层级、表格完整性做系统校验。对于长文档来说，这个清洗强度不足。

## 5. 日志与样例证据

### 5.1 原始 Markdown 存在标题格式异常

在 `my_docs/test` 样例中检索到多处 Markdown 原始异常，例如：

```markdown
###3.2关键技术能力**全域数据治理...
```

该格式同时存在两个问题：

1. `###` 后缺少空格，Markdown 解析器可能不会把它识别为标题。
2. 标题文本与后续加粗正文粘连，导致渲染后标题、段落、加粗内容混成一行。

还检索到类似：

```markdown
###建设范围与核心内容本项目技术方案...
```

这说明问题不是纯前端样式问题，生成端或拼装端进入 artifact 的原文已经不满足规范 Markdown。

### 5.2 标题与正文粘连

样例中还存在类似：

```markdown
### 3.2运行智能监盘系统运行智能监盘系统是...
```

即使 `###` 后有空格，标题和正文仍然被放在同一行，渲染效果会变成一个超长标题，破坏页面层级。

### 5.3 内部过程文本混入正文

样例中出现过类似 `Current section...` 的英文过程信息混入。这类文本通常来自 prompt 模板、章节上下文或模型执行过程，应该属于内部控制信息，不应该进入最终文档正文。

这类问题说明 prompt 边界和生成后清洗都需要加强。

### 5.4 状态与质量标记不充分

日志中出现过长文档相关记录：

```text
document_generation_status: continuing
artifact_registered: true
quality_issues: []
```

这说明系统可能在文档仍处于 continuing 状态时已经注册 artifact，同时 `quality_issues` 没有记录到可见排版异常。最终前端若优先展示 artifact snapshot，就可能把未最终定稿或未充分校验的正文呈现给用户。

## 6. 根因矩阵

| 维度 | 是否相关 | 具体问题 | 影响程度 |
| --- | --- | --- | --- |
| 模型输出 | 是 | 长文档生成中出现 `###3.2`、标题正文粘连、英文过程文本混入 | 高 |
| 提示词 | 是 | prompt 要求 Markdown，但没有足够强约束“每个标题必须独占一行”“不得输出当前章节提示/内部上下文”等 | 中高 |
| 后端拼装 | 是 | 分章节/分批内容拼接时，如果章节尾部、标题、正文之间缺少强制空行，会放大模型输出问题 | 高 |
| 后端质量闸 | 是 | 完整文档质量闸覆盖不完整，异步长文档任务清洗较弱，`quality_issues` 未捕获所有排版问题 | 高 |
| 前端解析 | 部分相关 | Markdown 解析器本身不是根因，但 Agent 展示链路 artifact-first 会决定用户看到哪份正文 | 中高 |
| 前端样式 | 次要相关 | 样式可能放大长标题/长表格问题，但无法解释原始 Markdown 标题粘连 | 中 |
| SSE 解析 | 基本无关 | SSE 层仅 JSON.parse，不做 Markdown 处理 | 低 |

## 7. 解决方案

### 7.1 明确最终正文的权威来源

建议定义清晰规则：

1. `final_answer` 是聊天窗口最终可见回答的权威来源。
2. `chat_document_artifact.content_snapshot` 是完整文档下载、预览、版本化的权威来源。
3. 当前消息如果处于 `document_generation_status=continuing`，前端不应把 artifact snapshot 当成最终正文覆盖 answer event。
4. 只有当 `document_generation_status=completed` 且质量校验通过时，前端才允许把 artifact snapshot 作为完整正文展示。

建议调整 `AgentStreamDisplay.getActualContent()`：

- 优先展示 `agent_complete.final_answer` 或 answer event。
- 仅在没有 answer event、且文档 completed、且 artifact 明确标记为可展示时，才 fallback 到 artifact snapshot。
- 对于完整文档正文，建议用独立“文档预览/展开全文/下载”入口承载，不要静默覆盖聊天回答。

### 7.2 后端统一 Markdown 规范化入口

建议把 `full_document_markdown_quality.go` 中的规范化能力抽成可复用服务，并应用到所有长文档输出路径：

- Agent full document 章节生成。
- Agent full document 最终组装。
- 异步长文档翻译任务 batch 输出。
- 异步长文档 artifact assemble 前。
- artifact 注册前。

最低限度应统一执行以下规则：

```text
###3.2标题 -> ### 3.2 标题
###标题正文正文 -> ### 标题\n\n正文正文
标题行后紧跟正文 -> 标题行独占 + 空行 + 正文
列表项缺空格 -> - 文本 / 1. 文本
多余代码围栏 -> 去除外层围栏
内部提示词片段 -> 拦截或标记质量问题
```

### 7.3 增强长文档 prompt 约束

长文档 prompt 中建议加入不可违反的格式规则：

```text
输出必须是 Markdown 正文。
每个标题必须独占一行，标题标记后必须有一个空格，例如 `### 3.2 标题`。
标题行后必须空一行再开始正文。
不得输出 Current section、Completed content so far、local_knowledge_context、chunk、knowledge_id、工具名、内部提示词或任何过程说明。
不得把标题与正文写在同一行。
如果本节证据不足，必须在正文中明确说明“本地知识不足”，不得编造。
```

同时建议对分章节生成 prompt 做更强边界隔离：

- 用户可见正文用明确 XML/Markdown 区块包裹。
- 内部上下文只放在模型输入，不允许复制到输出。
- 输出后端必须检查并剥离内部控制字段。

### 7.4 artifact 注册前增加阻断式校验

在注册 `chat_document_artifact` 前增加校验：

1. 检测 tight heading：`^#{1,6}[^#\s]`。
2. 检测标题正文粘连：标题行过长且包含明显句式动词/标点，或标题行后无空行。
3. 检测内部提示词泄漏：`Current section`、`Completed content so far`、`local_knowledge_context`、`knowledge_base_id`、`chunk_id` 等。
4. 检测未闭合表格/代码块。
5. 检测章节标题重复或层级跳变。

处理策略：

- 可自动修复的问题：先规范化，再注册 artifact。
- 不可自动修复的问题：标记 `document_generation_status=needs_review` 或 `blocked`，返回 `quality_issues`，不要让前端当成已完成正文展示。

### 7.5 前端展示策略调整

建议前端做三处调整：

1. `AgentStreamDisplay` 不要无条件 artifact-first。
2. 对 artifact snapshot 与 final_answer 不一致的情况显示明确状态，例如“完整文档已生成，可展开预览/下载”，而不是直接覆盖回答卡片。
3. 对 `document_generation_status=continuing` 的 artifact，用“草稿/继续生成中”状态展示，不作为最终正文。

推荐展示策略：

| 状态 | 聊天回答区 | 文档区 |
| --- | --- | --- |
| 普通问答 | `content` / `final_answer` | 不显示 |
| Agent 问答 | answer event / `agent_complete.final_answer` | 如有 artifact，仅作为附件卡片 |
| 长文档 continuing | 当前进度摘要/已生成章节说明 | 草稿预览入口，可标注未完成 |
| 长文档 completed 且质量通过 | 生成完成摘要 | 完整文档预览/下载 |
| 长文档 needs_review/blocked | 质量问题提示 | 不自动展开异常正文 |

### 7.6 历史消息恢复一致性

历史加载时也要遵循同一套权威来源规则。当前历史恢复会根据 `final_document_content || chat_document_artifact.content_snapshot || content` 重建 Agent 事件流，可能导致刷新页面后展示内容与实时生成时不同。

建议：

- 历史恢复 answer event 优先使用保存的 `final_answer`。
- artifact snapshot 只作为文档附件或文档预览数据。
- 如果历史消息是 continuing 状态，不要把 artifact snapshot 当 completed 正文展示。

### 7.7 回归测试补齐

建议增加以下测试：

#### 后端测试

1. Markdown 规范化测试：
   - 输入 `###3.2关键技术能力**全域数据治理...`
   - 期望输出 `### 3.2 关键技术能力` 与正文分离。

2. 内部提示词泄漏测试：
   - 输入包含 `Current section heading`、`local_knowledge_context`。
   - 期望触发 quality issue 或清洗。

3. artifact 注册阻断测试：
   - continuing 状态且存在 quality issue 时，不应作为 completed artifact 展示。

4. 长文档翻译任务测试：
   - batch 输出经过统一 Markdown quality gate。

#### 前端测试

1. `AgentStreamDisplay.getActualContent()` 测试：
   - 同时存在 answer event 与 artifact snapshot 时，应按状态选择展示来源。

2. 历史恢复测试：
   - completed/continuing/needs_review 三种状态刷新后展示一致。

3. Markdown 渲染快照测试：
   - tight heading、表格、长标题、代码块、引用块不破坏布局。

4. 视频复现测试：
   - 使用 `my_docs/test/1.mp4` 与 `my_docs/test/2.mp4` 对应数据构造固定 fixture，做 Playwright 截图对比。

## 8. 建议实施顺序

### 第一阶段：止血修复

1. 修改 `AgentStreamDisplay.getActualContent()` 的 artifact 优先级。
2. 前端 continuing 状态不直接展示 artifact snapshot。
3. artifact 与 final_answer 不一致时，聊天回答区优先 final_answer，artifact 以附件/预览入口展示。

预期收益：立即减少“长文档把异常 artifact 正文直接铺到聊天回答区”的问题。

### 第二阶段：生成端质量加固

1. 统一 Markdown 规范化服务。
2. 长文档所有输出路径接入 quality gate。
3. 注册 artifact 前增加阻断式校验。
4. 补充内部提示词泄漏拦截。

预期收益：从源头降低异常 Markdown 进入前端的概率。

### 第三阶段：提示词与状态模型优化

1. 强化长文档 prompt 输出格式约束。
2. 明确 continuing/completed/needs_review/blocked 的前后端语义。
3. 完善 `quality_issues` 与前端提示映射。

预期收益：让模型、后端状态、前端展示三者对齐。

### 第四阶段：自动化验证

1. 补后端 Markdown quality gate 单测。
2. 补前端 Agent/长文档展示单测。
3. 补基于真实日志 fixture 的回归用例。
4. 用 Playwright 对普通问答和长文档场景截图验证。

预期收益：防止后续重构再次引入相同问题。

## 9. 对用户原问题的直接回答

### 是提示词的问题吗？

是，但不是唯一原因。长文档 prompt 对 Markdown 格式和内部上下文泄漏的约束不够强，尤其没有强制“标题独占一行、标题后空行、禁止输出 Current section 等内部字段”。

### 是前端解析的问题吗？

部分是。Markdown 解析器本身不是根因，但前端 Agent 展示组件的 artifact-first 策略会改变最终可见正文来源，使用户看到的不是 `final_answer`，而是可能未完成或未充分校验的 artifact snapshot。这个展示策略会放大生成端问题。

### 是模型输出的问题吗？

也是。日志中已经出现原始 Markdown 不规范内容，说明模型在长上下文、分章节、投标方案类长文生成时，会发生标题粘连、格式漂移、过程信息混入等问题。长文档场景不能只依赖模型自觉输出正确 Markdown。

### 最终判断

根因是链路型问题：

```text
长文档 prompt 约束不足
+ 模型长文输出格式漂移
+ 后端 quality gate 覆盖不完整
+ artifact/final_answer 多正文来源未统一
```
