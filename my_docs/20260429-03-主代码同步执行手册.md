# 20260429 主代码同步执行手册

## 1. 文档目标

本文档基于 2026-04-29 当前仓库实际状态，给出一套可直接执行的主线同步方案，目标是：

1. 将当前 `dev` 分支从 `Tencent/WeKnora:main` 的最新基线补齐。
2. 保留本地分支已经落地的业务定制和前一轮同步成果。
3. 把同步动作控制在独立临时分支中，避免直接破坏当前 `dev`。
4. 在同步完成后，形成可验证、可回滚、可推送的交付结果。

本文档描述的是“如何同步”，不是直接修改当前分支指针。

## 3. 本轮同步建议策略

### 3.1 推荐方案

推荐使用：

1. 先为当前 `dev` 打备份分支。
2. 从当前 `dev` 切出新的临时同步分支。
3. 在临时同步分支上执行 `rebase upstream/main`。
4. 处理冲突并完成聚焦验证。
5. 验证通过后，再决定是否覆盖更新 `origin/dev`。

### 3.2 为什么不继续沿用旧同步分支

当前 `sync/dev-from-upstream` 是 `dev` 的祖先提交，`dev` 相对它新增了 2 个提交：

1. `aa944579` 解决编译错误
2. `82186fc6` 实现对话结果导出为文件的功能

因此本轮同步应以当前 `dev` 为起点，而不是回到旧同步分支继续叠加。

### 3.3 为什么优先选 rebase 而不是直接 merge

当前场景下优先选择 `rebase` 的原因：

1. 当前 `dev` 已经包含上一轮同步结果与适配提交，直接从 `dev` 重放到最新 `upstream/main` 更符合现状。
2. `rebase` 完成后，新的同步结果会形成一条更干净的线性历史，便于后续替换 `origin/dev`。
3. 当前工作区干净，且已有 `backup/dev-20260428` 可作历史参考，继续执行一次受控 rebase 的风险可控。

如果 rebase 过程中出现大面积语义冲突，再退回到“从 `upstream/main` 新开分支后按主题 cherry-pick”的兜底方案。

## 7. 推荐执行流程

### 7.1 同步前检查

在仓库根目录执行：

```bash
git checkout dev
git status --short --branch
git fetch origin --prune
git fetch upstream --prune
git pull --ff-only origin dev
git rev-list --left-right --count HEAD...upstream/main
```

预期结果：

1. 工作区必须保持 clean。
2. `dev` 必须先与 `origin/dev` 对齐。
3. 再次确认当前仍是 `ahead 24 / behind 14` 或接近该数量级。

### 7.2 备份当前 dev

```bash
git checkout dev
git branch backup/dev-20260429-before-sync
git tag sync-dev-20260429-pre
```

说明：

1. 备份分支用于随时回退。
2. tag 用于快速定位同步前基线。

### 7.3 创建本轮同步分支

```bash
git checkout -b sync/dev-20260429-main dev
```

分支命名建议：

1. 继续使用 `sync/` 前缀。
2. 带上日期，避免和旧的 `sync/dev-from-upstream` 混淆。

### 7.4 在同步分支上执行 rebase

```bash
git rebase upstream/main
```

冲突时按以下顺序处理：

1. 先处理 `internal/agent/tools/data_analysis.go` 和 `internal/container/container.go`。
2. 再处理 `internal/agent/think.go`。
3. 之后处理前端 i18n、聊天展示和 Agent 编辑器相关文件。
4. 最后处理 `.env.example`、`.gitignore` 这类低复杂度文件。

每轮冲突处理完成后执行：

```bash
git add <冲突文件>
git rebase --continue
```

如果需要放弃当前同步：

```bash
git rebase --abort
git checkout dev
```

### 7.5 冲突处理原则

处理冲突时统一遵循以下原则：

1. 上游新增的安全修复、超时控制、错误处理、重试与数据校验能力优先保留。
2. 本地已经验证过的业务扩展能力不能被简单回退。
3. 对于前端多语言文件，优先保留上游新增 key，再补回本地功能需要的文案。
4. 对于 `data_analysis` 链路，优先保留当前 upstream 的接口语义，再把本地 DuckDB / 导出 / 文件物化适配补齐。
5. 对于聊天展示组件，优先确保流式渲染、LaTeX 展示与导出行为都不退化。

## 8. 建议验证步骤

### 8.1 后端聚焦验证

优先执行与本轮热点最相关的测试或编译校验：

```bash
go test ./internal/agent/...
go test ./internal/handler/...
go test ./internal/container
go test ./internal/models/chat ./internal/application/service/chat_pipeline ./internal/handler/session
go test ./internal/agent/tools
```

如果需要更大范围验证，再执行：

```bash
make test
```

注意：

1. 全量测试可能受外部依赖影响。
2. 如果 `vectorstore` 或搜索相关测试依赖外部服务，应单独记录失败原因，不要把环境问题误判为同步问题。

### 8.2 前端验证

当前仓库前端位于 `frontend`，推荐执行：

```bash
cd frontend
pnpm install --frozen-lockfile
pnpm build
```

如果本机未启用 `pnpm`：

```bash
corepack enable
pnpm install --frozen-lockfile
pnpm build
```

前端验证重点：

1. Agent 编辑器是否正常打开与保存。
2. 聊天窗口流式输出是否正常。
3. LaTeX 展示是否仍存在闪烁或消失。
4. 对话导出功能是否可用。
5. 多语言构建是否存在缺失 key 或类型错误。

### 8.3 手工回归建议

同步完成后至少做以下手工回归：

1. 发起一次带流式输出的聊天。
2. 执行一次带知识文件的数据分析。
3. 打开 Agent 编辑器检查模型超时设置相关界面。
4. 执行一次对话导出。
5. 若启用了 wiki / docreader 相关能力，补一次对应链路验证。

## 9. 同步完成后的收口动作

验证通过后，先推送临时同步分支：

```bash
git push -u origin sync/dev-20260429-main
```

建议收口步骤：

1. 先在远端保留 `sync/dev-20260429-main`，作为可审阅版本。
2. 代码验证和人工检查通过后，再决定是否替换 `origin/dev`。
3. 如果需要用同步结果覆盖当前 `origin/dev`，建议使用 `--force-with-lease`，不要直接使用裸 `-f`。

示例：

```bash
git checkout sync/dev-20260429-main
git push --force-with-lease origin HEAD:dev
```

说明：

1. 这一步会改写远端 `dev` 历史。
2. 执行前必须确保备份分支和 tag 已存在。
3. 若团队内还有其他人共同使用 `origin/dev`，则应先通过评审或通知后再执行。

## 10. 回滚方案

### 10.1 本地回滚

如果本轮同步尚未推送，只需：

```bash
git rebase --abort
git checkout dev
git branch -D sync/dev-20260429-main
```

### 10.2 已推送同步分支但未覆盖 dev

```bash
git checkout dev
git branch -D sync/dev-20260429-main
git push origin --delete sync/dev-20260429-main
```

### 10.3 已覆盖 origin/dev

使用备份分支恢复：

```bash
git checkout backup/dev-20260429-before-sync
git push --force-with-lease origin HEAD:dev
```

## 11. 完成判定标准

满足以下条件，才算本轮主线同步完成：

1. `sync/dev-20260429-main` 已成功 rebase 到 `upstream/main` 之上。
2. `git rev-list --left-right --count HEAD...upstream/main` 中 behind 为 0。
3. 后端聚焦测试通过，或失败仅限已确认的环境依赖问题。
4. 前端构建通过。
5. 对话导出、流式聊天、data analysis 至少完成一次手工验证。
6. 已将同步结果推送到远端临时分支，或已经受控覆盖 `origin/dev`。

## 12. 执行摘要

针对当前仓库，最稳妥的同步路径不是重建 fork，也不是继续沿用旧的 `sync/dev-from-upstream`，而是：

1. 以当前 `dev` 为真实业务基线。
2. 先备份，再切出新的 `sync/dev-20260429-main`。
3. 在该分支上 rebase 到 `upstream/main`。
4. 重点处理 `frontend`、`internal/agent`、`internal/container` 的冲突。
5. 完成后端聚焦验证、前端构建验证和关键手工回归。
6. 最后再决定是否用同步结果覆盖 `origin/dev`。

这套流程与当前仓库实际状态一致，能兼顾安全性、可回滚性和后续交付效率。
