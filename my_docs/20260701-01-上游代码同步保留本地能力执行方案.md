# 20260701 上游代码同步保留本地能力执行方案

## 1. 文档目标

本文档基于 `my_docs/20260429-04-通用上游同步操作手册.md` 的通用原则，并结合当前 `dev` 与最新 `upstream/main` 的实际差异，形成本轮从本地 `0.6.1` 同步到上游 `0.6.3` 的专项执行方案。

本轮目标如下：

1. 将 `Tencent/WeKnora` 最新 `upstream/main` 同步到本地 `mrsteed/WeKnora` 的 `dev` 主开发线。
2. 明确保留本地已经存在的业务能力、服务能力和界面定制，尤其是上次同步后曾丢失的本地核心能力。
3. 以一次性 `merge` 为主线完成同步，避免在大跨度分叉场景下逐提交 `rebase` 反复冲突。
4. 在验证通过后，先推送临时同步分支，再受控落地到 `origin/dev`。

本文档是本轮同步前的执行方案和冲突裁决手册。当前只进行了只读对比、`fetch` 和 `merge-tree` 冲突预测，尚未在工作区执行真实 `git merge`。

## 3. 本轮采用 merge 的原因

通用手册中日常推荐可以使用 `rebase`，但本轮明确采用：

```bash
git merge upstream/main --no-edit
```

原因如下：

1. 当前上游新增 182 个提交，本地独有 66 个提交，差异文件 1180 个，逐提交 `rebase` 会在同一业务链路上反复解决冲突。
2. 本地独有能力不是简单补丁，而是长文档、数据库知识库、对话导出、组织权限、Agent 分享、管理员增强、界面定制等成套能力。
3. 上次同步后出现本地服务能力丢失，说明本轮需要集中做“上游修复吸收 + 本地能力保留”的人工裁决，而不是机械重放提交。
4. `merge` 可以保留本地历史，把本次同步作为一次可审计的整合提交，便于回滚和复盘。

本轮禁止使用以下方式：

```bash
git checkout --theirs .
git checkout --ours .
git reset --hard upstream/main
```

这些操作都会直接放大本地能力丢失风险。

## 8. 分批执行计划

### 8.1 执行前准备

执行前必须再次确认工作区干净：

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch
git remote -v
git branch -vv
git config --get branch.dev.remote
git config --get branch.dev.merge
git config --get remote.pushDefault
```

要求：

1. `dev` 仍跟踪 `origin/dev`。
2. `remote.pushDefault` 仍为 `origin`。
3. 工作区无未提交改动。

如果出现未提交改动，必须先提交或 stash，不能带脏工作区合并。

### 8.2 建立回退锚点

```bash
git checkout dev
git pull --ff-only origin dev
git branch backup/dev-20260701-before-sync
git tag sync-dev-20260701-pre
git fetch origin --prune
git fetch upstream --prune
```

建议执行后记录：

```bash
git rev-parse --short dev
git rev-parse --short upstream/main
git rev-list --left-right --count dev...upstream/main
```

### 8.3 创建临时同步分支并触发 merge

```bash
git checkout -b sync/dev-20260701-main dev
git merge upstream/main --no-edit
```

进入冲突后先记录现场：

```bash
git diff --name-only --diff-filter=U > my_docs/20260701-01-实际冲突文件清单.txt
git status --short > my_docs/20260701-01-合并冲突状态.txt
```

然后确认实际冲突是否与本文预测的 66 个文件一致。如果新增冲突，按模块追加到本文对应批次；如果少于 66 个，也不能跳过第 7 节自动合并复核。

### 10.1 完成冲突后提交

所有冲突解决并完成基础验证后：

```bash
git status
git commit
```

建议提交信息：

```text
merge: sync upstream main into dev 20260701
```

### 10.2 先推送临时同步分支

```bash
git push -u origin sync/dev-20260701-main
```

推送后先检查远端差异和 CI。如果 CI 或人工验证失败，不更新 `origin/dev`。

### 10.3 再受控落地 dev

验证通过后：

```bash
git checkout dev
git pull --ff-only origin dev
git merge sync/dev-20260701-main --ff-only
git push origin dev
```

如果 `dev` 在同步期间又有新提交进入，不能强行快进，应先回到同步分支吸收最新 `origin/dev`，重新验证后再落地。

## 11. 回滚方案

### 11.1 merge 未提交

```bash
git merge --abort
```

### 11.2 同步分支已提交但未落地 dev

保留失败现场：

```bash
git checkout dev
git branch sync/dev-20260701-main-failed sync/dev-20260701-main
```

重新开始：

```bash
git checkout dev
git checkout -b sync/dev-20260701-main-retry
git merge upstream/main --no-edit
```

### 11.3 已落地 origin/dev 后回滚

只有在确认 `origin/dev` 没有其他人新提交需要保留时，才能执行：

```bash
git checkout dev
git reset --hard backup/dev-20260701-before-sync
git push --force-with-lease origin dev
```

强制回滚前必须完成团队通知。
