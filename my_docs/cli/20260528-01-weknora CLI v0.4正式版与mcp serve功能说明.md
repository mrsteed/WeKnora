# weknora CLI v0.4 正式版与 mcp serve 功能应用说明

> 分析日期：2026-05-28
>
> 分析范围：当前仓库中的 `cli/` Go 模块、根目录发布说明、CLI 自身文档与 `weknora mcp serve` 的实现代码。
>
> 说明口径：根目录 `README_CN.md` 与 `CHANGELOG.md` 将 WeKnora v0.6.0 中的 `weknora` CLI v0.3 / v0.4 描述为 GA 能力；本文以当前代码实现为准整理可用命令、应用方式与 MCP 工具面。若实际安装的二进制版本不同，应先执行 `weknora version` 确认可用能力。

## 1. 一句话结论

`weknora` CLI 是 WeKnora 面向终端、脚本、CI 和 AI Agent 的官方命令行客户端。它把 WeKnora 的知识库、文档、检索、问答、会话、自定义 Agent、认证配置和诊断能力组织成清晰的命令树，并通过 `weknora mcp serve` 把一组经过筛选的只读能力暴露给支持 MCP 的 IDE 或 Agent 客户端。

重点可以理解为两层：

1. CLI 层：适合人、脚本和自动化任务直接执行，覆盖登录、管理知识库、上传/删除文档、检索、问答、查看会话、管理 Agent、健康检查、原始 API 调用等。
2. MCP 层：适合 Claude Code、Cursor、Gemini CLI 等 AI Agent 通过 JSON-RPC over stdio 调用 WeKnora，工具面只保留读取、检索、问答和 Agent 调用，不暴露上传、删除、编辑等破坏性操作。

## 2. CLI 的定位与适用对象

### 2.1 面向终端用户

用户可以用 CLI 快速完成：

- 登录 WeKnora 服务并管理多套连接配置。
- 列出、查看、创建、修改、置顶和删除知识库。
- 上传本地文件、从 URL 拉取文档、创建文本型文档、等待解析完成。
- 通过混合检索定位知识块或文档。
- 基于知识库发起 RAG 问答。
- 查看或删除历史会话。
- 创建、修改、检查和删除自定义 Agent。
- 运行 `doctor` 做连接、认证、版本和凭据存储诊断。

### 2.2 面向脚本与 CI

CLI 的默认输出面向机器读取，适合管道和自动化：

- 支持 `--format json`、`--format ndjson`、`--format text`。
- 支持 `--jq` 对 JSON 输出做过滤。
- 成功结果写到 stdout，错误写到 stderr，便于脚本区分数据流与错误流。
- 通过稳定的错误 code 和 exit code 表达失败类型。
- 非交互环境下遇到高风险写操作时返回 `input.confirmation_required` / exit 10，防止脚本误删资源。

### 2.3 面向 AI Agent

当前 CLI 明确考虑了 Agent 消费方式：

- 错误类型是结构化的，例如 `auth.unauthenticated`、`resource.not_found`、`network.error`、`input.confirmation_required`。
- 部分命令提供 Agent 可读帮助信息。
- 流式命令使用 NDJSON 事件流，便于 Agent 逐行解析。
- `mcp serve` 让 Agent 不需要每次 shell out，而是通过 MCP 工具直接调用 WeKnora。

## 3. 安装、构建与版本确认

### 3.1 从源码构建

CLI 位于独立 Go 模块 `cli/` 下。源码构建方式如下：

```bash
cd cli
go build -o weknora .
./weknora version
```

正式发布的二进制通常会通过构建参数注入版本、commit 与构建日期；源码开发环境中可能显示为 `dev`、`none`、`unknown`。

### 3.2 确认可用版本

建议任何脚本、文档或 Agent 集成前先执行：

```bash
weknora version --format json
```

用于确认当前 CLI 的版本、commit 和构建时间。

## 4. 全局行为与环境变量

### 4.1 全局 flag

所有子命令共享以下全局 flag：

| flag | 作用 |
|---|---|
| `--profile <name>` | 本次调用临时指定 profile，不写入配置文件。 |
| `--format text|json|ndjson` | 指定输出格式；默认是 JSON。 |
| `--jq <expr>` / `-q <expr>` | 对 JSON/NDJSON 输出执行 jq 过滤。 |
| `--log-level error|warn|info|debug` | 设置 SDK 调试日志级别。 |
| `-y` / `--yes` | 高风险写操作跳过确认，仅应在用户明确授权后使用。 |

### 4.2 常用环境变量

| 环境变量 | 作用 |
|---|---|
| `WEKNORA_PROFILE` | 默认 profile；低于显式 `--profile` 优先级。 |
| `WEKNORA_FORMAT` | 默认输出格式；显式 `--format` 会覆盖。 |
| `WEKNORA_KB_ID` | 作为需要知识库上下文命令的默认 KB ID。 |
| `WEKNORA_LOG_LEVEL` | 默认日志级别；显式 `--log-level` 会覆盖。 |
| `WEKNORA_AGENT_HELP=1` | `--help` 输出 Agent 友好的结构化帮助。 |

### 4.3 知识库上下文解析顺序

很多命令需要知识库 ID，例如 `chat`、`doc upload`、`doc create`、`doc fetch`。CLI 的 KB 解析顺序为：

1. 命令行 `--kb <id-or-name>`。
2. 环境变量 `WEKNORA_KB_ID`。
3. 当前目录或父目录中的 `.weknora/project.yaml`。
4. 都没有时返回 `local.kb_id_required`。

`--kb` 可以传 KB UUID，也可以传知识库名称；名称会通过 `kb list` 查询并解析成 ID。

## 5. 认证与多 profile

### 5.1 登录

交互式密码登录：

```bash
weknora auth login --host https://kb.example.com --name prod
```

CI 或 Agent 环境可通过 stdin 传入 API Key：

```bash
echo "sk-..." | weknora auth login --host https://kb.example.com --name prod --with-token
```

`auth login --with-token` 会先调用 `/auth/me` 验证 API Key，验证通过后才写入本地凭据，避免把错误 token 持久化。

### 5.2 凭据存储

CLI 优先使用系统 keyring：

- macOS：Keychain。
- Linux：libsecret。
- Windows：Wincred。

如果 keyring 不可用，会退化到 `$XDG_CONFIG_HOME/weknora/secrets/` 下的 0600 权限文件。

### 5.3 profile 与 auth 的区别

`profile` 管理连接目标，`auth` 管理认证凭据。

- `profile add` 只写 host 和可选 user，不写 token。
- `auth login` 会同时写 profile 与凭据。
- `profile use` 切换默认 profile。
- `auth logout` 删除本地凭据和 profile 条目，但不吊销服务端 token/API Key。

常用命令：

```bash
weknora profile add staging --host https://staging.example.com --user alice@example.com
weknora auth login --host https://prod.example.com --name prod
weknora profile list
weknora profile use prod
weknora auth status
weknora auth token
weknora auth refresh
weknora auth logout --name staging -y
```

## 7. 知识库命令：`weknora kb`

`kb` 是知识库生命周期管理入口。

### 7.1 列出知识库

```bash
weknora kb list
weknora kb list --pinned
weknora kb list --limit 10 --format json --jq '.data[] | {id, name}'
```

主要能力：

- 默认列出当前 profile 可见的知识库。
- `--pinned` 只显示置顶知识库。
- `--limit` 限制返回数量。
- JSON 输出包含 `id`、`name`、`description`、文档数量、分块数量、是否处理中等字段。

### 7.2 查看知识库详情

```bash
weknora kb view kb_abc
weknora kb view kb_abc --format json
```

适合确认：

- KB 名称与描述。
- 嵌入模型、摘要模型。
- 文档数、chunk 数、处理中数量。
- 是否临时、是否置顶。

### 7.3 创建知识库

```bash
weknora kb create "产品知识库" --description "面向客服和售前的产品文档"
weknora kb create "研发知识库" --embedding-model model_x --storage-provider local
```

可用参数：

| 参数 | 说明 |
|---|---|
| `<name>` | 知识库名称，必填位置参数。 |
| `--description` | 知识库描述。 |
| `--embedding-model` | 指定嵌入模型 ID；不传则服务端选择默认值。 |
| `--storage-provider` | 指定文档存储 provider；不传则服务端选择默认值。 |

### 7.4 编辑知识库

```bash
weknora kb edit kb_abc --name "新名称" -y
weknora kb edit kb_abc --description "更新后的描述" -y
```

注意：当前实现把 `kb edit` 视为高风险写操作，会触发确认协议。非交互或 JSON 模式下如果没有 `-y`，会返回 exit 10。自动化系统不要自动补 `-y`，必须先获得用户确认。

### 7.5 删除知识库

```bash
weknora kb delete kb_abc -y
```

这是不可逆操作，会触发 `input.confirmation_required` 确认协议。建议先执行：

```bash
weknora kb view kb_abc
weknora doc list --kb kb_abc --limit 5
```

确认目标无误后再删除。

### 7.6 置顶、取消置顶

```bash
weknora kb pin kb_abc
weknora kb unpin kb_abc
```

`pin` / `unpin` 是幂等设计：如果目标已经处于期望状态，会直接返回当前资源，不重复切换。

### 7.7 状态与深度检查

```bash
weknora kb status kb_abc
weknora kb check kb_abc
```

区别：

| 命令 | 成本 | 适用场景 |
|---|---|---|
| `kb status` | 约 1 次 HTTP | 快速查看可达性、文档数、处理中状态。 |
| `kb check` | 1+N 次 HTTP | 深度检查失败文档数量等聚合信息。 |

## 8. 文档命令：`weknora doc`

`doc` 是知识库内容管理入口，覆盖上传、文本创建、URL 导入、列表过滤、下载、删除和解析等待。

### 8.1 列出文档

```bash
weknora doc list --kb kb_abc
weknora doc list --kb kb_abc --status completed --limit 20
weknora doc list --kb kb_abc --keyword "方案" --file-type pdf
weknora doc list --kb kb_abc --start-time 2026-05-01T00:00:00Z --end-time 2026-05-28T23:59:59Z
```

主要参数：

| 参数 | 说明 |
|---|---|
| `--kb` | 知识库 ID 或名称；也可由环境变量或目录绑定提供。 |
| `--limit` / `-L` | 最大返回数量，默认 30。 |
| `--page-size` | 服务端分页大小，默认 50，范围 1..1000。 |
| `--all-pages` | 遍历所有服务端分页直到耗尽或达到 limit。 |
| `--status` | 过滤解析状态：`pending`、`processing`、`completed`、`failed`。 |
| `--keyword` | 服务端按标题或文件名做子串过滤。 |
| `--file-type` | 按扩展类型过滤，例如 `pdf`、`md`。 |
| `--source` | 按导入来源过滤。 |
| `--tag-id` | 按标签关联过滤。 |
| `--start-time` / `--end-time` | 按 `updated_at` RFC3339 时间范围过滤。 |

### 8.2 上传本地文件

```bash
weknora doc upload report.pdf --kb kb_abc
weknora doc upload notes.md --kb "产品知识库" --name "产品 FAQ.md"
weknora doc upload ./docs --recursive --glob '*.pdf' --kb kb_abc
weknora doc upload report.pdf --kb kb_abc --enable-multimodel --metadata team=alpha --metadata quarter=Q4
```

主要参数：

| 参数 | 说明 |
|---|---|
| `<file>` | 本地文件路径；`--recursive` 时可为目录。 |
| `--kb` | 知识库 ID 或名称。 |
| `--name` | 覆盖记录在服务端的文件名。 |
| `--recursive` | 将位置参数作为目录递归上传。 |
| `--glob` | 配合 `--recursive` 过滤文件名，默认 `*`。 |
| `--enable-multimodel` | 三态开关：不传为服务端默认；裸传或 true 为启用；false 为禁用。 |
| `--metadata key=value` | 可重复传入元数据；重复 key 后者覆盖前者。 |
| `--channel` | 覆盖导入渠道标签，默认 `api`。 |

### 8.3 从 URL 导入

```bash
weknora doc fetch https://example.com/whitepaper.pdf --kb kb_abc
weknora doc fetch https://example.com/no-ext --file-type pdf --title "Whitepaper" --kb kb_abc
weknora doc fetch https://example.com/report.pdf --kb kb_abc --enable-multimodel
```

服务端会抓取 URL 并导入到知识库。URL 有 `.pdf`、`.docx`、`.md`、`.txt` 等扩展时会进入文件下载模式；否则可通过 `--file-type` 或带扩展的 `--name` 明确文件类型。

### 8.4 从文本创建文档

```bash
weknora doc create --kb kb_abc --name "会议纪要" --text "# Meeting Notes\n\nAction items..."
weknora doc create --kb kb_abc --text "$(cat notes.md)" --name "Sprint Notes"
```

适合短文档、Agent 生成内容、结构化笔记，不需要先落地为本地文件。

### 8.5 查看和下载文档

```bash
weknora doc view doc_abc
weknora doc download doc_abc
weknora doc download doc_abc -O report.pdf
weknora doc download doc_abc --output -
weknora doc download doc_abc -O report.pdf --clobber
```

`doc download` 默认使用服务端建议文件名写入当前目录；已有文件不会覆盖，除非传 `--clobber`。`--output -` 会把原始字节写到 stdout，适合管道处理。

### 8.6 等待解析完成

```bash
weknora doc wait doc_abc
weknora doc wait doc_a doc_b doc_c --timeout 20m
weknora doc wait doc_a doc_b --format ndjson
```

退出码语义：

| exit code | 含义 |
|---|---|
| `0` | 全部解析完成。 |
| `1` | 至少一个文档解析失败。 |
| `124` | 达到 `--timeout`。 |
| `130` | 用户中断，例如 Ctrl-C。 |

多 ID 会并发轮询，最多 5 个并发。

### 8.7 删除文档

```bash
weknora doc delete doc_abc -y
weknora doc delete doc_a doc_b doc_c -y
weknora doc delete --all --kb kb_abc -y
```

删除是不可逆操作：

- 单 ID：一次确认，一次删除。
- 多 ID：一次确认，keep-going 语义；部分失败会汇总输出并返回非零。
- `--all --kb=<id>`：原子清空 KB 内全部文档，但保留知识库配置。

## 9. 分块调试命令：`weknora chunk`

`chunk` 面向 RAG 检索调试和管理，不是普通检索入口。普通检索应优先使用 `weknora search chunks`。

### 9.1 列出文档分块

```bash
weknora chunk list --doc doc_abc
weknora chunk list --doc doc_abc --limit 100 --all-pages
```

参数：

| 参数 | 说明 |
|---|---|
| `--doc` | 父文档 ID，必填。 |
| `--limit` / `-L` | 最大返回数量，范围 1..1000。 |
| `--page-size` | 服务端分页大小，范围 1..1000。 |
| `--all-pages` | 遍历所有分页直到耗尽或达到 limit。 |

### 9.2 查看单个 chunk

```bash
weknora chunk view chunk_abc
```

会输出 chunk 的 ID、序号、父文档、父知识库、类型、状态、时间、图片信息和完整内容。

### 9.3 删除 chunk

```bash
weknora chunk delete chunk_abc --doc doc_abc -y
weknora chunk delete chunk_a chunk_b --doc doc_abc -y
```

删除 chunk 会影响检索结果，属于高风险写操作，需要确认。

## 10. 检索命令：`weknora search`

### 10.1 混合检索 chunk

```bash
weknora search chunks "什么是 RRF" --kb kb_abc
weknora search chunks "embedding model" --kb kb_abc --limit 20
weknora search chunks "retry policy" --kb kb_abc --no-keyword
weknora search chunks "术语解释" --kb kb_abc --vector-threshold 0.3 --keyword-threshold 0.1
```

特点：

- 同时走向量检索和关键词检索，再进行融合排序。
- `--limit` 默认 8，范围 1..1000。
- `--no-vector` 可关闭向量通道。
- `--no-keyword` 可关闭关键词通道。
- 不能同时关闭两个通道。

### 10.2 搜索知识库

```bash
weknora search kb "marketing"
weknora search kb "team" --limit 5
```

该命令在客户端对可见知识库的名称和描述做大小写不敏感子串匹配，适合找 KB ID。

### 10.3 搜索文档

```bash
weknora search docs "Q3 forecast" --kb finance
weknora search docs "spec" --kb engineering --limit 5
weknora search docs "spec" --kb engineering --all-pages=false
```

该命令把 query 作为服务端 keyword filter，匹配文档标题或文件名，适合先定位文档 ID 再下载、查看或删除。

### 10.4 搜索会话

```bash
weknora search sessions "onboarding"
weknora search sessions "Q3 review" --limit 3
weknora search sessions "Q3 review" --all-pages=false
```

该命令分页获取会话后在客户端按 title/description 做大小写不敏感匹配。

## 11. RAG 问答：`weknora chat`

```bash
weknora chat "这个设计文档的核心结论是什么？" --kb kb_abc --format text
weknora chat "Summarise this design doc" --kb my-kb --format json
weknora chat "继续刚才的问题" --session sess_abc --kb kb_abc
```

行为特点：

- 面向知识库的 RAG 问答。
- 不传 `--session` 时自动创建新会话。
- 传 `--session` 时续接已有会话。
- `--format text` 是终端友好的实时流式输出。
- `--format json` / `--format ndjson` 输出 NDJSON 事件流：第一行是 CLI 注入的 `init` 事件，后续是 SDK 原始事件。

适用场景：

- 人在终端中快速问答：用 `--format text`。
- 脚本或 Agent 解析输出：用 `--format json` 或 `--format ndjson`。
- 需要连续追问：保存上一轮输出中的 session_id，下一轮传 `--session`。

## 12. 会话命令：`weknora session`

### 12.1 列出会话

```bash
weknora session list
weknora session list --since 7d
weknora session list --all-pages --limit 200
```

参数：

| 参数 | 说明 |
|---|---|
| `--limit` / `-L` | 最大返回数量，默认 30。 |
| `--page-size` | 服务端分页大小，范围 1..1000。 |
| `--all-pages` | 遍历所有服务端分页。 |
| `--since` | 只显示最近一段时间更新的会话，例如 `7d`、`24h`、`30m`。 |

### 12.2 查看会话

```bash
weknora session view sess_abc
weknora session view sess_abc --full
weknora session view sess_abc --full --limit 100
```

默认只显示会话元数据；`--full` 会加载消息历史，`--limit` 仅能与 `--full` 一起使用。

### 12.3 删除会话

```bash
weknora session delete sess_abc -y
weknora session delete sess_a sess_b -y
```

删除会话是高风险写操作，多 ID 删除采用 keep-going 语义。

### 12.4 通过 Agent 发起会话问答

```bash
weknora session ask --agent ag_abc "总结 Q3 销售情况" --format text
weknora session ask --agent ag_abc "多步分析任务" --format ndjson
weknora session ask --session sess_abc --agent ag_abc "继续上一轮" --format json
```

这是调用自定义 Agent 的主要入口，而不是 `weknora agent invoke`。`agent` 命令树主要负责 CRUD 和检查；真正执行 Agent 问答由 `session ask --agent` 完成。

## 13. 自定义 Agent 命令：`weknora agent`

### 13.1 列出和查看 Agent

```bash
weknora agent list
weknora agent list --limit 10
weknora agent view ag_abc
```

适合发现 Agent ID、查看 Agent 配置、确认系统提示词、模型、KB 作用域和运行模式。

### 13.2 创建 Agent

```bash
weknora agent create "客服助手" --model model_abc --description "回答客服常见问题" --kb kb_abc --kb-selection-mode selected
weknora agent create "推理助手" --model model_abc --agent-mode smart-reasoning --temperature 0.7
weknora agent create "模板" --generate-skeleton
weknora agent create "复制版" --from ag_base --model model_new
weknora agent create "配置版" --model model_abc --config-file ./agent.yaml
```

主要参数：

| 参数 | 说明 |
|---|---|
| `<name>` | Agent 名称；`--generate-skeleton` 时可不传。 |
| `--model` | LLM 模型 ID；创建真实 Agent 时必填。 |
| `--description` | 描述。 |
| `--system-prompt` | 直接传系统提示词。 |
| `--system-prompt-file` | 从文件或 stdin 读取系统提示词；与 `--system-prompt` 互斥。 |
| `--agent-mode` | `quick-answer` 或 `smart-reasoning`。 |
| `--kb` | 可重复传入，关联知识库 ID。 |
| `--kb-selection-mode` | `all`、`selected`、`none`。 |
| `--rerank-model` | Rerank 模型 ID。 |
| `--temperature` | 生成温度，范围 0.0..2.0。 |
| `--from` | 从已有 Agent 复制配置后再覆盖。 |
| `--config-file` | 读取完整 AgentConfig YAML/JSON。 |
| `--generate-skeleton` | 输出空白 AgentConfig YAML 后退出。 |

### 13.3 编辑 Agent

```bash
weknora agent edit ag_abc --name "新名称" -y
weknora agent edit ag_abc --add-kb kb_new --remove-kb kb_old -y
weknora agent edit ag_abc --config-file ./tuned.yaml --temperature 0.9 -y
```

`agent edit` 可能改变工具权限、KB 范围和模型行为，当前实现将其视为高风险写操作，需要确认。

### 13.4 检查 Agent

```bash
weknora agent status ag_abc
weknora agent check ag_abc
```

区别：

| 命令 | 含义 |
|---|---|
| `agent status` | 快速检查 Agent 可达性和模型配置。 |
| `agent check` | 进一步探测 Agent 作用域内的 KB 可达性。 |

### 13.5 删除 Agent

```bash
weknora agent delete ag_abc -y
```

删除 Agent 是不可逆操作，会触发确认协议。

## 14. 本地目录绑定：`link` / `unlink`

```bash
weknora link --kb kb_abc
weknora link --kb "产品知识库"
weknora unlink
```

`link` 会在当前目录写入 `.weknora/project.yaml`，之后该目录及子目录中的 `doc upload`、`doc create`、`doc fetch`、`chat` 等命令可以自动解析默认 KB。

应用建议：

- 一个项目目录长期对应一个知识库时，使用 `weknora link --kb ...`。
- CI 中建议显式传 `--kb` 或 `WEKNORA_KB_ID`，避免受工作目录影响。
- Agent 不应在用户未要求时自动执行 `link`，因为它会修改用户工作目录。

## 15. 原始 API 透传：`weknora api`

```bash
weknora api /api/v1/knowledge-bases
printf '{"name":"foo"}' | weknora api /api/v1/knowledge-bases --input -
weknora api /api/v1/knowledge-bases/kb_xxx -X DELETE -y
weknora api /api/v1/knowledge-bases --paginate --format json
```

能力：

- 默认 GET。
- 传 `--input` 时默认提升为 POST。
- `-X/--method` 可指定任意 HTTP 方法。
- 自动注入认证、租户和 request-id 相关 header。
- 默认直接输出响应 body。
- `--format json` 输出包含 status、headers、body 的 JSON 结构。
- `--paginate` 支持基于 `page` / `page_size` 的 offset 分页聚合。

安全边界：

- `api` 是逃生舱，适合调用 CLI 尚未包装的高级接口。
- 当前实现对 DELETE 触发确认协议；PUT/PATCH/POST 是否危险由调用者自己判断。
- 在 Agent 或 CI 中使用 `api` 时，应优先选择只读 GET；写操作必须由人确认具体 path 和 body。

## 16. 自诊断：`weknora doctor`

```bash
weknora doctor
weknora doctor --format json
weknora doctor --offline
weknora doctor --no-cache
```

检查项：

1. Base URL 可达性。
2. 当前认证是否可用。
3. 服务端与 CLI 版本是否兼容。
4. 本地凭据存储是否可用。

输出状态：

| 状态 | 含义 |
|---|---|
| `ok` | 检查通过。 |
| `warn` | 有警告但不阻塞。 |
| `fail` | 检查失败。 |
| `skip` | 离线模式或依赖前置检查失败。 |

建议把 `doctor` 作为所有自动化流程的前置检查：

```bash

```

## 17. MCP Server：`weknora mcp serve`

### 17.1 定位

`weknora mcp serve` 把 WeKnora CLI 的一组只读能力作为 MCP 工具暴露给 Agent 客户端。它使用 JSON-RPC 2.0 over stdio：

- stdin/stdout 是协议数据通道。
- stderr 用于日志和错误提示。
- 启动时会立即解析 SDK client 和认证配置。
- 如果没有可用 profile 或凭据，会在 MCP 握手前失败，返回清晰的 `auth.unauthenticated`。

### 17.2 客户端配置示例

通用 MCP 配置示例：

```json
{
  "mcpServers": {
    "weknora": {
      "command": "weknora",
      "args": ["mcp", "serve"]
    }
  }
}
```

如果需要指定 profile：

```json
{
  "mcpServers": {
    "weknora-prod": {
      "command": "weknora",
      "args": ["--profile", "prod", "mcp", "serve"]
    }
  }
}
```

### 17.3 为什么 MCP 工具面只读

当前 MCP Server 是手工维护的 curated 工具面，而不是自动把所有 cobra 命令暴露出去。原因是 Agent 一旦持有用户 token，就具备用户权限；在服务端细粒度 token scope 完整落地前，CLI 通过不暴露上传、删除、编辑等 mutation 工具降低误操作风险。

因此 MCP 中没有：

- `kb create/edit/delete`。
- `doc upload/create/fetch/delete`。
- `agent create/edit/delete`。
- `chunk delete`。
- `api` 透传。
- `auth` / `profile` / `link` 等本地状态变更命令。

需要写操作时，Agent 应回到 CLI 命令路径，并让用户明确确认 `-y`。

### 17.4 MCP 工具清单

当前代码中 `registerTools` 注册了 10 个工具：

| 工具 | 输入 | 输出重点 | 适合场景 |
|---|---|---|---|
| `kb_list` | 无 | `items[]`，包含 KB 基本信息。 | 发现可用知识库，选择 `kb_id`。 |
| `kb_view` | `kb_id` | KB 完整记录，含模型、chunk 配置、数量等。 | 在回答前确认 KB 配置和规模。 |
| `doc_list` | `kb_id`、分页与过滤条件 | 文档列表、page、page_size、total。 | 找到某个文档，检查解析状态。 |
| `doc_view` | `doc_id` | 文档记录，含文件名、标题、类型、状态、大小等。 | 查看文档元数据。 |
| `doc_download` | `doc_id` | 文档内容、文件名、字节数、是否 base64。 | 小文件全文读取。 |
| `chunk_list` | `doc_id`、`limit` | 文档 chunks、total、是否截断。 | 调试分块，检查原文切片。 |
| `search_chunks` | `kb_id`、`query`、`limit`、阈值 | 混合检索结果。 | 在回答前找证据片段。 |
| `chat` | `kb_id`、`query`、可选 `session_id` | 完整回答、引用、思考、session_id。 | 基于知识库生成 RAG 答案。 |
| `agent_list` | 无 | 自定义 Agent 列表。 | 发现可调用 Agent。 |
| `agent_invoke` | `agent_id`、`query`、可选 `session_id` | 完整回答、引用、工具事件、思考。 | 让服务端 Agent 执行复杂任务。 |

### 17.5 各 MCP 工具详细说明

#### kb_list

功能：列出当前租户/当前 profile 可见的知识库。

输入：无。

输出：

```json
{
  "items": [
    {
      "id": "kb_abc",
      "name": "产品知识库",
      "description": "...",
      "knowledge_count": 12,
      "is_pinned": true,
      "updated_at": "..."
    }
  ]
}
```

典型用法：Agent 首先调用 `kb_list`，让用户或模型选择后续工具需要的 `kb_id`。

#### kb_view

功能：按 `kb_id` 获取知识库详情。

输入：

```json
{"kb_id":"kb_abc"}
```

输出包含 KB 的完整记录，例如模型 ID、分块配置、文档数、chunk 数、处理中数量等。

#### doc_list

功能：列出知识库内文档，支持分页和过滤。

输入字段：

| 字段 | 说明 |
|---|---|
| `kb_id` | 必填，知识库 ID。 |
| `page` | 可选，默认 1。 |
| `page_size` | 可选，默认 20，最大 1000。 |
| `status` | 可选，解析状态。 |
| `keyword` | 可选，标题/文件名关键字。 |
| `file_type` | 可选，文件类型。 |
| `source` | 可选，导入来源。 |
| `tag_id` | 可选，标签 ID。 |
| `start_time` / `end_time` | 可选，RFC3339 更新时间范围。 |

输出包含 `items`、`page`、`page_size`、`total`。

#### doc_view

功能：按 `doc_id` 获取文档元数据。

输入：

```json
{"doc_id":"doc_abc"}
```

适合确认文档是否解析完成、文件名是否匹配、文档类型是否正确。

#### doc_download

功能：下载文档原始内容。

输入：

```json
{"doc_id":"doc_abc"}
```

输出：

```json
{
  "doc_id": "doc_abc",
  "file_name": "report.md",
  "bytes": 1024,
  "content": "...",
  "is_base64": false
}
```

限制：

- 单次最大 1 MiB。
- 如果检测到二进制内容，会把 `content` 以 base64 编码返回，并设置 `is_base64=true`。
- 大文档应优先使用 `search_chunks` 找相关片段，而不是全文下载。

#### chunk_list

功能：列出某个文档的 chunks。

输入：

```json
{"doc_id":"doc_abc","limit":50}
```

输出：

```json
{
  "chunks": [],
  "total": 120,
  "truncated_at_limit": true
}
```

限制：

- 默认 `limit=50`。
- 最大 `limit=1000`。
- 如果 `truncated_at_limit=true`，说明只返回了前一部分；需要更完整分页时应回到 CLI 的 `weknora chunk list --doc ... --all-pages`。

#### search_chunks

功能：对知识库做混合检索。

输入：

```json
{
  "kb_id": "kb_abc",
  "query": "什么是 RRF",
  "limit": 10,
  "vector_threshold": 0,
  "keyword_threshold": 0
}
```

限制：

- `query` 不能为空。
- `limit` 默认 10，最大 1000。

适用方式：先用 `search_chunks` 找证据，再调用 `chat` 生成最终答案；或者直接把检索结果作为 Agent 的引用依据。

#### chat

功能：基于指定 KB 发起 RAG 问答。

输入：

```json
{
  "kb_id": "kb_abc",
  "query": "总结这批文档的关键风险",
  "session_id": "sess_abc"
}
```

行为：

- `session_id` 可选；不传时自动创建会话。
- 底层是 SSE 流，但 MCP `tools/call` 没有标准部分结果流，因此 MCP 工具会在服务端累积完整结果后一次性返回。

输出重点：

- `answer`：最终回答。
- `references`：引用片段。
- `thinking`：思考模型返回的 reasoning 内容，可能为空。
- `session_id`：后续追问可复用。
- `assistant_message_id`：助手消息 ID，可能为空。

#### agent_list

功能：列出当前租户可见的自定义 Agent。

输入：无。

输出：`items[]`，包含 Agent ID、名称、描述、是否内置等。

#### agent_invoke

功能：调用服务端自定义 Agent 执行问题。

输入：

```json
{
  "agent_id": "ag_abc",
  "query": "分析这些文档并给出行动建议",
  "session_id": "sess_abc"
}
```

行为：

- `session_id` 可选；不传时自动创建会话。
- 底层 Agent SSE 事件会被累积为一次 MCP 返回。

输出重点：

- `answer`：最终回答。
- `references`：知识引用。
- `tool_events`：Agent 工具调用轨迹。
- `thinking`：思考内容，可能为空。
- `session_id`、`agent_id`、`query`。

## 18. 输出与错误契约

### 18.1 成功输出

当前代码中的 JSON 输出通过统一 envelope 输出，常见形态为：

```json
{
  "ok": true,
  "data": {},
  "meta": {},
  "profile": "prod"
}
```

列表命令通常在 `data` 中放数组，在 `meta.count` 中放数量。mutation 成功可能返回被删除 ID、更新后的资源或批处理结果。

`--format ndjson` 则按行输出 JSON 对象，适合流式或列表逐项处理。

### 18.2 错误输出

错误写到 stderr。文本模式常见形态：

```text
auth.unauthenticated: no current profile configured
hint: run `weknora auth login`
```

JSON 模式下错误也会带结构化 detail，MCP 工具错误则通过 `StructuredContent` 返回同类结构。

### 18.3 退出码

| exit code | 含义 | 建议处理 |
|---|---|---|
| `0` | 成功 | 继续流程。 |
| `1` | 本地错误、操作失败、未分类错误等 | 读取 stderr，决定是否人工处理。 |
| `2` | flag 或参数错误 | 检查 `weknora <cmd> --help`。 |
| `3` | 认证/权限错误 | 登录、刷新 token 或切换权限。 |
| `4` | 资源不存在 | 检查 ID。 |
| `5` | 输入错误 | 调整参数。 |
| `6` | 服务端限流 | 等待后重试。 |
| `7` | 服务端或网络错误 | 退避重试，必要时执行 `doctor`。 |
| `10` | 高风险写操作缺少确认 | 让用户确认后再加 `-y`。 |
| `124` | CLI 轮询超时 | 增大 `--timeout` 或检查任务状态。 |
| `130` | 用户中断 | 停止流程。 |

## 19. 推荐应用流程

### 19.1 第一次接入 WeKnora

```bash
weknora auth login --host https://kb.example.com --name prod
weknora doctor
weknora kb list
```

如果是 CI：

```bash
echo "$WEKNORA_API_KEY" | weknora auth login --host "$WEKNORA_HOST" --name ci --with-token
weknora doctor --format json
```

### 19.2 给项目目录绑定默认知识库

```bash
weknora kb list
weknora link --kb kb_abc
weknora doc upload README.md
weknora chat "这个项目的核心模块有哪些？" --format text
```

适合开发团队把某个代码仓库、文档目录或项目资料夹长期绑定到一个知识库。

### 19.3 批量导入资料并等待解析

```bash
weknora doc upload ./docs --recursive --glob '*.pdf' --kb kb_abc --metadata source=manual
weknora doc list --kb kb_abc --status processing --format json
weknora doc wait doc_a doc_b doc_c --timeout 30m
```

实际批量场景中，建议上传后从 JSON 输出中收集文档 ID，再调用 `doc wait`。

### 19.4 调试检索效果

```bash
weknora search chunks "关键问题" --kb kb_abc --limit 10 --format json
weknora doc view doc_abc
weknora chunk list --doc doc_abc --limit 50
weknora chunk view chunk_abc
```

调试思路：

1. 用 `search chunks` 看检索是否命中正确文档。
2. 用 `doc view` 看解析状态是否完成。
3. 用 `chunk list` / `chunk view` 看切块是否合理。
4. 必要时回到服务端调整 KB 分块配置或模型配置。

### 19.5 使用自定义 Agent

```bash
weknora agent list
weknora agent view ag_abc
weknora session ask --agent ag_abc "请分析这些知识库中的风险点" --format text
```

若需要创建或编辑 Agent：

```bash
weknora agent create "项目分析助手" --model model_abc --agent-mode smart-reasoning --kb kb_abc --kb-selection-mode selected
weknora agent edit ag_abc --temperature 0.7 -y
weknora agent check ag_abc
```

### 19.6 给 IDE / Agent 配置 MCP

1. 先在终端完成登录：

```bash
weknora auth login --host https://kb.example.com --name prod
weknora doctor
```

2. 在 MCP 客户端配置：

```json
{
  "mcpServers": {
    "weknora": {
      "command": "weknora",
      "args": ["--profile", "prod", "mcp", "serve"]
    }
  }
}
```

3. 在 Agent 中推荐调用顺序：

```text
kb_list -> kb_view -> doc_list/search_chunks -> chat 或 agent_invoke
```

4. 如需删除、上传、编辑，回到 CLI 命令并要求用户确认。

## 20. 安全与权限建议

1. 不要把 `auth token` 输出粘贴到聊天窗口或日志系统。
2. API Key 是长效凭据，泄露后需要在服务端或身份系统中轮换。
3. Agent 或脚本遇到 exit 10 时，不得自动添加 `-y` 重试。
4. `weknora api` 是高级逃生舱，写操作风险比普通命令更高。
5. MCP 工具面虽然只读，但仍使用用户凭据读取知识库和会话数据，应只配置给可信 Agent 客户端。
6. 多租户环境中先用 `auth status`、`kb list` 确认当前 profile 和租户上下文。
7. 删除知识库、文档、chunk、Agent、会话前，先用 `view/list/status/check` 确认目标。

## 21. 当前实现中的边界与注意事项

1. `mcp serve` 当前是 stdio-only；代码注释中提到 HTTP / SSE transport 可能后续加入。
2. MCP 的 `chat` 和 `agent_invoke` 不是实时流式返回，而是服务端累积完整 SSE 后一次性返回。
3. MCP 的 `doc_download` 有 1 MiB 上限，大文档应使用 `search_chunks` 或 CLI 下载。
4. MCP 只暴露 curated 只读工具，不覆盖完整 CLI 命令树。
5. `api` 可以访问 CLI 未封装的深层接口，但需要调用者理解服务端 API 结构。
6. `doc wait` 是 wait-all 语义，不是 fail-fast；需要 fail-fast 时用 shell 串联多个单 ID 调用。
7. `profile add` 不写凭据；新增 profile 后还需要 `auth login --name <profile>`。
8. `link` 写入当前工作目录，应避免在不确定目录中执行。
9. 当前 CLI 代码中的 JSON 输出统一经过 envelope；如果外部旧文档提到 bare JSON，应以实际安装版本和 `weknora <cmd> --format json` 结果为准。
