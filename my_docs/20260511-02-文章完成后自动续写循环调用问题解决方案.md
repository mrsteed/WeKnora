# 文章完成后自动续写循环调用问题解决方案

## 1. 问题结论

本次问题不是单一的前端自动续写误触发，而是 Agent 流式收口、最终答案降级合成、文档级完成判定三条链路叠加导致的循环调用。

结合日志可以确认，Agent 第 6 轮已经向前端输出了大量 `answer` chunk，但流在尾部遇到 `context deadline exceeded`。当前后端在统计可用内容时只看 `content_len` 和 `tool_calls`，而本轮 `answer` chunk 很可能带有 `source` 元数据，被 `streamLLMToEventBus()` 排除在 `result.Content` 之外，最终日志呈现为：

```text
Completed: chunks=3838, content_len=0, tool_calls=0,
type_distribution=map[answer:3586 error:1 thinking:250 tool_call:1]
Iteration-6 failed: LLM stream error: context deadline exceeded
LLM failed but have 5 steps with 7 tool calls — attempting final answer synthesis
```

这说明系统实际上已经收到并转发了 3586 个 answer 类型 chunk，却仍按“本轮没有可用答案”进入降级合成。降级合成又重新向模型发起一次 `final_answer` LLM 请求，且该请求使用的是通用最终答案提示词，不稳定继承“文档续写只输出新增内容、完成后输出 `<!-- document_complete -->`”的协议，因此容易重新生成标题和前文内容。前端收到的 artifact 仍被标记为 `continuing`，于是 `maybeContinueDocumentAutomatically()` 判断 `auto_continue_next=true`，继续发起下一轮自动续写，形成“文章应该结束但仍循环调用”的现象。

## 2. 现有代码链路

### 2.1 Agent 流式阶段

相关文件：`internal/agent/think.go`

`streamLLMToEventBus()` 会遍历模型流：

1. `ResponseTypeAnswer` 通过回调发送到事件总线，前端能看到内容。
2. 但如果 chunk 带有 `Data["source"]`，当前逻辑会认为它是提取型内容，不追加到 `result.Content`。
3. 当尾部出现 `ResponseTypeError` 且 `result.Content == "" && len(result.ToolCalls) == 0` 时，函数返回错误。

本次日志里的 `answer:3586` 与 `content_len=0` 正好命中这个缺陷：可见答案已经流出，但内部结果对象没有记录“已有可用最终答案”。

### 2.2 超时降级合成阶段

相关文件：`internal/agent/think.go`、`internal/agent/finalize.go`

`callLLMWithRetry()` 在 LLM 超时且已有历史工具结果时，会调用 `streamFinalAnswerToEventBus()` 做最终答案合成。该逻辑本意是避免用户什么都拿不到，但在本次场景中有两个副作用：

1. 当前轮已经输出过答案，降级合成相当于第二次生成同一篇文章，容易造成重复标题、重复章节或从头开始。
2. `streamFinalAnswerToEventBus()` 构造的是通用 final prompt，没有显式携带文档续写完成协议，也没有强制“不要从头重写上一份文档”。

### 2.3 文档完成状态阶段

相关文件：`internal/application/service/chat_document_artifact.go`、`internal/handler/session/qa.go`、`frontend/src/views/chat/index.vue`

现有文档自动续写已经有完成标记协议：续写上下文要求模型在整篇文档完成时输出 `<!-- document_complete -->`。后端检测到该标记后会把 artifact 的 `DocumentGenerationStatus` 设置为 `completed`，并在 complete payload 中写入：

```json
{
  "document_generation_status": "completed",
  "auto_continue_next": false,
  "auto_continue_reason": "document_complete_marker"
}
```

前端 `maybeContinueDocumentAutomatically()` 看到 `document_generation_status === 'completed'` 或 `auto_continue_next === false` 后停止续写。

当前缺口是：完成判定强依赖模型主动输出标记。一旦降级合成没有输出标记，或者模型在文章实际结束时没有输出标记，`NormalizeChatDocumentGenerationStatus()` 会默认返回 `continuing`，前端就会继续自动发起下一轮。

## 3. 根因拆解

### 3.1 根因一：流式 answer 已输出但未进入可用内容统计

日志里的核心矛盾是：`answer` 类型 chunk 数量很大，但 `content_len=0`。这会导致 `streamLLMToEventBus()` 认为“没有可用内容”，从而把尾部超时升级为整个 LLM 调用失败。

这类场景下，系统不应该直接进入“从工具结果重新合成完整答案”，而应该优先复用已经流出的 answer 内容，并补发终止事件，保证本轮请求收口。

### 3.2 根因二：降级合成缺少文档续写协议

`streamFinalAnswerToEventBus()` 的 final prompt 只要求“基于工具结果生成完整答案、引用来源、结构化组织”。对普通问答是合理的，但对文档续写不够：

1. 没有告诉模型当前是 `continue_document`。
2. 没有带入上一版文档的尾部窗口、目录或已生成章节。
3. 没有要求完成时输出 `<!-- document_complete -->`。
4. 没有禁止重新输出标题和已完成章节。

因此降级合成很容易把“续写剩余内容”误理解为“重新输出完整技术方案”。截图中 `7.2 实施方能力保障` 后又出现 `## 北海电厂二期智慧电厂项目 — 智慧运行技术方案` 和 `一、智慧运行总体概述`，就是典型的标题回退/从头重写信号。

### 3.3 根因三：缺少后端确定性防循环判定

当前前端已有以下保护：

1. `document_generation_status=completed` 停止。
2. `auto_continue_next=false` 停止。
3. artifact 未变化停止。
4. 连续低增长停止。
5. 达到最大轮次停止。

但这些保护偏结果侧。如果模型继续输出重复内容，后端仍会注册新 artifact，snapshot 字数也可能增长，低增长保护不会立即命中。后端需要在 artifact 注册阶段增加确定性的文档级防循环信号，把“重复开头、章节编号回退、续写 delta 近似重复、已经出现收尾章节后继续生成”等情况标记为 `completed` 或 `needs_review`，并返回 `auto_continue_next=false`。

## 4. 修复目标

本次修复应达到以下目标：

1. LLM 尾部超时但已流出 answer 时，不再盲目二次合成完整答案。
2. 必须降级合成时，合成提示词继承文档续写协议，不能从头重写。
3. 文档实际结束但缺少完成标记时，后端能通过结构化规则停止自动续写。
4. 所有停止原因都通过 `document_generation_status`、`auto_continue_next`、`auto_continue_reason` 返回前端。
5. 前端只负责执行后端明确的文档级状态，不再猜测文章是否完成。

## 5. 推荐修复方案

### 5.1 修复流式 answer 统计与尾部超时收口

修改文件：`internal/agent/think.go`

为 `streamLLMResult` 增加可见答案统计字段：

```go
type streamLLMResult struct {
    Content              string
    ReasoningContent     string
    VisibleAnswerContent string
    AnswerStreamed       bool
    FinalAnswerStreamed  bool
    SawDone              bool
    ToolCalls            []types.LLMToolCall
    Usage                *types.TokenUsage
    FinishReason         string
    StreamError          string
}
```

处理 `ResponseTypeAnswer` 时，不论是否带 `source`，都追加到 `VisibleAnswerContent`：

```go
if chunk.ResponseType == types.ResponseTypeAnswer && chunk.Content != "" {
    result.AnswerStreamed = true
    result.VisibleAnswerContent += chunk.Content
    if source == "final_answer_tool" {
        result.FinalAnswerStreamed = true
    }
}
if chunk.Done {
    result.SawDone = true
}
```

尾部错误判断改为：

```go
if result.StreamError != "" {
    hasUsableAnswer := strings.TrimSpace(result.Content) != "" || strings.TrimSpace(result.VisibleAnswerContent) != ""
    if hasUsableAnswer || len(result.ToolCalls) > 0 {
        result.FinishReason = "stream_error_after_answer"
        return result, nil
    }
    return result, fmt.Errorf("LLM stream error: %s", result.StreamError)
}
```

这样本次日志中的 `answer:3586` 不会再被当成“无可用内容”。

### 5.2 在流式答案尾部超时时补发 Done 事件

修改文件：`internal/agent/think.go`

`streamThinkingToEventBus()` 拿到 `llmResult.StreamError != "" && llmResult.AnswerStreamed` 时，应补发一个 `Done=true` 的 final answer 事件，保证前端和持久化链路收口：

```go
if llmResult.StreamError != "" && llmResult.AnswerStreamed && !llmResult.SawDone {
    e.eventBus.Emit(ctx, event.Event{
        ID:        answerID,
        Type:      event.EventAgentFinalAnswer,
        SessionID: sessionID,
        Data: event.AgentFinalAnswerData{
            Content:          "",
            Done:             true,
            CompletionStatus: types.MessageCompletionStatusPartial,
            FinishReason:     "stream_error_after_answer",
            IsPartial:        true,
            AllowIndexing:    false,
            AllowComplete:    false,
            FailureReason:    "stream_error_after_answer",
        },
    })
}
```

同时返回 `ChatResponse`：

```go
resp.Content = firstNonEmpty(llmResult.Content, llmResult.VisibleAnswerContent)
resp.AnswerStreamed = llmResult.AnswerStreamed
resp.FinalAnswerStreamed = llmResult.FinalAnswerStreamed
resp.FinishReason = normalizeFinishReason(llmResult.FinishReason)
```

`callLLMWithRetry()` 收到这种响应后应进入 `observe` 分析，而不是执行“已有工具结果则重新合成最终答案”的错误分支。

### 5.3 禁止已流出最终答案后再次降级合成

修改文件：`internal/agent/think.go`

在 `callLLMWithRetry()` 的错误处理前增加守卫：

1. 如果当前轮已经通过 `final_answer_tool` 输出过答案，则不再调用 `streamFinalAnswerToEventBus()`。
2. 如果当前轮已有可见答案但尾部超时，只允许标记 partial/fallback stop，不再二次生成。

建议新增错误类型或结果标志，而不是靠字符串判断：

```go
type streamPartialAnswerError struct {
    Reason string
}
```

若短期不引入新类型，也可先通过 `llmResult.StreamError` 与 `VisibleAnswerContent` 组合完成最小改动。

### 5.4 让最终答案合成继承文档协议

修改文件：`internal/agent/finalize.go`、`internal/handler/session/qa.go`

为 Agent state 或 final answer synthesis 参数补充文档上下文：

```go
type AgentDocumentContext struct {
    Intent             string
    Operation          string
    OutputMode         string
    QuotedContext      string
    AutoContinue       bool
    AutoContinueRound  int
    CompletionMarker   string
}
```

在 `streamFinalAnswerToEventBus()` 构造 final prompt 时，如果是 `continue_document` 或自动续写，追加强约束：

```text
当前任务是同一文档的续写，不是重新生成全文。
只输出上一版文档之后缺失的新增正文。
不要重复文档标题、一级标题、已完成章节或上一版末尾已经存在的段落。
如果你判断整篇文档已经完整输出，请在正文最后单独输出 <!-- document_complete -->。
如果没有新增内容可输出，只输出 <!-- document_complete -->。
```

同时把上一版 artifact 的目录、尾部窗口和原始用户目标带入合成上下文。这样即使必须走降级合成，也会输出“剩余尾部”或完成标记，而不是重启整篇文章。

### 5.5 后端增加文档级防循环判定

修改文件：`internal/application/service/chat_document_artifact.go`

在 `RegisterFromAssistantMessage()` 或 `buildChatDocumentSnapshotResult()` 后增加 `inferDocumentGenerationStatus()`：

```go
type DocumentGenerationDecision struct {
    Status string
    Reason string
}
```

判定优先级建议如下：

1. 显式完成标记：命中 `<!-- document_complete -->`，状态为 `completed`，原因为 `document_complete_marker`。
2. 空 delta 或纯提示语：自动续写轮输出为空、只有“已完成/无需继续”等语义，状态为 `completed`。
3. 标题回退：续写 delta 重新出现与根文档相同的 H1/H2 标题，且 base snapshot 已包含该标题，状态为 `needs_review` 或 `blocked`，`auto_continue_next=false`。
4. 章节编号回退：base 已经到 `7.2`，delta 又从 `一、`、`1.`、`## 总体概述` 等早期章节开始，状态为 `needs_review`，停止自动续写。
5. 重复率过高：delta 与 base 尾部或全文相似度超过阈值，状态为 `needs_review`，停止自动续写。
6. 收尾章节后继续：base 末尾章节标题命中 `实施方能力保障|保障措施|结束语|总结|附录`，且 delta 没有新章节推进，状态为 `completed` 或 `needs_review`。
7. 默认：状态为 `continuing`。

对本次截图，可由第 3 和第 4 条直接拦截：在 `7.2 实施方能力保障` 后再次出现项目标题和 `一、智慧运行总体概述`，属于明显的标题回退/章节编号回退，应停止自动续写，并提示用户检查最终文档。

建议返回原因枚举：

```go
const (
    ChatDocumentAutoContinueReasonMarker        = "document_complete_marker"
    ChatDocumentAutoContinueReasonDuplicateHead = "duplicate_document_head"
    ChatDocumentAutoContinueReasonSectionReset  = "section_number_reset"
    ChatDocumentAutoContinueReasonLowNovelty    = "low_novelty_delta"
    ChatDocumentAutoContinueReasonTerminalTail  = "terminal_section_tail"
)
```

`addChatDocumentGenerationPayload()` 根据状态和原因输出：

```json
{
  "document_generation_status": "needs_review",
  "auto_continue_next": false,
  "auto_continue_reason": "section_number_reset"
}
```

前端收到后停止，并在 banner 中显示“自动续写已暂停：检测到章节回退，请检查最终文档”。

### 5.6 降低前端最大循环风险

修改文件：`frontend/src/views/chat/index.vue`

当前 `AUTO_CONTINUE_MAX_ROUNDS = 50` 对投标方案类长文档过高。建议第一阶段降到 12 或 15，并把后端 stop reason 作为主要停止依据。

前端补充两个轻量保护：

1. 如果 `payload.auto_continue_reason` 存在且不是空字符串，直接停止并展示该原因。
2. 如果完整快照尾部出现 `DOCUMENT_COMPLETE_MARKER` 的残留，也立即停止并本地清理 marker。

示例：

```ts
if (payload.auto_continue_next === false || payload.document_generation_status !== 'continuing') {
    stopAutoContinue(formatAutoContinueStopReason(payload.auto_continue_reason));
    return;
}
```

### 7.1 流式结果收口

```go
// internal/agent/think.go
if chunk.ResponseType == types.ResponseTypeAnswer && chunk.Content != "" {
    result.AnswerStreamed = true
    result.VisibleAnswerContent += chunk.Content
    source := sourceFromChunk(chunk)
```

### 7.2 文档防循环判定

```go
func inferDocumentGenerationStatus(base, artifact *types.ChatDocumentArtifact, delta string, explicitStatus string) (status, reason string) {
    if types.NormalizeChatDocumentGenerationStatus(explicitStatus) == types.ChatDocumentGenerationStatusCompleted {
        return types.ChatDocumentGenerationStatusCompleted, "document_complete_marker"
    }
    if base == nil || strings.TrimSpace(delta) == "" {
        return types.ChatDocumentGenerationStatusContinuing, ""
    }
    if repeatsDocumentHead(base.ContentSnapshot, delta) {
        return types.ChatDocumentGenerationStatusNeedsReview, "duplicate_document_head"
    }
    if sectionNumberResets(base.ContentSnapshot, delta) {
        return types.ChatDocumentGenerationStatusNeedsReview, "section_number_reset"
    }
    if deltaNoveltyTooLow(base.ContentSnapshot, delta) {
        return types.ChatDocumentGenerationStatusNeedsReview, "low_novelty_delta"
    }
    if terminalTailReached(base.ContentSnapshot) && !hasMeaningfulForwardProgress(delta) {
        return types.ChatDocumentGenerationStatusCompleted, "terminal_section_tail"
    }
    return types.ChatDocumentGenerationStatusContinuing, ""
}
```

## 10. 建议实施顺序

建议按以下顺序落地：

1. 先修 `streamLLMToEventBus()` 的可见 answer 统计和尾部超时收口，这是本次日志的直接触发点。
2. 再修 `streamFinalAnswerToEventBus()` 的文档续写协议，防止必须降级时重新生成全文。
3. 然后补 artifact 注册阶段的重复开头、章节回退、低新颖度判定，形成确定性防循环闸门。
4. 最后调整前端最大轮次和停止原因展示。

完成后，即使模型没有严格输出完成标记，系统也能通过后端结构化规则停止自动续写；即使模型在尾部超时，系统也会优先收口已有 answer，而不是重新调用一次模型生成重复文章。
