# 长文档输出不完整原因审查报告

## 1. 审查背景

针对如下长文档生成日志，需分析“输出不完整”的直接原因、控制链路和非根因项，并给出整改建议。

本次现象对应的任务为“北海电厂二期智慧电厂详细设计方案”全文生成。在运行到第 7 章“智能安全监控应急中心设计”时，系统输出终止，最终消息以 partial 状态持久化。

## 2. 结论摘要

本次“输出不完整”的直接原因，不是前端流丢失，也不是 artifact 落库失败，而是第 7 章单章生成阶段触发了 LLM 调用超时，系统按既定降级逻辑将本轮结果标记为 partial，并停止自动续写。

更准确地说，日志虽然显示本章预算超时为 180 秒，但实际执行在约 120 秒时结束，说明真正生效的超时不是日志里打印的 budget section timeout，而是更小的 agent LLMCallTimeout。代码中长文档章节调用使用“预算超时”和“agent 全局 LLM 超时”二者的较小值，因此本轮极大概率被 120 秒全局超时提前截断。

## 3. 关键证据链

### 3.1 第 7 章进入单章生成

日志明确显示当前正在生成第 7 章，且本章属于全文 10 章中的第 7 章：

- section_current: 7
- section_total: 10
- section_title: 智能安全监控应急中心设计
- budget_section_tokens: 4096
- evidence_top_k: 8
- section_timeout_seconds: 180

这说明问题发生在“章节级生成”阶段，而不是大纲规划阶段，也不是全文收尾阶段。

### 3.2 模型并非完全无响应，而是首个可见正文较晚

日志显示：

- First reasoning_content 到达时间约 1189ms
- First delta.Content 到达时间约 22879ms

这表明模型调用已经启动，并且先输出了 reasoning，再在约 22.9 秒后才开始输出正文内容。也就是说，生成不是启动失败，而是在有正文流出的前提下，后续章节内容未能在有效超时窗口内完成。

### 3.3 终态直接指向章节超时

完成摘要日志中给出了明确的结束原因：

- completion_status: partial
- finish_reason: section_generation_timeout
- failure_reason: llm_timeout
- document_generation_status: continuing
- auto_continue_next: false

这组字段已经足以判定：本轮输出不完整的直接原因是章节生成超时，而不是正常 stop、长度截断或流式传输异常。

### 3.4 运行时反馈进一步证明系统识别到了“超时型过载”

日志中的运行时反馈为：

- reasons=[section_top_k_down_timeout section_timeout_up_timeout section_batch_limit_down_timeout]
- next_section_top_k=7
- next_timeout=210
- recommended_section_limit=2

这说明系统在本轮结束后，已经根据“超时”这一运行时信号给出下一轮收缩/补偿策略：

- 减少检索证据数量
- 增加章节超时预算
- 缩小单轮处理章节数

因此，系统内部对这次失败的归因本身也是“timeout pressure”，而不是数据库、SSE 或 Markdown 修复导致的截断。

## 4. 根因分析

### 4.1 直接根因：第 7 章章节流在有效 LLM deadline 内未完成

长文档章节流的消费逻辑位于 internal/application/service/session_agent_qa.go 中的 consumeFullDocumentSectionStream。该函数在 ctx.Done 且错误为 context.DeadlineExceeded 时，会直接返回：

- completionStatus = partial
- finishReason = section_generation_timeout
- failureReason = llm_timeout
- documentGenerationState = continuing

这与本次日志完全一致，说明问题就是由章节调用 deadline 触发。

### 4.2 深层原因一：日志打印的是预算超时，不一定是最终生效超时

章节调用超时通过 effectiveDocumentGenerationCallTimeout 计算，逻辑是：

- 先取 budget.SectionCallTimeoutSeconds
- 再判断 agentConfig.LLMCallTimeout
- 如果 agentConfig.LLMCallTimeout 更小，则最终以更小值为准

也就是说，日志中的 section_timeout_seconds: 180 只是预算值，不代表真实执行 deadline 一定是 180 秒。

而本次从第 7 章 section config 打印时间 09:28:28.522 到 timeout 总结时间 09:30:28.886，仅约 120.4 秒。这与 180 秒不符，却与系统默认单次 LLM 超时 120 秒高度吻合。

结合代码可进一步推断：

- 若自定义 agent 未配置 llm_call_timeout，则会回退到全局 agent 配置
- agent 默认单次 LLM 调用超时为 120 秒
- 环境变量 WEKNORA_AGENT_LLM_TIMEOUT 也可能覆盖该值

因此，本次最可能的真实触发链路是：

1. 本章预算超时为 180 秒。
2. agent 或全局配置中的 LLMCallTimeout 为 120 秒。
3. effective timeout 取二者较小值，即约 120 秒。
4. 第 7 章在该窗口内未生成完成。
5. 系统落为 partial，并以 section_generation_timeout 收口。

### 4.3 深层原因二：第 7 章提示词和证据负载较重，首个正文较晚压缩了可用生成时间

从请求体可见，本轮属于“结合本地知识库生成客户投标式长文档”的重负载场景，提示词具有以下特点：

- 全量文档标题、完整目录、已完成章节摘要同时注入
- 明确要求逐系统展开、详细设计、区分事实/推断/待补充项
- evidence_top_k = 8，本地知识证据负载不低
- 当前章节本身还包含 7.1 至 7.5 多个子系统小节

再结合日志可见：

- reasoning 很快开始
- 但首个可见正文要到 22.9 秒后才到达

这意味着真正留给正文连续输出的时间，只有约 97 秒左右。对于章节体量较大的投标型设计文档，这个窗口偏紧，尤其在模型开启深度推理或输出风格偏长时更容易超时。

### 4.4 深层原因三：超时后自动续写被主动抑制，导致用户看到“停在半截”

日志显示：

- document_generation_status: continuing
- auto_continue_next: false

这不是矛盾，而是系统设计使然。自动续写判断要求“允许继续且 failure_reason 为空”。一旦 failure_reason 为 llm_timeout，系统即便知道文档理论上还可继续，也不会自动发起下一轮续写。

因此，用户侧看到的结果是：

- 本轮消息已保留部分输出
- 系统承认全文尚未完成
- 但不会自动接着跑下一轮

这正是“输出不完整且停住”的直接产品表现。

## 5. 非根因排除

### 5.1 不是 SSE 流提前关闭导致正文丢失

日志末尾有一条：

- Stream completed for session=..., message=...; closing immediately without waiting for title event

这条日志对应的是 SSE 收尾逻辑。代码说明“session title generation is best-effort metadata and must not delay closing the answer stream”，即 complete 事件到达后，流会立即关闭，而不会等待标题事件。

这意味着：

- 流关闭发生在 complete 之后
- 它只是不等待 title 元数据
- 不会反向导致正文在生成中的 answer chunk 被截断

所以，这条日志是结果，不是根因。

### 5.2 不是 artifact 查询 record not found 导致输出丢失

日志中出现：

- SELECT * FROM chat_document_artifacts ... record not found
- 随后又出现 Registered chat document artifact ... operation: create

仓储代码对 GetArtifactBySourceMessageID 的语义就是：查不到返回 nil，不视为错误。随后系统执行首次 artifact 注册，这是正常的“先查旧版本、查不到则新建”流程。

因此，该条 record not found 是首次创建 artifact 的正常前置现象，不是导致文档不完整的异常。

### 5.3 不是 request_max_completion_tokens = 0 直接导致被截断

本次 section config 日志中 request_max_completion_tokens: 0 容易引发误判，但从代码调用点看，长文档章节日志这里传入的就是 0，占位意义强于控制意义。

真正参与章节调用控制的是：

- budget.SectionMaxCompletionTokens
- budget.SectionCallTimeoutSeconds
- agentConfig.LLMCallTimeout

因此，不能把 request_max_completion_tokens = 0 当成这次截断的直接原因。

### 5.4 不是 Markdown 质量修复导致正文被吞掉

日志中质量问题仅出现：

- quality_issues: [markdown_heading_normalized]

这说明系统对 Markdown 标题做过规范化修正，但完成态仍然是 section_generation_timeout / llm_timeout。质量修复是附加处理，不是主因。它最多解释“标题被规范化”，不能解释“第 7 章后续内容没有继续生成完”。

## 6. 代码级控制链路说明

本次问题涉及的核心控制链路如下：

1. runKnowledgeGroundedFullDocumentGenerationPath 进入按章生成流程。
2. 生成到第 7 章时，打印 section config。
3. 调用 withDocumentGenerationCallTimeout 创建章节调用上下文。
4. effectiveDocumentGenerationCallTimeout 取 budget 超时与 agentConfig.LLMCallTimeout 的较小值。
5. consumeFullDocumentSectionStream 消费模型流。
6. 若 ctx 因 deadline exceeded 结束，则返回 partial + section_generation_timeout + llm_timeout。
7. 运行时反馈识别为 timeout，给出下轮预算调整建议。
8. emitFullDocumentCompletion 输出 partial 完成事件。
9. completeAssistantMessage 以 partial 状态持久化消息，并设置 allow_complete=false、allow_indexing=false。
10. 后续注册 artifact，SSE 在收到 complete 后正常关闭。

该链路与本次日志逐条对应，闭环完整。

## 7. 为什么是“第 7 章”更容易出问题

从章节定义看，第 7 章“智能安全监控应急中心设计”包含如下小节：

- 7.1 集控室升级为智慧指挥大厅
- 7.2 一屏统揽可视化展示
- 7.3 事件监控与应急研判
- 7.4 智能讲解机器人部署
- 7.5 通信扩声与辅助系统

这类章节有两个特征：

- 子系统多，天然容易写长
- 既要写展示体系，也要写事件联动、应急研判、机器人与辅控系统，内容跨度大

而且此时提示词还携带前 6 章已完成摘要，模型需要同时保证与既有章节连续一致。对 deepseek-v4-pro 这类带 reasoning 的模型来说，章节体量大、结构要求细、证据输入多，会明显抬高单章耗时。

因此，第 7 章不是随机出问题，而是刚好落在“章节复杂度与有效超时窗口冲突”的高风险点上。

## 8. 整改建议

### 8.1 优先级 P0：补齐“实际生效超时”的日志观测

当前日志只打印：

- section_timeout_seconds（预算值）

但没有打印：

- effective_section_timeout_seconds（本轮调用真实采用的 deadline）
- agent_llm_call_timeout_seconds（当前 agent 配置值）
- global_agent_llm_call_timeout_seconds（全局回退值）
- effective_timeout_source（最终超时来源）

这会直接误导排障，因为仅看到 180 秒预算值，并不能判断最终是：

- 直接使用了 budget.SectionCallTimeoutSeconds
- 被 agent 自定义 llm_call_timeout 压低
- 被全局 Agent 配置回退值压低
- 或命中了默认 120 秒超时

建议在 section config 日志中同时打印：

- budget_section_timeout_seconds
- agent_llm_call_timeout_seconds
- global_agent_llm_call_timeout_seconds
- effective_section_timeout_seconds
- effective_timeout_source（budget / agent_config / global_agent / default）

如果还要进一步降低复盘成本，建议补充一条 timeout resolved 日志，在真正创建章节调用上下文前输出最终决议，例如：

- budget_section_timeout_seconds=180
- agent_llm_call_timeout_seconds=0
- global_agent_llm_call_timeout_seconds=120
- effective_section_timeout_seconds=120
- effective_timeout_source=global_agent

这样排障时就不需要再通过时间差反推“180 秒预算为何 120 秒就终止”，而是可以直接从日志确认真实生效超时及其来源。

### 8.2 优先级 P0：提高长文档路径的单章 LLM 超时上限

若当前 agent 或全局 LLMCallTimeout 为 120 秒，建议针对长文档任务至少提升到 210 至 300 秒，或者确保其不低于运行时协商出来的章节预算超时。

否则会持续出现以下现象：

- 预算层认为本章还有时间
- 实际 LLM deadline 已先到达
- 运行时反馈不断建议加 timeout
- 但当轮依旧只能在较小超时上限被截断

### 8.3 优先级 P1：在 llm_timeout 场景允许基于 artifact 自动续写下一轮

当前逻辑中 failure_reason 一旦非空，auto_continue_next 就会被抑制。对长文档而言，这会把“可恢复的超时”直接暴露成“用户看到的半截结果”。

建议区分两类失败：

- 不可恢复失败：如本地知识为空、权限问题、参数错误
- 可恢复失败：如 llm_timeout、section_batch_limit、瞬时流截断

对可恢复失败，若已有 artifact 和 generation run，则可继续自动续写下一章或下一轮，而不是直接停住。

### 8.4 优先级 P1：在超时高风险章节提前收缩负载

当前系统是在 timeout 发生后，才通过 runtime feedback 下降 top_k 和 section limit。建议前置处理：

- 对后半程复杂章节先把 evidence_top_k 从 8 调到 6 或 7
- 对章节体量大的投标型文档，默认按 1 至 2 章一轮推进
- 对章节内 H3 数较多的章节，启用“章内分段续写”而不是整章一次完成

这样更符合长文档重负载场景。

### 8.5 优先级 P2：增强 timeout 后的用户可见提示

当前用户只会看到内容停在半截，排障上不够直观。建议在 complete 事件或页面提示中增加明确描述，例如：

- 当前已输出第 7 章部分内容
- 本轮因模型响应超时暂停
- 可点击“继续生成”从第 7 章后续或第 8 章继续

这样能把“系统失败感”降为“系统分轮继续生成”的可理解行为。

## 9. 最终结论

本次长文档输出不完整的根因已经明确：

- 发生点在第 7 章单章生成阶段
- 直接原因是 LLM 调用超时
- 完成态由系统按 partial 正常持久化
- artifact 注册和 SSE 关闭都发生在超时之后，不是根因

其中最关键的隐含问题在于：日志显示的章节预算超时为 180 秒，但真实生效 deadline 大概率只有 120 秒。这个“观测值与真实执行值不一致”的问题，才是本次排障最容易误判的核心。

如果只做表面判断，容易以为模型在 180 秒预算内无故提前结束；但从代码和时间差联合判断，更合理的结论是：较小的 agent 级 LLM 超时覆盖了章节预算，导致第 7 章在约 120 秒时被提前截断。

因此，本问题的整改重点不是修数据库、不是修 artifact、也不是修 SSE，而是统一长文档路径中的“预算超时”和“实际调用超时”，并在 llm_timeout 后提供可恢复续写能力。
