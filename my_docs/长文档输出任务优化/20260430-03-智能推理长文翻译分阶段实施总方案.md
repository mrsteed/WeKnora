# 智能推理长文翻译分阶段实施总方案

## 1. 方案定位

基于当前审查结论，最合理可行的落地方式不是一次性大改，而是采用三阶段渐进实施：

1. P0 先止血，优先解决错误完成和错误入库。
2. P1 再修协议和状态机，把 partial、failed、cancelled 从隐式行为提升为显式语义。
3. P2 最后重构长文翻译任务流，把“全文翻译并输出 markdown 文件”从在线问答链路中剥离出来。

这个顺序的核心原因有三点：

1. 当前最严重的问题不是“翻译不完整”，而是“半成品被当成完整结果并写回知识库”。
2. 当前前后端已经对 IsCompleted 和 complete 事件形成了耦合，直接重构任务流风险过高。
3. 长文翻译异步化是正确方向，但它依赖前面两阶段先把状态、协议和入库语义稳定下来。

## 2. 分阶段目标

### 2.1 P0 止损修复

目标：阻止错误完成和脏数据继续扩散。

本阶段必须达成的结果：

1. 长文被截断时，不再被视为 completed。
2. partial、failed、cancelled 结果不再写入聊天历史知识库。
3. 现有短问答链路不被破坏。

本阶段刻意不做的事：

1. 不修改 messages 表结构。
2. 不引入复杂前端状态面板。
3. 不新增长文翻译异步任务系统。

### 2.2 P1 协议与状态机修复

目标：统一后端完成语义、前端展示语义和流式事件协议。

本阶段必须达成的结果：

1. answer、thought、complete 三类事件边界稳定。
2. 消息具备 completion_status、finish_reason、is_partial 等显式状态。
3. 前端能够正确展示 partial、failed、cancelled，而不是一律显示“已完成”。

本阶段刻意不做的事：

1. 不提供全文翻译后台任务调度。
2. 不直接承诺输出工作区 markdown 文件。

### 2.3 P2 长文翻译任务流重构

目标：为全文翻译提供独立能力边界和文件交付能力。

本阶段必须达成的结果：

1. 长文翻译不再依赖单次 SSE 文本输出。
2. 翻译过程可以断点推进、持久化中间结果、最终生成 markdown 文件。
3. Chat 只作为任务发起和结果查看入口，不再承担整篇文档正文承载。

## 3. 推荐实施节奏

### 3.1 第一周

实施 P0。

推荐原因：

1. 改动集中在 internal/agent、internal/handler/session、internal/application/service。
2. 不需要数据库迁移。
3. 可以最快阻断线上脏数据继续生成。

### 3.2 第二周到第三周

实施 P1。

推荐原因：

1. P0 稳定后，才能安全补齐 completion_status 和 finish_reason 等显式状态。
2. 前端要基于稳定的后端协议做 partial 展示，否则容易反复返工。

### 3.3 独立迭代

实施 P2。

推荐原因：

1. P2 不是简单修 bug，而是能力重构。
2. 会引入异步任务、产物管理、进度展示、失败恢复等新模块。
3. 与当前普通问答链路的开发节奏应该分离。

## 4. 阶段间依赖关系

### 4.1 P0 对 P1 的前置条件

P1 依赖 P0 先完成以下收敛：

1. analyzeResponse 能正确识别 length。
2. completeAssistantMessage 不再无条件 completed。
3. IndexMessageToKB 具备 allowIndexing 语义。

如果这些点不先稳定，P1 的 completion_status 很容易只是在表面加字段，实际语义仍然不可靠。

### 4.2 P1 对 P2 的前置条件

P2 依赖 P1 提供以下稳定基础：

1. 前端和后端都能正确表达 partial、failed、cancelled。
2. 答案流和思考流已经解耦。
3. 完成事件只负责生命周期，不再补正文。

如果 P1 不先完成，P2 生成任务流后仍可能在结果回传时复用旧协议，问题会被重新带回来。

## 5. 总体交付清单

### 5.1 P0 交付物

1. 后端 completion metadata 的最小透传。
2. finish_reason=length 识别和 partial 收尾。
3. 禁止 partial/failed 结果入库。
4. 针对长文翻译场景的基础回归测试。

### 5.2 P1 交付物

1. event_data、消息持久化模型、SSE 事件协议统一。
2. front chat 流式状态机重构。
3. partial/truncated/failed/cancelled UI 展示。
4. answer/thought/complete 事件职责边界固定。

### 5.3 P2 交付物

1. 长文翻译后台任务服务。
2. 任务状态、批次结果和产物文件模型。
3. markdown 文件生成与下载能力。
4. 前端任务发起、进度查询和结果查看界面。

## 6. 风险控制策略

### 6.1 P0 风险控制

1. 不改表结构，避免迁移风险。
2. 以最小增量扩展事件 metadata 和 handler 逻辑。
3. 先做严格门禁，再考虑优化用户体验。

### 6.2 P1 风险控制

1. 前后端协议同时改，但通过兼容字段保留旧逻辑兜底。
2. complete 事件只减职责，不再增加隐式行为。
3. 对历史消息回放路径做专项回归。

### 6.3 P2 风险控制

1. 与现有 chat QA 链路隔离部署。
2. 新任务流先只支持“发起任务 + 下载文件”，再逐步丰富交互。
3. 中间结果持久化优先于实时渲染，避免长任务失败后全部丢失。

## 7. 决策建议

如果只能选择一个最优先动作，建议先实施 P0。

如果可以安排连续两个迭代，建议按以下顺序：

1. 第一迭代完成 P0 全量。
2. 第二迭代完成 P1 全量。
3. P2 单独立项评审后进入开发。

这是当前代码现状下收益最高、风险最低、返工最少的实施路径。
