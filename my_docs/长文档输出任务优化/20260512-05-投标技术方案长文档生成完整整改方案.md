# 投标技术方案长文档生成完整整改方案

## 1. 背景与目标

本方案合并 [my_docs/20260512-03-投标技术方案长文档生成实现审查报告.md](my_docs/20260512-03-投标技术方案长文档生成实现审查报告.md) 与 [my_docs/20260512-04-长文档规划展示与正文排版优化修正方案.md](my_docs/20260512-04-长文档规划展示与正文排版优化修正方案.md) 中需要整改的内容，统一解决投标技术方案长文档生成中的主链路质量、页面过程可见性、正文排版质量、历史索引健壮性和运维观测问题。

本次日志 [my_docs/test/13.md](my_docs/test/13.md) 证明主链路已经基本跑通：请求被识别为 `full_document`，智能体配置中的知识库被纳入有效 KB 范围，后端路由到 `knowledge_grounded_full_document`，并按“本地知识检索 -> 大纲规划 -> 逐章节检索 -> 逐章节流式生成 -> 完成落库 -> 注册文档产物”的受控路径完成。最终状态为 `completed/stop`，文档 artifact 注册成功。

但从“提交甲方的投标技术方案”交付标准看，当前实现仍存在四类必须合并处理的问题：

1. 页面没有稳定展示完整规划大纲，用户只能看到折叠后的步骤摘要。
2. 章节正文 Markdown 排版不规范，出现 `###3.1...` 这类标题不渲染、标题正文粘连的问题。
3. 长文档完成后的聊天历史索引把 50249 字符 Q&A 作为单 chunk 送入 embedding，触发输入长度超过 8192 的错误。
4. 生成耗时接近 9 分钟，虽有进度心跳，但前端需要更明确地展示规划、章节进度与收尾状态，避免用户误判卡死。

## 2. 当前实现结论

### 2.1 已符合设计的部分

| 设计要求 | 当前实现表现 | 结论 |
| --- | --- | --- |
| `full_document` 且有效 KB 范围非空时走知识库长文档路径 | 日志显示 `document_output_mode=full_document`，请求体 `knowledge_base_ids=null`，但智能体解析出 1 个配置知识库，并路由到 `knowledge_grounded_full_document` | 符合 |
| 本地知识检索由服务端确定性执行 | 规划前检索 3 组查询，每个章节生成前继续按章节检索，并把 `local_knowledge_context` 注入 Prompt | 符合 |
| 首轮完整文档覆盖全部大纲章节 | 大纲包含 8 个 H2，`fullDocumentSectionsForInitialRun` 返回全部章节，最终以 completed 收尾 | 符合 |
| 完成后落库并注册文档产物 | 日志显示 completion persisted、`agent_steps_count=70`、artifact 注册成功 | 符合 |
| 完成判定不只依赖模型 stop | 后端已有 H1/H2 结构完整性校验和 handler 层兜底 | 基本符合 |

### 2.2 不应判定完全达标的部分

1. 规划过程没有直观展示给用户。后端仅发送“已识别 N 个章节”等进度摘要，前端已有大纲解析组件但默认藏在折叠树中，且大纲内容可能被 `Replace=true` 的进度事件覆盖。
2. 正文内部 H3/H4 格式质量不足。Prompt 只写了“可以使用 ###”，没有强制标题空格、独占一行、标题后空行、段落和列表排版。
3. 聊天历史索引对长文档不安全。DocReader 输出单 chunk，embedding 输入长度 50249 超过 8192，后台日志虽继续显示保存成功，但索引实际质量不可控。
4. 进度体验仍偏弱。生成耗时约 8 分 58 秒，如果页面只显示折叠摘要，会削弱长任务的可信度。
5. 尾部 tracing exporter 出现 `context deadline exceeded`，不影响主功能，但会影响链路追踪完整性。

## 3. 总体整改思路

本次整改不应只修某一个局部，而应按以下原则整体收口：

1. **规划产物和实时进度分离**：完整大纲是用户可见产物，必须稳定展示；章节检索、生成、收尾是实时状态，可继续使用可替换进度事件。
2. **模型约束、服务端兜底、前端兼容三层处理排版**：Prompt 从源头减少错误；后端保证 artifact snapshot 质量；前端保证流式和历史页面能正确渲染。
3. **长文档历史索引走摘要或安全切分**：不要把 5 万字符 Q&A 单段送入 embedding；索引目标应服务于历史检索，而不是复制完整正文。
4. **完成判定扩展到交付质量**：H1/H2 完整只是结构底线，H3/H4 标题格式、明显正文粘连、过长标题也应进入轻量质量校验。
5. **前端体验围绕长任务设计**：用户应能看到“规划已完成、正在生成第几章、剩余几章、最终文档正在收尾”。

## 4. 后端整改方案

### 4.1 保持现有知识库长文档路由，不做大范围改造

现有 `AgentQA` 路由已经能覆盖“请求体无知识库 ID，但智能体配置了知识库”的场景，应保留当前判断顺序：

1. 知识库续写；
2. 文档编辑；
3. 知识库完整文档生成；
4. 纯模型完整文档生成；
5. 通用 Agent 或普通问答。

本次后端整改重点不是重写路由，而是在知识库完整文档路径和纯模型完整文档路径中补齐可见规划、格式质量、历史索引和观测能力。

### 4.2 新增完整大纲格式化函数

新增 `formatFullDocumentOutlineMarkdown(outline dedicatedFullDocumentOutline) string`，统一输出严格 H1/H2：

```go
func formatFullDocumentOutlineMarkdown(outline dedicatedFullDocumentOutline) string {
    var builder strings.Builder
    title := strings.TrimSpace(outline.Title)
    if title != "" {
        builder.WriteString("# ")
        builder.WriteString(title)
    }
    for _, section := range outline.Sections {
        section = strings.TrimSpace(section)
        if section == "" {
            continue
        }
        if builder.Len() > 0 {
            builder.WriteString("\n")
        }
        builder.WriteString("## ")
        builder.WriteString(section)
    }
    return strings.TrimSpace(builder.String())
}
```

要求：

- 第一行只能是一个 H1；
- 后续每个章节独占一行 H2；
- 不输出解释性文本；
- 不把大纲写入最终正文，避免重复。

### 4.3 新增独立大纲展示事件

在 `fullDocumentProgressReporter` 中新增 `PublishOutline`，不要复用当前进度事件的 `progressEventID`。

```go
func (r *fullDocumentProgressReporter) PublishOutline(outline dedicatedFullDocumentOutline) {
    if r == nil || r.eventBus == nil {
        return
    }
    outlineMarkdown := formatFullDocumentOutlineMarkdown(outline)
    if strings.TrimSpace(outlineMarkdown) == "" {
        return
    }
    _ = r.eventBus.Emit(r.ctx, event.Event{
        ID:        generateEventID("document-outline"),
        Type:      event.EventAgentThought,
        SessionID: r.sessionID,
        Data: event.AgentThoughtData{
            Content:   outlineMarkdown,
            Iteration: 0,
            Done:      true,
            Replace:   false,
            Synthetic: true,
            Stage:     "planning",
        },
    })
}
```

关键语义：

- `Replace=false`：保证大纲不被后续“第 N/M 章生成中”覆盖。
- `Done=true`：表示大纲是已确定的规划产物，不再追加。
- `Stage=planning`：便于前端识别为规划阶段产物。
- 独立 event id：保证其生命周期与实时进度分离。

### 4.4 在两个完整文档路径发布大纲

在大纲生成并校验通过之后调用：

```go
outline, err := generateValidatedFullDocumentOutline(...)
if err != nil {
    return err
}
progress.PublishOutline(outline)
```

调用点：

1. `runKnowledgeGroundedFullDocumentGenerationPath`：大纲校验通过后、创建 generation run 或章节循环前。
2. `runDedicatedFullDocumentGenerationPath`：大纲校验通过后、进入章节循环前。

这样用户在章节正文开始前就可以看到完整规划。

### 4.5 在完成事件和 generation run extra 中保留结构化大纲

为支持历史回放，`emitFullDocumentCompletion` 的 `Extra` 或 generation run extra 中应稳定包含：

```json
{
  "outline": {
    "title": "北海电厂二期智慧电厂项目技术方案",
    "sections": ["项目背景与建设目标", "总体技术架构", "数据湖与基础算力平台", "智慧运行系统", "智慧安防系统", "智能安全监控应急中心", "AR眼镜智能作业", "实施与运维保障"]
  },
  "document_generation_status": "completed"
}
```

即使历史消息缺失 thought stream，前端也能从完成事件或 artifact 元数据恢复规划卡片。

### 4.6 增强章节正文 Prompt

抽取共享函数，供 `buildDedicatedFullDocumentSectionMessages` 与 `buildKnowledgeGroundedFullDocumentSectionMessages` 复用：

```go
func fullDocumentSectionMarkdownStyleInstructions() string {
    return strings.Join([]string{
        "Markdown layout requirements:",
        "1. Use polished proposal-style writing suitable for customer-facing technical bids.",
        "2. Do not repeat the current H2 section heading; it has already been emitted.",
        "3. Use H3/H4 headings only when they improve readability.",
        "4. Every markdown heading must be on its own line and must contain exactly one space after heading markers, for example: ### 3.1 全域数据湖建设.",
        "5. Never write malformed headings such as ###3.1标题, ### 标题正文粘连, or headings and body text on the same line.",
        "6. Add one blank line after each heading before body paragraphs.",
        "7. Keep paragraphs concise; prefer 2-4 sentences per paragraph.",
        "8. Use bullet lists for capability groups, implementation steps, constraints, and value points.",
        "9. Use bold lead-in labels such as **建设内容：** or **技术价值：** when it helps scanning.",
        "10. Do not output HTML, code fences, JSON, hidden reasoning, or raw prompt text.",
    }, " ")
}
```

并同时写入 system prompt 和 user prompt 末尾，降低模型忽略概率。

推荐中文 Prompt 片段：

```text
## 排版与行文要求
- 本节正文用于提交甲方的投标技术方案，请使用正式、清晰、专业的技术方案文风。
- 当前 H2 章节标题已经由系统输出，请不要重复输出 H1/H2 标题。
- 如需拆分小节，只能使用 H3/H4 标题；标题必须独占一行，且 `###` 或 `####` 后必须有一个空格。
- 正确示例：`### 3.1 全域数据湖建设`。错误示例：`###3.1全域数据湖建设`、`### 3.1 全域数据湖建设正文内容...`。
- 每个标题后必须空一行再写正文。
- 每个自然段控制在 2-4 句，避免大段堆叠。
- 对平台能力、建设内容、技术路径、业务价值、保障措施等内容，优先使用项目符号拆分。
- 可使用加粗标签提升可读性，例如：`**建设内容：**`、`**技术价值：**`、`**实施要点：**`。
- 不要输出 HTML、代码块、JSON、提示词说明或隐藏推理。
```

### 4.7 增加 Markdown 标题归一化与质量检测

新增轻量归一化函数，用于最终 artifact snapshot 入库前：

```go
var markdownHeadingWithoutSpaceRE = regexp.MustCompile(`(?m)^(#{1,6})([^#\s].*)$`)

func normalizeGeneratedMarkdownHeadings(content string) string {
    normalized := strings.ReplaceAll(content, "\r\n", "\n")
    normalized = strings.ReplaceAll(normalized, "\r", "\n")
    return markdownHeadingWithoutSpaceRE.ReplaceAllString(normalized, "$1 $2")
}
```

同时增加质量检测：

```go
var malformedMarkdownHeadingRE = regexp.MustCompile(`(?m)^#{1,6}[^#\s].*$`)
var overlongMarkdownHeadingRE = regexp.MustCompile(`(?m)^#{1,6}\s+.{80,}$`)
```

处理策略：

1. 可确定问题自动修复，例如 `###3.1 全域数据湖建设` -> `### 3.1 全域数据湖建设`。
2. 疑似标题正文粘连只记录质量问题，不盲目截断正文。
3. 完成事件或 artifact 元数据可记录 `quality_issues`，例如 `malformed_heading_or_body_joined`。
4. `completedFullDocumentIntegrityFailureReason` 继续负责 H1/H2 完整性；新增格式质量检测不应误伤正常完成链路，可先以 warning 或 `needs_review` 形式落地。

### 4.8 修复聊天历史索引 embedding 超长

这是 03 文档中 P1 问题，应与页面和排版一起解决。

推荐两种可选实现，优先采用方案 A。

#### 方案 A：长文档消息摘要索引

对 `full_document` 或 `document_generation_status in (completed, partial)` 的助手消息，不把完整正文和用户问题拼成单 passage。改为索引摘要对象：

```text
标题：北海电厂二期智慧电厂项目技术方案
大纲：项目背景与建设目标；总体技术架构；数据湖与基础算力平台；智慧运行系统；智慧安防系统；智能安全监控应急中心；AR眼镜智能作业；实施与运维保障
状态：completed
文档产物：artifact_id=...
用户问题：这是一份提交给甲方的投标技术方案...
摘要：本轮生成了面向北海电厂二期智慧电厂项目的完整投标技术方案，覆盖数据湖、基础算力、智慧运行、智慧安防、应急中心、AR 作业和实施运维保障。
```

优点：

- 输入短，稳定低于 embedding 限制；
- 历史检索能定位到这次完整文档；
- 不重复索引大段正文，降低成本。

#### 方案 B：安全切分后索引

如果必须索引全文，则在 `MessageService.IndexMessageToKB` 或 `CreateKnowledgeFromPassage` 入口增加 embedding 上限感知切分：

1. 按 Markdown H2/H3 先切章节；
2. 章节仍超长时按段落切；
3. 最后按 rune/token 预算硬切；
4. 每个 chunk 进入 embedding 前再次校验长度，超过上限则跳过并记录 warning。

无论采用哪种方案，都要修复“embedding 报 INVALID 后仍显示 processChunks successfully”的误导性日志。批处理结果应区分：

- `embedded_chunk_count`；
- `skipped_chunk_count`；
- `failed_chunk_count`；
- `partial_success`。

### 4.9 保留并增强长任务进度心跳

现有 `progress.UpdateStage` 和 outline/section stream heartbeat 仍应保留，但职责要调整：

- 完整大纲：由 `PublishOutline` 作为稳定产物输出。
- 实时阶段：继续用 `UpdateStage` 输出当前动作。
- 章节进度：建议统一为 `正在生成第 N/M 章：章节名`、`第 N/M 章已完成`。
- 收尾阶段：输出 `正在整理最终文档并注册文档产物`。

这样页面可以同时显示“文档规划”和“当前阶段”，用户不会把长时间生成误判为无响应。

## 5. 前端整改方案

### 5.1 新增常驻“文档规划”卡片

在 [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue) 中新增 `visiblePlanningOutline`：

```ts
const visiblePlanningOutline = computed(() => {
  // 1. 优先从 thinking event 中解析完整大纲
  // 2. 其次从 agent_complete.extra.outline 恢复
  // 3. 再从 session.chat_document_artifact 的结构信息恢复
  // 4. 最后从 final_document_content 的 H1/H2 中恢复
});
```

模板位置放在折叠树和答案之前：

```vue
<PlanningOutlineCard
  v-if="visiblePlanningOutline"
  :outline="visiblePlanningOutline"
  class="full-document-planning-card"
/>
```

展示内容：

- `文档规划` 标题；
- 文档 H1；
- H2 章节列表；
- `8 个章节` 等数量提示；
- 当前状态：规划完成、生成中、已完成。

### 5.2 不再依赖中间步骤展开才能看到规划

当前 `showIntermediateSteps=false`，答案开始或会话完成后中间步骤会被折叠。新增规划卡片后，大纲不应再只出现在 `.thinking-detail-content` 内。

处理方式：

1. `visiblePlanningOutline` 负责常驻展示完整大纲。
2. `intermediateEvents` 中过滤大纲专用 thinking 事件，避免重复。
3. 普通 `planning/retrieving/generating/finalizing` 进度仍保留在折叠树中。

示例：

```ts
const hiddenPlanningOutlineEventIds = computed(() => {
  const hidden = new Set<string>();
  for (const event of eventStream.value || []) {
    if (event.type === 'thinking' && hasPlanningOutline(event) && event.synthetic) {
      hidden.add(event.event_id);
    }
  }
  return hidden;
});
```

### 5.3 历史回放恢复规划

历史消息可能没有完整 thought stream，因此需要 fallback：

1. `agent_complete.extra.outline`；
2. `session.chat_document_artifact.structure_info.heading_titles`；
3. `session.final_document_content` 的 H1/H2；
4. 如仍无法恢复，则不显示规划卡片，只保留原有折叠步骤。

优先使用结构化数据，避免从正文中误提取 H3/H4。

### 5.4 Markdown 渲染前修复标题空格

在 `preprocessMarkdown` 或 `renderMarkdownContent` 入口增加：

```ts
const normalizeMarkdownHeadings = (content: string): string => {
  return content
    .replace(/\r\n?/g, '\n')
    .replace(/(^|\n)(#{1,6})(?![#\s])([^\n]+)/g, '$1$2 $3')
    .replace(/([^\n])\n?(#{1,6}\s+)/g, '$1\n\n$2')
    .replace(/(#{1,6}\s+[^\n]+)\n(?!\n)/g, '$1\n\n');
};
```

目标：

- `###3.1全域数据湖建设` 能渲染为 H3；
- H3 前后空行更稳定；
- 流式阶段也能尽量正常显示；
- 不替代后端 artifact snapshot 归一化。

### 5.5 强化长任务当前阶段展示

前端已有 `stage` metadata 和 `getThinkingStageLabel`，建议在规划卡片或折叠树摘要旁展示：

- 当前阶段：规划中、检索中、生成中、收尾中；
- 当前章节：第 N/M 章；
- 最近完成章节；
- 会话完成后显示 `已生成完整文档`。

这可以和常驻大纲卡片组合成“规划 + 当前进度”的长任务体验。

## 6. 质量与验收标准

### 6.1 页面可见性验收

1. 长文档规划完成后，页面直接显示 H1 和全部 H2 大纲。
2. 答案开始生成、折叠树出现后，大纲仍在页面上方可见。
3. 会话完成后刷新历史，大纲仍能恢复展示。
4. 大纲不重复出现在常驻卡片和折叠树中。
5. 当前阶段能体现规划、检索、生成、收尾中的至少一种状态。

### 6.2 正文排版验收

1. 页面、artifact snapshot、复制导出内容均不出现 `###3.1`、`####标题` 等非标准标题。
2. “数据湖与基础算力平台”等章节的 H3 标题能正确渲染。
3. 标题独占一行，标题后有空行。
4. 正文包含合理的短段落、要点列表和加粗标签。
5. 章节正文不重复当前 H2，不重启全文。

### 6.3 历史索引验收

1. 长文档完成后不再出现 embedding `INVALID length=50249` 这类错误。
2. 单个 embedding 输入稳定低于模型上限。
3. 索引日志能区分成功、跳过、失败和部分成功。
4. 历史检索能召回长文档标题、大纲、摘要和 artifact 信息。

### 6.4 主链路回归验收

1. `full_document` + 有效 KB 范围仍进入 `knowledge_grounded_full_document`。
2. 首轮完整文档仍覆盖全部大纲章节。
3. 完成状态仍能正确落库为 completed 或 partial。
4. artifact 注册不回退。
5. 自动续写判断不回退。
6. 超时、length、取消等异常路径仍能正确返回 partial/failed/cancelled。

## 9. 风险与兼容性

1. **大纲重复展示**：新增常驻卡片后必须隐藏大纲专用 thinking 事件，普通进度仍应保留。
2. **旧历史数据缺字段**：旧消息可能没有 `extra.outline`，前端需要从 artifact 或最终正文 fallback。
3. **标题归一化误修复**：自动修复只处理明确的 `#{1,6}` 后缺空格，不对疑似粘连正文做截断。
4. **流式体验与后端修复一致性**：前端修复解决显示，后端修复保证最终 artifact；两者都要做。
5. **摘要索引召回粒度降低**：摘要索引成本低且安全，但不适合全文语义召回；若需要全文召回，应采用安全切分方案。
6. **质量检测误判 completed**：格式质量问题初期建议只记录 warning，不直接把 completed 改为 failed，避免影响主链路稳定性。
