# 长文档章节扩写修正模式审查

**审查日期**: 2026-05-15  
**审查对象**: 选中文档基线后，要求对“第7章 人工智能平台与智能应用扩展”章节扩写/生成功能清单的执行链路  
**请求特征**: `intent_hint=revise_document`、`base_artifact_id=4d6dd414-e91e-4447-9766-3e8e00d74918`、`document_output_mode=delta_only`、`document_target_heading=人工智能平台与智能应用扩展`、`document_merge_mode=append_to_section`、显式知识库范围存在

## 1. 结论

| 问题 | 结论 |
|------|------|

## 2. 日志链路判断

### 2.1 请求本身满足“基线章节修正”条件

请求中已经携带以下结构化参数：

```json
{
  "intent_hint": "revise_document",
  "base_artifact_id": "4d6dd414-e91e-4447-9766-3e8e00d74918",
  "document_output_mode": "delta_only",
  "document_target_heading": "人工智能平台与智能应用扩展",
  "document_merge_mode": "append_to_section"
}
```

这说明前端/调用方已经明确表达：以上一个 artifact 为基线，只生成增量内容，并追加到指定章节。

### 2.2 路由模型识别正确，但路由应用被守卫拦截

日志关键片段：

```text
[ChatRouter][Model] raw_kind=document_edit normalized_kind=document_edit confidence=0.95
[ChatRouter] document route not applied kind=document_edit ... blocked_by=has_knowledge_scope
[ChatRouter] decision kind=document_edit ... requested_output_mode=delta_only effective_agent_kb=true ... applied=false endpoint=agent_qa
```

代码对应位置：`internal/handler/session/chat_route.go` 中 `applyDocumentRouteDecisionWithReason` 对 `ChatRouteDocumentEdit` 会调用 `canApplyDocumentEditRouteDecisionWithReason(reqCtx, false)`；后者在存在显式 `KnowledgeBaseIDs` / `KnowledgeIDs` 时直接返回 `has_knowledge_scope`。因此，带知识库的章节修正请求被排除在专用 `document_edit` 路由之外。

### 2.3 后续也没有进入专用文档编辑执行器

专用执行器的日志入口是：

```text
document_edit_path_selected: session_id=... base_artifact_id=... output_mode=...
```

本次日志没有该记录。代码上，`internal/application/service/session_agent_qa.go` 的 `shouldUseDedicatedDocumentEditPath` 同样要求 `len(req.KnowledgeBaseIDs) == 0 && len(req.KnowledgeIDs) == 0`，所以显式知识库范围再次阻断专用编辑路径。

同时，`shouldUseKnowledgeGroundedDocumentContinuationPath` 只处理 `AutoContinue=true` 的续写场景；本次是用户主动的章节修正/扩写，`auto_continue=false`，因此也不会进入知识增强长文档续写路径。

### 2.4 实际进入的是普通 AgentQA

日志显示：

```text

Calling agent QA service
Agent configured with 1 knowledge base(s)
knowledge_search tool not enabled, skipping rerank model initialization
AllowedTools=[final_answer]
```

这意味着本轮只是普通 Agent 单轮推理。模型先调用了 `read_skill`，由于 `MaxIterations=1`，后续没有机会再调用 `final_answer` 或任何检索工具，最终进入“max iterations fallback final synthesis”。

## 3. 合并链路分析

### 3.1 基线确实被加载，但没有稳定进入生成上下文

请求解析阶段 `prepareDocumentRequest` 会根据 `base_artifact_id` 加载 artifact，并调用 `BuildQuotedContext` 构建 `<document_revision_context>`。对于本请求，理论上上下文会包含：

- 上一版文档元数据；
- 目标章节 `人工智能平台与智能应用扩展`；
- `document_merge_mode: append_to_section`；
- 目标章节窗口或文档截断窗口；
- 要求优先输出 `<document_patch>`。

但是普通 Agent 路径中存在 stopgap 行为：`shouldInlineQuotedContext(req)` 在有 base artifact 且为 delta 修正时返回 false，因此 `<document_revision_context>` 不会拼进主 Agent 用户消息，只被放入 `AgentDocumentContext`。

进一步看最终合成：`internal/agent/finalize.go` 只给文档修正添加“Document revision requirements”，没有像续写一样调用上下文追加函数。现有测试 `TestStreamFinalAnswerToEventBus_UsesCompactUserGoalForDocumentRevision` 还明确断言最终合成消息不包含 `<document_revision_context>`。这与日志中最终合成请求只包含用户目标、技能摘要和修订规则一致。

因此，本次最终答案缺少“第 7 章原文/章节边界/目标标题完整形态”的输入，无法稳定生成可合并 patch。

### 3.2 artifact 注册器具备合并能力，但依赖输出格式

`RegisterFromAssistantMessage` 最终会调用 `buildChatDocumentSnapshotResult`：

- `operation=revise` 且有 base artifact 时，调用 `mergeChatDocumentRevisionDelta(base, delta)`；
- 如果 delta 是合法 `<document_patch>`，会按 `<append heading="...">` 找目标章节并追加；
- 如果 delta 以 Markdown 标题开头，会尝试按同级同名标题替换章节；
- 如果 delta 没有可识别标题/patch，会调用 `appendChatDocumentFragment(base, delta)`，把内容追加到文档末尾，并标记 `delta_merge_uncertain`。

所以，只有最终答案格式稳定时，才能正确合并到扩写章节。本次日志里最终答案合成是在缺少章节上下文的情况下产生，且开头已经出现“好的...”类自然语言片段，命中合法 patch 的概率较低。

### 3.3 即使产生 artifact，也可能是 partial

注册 artifact 时，如果出现 `delta_merge_uncertain`，artifact 状态会被置为 `partial`。如果修订结果过短、缺少标题，还可能附加 `revision_too_short` / `revision_missing_heading`。这能避免把不可靠合并标成完全可用，但不能满足用户“正确合并到第 7 章”的预期。

## 4. 根因

### R1. `document_edit` 路由不支持显式知识库范围

用户选择了知识库并不代表应该退出文档修正模式；本场景实际需要“知识增强的章节修正”。当前 guard 将 `has_knowledge_scope` 作为硬阻断，导致正确的 `document_edit` 识别结果不能生效。

### R2. 只存在“知识增强自动续写”，缺少“知识增强人工修正/扩写”专用路径

`shouldUseKnowledgeGroundedDocumentContinuationPath` 要求 `AutoContinue=true`。用户主动要求扩写某章节时，即使有 base artifact、target heading、merge mode 和知识库，也没有对应的专用执行器。

### R3. 普通 Agent fallback final synthesis 丢失修订上下文

`AgentDocumentContext.QuotedContext` 对 revise 场景没有被注入最终合成消息。普通 Agent 如果因为 max iterations 进入 fallback，就只能根据用户目标和工具摘要生成回答，无法安全生成章节 patch。

### R4. 合并器对“目标章节 + append_to_section + 非结构化 delta”没有确定性兜底

注册器知道 `TargetHeading` 和 `MergeMode`，但 `operation=revise` 时没有在非 patch 输出场景下自动包装为 `<append heading="target">...</append>`。因此模型输出普通清单时，系统更可能追加到全文末尾。

## 5. 可落地解决方案

### 方案 A：新增“知识增强章节修正”专用路径（推荐）

**目标**: 对 `base_artifact_id + delta_only + target_heading + append_to_section` 的请求，无论是否选择知识库，都进入专用文档修正执行器，而不是普通 ReAct Agent。

实施步骤：

1. 在 `session_agent_qa.go` 新增 `shouldUseKnowledgeGroundedDocumentEditPath(req, agentConfig)`：
   - `BaseArtifactID != ""`；
   - `DocumentOutputMode == delta_only`；
   - `DocumentIntent/Operation` 为 `revise`；
   - `DocumentTargetHeading != ""` 或 `QuotedContext` 中存在 `target_heading`；
   - 存在有效本地知识范围；
   - 非附件/图片场景。

2. 在 AgentQA 路由顺序中，将该路径放在普通 Agent 之前：
   - translation continuation；
   - translation full document；
   - knowledge grounded document continuation；
   - **knowledge grounded document edit**；
   - dedicated document edit；
   - knowledge grounded full document；
   - dedicated full document；
   - generic agent。

3. 实现 `runKnowledgeGroundedDocumentEditPath`：
   - 使用 `QuotedContext` 中的目标章节窗口作为主上下文；
   - 根据用户目标和目标标题做一次受控知识检索，限制 chunk 数与字符预算；
   - 构造两段上下文：`Document editing context` + `Local knowledge context`；
   - 单次 LLM 生成 `<document_patch><append heading="已解析目标标题">...</append></document_patch>`；
   - 禁止 ReAct 工具调用，避免 `MaxIterations=1` 被技能读取消耗；
   - completion event 写入 `document_patch_metadata`，包括 `structured`、`resolved_heading`、`merge_confidence`、`local_knowledge_used`、`evidence_refs`。

4. 日志新增：

```text
[DocumentEdit][Router] selected=knowledge_grounded_document_edit base_artifact_id=... target_heading=... kb_scope=...
document_edit_patch_generated structured=true merge_confidence=high target_heading=...
document_edit_artifact_registered artifact_id=... status=available quality_issues=[]
```

收益：既保留知识库 grounding，又保证输出是可合并 patch，符合用户“选中文档基线后扩写某章”的真实语义。

### 方案 B：调整 route guard，让 `document_edit` 可在知识库范围下生效

`canApplyDocumentEditRouteDecisionWithReason` 不应一律用 `has_knowledge_scope` 阻断。建议改为：

- 无知识库：进入 `dedicated_document_edit`；
- 有知识库：保持 `routeDecisionApplied=true`，但标记 execution mode 为 `knowledge_grounded_document_edit`；
- 只有附件/图片、空 query、缺失 base artifact 上下文时才阻断。

这样日志中的 `kind=document_edit` 不会停留在“影子决策”，而是真正驱动后续执行路径。

### 方案 C：修复普通 Agent fallback 的修订上下文缺失

即使新增专用路径，也应修补 fallback：

1. 在 `internal/agent/finalize.go` 增加 `appendDocumentRevisionContext`；
2. 对 `Intent=revise` / `Operation=revise` 且存在 `QuotedContext` 的场景，把压缩后的修订上下文加入最终合成；
3. 保留预算上限，优先保留：`document_outline`、`target_section_heading`、`target_section`、`nearby_siblings`、`original_user_goal`、`document_edit_target`；
4. 修改测试：不要再断言 revision final synthesis 不包含 `<document_revision_context>`，改为断言包含目标章节上下文且总字符数受控。

这能避免普通 Agent 因 max iterations、工具调用失败、模型 fallback 时生成无法合并的自然语言回答。

### 方案 D：在 artifact 合并层增加确定性兜底

在 `buildChatDocumentSnapshotResult` 的 `operation == revise` 分支增加兜底逻辑：

```go
targetHeading := resolveDocumentTargetHeading(options.UserQuery, options.TargetHeading)
mergeMode := normalizeChatDocumentMergeMode(options.MergeMode, options.Intent, targetHeading)
if targetHeading != "" && mergeMode == types.ChatDocumentMergeModeAppendToSection {
    if !looksLikeStructuredPatch(trimmed) && !startsWithMatchingHeading(trimmed, targetHeading) {
        merged := mergeChatDocumentRevisionDelta(base.ContentSnapshot, buildChatDocumentAppendPatch(targetHeading, trimmed))
        ...
        return chatDocumentSnapshotBuildResult{Snapshot: merged.Snapshot, QualityIssues: merged.QualityIssues}
    }
}
```

注意事项：

- 先清理“好的/以下是/已根据要求”等短前缀；
- 如果内容为空或只有解释性话术，不创建 artifact；
- 如果目标标题匹配不唯一，仍标记 `target_section_uncertain` 并降级 partial；
- 如果合并成功，不要把 `<document_patch>` 原文写入快照。

该兜底不能替代专用生成路径，但能显著降低“内容追加到全文末尾”的风险。

### 方案 E：完成态质量闸前移到消息完成状态

如果合并结果含 `delta_merge_uncertain`、`target_section_uncertain`、`revision_too_short`，不仅 artifact 应为 `partial`，assistant message 也不应保持 `completed=true`。建议在完成持久化阶段：

- artifact 注册返回质量问题后，回写 completion options；
- 对不确定合并设置 `CompletionStatus=partial`、`AllowComplete=false`；
- SSE complete payload 中带上 `quality_issues`、`user_hint`、`final_document_artifact_id`。

这样前端可以提示“已生成修订草稿，但章节合并需要确认”，避免用户误以为已经正确写入目标章节。

## 8. 本次日志的最终判定

本次请求“选中文档基线，并对第 7 章扩写功能清单”在语义上应该进入长文档章节修正模式；系统也已经识别出 `document_edit`。但由于显式知识库范围触发 `has_knowledge_scope` 守卫，路由没有应用，执行落入普通 AgentQA。

普通 AgentQA 又因为 `AllowedTools=[final_answer]` 叠加技能工具、`MaxIterations=1`，首轮调用 `read_skill` 后直接进入 fallback final synthesis。最终合成缺少基线章节上下文，无法稳定生成可合并 patch。

因此，本次结果不能被认为“已正确进入长文档修正模式”，也不能保证“正确合并到第 7 章”。当前最多只能依赖 artifact 注册阶段的后处理兜底；若输出不是合法 patch 或章节片段，新内容很可能追加到文档末尾并标记不确定合并。
