# 智能推理长文翻译 P0 止损修复详细设计

## 1. 阶段目标

P0 的目标不是把长文翻译体验做到理想，而是先阻断两个最严重的问题：

1. 被截断的结果被错误标记为完成。
2. 被截断或异常结果被错误写入聊天历史知识库。

本阶段必须遵守一个实现原则：尽量不引入数据库迁移，不扩大改动面，不改变现有普通问答主流程。

## 2. 设计范围

### 2.1 纳入范围

1. internal/agent/observe.go
2. internal/agent/engine.go
3. internal/agent/finalize.go
4. internal/event/event_data.go
5. internal/handler/session/qa.go
6. internal/application/service/message.go
7. config/builtin_agents.yaml

### 2.2 不纳入范围

1. messages 表结构变更。
2. 前端完整 partial 状态展示。
3. 长文翻译后台任务系统。

## 3. 核心设计原则

### 3.1 completed 必须变成“显式判定结果”

当前系统把结束等同于完成，这是根本问题。P0 必须改成：

1. 结束只是生命周期结束。
2. completed 必须由 completionStatus 显式给出。

### 3.2 入库必须建立硬门禁

即使上层状态判断失误，也要确保 messageService.IndexMessageToKB 能挡住明显脏数据。

### 3.3 普通问答体验优先保持不变

P0 的任何变更都不能显著破坏当前短问答和普通知识问答路径。

## 4. 详细设计

## 4.1 completion metadata 最小扩展

### 4.1.1 设计目标

在不改数据库的前提下，先通过内存态和事件态补充完成语义。

### 4.1.2 修改点

文件：internal/event/event_data.go

为 AgentFinalAnswerData 和 AgentCompleteData 新增字段：

```go
CompletionStatus string `json:"completion_status,omitempty"`
FinishReason     string `json:"finish_reason,omitempty"`
IsPartial        bool   `json:"is_partial,omitempty"`
AllowIndexing    bool   `json:"allow_indexing,omitempty"`
AllowComplete    bool   `json:"allow_complete,omitempty"`
FailureReason    string `json:"failure_reason,omitempty"`
```

### 4.1.3 字段语义

1. CompletionStatus：completed、partial、failed、cancelled。
2. FinishReason：直接透传 stop、length、content_filter 等模型结束原因。
3. IsPartial：便于前端或 handler 做快速分支。
4. AllowIndexing：是否允许写入聊天历史知识库。
5. AllowComplete：是否允许置消息为 completed。
6. FailureReason：兜底描述失败原因，P0 可选填。

## 4.2 analyzeResponse 增加截断识别

### 4.2.1 设计目标

当模型因为输出长度上限停止时，必须识别为 partial，而不是自然完成。

### 4.2.2 修改点

文件：internal/agent/observe.go

扩展 responseVerdict：

```go
type responseVerdict struct {
    isDone           bool
    finalAnswer      string
    emptyContent     bool
    isPartial        bool
    completionStatus string
    finishReason     string
    allowIndexing    bool
    allowComplete    bool
    step             types.AgentStep
}
```

### 4.2.3 新判定规则

1. finish_reason=stop 且无 tool calls：
   completionStatus=completed
   allowComplete=true
   allowIndexing=true
2. finish_reason=length 且无 tool calls：
   completionStatus=partial
   allowComplete=false
   allowIndexing=false
3. finish_reason=content_filter：
   completionStatus=failed
   allowComplete=false
   allowIndexing=false
4. final_answer 工具调用：
   completionStatus=completed
   allowComplete=true
   allowIndexing=true

### 4.2.4 P0 特别说明

P0 暂不在 Agent 内做复杂续写循环，只先正确分类 length。续写放到 P1 处理，这样可以降低本阶段风险。

## 4.3 Agent 完成事件透传 completion metadata

### 4.3.1 设计目标

让 handler 能基于 verdict 的真实语义完成消息收尾，而不是自行猜测。

### 4.3.2 修改点

文件：internal/agent/engine.go、internal/agent/finalize.go

调整方式：

1. state 中增加本轮最终 completionStatus、finishReason、allowIndexing、allowComplete。
2. emitCompletionEvent 输出这些字段。
3. handleMaxIterations 的兜底完成逻辑默认设为 partial 或 failed，而不是 completed。

### 4.3.3 推荐取值

1. 正常自然停止：completed。
2. max iterations 后只能拼凑总结：默认 partial。
3. 异常中止：failed。

## 4.4 completeAssistantMessage 收尾逻辑收敛

### 4.4.1 设计目标

统一由 completion metadata 决定消息是否 completed、是否允许入库。

### 4.4.2 修改点

文件：internal/handler/session/qa.go

函数签名改为：

```go
func (h *Handler) completeAssistantMessage(
    ctx context.Context,
    assistantMessage *types.Message,
    userQuery string,
    completionStatus string,
    finishReason string,
    allowIndexing bool,
    allowComplete bool,
)
```

### 4.4.3 新行为

1. allowComplete=true 时才设置 assistantMessage.IsCompleted=true。
2. allowComplete=false 时保留内容，但 IsCompleted 维持 false。
3. allowIndexing=true 时才调用 IndexMessageToKB。
4. 对 finishReason=length 打专门日志。

### 4.4.4 兼容策略

对于普通非 Agent QA 流程，如果当前没有补齐 completion metadata，可使用默认值：

1. completionStatus=completed
2. finishReason=stop
3. allowIndexing=true
4. allowComplete=true

这样可以避免大范围连锁改动。

## 4.5 聊天历史入库硬门禁

### 4.5.1 设计目标

即使上层误传，也要在最后一道口子挡住脏回答。

### 4.5.2 修改点

文件：internal/application/service/message.go

推荐将 IndexMessageToKB 改为：

```go
type MessageIndexOptions struct {
    CompletionStatus string
    FinishReason     string
    AllowIndexing    bool
}
```

```go
func (s *messageService) IndexMessageToKB(
    ctx context.Context,
    userQuery string,
    assistantAnswer string,
    messageID string,
    sessionID string,
    options MessageIndexOptions,
)
```

### 4.5.3 门禁规则

1. AllowIndexing=false 直接返回。
2. CompletionStatus!=completed 直接返回。
3. FinishReason==length 直接返回。
4. assistantAnswer 为空直接返回。
5. 命中规划性文本模式时直接返回。

### 4.5.4 文本模式建议

建议拦截如下模板片段：

1. 让我整理
2. 我需要将其完整翻译
3. 由于内容太多
4. Let me organize
5. I need to provide the full translation

这类规则是兜底，不应替代状态治理。

## 4.6 临时参数调整

### 4.6.1 设计目标

降低一般性长回答被截断的概率，但不把它当成根治方案。

### 4.6.2 修改点

文件：config/builtin_agents.yaml

建议：

1. builtin-smart-reasoning.max_completion_tokens 从 2048 提升到 4096。
2. 如线上模型验证稳定，可再提到 8192。

### 4.6.3 说明

该调整只作为缓解措施，不能视为 P0 的主要成果。

## 5. 联调方案

### 5.1 后端联调要点

1. 验证 stop 正常 completed。
2. 验证 length 进入 partial。
3. 验证 partial 结果不会触发聊天历史入库。
4. 验证短回答仍然 completed 且可入库。

### 5.2 前端联调要点

P0 不强制要求前端完整支持 partial UI，但至少要满足：

1. partial 时不要因为 complete 事件直接把消息伪装成 completed。
2. 若暂时无法展示 partial，可先保持消息未完成状态。

## 6. 测试设计

### 6.1 单测

建议新增：

1. analyzeResponse 对 length 的 verdict 测试。
2. completeAssistantMessage 不同 allowComplete/allowIndexing 分支测试。
3. IndexMessageToKB 硬门禁测试。

### 6.2 集成测试

准备以下场景：

1. 正常短回答。
2. 人为构造 length 截断回答。
3. final_answer 工具回答。
4. content_filter 回答。

## 7. 验收标准

P0 完成后，满足以下条件才算通过：

1. finish_reason=length 的回答不会被标记为 completed。
2. partial/failed 结果不会再进入聊天历史知识库。
3. 普通问答路径无明显回归。
4. 线上日志可明确区分 completed 和 partial。

## 8. 本阶段产出定义

P0 的产出不是“长文翻译能力完成”，而是“错误完成与错误入库已被阻断”。这是后续 P1 和 P2 能够继续推进的前提。
