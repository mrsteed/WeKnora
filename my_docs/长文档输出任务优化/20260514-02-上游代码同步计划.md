# 20260514 上游代码同步计划

## 1. 文档目标

本文档基于 `my_docs/20260429-04-通用上游同步操作手册.md`，形成一份面向本仓库实际执行的上游代码同步计划，用于完成以下目标：

1. 同步 `Tencent/WeKnora` 上游主线的最新代码。
2. 在解决代码冲突时，保证本地既有功能和业务语义优先落地。
3. 对于纯代码冲突和代码功能冲突，最终裁决原则均以本地功能可用、业务行为不回退为准。
4. 在完成验证后，将同步结果提交到当前仓库的 `dev` 分支。

本文档是执行计划，不是一次性命令清单。执行过程中必须保留回退锚点，并在每个阶段留下可审计的 Git 痕迹。

---

## 2. 同步范围与目标分支

### 2.1 同步来源

默认远端约定如下：

```bash
origin   https://github.com/mrsteed/WeKnora.git
upstream https://github.com/Tencent/WeKnora.git
```

本次同步目标是吸收：

```bash
upstream/main
```

### 2.2 最终落点

最终交付分支为：

```bash
origin/dev
```

需要特别强调：`dev` 是本仓库的主开发分支，跟踪对象应为 `origin/dev`，不是 `upstream/main`。本次同步的本质，是把上游主线能力受控引入本地 `dev`，而不是直接把 `dev` 替换成上游代码。

---

## 3. 本次同步的核心原则

### 3.1 上游代码必须同步

本次计划要求显式同步上游最新代码，不能只做形式上的 `fetch`。执行结果必须体现为本地同步分支已经建立在最新 `upstream/main` 基线之上。

### 3.2 功能冲突以本地功能为准

当出现以下冲突时，统一按“本地功能优先”处理：

1. 上游代码改动与本地业务能力存在行为冲突。
2. 上游接口调整会导致本地已上线功能失效。
3. 上游默认实现与本地定制化流程不一致。
4. 上游逻辑会削弱或移除本地租户、权限、知识库、Agent、导出、数据分析等既有能力。

这里的“以本地功能为准”，并不等于简单选择 `ours` 覆盖全部文件，而是要求：

1. 尽量吸收上游新增结构、修复和公共能力。
2. 在新的上游代码结构上恢复本地业务语义。
3. 如果两边实现无法直接兼容，优先保证本地功能行为不变，再补齐必要的兼容层。

### 3.3 结果必须可回退、可验证、可提交

本次同步不是只解决编译冲突，而是要形成可提交到 `dev` 的完整结果，因此必须满足：

1. 同步前可回退。
2. 同步后可验证。
3. 验证通过后可提交并推送到 `dev`。

---

## 4. 建议执行策略

### 4.1 采用“本地 dev 为基线、rebase 上游 main”的方式同步

建议流程：

1. 先将本地 `dev` 与 `origin/dev` 对齐。
2. 从本地 `dev` 切出临时同步分支。
3. 在临时同步分支上执行 `rebase upstream/main`。
4. 在冲突解决过程中，将本地功能逐步迁移到上游最新结构上。

采用该方式的原因如下：

1. 本地 `dev` 承载当前真实可用的业务能力。
2. `rebase upstream/main` 可以明确体现“同步上游”的动作。
3. 冲突会暴露在逐提交重放阶段，更利于逐项判断哪些能力必须按本地行为保留。

### 4.2 当 rebase 成本失控时，切换为“上游新基线 + 本地主题回放”

若出现以下任一情况，应切换方案：

1. 冲突文件数量过多，超过团队当次可控范围。
2. 同一条核心链路已被上游重写，继续 `rebase` 会持续重复冲突。
3. 本地历史存在较多临时性提交，导致逐提交重放价值不高。

切换后执行：

1. 从 `upstream/main` 新建整合分支。
2. 以功能主题维度回放本地提交或手工迁移本地能力。
3. 仍然遵循“功能冲突以本地功能为准”。

---

## 5. 详细执行计划

## 5.1 阶段一：同步前检查与备份

执行命令：

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch
git remote -v
```

检查要求：

1. `dev` 必须先对齐 `origin/dev`。
2. 工作区必须干净；若不干净，先提交或 `stash`。
3. 远端必须确认 `origin` 指向本仓库，`upstream` 指向腾讯上游仓库。

完成检查后，建立回退锚点：

```bash
git branch backup/dev-$(date +%Y%m%d)-before-upstream-sync
git tag sync-dev-$(date +%Y%m%d)-pre
```

该阶段产出：

1. 一个可随时回退的 `backup` 分支。
2. 一个记录同步前基线的 tag。

## 5.2 阶段二：拉取上游并创建同步分支

执行命令：

```bash
git fetch origin --prune
git fetch upstream --prune
git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main
```

建议同时记录差异规模，便于评估冲突成本：

```bash
git rev-list --left-right --count dev...upstream/main
git log --oneline dev..upstream/main -n 20
git log --oneline upstream/main..dev -n 20
```

该阶段目标：

1. 把上游最新主线作为新的历史基线。
2. 把本地 `dev` 的能力重放到上游最新基线之上。

## 5.3 阶段三：冲突处理

本阶段是本次同步的重点，处理规则如下。

### 5.3.1 普通代码冲突处理规则

当冲突只涉及导入、函数签名、配置项位置、结构体字段、初始化顺序等纯代码层调整时：

1. 优先接受上游新的结构组织方式。
2. 再把本地改动迁移到新的结构上。
3. 不允许因为图省事而回退上游整体文件版本。

### 5.3.2 代码功能冲突处理规则

当冲突已经影响业务功能行为时，统一执行以下决策顺序：

1. 先判断本地功能是否已上线、已依赖或已被验证可用。
2. 若本地功能已形成真实业务依赖，则必须保留本地功能行为。
3. 在保留本地功能行为的前提下，再吸收上游可复用的非业务层改进。
4. 如果上游新结构要求改造本地逻辑，则改造实现，不改变本地功能结果。

可直接按“本地功能优先”处理的典型场景包括：

1. 本地 Agent 行为与上游默认流程不同。
2. 本地知识库、组织、租户、权限逻辑与上游实现不一致。
3. 本地导出、文档处理、数据分析流程与上游存在行为差异。
4. 上游更新会导致本地前端交互、接口契约或页面路径失效。

### 5.3.3 冲突解决动作

每解决一批冲突后执行：

```bash
git add <冲突文件>
git rebase --continue
```

如果确认本轮 `rebase` 路径不可控，则立即停止并切换到兜底方案：

```bash
git rebase --abort
git checkout -b sync/dev-$(date +%Y%m%d)-rebuild upstream/main
```

## 5.4 阶段四：同步结果收口

当 `rebase` 或重建整合完成后，执行以下检查：

```bash
git status --short
git log --oneline --decorate -n 20
```

如果还存在为解决冲突而追加的手工调整，需形成明确提交：

```bash
git add .
git commit -m "merge: sync upstream main and preserve local features"
```

该提交的职责是：

1. 记录冲突后的最终整合结果。
2. 明确表明此次同步的策略是“同步上游，同时保留本地功能”。

---

## 6. 验证计划

## 6.1 后端验证

优先执行与本轮冲突热点相关的聚焦测试；若无法快速判断热点模块，则执行全量 Go 测试：

```bash
go test -v ./...
```

如果本轮冲突主要落在以下区域，可优先补充聚焦验证：

```bash
go test ./internal/agent/...
go test ./internal/handler/...
go test ./internal/container/...
```

说明：

1. 若测试失败是由于外部依赖环境缺失，必须单独记录。
2. 若失败源于同步引入的编译、接口或行为问题，则不得进入提交流程。

## 6.2 前端验证

执行命令：

```bash
cd frontend
pnpm install --frozen-lockfile
pnpm build
```

若本机尚未启用 `pnpm`，先执行：

```bash
corepack enable
```

## 6.3 手工回归重点

至少覆盖以下能力：

1. 聊天和流式输出是否正常。
2. Agent 相关配置、调用链路和工具流程是否正常。
3. 知识库、组织、租户、权限等本地扩展能力是否保持原有行为。
4. 文档处理、导出、数据分析等本地定制功能是否正常。
5. 本轮出现冲突的前端页面和后端接口是否完成回归。

---

## 7. 提交到 dev 分支的计划

### 7.1 先推送同步分支留痕

建议先推送临时同步分支，保留一次可复查的同步结果：

```bash
git push -u origin sync/dev-$(date +%Y%m%d)-upstream
```

### 7.2 再将同步结果提交到 dev 分支

确认验证通过后，在当前同步分支上执行：

```bash
git push --force-with-lease origin HEAD:dev
```

之所以使用 `--force-with-lease`，是因为本次同步的推荐路径是 `rebase upstream/main`，会改写本地 `dev` 历史对应的提交哈希。若不用该参数，容易在多人协作时误覆盖远端最新状态；若直接使用裸 `-f`，风险过高。

### 7.3 提交完成判定

当且仅当以下条件同时满足，才认为本次同步完成：

1. 同步分支已经建立在最新 `upstream/main` 基线之上。
2. 本地核心功能行为未回退。
3. 后端测试与前端构建已完成，失败项仅限已说明的环境依赖问题。
4. 已保留同步分支留痕。
5. 最终结果已成功推送到 `origin/dev`。

---

## 8. 回滚预案

### 8.1 尚未推送到 dev

```bash
git rebase --abort
git checkout dev
git branch -D sync/dev-$(date +%Y%m%d)-upstream
```

### 8.2 已推送同步分支但尚未更新 dev

```bash
git push origin --delete sync/dev-$(date +%Y%m%d)-upstream
git checkout dev
git branch -D sync/dev-$(date +%Y%m%d)-upstream
```

### 8.3 已更新 dev

使用同步前备份分支恢复：

```bash
git checkout backup/dev-$(date +%Y%m%d)-before-upstream-sync
git push --force-with-lease origin HEAD:dev
```

---

## 9. 建议执行顺序摘要

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch
git remote -v

git branch backup/dev-$(date +%Y%m%d)-before-upstream-sync
git tag sync-dev-$(date +%Y%m%d)-pre

git fetch origin --prune
git fetch upstream --prune

git checkout -b sync/dev-$(date +%Y%m%d)-upstream dev
git rebase upstream/main

# 解决冲突，功能冲突以本地功能为准
# 完成后执行测试和构建

git push -u origin sync/dev-$(date +%Y%m%d)-upstream
git push --force-with-lease origin HEAD:dev
```

本计划的最终要求可以概括为一句话：在明确吸收上游最新代码的前提下，把所有影响本地业务能力的冲突都收敛为“上游结构兼容、本地功能优先”，并在验证通过后把结果提交到 `dev` 分支。
