# 长文档模式续写不完整原因分析与修正方案

## 1. 问题概述

本次问题发生在长文档模式的“继续生成剩余章节”场景，接口实际走的是文档修订专用链路，而不是完整长文档章节生成链路。根据日志，模型在输出过程中被 `max_completion_tokens` 硬截断，服务端却仍将结果按 `completed` 持久化，随后质量闸又识别出正文混入内部提示词，因此前端最终展示为“部分完成”并提示人工复核。

从现象上看，用户感知是“长文档总是生成不完整”；从代码链路上看，实质是“文档修订流缺少长文档续写预算和自动续写机制，且流式输出中的思考内容与内部上下文污染了后续修订输入”。

## 2. 日志结论

### 2.1 直接证据

本次日志中的关键信号如下：

1. 请求进入文档修订专用路径，而非完整长文档章节生成路径。
   - `document_edit_path_selected`
   - `base_artifact_id=90e8c9ea-9bb0-4456-80df-441e84091840`
   - `output_mode=delta_only`

2. 修订上下文很大。
   - `quoted_context_len=7550`
   - 上一版快照大小：`snapshot_char_count: 6482`
   - `continuation_context_mode: inline_full`

3. 模型输出被长度上限截断。
   - `document_edit_stream_started ... max_completion_tokens=2048`
   - `document_edit_complete ... finish_reason=length`
   - `completion_tokens=2048`

4. 流式首个可见内容就是思考标签。
   - `First delta.Content ... preview="<think>"`

5. 服务端仍以完成态持久化消息。
   - `completion_status: completed`
   - `finish_reason: length`
   - `allow_complete: true`

6. 产物已注册，但文档状态被打为需人工复核。
   - `document_generation_status: needs_review`
   - `quality_issues: [markdown_heading_normalized internal_prompt_leakage]`

7. 索引被跳过。
   - `Skipped indexing ... reason: finish_reason_length`

### 2.2 结论摘要

这不是单一的模型能力问题，而是以下三个因素叠加导致：

1. 文档修订路径沿用了普通摘要问答的 2048 输出上限。
2. 修订路径将上一版全文和修订上下文整体塞回模型，导致输出空间进一步受限。
3. 修订路径在 `finish_reason=length` 时没有自动续写，反而直接按完成态落库。

## 3. 根因分析

### 3.1 修订链路错误复用普通摘要输出预算

文档修订入口 [internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go) 中的 `runDedicatedDocumentEditPath` 直接读取全局摘要配置作为输出上限：

- `maxCompletionTokens := s.cfg.Conversation.Summary.MaxCompletionTokens`
- 当配置未设置时才回退到 4096

当前默认配置 [config/config.yaml](../config/config.yaml) 中明确是：

- `conversation.summary.max_completion_tokens: 2048`

这意味着“继续生成剩余章节”虽然是长文档续写语义，但预算上却按普通摘要问答处理。对需要输出结构化 patch 或大段追加正文的请求而言，2048 completion tokens 明显偏低。

### 3.2 修订消息构造直接拼接完整上下文，压缩了可用输出空间

`buildDedicatedDocumentEditMessages` 位于 [internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go)，其实现非常直接：

1. 用户请求写入 `User goal`
2. 原样拼接 `QuotedContext`
3. 作为单条 user message 发给模型

本次日志已经给出：

- `quoted_context_len=7550`
- `snapshot_char_count=6482`
- `continuation_context_mode=inline_full`
- `prompt_tokens=3775`

也就是说，本轮“继续生成”不是从干净的章节锚点续写，而是把上一版大块正文、修订规则、合并规则、目标 heading 等信息整体内联回去。模型先要消化一大段上下文，再在 2048 completion tokens 内输出结果，最终被 `finish_reason=length` 截断是高概率事件，不是偶发问题。

### 3.3 修订流在被截断后没有自动续写，而是直接收口为 completed

`consumeDedicatedDocumentEditStream` 位于 [internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go)。当前逻辑只要收到流式 `Done` 事件，就会：

1. 把已收到内容拼成 `FinalAnswer`
2. 直接发出 `AgentComplete`
3. 强制写入 `CompletionStatus: completed`
4. `AllowIndexing: true`
5. `AllowComplete: true`

这里没有针对 `finish_reason=length` 做任何修订流续写分支，也没有把状态降为 `partial` 或 `continuing`。因此一旦模型因为 token 上限被截断，系统会把“不完整输出”当作“已完成输出”继续向后传递。

这就是用户看到“已经完成但正文不全”的直接原因。

### 3.4 流式链路未剥离 think，导致修订内容被污染

日志里有一个非常关键的信号：

- `First delta.Content ... preview="<think>"`

非流式场景下，[internal/models/chat/remote_api.go](../internal/models/chat/remote_api.go) 中的 `parseCompletionResponse` 会调用 `removeThinkingContent`，对以 `<think>` 开头的模型输出做兜底剥离。

但流式场景下，同文件中的 `processStreamDelta` 直接透传 `delta.Content`，不会先移除 `<think>...</think>`。这说明：

1. 即使 `Thinking=false`，部分模型仍可能输出 `<think>`。
2. 文档修订流的 `finalContent` 会把这类内容直接拼进最终答案。
3. 后续修订若再把该答案当作 base artifact 或 quoted context 输入，就会继续放大污染。

这也是为什么本次“上一份文档内容”里已经能看到明显的 `<think>` 与内部提示语痕迹。

### 3.5 内部提示词泄漏已经被质量闸捕获，但捕获发生得太晚

Markdown 质量校验位于 [internal/application/service/full_document_markdown_quality.go](../internal/application/service/full_document_markdown_quality.go)。

其中 `containsInternalPromptLeakage` 会检查如下内容：

1. 文档上下文标签，如 `document_revision_context`、`document_patch`、`target_heading` 等。
2. 内部元数据行，如 `document_merge_mode`、`source_heading`、`continuation_context_mode` 等。

本次日志中已经出现：

- `quality_issues: [markdown_heading_normalized internal_prompt_leakage]`

说明系统最终识别到了“正文混入了内部提示词或上下文标签”。但当前问题在于：

1. 这些污染内容已经进入本轮输出。
2. 修订路径没有在进入模型前做上下文清洗。
3. 质量闸只是在产物注册阶段给出 `needs_review` 标记，并没有阻断前面错误的完成态收口。

因此质量闸起到的是“告警”作用，不是“防错”作用。

## 4. 非根因项说明

以下日志不属于本次“不完整”的根因：

### 4.1 `chat_document_artifacts record not found`

日志中的 SQL：

- `SELECT * FROM chat_document_artifacts WHERE source_message_id = ... LIMIT 1`
- 结果 `record not found`

这表示系统在注册当前消息对应的新 artifact 前，先查了一次是否已有旧记录。随后日志已经显示：

- `Registered chat document artifact`
- `revision_no: 2`
- `operation: revise`

因此这里是“先查无，再新增”的正常流程，不是失败点。

### 4.2 索引跳过不是根因，而是后果

`Skipped indexing ... reason: finish_reason_length` 只是因为本轮输出被 length 截断后，系统避免把可疑结果写入聊天历史知识库。它是保护动作，不是生成不完整的原因。

## 5. 修正方案

### 5.1 P0：文档修订路径改为使用长文档预算，而不是摘要预算

建议修改 [internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go) 中的 `runDedicatedDocumentEditPath`，不要再直接使用 `Conversation.Summary.MaxCompletionTokens`。

建议策略：

1. 文档修订默认最小输出预算提升到 4096。
2. 若已存在长文档预算协商结果，则优先复用 continuation budget。
3. 若模型能力声明了更高 `max_output_tokens`，按模型能力上限协商。
4. 仅在模型自身输出上限低于目标预算时，才向下收缩。

目标是让“继续生成剩余章节”与完整长文档续写共享同一套预算语义，而不是走普通摘要档位。

### 5.2 P0：`finish_reason=length` 时不要直接 completed，必须进入续写分支

建议在 [internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go) 的 `consumeDedicatedDocumentEditStream` 中增加修订流专用分支：

1. 若 `finish_reason=length` 且当前 `finalContent` 非空，不直接发 completed。
2. 将状态落为 `partial` 或文档态 `continuing`。
3. 自动构造一次“继续输出剩余 patch/剩余章节”的 follow-up revise 请求。
4. 将本轮已产出的 patch 作为基线，直到出现 `finish_reason=stop`、显式完成标记、或达到自动续写轮次上限。

没有这一步，再高的预算也只是“降低截断概率”，不能从机制上消除“不完整”。

### 5.3 P0：流式输出进入文档缓冲前必须剥离 think 块

建议在 [internal/models/chat/remote_api.go](../internal/models/chat/remote_api.go) 的流式处理逻辑中增加统一净化：

1. 对 `delta.Content` 使用与非流式一致的 think 剥离策略。
2. 或者在文档修订流拼接 `finalContent` 前统一执行 `SplitContentAndReasoning` / think strip。
3. 若收到不完整的 `<think>` 起始片段，需要跨 chunk 维护状态，避免把半截 think 混入最终正文。

如果不做这一步，后续所有基于 artifact 的 revise/continue 都会被污染上下文反复放大。

### 5.4 P1：进入模型前对修订上下文做瘦身和净化

建议在 `buildDedicatedDocumentEditMessages` 之前增加专用的 quoted context 预处理：

1. 移除 `<think>...</think>`。
2. 移除明显的内部标签与元数据行。
3. 对于 `append_to_section` 这类场景，仅保留目标章节、相邻章节、必要的合并规则。
4. 避免把上一版全文以 `inline_full` 形式全部塞回。

推荐目标：

1. 优先传“目标章节片段 + 目标 heading + 合并规则”。
2. 不再默认传整个 base artifact 快照。
3. 对于超长上下文，改为“结构化锚点 + 摘要 + 目标片段”的组合。

### 5.5 P1：完成态和展示态要与真实完整性一致

当前服务端会出现：

1. 消息 `completion_status=completed`
2. 但文档 `document_generation_status=needs_review`
3. 且 `finish_reason=length`

这个组合会让前端很难准确表达“其实被截断了”。建议收口规则调整为：

1. `finish_reason=stop` 且无完整性风险，才允许 completed。
2. `finish_reason=length` 且未完成自动续写，统一落为 partial。
3. `needs_review` 只表达质量问题，不承担“内容未完成”的语义。

这样可以避免“系统显示完成，但正文没写完”的用户困惑。

## 6. 建议实施顺序

### 6.1 第一阶段

优先修复三项 P0：

1. 修订流预算切换到长文档预算。
2. `finish_reason=length` 自动续写。
3. 流式 think 剥离。

这三项完成后，能先解决绝大多数“继续生成剩余章节仍然不完整”的主路径问题。

### 6.2 第二阶段

补齐两项 P1：

1. quoted context 瘦身与净化。
2. 完成态与文档态统一。

这两项会显著改善稳定性和用户认知一致性。

## 7. 验收标准

修复后建议按以下标准回归：

1. 在“请继续生成剩余的章节”场景下，若首轮命中 `finish_reason=length`，系统必须自动发起下一轮续写，直到 `stop` 或达到明确上限。
2. 修订流日志中不应再出现 `First delta.Content ... preview="<think>"` 被直接写入最终 artifact 的情况。
3. 新产物不应再出现 `internal_prompt_leakage`。
4. 同样的北海电厂技术方案场景下，前端不应再稳定复现“部分完成 + 内部提示词混入”的组合告警。
5. 若确实达到自动续写轮次上限，前端应明确显示“续写未完成”，而不是“已完成但需复核”。

## 8. 最终判断

本次“总是生成不完整”的根因已经比较明确：

1. 文档修订链路预算过小。
2. 修订上下文过重。
3. `length` 截断后没有自动续写。
4. 流式 think 与内部提示词污染了修订输入和最终产物。

因此，单纯换模型或单纯提高前端重试次数都无法根治问题。必须在服务端把“修订流”真正升级为“具备长文档预算、流式净化和自动续写能力的文档续写链路”。
