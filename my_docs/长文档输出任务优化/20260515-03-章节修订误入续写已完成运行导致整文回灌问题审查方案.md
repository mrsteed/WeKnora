# 章节修订误入续写已完成运行导致整文回灌问题审查方案

## 1. 审查结论

本次“补充第5章 智慧安防系统方案的功能清单”请求虽然表面表现为“续写失败”，但从日志和当前实现看，实际不是模型没生成，也不是知识库没有命中，而是请求在进入执行链路前后发生了三层语义错配：

1. 用户请求本质是“基于现有文档，对指定章节做增量修订”，应走 document edit 语义。
2. 路由输入构造阶段却先把该请求标记成 `user_requested_route=knowledge_grounded_document_continuation`，导致模型和后续路由应用阶段被强烈引导到 continuation。
3. continuation 运行阶段加载到一个“已无剩余章节”的 generation run 后，直接按 completed 返回整文快照，且没有任何 answer chunk 流出，最终由 SSE fallback 把整篇文档一次性回灌给前端。

因此，本次问题不是“续写模型输出失败”，而是：

**章节修订请求被误识别成知识库续写；而续写主路径在命中已完成 generation run 时，又把“整文已完成”错误地当成“本次请求已成功完成”，最终把整篇文档作为 fallback answer 回灌给前端。**

## 2. 日志证据链

从日志可以直接串出完整错误链路。

### 2.1 请求原始语义是章节修订，不是整文续写

请求体中已经给出了明确的结构化修订语义：

1. `intent_hint=revise_document`
2. `base_artifact_id=402f97d6-334d-4bde-8ef3-a33cbec579e3`
3. `document_output_mode=delta_only`
4. `document_target_heading=智慧安防`
5. `document_merge_mode=append_to_section`

这组参数组合的语义非常明确：

1. 用户不是要继续自动补齐整篇剩余章节。
2. 用户是要对“智慧安防”对应章节执行定向增补。
3. 返回结果应优先是章节级 delta，而不是整篇 full snapshot。

### 2.2 路由输入阶段先把请求写成了 continuation

日志中的 ChatRouter prompt 里已经出现：

1. `user_requested_route = knowledge_grounded_document_continuation`
2. `latest_artifact_summary.document_generation_status = continuing`

这说明在模型参与路由分类前，服务端已经把该请求包装成“用户请求 continuation”。

也就是说，本次误路由并不是大模型凭空判断错了，而是：

**服务端路由输入本身先把 revise 请求描述成了 continuation 请求。**

### 2.3 路由模型随后返回 continuation，属于被输入诱导后的顺势结果

路由日志显示：

1. `raw_kind=knowledge_grounded_document_continuation`
2. `normalized_kind=knowledge_grounded_document_continuation`
3. `reason="查询要求补充已有方案的指定章节，存在续写中的base artifact，知识库有效，应按续写路线处理。"`

这一步虽然看似是模型判定，但由于输入上下文已包含：

1. `explicit_base_artifact_id`
2. `latest_artifact_summary.document_generation_status=continuing`
3. `user_requested_route=knowledge_grounded_document_continuation`

所以这更像是模型在沿着服务端预设的 continuation 语义做确认，而不是独立识别出了正确的业务意图。

### 2.4 continuation 主路径没有真正生成章节，runtime_sections=0

关键日志：

1. `[LongDocument] completion summary`
2. `completion_status: completed`
3. `document_generation_status: completed`
4. `runtime_sections: 0`
5. `generation_run_id: 317bd3ae-document-generation-run`

`runtime_sections=0` 说明本轮根本没有进入任何章节生成。

因此这次 completed 不是“补充第5章成功完成”，而是：

**续写主路径加载到 generation run 后，发现它没有剩余章节，于是直接返回 completed。**

### 2.5 SSE 没有收到 answer chunk，只能 fallback 整文

关键日志：

1. `No answer events were streamed, emitting fallback answer (len=62973)`
2. `content_len: 23267`
3. 持久化后 artifact revision 注册成功

这说明执行链路里没有任何 answer event 流向前端。最终 complete event 携带的 authoritative answer 被 SSE fallback 机制直接作为 answer 发出。

由于 continuation completed 分支返回的是整文快照，所以 fallback answer 也变成了整篇文档，而不是用户预期的“第5章功能清单补充 delta”。

## 3. 当前代码对应的问题点

## 3.1 路由输入构造错误地把“有 baseArtifact + 有知识库”的请求统统标记为 continuation

在 [internal/handler/session/chat_route.go](internal/handler/session/chat_route.go) 的 `buildChatRouteInput(...)` 中，当前逻辑是：

1. 只要 `request.AutoContinue` 为真，就设置 `requestedRoute=knowledge_grounded_document_continuation`
2. 或者只要 `BaseArtifactID` 非空且存在知识库范围，也同样设置为 `knowledge_grounded_document_continuation`
3. 只有 `BaseArtifactID` 非空但没有知识库范围时，才设置为 `document_edit`

这段逻辑遗漏了最关键的业务分叉：

1. `intent_hint=revise_document`
2. `document_target_heading` 非空
3. `document_merge_mode=append_to_section / replace_section`
4. `document_output_mode=delta_only`

这些字段已经足以说明这是“结构化章节修订请求”，不能因为同时选择了知识库就自动降格为 continuation。

### 直接影响

这会造成两个问题：

1. route model prompt 被预先注入错误的 `user_requested_route`
2. 路由模型就算想返回 document_edit，也会被显著干扰

## 3.2 document_edit 路由还会被无条件提升为 knowledge grounded continuation

在 [internal/handler/session/chat_route.go](internal/handler/session/chat_route.go) 中：

1. `shouldPromoteDocumentEditToKnowledgeGroundedContinue(...)`
2. `canApplyKnowledgeGroundedContinuationRouteDecisionWithReason(...)`

当前条件基本等价于：

1. 有 `baseArtifactID` 或 `autoContinue`
2. 有知识库范围

就允许 document_edit 被提升为 knowledge grounded continuation。

这里同样没有把下列请求视作必须保留 document_edit 语义的强信号：

1. revise intent
2. 非空 `document_target_heading`
3. `append_to_section / replace_section`
4. `delta_only`

因此，即使路由模型返回了 document_edit，只要命中了“带知识库的基线文档请求”，仍可能被服务端二次提升成 continuation。

### 更准确地说

当前系统里，knowledge grounded continuation 被当成了“带知识库的所有基线文档请求”的统一路由，但这与章节修订的业务语义并不一致。

## 3.3 continuation run-path 在 remainingSections=0 时直接 completed，忽略了当前请求的章节修订语义

在 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runKnowledgeGroundedDocumentContinuationRunPath(...)` 中，当前逻辑是：

1. 从 generation run 读取 `CompletedSectionsJSON`
2. 计算 `remainingSections`
3. 若 `len(remainingSections)==0`，立即返回 completed
4. 若存在 `BaseArtifact.ContentSnapshot`，则直接把整文快照作为 final answer

这段逻辑对于“真正的整文续写”是合理的，但对于本次请求存在两个问题：

1. 当前请求不是“继续剩余章节”，而是“补充第5章的功能清单”
2. generation run 的“无剩余章节”只能说明整文大纲已完成，不能说明“针对既有章节的修订需求已经完成”

也就是说：

**run 已完成，不等于本次章节修订请求也完成。**

当前实现把这两个概念混为一谈了。

## 3.4 SSE fallback 对“无 answer chunk 的 completed 长文档”缺少语义保护

在 [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go) 中：

1. `shouldUseAuthoritativeCompleteAnswer(...)`
2. `shouldFallbackAnswerFromComplete(...)`

当前策略是：

1. 只要 complete event 有 `FinalAnswer`
2. 且前面没有 answer chunk
3. completed 场景就允许 fallback answer

这对于“自然结束但只发了 complete event”的正常长文档生成是合理的。

但对于本次这种“误路由 + 无 runtime section + 返回整文快照”的异常 completed 场景，就会放大错误结果：

1. 前端没有看到章节修订 delta
2. 却被一次性灌入整篇完整文档
3. 用户主观体验就会变成“续写失败/返回错结果”

## 3.5 `artifact.CanContinue()` 只能说明 artifact 可继续承载增量，不代表 generation run 还有剩余章节

在 [internal/types/chat_document_artifact.go](internal/types/chat_document_artifact.go) 中，`CanContinue()` 的条件主要依赖：

1. artifact kind 合法
2. `document_generation_status` 不是 blocked / needs_review
3. snapshot 长度未超过 continuation 上限

这只能说明 artifact 还能作为“文档编辑/续写的基线容器”。

它并不能说明：

1. generation run 还有剩余章节
2. continuation 主路径适用于当前请求
3. 章节修订应被当成整文续写处理

本次日志里 artifact summary 仍是 `continuing`，很容易误导路由层继续走 continuation，但这只是 artifact 的历史状态，不是本次请求的执行方式判定依据。

## 4. 根因归纳

本次问题至少包含三个根因。

### 4.1 根因一：路由输入先验错误

服务端在构造 ChatRouter 输入时，使用“只要有 baseArtifact + 知识库，就默认 requestedRoute=knowledge_grounded_document_continuation”的规则，覆盖了结构化 revise 请求的真实语义。

### 4.2 根因二：document_edit 到 continuation 的提升条件过宽

当前只要命中知识库范围，带基线文档的 edit 请求就容易被二次提升为 continuation，没有把“目标章节修订”视作不可跨越的业务边界。

### 4.3 根因三：continuation run-path 的 completed 分支缺少“请求语义防线”

`remainingSections==0` 只能推出“整文续写无需继续”，不能推出“本次章节修订请求已完成”。当前实现直接返回 full snapshot completed，导致错误完成态成立。

## 5. 修正目标

修正后的行为应满足以下目标：

1. 带 `revise_document + document_target_heading + append_to_section/replace_section + delta_only` 的请求，优先保留 document_edit 语义。
2. 即使同时存在知识库范围，也不能自动把章节修订请求升级为 continuation。
3. continuation run-path 命中“已无剩余章节”时，必须先判断当前请求是不是整文续写，而不是无条件返回 completed full snapshot。
4. SSE fallback 不能把“本来应返回 delta 的请求”错误放大成“整篇文档回灌”。

## 6. 修正方案

## 6.1 P0：修正 ChatRouter 输入中的 requested_route 构造逻辑

建议修改 [internal/handler/session/chat_route.go](internal/handler/session/chat_route.go) 的 `buildChatRouteInput(...)`：

当前错误规则：

1. `BaseArtifactID != "" && hasKnowledgeScope => knowledge_grounded_document_continuation`

建议改为分层判断：

1. 若 `request.AutoContinue == true`，才默认 `requested_route=knowledge_grounded_document_continuation`
2. 若存在 `BaseArtifactID`，但同时满足以下任一条件，则 `requested_route=document_edit`
   - `intent_hint == revise_document`
   - `document_target_heading != ""`
   - `document_merge_mode == append_to_section`
   - `document_merge_mode == replace_section`
   - `document_output_mode == delta_only` 且意图是对既有章节补充/修改
3. 只有在“基线文档 + 知识库 + 明确整文续写语义”同时成立时，才把 `requested_route` 设为 continuation

也就是说：

**知识库存在与否，只影响是否需要 knowledge grounding，不应直接决定 edit 与 continuation 的语义分支。**

## 6.2 P0：收紧 document_edit -> knowledge_grounded_continue 的提升条件

建议修改 [internal/handler/session/chat_route.go](internal/handler/session/chat_route.go) 中：

1. `shouldPromoteDocumentEditToKnowledgeGroundedContinue(...)`
2. `canApplyKnowledgeGroundedContinuationRouteDecisionWithReason(...)`

增加明确的拒绝提升条件：

1. `reqCtx.documentIntent == revise_document`
2. `reqCtx.documentOperation == revise`
3. `reqCtx.documentTargetHeading != ""`
4. `reqCtx.documentMergeMode == append_to_section`
5. `reqCtx.documentMergeMode == replace_section`
6. `reqCtx.documentOutputMode == delta_only` 且非显式 auto continue

只要命中上述任一条件，就必须保留 document_edit 路由。

## 6.3 P0：continuation run-path 对“已完成 run”增加请求语义保护

建议修改 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runKnowledgeGroundedDocumentContinuationRunPath(...)`：

当前行为：

1. `remainingSections == 0 => completed + full snapshot`

建议增加守卫：

1. 若当前请求存在明确章节修订语义：
   - `DocumentIntent == revise_document`
   - 或 `DocumentOperation == revise`
   - 或 `DocumentTargetHeading != ""`
   - 或 `DocumentMergeMode == append_to_section/replace_section`
2. 则不能直接把“已完成 run”当成本次请求的成功结果

建议处理方式二选一：

1. 优先方案：直接拒绝 continuation 路径，回退到 dedicated document edit 路径
2. 次优方案：返回 blocked/manual_retry，并明确 reason 为 `completed_generation_run_not_applicable_to_revision`

这里不建议继续返回 full snapshot completed，因为那会继续掩盖误路由问题。

## 6.4 P1：SSE fallback 增加“delta-only 章节修订”保护条件

建议修改 [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go) 中：

1. `shouldUseAuthoritativeCompleteAnswer(...)`
2. `shouldFallbackAnswerFromComplete(...)`

增加保护逻辑：

1. 若 complete payload 对应的是 delta-only 修订请求
2. 且没有任何 answer chunk 流出
3. 且 authoritative answer 明显是 full snapshot

则不要直接 fallback 整文 answer；应优先：

1. 依赖 artifact card 展示最终整文
2. 或 fallback 为明确错误提示，而不是整文回灌

这样即便上游再误路由一次，也不会把整篇文档直接灌回聊天正文。

## 6.5 P1：artifact summary 不能单独作为 continuation 优先级依据

建议补充约束：

1. `latest_artifact_summary.document_generation_status=continuing` 只能说明“历史产物未终结”
2. 不能单独据此把新请求当成 continuation

真正的 continuation 语义至少还应同时满足：

1. 当前请求 intent/operation 是 continue
2. 没有章节级 target heading
3. merge mode 不是 section-scoped revise mode
4. 或者显式 `auto_continue=true`

## 7. 建议回归用例

修复后至少补以下回归。

### 7.1 路由输入与路由应用

1. `revise_document + base_artifact + kb + target_heading + append_to_section + delta_only`
   - 期望：`requested_route=document_edit`
   - 期望：最终应用 `document_edit`

2. `continue_document + base_artifact + kb + 无 target_heading + auto_continue=false`
   - 期望：可走 `knowledge_grounded_document_continuation`

3. `revise_document + base_artifact + kb` 但 `target_heading` 为空、`merge_mode` 为空
   - 期望：仍优先按 revise/document_edit 处理，除非存在明确 continuation 信号

### 7.2 continuation run-path 守卫

1. generation run 已 completed，当前请求是章节 revise
   - 期望：不得直接返回 completed full snapshot

2. generation run 已 completed，当前请求是整文 continuation
   - 期望：可以直接 completed，并返回整文权威结果

### 7.3 SSE fallback

1. completed 但无 answer chunk，authoritative answer 是整文 full snapshot，当前请求是 delta-only revise
   - 期望：不直接 fallback 整文 answer

2. completed 但无 answer chunk，当前请求是合法整文 continuation/full document
   - 期望：允许 fallback authoritative answer

## 8. 最终判断

本次问题的核心不是“模型没补出来内容”，而是系统把一个“章节修订请求”错误地套进了“整文续写已完成”的执行框架：

1. 路由输入先把 revise 请求包装成 continuation
2. 路由应用阶段继续放宽 document_edit 到 continuation 的提升条件
3. continuation run-path 又把“无剩余章节”误判为“本次请求成功完成”
4. SSE fallback 最终把整篇文档回灌给前端，放大错误表现

因此，修复重点不在大模型提示词，而在：

1. **修正 requested_route 构造逻辑**
2. **给章节修订请求保留 document_edit 语义边界**
3. **为 continuation completed 分支增加请求语义防线**
4. **限制 fallback answer 对异常 full snapshot 的放大**

只有把这四层同时收紧，才能避免“章节补充请求误走续写并把整篇文档回灌”的问题再次出现。
