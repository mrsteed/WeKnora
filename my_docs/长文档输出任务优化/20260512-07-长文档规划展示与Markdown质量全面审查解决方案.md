# 长文档规划展示与 Markdown 质量全面审查解决方案

## 1. 审查背景

本次审查对象包括：

- 最新输出文档：`my_docs/test/北海电厂二期智慧电厂项目投标技术方案_2026-05-12_141736.md`
- 对应日志：`my_docs/test/14.md`
- 后端长文档生成链路：`internal/application/service/session_agent_qa.go`
- 后端 Agent SSE 事件处理：`internal/handler/session/agent_stream_handler.go`
- 前端 Agent 流式展示组件：`frontend/src/views/chat/components/AgentStreamDisplay.vue`
- 前端 SSE 消息处理入口：`frontend/src/views/chat/index.vue`
- 文档 Artifact 质量与导出链路：`internal/application/service/chat_document_artifact.go`

用户反馈的四类问题可以归纳为两个层面：

1. 生成质量层面：规划、章节编号、正文 Markdown 排版没有形成统一约束，导致输出文档像多个片段拼接而成。
2. 展示协议层面：规划阶段已经完成的大纲没有在实时流中及时显示，而是在完成输出后才通过兜底数据源显示出来。

## 2. 总体结论

### 2.1 文档质量问题结论

最新文档确实存在较明显的 Markdown 结构问题：

- 多处 H3 标题与正文粘连，例如 `### 项目背景与行业机遇当前...`。
- 二级章节没有统一章节编号，三级标题有的带 `3.1`，有的完全不带编号。
- 子列表缩进不稳定，存在单空格缩进、列表符号后缺空格等问题。
- 加粗标签后正文直接粘连，例如 `**总体愿景：**建成一座...`，虽然多数 Markdown 解析器可渲染，但观感不符合正式技术文档要求。
- 后端当前对章节输出主要依赖 Prompt 约束，没有在写入最终答案前进行章节级 Markdown 规范化和质量门禁。

### 2.2 规划展示问题结论

规划大纲“完成输出后才显示”的直接原因在前端 SSE 处理：

- 后端 `fullDocumentProgressReporter.PublishOutline` 会发送一个一次性 synthetic thinking 事件，事件带有大纲 `content`，同时 `done=true`。
- 前端 `frontend/src/views/chat/index.vue` 的 `thinking` 分支只在 `data.done === false` 时创建 thinking 事件。
- 当首个 thinking 事件就是 `done=true + content` 时，前端只尝试完成已有事件，但 `_eventMap` 里没有该事件，因此内容被丢弃。
- 完成事件到达后，`upsertAgentCompleteEvent` 又从 complete payload 的 `outline` 字段拿到大纲，所以用户看到的是“最后才显示”。

这不是后端完全没有发规划，而是前端实时流没有兼容“一次性完成的 thinking 事件”。

## 3. 当前链路梳理

### 3.1 后端长文档生成路径

主要入口在 `internal/application/service/session_agent_qa.go`：

- `runKnowledgeGroundedFullDocumentGenerationPath`：基于知识库证据生成完整文档。
- `runDedicatedFullDocumentGenerationPath`：不依赖知识库的完整文档生成。
- `buildKnowledgeGroundedFullDocumentOutlineMessages`：生成大纲 Prompt。
- `buildDedicatedFullDocumentOutlineMessages`：通用大纲 Prompt。
- `generateValidatedFullDocumentOutline`：调用模型流式生成大纲并解析为 `dedicatedFullDocumentOutline`。
- `fullDocumentProgressReporter.PublishOutline`：将已解析大纲作为 synthetic thought 发给前端。
- `buildKnowledgeGroundedFullDocumentSectionMessages` / `buildDedicatedFullDocumentSectionMessages`：按章节生成正文。
- `consumeFullDocumentSectionStream`：消费章节流式内容并直接向前端输出 answer chunk。

当前 `dedicatedFullDocumentOutline` 只有：

```go
type dedicatedFullDocumentOutline struct {
    Title    string
    Sections []string
}
```

这个结构只能表达“标题 + H2 列表”，不能表达章节编号、章节层级、子标题规划、章节写作要求、每章编号前缀等信息。

### 3.2 前端流式展示路径

主要入口在 `frontend/src/views/chat/index.vue`：

- `case 'thinking'`：接收后端 thinking SSE。
- `message.agentEventStream`：维护 Agent 展示事件流。
- `_eventMap`：按 event_id 合并 thinking 流式片段。
- `case 'complete'`：收到完成事件后，写入 complete payload、更新 Artifact、触发自动续写。

展示组件在 `frontend/src/views/chat/components/AgentStreamDisplay.vue`：

- `visiblePlanningOutline`：决定置顶规划卡片显示内容。
- `planningOutlineFromStreamEvent`：优先从实时 thinking 事件中解析大纲。
- `extractPlanningOutlineFromCompleteEvent`：从 complete 事件读取结构化 outline。
- `extractPlanningOutlineFromText`：从最终文档或 thinking markdown 里反解析 H1/H2。

因此，规划卡片的理想实时路径应是：

后端 `PublishOutline` -> SSE thinking -> 前端 `case 'thinking'` 创建事件 -> `AgentStreamDisplay` 从 thinking 内容解析 outline -> 置顶卡片立即显示。

当前断点发生在前端 `case 'thinking'` 对 `done=true + content` 的处理。

## 4. 根因分析

### 4.1 Markdown 格式混乱

#### 现象

最新产物中存在以下典型问题：

```markdown
### 项目背景与行业机遇当前，能源革命与数字革命...

### 总体建设目标本项目旨在依托...

- **全量数据采集与汇聚**
 - **采集范围覆盖**：接入 DCS、NCS等...
 - **采集方式多样**：支持 OPC UA/DA...

- **数据清洗与标准化治理**
 -在线部署数据清洗流水线...
 -建立电厂级数据标准字典...
```

#### 根因

1. 章节正文完全流式直出，模型一旦产生格式问题，后端没有在最终答案写入前进行统一修复。
2. Prompt 只描述“标题独占一行”“空一行”等规则，但没有强制“输出前自检”和“格式不合规必须重写”。
3. 章节流式消费函数 `consumeFullDocumentSectionStream` 按 token/chunk 直接写入 `finalContent`，没有章节级缓冲、规范化、校验、修复步骤。
4. `sanitizeGeneratedMarkdown` 只用于另一套 long document task 链路，不作用于当前 Agent full_document 链路。

### 4.2 章节编号不统一

#### 现象

- H2 标题多为 `## 数据湖与基础算力平台技术方案`，不带章节号。
- H3 标题中出现 `### 3.1总体架构与设计思路`、`### 3.2运行智能监盘系统`，但前面的章节并未统一编号。
- 用户希望规划阶段明确章节、标题、编号，后续生成严格继承，例如某章是第四章，则所有下级标题都使用 `4.x`。

#### 根因

1. 当前大纲结构只保存 H2 字符串，没有保存 `chapter_no`。
2. 章节生成 Prompt 只告诉模型“当前 H2 已经由系统输出”，没有传递“本章编号前缀必须是 N”。
3. Prompt 示例中出现 `### 3.1 全域数据湖建设`，模型容易照搬 `3.1`，但系统没有动态替换成当前章节真实编号。
4. 后端输出 H2 时直接使用 `headingChunk := "## " + section`，没有对 `section` 注入章节编号。

### 4.3 加粗标签与正文粘连

#### 现象

例如：

```markdown
**总体愿景：**建成一座...
- **数据湖与基础算力平台建设：**构筑全厂...
```

#### 根因

1. Prompt 允许使用 `**建设内容：**` 等标签，但没有明确要求标签后必须留一个空格再接正文。
2. 没有后处理规则统一将 `**标签：**正文` 修正为 `**标签：** 正文`。
3. 列表项中加粗标签与正文混写较多，在长文档中会放大视觉割裂感。

### 4.4 规划大纲延迟显示

#### 现象

截图显示“文档规划”卡片直到完善输出完毕后才出现。

#### 直接原因

后端发送规划大纲的事件形态类似：

```json
{
  "response_type": "thinking",
  "done": true,
  "content": "# 标题\n## 章节一\n## 章节二",
  "data": {
    "event_id": "document-outline-...",
    "synthetic": true,
    "stage": "planning"
  }
}
```

前端实时处理逻辑当前是：

```js
if (!data.done) {
  // 创建 thinkingEvent 并累加 content
} else {
  // 只完成已有 thinkingEvent
}
```

当首包就是 `done=true` 时，`thinkingEvent` 不存在，content 没有进入 `agentEventStream`。完成事件到达后，前端才从 complete payload 的 `outline` 或 artifact/final document 中兜底恢复大纲。

## 5. 完整解决方案

建议按“协议修复、规划结构化、章节生成约束、格式后处理、质量门禁、测试验收”六层落地。

### 5.1 第一层：修复前端 thinking 一次性完成事件

#### 目标

保证 `done=true + content` 的 thinking 事件也会被创建、展示、解析为规划卡片。

#### 修改位置

`frontend/src/views/chat/index.vue` 的 `case 'thinking'`。

#### 建议逻辑

将 thinking 事件处理改成“先 upsert，再根据 done 标记完成”：

```js
const eventId = data.data?.event_id;
if (!eventId) break;

let thinkingEvent = message._eventMap.get(eventId);
if (!thinkingEvent && (data.content || data.done)) {
  thinkingEvent = {
    type: 'thinking',
    event_id: eventId,
    content: '',
    done: false,
    startTime: Date.now(),
    thinking: !data.done,
    synthetic: !!data.data?.synthetic,
    stage: typeof data.data?.stage === 'string' ? data.data.stage : ''
  };
  message.agentEventStream.push(thinkingEvent);
  message._eventMap.set(eventId, thinkingEvent);
}

if (thinkingEvent) {
  if (data.data?.synthetic !== undefined) {
    thinkingEvent.synthetic = !!data.data.synthetic;
  }
  if (typeof data.data?.stage === 'string') {
    thinkingEvent.stage = data.data.stage;
  }
  if (data.content || data.data?.replace) {
    if (data.data?.replace) {
      thinkingEvent.content = data.content || '';
    } else {
      thinkingEvent.content += data.content || '';
    }
  }
  if (data.done) {
    thinkingEvent.done = true;
    thinkingEvent.thinking = false;
    thinkingEvent.duration_ms = data.data?.duration_ms || (Date.now() - thinkingEvent.startTime);
    thinkingEvent.completed_at = data.data?.completed_at || Date.now();
  }
}
```

#### 预期效果

- `PublishOutline` 发出的规划大纲会在规划完成后立即显示。
- 不再等待 complete 事件或 artifact 兜底。
- 对普通多段 thinking 流没有破坏，因为 `replace` 与累加逻辑保持兼容。

### 5.2 第二层：增强后端大纲事件为结构化 outline

#### 目标

减少前端从 markdown 文本反解析大纲的不确定性。

#### 修改位置

- `internal/event/event_data.go`
- `internal/application/service/session_agent_qa.go`
- `internal/handler/session/agent_stream_handler.go`
- `frontend/src/views/chat/components/AgentStreamDisplay.vue`

#### 建议改造

在 `AgentThoughtData` 中新增可选字段：

```go
type AgentThoughtData struct {
    Content   string                 `json:"content"`
    Iteration int                    `json:"iteration"`
    Done      bool                   `json:"done"`
    Replace   bool                   `json:"replace,omitempty"`
    Synthetic bool                   `json:"synthetic,omitempty"`
    Stage     string                 `json:"stage,omitempty"`
    Outline   map[string]interface{} `json:"outline,omitempty"`
}
```

`PublishOutline` 发送：

```go
Data: event.AgentThoughtData{
    Content: outlineMarkdown,
    Done: true,
    Synthetic: true,
    Stage: "planning",
    Outline: dedicatedFullDocumentOutlineData(outline),
}
```

`agent_stream_handler.handleThought` 将 `outline` 透传到 metadata。

前端 `getPlanningOutline(event)` 优先读取 `event.outline` 或 `event.data.outline`，再 fallback 到 markdown 解析。

#### 预期效果

- 前端规划卡片不依赖 markdown 标题反解析。
- 大纲展示、complete 事件、历史 artifact 使用同一份结构化数据。
- 后续支持章节编号、子标题、进度定位更容易。

### 5.3 第三层：将大纲从“字符串列表”升级为“编号化章节计划”

#### 目标

规划阶段一次性确定章节编号、标题、下级标题规则，正文生成严格按规划执行。

#### 建议结构

将当前：

```go
type dedicatedFullDocumentOutline struct {
    Title    string
    Sections []string
}
```

升级为兼容结构：

```go
type dedicatedFullDocumentOutline struct {
    Title    string
    Sections []dedicatedFullDocumentSection
}

type dedicatedFullDocumentSection struct {
    Number      int
    Title       string
    Heading     string
    Subsections []dedicatedFullDocumentSubsection
}

type dedicatedFullDocumentSubsection struct {
    Number string
    Title  string
}
```

兼容策略：

- 旧字段 `Sections []string` 可先保留为 `LegacySections` 或通过 helper 输出。
- 对旧大纲解析结果自动补编号：第 1 个 H2 -> `Number=1`，第 2 个 H2 -> `Number=2`。
- `Heading` 统一由后端生成，不让模型自由决定：`第%d章 %s` 或 `%d. %s` 二选一。

#### 推荐标题格式

建议投标技术方案采用中文章格式：

```markdown
# 北海电厂二期智慧电厂项目投标技术方案

## 第1章 项目背景与建设目标
### 1.1 项目背景与行业机遇
### 1.2 总体建设目标
### 1.3 建设范围与核心内容

## 第2章 数据湖与基础算力平台技术方案
### 2.1 全域数据湖建设
### 2.2 标准化数据管控体系
```

如果用户明确希望“第四章”，后端必须以规划结果为准：

- 规划里 `数据湖与基础算力平台技术方案` 是第 4 章，则 H2 为 `## 第4章 数据湖与基础算力平台技术方案`。
- 本章所有 H3 必须为 `### 4.1 ...`、`### 4.2 ...`。
- H4 必须为 `#### 4.1.1 ...`。

### 5.4 第四层：重写大纲 Prompt，要求编号与子标题计划

#### 当前问题

当前大纲 Prompt 只要求 H1 + H2，不要求编号，也不要求子标题计划。正文阶段只能靠模型临时发挥。

#### 建议大纲 Prompt

将大纲输出从纯 markdown 改为“严格 JSON + 后端渲染 markdown”。这样能从源头保证章节编号清晰。

推荐 System Prompt：

```text
You are planning a complete customer-facing technical bid document.
Return JSON only. Do not write the document body. Do not use markdown fences.
The outline is a contract for later generation. Every later section must follow the numbers and titles in this outline.
```

推荐 User Prompt：

```text
请为用户目标规划完整技术方案大纲。

输出要求：
1. 只输出 JSON，不要输出 Markdown、解释或代码块。
2. title 为文档 H1 标题。
3. sections 为 4-8 个一级章节，每个章节必须包含 number、title、heading、subsections。
4. number 从 1 开始连续递增，不能跳号、重号。
5. heading 必须等于 "第{number}章 {title}"。
6. subsections 至少 3 个，最多 6 个。
7. 每个 subsection.number 必须使用 "{chapter}.{index}"，例如第二章第三节为 "2.3"。
8. 后续正文生成必须严格使用这些编号，不允许自行改名或新增同级章节。
9. 如果本地知识不足，也要在对应章节中规划“资料缺口与补充建议”，不要编造事实。

JSON Schema:
{
  "title": "string",
  "sections": [
    {
      "number": 1,
      "title": "string",
      "heading": "第1章 string",
      "subsections": [
        {"number": "1.1", "title": "string"}
      ]
    }
  ]
}
```

#### 后端渲染规则

后端不要直接信任模型 markdown，而是使用结构化 JSON 渲染：

```go
func renderFullDocumentOutlineMarkdown(outline dedicatedFullDocumentOutline) string {
    // # title
    // ## 第1章 xxx
    // ### 1.1 xxx
}
```

正文生成时，系统已经知道当前章编号与子标题列表，模型只负责填充内容。

### 5.5 第五层：重写章节生成 Prompt，强约束 Markdown 风格

#### 核心原则

章节生成 Prompt 必须从“泛泛提醒格式”变成“带编号契约的写作任务”。

#### 建议章节 Prompt

```text
你正在撰写完整技术方案中的一个章节正文。

硬性约束：
1. 当前 H2 章节标题已经由系统输出，请不要重复 H1/H2。
2. 当前章节编号为：{{chapter_number}}。
3. 当前章节标题为：{{chapter_heading}}。
4. 本章只能使用以下 H3 小节标题，且必须按顺序输出：
   - {{chapter_number}}.1 {{subsection_1}}
   - {{chapter_number}}.2 {{subsection_2}}
   - {{chapter_number}}.3 {{subsection_3}}
5. 所有 H3 必须写成：### {{subsection_number}} {{subsection_title}}
6. H3 标题必须独占一行，标题后必须空一行再写正文。
7. 禁止输出 `###标题正文`、`### 1.1 标题正文粘连`、`### 1.1标题`。
8. 禁止输出 H1/H2，禁止新增未规划的 H3。
9. 列表规则：
   - 顶层列表使用 `- `。
   - 二级列表使用两个空格缩进后再写 `- `。
   - 禁止使用单空格缩进的 ` - `。
   - 禁止使用 `-正文`，列表符号后必须有一个空格。
10. 加粗标签规则：
   - 可使用 `**建设内容：** 正文`。
   - `**标签：**` 后必须有一个空格再写正文。
   - 禁止 `**标签：**正文`。
11. 每个自然段 2-4 句，段落之间空一行。
12. 只输出当前章节正文，不要输出总结性解释、代码块、JSON 或提示词。
13. 如证据不足，写明“本地知识库资料不足以支撑该点”，不要编造事实。

输出前自检：
- 是否所有标题独占一行？
- 是否所有 H3 编号都以 {{chapter_number}}. 开头？
- 是否存在 `**标签：**正文`？如有，改为 `**标签：** 正文`。
- 是否存在 ` - ` 单空格缩进？如有，改为两个空格缩进。
```

### 5.6 第六层：新增章节级 Markdown 规范化与质量门禁

#### 目标

Prompt 只能降低错误概率，不能作为唯一质量保障。正式投标技术文档必须在后端最终写入前做格式校验。

#### 建议新增函数

在 `session_agent_qa.go` 或拆分新文件 `document_markdown_quality.go`：

```go
type markdownQualityIssue struct {
    Code    string
    Line    int
    Message string
}

func normalizeGeneratedSectionMarkdown(input string, chapterNumber int, allowedSubsections []string) string
func validateGeneratedSectionMarkdown(input string, chapterNumber int, allowedSubsections []string) []markdownQualityIssue
func repairGeneratedSectionMarkdown(ctx context.Context, chatModel chat.Chat, input string, issues []markdownQualityIssue) (string, error)
```

#### 确定性规范化规则

先做不改变事实的确定性修复：

1. 统一换行：将 CRLF 与 CR 统一转换为 LF，即把 `\\r\\n`、`\\r` 统一为 `\\n`。
2. 去除 markdown fence。
3. 标题标记修复：`###3.1标题` -> `### 3.1 标题`。
4. 标题前后空行：标题前至少一个换行，标题后一个空行。
5. 列表符号修复：`-正文` -> `- 正文`。
6. 单空格子列表修复：`^ - ` -> `  - `。
7. 加粗标签修复：`**([^*\n]{1,40}[：:])**([^\s\n])` -> `**$1** $2`。
8. 多余空行压缩：连续 3 个以上空行 -> 2 个。

#### 校验规则

如果仍存在以下问题，进入修复重试或阻断：

- 标题与正文同一行。
- H3 编号不以当前章编号开头。
- 出现 H1/H2，因为当前章节正文不允许重复 H1/H2。
- 出现未规划的 H3 标题。
- 列表符号后无空格。
- 单空格缩进列表。
- 输出为空或过短。

#### 修复策略

建议采用两级策略：

1. 默认使用确定性规范化，成本低且不会改变事实。
2. 对无法确定性修复的问题，调用一次“格式修复模型”，要求不增删事实，只修复 Markdown。修复后再次校验；仍失败则标记 `needs_review`，但不要把明显混乱的正文标记为 `completed`。

### 5.7 第七层：调整流式输出与最终内容一致性

当前章节流式内容直接写入 `finalContent` 并同步给前端。如果只在最终 complete 中修复，用户流式期间可能短暂看到原始格式，但最终答案和 Artifact 可以被覆盖为修复后的版本。

推荐分两步实施：

#### P1：最终内容质量优先

- `consumeFullDocumentSectionStream` 仍可流式输出原始内容。
- 每章结束后，对该章累计内容进行 normalize/validate。
- `finalContent` 中保存规范化后的章节内容。
- complete 事件的 `final_answer` 使用规范化全文。
- 前端现有逻辑在 complete 时会用 `final_answer` 覆盖 `message.content` 与 answerEvent，因此最终展示、Artifact、导出均为规范版本。

#### P2：流式展示也完全规范

- 章节正文改为“章节级缓冲”，生成期间只显示进度，不实时显示正文 token。
- 每章完成并规范化后，一次性发送该章规范化正文。
- 优点是用户从头到尾看到的都是干净 Markdown。
- 缺点是正文可见首 token 延迟增加，但长文档投标方案更看重质量，建议可做成 `document_quality_mode=section_buffered` 配置。

### 阶段五：Markdown 规范化与质量门禁

1. 新建 markdown quality helper。
2. 对每章输出进行 normalize。
3. 对 normalize 后内容进行 validate。
4. validate 失败时执行一次 repair pass。
5. repair 仍失败时：
   - `completion_status=partial`
   - `document_generation_status=needs_review`
   - `failure_reason=markdown_quality_failed`
   - complete extra 中带 `quality_issues`
6. Artifact 注册时保留这些 quality signals，前端可提示“文档格式需复核”。

### P1：质量提升

1. 结构化 outline metadata。
2. JSON 大纲规划。
3. Markdown quality issue 写入 complete extra 和 artifact quality signals。
4. 前端规划卡片优先读取结构化 outline。

### P2：体验优化

1. 章节级 buffered streaming，保证用户流式期间也只看到规范正文。
2. 规划卡片显示当前生成章节进度与当前 H3 小节。
3. 文档完成后提供“格式检查通过/需复核”状态。

## 8. 风险与兼容性

### 8.1 JSON 大纲解析失败风险

模型偶尔可能输出非 JSON。建议保留旧 markdown outline parser 作为 fallback，并在 fallback 后由后端补齐编号。

### 8.2 流式延迟风险

如果启用章节级 buffered streaming，正文显示会从 token 级变成章节级。建议先 P1 只修复最终内容，P2 再引入可配置 buffered 模式。

### 8.3 历史消息兼容

历史消息主要通过 `agent_steps`、complete event、artifact snapshot 重建。新增 outline metadata 不影响旧数据。前端读取顺序建议为：

1. thinking event 的结构化 outline。
2. complete event 的结构化 outline。
3. artifact 的 structure_info。
4. final_document_content markdown 反解析。

### 8.4 内容事实风险

Markdown 修复不应改变业务事实。确定性修复只处理空格、换行、标题标记、列表缩进。模型 repair pass 必须明确“不得增删事实、不得改写技术内容，只修复 Markdown 格式”。

## 10. 本次问题对应关系

| 用户问题 | 根因 | 解决措施 |
| --- | --- | --- |
| Markdown 格式混乱、文档割裂 | 章节流式直出，无最终格式门禁 | 章节级 normalize/validate/repair，最终 complete 使用规范化全文 |
| 规划阶段未明确编号 | outline 只有 H2 字符串，无章节编号结构 | JSON 结构化大纲，后端生成 `第N章` 与 `N.x` |
| 正文格式错误，例如加粗标签粘连 | Prompt 约束弱，无后处理 | Prompt 增加强制格式规则，后处理插入必要空格 |

## 11. 最终验收标准

完成修复后，同类完整技术方案生成必须满足：

1. 规划大纲在正文开始前实时显示。
2. 大纲显示的章节与最终文档 H2 一致。
3. H2 统一为 `第N章 标题` 或统一的数字标题格式。
4. H3/H4 编号严格继承所属章节编号。
5. 标题独占一行，标题后空一行。
6. 列表缩进符合 Markdown 标准。
7. 加粗标签后留空格，正文不粘连。
8. 完成态 Artifact 和导出 Markdown 均为规范化内容。
9. 若质量校验失败，不能标记为 completed，必须进入 partial/needs_review 并携带质量问题。
