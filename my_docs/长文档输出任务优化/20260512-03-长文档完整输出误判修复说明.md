# 长文档完整输出误判修复说明

## 1. 问题结论

本次现象不是大模型真正发生了“输出截断”，而是后端在长文档生成完成后的持久化阶段，将已经完整的最终正文误判为“章节不完整”，从而把消息状态从已完成降级成了 partial，并将 `document_generation_status` 改写为 `continuing`。

因此，界面上出现“输出被截断，结果不完整”的提示，本质上是后端完成态判定错误，不是模型中途停止，也不是正文真的缺失。

## 2. 日志证据链

从本次日志可以直接看到两段相互矛盾、但能串起来解释问题的证据：

1. 模型侧已经正常结束。

   - `Natural-stop at OpenAI layer (finish=stop)`
   - `[LongDocument] completion summary ... completion_status: completed, finish_reason: stop, document_generation_status: completed`

2. 持久化阶段又被后端拦截并降级。

   - `Blocked invalid completed full document persistence`
   - `finish_reason: outline_or_section_incomplete`
   - `document_generation_status: continuing`
   - `completion_status: partial`

这说明：

- 大模型输出流程已经完整结束。
- 长文档运行态也认为已经完成。
- 真正把结果改成“不完整”的，是 handler 在落库前的二次完整性校验。

## 3. 根因分析

本次误判由两个条件叠加触发。

### 3.1 handler 没有始终采用 authoritative finalAnswer

agent 长文档链路在服务层会生成一个经过规范化的最终正文 `FinalAnswer`。这个正文才是完成事件里的权威版本。

但原有逻辑在 `internal/handler/session/qa.go` 中只有当 `FinalAnswer` 的字符数比当前流式累积内容更长时，才会用 `FinalAnswer` 覆盖消息正文。

这会带来一个问题：

- 流式正文可能包含提示语、重复空行、未修正标题等“脏内容”，长度更长。
- 最终整理后的 `FinalAnswer` 经过 Markdown 规范化后，反而可能更短，但结构更正确。

一旦出现这种情况，handler 会继续拿旧的流式正文去做完整性校验，而不是拿已经修正过的最终正文去校验。

### 3.2 handler 的章节比较规则比服务层更粗糙

服务层在校验完整文档时，比较的是“章节标题 title”，会把：

- `第1章 项目背景与建设目标`

归一化为：

- `项目背景与建设目标`

但 handler 的兜底校验原先直接比较原始 H2 文本，导致：

- outline 元数据里的章节：`项目背景与建设目标`
- 实际正文里的 H2：`第1章 项目背景与建设目标`

被错误视为不相等，进而触发：

- `outline_or_section_incomplete`

这就造成了“服务层认为完整，handler 又把它打回 partial”的判定冲突。

## 4. 为什么界面会显示“输出被截断，结果不完整”

前端显示的是后端最终持久化后的消息状态，而不是单纯显示模型的 `finish=stop`。

本次链路中：

1. 模型正常输出完成。
2. 服务层生成了完整文档，并记录 `completion_status=completed`。
3. handler 在 `completeAssistantMessage` 中再次做完整性检查。
4. 由于使用了非权威正文，且章节名比较规则不一致，命中了 `outline_or_section_incomplete`。
5. 消息最终以 `partial + continuing` 落库。
6. 前端据此显示橙色提示：“输出被截断，结果不完整”。

所以，界面提示是“后端状态误判”的结果，不是“模型真的没写完”。

## 5. 本次修复内容

本次已完成两处修复。

### 5.1 full-document agent 完成时优先采用最终正文

修复文件：`internal/handler/session/qa.go`

修复要点：

- 对于 `OutputMode=full` 的长文档完成事件，只要 `FinalAnswer` 非空且不是 failed/cancelled，就优先用它作为持久化正文。
- 不再依赖“最终正文必须比流式正文更长”这一不可靠条件。

修复后可确保：

- 落库校验使用的是服务层已经规范化过的权威正文。
- 不会因为最终正文更精炼、更短而被错误忽略。

### 5.2 统一章节标题比较语义

修复文件：`internal/handler/session/qa.go`

修复要点：

- 对 handler 的章节提取与缺失章节比对增加统一归一化。
- 比较时忽略 `第N章`、`1.` 等章节编号，只按章节标题 title 比较。

修复后可确保：

- handler 与服务层对“章节是否完整”的判断标准一致。
- 不再把 `第8章 运维保障与验收建议` 与 `运维保障与验收建议` 判定为不同章节。

## 6. 验证结果

已针对本次问题补充并通过窄范围回归测试：

1. `TestCompleteAssistantMessage_DowngradesInvalidCompletedFullDocument`
2. `TestCompleteAssistantMessage_PrefersAuthoritativeFinalAnswerForFullDocument`
3. `TestCompleteAssistantMessage_AllowsCompletedNeedsReviewFullDocument`

验证结果：全部通过。

其中新增测试专门覆盖以下场景：

- 流式正文更长但格式不规范。
- `FinalAnswer` 更短但结构正确。
- full-document 持久化必须采用 `FinalAnswer`，且最终状态保持 completed。

## 7. 对本次现象的最终判断

本次“输出被截断，结果不完整”属于后端完成态误判。

更准确地说：

- 文档正文实际上已经完整输出。
- 导出 PDF 成功也进一步侧面说明正文可用。
- 错误出在完成后落库时的二次完整性校验，而不是生成过程本身。

## 8. 后续建议

建议后续再补两项加固：

1. 在长文档完成链路增加结构化日志，显式打印“用于最终校验的正文来源”是 `stream_content` 还是 `final_answer`。
2. 将 handler 的兜底完整性校验继续向服务层标准化逻辑收敛，避免再次出现“双重校验标准不一致”的问题。
