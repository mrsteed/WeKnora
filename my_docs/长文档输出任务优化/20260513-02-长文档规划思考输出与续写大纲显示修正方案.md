# 长文档规划思考输出与续写大纲显示修正方案

## 1. 审查背景

本次审查针对两个录屏现象和一份日志：

1. `my_docs/test/屏幕录制 2026-05-13 104726.mp4`：首次发起完整长文档生成后，页面长时间只显示“正在规划完整文档大纲，已等待 N 秒”，用户看不到大模型已经输出的思路过程；最终希望能显示规划出来的大纲。
2. `my_docs/test/17.md`：对应首个录屏的后端日志，明确记录模型在规划阶段已经输出 `reasoning_content`，但正文 JSON 大纲很久之后才出现。
3. `my_docs/test/屏幕录制 2026-05-13 112306.mp4`：文章续写或修订时，页面能够较快显示思路过程，但思路区域里会出现修订补丁、append heading、章节追加计划等内容，容易被理解为“大纲显示异常”。

本次目标不是重新分析第 7 章输出不完整的超时问题；该问题已在 `20260513-01-长文档输出不完整原因审查报告.md` 中归因。本次重点是“规划/续写过程中的思考可见性”和“大纲展示边界”。

## 2. 结论摘要

首个录屏中，模型并不是没有输出。日志显示，完整文档大纲规划阶段在 10:46:59 已经收到首个 `reasoning_content`，但直到 10:48:20 才收到首个 `delta.Content`。这中间约 81 秒，后端长文档大纲消费函数直接忽略 `ResponseTypeThinking`，前端只能看到系统合成的等待心跳，所以用户感觉“开始规划后很久不输出内容”。

第二个录屏中，文档修订链路已经有一条可工作的样板：`runDedicatedDocumentEditPath` 的 `consumeDedicatedDocumentEditStream` 会透传 `ResponseTypeThinking`，因此用户能立刻看到思路。但当前前端大纲识别逻辑仍会对所有 thinking-like 事件尝试做大纲解析；同时，续写完成事件在后端会把基线文档大纲以通用 `outline` 语义放进 complete payload。需要注意的是，前端实时流当前只显式读取 complete payload 里的 `outline` 字段，没有消费未来可能新增的 `base_outline/planning_outline` 等字段，因此这里既有“当前语义混用”的问题，也有“后续继续扩展时容易误接线”的问题。

因此，本次应按两条线修正：

1. 完整长文档生成链路：在大纲规划和章节生成阶段透传模型返回的 `reasoning_content`，不要只显示等待心跳。
2. 续写/修订展示链路：区分“生成出来的新大纲”“基线文档大纲”“模型思路内容”和“修订补丁计划”，前端只在明确是规划大纲时显示大纲卡片。

## 3. 关键证据

### 3.1 首次完整文档规划阶段已有模型 reasoning

`my_docs/test/17.md` 中，完整文档大纲规划请求开始后，模型很快返回了 reasoning：

```text
INFO [2026-05-13 10:46:59.722] [Yuqzw3sdpopj] remote_api.go:833[processStreamDelta] | [LLM Stream] First reasoning_content at OpenAI layer (len=6, preview="我们", elapsed_ms=621)
```

但真正的可见正文，也就是 JSON 大纲内容，要到 10:48:20 才开始：

```text
INFO [2026-05-13 10:48:20.423] [Yuqzw3sdpopj] remote_api.go:852[processStreamDelta] | [LLM Stream] First delta.Content at OpenAI layer (len=2, preview="{\n", tool_call_seen=false, thinking_seen=true, elapsed_ms=80987)
```

随后大纲流在 10:49:03 自然结束：

```text
INFO [2026-05-13 10:49:03.373] [Yuqzw3sdpopj] remote_api.go:819[processStreamDelta] | [LLM Stream] Natural-stop at OpenAI layer (finish=stop, tool_calls field never observed, thinking_seen=false, first_content_seen=true, elapsed_ms=123829)
```

这说明，用户等待期间并不是模型完全静默，而是模型输出了 reasoning，但长文档规划链路没有把它转成用户可见事件。

### 3.2 首次章节生成阶段也先有 reasoning

同一日志中，开始生成第 1 章后也出现了相同模式：

```text
INFO [2026-05-13 10:49:05.193] [Yuqzw3sdpopj] remote_api.go:833[processStreamDelta] | [LLM Stream] First reasoning_content at OpenAI layer (len=6, preview="好的", elapsed_ms=890)
INFO [2026-05-13 10:49:17.364] [Yuqzw3sdpopj] remote_api.go:852[processStreamDelta] | [LLM Stream] First delta.Content at OpenAI layer (len=3, preview="###", tool_call_seen=false, thinking_seen=true, elapsed_ms=12929)
```

也就是说，长文档链路中不仅“大纲规划”会先输出 reasoning，“章节正文生成”也可能先输出 reasoning。如果只消费 `answer`，用户仍会看到一段等待空窗。

### 3.3 录屏画面与日志一致

首个录屏抽帧显示，界面只出现：

```text
正在规划完整文档大纲，已等待 8 秒。
正在规划完整文档大纲，已等待 16 秒。
```

下方状态仅为“规划中”，没有模型真实思路。结合日志可知，这不是前端没有连上 SSE，而是后端长文档规划消费者把 `thinking` 流丢弃了，只保留了合成进度心跳。

### 3.4 续写/修订路径已经能透传 thinking

第二个录屏显示，修订请求发出后，思考框很快出现如下模型内容：

```text
正在分析基线文档并生成修订补丁，请稍候。
我们被要求补充第4章 智慧运行系统设计，完善功能描述...
```

当前代码中 `runDedicatedDocumentEditPath` 的 `consumeDedicatedDocumentEditStream` 已经处理 `ResponseTypeThinking`：

```go
if response.ResponseType == types.ResponseTypeThinking {
    if response.Content != "" {
        markFirstVisibleStream(string(types.ResponseTypeThinking))
        modelThinkingStarted = true
        emitDedicatedDocumentEditThought(ctx, req, eventBus, modelThinkingEventID, response.Content, false, false, false)
    }
    if response.Done {
        closeModelThinking()
    }
    continue
}
```

这解释了为什么第二个录屏能立刻看到思路，而第一个完整文档规划仍然不能。

## 4. 代码级根因

### 4.1 大纲规划函数直接忽略 `ResponseTypeThinking`

位置：`internal/application/service/session_agent_qa.go`

函数：`chatFullDocumentOutlineWithProgress`

当前处理逻辑为：

```go
if response.ResponseType == types.ResponseTypeThinking {
    continue
}
```

因此完整文档大纲阶段的真实模型 reasoning 只停留在 `remote_api.go` 到上游 stream channel 之间，没有进一步转换成 `AgentThoughtData`，也不会进入 `StreamManager`，前端没有机会展示。

这就是首个录屏“已经有 reasoning，但页面仍只显示等待心跳”的直接原因。

### 4.2 章节生成函数同样忽略 `ResponseTypeThinking`

位置：`internal/application/service/session_agent_qa.go`

函数：`consumeFullDocumentSectionStream`

当前处理逻辑同样是：

```go
if response.ResponseType == types.ResponseTypeThinking {
    continue
}
```

这会导致章节正文开始前的模型思路也不可见。日志中第 1 章首个 reasoning 到首个 `###` 正文之间约 12 秒，当前 UI 只能依赖“第 X/Y 章仍在生成中”的心跳，而不能显示模型真实思路。

### 4.3 合成进度和模型思路共用 thinking 类型，但语义不同

后端 `fullDocumentProgressReporter.UpdateStage` 会发送合成 thinking：

```go
Data: event.AgentThoughtData{
    Content: trimmedContent,
    Replace: true,
    Synthetic: true,
    Stage: trimmedStage,
}
```

这类事件本质上是系统状态，例如“正在检索本地知识库”“正在规划完整文档大纲，已等待 16 秒”。它应该被前端当作可替换的状态标签或进度条。

模型 reasoning 则是供应商返回的流式内容，应该是非 synthetic、可追加的思路流。

当前完整文档路径只发 synthetic progress，不发 model thinking；文档编辑路径则两者都会发，所以两个录屏表现不同。

### 4.4 当前大纲显示逻辑对事件来源区分不够

位置：`frontend/src/views/chat/components/AgentStreamDisplay.vue`

当前 `getPlanningOutline` 对任何 thinking-like 事件都尝试解析大纲：

```ts
const getPlanningOutline = (event: any): PlanningOutlinePreview | null => {
  if (!isThinkingLikeEvent(event)) {
    return null;
  }
  return getPlanningOutlineFromEvent(event, getThinkingContent(event));
};
```

这会带来两个风险：

1. 如果模型思路中出现 Markdown 标题，前端可能把普通思路误识别为“文档规划”。
2. 如果思路内容恰好是修订补丁说明、append heading、章节追加计划，用户会把它和正式大纲混淆。

第二个录屏中，模型思路里出现了类似：

```text
append heading="第4章 智慧运行系统设计"
4.7 智慧运行系统集成与人机交互
```

这些内容属于修订补丁构造过程，不等于完整文档规划大纲，也不应该驱动顶部“文档规划”卡片。

### 4.5 续写完成事件里的 `outline` 目前承载的是基线大纲，不是本轮规划

位置：`runKnowledgeGroundedDocumentContinuationPath`

当前续写路径会从基线 artifact 提取大纲：

```go
baseOutline := extractFullDocumentOutlineFromArtifact(req.BaseArtifact)
```

并在 complete extra 中写入：

```go
extra["outline"] = dedicatedFullDocumentOutlineData(baseOutline)
```

这个 `outline` 的语义是“基线文档结构”，不是“本轮续写规划”。后端 SSE complete 事件会把 `data.Extra` 扁平展开，因此这个字段会出现在 complete payload 顶层。当前前端实时流的 `upsertAgentCompleteEvent` 只显式读取 `completePayload.outline`，所以它已经具备把该字段视为规划大纲来源的接线条件；而对 `base_outline/planning_outline/outline_role` 这类新字段，前端目前还没有对应接收逻辑。

因此，文章续写时存在两层风险：

1. 一旦 complete 事件或历史恢复路径继续直接消费这个 `outline`，顶部规划卡片就可能显示旧文档完整结构。
2. 当前用户只是要求补充第 4 章，但卡片标题仍可能被理解为整篇文档规划。
3. 用户看到模型思路中的 append 计划，又看到基线 outline，难以区分哪个才是本轮输出计划。

## 5. 修正方案

### 5.1 P0：完整文档大纲阶段透传模型 reasoning

在 `chatFullDocumentOutlineWithProgress` 中新增一个模型思路事件 ID，例如：

```go
modelThinkingEventID := generateEventID("document-outline-model-thinking")
```

当收到 `ResponseTypeThinking` 且 content 非空时：

1. 发送 `EventAgentThought`。
2. `Synthetic=false`。
3. `Replace=false`。
4. `Stage="planning"`。
5. 复用 `AgentThoughtData` 的 `Content` 承载增量思路。
6. 收到 `response.Done` 时发送同 event_id 的 done 事件。

示意逻辑：

```go
if response.ResponseType == types.ResponseTypeThinking {
    if strings.TrimSpace(response.Content) != "" {
        emitFullDocumentModelThinking(ctx, req, eventBus, modelThinkingEventID, "planning", response.Content, false)
        resetHeartbeat()
    }
    if response.Done {
        emitFullDocumentModelThinking(ctx, req, eventBus, modelThinkingEventID, "planning", "", true)
    }
    continue
}
```

不要通过 `progress.UpdateStage` 发送模型思路，因为 `UpdateStage` 会记录到结构化进度步骤，并默认 `Replace=true/Synthetic=true`。模型 reasoning 应作为单独的非 synthetic 流式思考事件，避免污染 `AgentSteps`。

### 5.2 P0：章节生成与续写生成阶段也透传 reasoning

在 `consumeFullDocumentSectionStream` 中也处理 `ResponseTypeThinking`。建议扩展函数签名，传入当前 `session_id`、`message_id` 或一个专门的 emit 回调，避免该低层函数直接依赖过多请求对象。

推荐形态：

```go
type fullDocumentThinkingEmitter func(stage string, content string, done bool)

func consumeFullDocumentSectionStream(..., onThinking fullDocumentThinkingEmitter, onAnswer func(string)) fullDocumentSectionStreamResult
```

章节生成时：

1. `stage="generating"`。
2. metadata 中保留 `section_current`、`section_total`、`section_title`。
3. reasoning 到达时重置等待心跳，避免同时显示“仍在生成中，已等待 N 秒”和真实思路造成噪声。
4. answer 到达时关闭该模型思考事件，前端自然折叠到中间步骤。

这样完整文档生成的用户体验会与第二个录屏中的文档编辑路径一致：模型一开始思考，页面就有内容可看。

### 5.3 P0：规划大纲必须用结构化事件明确发布

当前 `progress.PublishOutline(outline)` 的方向是正确的，应继续保留，并作为“最终规划大纲”的唯一强信号。

建议增强 outline metadata：

```json
{
  "outline": {...},
  "outline_role": "generated_plan",
  "outline_source": "model_validated_outline"
}
```

前端顶部“文档规划”卡片只应该优先展示 `outline_role=generated_plan` 的 outline。这样即使模型思路里有标题、补丁或 append heading，也不会误触发大纲卡片。

### 5.4 P1：续写路径先补语义标识，再逐步拆分 `base_outline` / `planning_outline`

这里建议分两步做，而不是直接一次性改字段名。

第一步是低风险兼容修正：保留当前 complete payload 里的 `outline`，但显式补一个角色字段，先把它标识为基线结构，而不是本轮规划：

```go
extra["outline"] = dedicatedFullDocumentOutlineData(baseOutline)
extra["outline_role"] = "base_document"
```

这样可以先让前端在不破坏自动续写等现有链路的前提下，学会忽略“基线大纲”作为顶部规划卡片来源。

第二步再做语义拆分：如果后续确实会生成“本轮续写计划”，再新增独立字段：

```go
extra["base_outline"] = dedicatedFullDocumentOutlineData(baseOutline)
extra["planning_outline"] = currentRoundOutline
extra["outline_role"] = "continuation_plan"
```

前端展示规则建议明确为：

1. `generated_plan`：显示为“文档规划”。
2. `continuation_plan`：显示为“续写计划”或“本轮补充计划”。
3. `base_document`：不要显示为规划卡片，可作为基线结构信息使用。
4. 无 role 的旧数据保持兼容，但优先级低于显式 role。

这样第二个录屏中的“第 4 章补充”就不会被旧文档完整大纲或修订补丁思路干扰，同时也避免一开始就改动过大的字段迁移。

### 5.5 P1：前端大纲解析只信任明确的规划来源

修改 `AgentStreamDisplay.vue` 的 `getPlanningOutline`，不要对所有 thinking 内容做 Markdown 大纲兜底解析。

建议规则：

```ts
const getPlanningOutline = (event: any): PlanningOutlinePreview | null => {
  if (!isThinkingLikeEvent(event)) {
    return null;
  }
  if (event.outline) {
    return getPlanningOutlineFromEvent(event, '');
  }
  const stage = typeof event.stage === 'string' ? event.stage.trim() : '';
  const isSyntheticPlanning = event.synthetic === true && stage === 'planning';
  if (!isSyntheticPlanning) {
    return null;
  }
  return getPlanningOutlineFromEvent(event, getThinkingContent(event));
};
```

这条规则可以避免以下误判：

1. 普通模型 reasoning 中出现 `#` / `##` 被当成正式大纲。
2. 文档修订补丁中的 append heading 被理解为完整规划。
3. 完整文档正文中的标题被重复提取成顶部规划卡片。

同时还要同步收紧 `visiblePlanningOutline` 的回退来源。当前实现会按“stream event -> complete event -> artifactDocumentContent -> artifact metadata”的顺序取规划卡片，其中 `artifactDocumentContent` 和 `artifact.structure_info.heading_titles/title` 都可能只是最终文档结构，不等于本轮规划。因此需要至少增加场景判断：只有完整长文档首轮生成场景才允许从最终文档或 artifact 回退推导规划卡片；续写/修订场景不应无条件复用这条回退链。

### 5.6 P1：模型思路与系统进度在 UI 上分层展示

建议前端展示上区分三类信息：

1. 系统进度：`synthetic=true`，例如检索、规划、生成第几章，适合显示为状态条或紧凑步骤。
2. 模型思路：`synthetic=false` 且 `type=thinking`，适合显示在可折叠“思考”框中。
3. 规划大纲：`outline_role=generated_plan` 或 `continuation_plan`，适合显示为固定的大纲卡片。

这能同时满足两个录屏诉求：

1. 完整文档规划时，用户立即看到模型思路，不再只看等待秒数。
2. 规划完成后，用户看到干净、结构化的大纲卡片。
3. 续写/修订时，用户能看到思路，但不会把补丁构造过程误当成正式大纲。

## 6. 建议落地顺序

### 6.1 第一步：先修后端 P0，可见性问题优先

先改后端完整长文档链路，把“用户长时间只看到等待心跳”的问题解决掉：

1. `internal/application/service/session_agent_qa.go`
  - `chatFullDocumentOutlineWithProgress`：不要继续忽略 `ResponseTypeThinking`，改为透传非 synthetic 模型思路事件。
  - `consumeFullDocumentSectionStream`：增加 thinking emitter，透传章节生成和知识库续写生成中的 reasoning。
2. 这一步先不改 outline 字段语义，只解决“思考不可见”的主问题。
3. 同步补后端单测，覆盖“大纲阶段 thinking 先于 answer”“章节阶段 thinking 到达后重置等待心跳”两类行为。

### 6.2 第二步：收紧前端 thinking 卡片的大纲识别

在用户已经能看到模型思路后，立刻限制 thinking 卡片的误判范围：

1. `frontend/src/views/chat/components/AgentStreamDisplay.vue`
  - `getPlanningOutline` 不再对所有 thinking 内容做 Markdown 大纲兜底解析。
  - 仅信任结构化 `outline`，或明确属于 synthetic planning 的进度事件。
2. `frontend/src/views/chat/utils/planningOutline.js`
  - 保留结构化 `outline` 优先级。
  - 降低普通自由文本 `thinking` 的大纲推断优先级，避免 append heading / patch 说明误判。
3. 这一步完成后，即使后端开始透传 reasoning，也不会把思考正文直接顶成“文档规划”卡片。

### 6.3 第三步：补齐 complete 事件到前端的字段接线

当前后端 SSE complete 事件会扁平展开 `data.Extra`，但前端实时流只显式提取 `outline`。因此要补齐接包层：

1. `frontend/src/views/chat/index.vue`
  - `upsertAgentCompleteEvent` 需要保存 `outline_role`、`outline_source`、`base_outline`、`planning_outline` 等字段，而不是只拿 `outline`。
2. `frontend/src/views/chat/utils/thinkingEvent.js`
  - 若 thinking 事件后续也补充这些 metadata，需要同步存储。
3. 这一步是后续区分“生成规划”“基线结构”“续写计划”的前提，应该排在字段语义拆分之前。

### 6.4 第四步：再做续写 outline 语义拆分

等前端已经能识别角色字段后，再处理续写链路里的 `outline` 语义混用：

1. 先用兼容方式，把当前续写 complete payload 中的 `outline` 明确标记为 `outline_role=base_document`。
2. 前端确认不会再把 `base_document` 渲染成“文档规划”后，再考虑拆成 `base_outline` 和 `planning_outline`。
3. 如果本轮续写没有独立规划，就不要制造 `planning_outline`，避免引入新的伪结构化数据。

### 6.5 第五步：收紧置顶规划卡片的回退来源

当前 `visiblePlanningOutline` 会从 stream event、complete event、`artifactDocumentContent`、artifact metadata 依次回退。这里建议在第四步之后再收口：

1. 完整长文档首轮生成：允许从 complete / artifact 做有限回退，保证刷新后仍能看到规划卡片。
2. 续写、修订、补章节场景：不要再无条件从最终文档正文或 artifact 结构推导“文档规划”。
3. 若有独立 `continuation_plan`，单独展示为“续写计划”或“本轮补充计划”。

### 6.6 第六步：最后补回归测试与验收

1. 后端测试：覆盖 outline thinking、section thinking、complete extra 扁平字段。
2. 前端测试：覆盖 `thinkingEvent.js`、`planningOutline.js`、`AgentStreamDisplay.vue` 的规划识别优先级。
3. 联调验收按第 7 章执行，重点看三类场景：首轮完整生成、章节生成中途、文章续写/修订。

## 7. 验收标准

### 7.1 首次完整文档生成

发起如下请求：

```text
结合知识库内容，给出北海电厂详细的设计方案，重点方案章节要逐个系统进行展开。
```

期望表现：

1. 若模型返回 `reasoning_content`，页面 1 秒左右出现模型思路流，而不是只显示“已等待 N 秒”。
2. 系统进度仍能显示检索、规划、生成阶段，但不覆盖模型真实思路。
3. 大纲模型完成后，页面显示结构化“文档规划”卡片，包含标题和章节列表。
4. 章节正文开始生成后，思考框可折叠，正文正常流式追加。

### 7.2 章节生成阶段

期望表现：

1. 每章 answer 正文前，如果模型先输出 reasoning，前端能看到该 reasoning。
2. reasoning 到达后，不再继续刷“仍在生成中，已等待 N 秒”的重复等待提示。
3. 正文开始后，模型思路自动收口为中间步骤。

### 7.3 文章续写/修订

发起类似请求：

```text
请补充第4章 智慧运行系统设计，完善功能描述。
```

期望表现：

1. 思路过程仍能快速出现。
2. 修订补丁中的 `<append heading="...">`、章节追加推导、XML patch 片段不触发顶部“文档规划”卡片。
3. 如果展示计划，应命名为“本轮补充计划”或“续写计划”，而不是“文档规划”。
4. 基线文档的 51 个标题只作为文档版本结构信息，不作为本轮新大纲展示。

## 8. 最终结论

首个录屏的核心问题是后端完整长文档链路丢弃了模型 `reasoning_content`：模型 621ms 就开始思考，但大纲正文 80 秒后才出现；期间页面只能显示 synthetic 等待心跳。

第二个录屏说明文档编辑路径已经证明“透传模型思路”是可行的，但也暴露出展示边界问题：模型思路、补丁计划、基线大纲和正式规划大纲需要明确区分，不能都用 `thinking` 内容或通用 `outline` 字段混合驱动同一个大纲卡片。

因此，推荐的修正方向是：后端把完整长文档规划/章节生成中的模型 thinking 透传出来，同时给正式大纲增加明确 role；前端只把显式规划来源渲染为大纲卡片，把续写和修订过程中的模型思路作为普通思考内容展示。这样既能满足“规划时立即有思路输出”，也能保证“最终显示规划大纲”干净准确。
