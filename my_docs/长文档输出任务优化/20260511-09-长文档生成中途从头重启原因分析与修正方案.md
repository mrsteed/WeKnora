# 长文档生成中途从头重启原因分析与修正方案

## 1. 背景

根据截图和后台日志 `my_docs/test/9.md`，用户发起的是一次完整投标技术方案生成请求：

```json
{
  "query": "这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。",
  "agent_enabled": true,
  "intent_hint": "normal",
  "document_output_mode": "full_document",
  "web_search_enabled": true,
  "summary_model_id": "a0d12381-01c7-4243-8c48-6ce15a7d66a1"
}
```

截图中的表现是：文档已经生成到 `3.3 核心价值`，随后没有顺着后续章节继续，而是在同一个回答流里重新出现：

```text
一、项目背景与总体思路
```

这说明用户看到的不是“自动续写从下一节继续”，而是“同一个回答流中追加了一段新的文档开头”。

## 2. 关键结论

本次问题的核心不是模型单纯写错标题，而是后端智能体循环没有在已经流出 `final_answer` 内容后及时终止。

准确链路是：

```text
模型通过 final_answer 工具流式输出完整文档片段
-> 输出过程中发生 stream_error_after_answer
-> 后端已把 final_answer_tool 的正文流给前端
-> 但本轮返回 ToolCalls=0，finish_reason=stream_error_after_answer
-> analyzeResponse 没有把这类“已流式 final_answer”当作终止态
-> ReAct 进入下一轮
-> 下一轮上下文包含上一轮已生成的完整文档开头
-> 模型再次从文档标题/第一章开始输出
-> 前端继续按 answer 流追加
-> 用户看到“生成到一半，又从头开始生成”
```

因此，修正重点应放在后端终止条件、流尾异常后的状态收口和取消/完成竞态上，而不是只改前端显示。

## 3. 日志证据

### 3.1 本次请求走的是普通 Agent 长文生成，不是专用文档编辑路径

日志中请求参数为：

```text
intent_hint="normal"
document_output_mode="full_document"
agent_enabled=true
```

这意味着它没有进入前面已经治理过的 dedicated document edit path，而是走通用 ReAct Agent。

### 3.2 第 9 轮已经从文档开头重新生成

日志显示第 9 轮 LLM 结果：

```text
[Agent][Round-9] LLM responded: finish=stream_error_after_answer, content=7439 chars, tool_calls=0
```

紧接着预览内容是：

```text
# 北海电厂二期智慧电厂项目 —— 投标技术方案

---

## 一、项目背景与总体思路
```

这与截图红框后的现象一致：不是继续生成下一章，而是重新输出文档标题和第一章。

### 3.3 第 9 轮虽然已经流出了 final_answer 内容，但没有终止 Agent 循环

同一段日志还显示：

```text
[LLM Stream] First tool_calls delta ... first_name="final_answer"
[Agent][Thinking] Iteration-9 completed: ... emitted_events=map[final_answer_tool_chunk:2112 thought_chunk:1133 tool_call_pending:1]
```

这说明模型确实已经开始调用 `final_answer`，并且 `final_answer_tool_chunk` 已经被后端当作 answer chunk 发给前端。

但该轮汇总却是：

```text
tool_calls=0
finish_reason="stream_error_after_answer"
```

也就是说：

1. 流式过程中检测到了 `final_answer` 工具调用。
2. 工具参数内容被拆成 `final_answer_tool_chunk` 流式输出。
3. 流尾异常后，最终 `ChatResponse.ToolCalls` 没有保留完整工具调用。
4. `analyzeResponse()` 只在 `len(response.ToolCalls) > 0` 时进入 final_answer 终止分支。
5. 因此第 9 轮没有被判定为终止。

### 3.4 第 10 轮把第 9 轮的文档开头作为上下文继续发送

日志显示第 10 轮启动：

```text
[Agent][Round-10/40] Starting: 20 messages, 24 tools, est_tokens=34317
```

并且第 10 轮尾部消息包含：

```text
msg[19]: role=assistant, len=7439, content=# 北海电厂二期智慧电厂项目 —— 投标技术方案
```

这说明第 9 轮已经生成的文档开头被加入了下一轮上下文，模型自然会继续围绕这份“刚从头生成的文档”输出，而不是回到用户感知中的中断位置。

### 3.5 用户停止触发了取消/完成竞态

后续日志显示：

```text
Stop generation request ... message: 5151a5dc-4153-467d-a21e-60328aa992d8
Preparing assistant completion persistence ... completion_status: cancelled
...
Preparing assistant completion persistence ... completion_status: completed, finish_reason: stop
...
Persisted assistant completion state ... completion_status: cancelled
...
Persisted assistant completion state ... completion_status: completed
```

同一条 assistant message 先被写成 `cancelled`，随后又被写成 `completed`。同时：

```text
Append complete event to stream failed: failed to append event to Redis: context canceled
```

这说明停止按钮触发后，Agent 主流程仍继续走了完成收口，导致取消态和完成态竞争。这个问题不一定是“从头开始”的第一因，但会放大用户侧混乱：前端可能收不到完整 complete 事件，数据库里却可能被最终 completed 覆盖。

## 4. 代码层根因

### 4.1 `stream_error_after_answer` 被设计为可用输出，但没有被 Agent 终止逻辑接住

`streamLLMToEventBus()` 中已有逻辑：如果 stream 出错前已经产生可见回答，则不直接返回错误，而是返回一个可用结果，并设置：

```text
finish_reason = stream_error_after_answer
```

这本身是合理的：不能因为流尾异常就丢掉用户已经看到的正文。

问题在于 `analyzeResponse()` 的终止条件主要包括：

1. `finish_reason == "stop" && len(tool_calls) == 0`
2. `finish_reason == "length" && len(tool_calls) == 0`
3. `len(tool_calls) > 0 && tool_name == final_answer`

但本次返回形态是：

```text
finish_reason = stream_error_after_answer
tool_calls = 0
FinalAnswerStreamed = true
AnswerStreamed = true
```

它不命中任何终止分支，于是被当成“未完成的一轮思考”，继续进入下一轮。

### 4.2 `final_answer_tool_chunk` 是终止信号的一部分，不能只当普通 answer chunk

当前流式层已经能识别：

```go
source == "final_answer_tool"
```

并设置：

```go
FinalAnswerStreamed = true
```

这说明后端知道这些 answer chunk 来自 `final_answer` 工具，而不是普通自然语言前言。

但后续终止逻辑没有使用 `FinalAnswerStreamed`，仍依赖最终聚合出的 `ToolCalls`。在流尾异常、工具参数 JSON 未闭合、provider 未发完整 done 的情况下，`ToolCalls` 可能为空；此时如果忽略 `FinalAnswerStreamed`，就会重复进入下一轮。

### 4.3 前端只是持续追加 answer 流，不负责判断 Agent 是否应该进入下一轮

截图中的视觉效果来自前端把新 answer chunk 继续追加到同一条消息。前端这样做符合流式协议，因为后端确实还在推送新的 answer 事件。

因此前端不是根因。根因是后端本应在第 9 轮 `final_answer_tool_chunk` 已经流出并发生流尾异常时，结束本次 Agent，而不是继续第 10 轮。

### 4.4 取消与完成没有建立优先级

停止按钮后，日志出现同一 message 的 `cancelled` 与 `completed` 双重持久化。说明收口层缺少“取消态优先”保护：一旦 stop/cancel 已经写入或上下文已取消，后续 completed complete event 不应覆盖取消态。

## 5. 修正方案

## 5.1 P0：把 `FinalAnswerStreamed + stream_error_after_answer` 作为终止态

在 `internal/agent/observe.go` 的 `analyzeResponse()` 中，在普通 natural stop 分支前增加专门处理：

```go
if response.FinalAnswerStreamed &&
    response.AnswerStreamed &&
    len(response.ToolCalls) == 0 &&
    response.FinishReason == "stream_error_after_answer" {
    // 已有 final_answer 工具内容流给用户，但流尾异常导致工具调用未完整聚合。
    // 必须终止 ReAct，避免下一轮重复输出文档开头。
}
```

建议收口语义：

```text
completion_status = partial
finish_reason = stream_error_after_answer
allow_indexing = false
allow_complete = false
is_partial = true
final_answer = response.Content
```

原因：

1. 用户已经看到正文，不能丢弃。
2. 流尾异常说明内容完整性不确定，不应标记 completed。
3. 终止 Agent 循环可以避免下一轮从头生成。
4. `allow_complete=false` 可以防止把半截长文注册成完成版基线。

同时需要发送 answer done event：

```go
EventAgentFinalAnswer{Content: "", Done: true, CompletionStatus: partial, FinishReason: stream_error_after_answer}
```

这样前端能关闭 answer 流，展示“部分生成”状态，而不是继续等待下一轮。

## 5.2 P0：`streamThinkingToEventBus()` 应保留 final_answer 工具流的终止意图

当前已有：

```go
FinalAnswerStreamed: finalAnswerStreamed || llmResult.FinalAnswerStreamed
```

建议补强两点：

1. 当收到 `source=final_answer_tool` 的 answer chunk 时，记录该轮已经进入 terminal final answer stream。
2. 当后续收到 stream error 且没有完整 `ToolCalls` 时，不要让该信息丢失。

可以增加更明确的字段或注释，例如：

```go
FinalAnswerStreamed = true // means terminal final_answer tool args have already been streamed
```

并由 `analyzeResponse()` 使用该字段终止循环。

## 5.3 P0：取消态优先，禁止 completed 覆盖 cancelled

在 `completeAssistantMessage()` 或调用层增加保护：

```text
如果当前 message 已经是 cancelled/failed，且新的 complete event 不是同一请求的更高优先级恢复事件，则不要覆盖为 completed。
```

推荐优先级：

```text
cancelled > failed > partial > completed
```

更具体地说：

1. 用户主动 stop 后，后续 Agent complete event 如果来自同一个 request 且 ctx 已取消，应降级为 cancelled 或直接忽略。
2. `emitCompletionEvent()` 在 `ctx.Err()!=nil` 或 `state.IsComplete=false` 时，不应发 completed。
3. `handleComplete()` 追加 complete SSE 时，如果 request context 已取消，应用 background context 或有序关闭机制保证状态事件能写入；但数据库状态不能被错误覆盖。

## 5.4 P1：长文档完整生成应走文档 artifact 管线，而不是普通 Agent 多轮最终回答

本次请求虽然带了：

```text
document_output_mode = full_document
```

但 `intent_hint=normal`，所以走了通用 Agent。通用 Agent 的 ReAct 轮次适合问答和检索，不适合长文档从 0 到 1 的稳定生成。

建议对以下请求建立长文档生成专用路径：

```text
agent_enabled=true
document_output_mode=full_document
query 包含 输出完整技术方案 / 完整文档 / 投标技术方案 / 方案文档
```

专用路径应具备：

1. 明确的大纲生成阶段。
2. 分章节生成阶段。
3. 每一章节作为可合并片段注册到 `chat_document_artifacts`。
4. 超长时以 `partial + can_continue` 收口，而不是让同一 ReAct answer 流持续多轮追加。
5. 后续继续生成时从 artifact 的末尾窗口续写，而不是从 Agent 历史里猜测续写位置。

## 5.5 P1：对“文档头重复/章节编号回退”做流式级别拦截

仓库中已有续写质量检测：

```text
ChatDocumentQualityIssueDuplicateDocumentHead
ChatDocumentQualityIssueSectionNumberReset
ChatDocumentQualityIssueLowNoveltyDelta
```

但本次问题发生在普通 Agent answer 流阶段，尚未进入 artifact 质量治理。

建议在 Agent answer 流中增加轻量保护：

1. 当前 answer 已经输出超过一定长度后，如果新一轮 answer 又以 `# 标题`、`## 一、`、`## 1.` 等文档头模式开头，判断为疑似重启。
2. 若上一轮 answer 已含相同标题，立即停止后续追加，标记为 `partial_duplicate_head`。
3. 对用户提示：检测到模型重新输出文档开头，已暂停生成，请从当前版本继续或重新生成。

这不是根修，但能在模型再次跑偏时减少污染。

## 5.6 P1：前端展示层区分“同一轮续写”和“新一轮重启”

前端可以做防御性优化，但不应作为主要修复：

1. 如果同一 assistant message 中出现多个 answer event ID，且新 event 以文档标题/第一章开头，可以给出“疑似重新开始”的状态提示。
2. 对 partial answer done event 明确展示“生成中断，已保留当前内容”。
3. 不要在 complete event 丢失时无限展示 loading。

前端只能降低困惑，不能替代后端终止 ReAct 循环。

## 6. 推荐落地顺序

### 阶段一：修复 ReAct 终止条件

目标：避免 `final_answer_tool_chunk` 已流出后继续下一轮。

改动点：

1. `internal/agent/observe.go`
   - 新增 `FinalAnswerStreamed + stream_error_after_answer` 终止分支。
2. `internal/agent/observe_test.go`
   - 新增回归测试：
     - 输入 `ChatResponse{Content: "# 文档...", FinishReason: "stream_error_after_answer", AnswerStreamed: true, FinalAnswerStreamed: true, ToolCalls: nil}`
     - 断言 `isDone=true`
     - 断言 `completionStatus=partial`
     - 断言不会继续下一轮。

### 阶段二：修复取消/完成竞态

目标：用户点击停止后，状态不能被后续 completed 覆盖。

改动点：

1. `internal/application/service/session/qa.go` 或对应 completion 持久化入口。
2. `internal/agent/finalize.go` 的 complete event 生成逻辑。
3. 增加测试：同一 message 先 cancelled 后 completed，最终状态仍为 cancelled。

### 阶段三：长文档 full_document 专用生成管线

目标：把长文档生成从普通 ReAct answer 流中拆出来，以 artifact 分段生成和续写。

改动点：

1. 请求路由：识别 `document_output_mode=full_document` 的长文档生成。
2. 后端：按大纲/章节生成 artifact partial。
3. 前端：基于 artifact 展示完整文档和继续生成入口。

### 阶段四：重复文档头质量拦截

目标：即使模型异常从头输出，也不要污染当前消息。

改动点：

1. Agent answer stream 中维护已输出标题摘要。
2. 新 answer event 开头命中文档头重复时，停止或降级为 partial。
3. 给用户明确提示。

## 7. 验收标准

修复后应满足：

1. 发生 `stream_error_after_answer` 且已经有 `FinalAnswerStreamed=true` 时，Agent 必须停止本次 ReAct 循环。
2. 前端不应在同一条回答中继续追加第二份从标题开始的文档。
3. 此类流尾异常应以 `partial` 状态收口，不应标记为 completed。
4. 用户点击停止后，最终消息状态应保持 cancelled，不应被后续 completed 覆盖。
5. `document_output_mode=full_document` 的长文档生成应能注册 artifact；生成不完整时应提供继续生成能力，而不是把半截长文当普通问答落库。
6. 回归测试应覆盖：
   - final_answer 工具流式输出后 stream error 不再进入下一轮。
   - stream error after answer 以 partial done event 关闭流。
   - stop/cancelled 不被 completed 覆盖。
   - answer 流重复文档头被识别为异常。

## 8. 最终结论

截图中的“生成到一半，又从头开始生成”由后端 Agent 循环失控引起。

最关键的证据是日志中第 9 轮已经出现：

```text
final_answer_tool_chunk:2112
finish_reason="stream_error_after_answer"
tool_calls=0
content preview="# 北海电厂二期智慧电厂项目 —— 投标技术方案"
```

这表示模型已经通过 `final_answer` 工具把文档内容流式发出，但流尾异常导致最终工具调用没有完整聚合；后端没有把 `FinalAnswerStreamed=true` 当作终止信号，于是继续第 10 轮，模型又从文档开头生成。

因此最优先修复是：

```text
FinalAnswerStreamed + stream_error_after_answer 必须终止 Agent 循环，并以 partial 状态收口。
```

随后再治理 stop/cancelled 与 completed 的状态竞态，并把 `full_document` 长文档生成迁移到 artifact 专用管线。
