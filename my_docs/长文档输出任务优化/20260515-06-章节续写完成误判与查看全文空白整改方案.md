# 章节续写完成误判与查看全文空白整改方案

## 1. 结论

基于本次日志，本轮“请为第7章 AR眼镜智能作业方案补充功能清单”的模型输出层面已经自然结束，不属于上一轮那种 `max_completion_tokens=2048` 导致的长度截断问题。

本次关键证据是：

1. dedicated document edit 实际请求上限为 `8192`，且 `request_max_completion_tokens_matches_budget=true`。
2. 模型流式结束原因是 `finish=stop`，不是 `length`。
3. 本轮 `completion_tokens=1556`，没有跑满 `8192`。
4. 后端最终把消息持久化为 `completion_status=partial`、`finish_reason=revision_too_short`、`document_generation_status=needs_review`。
5. artifact 已注册为 V3，但质量问题包含 `revision_too_short`，因此系统提示“部分完成/建议补齐缺失章节”。

因此，这次现象的准确判断是：

**章节续写正文看起来已经生成完成，但后端把“章节级 delta 补丁较短”误当成“整篇修订结果过短”，触发 `revision_too_short` 质量闸，并将一次自然完成的章节修订降级为 partial/needs_review。由于该状态又阻断了最终全文快照交付，前端点击“查看全文”无法稳定展示完整文档。**

这不是模型没有继续写完，而是完成判定、修订质量闸、全文快照展示三者的状态边界没有分清。针对“修订某个章节”“补充某个章节”“替换某个章节”这类明确章节范围的操作，后端不应再校验补充内容长度；只需要确认目标章节定位正确，并把模型输出直接插入或替换到该章节的正确位置。

## 2.4 “查看全文”没有输出全文的直接触发条件

当前后端完整文档交付逻辑对 revise artifact 很严格。`buildFinalDocumentDelivery` 对 `operation=revise` 有三层门禁：

1. `document_generation_status` 必须是 `completed`。
2. artifact `status` 必须是 `available`。
3. artifact `completion_status` 如果非空，必须是 `completed`。

本轮 artifact 状态为：

1. `completion_status=partial`
2. `document_generation_status=needs_review`
3. 质量问题包含 `revision_too_short`

因此 complete 事件不会携带：

1. `final_document_mode`
2. `final_document`
3. `final_document_artifact_id`

前端收到的 `chat_document_artifact` 也主要是元数据，流式 complete 事件中的 artifact metadata 不包含 `content_snapshot`。前端点击“查看全文”时，只能再通过 `getChatDocumentArtifact(artifact_id)` 拉取完整 artifact 的 `content_snapshot`。

如果这一步拿不到 `content_snapshot`，或 artifact 快照本身只保存了 delta 片段而不是合并后的整文，就会表现为“点击查看全文没有输出全文”。

## 3. 根因分析

## 3.1 根因一：章节级 delta 输出被整文长度规则误判

当前 `evaluateRevisionArtifactQuality` 对 revise 操作有如下规则：

```go
if snapshotLen < int(float64(baseLen)*0.4) && !strings.Contains(options.UserQuery, "精简") && !strings.Contains(options.UserQuery, "摘要") && !strings.Contains(options.UserQuery, "压缩") {
    status = types.ChatDocumentArtifactStatusPartial
    issues = append(issues, types.ChatDocumentQualityIssueRevisionTooShort)
}
```

这条规则适合判断“整篇文档重写后明显缺失大段内容”，但不适合直接判断“delta-only 章节追加”。

本轮用户只要求给第 7 章补充功能清单。对于这种请求，合理输出本来就可能只有几千字，远短于 2.3 万字的基线全文。如果质量闸拿 delta 片段长度与基线全文长度比较，就会天然触发 `revision_too_short`。

正确的判断对象应该是：

1. 是否识别到这是章节级修订或章节级补充。
2. 是否能确定目标章节位置。
3. 插入、追加或替换操作是否执行到目标章节内。
4. 合并后的 artifact 快照是否仍保留基线全文。

不应该用“本轮模型输出长度 / 上一版整文长度”直接判断是否未完成。对于章节级操作，也不应该再用“补充内容自身长度”判断质量是否合格；用户要求补充一个短清单、一个短段落或一个长小节都应被允许，系统职责是准确合并，而不是替用户规定补充字数。

## 3.2 根因二：完成状态和质量复核状态被耦合

本轮模型结束状态是 `stop`，请求预算也匹配，因此生成过程应当可以进入 completed。

但 `dedicatedDocumentEditQualityOutcome` 遇到 `revision_too_short` 后，会把完成状态降级为 partial：

1. `finish_reason=revision_too_short`
2. `failure_reason=revision_too_short`
3. `document_generation_status=needs_review`
4. `next_action=manual_retry`

这导致用户看到的是“没有完成”，而不是“已生成，但需复核”。

更合理的状态模型应区分两类情况：

1. **生成未完成**：`length`、超时、空输出、流中断、请求预算不匹配。这类应为 `partial/continuing`，建议继续生成。
2. **生成已完成但需复核**：自然 `stop`，内容已落 artifact，但存在质量告警。这类应为 `completed/needs_review` 或 `completed` 加质量告警，不应提示继续补缺失章节。

本轮属于第二类。

## 3.3 根因三：needs_review 状态阻断了全文快照交付

为了避免把不可信整文直接作为最终答案，后端目前会在 revise artifact 非 completed/available 时抑制 `final_document_mode`。

这个思路本身是安全的，但副作用是：

1. 用户不能在完成事件中直接拿到全文。
2. 前端只能依赖点击时 GET artifact。
3. 如果 GET 返回的是元数据、空快照、或 delta 快照，用户就看不到全文。
4. UI 文案又显示“查看全文”，但实际没有可展示内容，体验上像是按钮失效。

安全策略应从“禁止查看全文”调整为“允许查看快照，但标明需复核”。

也就是说：

1. 不要把 needs_review artifact 自动设为可信最终答案。
2. 不要自动索引。
3. 不要自动作为下一轮基线，除非用户确认。
4. 但应允许用户查看合并后的完整快照，便于人工判断。

## 3.4 根因四：artifact 快照完整性缺少强校验和观测

本次日志有 `final_answer_len=2785`，但没有直接打印：

1. `base_snapshot_char_count`
2. `raw_delta_char_count`
3. `merged_snapshot_char_count`
4. `artifact_snapshot_char_count`
5. `patch_detected`
6. `patch_valid`
7. `target_heading_found`
8. `merge_quality_issues`

因此排查时无法只靠日志确认 artifact 中保存的是：

1. 合并后的完整文档。
2. 目标章节片段。
3. 原始 `<document_patch>`。
4. 或者普通 delta 文本追加结果。

这也是“修订文本看起来完成，但查看全文为空/不完整”难以一眼定位的原因。

## 4. 修正目标

本次整改目标是明确章节级修订的处理边界：**只要用户目标指向某个章节、某个小节或某个标题范围，后端就不再验证补充内容长度，直接按用户指定的合并模式插入、追加或替换到目标章节正确位置。**

整改后应把章节修订链路拆成三个独立判断：

1. **生成是否完成**：由模型 `finish_reason`、请求预算、流状态决定。
2. **合并是否可靠**：由 patch 解析、目标标题匹配、合并结果结构决定。
3. **快照是否可展示/可作为基线**：由 artifact 快照完整性和质量状态决定。

修正后应达到：

1. `finish_reason=stop` 且请求预算匹配时，不因 delta 短而提示“未完成”。
2. 章节级修订、章节级补充、章节级替换不做补充内容长度校验。
3. delta-only 章节追加只要能确定合并到目标章节，就生成完整 artifact 快照。
4. `revision_too_short` 只保留给整文级 revise/full 输出，不能作用于确定目标章节的 append/replace/insert_after。
5. needs_review artifact 可以点击查看全文快照，但不能自动索引或自动设为可信基线。
6. 前端在快照缺失时给出明确提示，并记录可排查日志。

## 5. 落地步骤

## 5.0 建议实施顺序

建议按“先解除误判，再保证快照，再修展示，最后补观测”的顺序实施，避免一次性改动过大导致状态链路更难排查。

1. **P0-1：先移除章节级长度校验。** 在 artifact 质量判断中识别章节级 append/replace/insert_after，只要目标章节可定位，就跳过 `revision_too_short`，不再用补充内容长度或相对基线比例判定完成度。
2. **P0-2：保证章节合并位置正确。** 优先加固 `mergeChatDocumentRevisionDelta` 和 patch selector 命中逻辑，确保追加内容进入目标章节末尾，替换内容覆盖目标章节范围，未命中时才标记 `target_section_uncertain`。
3. **P0-3：保证 artifact 保存完整快照。** 修订消息正文可以保留 delta，但 artifact `ContentSnapshot` 必须保存合并后的完整文档，作为“查看全文”和后续基线的唯一来源。
4. **P0-4：恢复查看全文链路。** 对有完整快照的 needs_review/partial artifact，也允许返回 `final_document_artifact_id` 或通过 GET 拉取快照，前端点击“查看全文”必须可展示。
5. **P1-1：调整完成状态与文案。** `finish_reason=stop` 且合并成功时，不再显示“部分完成/补齐缺失章节”；如存在非阻断告警，显示“已生成，需复核”。
6. **P1-2：补充日志与测试。** 增加目标章节命中、合并动作、快照长度、质量问题日志，并补齐章节补充不校验长度、章节替换不校验长度、全文修订仍保留必要保护的测试。

## 5.1 后端步骤一：移除章节级修订的长度校验

修改位置：

1. `internal/application/service/chat_document_artifact.go`
2. `internal/application/service/chat_document_patch_merger.go`
3. `internal/types/chat_document_artifact.go`

建议新增一个章节修订判定上下文，重点不再是补充内容长度，而是判断本轮是否属于确定目标章节的局部编辑：

```go
type revisionArtifactQualityContext struct {
   OutputMode         string
   Operation          string
   MergeMode          string
   TargetHeading      string
   PatchDetected      bool
   PatchValid         bool
   TargetHeadingFound bool
   PatchAction        string
   MergeQualityIssues []string
}
```

然后调整 `evaluateRevisionArtifactQuality` 或在调用它之前短路：

1. 如果 `output_mode=delta_only/delta`，且本轮是章节级 append/replace/insert_after，直接跳过 `revision_too_short`。
2. 如果请求中存在 `document_target_heading`，或上下文解析出稳定目标章节，直接跳过补充内容长度校验。
3. 如果 patch selector 命中目标标题，直接执行插入或替换，不再比较 raw delta 长度、补充片段长度、模型输出长度。
4. 如果目标标题未命中，只记录 `target_section_uncertain` 或 `delta_merge_uncertain`，进入人工复核；仍不应把问题归因为 `revision_too_short`。
5. `revision_too_short` 仅保留给整文级 revise/full 输出，即用户要求重写全文、输出完整文档或未指定章节范围的整体修订。

建议伪代码：

```go
if isSectionScopedRevision(options, mergeResult) {
   return revisionArtifactQualityResult{ShouldCreate: true}
}

// 仅整文级修订继续执行原有长度保护。
if isFullDocumentRevision(options) && snapshotLen < int(float64(baseLen)*0.4) {
   ...
}
```

验收标准：

1. 第 7 章追加功能清单，无论输出是 300 字、2000 字还是 8000 字，都不因长度被标记为未完成。
2. 第 7 章替换为一段短文本，也不因短于原章节或短于全文被标记为未完成。
3. 目标章节未命中时提示“合并位置需复核”，不提示“内容过短”。
4. 整篇文档重写只输出几段内容，仍可保留整文级 `revision_too_short` 保护。

## 5.2 后端步骤二：让 patch 合并结果显式返回诊断信息

当前 `mergeChatDocumentRevisionDelta` 只返回 `Snapshot` 和 `QualityIssues`，建议扩展为：

```go
type chatDocumentRevisionMergeResult struct {
    Snapshot            string
    QualityIssues       []string
    PatchDetected       bool
    PatchValid          bool
    TargetHeadingFound  bool
    AppliedOperations   []string
    BaseSnapshotChars   int
    DeltaChars          int
    SnapshotChars       int
}
```

在 `applyChatDocumentStructuredPatch` 中记录：

1. 每个操作是 `append`、`replace` 还是 `insert_after`。
2. selector 是否命中目标标题。
3. 是否发生歧义命中。
4. 是否退化为文末追加。

验收标准：

1. 日志能明确看到本轮是 `append` 到 `## 第7章 AR眼镜智能作业方案`。
2. 如果目标标题未命中，日志出现 `target_heading_found=false` 和 `delta_merge_uncertain`。
3. 如果成功命中目标章节，质量闸不能再根据 raw delta 长度、补充片段长度或相对基线比例判 partial。

## 5.3 后端步骤三：拆分“生成完成”和“质量需复核”状态

修改位置：

1. `internal/application/service/session_agent_qa.go`
2. `internal/handler/session/qa.go`
3. `frontend/src/views/chat/utils/documentCompletion.js`

建议规则：

1. `finish_reason=length`、首 token 超时、空输出、流错误、请求预算不匹配，仍然判为 `completion_status=partial` 或 failed。
2. `finish_reason=stop` 且请求预算匹配时，生成状态先判 completed。
3. 章节级修订不产生 `revision_too_short`；如果仍存在位置不确定、结构异常等问题，只进入复核，不降级为“未完成”。
4. 如果仍存在 `delta_merge_uncertain`、`target_section_uncertain`、`markdown_structure_invalid` 等问题，可设置 `document_generation_status=needs_review`，但不要把 `completion_status` 改为 partial。
5. needs_review 的 `next_action` 应为 `manual_review`，不是 `manual_retry`。

建议新增或复用文案：

```text
文档正文已生成，但存在质量告警，建议人工复核后再设为基线。
```

不要提示：

```text
本次修改结果明显短于上一版，建议继续补齐缺失章节后再作为新基线。
```

除非确实是整文合并结果缺失。

验收标准：

1. 本轮同类日志应持久化为 `completion_status=completed`。
2. 如果只有轻微质量告警，`document_generation_status=needs_review` 可以保留，但 UI 不应显示“部分完成”。
3. `allow_indexing` 可继续为 false，防止未复核版本进入知识索引。

## 5.4 后端步骤四：artifact 必须保存合并后的完整快照

修改位置：

1. `internal/application/service/chat_document_artifact.go`
2. `internal/handler/session/qa.go`

对于 revise + delta 输出，artifact 保存规则应固定为：

1. `message.Content` 保存本轮用户可见的修改片段或 patch。
2. `artifact.ContentSnapshot` 保存合并后的完整文档快照。
3. `snapshot_char_count` 使用合并后快照长度。
4. 若合并失败，才允许保存“基线 + delta 文末追加”的降级快照，并记录 `delta_merge_uncertain`。
5. 不允许 artifact 的 `content_snapshot` 只保存原始 delta，除非没有 base artifact。

补充日志：

```text
[ChatDocumentArtifact][RevisionMerge] session_id=... message_id=... base_artifact_id=... operation=revise output_mode=delta_only target_heading="## 第7章 AR眼镜智能作业方案" merge_mode=append_to_section patch_detected=true patch_valid=true target_heading_found=true raw_delta_chars=2785 base_snapshot_chars=23267 merged_snapshot_chars=26052 quality_issues=[]
```

验收标准：

1. GET 新 artifact 时，`content_snapshot` 包含第 1 到第 9 章。
2. `content_snapshot` 中第 7 章包含新增“功能清单”。
3. `content_snapshot` 不包含 `<document_patch>` 标签。
4. `snapshot_char_count` 大于或接近上一版基线长度，而不是等于 `final_answer_len`。

## 5.5 后端步骤五：允许 needs_review artifact 查看全文快照

修改位置：

1. `internal/handler/session/qa.go` 中的 `buildFinalDocumentDelivery`
2. `internal/handler/session/agent_stream_handler.go`
3. `internal/handler/session/stream.go`

当前 revise artifact 只要不是 completed/available 就完全不交付全文信息。建议调整为两层能力：

1. **promote as final answer**：仍要求 completed/available。
2. **view snapshot**：只要 artifact 有非空 `ContentSnapshot`，就允许返回 `final_document_artifact_id`，必要时返回 `fetch_artifact_snapshot`。

建议策略：

4. 前端展示时标注“需复核”，但允许查看。

验收标准：

1. 本轮 V3 即使仍是 needs_review，也能点击“查看全文”加载 artifact 快照。
2. complete 事件中至少有 `final_document_artifact_id`。
3. 不会把 needs_review 内容自动放入普通答案正文。
4. 不会自动索引 needs_review artifact。

## 9. 本次问题的最终判定口径

面向用户和产品侧，建议统一口径：

1. **功能续写是否完成**：从模型流式输出看，本轮已自然完成；从系统当前状态看，被质量闸误判为未完成。
2. **为什么提示没完成但文本像完成了**：因为当前 `revision_too_short` 用整文长度规则判断章节 delta，把短补丁误认为缺失整文。后续章节级修订或补充不再验证补充内容长度。
3. **为什么查看全文没有输出全文**：因为 artifact 被标记为 partial/needs_review 后，后端没有在 complete 事件交付全文快照；前端点击时若没有成功获取 `content_snapshot`，就没有可渲染的全文。
4. **应该怎么修**：将生成完成、合并质量、快照展示拆开；章节 delta 成功定位后直接插入或替换到目标章节，不做长度校验；needs_review 允许查看全文但不自动索引和设为可信基线。

## 10. 优先级

P0 必须修：

1. delta-only 章节追加、章节替换、章节后插入不再执行补充内容长度校验。
2. artifact `ContentSnapshot` 必须保存合并后的完整文档。
3. 点击“查看全文”必须能从 artifact 拉取并展示快照。

P1 应尽快修：

1. completed + needs_review 的状态和文案区分。
2. complete 事件返回 `final_document_artifact_id` 或 `can_view_snapshot`。
3. 增加 merge 诊断日志。

P2 可后续优化：

1. 新增 artifact snapshot 专用接口。
2. 增加人工确认后设为基线的显式操作。
3. 对不同质量问题做更细粒度的 UI 文案。

## 11. 验收标准

完成整改后，复现同类请求应满足：

1. 日志中 `finish_reason=stop` 时，不再出现 `finish_reason=revision_too_short` 覆盖自然结束原因。
2. 如果章节追加、章节替换或章节后插入成功合并，消息不显示“部分完成”。
3. artifact V3 的 `content_snapshot` 是完整文档，不是 delta 片段。
4. 点击“查看全文”能显示完整文档。
5. 第 7 章中能看到新增功能清单。
6. 章节级补充内容无论长短，都不会触发 `revision_too_short`。
7. 存在质量告警时，UI 显示“需复核”，而不是“继续补齐缺失章节”。
8. needs_review artifact 不自动索引。
9. 刷新页面后，历史版本链仍可查看全文。
