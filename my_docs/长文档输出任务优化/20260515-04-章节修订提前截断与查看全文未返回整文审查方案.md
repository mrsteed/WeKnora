# 章节修订提前截断与查看全文未返回整文审查方案

## 1. 审查结论

基于本次日志和当前代码实现，这两个现象并不是彼此独立的问题，而是同一条 dedicated document edit 链路在“结果收口”和“整文交付”两端都没有收紧导致的。

### 1.1 针对某一章的修改为什么没有真正完成，而是被提前终止

这次“补充第5章 智慧安防系统方案的功能清单”不能简单归因为“模型没有遵守 8192 的最大输出长度”，而是至少存在两层上限错配：

1. dedicated edit / long document 路径已经正常启动。
2. 新增日志 `[DocumentBudget] source=capability ... section_tokens=4096 ... continuation_tokens=4096 ... max_output=8192` 说明：`8192` 只是模型 capability 上限，当前服务端真正下发给章节生成的预算只有 `4096`。
3. `source=capability` 进一步说明这轮并没有把预算协商结果提升到 `8192`，而是停留在静态预算映射阶段。
4. 因此，模型并不是“没有按 8192 输出”，而是后端根本没有按 `8192` 发起章节 / 续写请求。
5. 现场另一组日志若仍显示 `[LLM Usage] completion_tokens=2048` 且 `finish_reason=length`，则说明在 `4096` 之外还存在第二层实际请求裁剪或供应商侧硬上限，但这已经是后续链路问题，不改变“首个截断点发生在服务端预算没有跟随 capability.MaxOutputTokens 提升”这一结论。
6. 在结果收口阶段，后端仍把这次结果按 `completion_status=completed` 持久化；artifact 注册后又根据快照质量把该版本判成 `partial`，并打上 `revision_too_short`。

所以用户看到的不是“系统明确告诉你这次只完成了一部分”，而是：

1. capability 日志显示模型支持 `8192`。
2. 实际章节预算却只有 `4096`。
3. 消息完成态被写成 completed。
4. 版本卡状态又显示“部分完成”。

这说明本次章节修订的真实结果应当被归类为：

**服务端没有把模型 `max_output=8192` 兑现为实际章节请求预算，导致生成链路先天更早触发 `finish_reason=length`；随后 dedicated edit 的消息完成态又没有按 partial/manual_retry 收口。**

### 1.2 点击“查看全文”为什么没有把完整文章内容返回

“查看全文”按钮当前并不是去重新拼接消息正文，也不是临时从基线文档和 patch 现场重算整文，而是优先依赖后端 complete payload 或 artifact 中已经准备好的“最终整文快照”。

本次链路里，后端实际交给前端的不是一个可靠的整文权威结果，而是：

1. assistant message 持久化内容长度只有 3398 字符。
2. artifact 注册后被判定为 `partial`，并带 `revision_too_short`。
3. complete/stream payload 中的 `final_document_mode`、`final_document`、`final_document_artifact_id` 最终都仍然依赖 artifact 的 `content_snapshot`。
4. 因为这份 artifact 快照本身不是已经确认完成的整文权威版本，所以前端“查看全文”只能展示这份不完整快照，而不是整篇文章。

因此第二个问题的本质不是前端按钮没有请求数据，而是：

**后端没有在这次章节修订完成时，稳定地产生一个可供“查看全文”消费的完整整文快照。**

## 2. 日志证据链

## 2.1 dedicated edit 实际已经进入执行

日志显示：

1. `document_edit_stream_started`
2. `document_edit_first_visible_stream`
3. `document_edit_first_content`

说明这次请求并没有再次误入 continuation completed 分支，而是已经进入 dedicated document edit 主路径。

## 2.2 预算日志已经表明 `8192` 不是实际请求上限

新增关键日志：

1. `[DocumentBudget] source=capability ... section_tokens=4096 ... continuation_tokens=4096 ... context_window=64000 max_output=8192`

这行日志对应的不是“已经协商完成并按 8192 发起请求”，而是：

1. `source=capability` 代表当前只是把模型能力写入 budget。
2. 真正的章节预算字段仍然是 `section_tokens=4096`、`continuation_tokens=4096`。
3. `max_output=8192` 在当前实现里只是 capability ceiling，不是本轮 section request ceiling。

因此，从这条日志本身就可以直接否定一个错误前提：

**这次请求并不是“已经协商到 8192，但模型没按 8192 输出”；而是“服务端只给了 4096 的章节预算，同时把 capability 的 8192 一并记录了下来”。**

## 2.3 模型最终确实被长度上限截断，不是自然完成

关键日志：

1. `[LLM Usage] completion_tokens=2048`
2. `document_edit_complete ... finish_reason=length`

这两行已经足够说明：

1. 本轮生成被 token 上限截断。
2. 不是 `finish_reason=stop` 的自然结束。
3. 当前结果天然更接近“部分完成”而不是“完整完成”。
4. 但它们还不足以单独证明“模型违背了 8192”，因为在当前代码里，真正该对齐的是 section request 的 `MaxCompletionTokens`，而不是 capability 日志里的 `max_output` 字段。

## 2.4 持久化层仍把消息写成 completed

关键日志：

1. `Preparing assistant completion persistence ... completion_status: completed, finish_reason: length`
2. `Persisted assistant completion state ... completion_status: completed, finish_reason: length`

说明 dedicated edit 在 `finish_reason=length` 下没有把消息降级为 partial，而是直接落成 completed。

## 2.5 artifact 层又把结果判成 partial

关键日志：

1. `Registered chat document artifact ... operation: revise`
2. `[LongDocument] persistence summary ... quality_issues: [markdown_heading_normalized revision_too_short]`

结合当前 artifact 逻辑可知：

1. `revision_too_short` 会让 revise artifact 进入 `partial`。
2. 前端卡片状态显示来源正是 artifact 的 `status`。

所以这次请求最终形成了一个明显的状态分裂：

1. message completion 是 completed
2. artifact status 是 partial

## 3. 代码对应关系

## 3.1 为什么日志里的 `max_output=8192` 没有变成实际章节请求上限

在 `internal/application/service/session_agent_qa.go` 中，真实预算链路是：

1. `fallbackDocumentGenerationBudget(...)` 先把章节 / 续写默认预算设为 `4096`。
2. `applyStaticModelCapabilityToBudget(...)` 在 `capability.MaxOutputTokens >= 4096` 时，只会执行：
   - `SectionMaxCompletionTokens = max(existing, 4096)`
   - `ContinuationMaxCompletionTokens = max(existing, 4096)`
3. 随后又执行：
   - `SectionMaxCompletionTokens = min(current, capability.MaxOutputTokens)`
   - `ContinuationMaxCompletionTokens = min(current, capability.MaxOutputTokens)`

这意味着：

1. 当 fallback baseline 本来就是 `4096` 时，`capability.MaxOutputTokens = 8192` 不会把 section/continuation 自动抬升到 `8192`。
2. `8192` 只起“上限夹子”作用，没有起“默认预算提升”作用。
3. 所以日志里会同时出现：
   - `max_output=8192`
   - `section_tokens=4096`
   - `continuation_tokens=4096`

这正是当前问题的第一现场根因：

**服务端把模型的 `MaxOutputTokens` 记录下来了，但没有把它兑现成章节生成请求的实际 `MaxCompletionTokens`。**

## 3.2 为什么这轮大概率没有进入真正的预算协商

在同一文件中：

1. `resolveDocumentGenerationBudget(...)` 会先调用 `resolveStaticDocumentGenerationBudget(...)`
2. 再决定是否调用 `negotiateDocumentGenerationBudget(...)`
3. `shouldNegotiateDocumentGenerationBudget(...)` 只有在以下场景才会返回 true：
   - `DocumentOutputMode == full`
   - `AutoContinue == true`
   - `DocumentIntent == continue`
   - `profile.AutoContinue == true`

这说明 revise / delta 类章节修订如果既不是 full output，也不是 continue / auto-continue，就很可能根本不会进入预算协商。

因此，当前日志中的：

1. `source=capability`
2. 没有 `source=negotiated`

本身就已经说明：

**本轮并不是“协商到了 8192 却没生效”，而是“停留在静态 capability budget，实际章节预算仍是 4096”。**

## 3.3 章节修订提前终止但仍被写成 completed 的原因

在 `internal/application/service/session_agent_qa.go` 中：

1. `consumeDedicatedDocumentEditStream(...)` 在 `response.Done` 时，不区分 `finish_reason=stop` 还是 `finish_reason=length`，都会直接走完成分支。
2. `finalizeDedicatedDocumentEditAnswer(...)` 默认 `completionStatus := completed`，对 revise/delta 路径没有“length 即 partial”的专门收口。

也就是说，当前 dedicated edit 的完成判定更像是：

1. 只要上游流结束并且有内容。
2. 就把本轮当成 completed。

这与章节修订场景下的业务语义不一致，因为：

1. `finish_reason=length` 表示输出被截断。
2. 截断结果不能自动等价于“当前章节修订已经完整完成”。

## 3.4 版本卡为什么显示“部分完成”

在 `internal/application/service/chat_document_artifact.go` 中：

1. `RegisterFromAssistantMessage(...)` 会根据快照质量重新评估 artifact status。
2. `evaluateRevisionArtifactQuality(...)` 对 revise 结果存在“过短修订”保护：
   - 当 `snapshotLen < baseLen * 0.4` 时，打 `revision_too_short`
   - 同时把 artifact status 设为 `partial`

在前端 `frontend/src/views/chat/components/ChatDocumentArtifactCard.vue` 中：

1. 卡片状态文案直接取 `artifact.status`
2. `partial => 部分完成`

所以本次 UI 上的“部分完成”并不是前端误判，而是后端 artifact 质量闸的直接结果。

## 3.5 “查看全文”为什么只能拿到不完整内容

后端：

1. `internal/handler/session/qa.go` 的 `buildFinalDocumentDelivery(...)` 只会从 artifact 的 `ContentSnapshot` 派生 `final_document_mode/final_document/final_document_artifact_id`
2. `internal/handler/session/agent_stream_handler.go` 和 `internal/handler/session/stream.go` 都是把这组字段原样塞进 complete payload

前端：

1. `frontend/src/views/chat/index.vue` 的 `hydrateFinalDocumentFromComplete(...)` 先看 `payload.final_document`
2. 若没有 inline snapshot，则按 `final_document_artifact_id` 拉 artifact 再取 `content_snapshot`
3. `frontend/src/views/chat/components/botmsg.vue` 在 `document_display_mode=full` 时，也是直接显示 `final_document_content || artifact.content_snapshot`

所以“查看全文”拿不到整文，不是按钮没发请求，而是：

1. 后端没有提供一份可靠的整文快照。
2. 前端只能老老实实展示当前 artifact snapshot。

## 4. 根因归纳

本次问题至少包含四个直接根因。

### 4.1 根因一：静态预算把 `max_output` 当成 capability ceiling，而不是实际章节预算目标值

这导致：

1. 模型 capability 里明明有 `MaxOutputTokens=8192`
2. 但 section / continuation 默认预算仍停留在 `4096`
3. 日志容易让人误以为“系统已经按 8192 发起请求”

### 4.2 根因二：预算协商入口过窄，章节修订场景经常停留在 `source=capability`

这导致：

1. revise / delta 场景无法稳定把预算从 `4096` 提升到更接近模型能力上限
2. `source=capability` 日志成为常态
3. 预算协商名义存在，但对本次故障路径没有真正生效

### 4.3 根因三：dedicated edit 的终态判定只看“流结束”，没有把 `finish_reason=length` 当成截断

这导致：

1. 模型被长度上限提前截断。
2. 但消息仍然被持久化成 completed。
3. 没有显式告诉用户“这次只是部分完成，需要继续补齐”。

### 4.4 根因四：查看全文依赖 artifact 整文快照，但截断场景没有稳定产出整文权威结果

只要这次 revise 没有形成一个可信的 merged full snapshot：

1. `final_document_mode` 就不能真正代表“完整文章已可展示”
2. 前端“查看全文”自然只能看到短修订或不完整快照

## 5. 修正建议

## 5.1 P0：章节 / 续写预算默认跟随模型 `MaxOutputTokens`，不要固定停在 `4096`

建议修改 `internal/application/service/session_agent_qa.go` 中的预算映射逻辑：

1. `fallbackDocumentGenerationBudget(...)`
2. `applyStaticModelCapabilityToBudget(...)`

建议规则：

1. 当 `capability.MaxOutputTokens > 0` 且上下文窗口足够时，`SectionMaxCompletionTokens` 与 `ContinuationMaxCompletionTokens` 应默认提升到 capability 允许的上限，而不是只做 `max(existing, 4096)`。
2. 对本次 DeepSeek 能力值 `8192` 的场景，实际请求预算应跟随提升到 `8192`。
3. 若未来仍需保留平台级安全阈值，应通过单独的全局 hard cap 控制，而不是把默认 budget 固定写死在 `4096`。

这样才能满足你的目标：

**按照模型最大输出长度输出内容，而不是在服务端预算层提前截断。**

## 5.2 P0：revise / delta 场景也要允许预算协商，或直接继承 capability max output

建议修改 `shouldNegotiateDocumentGenerationBudget(...)` 或相邻调用方：

1. 对 `BaseArtifact != nil` 的章节修订请求
2. 对 `Intent = revise` 且目标是“补充某章/补写某节”的请求
3. 即使不是 `DocumentOutputMode = full`，也应允许预算协商，或直接使用 capability 驱动的高预算默认值

否则 dedicated revise 路径会持续出现：

1. capability 日志显示 `8192`
2. 实际请求仍然只发 `4096`

## 5.3 P0：把 `request_max_completion_tokens` 作为事故判定主证据，并在不匹配时阻断 completed 收口

当前代码已经有观测字段：

1. `[LongDocument] section config ... request_max_completion_tokens=...`
2. `request_max_completion_tokens_expected=...`
3. `request_max_completion_tokens_matches_budget=...`

建议在生产链路里进一步收紧：

1. 若 capability / negotiated budget 为 `8192`，则章节请求日志必须出现 `request_max_completion_tokens=8192`
2. 若最终又只在 `2048` 或其他更低值触发 `finish_reason=length`，必须额外记录“二次裁剪来源”，例如 provider cap、request overwrite、SDK fallback
3. 对 `finish_reason=length` 且 `request_max_completion_tokens < expected_capability_tokens` 的场景，不能收口为 completed

这样可以把“预算没抬上去”和“预算抬上去了但下游又被改小”明确区分开。

## 5.4 P0：dedicated edit 对 `finish_reason=length` 明确按 partial/manual_retry 收口

建议修改 `internal/application/service/session_agent_qa.go`：

1. `consumeDedicatedDocumentEditStream(...)`
2. `finalizeDedicatedDocumentEditAnswer(...)`

建议规则：

1. 对章节修订 / revise delta 请求，若 `finish_reason=length`
2. 或质量闸命中 `revision_too_short`
3. 或 patch merge 结果存在 `delta_merge_uncertain`

则不能返回：

1. `completion_status=completed`
2. `document_generation_status=completed`
3. `allow_complete=true`

建议改为：

1. `completion_status=partial`
2. `finish_reason=length` 或标准化后的 `section_generation_truncated`
3. `next_action=manual_retry`
4. `auto_continue_next=false`
5. `allow_complete=false`

这样 UI 顶部状态、版本卡状态和用户认知才能一致。

## 5.5 P0：章节修订完成后必须区分“delta answer”和“full snapshot delivery”两条输出

建议修改 dedicated edit finalize 逻辑：

1. 聊天正文仍可保留 delta/patch 作为本轮回答
2. 但只要存在 `BaseArtifact`，都应尝试构造 merged full snapshot
3. 如果 merged full snapshot 通过完整性校验，再把它作为 `final_document` / `final_document_artifact_id` 的权威来源

当前问题在于：

1. 聊天回答和 artifact 快照共用同一个 finalize 收口
2. 截断时二者一起退化

建议改成：

1. `answer_content` 允许是 delta
2. `final_document_delivery` 只允许是完整 merged snapshot
3. 若拿不到可信整文，则前端不要展示“查看全文”，而应展示“查看修改内容”或“结果未完整收口，请继续补齐”

## 7. 最终判断

针对本次日志，两个问题的答案可以直接收敛为：

1. **你看到的 `max_output=8192` 不是本轮已经协商成功的实际请求上限，而只是模型 capability。当前代码真正给 section / continuation 下发的预算仍是 `4096`，所以模型并不是“没有按 8192 输出”，而是服务端没有按 8192 请求。**
2. **章节修订并没有真正完成，而是在更低的实际请求上限处触发了 `finish_reason=length`；系统却又把消息错误收口成 completed。**
3. **“查看全文”拿不到完整文章，不是按钮本身有问题，而是后端没有在这次截断修订后稳定产出完整整文快照，前端只能展示 partial artifact 的 snapshot。**

因此，这次修复重点不在前端按钮本身，也不在继续扩大 route 守卫，而在：

1. **让章节 / 续写预算真正跟随模型 `MaxOutputTokens`，不要在 `4096` 提前卡死**
2. **让 revise 场景真正进入 budget negotiation，或至少继承 capability-driven 高预算**
3. **dedicated edit 截断场景的 completion status 收口**
4. **delta answer 与 full snapshot delivery 的分离**
5. **message status、artifact status、查看全文能力三者的统一判定**

只要这几层不统一，后续仍会继续出现：

1. 顶部提示显示已完成
2. 卡片显示部分完成
3. 查看全文又拿不到整文

这类表面看似互相矛盾、实则来自同一收口缺陷的问题。
