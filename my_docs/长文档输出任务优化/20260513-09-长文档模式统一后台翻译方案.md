# 长文档模式统一后台翻译方案

日期：2026-05-13

## 1. 结论

当前的长文档模式具备替代“后台翻译分支”的主体条件，但不建议直接删除旧分支后把翻译请求简单改投到现有 `full_document` 链路。

原因是两条链路的任务语义不同：

- 旧后台翻译分支是“按单个知识文件的全部 chunk 顺序覆盖翻译”，核心目标是忠实翻译原文并生成 Markdown 文件。
- 当前长文档模式是“围绕用户目标规划大纲、按章节生成完整文档”，知识增强路径依赖 TopK 检索证据，不保证覆盖原文件的每一个 chunk。

因此推荐方案是：以长文档模式作为唯一用户入口和唯一产物体系，把“全文翻译”改造成长文档模式下的一个 `translation` 任务类型。迁移完成后，前端不再走 `/api/v1/long-document-tasks`，后台不再保留独立的 `long_document_tasks / long_document_task_batches / long_document_artifacts` 任务体系；但在删除旧代码前，必须先补齐长文档模式的“源文件顺序快照、分段翻译、断点续写、产物存储、历史任务迁移”能力。

一句话目标：保留长文档模式的会话体验、artifact、版本链、自动续写和导出能力，移除后台翻译分支的独立任务表、独立 API、独立卡片和独立 worker。

## 2. 术语边界

本文中的“后台翻译分支”指已删除的旧异步任务链路：此前由前端显式翻译模式图标触发，调用 `/api/v1/long-document-tasks`，不属于普通 Agent/Knowledge QA 的流式回答。

本文中的“长文档模式”指会话 QA 请求中的 `document_output_mode=full_document` 及其配套能力，包括：ChatRoute 路由、`runKnowledgeGroundedFullDocumentGenerationPath` / `runDedicatedFullDocumentGenerationPath`、`chat_document_artifacts`、`chat_document_generation_runs`、自动续写、质量状态和前端 `ChatDocumentArtifactCard`。

## 3. 现状代码梳理

### 3.1 后台翻译分支

前端入口（旧链路，现已删除）：

- `frontend/src/components/Input-field.vue`：曾维护显式“长文档翻译”开关图标。
- `frontend/src/views/chat/index.vue`：曾接收翻译模式开关；开启后校验只能选一个文件、不能有图片/附件，然后调用 `createLongDocumentTask`，不进入普通 `startStream`。
- `frontend/src/api/chat/index.ts`：提供 `createLongDocumentTask / getLongDocumentTask / getLongDocumentTaskArtifact / getLongDocumentTaskBatches / retryLongDocumentTask / cancelLongDocumentTask / downloadLongDocumentArtifact`。
- `frontend/src/views/chat/components/LongDocumentTaskCard.vue`：单独展示任务状态、SSE/轮询进度、重试、取消和下载。
- `frontend/src/views/chat/components/botmsg.vue`：如果消息带 `long_document_task`，优先渲染 `LongDocumentTaskCard`，并隐藏普通 Markdown 内容。

后端入口：

- `internal/router/long_document.go`：注册 `/api/v1/long-document-tasks` 及子路由。
- `internal/handler/long_document.go`：创建任务、查询任务、查询批次、查询 artifact、下载 artifact、取消、重试、SSE 事件流。
- `internal/application/service/long_document.go`：核心服务。`CreateTask` 校验单文件知识、读取所有 chunk、按 `BatchChunkSize / BatchMaxChars` 规划批次，入队 `long_document:task`；`HandleTask` 执行每个批次，调用模型翻译，再拼装 Markdown artifact。
- `internal/application/repository/long_document.go`：独立任务、批次、产物三套 repository。
- `internal/types/long_document.go` 和 `internal/types/interfaces/long_document.go`：独立类型和接口。
- `internal/router/task.go`、`internal/router/sync_task.go`：注册 `types.TypeLongDocumentTask` 到 asynq 或同步任务执行器。
- `internal/config/config.go`：`LongDocumentConfig` 及 `WEKNORA_ENABLE_LONG_DOCUMENT_TASK_ROUTER / WORKER / ARTIFACT_DOWNLOAD`。
- `migrations/versioned/000045_long_document_tasks.up.sql`、`migrations/sqlite/000003_long_document_tasks.up.sql`：独立建表。

该链路的优点：

- 按知识文件全量 chunk 顺序处理，适合“完整翻译”。
- 任务状态、批次状态、失败批次、重试、取消比较清晰。
- artifact 是文件存储，下载大文件更自然。
- worker 可以在 HTTP 请求结束后继续执行。

该链路的问题：

- 与会话长文档体系割裂：产物不是 `chat_document_artifacts`，没有版本链、基线续写、统一完成状态和统一质量语义。
- 前端需要维护两种消息形态：`long_document_task` 卡片和 `chat_document_artifact` 卡片。
- API、repository、migration、config、worker 都是独立分支，长期维护成本高。
- `sendLongDocumentTaskEvent` 目前只看到定义，主要实际入口来自前端直接调用任务 API，说明会话流内的统一事件分支并未真正承接这条路径。

### 3.2 当前长文档模式

前端入口：

- `frontend/src/api/chat/streame.ts`：`startStream` 支持 `document_output_mode`、`intent_hint`、`base_artifact_id`、`generation_run_id`、`auto_continue` 等字段。
- `frontend/src/views/chat/index.vue`：普通发送走 `startStream`；对续写/修订会解析 `intent_hint`、`base_artifact_id`、`document_target_heading`、`document_merge_mode`。
- `frontend/src/views/chat/components/ChatDocumentArtifactCard.vue`：展示长文档 artifact，支持预览、导出、设为基线、版本链。
- `frontend/src/views/chat/components/AgentStreamDisplay.vue`、`botmsg.vue`：处理 agent 完成事件、artifact 信息、最终文档内容展示。

后端入口：

- `internal/handler/session/types.go`：QA 请求支持 `document_output_mode`、`base_artifact_id`、`auto_continue`、`generation_run_id` 等长文档字段。
- `internal/handler/session/chat_route.go`：根据 ChatRoute 决策把请求改写为 `full_document`、`knowledge_grounded_full_document`、`document_edit` 或 `knowledge_grounded_document_continuation`。
- `internal/application/service/chat_route_service.go`：路由模型和正则兜底识别“输出完整方案/报告/文档/Markdown”等请求。
- `internal/application/service/session_agent_qa.go`：`AgentQA` 中选择专用长文档路径。知识增强完整文档会先检索证据、生成大纲、逐章生成正文，最后通过 `emitFullDocumentCompletion` 发送完成事件。
- `internal/application/service/chat_document_artifact.go`：注册会话文档 artifact，支持续写、修订、合并、状态归一、质量问题记录。
- `internal/handler/session/qa.go` 和 `agent_stream_handler.go`：在完成阶段持久化 assistant message、注册 `chat_document_artifact`、索引长文档消息、把 `chat_document_artifact` 放入完成事件。
- `migrations/versioned/000048_chat_document_artifacts.up.sql`、`000049_chat_document_generation_status.up.sql`、`000050_chat_document_generation_runs.up.sql`、`000052_chat_document_generation_run_feedback.up.sql`：会话长文档 artifact、生成状态和 generation run。

该链路的优点：

- 用户体验统一：同一个对话流、同一个完成事件、同一个 artifact 卡片、同一个导出组件。
- 有 `completed / continuing / blocked / needs_review` 的文档生成状态。
- 有 `chat_document_generation_runs` 保存大纲、已完成章节、预算、运行反馈，可支撑自动续写和恢复。
- Artifact 可以作为后续修改/续写的基线。
- 已有较多质量闸：Markdown 结构、内部标签泄漏、章节完整性、artifact-first 完成语义等。

该链路的限制：

- 知识增强完整文档通过 HybridSearch TopK 取证据，默认最多 4-12 条证据，不覆盖原文件全部 chunk。
- 路由正则和提示词主要面向“方案/报告/标书/设计文档”，没有把“全文翻译”作为长文档任务类型建模。
- 当前长文档执行依赖 HTTP/SSE 对话流和服务 goroutine；虽然有 generation run 和自动续写，但并不等价于后台 worker 离线执行。
- `chat_document_artifacts.content_snapshot` 是内联正文快照，超大翻译结果需要评估数据库字段和导出策略，不能简单照搬旧文件存储能力。

## 4. 能否替代：能力矩阵

| 能力 | 后台翻译分支现状 | 长文档模式现状 | 结论 |
| --- | --- | --- | --- |
| 用户入口 | 独立翻译开关和任务卡片 | 普通对话流 + artifact 卡片 | 可统一到长文档模式 |
| 单文件全量翻译 | 读取目标 knowledge 全部 chunks，按顺序翻译 | TopK 检索证据，不保证全量 | 必须补齐源文件顺序快照能力 |
| 大文件分段 | 有批次表和 batch prompt | 按章节生成，非按源 chunk | 需新增 translation 分段策略 |
| 进度 | task/batch 状态 + SSE/轮询 | thinking/progress 事件 + generation run | 可统一，但前端展示要改造 |
| 失败恢复 | retry failed/partial/cancelled task | continuing/auto_continue/generation_run | 可统一，但需记录翻译段游标 |
| 取消 | task cancel API | stop session stream | 可统一到 stop session，但需同步 generation run 状态 |
| 产物 | 文件存储 + long_document_artifacts | chat_document_artifacts 内联快照 + 导出组件 | 可统一；超大文档需外部存储补强 |
| 历史任务 | long_document_tasks 列表恢复卡片 | message/artifact 历史加载 | 需迁移或兼容窗口 |
| 后台离线执行 | asynq worker 可离线跑 | 当前主要是请求内流式执行 | 如果仍要求离线后台，需做通用 generation-run worker，而不是保留翻译专属 worker |

判断：可以替代，但替代动作应是“把翻译能力迁入长文档模式”，不是“把旧按钮简单改为 `document_output_mode=full_document`”。

## 5. 目标架构

### 5.1 统一后的请求模型

建议在会话 QA 请求中新增显式任务字段，而不是复用自然语言判断：

```json
{
  "query": "请把这篇文档完整翻译成中文 Markdown",
  "knowledge_ids": ["knowledge-id"],
  "document_output_mode": "full_document",
  "document_task_kind": "translation",
  "translation_options": {
    "target_language": "Chinese (Simplified)",
    "source_language": "auto",
    "preserve_structure": true,
    "output_format": "markdown"
  }
}
```

后端对应扩展：

- `CreateKnowledgeQARequest` 增加 `DocumentTaskKind`、`TranslationOptions`。
- `types.QARequest` 增加同名字段，传入 `AgentQA`。
- 新增常量：`ChatDocumentTaskKindWriting = "writing"`、`ChatDocumentTaskKindTranslation = "translation"`。
- ChatRoute 对显式 `document_task_kind=translation` 不再依赖大模型路由判断，直接进入翻译专用长文档路径。

### 5.2 翻译作为长文档模式的一个专用路径

在 `session_agent_qa.go` 中新增或拆分一个专用路径：

```go
shouldUseLongDocumentTranslationPath(req, agentConfig)
runLongDocumentTranslationPath(ctx, req, eventBus, chatModel, agentConfig)
```

进入条件：

- `DocumentOutputMode == full_document`
- `DocumentTaskKind == translation`
- `len(KnowledgeIDs) == 1`
- 无图片、无附件
- knowledge parse status completed
- 输出格式为 Markdown

运行方式：

1. 读取目标 knowledge 的全部 chunks。
2. 用 chunk 顺序构造 `source_document_snapshot`，避免 HybridSearch TopK。
3. 按字符数或 token 预算切分翻译段，沿用旧链路 `BatchMaxChars` 的思想，但状态写入 `chat_document_generation_runs`，不写 `long_document_task_batches`。
4. 每段调用模型翻译，提示词沿用旧 `buildBatchPrompt` 的忠实翻译约束，并补充长文档模式已有的 Markdown 质量约束。
5. 每完成一段就流式输出并更新 generation run 的完成游标。
6. 全部完成后注册 `chat_document_artifact`，完成事件中携带 `chat_document_artifact`。
7. 如果中断或长度停止，返回 `completion_status=partial`、`document_generation_status=continuing`，前端可用自动续写继续翻译剩余分段。

### 5.3 generation run 扩展

建议新增 migration 扩展 `chat_document_generation_runs`，不要复用旧任务表：

- `task_kind`：`writing` / `translation`
- `source_knowledge_id`
- `source_snapshot_hash`
- `source_chunk_cursor_json`：记录已完成 chunk 范围或段落序号
- `target_language`
- `output_format`
- `quality_status`

也可以把部分字段写入已有 `runtime_feedback_json`，但不推荐。显式列更利于排障、迁移和后台恢复。

### 5.4 artifact 存储策略

统一产物必须落到 `chat_document_artifacts`，否则前端仍会分裂。

对于普通长度结果：继续写 `content_snapshot`。

对于超大翻译结果：建议给 `chat_document_artifacts` 增加可选外部存储字段，或增加关联内容表：

- `content_storage_backend`
- `content_storage_path`
- `content_size`
- `content_snapshot_mode`：`inline` / `external`

前端导出时优先使用 `content_snapshot`；若是 external，则通过统一 artifact 下载接口取正文。这样可以替代旧 `long_document_artifacts.file_path` 的大文件优势，同时不把旧任务体系保留下来。

### 5.5 前端统一体验

前端不再创建 `long_document_task` 消息，而是让“全文翻译”走普通会话流：

- 输入框保留一个用户可见能力入口，但语义改为“完整文档/全文翻译模式”，不再叫后台任务。
- 开启翻译时仍校验单文件、无图片/附件。
- 请求调用 `startStream`，携带 `document_output_mode=full_document`、`document_task_kind=translation`、`translation_options`。
- 结果展示使用 `AgentStreamDisplay`、普通 Markdown 流式内容和 `ChatDocumentArtifactCard`。
- 进度展示复用长文档 progress/thinking 事件，不再使用 `LongDocumentTaskCard`。
- 导出复用 `ExportDropdown`，不再调用 `/long-document-tasks/:id/download`。

## 7. 分阶段实施步骤

### 阶段 0：冻结旧入口

目标：新任务立即停止进入旧分支。

步骤：

1. 前端停止调用 `createLongDocumentTask`，统一改走会话 SSE 流。
2. 输入框保留“全文翻译”能力入口，但语义改为长文档翻译，不再保留后台任务形态。
3. 后端不再暴露 `/long-document-tasks` 兼容查询、下载、重试入口。

### 阶段 1：扩展会话请求协议

目标：让翻译任务可以通过普通会话流表达。

步骤：

1. `internal/handler/session/types.go` 增加 `DocumentTaskKind` 和 `TranslationOptions`。
2. `internal/types/qa_request.go` 增加同名字段。
3. `frontend/src/api/chat/streame.ts` 支持透传 `document_task_kind`、`translation_options`、`document_target_heading`、`document_merge_mode`。当前 `streame.ts` 类型里没有 target/merge 字段，但 `index.vue` 已构造这两个字段，建议一并修正。
4. `frontend/src/views/chat/index.vue` 中将旧翻译分支从 `createLongDocumentTask` 改为 `startStream`。
5. 对显式 `document_task_kind=translation`，后端跳过普通 ChatRoute 的“方案/报告”判断，直接进入翻译路径。

验收：打开翻译开关后，请求仍是 `/api/v1/agent-chat/:session_id` 或 `/api/v1/knowledge-chat/:session_id` 的 SSE 流，不再出现 `/api/v1/long-document-tasks` 创建请求。

### 阶段 2：实现长文档翻译专用路径

目标：用长文档模式承接旧后台翻译的核心能力。

当前落地边界：已实现首轮完整翻译、按段 partial 输出、generation run 进度记录、`generation_run_id` 回传、基于 generation run 的 translation 续写恢复，以及 `AgentQA` / `KnowledgeQA` 双入口承接。

步骤：

1. 在 session service 中新增 `shouldUseLongDocumentTranslationPath`。
2. 在 `AgentQA` 的专用路径选择中，把 translation path 放在 `knowledge_grounded_full_document` 之前，避免翻译请求被当成方案生成。
3. 实现 `loadSourceDocumentChunksForTranslation`，复用旧链路的 `ListChunksByKnowledgeID`、`uniqueChunkContent`、`buildLongDocumentSnapshotHash` 思路。
4. 实现 `planTranslationSegments`，按 chunk 顺序、字符数、模型上下文预算切分。
5. 创建/更新 `chat_document_generation_runs`，记录 `task_kind=translation`、源文件 ID、snapshot hash、目标语言、分段进度。
6. 迁移旧 `buildBatchPrompt` 为新 `buildLongDocumentTranslationSegmentMessages`，提示词要求：忠实翻译、保留 Markdown/表格/列表/引用、去页眉页脚/OCR 噪声、不添加无关解释、不泄漏内部标签。
7. 每段翻译完成后 emit answer chunk，更新 generation run 进度。
8. 完成后调用现有 `emitFullDocumentCompletion`，让 handler 注册 `chat_document_artifacts`。
9. 中断时输出 partial，并设置 `document_generation_status=continuing`；前端自动续写用 `generation_run_id` 继续剩余分段。

验收：同一份长文件翻译结果按源文顺序覆盖所有 chunk；刷新页面后可从 artifact/generation run 恢复，不依赖 `long_document_tasks`。

### 阶段 3：统一进度和续写体验

目标：替代旧任务卡片里的进度、取消、重试。

当前落地边界：已接入 translation 专用 continuation path；completion extra 会回传 `translation_progress`；前端已展示翻译段进度，并将手动重试/自动续写统一到 `generation_run_id` 驱动的继续生成链路；首段失败无 artifact 场景和页面刷新后仅依赖 `agent_complete` 事件的恢复场景都已覆盖。

步骤：

1. 将翻译分段进度转成 `thinking/progress` 事件，例如“正在翻译第 3/18 段”。
2. 在 completion extra 中增加 `translation_progress`，包括 total segments、completed segments、source chunk range。
3. 前端 `AgentStreamDisplay` 或长文档进度展示读取该结构化进度。
4. “取消任务”统一为已有 `stopSession`。
5. “重试任务”统一为对最后一次 partial/failed 消息发起继续生成，携带 `generation_run_id`。
6. 自动续写逻辑使用 `document_generation_status=continuing` 和 generation run 剩余游标，不再关心旧 task status。

验收：用户可看到翻译进度；停止后状态变为 blocked/cancelled 或 continuing；继续生成不会从头翻译。

### 阶段 4：统一 artifact 和导出

目标：删除旧 file artifact 依赖。

当前落地边界：translation 结果已统一注册为 `chat_document_artifacts`；artifact 已补充 `document_task_kind / source_title / target_language / output_format` 元数据；`ChatDocumentArtifactCard` 已展示翻译 artifact 的标题、状态、目标语言和源文件名；导出统一走 `ExportDropdown` + artifact 内容；消息历史加载已停止额外调用 `getLongDocumentTasksBySession`。超大结果的“外部存储 + storage path” 仍未落地，当前仍以 `content_snapshot` 为主，并在长文档场景下通过 artifact API 拉取完整正文。

步骤：

1. 短中等结果写入 `chat_document_artifacts.content_snapshot`。
2. 超大结果写外部存储，并在 artifact 上记录 storage path；前端导出时通过 artifact 下载接口获取正文。
3. `ChatDocumentArtifactCard` 显示翻译 artifact 的标题、状态、目标语言、源文件名。
4. 导出统一使用 `ExportDropdown`，不再调用旧 `downloadLongDocumentArtifact`。
5. 消息历史加载时只依赖 message + `chat_document_artifact`，不再额外调用 `getLongDocumentTasksBySession`。

验收：完成翻译后展示的是“文档版本”卡片，可预览、导出、设为基线、查看版本链。

### 阶段 5：历史任务迁移

目标：明确不迁移历史任务，直接删除兼容逻辑。

当前落地边界：已删除前端旧任务卡片与旧 API、后端 `/long-document-tasks` 路由与任务处理链路、旧 task type 注册、旧 long_document service/repository/types/interfaces，以及只服务旧链路的配置开关；数据库通过 drop migration 直接删除旧表，新库初始化也不再创建旧表。

步骤：

1. 不再读取 `long_document_tasks` 历史记录恢复 UI。
2. 不再保留旧任务下载、重试、取消接口。
3. 新增 drop migration 删除旧三张表。
4. 同步移除 sqlite init 中旧表定义。

验收：仓库内不再存在旧 long_document task 兼容代码；新会话和新库初始化都不再接触旧任务体系。

### 阶段 6：删除旧分支

目标：阶段 5 后的剩余收尾项。

步骤：

1. 更新 swagger/openapi、README 或开发文档，移除 `/long-document-tasks` 说明。
2. 继续补齐超大结果外部存储方案。
3. 根据需要清理遗留文档中的旧方案描述。

## 8. 测试计划

后端单元测试：

- `shouldUseLongDocumentTranslationPath`：单文件、无附件、显式 translation 才进入。

- ChatRoute：翻译请求不被误判为普通 QA 或方案生成。

## 9. 风险与注意事项

1. 不能用 HybridSearch TopK 翻译全文。翻译必须走源文件顺序快照，否则会漏段、乱序或只翻译相关证据。
2. 旧后台翻译有离线 worker 语义。如果产品仍要求用户关闭页面后继续执行，应实现“通用 generation-run worker”，而不是保留翻译专属 `long_document:task`。
3. 超大文档产物不能只依赖内联 `content_snapshot`。需要外部存储能力，或至少明确最大输出限制。
4. 当前阶段已接受历史任务直接废弃，不再提供兼容窗口或迁移脚本；上线前需明确告知该行为变更。
5. 路由保护要足够显式。`document_task_kind=translation` 应优先于“方案/报告”路由，避免全文翻译被规划成技术方案。
6. 自动续写必须基于 generation run 游标，而不是基于模型猜测“从哪里继续”。
7. 删除 migration 要谨慎。历史 migration 文件通常不改，新增迁移完成收敛。

## 10. 推荐落地顺序

按当前实际落地，顺序已经收口为：

1. 新增请求协议和 translation path。
2. 全文翻译切到统一会话流，并补齐自然语言路由。
3. 接入 generation run 进度、续写恢复和 artifact 持久化。
4. 删除前端显式翻译/自动续写图标及旧任务卡片。
5. 删除旧前后端 long_document task 体系。
6. 通过 drop migration 和 sqlite init 收口旧表。
7. 后续只保留超大结果外部存储等增强项，不再保留历史查询下载或兼容迁移窗口。

这样做能把维护面统一到长文档模式，同时避免“为了删分支而牺牲全文翻译完整性”。
