# 文档编辑无模型流式输出原因分析与解决方案

## 1. 背景

根据截图与 `my_docs/test/8.md` 日志，本次基线文档编辑请求：

```text
把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

当前表现为：

1. 聊天主区域没有看到模型真实的流式输出内容。
2. 页面只显示系统生成的等待提示：
   - 正在分析基线文档并生成修订补丁，请稍候。
   - 仍在等待模型输出首个修订片段，将在超时后自动结束本次生成。
3. 等待 30 秒后直接失败，失败原因为：
   - `first_content_timeout`
   - `timed out waiting for first document edit content after 30s`
4. 等待提示在界面上重复出现多行，用户感知为“既没有模型流式内容，又一直在重复等待”。

用户的核心诉求不是“增加等待提示”，而是“把模型的流式输出真正展示出来，而不是仅仅显示等待状态”。

## 2. 关键结论

本次问题的本质不是“模型完全没有流式输出”，而是：

1. 模型实际上已经开始流式输出，只不过输出的是 `reasoning_content`，不是 `delta.Content`。
2. 专用文档编辑链路当前只把 `ResponseTypeAnswer` 当作用户可见流式内容，直接丢弃了 `ResponseTypeThinking`。
3. `first_content_timeout` 当前盯的是“首个 answer token”，不是“首个可见流事件”或“首个模型流事件”。
4. 因此即使模型已经在流式发送 reasoning，系统仍然把它视为“还没有开始输出”，并在 30 秒后强制失败。
5. 页面上的多条“仍在等待模型输出首个修订片段”不是模型输出，而是后端心跳提示被前端按追加文本累积展示了。

所以当前用户看到的不是模型流式输出失败，而是：

```text
模型有流式信号 -> 后端丢弃 reasoning 流 -> 前端只收到等待心跳 -> 30 秒后被 first_content_timeout 终止
```

## 3. 日志证据

### 3.1 模型实际上很快就开始流式输出了

`my_docs/test/8.md` 中明确记录：

```text
14:46:50.999 document_edit_stream_started
14:46:51.491 [LLM Stream] First reasoning_content at OpenAI layer (len=6, preview="我们", elapsed_ms=492)
```

这说明：

1. 模型在 492ms 时就已经开始输出。
2. 输出类型是 `reasoning_content`，不是 `delta.Content`。
3. 从 LLM provider 角度看，这已经是流式响应开始，而不是完全静默等待。

因此，“模型为什么没有流式输出”的准确答案是：

```text
模型有流式输出，但当前系统没有把这段流式输出转成用户可见内容。
```

### 3.2 后端只接受 answer，不接受 thinking

当前 `internal/application/service/session_agent_qa.go` 中的 `consumeDedicatedDocumentEditStream()`：

```go
if response.ResponseType != types.ResponseTypeAnswer {
    continue
}
```

这段逻辑直接忽略了：

1. `ResponseTypeThinking`
2. provider 透出的 `reasoning_content`
3. 任何首个 answer 之前的模型流信号

而 `internal/models/chat/remote_api.go` 明确会把 reasoning 内容向上游送出：

```go
streamChan <- types.StreamResponse{
    ResponseType: types.ResponseTypeThinking,
    Content:      reasoningContent,
    Done:         false,
}
```

因此现状是：

```text
remote_api 已经在流式发 thinking -> dedicated document edit path 把它 continue 掉 -> 用户看不到模型真正流式内容
```

### 3.3 当前超时逻辑监控的是“首个 answer”，不是“首个流事件”

日志中的超时链路：

```text
14:46:59.000 document_edit_waiting_for_model elapsed_ms=8001
14:47:07.001 document_edit_waiting_for_model elapsed_ms=16002
14:47:15.758 document_edit_waiting_for_model elapsed_ms=24004
14:47:22.852 document_edit_first_content_timeout timeout_ms=30000
```

这说明当前超时条件是：

```text
30 秒内没有收到首个 ResponseTypeAnswer -> 失败
```

而不是：

```text
30 秒内没有收到任何模型流式活动 -> 失败
```

因此即使模型已经在输出 reasoning，系统仍认为“没有内容”。

这也是本次失败的直接原因。

### 3.4 前端重复显示等待提示，是因为心跳被当成正文追加

当前 `frontend/src/views/chat/index.vue` 对 `thinking` 事件的处理是：

```js
thinkingEvent.content += data.content;
```

而后端当前发送等待提示时，复用了同一个 `progressEventID`，每 8 秒再次发送：

```text
仍在等待模型输出首个修订片段，将在超时后自动结束本次生成。
```

结果就是前端把相同的等待提示不断 `+=` 到同一个思考块里，最终在界面上累积成多行重复文本。

所以截图中出现的三行等待提示，不代表模型输出了三段不同内容，而是：

```text
同一条系统等待消息被前端连续追加了三次
```

## 4. 根因拆解

### 4.1 根因一：专用文档编辑链路没有透传模型 reasoning 流

这是最核心的根因。

当前 dedicated document edit path 的目标是：

1. 尽快拿到 `<document_patch>`
2. 在首个 patch 前给用户一些进度反馈

但实际实现变成了：

1. 丢弃所有 `thinking`/`reasoning_content`
2. 自己生成等待提示
3. 等首个 `answer`
4. 30 秒内没有 `answer` 就失败

问题在于：

- 深度推理模型在复杂文档编辑场景下，往往会先持续输出 reasoning，再开始 answer。
- 当前链路相当于把模型真正的早期流式信号全屏蔽掉了。

这与用户要求的“请给出模型的流式输出，而不是等待”正好相反。

### 4.2 根因二：`first_content_timeout` 绑定错了信号源

当前 timeout 名义上叫 `first_content_timeout`，但实际监控对象是：

```text
首个 ResponseTypeAnswer
```

而不是：

```text
首个模型可见流信号（thinking 或 answer）
```

所以在日志中出现了一个明显矛盾：

```text
14:46:51 模型已经有 First reasoning_content
14:47:22 系统仍判定 first_content_timeout
```

这说明 timeout 语义与真实流状态不一致。

### 4.3 根因三：等待心跳是“追加文本”，不是“状态更新”

等待提示当前被作为一个普通 `thinking` 内容片段发送给前端。

但对于这种系统状态提示，正确语义应是：

1. 覆盖更新同一状态
2. 或作为单独的 status event 展示
3. 或仅在 UI 上显示一个动态状态标签

而不是不断往正文中追加相同句子。

当前实现等于把：

```text
系统状态：仍在等待模型首字
```

误当成：

```text
模型新增输出：仍在等待模型首字
```

于是造成截图中的重复等待文本。

### 4.4 根因四：当前“Thinking=false”并不等于 provider 不会返回 reasoning_content

当前 dedicated path 在请求参数中设置了：

```go
Thinking: &thinking // thinking=false
```

但从日志看，provider 仍然返回了：

```text
First reasoning_content at OpenAI layer
```

说明：

1. `Thinking=false` 不是所有 provider 的强约束。
2. provider 仍可能返回 reasoning_content。
3. 上层不能假设“thinking=false 就一定没有 reasoning stream”。

因此，当前实现对 provider 行为的假设过强。

## 5. 为什么用户没有看到“模型流式输出”

可以用一句话概括：

```text
模型流式输出的是 reasoning_content，但当前专用文档编辑路径没有把 reasoning_content 当作用户可见流式输出，而是只认可 answer，所以用户只能看到系统等待提示。
```

也就是说，截图里的“正在分析基线文档并生成修订补丁，请稍候。”和“仍在等待模型输出首个修订片段……”并不是模型输出，而是系统自行补出来的占位提示。

## 6. 解决方案

## 6.1 P0：把模型真实流式输出透传给前端，而不是只显示等待提示

### 目标

让用户在 dedicated document edit path 中看到真实模型流，而不是仅看到系统等待占位文案。

### 改造建议

在 `consumeDedicatedDocumentEditStream()` 中增加对 `ResponseTypeThinking` 的处理：

1. 收到 `ResponseTypeThinking` 时，不要直接丢弃。
2. 将其转换为 `EventAgentThought` 发给前端。
3. 使用单独的 `modelThinkingEventID`，与系统等待提示分离。
4. 收到 `thinking done` 时，关闭对应 thought event。

建议逻辑：

```text
ResponseTypeThinking -> EventAgentThought(模型真实流)
ResponseTypeAnswer   -> EventAgentFinalAnswer(正文 patch 流)
系统等待提示         -> 单独 status event 或单次提示，不与模型流复用
```

### 直接收益

1. 492ms 左右即可看到模型开始输出，而不是空等 8 秒后才看到第一条等待提示。
2. 用户可以明确区分“模型在思考/组织 patch”与“系统在等待”。
3. 体验从“无输出”变成“模型在流式输出，只是还没进入 patch 正文”。

## 6.2 P0：把超时语义从“首个 answer”改成“首个可见流事件”

### 当前错误语义

```text
30 秒内没等到 answer -> 失败
```

### 建议语义

```text
30 秒内没有任何模型流事件（thinking/answer） -> 失败
```

更细一点可以拆成两个阈值：

1. `first_visible_stream_timeout`
   - 监控首个可见事件（thinking 或 answer）
   - 例如 5~10 秒
2. `first_answer_timeout`
   - 监控首个 answer patch
   - 但只有在完全没有 thinking 进展时才作为 hard timeout
   - 如果 reasoning 在持续流动，则改为告警日志，不要立刻失败

### 实施建议

在 `consumeDedicatedDocumentEditStream()` 中：

1. 收到第一条 `ResponseTypeThinking` 时，清空“空白等待超时”。
2. 如果后续持续收到 thinking，则说明模型仍在工作，不应因 answer 未到而直接失败。
3. `first_answer_timeout` 可以只做日志，不作为强制终止条件。
4. 真正的失败条件应是：
   - 上游 stream 中断
   - provider 返回 error
   - 整体 LLM 调用超时
   - thinking 和 answer 都长时间无任何新 token

### 直接收益

避免“模型正在流式推理，系统却把它判成超时”的错误收口。

## 6.3 P0：等待提示不要再用“正文追加”方式展示

### 当前问题

等待心跳使用同一个 `progressEventID`，而前端对 thinking 事件做：

```js
thinkingEvent.content += data.content;
```

所以等待文本被累计成多行重复内容。

### 建议方案

方案 A：新增 status event 类型

```text
document_edit_status_started
document_edit_status_waiting
```

前端把它们渲染为单行状态，不进入 markdown 正文流。

方案 B：保留 thinking 事件，但增加 replace 语义

例如在 metadata 中加：

```json
{
  "event_id": "document-edit-progress",
  "replace": true,
  "synthetic": true
}
```

前端遇到 `replace=true` 时：

```text
thinkingEvent.content = data.content
```

而不是：

```text
thinkingEvent.content += data.content
```

### 建议优先级

优先推荐方案 A，因为状态类信息和模型正文/思考本来就不应该混在同一类事件里。

## 6.4 P1：把真实模型流和系统补充提示分层展示

建议把专用文档编辑前端展示拆成三层：

1. 模型真实流
   - 来自 provider 的 reasoning_content / answer
2. 系统状态流
   - 例如：已进入专用文档编辑路径、已基于 V4 构造上下文、正在等待首个 patch
3. 失败/恢复控制
   - 重试、切换轻量模式、降级为确定性 patch

推荐展示顺序：

```text
[系统状态] 已基于当前基线 V4 构造文档编辑上下文
[模型流式输出] 我们先定位 2.5.5 小节与第二章的位置……
[模型流式输出] 接下来生成 append patch……
[正文 patch] <document_patch>...</document_patch>
```

而不是：

```text
[系统状态] 正在分析...
[系统状态] 仍在等待...
[系统状态] 仍在等待...
[系统状态] 仍在等待...
[失败] first_content_timeout
```

## 6.5 P1：对“章节移动/合并”场景增加确定性 patch 降级

当前请求属于高结构化场景：

```text
把 2.5.5 后续内容合并到第二章
```

这类请求不一定需要大模型长时间推理后再决定 patch。

建议：

1. 先由后端根据双锚点上下文确定：
   - source heading
   - destination heading
   - source section body
2. 直接生成基础 patch 框架：

```xml
<document_patch>
  <append heading="第二章">
    ...源章节后续内容...
  </append>
</document_patch>
```

3. 再让 LLM 仅做：
   - 局部措辞整理
   - 结构清洗
   - 编号修正

这样可以把“等待首个正文”的时间从深推理改为更快的结构化输出。

## 7. 建议实施顺序

### 阶段一：先把真实模型流透出来

1. dedicated path 支持 `ResponseTypeThinking`
2. 前端显示模型真实 thought stream
3. `first_content_timeout` 改为“首个可见流事件超时”

### 阶段二：修正等待提示展示方式

1. 把等待心跳改成 status event
2. 前端不再对等待提示做 `+=`
3. 截图中的重复等待文案消失

### 阶段三：再做结构化降级

1. move/append 类请求直接先构造 deterministic patch
2. LLM 只做轻量润色
3. 减少 reasoning-only 长等待

## 8. 验收标准

修复后应满足：

1. 发起 dedicated document edit 后，1 秒内前端能看到模型真实流式输出或明确的单行状态。
2. 如果 provider 返回 reasoning_content，前端必须可见，而不是被后端丢弃。
3. 等待提示不能再出现重复三行、四行累计追加。
4. 如果模型已有持续 reasoning 流，不应在 30 秒因为“没有 answer”而直接失败。
5. 对 move/append 类请求，首个可见正文或结构化 patch 的时间应明显缩短。

## 9. 最终结论

本次现象不是“模型没有流式输出”，而是“系统没有把模型已经产生的流式输出展示出来”。

日志已经明确证明：

```text
模型在 492ms 时就产生了 reasoning_content
```

但 dedicated document edit path 当前只承认 `ResponseTypeAnswer`，丢弃了 `ResponseTypeThinking`，并把首字超时绑定在 answer 上，于是最终表现成：

```text
模型已在流式推理
-> 后端不转发 reasoning
-> 前端只能看到等待提示
-> 等待提示被重复追加
-> 30 秒后 first_content_timeout
```

所以要解决“请给出模型的流式输出，而不是等待”，关键不是继续优化等待文案，而是：

```text
把模型真实的 thinking/reasoning 流透传给前端，并把超时逻辑从“等首个 answer”改为“等首个可见流事件”。
```
