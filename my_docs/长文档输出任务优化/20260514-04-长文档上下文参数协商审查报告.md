# 长文档上下文参数协商审查报告

## 1. 审查结论

基于给定日志和当前实现，结论如下：

1. 当前长文档流程中，存在“先协商、后生成”的预算协商过程。
2. 这一步协商的不是底层模型原生 `context_window` 配置本身，而是基于模型能力和长文档场景推导出的“文档生成预算参数”。
3. 协商结果的一部分会成为后续轮次生成的事实标准，例如证据检索条数、章节调用超时时间、续写轮次的持久化预算基线。
4. 但协商出的 `section_max_completion_tokens` 和 `continuation_max_completion_tokens` 当前并没有直接作为章节流式生成请求的 `MaxCompletionTokens` 传给模型，因此它们目前不是“严格生效的章节生成上限”。
5. 因此，如果需求是“长文档开启后，必须先对模型协商上下文参数，并把协商结果作为后续生成统一标准”，那么当前实现只能算“部分满足”，还没有形成完整闭环。

## 2. 日志审查结果

### 2.1 是否存在协商过程

存在，且发生在长文档正式生成之前。

关键日志顺序如下：

1. `18:08:05.465` 输出静态能力预算：

   `session_agent_qa.go:1769[resolveStaticDocumentGenerationBudget] | [DocumentBudget] source=capability ... outline_tokens=1536 section_tokens=4096 continuation_tokens=4096 ... context_window=64000 max_output=8192`

   这说明系统先根据模型能力拿到一份静态预算基线。

2. `18:08:05.472` 发起一次单独的 LLM 请求：

   `remote_api.go:441[chatWithRawHTTP] | [LLM Request] ... You are estimating safe generation budgets for a long markdown document. Return JSON only.`

   请求内容明确要求模型根据已知约束返回 JSON，包括：

   - `outline_max_completion_tokens`
   - `section_max_completion_tokens`
   - `continuation_max_completion_tokens`
   - `outline_evidence_top_k`
   - `section_evidence_top_k`
   - `continuation_evidence_top_k`
   - `section_call_timeout_seconds`
   - `reason`

3. `18:08:09.211` 协商完成：

   `session_agent_qa.go:1814[negotiateDocumentGenerationBudget] | [DocumentBudget] source=negotiated ... section_tokens=4096 section_top_k=8 timeout=210 reason="Using current conservative defaults ..."`

4. `18:08:09.215` 才开始知识库检索：

   `knowledgebase_search.go:95[HybridSearch] | Hybrid search parameters ...`

这说明当前链路确实是先完成预算协商，再进入后续知识检索和正文生成。

### 2.2 本次协商的内容是什么

从日志和实现综合来看，本次协商输入包括：

- 模型上下文窗口：`model_context_window_tokens: 64000`
- 模型最大输出：`model_max_output_tokens: 8192`
- 当前默认大纲输出上限：`1536`
- 当前默认章节输出上限：`4096`
- 当前默认续写输出上限：`4096`
- 当前默认大纲检索 top_k：`8`
- 当前默认章节检索 top_k：`8`
- 当前默认续写检索 top_k：`8`
- 当前章节调用超时：`210s`
- 用户目标：生成北海电厂技术文档
- 预估文档类型：`technical_proposal`
- 预估章节数：`6`
- 是否有本地知识：`true`
- 知识库范围数量：`1`
- 是否自动续写：`false`

协商输出在实现里要求必须返回完整 JSON，字段如下：

- 大纲生成预算
- 章节生成预算
- 续写生成预算
- 大纲证据检索规模
- 章节证据检索规模
- 续写证据检索规模
- 章节调用超时时间
- 协商原因

### 2.3 本次日志能否证明协商结果真正落地

可以证明“协商发生了”，也可以证明“部分协商结果被后续流程使用了”，但不能仅凭这段日志证明“所有协商字段都成为后续轮次统一标准”。

原因如下：

1. 协商成功日志只打印了 `section_tokens`、`section_top_k`、`timeout` 和 `reason`，没有把全部返回字段打出来。
2. 但从实现看，只要协商成功，返回 JSON 中所有字段必须齐全且大于 0，否则会被判定为失败并回退到基线预算。
3. 所以这条成功日志间接证明：模型确实返回了一份完整预算 JSON，只是日志没有完整展开。

另外，本次协商结果从日志可见部分看，与静态基线没有明显差异：

- 静态预算：`section_tokens=4096`、`section_top_k=8`、`timeout=210`
- 协商预算：`section_tokens=4096`、`section_top_k=8`、`timeout=210`

也就是说，本次协商是“有协商，但选择沿用保守默认值”，而不是“有协商且显著调整了预算”。

## 3. 当前实现分析

### 3.1 协商入口与校验逻辑

预算协商主链路如下：

1. `resolveStaticDocumentGenerationBudget`
   - 根据模型能力先构造一份静态预算。

2. `negotiateDocumentGenerationBudget`
   - 在长文档场景下向同一个聊天模型发起一次预算协商请求。

3. `validateAndClampNegotiatedDocumentBudget`
   - 对协商结果做上下限裁剪，避免超出模型能力或低于系统最小值。

4. `resolveDocumentGenerationBudget`
   - 返回最终预算，供后续大纲、章节、续写链路使用。

从实现上看，协商是严格校验的，不是“有返回就信”。如果响应不是合法 JSON，或任何预算字段缺失/非正数，都会退回静态能力预算。

### 3.2 首轮长文档生成如何使用协商结果

在知识库长文档生成路径中，预算会在正文开始前被解析出来，然后进入后续流程。

已确认直接生效的项：

1. `outline_max_completion_tokens`
   - 用于大纲生成请求的 `MaxCompletionTokens`。

2. `outline_evidence_top_k`
   - 用于大纲阶段本地知识检索。

3. `section_evidence_top_k`
   - 用于每章正文生成前的知识检索。

4. `section_call_timeout_seconds`
   - 用于章节和续写流式生成时的上下文超时控制。

未直接生效的项：

1. `section_max_completion_tokens`
2. `continuation_max_completion_tokens`

当前章节流式生成调用 `chatModel.ChatStream(...)` 时，没有把这两个值写入 `ChatOptions.MaxCompletionTokens`。测试也明确验证了这一点：

- 大纲请求 `MaxCompletionTokens=1536`
- 章节请求 `MaxCompletionTokens=0`

这说明协商出的章节 token 预算目前更多体现在：

- 日志观测
- runtime feedback 调整
- 持久化存储
- 后续预算合并

但没有直接约束章节流式生成请求本身。

### 3.3 后续轮次是否沿用协商结果

结论是：会沿用，但沿用得并不完整。

#### 3.3.1 会被持久化的内容

首轮知识库长文档生成在创建 `generation run` 时，会把当前预算写入 `BudgetJSON`，同时把 runtime feedback 写入 `RuntimeFeedbackJSON`。

这意味着预算不是只挂在一次性 completion extra 上，而是被保存为跨轮次状态。

#### 3.3.2 后续轮次如何恢复

在 `runKnowledgeGroundedDocumentContinuationRunPath` 中，系统会先重新调用 `resolveDocumentGenerationBudget` 获取当前轮预算，然后执行：

- `mergePersistedDocumentGenerationBudget(...)`

这一步会把上次 run 中持久化下来的预算重新合并回来，并按当前模型能力重新裁剪。

所以从机制上看，后续轮次确实把前一轮预算当作续写基线，而不是每轮完全从零开始。

#### 3.3.3 哪些预算会在后续轮次继续生效

明确继续生效的项：

1. `continuation_evidence_top_k`
   - 直接用于续写阶段的知识检索。

2. `section_evidence_top_k`
   - 直接用于续写 run 中各章节的知识检索。

3. `section_call_timeout_seconds`
   - 直接用于续写阶段章节流式生成超时。

4. runtime feedback 调整后的预算
   - 会继续影响下一轮的检索规模、超时和分批章节数。

未形成严格统一标准的项：

1. `section_max_completion_tokens`
2. `continuation_max_completion_tokens`

原因仍然相同：续写链路的 `ChatStream(...)` 也没有把这两个预算直接传到 `MaxCompletionTokens`。

因此，“预算被保存下来”与“预算真正作为模型输出上限执行”是两件不同的事。当前实现完成了前者，但后者只在大纲阶段成立，在章节/续写阶段并不成立。

## 4. 现状判断

### 4.1 如果需求只是“长文档开启后先做协商”

当前实现满足。

因为：

- 预算协商发生在长文档正式检索和生成之前。
- 协商失败有回退机制。
- 协商结果会进入后续流程，并被持久化到 generation run。

### 4.2 如果需求是“协商结果作为后续所有轮次生成统一标准”

当前实现不完全满足。

原因：

1. 大纲阶段满足，`outline_max_completion_tokens` 直接生效。
2. 检索规模和超时阶段基本满足，`top_k` 与 `timeout` 会持续生效并跨轮次传递。
3. 章节正文生成和续写正文生成阶段不完全满足，`section_max_completion_tokens` / `continuation_max_completion_tokens` 没有直接下发给模型请求。

因此，当前更准确的描述应该是：

“系统已经实现了长文档预算协商，并把协商结果作为后续流程的重要预算基线；但该预算尚未完整绑定到章节/续写的流式生成请求参数上。”

## 5. 风险点

### 5.1 预算名义生效但实际未完全约束模型输出

日志里能看到 `section_tokens` 和 `continuation_tokens`，容易让人以为这些值已经直接成为章节生成上限；但从实现上看，目前它们更多是观测与调优预算，而不是强制请求参数。

### 5.2 协商日志不完整，排查成本较高

协商成功日志没有输出全部预算字段，导致线上排查时无法直接确认：

- 大纲 token 是否变化
- 续写 token 是否变化
- outline / continuation top_k 是否变化

这会增加“日志看起来协商过了，但不知道协商结果是否完整”的分析成本。

## 6. 审查建议

### 6.1 若要满足“协商结果即统一标准”，应补齐章节与续写请求参数传递

建议在章节生成和续写生成的 `ChatStream(...)` 调用中，明确传入：

- `MaxCompletionTokens: currentBudget.SectionMaxCompletionTokens`
- 续写场景使用 `currentBudget.ContinuationMaxCompletionTokens`

这样协商结果才会真正成为正文生成阶段的硬约束，而不是只停留在预算记录与日志层。

### 6.2 补充协商成功日志字段

建议在协商成功日志中同时输出：

- `outline_tokens`
- `section_tokens`
- `continuation_tokens`
- `outline_top_k`
- `section_top_k`
- `continuation_top_k`
- `timeout`

这样可以直接从线上日志判断“协商是否完整生效”。

### 6.3 增加回归测试，防止预算只存不生效

建议补充两类测试：

1. 章节生成请求必须带上协商后的 `MaxCompletionTokens`。
2. 续写请求必须带上协商后的 `ContinuationMaxCompletionTokens`。

否则后续即便继续优化 runtime feedback，也仍然存在“预算持续被调整，但模型请求没有真正使用”的问题。

## 7. 最终判断

本次北海电厂日志中，长文档模式下确实发生了预算协商，而且协商发生在知识检索和正文生成之前。协商的核心内容包括输出 token 预算、检索 top_k、章节超时以及协商理由。

但是，当前实现只有“大纲 token 上限、检索规模、章节超时”形成了较明确的执行闭环；章节正文和续写正文对应的 token 上限虽然被协商、记录、持久化并跨轮次恢复，但并没有直接下发给流式生成请求，因此还不能认定为“后续轮次生成的完整统一标准”。

换句话说，当前实现是“有协商、有持久化、部分执行”，还不是“协商结果全量强约束后续轮次生成”。
