# 智能推理长文翻译 P2 任务流重构详细设计

## 1. 阶段目标

P2 的目标是把“全文翻译并输出 markdown 文件”从当前在线问答链路中彻底拆出来，建设为独立的后台任务能力。

这不是一次 bug fix，而是能力边界重构。P2 要解决的问题不是“如何减少截断”，而是“如何让系统正确承载长文加工任务”。

## 2. 设计原则

### 2.1 Chat 负责发起，不负责承载全文正文

Chat 窗口只做两件事：

1. 发起翻译任务。
2. 查看任务进度和结果。

不再试图把整篇译文通过 SSE 一次性流回对话框。

### 2.2 中间结果必须持久化

长文翻译不是一次调用，而是多批次加工过程。任何阶段失败后，都不能让已完成结果丢失。

### 2.3 产物交付必须文件化

最终结果应该是 markdown 文件、导出记录或对象存储文件，而不是一段超长聊天正文。

## 3. 能力边界设计

## 3.1 任务输入

任务发起时至少需要：

1. knowledge_id 或 document_id
2. 源语言或自动检测
3. 目标语言，当前为中文
4. 输出格式，当前为 markdown
5. 是否翻译图片标题、图注、表格、参考文献

## 3.2 任务输出

任务完成后返回：

1. task_id
2. 执行状态
3. 结果文件路径或下载链接
4. 失败原因
5. 批次统计

## 3.3 不再复用的旧能力

以下行为不应再作为主路径：

1. builtin-smart-reasoning 直接在聊天中输出全文翻译。
2. 把全文翻译结果写入聊天历史知识库。
3. 用单次 completion 承载整篇论文内容。

## 4. 架构设计

## 4.1 后端模块拆分

建议新增以下模块：

1. TranslationTaskService：任务创建、状态推进、结果汇总。
2. TranslationChunkWorker：按批次执行翻译。
3. TranslationAssembler：拼装最终 markdown 文件。
4. TranslationArtifactService：管理文件产物和下载。

建议目录：

1. internal/application/service/document_translation.go
2. internal/application/service/document_translation_worker.go
3. internal/application/service/document_translation_assembler.go
4. internal/handler/document_translation.go

## 5. 处理流程设计

## 5.1 任务创建流程

1. 用户在 Chat 中触发“全文翻译为 markdown”。
2. 后端不直接走 AgentQA，而是识别为 document translation task。
3. 创建任务记录和批次记录。
4. 向前端返回 task_id。

## 5.2 文档读取与分批策略

### 5.2.1 读取来源

优先使用已解析文档 chunk，而不是重新 OCR 或重新切分。

### 5.2.2 分批原则

建议按 token 预算和结构边界共同分批：

1. 保证单批不超过稳定输出窗口。
2. 优先在章节、小节、表格、参考文献边界切分。
3. 图表和图注尽量与正文同批。

### 5.2.3 建议批次大小

初版建议：

1. 每批 5 到 10 个 chunk。
2. 同时增加 token 预算校验。

## 5.3 批次翻译流程

每个批次执行如下步骤：

1. 读取该批次原文。
2. 拼装专用翻译 prompt。
3. 调用翻译模型生成 markdown 片段。
4. 将批次结果落库。
5. 更新 task.progress。

### 5.3.1 专用 prompt 要求

prompt 中要明确：

1. 必须逐字翻译为中文。
2. 保留标题层级。
3. 保留图表和参考文献结构。
4. 输出 markdown 片段，不添加额外解释。

### 5.3.2 重试策略

建议每批最多重试 2 到 3 次：

1. 超时可重试。
2. 长度截断可缩小批次重试。
3. 连续失败则该批次标记 failed，任务进入 partial 或 failed。

## 5.4 最终汇总流程

全部批次完成后：

1. 按 batch_no 顺序拼接 markdown。
2. 追加文档头信息和任务元数据。
3. 生成最终 markdown 文件。
4. 写入 artifact_path。
5. 更新任务状态为 completed。

如果存在失败批次但可交付：

1. 任务状态设为 partial。
2. 文件中标记缺失区段。
3. 前端提示“部分段落翻译失败”。

## 6. 接口设计

## 6.1 创建任务接口

建议：

1. POST /api/v1/document-translation/tasks

请求参数：

1. knowledge_id
2. target_language
3. output_format
4. options

返回：

1. task_id
2. status

## 6.2 查询任务接口

建议：

1. GET /api/v1/document-translation/tasks/:id

返回：

1. status
2. progress
3. total_batches
4. completed_batches
5. failed_batches
6. artifact_path
7. error_message

## 6.3 下载产物接口

建议：

1. GET /api/v1/document-translation/tasks/:id/download

## 7. 前端设计

## 7.1 Chat 入口设计

Chat 中识别以下意图时，不再走普通 Agent：

1. 全文翻译
2. 输出 markdown 文件
3. 保持原文结构

触发后：

1. 创建任务。
2. 插入一条“任务已创建”的 assistant 消息。
3. 展示任务进度卡片。

## 7.2 任务进度展示

建议展示：

1. 当前状态
2. 批次进度
3. 已完成页数或 chunk 数
4. 失败批次数
5. 下载按钮

## 7.3 失败恢复交互

建议支持：

1. 重试失败批次。
2. 下载 partial 文件。
3. 查看失败原因。

## 8. 运行与调度设计

## 8.1 异步执行方式

优先复用现有任务队列体系，例如 Asynq。

调度建议：

1. 创建任务时投递 translation task。
2. worker 逐批执行。
3. 每批完成后更新进度。

## 8.2 并发策略

初版建议保守：

1. 单任务串行批次执行。
2. 多任务可并发，但受租户配额控制。

原因：

1. 章节连续性更好控制。
2. 汇总顺序简单。
3. 失败恢复更清晰。

## 9. 风险点与处理策略

### 9.1 风险：批次边界破坏格式连续性

处理：

1. 按结构边界优先切分。
2. 每批附带前一批的结尾上下文摘要。

### 9.2 风险：某些批次仍然超长

处理：

1. 批次执行前做 token 预算预估。
2. 超阈值自动再切分。

### 9.3 风险：用户误以为任务已完成

处理：

1. 进度卡片明确区分 running、partial、completed。
2. 只有 artifact 可下载且状态 completed 时才显示“已完成”。

## 10. 测试设计

### 10.1 后端测试

1. 任务创建测试。
2. 批次切分测试。
3. 批次翻译重试测试。
4. 汇总 markdown 文件测试。
5. partial 任务生成可下载文件测试。

### 10.2 前端测试

1. 创建任务后展示进度卡片。
2. running 到 completed 状态切换。
3. partial 状态下载按钮展示。
4. 失败批次重试交互。

## 11. 验收标准

P2 完成后，满足以下条件才算通过：

1. 用户发起全文翻译时，不再走普通在线问答全文输出。
2. 长文翻译结果以 markdown 文件形式交付。
3. 任务失败时可定位到批次并保留中间成果。
4. Chat 窗口只展示任务进度，不再承载整篇正文。

## 12. 本阶段结论

P2 完成后，长文翻译才真正从“高风险问答行为”升级为“可交付文档任务”。这一步才是问题的最终解法，而 P0 和 P1 是为它铺路的必要前置阶段。
