# 续写基线与最终文档加载修正方案

## 1. 背景与目标

基于页面现象和 2026-05-09 18:38 日志，本轮问题已经不是单纯的 `tool_error` 硬失败。当前后端已经能把可恢复问题收口为：

```text
completion_status: partial
finish_reason: fallback_stop
failure_reason: 
allow_complete: false
allow_indexing: false
```

页面也已经展示出新的 recovered 提示：

```text
工具执行中出现了可恢复问题，系统已保留当前生成内容。你可以继续补全，或重新生成。
```

但从页面和日志继续看，用户体验仍然没有闭环：

1. 页面仍显示“当前基线 北海电厂二期智慧电厂项目技术方案 V1”，说明本轮已经生成的新内容没有成为下一轮基线。
2. 用户请求“请继续补齐”时，系统仍使用 `document_revision_context`，要求模型输出“修改后的完整 Markdown 文档”，导致每次修改都从文档开头重新流式输出。
3. 日志显示 final answer 已生成 428 字，但持久化内容仍可能是较短的中间正文，例如 356 字；即使后续代码已补强 FinalAnswer 覆盖，前端仍缺少“修改完成后一次性加载完整文件”的明确协议。

本方案只基于当前代码实现情况输出修正文档，不修改代码。

## 2. 当前实现状态审查

### 2.1 可恢复错误状态已基本收口

当前 Agent final answer 链路已经具备以下能力：

1. `internal/agent/finalize.go` 中通过 `normalizeCompletionEventState()` 将 `FinalAnswerSynthesized=true` 且正文非空的 stale `failed/tool_error` 归一化为 `partial/fallback_stop`。
2. `internal/handler/session/qa.go` 的 `completeAssistantMessage()` 在 Agent 模式下会调用 `shouldUseAgentFinalAnswerContent()`，允许 `fallback_stop` 场景使用 `FinalAnswer` 覆盖当前消息正文。
3. `internal/handler/session/agent_stream_handler.go` 当前已经允许 `partial/fallback_stop` 使用 complete payload 中的 `FinalAnswer` 作为权威正文，并在没有 answer chunk 时补发 fallback answer event。
4. `frontend/src/views/chat/index.vue` 的 `syncMessageCompletionState()` 会把 `fallback_stop` 且非 failed 的状态识别为 recovered，并清理旧的 `error_message`。
5. `frontend/src/views/chat/components/botmsg.vue` 已经能显示 recovered partial 提示，而不是继续显示硬失败红条。

这说明“工具执行可恢复问题”已经从硬失败修正为可继续状态。

### 2.2 新内容未成为下一轮基线的原因

当前 artifact 创建链路位于：

1. `internal/handler/session/qa.go`：`completeAssistantMessage()` 持久化后调用 `RegisterFromAssistantMessage()`。
2. `internal/application/service/chat_document_artifact.go`：`RegisterFromAssistantMessage()` 根据消息内容和 options 创建 `ChatDocumentArtifact`。
3. `shouldRegisterChatDocumentArtifact()` 当前有长文档门槛：正文长度小于 2000 字直接不创建 artifact。
4. `buildChatDocumentSnapshot()` 只对 `continue` 操作做 `base + delta` 合并；对 `revise` 操作直接把本轮输出当完整快照。

日志中本轮 final answer 只有 428 字，持久化正文约 356 字，低于 2000 字门槛，因此不会创建新的 artifact revision。于是前端只能继续使用上一版 V1 作为基线。

这就解释了页面现象：虽然系统提示“已保留当前生成内容”，但 artifact 版本链没有推进，下一轮仍会基于旧 V1，而不是当前已生成的新内容。

### 2.3 修改场景每次从头输出的原因

当前 `BuildQuotedContext()` 对 `revise_document` 使用的是完整文档改写 prompt：

```text
修改规则：
1. 以上一份文档为基线进行修改。
2. 不要丢失用户没有要求删除的章节和内容。
3. 按用户本轮要求调整结构、补充细节或修正文案。
4. 输出修改后的完整 Markdown 文档。
5. 不要输出 diff 标记，不要输出修改说明，除非用户明确要求。
```

这会强制模型从标题开始重新输出整份 Markdown。对于 5 万字级别的基线文档，这会产生三个问题：

1. 首屏持续重复已有内容，用户无法快速看到新增或修改部分。
2. 流式 token 消耗很大，更容易触发超时或截断。
3. 即使用户只是“继续补齐某一节”，系统也可能走 `revise_document`，导致完整重写。

当前前端意图识别也存在边界：

1. `inferDocumentContinuationIntent()` 能识别“继续生成、接着写、续写、补全剩余、继续输出”。
2. 但“请继续补齐”“继续补充某章节”这类表达不一定稳定命中 continue。
3. 如果用户手动选中了基线 artifact，且 intent 仍为 normal，前端会把 intent 改成 `revise_document`。

因此“请继续补齐”容易被送进 revision prompt，进而每次从头输出。

### 2.4 完整文件没有一次性加载的原因

当前 SSE complete 事件携带的 artifact 元数据来自 `chatDocumentArtifactMetadata()`，字段包括：

1. artifact id
2. revision no
3. status
4. completion status
5. operation
6. snapshot char count
7. structure info

但它不包含 `content_snapshot`。

前端在收到 complete 事件时会：

1. 如果有 `final_answer`，用 `final_answer` 覆盖消息正文。
2. 如果有 `chat_document_artifact`，只把 artifact 元数据绑定到消息上。
3. 不会在 complete 后自动调用 `getChatDocumentArtifact(artifact_id)` 拉取完整 `content_snapshot`。

因此当前前端能看到“本轮流式输出的正文”，但并没有一个明确机制在修改完成后一次性加载“后端合成后的完整文件”。

## 3. 修正目标

本次优化应同时满足三个目标。

### 3.1 可恢复 partial 也应推进基线

当系统提示：

```text
工具执行中出现了可恢复问题，系统已保留当前生成内容。你可以继续补全，或重新生成。
```

只要当前内容可以形成新的文档状态，就应创建或更新一个新 artifact revision。下一轮继续补全时，应默认以这个新 revision 为基线，而不是退回更早的 V1。

### 3.2 修改和补齐时只流式输出变化部分

对基于文件或 artifact 的续写、补齐、局部修改，不应要求模型每次流式输出完整文档。

推荐改为：

1. 流式阶段只输出本轮新增内容、修改片段或目标章节。
2. 后端根据 base artifact 和本轮 delta 合成新的完整 `ContentSnapshot`。
3. 聊天消息中可保留本轮 delta，用于解释用户看到的增量过程。
4. artifact 中保存完整文档版本，用于下一轮基线和最终预览。

### 3.3 完成后一次性加载完整文件

修改完成后，前端不应依赖流式片段作为最终完整文档。

推荐改为：

1. complete 事件带上新 artifact id、revision、operation、snapshot char count，以及一个明确的 `final_document_mode`。
2. 如果完整快照大小可控，complete 事件可直接带 `final_document`。
3. 如果完整快照较大，前端在 complete 后根据 artifact id 调用 `GET /api/v1/chat-document-artifacts/{artifact_id}` 拉取 `content_snapshot`。
4. 前端一次性用完整快照刷新文档展示区或 artifact 预览，而不是继续从头流式刷屏。

## 4. 后端修正方案

### 4.1 增加文档变更输出模式

建议在请求和内部上下文中引入文档输出模式：

```go
type DocumentOutputMode string

const (
    DocumentOutputModeFull  = "full_document"
    DocumentOutputModeDelta = "delta_only"
)
```

推荐规则：

1. `continue_document` 默认使用 `delta_only`。
2. `revise_document` 默认也使用 `delta_only`，除非用户明确要求“输出完整新版全文”。
3. `regenerate_document` 使用 `full_document`。
4. 首次创建文档使用 `full_document`。

该字段可落在：

1. `CreateKnowledgeQARequest`：可选接收前端提示。
2. `QARequest`：传给 Agent/Knowledge pipeline。
3. `RegisterChatDocumentArtifactOptions`：用于登记 artifact 时决定如何合成 snapshot。

### 4.2 修正意图识别

当前“请继续补齐”不一定被识别为 continue。建议扩展前后端规则。

后端 `chatDocumentContinueIntentRE` 增加：

```text
继续补齐|继续补充|接着补齐|接着补充|补齐剩余|补充剩余|继续完善|继续扩写
```

前端 `inferDocumentContinuationIntent()` 同步增加这些表达。

优先级建议：

1. 明确“重新生成、从头生成、重写一版”优先归为 `regenerate_document`。
2. “继续、接着、补齐、补充剩余、续写”优先归为 `continue_document`。
3. “修改、调整、替换、改成、完善上一版”归为 `revise_document`。
4. 如果用户只选中了基线但 query 没有明确修改词，不应直接升级为 `revise_document`，可保持 normal 或提示用户确认。

### 4.3 新增 delta prompt

对 `continue_document` 和默认 `revise_document`，建议把 prompt 改为 delta 输出协议。

#### 4.3.1 续写 delta prompt

```text
<document_continuation_context>
你正在继续生成同一会话中的上一份文档。

输出规则：
1. 只输出本轮需要新增的 Markdown 内容。
2. 不要从文档标题或第一章重新开始。
3. 从上一份文档末尾或用户指定章节自然继续。
4. 如果上一份文档末尾句子不完整，先补齐该句，再继续后续内容。
5. 保持标题层级、编号、术语、表格和 Markdown 风格一致。
6. 不要解释“我将继续”，直接输出可合并进文档的正文片段。

上一份文档元数据：
...

上一份文档内容：
<document>
...
</document>
</document_continuation_context>
```

#### 4.3.2 修改 delta prompt

```text
<document_revision_context>
你正在修改同一会话中的上一份文档。

输出规则：
1. 只输出本轮新增或修改后的章节片段，不要输出完整文档全文。
2. 如果用户指定了章节，只输出该章节的最终 Markdown 片段。
3. 如果需要替换已有章节，输出替换后的完整章节内容，并保留该章节标题。
4. 不要输出 diff 标记，不要输出修改说明。
5. 不要重复未修改章节。
6. 输出内容必须能被系统合并回上一份文档，形成新的完整版本。

上一份文档元数据：
...

上一份文档内容：
<document>
...
</document>
</document_revision_context>
```

### 4.4 增加 delta 合成完整快照能力

当前 `buildChatDocumentSnapshot()` 只对 continue 做简单追加，对 revise 直接返回本轮内容。需要扩展为：

```go
func buildChatDocumentSnapshot(content string, options RegisterChatDocumentArtifactOptions) string {
    base := options.BaseArtifact
    operation := normalizeChatDocumentOperation(options.Operation)
    outputMode := normalizeDocumentOutputMode(options.OutputMode)

    if base == nil || strings.TrimSpace(base.ContentSnapshot) == "" {
        return strings.TrimSpace(content)
    }

    if outputMode == DocumentOutputModeFull {
        return strings.TrimSpace(content)
    }

    switch operation {
    case ChatDocumentOperationContinue:
        return mergeChatDocumentContinuation(base.ContentSnapshot, content)
    case ChatDocumentOperationRevise:
        return mergeChatDocumentRevisionDelta(base.ContentSnapshot, content, options.UserQuery)
    default:
        return strings.TrimSpace(content)
    }
}
```

`mergeChatDocumentRevisionDelta()` 的 P0 推荐策略：

1. 如果 delta 以 Markdown 标题开头，提取第一个标题。
2. 在 base 中查找相同标题层级与标题文本。
3. 如果找到同名章节，则替换该章节到下一个同级或更高级标题之前的内容。
4. 如果找不到同名章节，则根据用户 query 或标题层级追加到最相近章节末尾。
5. 如果无法判断位置，则追加到文档末尾，并把 artifact 标记为 partial，增加 quality issue。

P1 可引入更严格的 patch envelope，例如：

```text
<document_patch>
<replace heading="## 智慧运行（AI智能运行体系）">
...替换后的章节内容...
</replace>
</document_patch>
```

这样后端可以结构化合并，减少误替换。

### 4.5 可恢复 partial 也创建新 artifact revision

当前 `shouldRegisterChatDocumentArtifact()` 有 2000 字门槛，导致本轮 428 字 recovered partial 不会生成新 artifact。建议新增特殊放行规则：

1. 如果存在 `BaseArtifact`。
2. `Operation` 为 `continue` 或 `revise`。
3. `CompletionStatus` 为 `partial` 或 `completed`。
4. `FinishReason` 为 `fallback_stop`、`timeout` 或 `stop`。
5. 本轮内容非空。
6. 能与 base 合成新的 `ContentSnapshot`。

则允许创建新 revision，即使本轮 delta 少于 2000 字。

注意：门槛应作用于“合成后的完整 snapshot”，而不是只看本轮 delta 长度。也就是说，短 delta 修改了 5 万字文档中的一个章节，也应该生成一个新的完整 artifact revision。

推荐登记规则：

1. `ContentSnapshot = merge(base.ContentSnapshot, delta)`。
2. `ParentArtifactID = base.ID`。
3. `RevisionNo = base.RevisionNo + 1`。
4. `Status = partial`，如果本轮是 recovered 或合并位置不确定。
5. `CompletionStatus = partial`。
6. `Operation = continue` 或 `revise`。
7. `QualityIssues` 增加 `recovered_partial` 或 `delta_merge_uncertain`。

这样页面提示“已保留当前生成内容”才有数据层含义：当前内容已经成为新的可选基线。

### 4.6 latest artifact 应优先返回当前最新 revision

当前前端调用 `getLatestChatDocumentArtifact(session_id)` 作为下一轮基线。后端 repository 需要确保 latest 规则符合版本链语义：

1. 同一 session 内按 `created_at desc` 或 `revision_no desc` 返回最新可继续 artifact。
2. `status=partial` 的 recovered revision 也应参与 latest 选择。
3. 不应因为 `allow_complete=false` 或 `completion_status=partial` 排除它。
4. 如果存在用户手动选择的 `selectedBaseArtifact`，则以前端选择为准。

### 4.7 complete 事件携带最终文档加载指令

建议扩展 `EventAgentComplete` / SSE complete payload：

```json
{
  "final_answer": "本轮流式增量内容",
  "completion_status": "partial",
  "finish_reason": "fallback_stop",
  "chat_document_artifact": {
    "id": "new-artifact-id",
    "revision_no": 2,
    "parent_artifact_id": "old-artifact-id",
    "status": "partial",
    "operation": "revise",
    "snapshot_char_count": 52300
  },
  "document_output_mode": "delta_only",
  "final_document_mode": "fetch_artifact_snapshot",
  "final_document_artifact_id": "new-artifact-id"
}
```

如果快照较小，也可以直接返回：

```json
{
  "final_document_mode": "inline_snapshot",
  "final_document": "# 完整文档..."
}
```

建议阈值：

1. 小于等于 80k 字符：允许 `inline_snapshot`。
2. 大于 80k 字符：使用 `fetch_artifact_snapshot`。
3. 超过 `ChatDocumentArtifactSnapshotMaxChars` 的文档仍按现有截断策略处理，并在 metadata 中带 `quality_issues`。

## 5. 前端修正方案

### 5.1 发送请求时区分输出模式

`frontend/src/views/chat/index.vue` 在 `sendMsg()` 中构造请求时，应增加：

```ts
const documentOutputMode = inferDocumentOutputMode(value, intentHint, latestArtifact);
```

规则：

1. `continue_document`：`delta_only`。
2. `revise_document`：默认 `delta_only`。
3. 用户明确输入“输出完整新版全文、给我完整文档、重新生成完整方案”：`full_document`。
4. `regenerate_document`：`full_document`。

请求体增加：

```json
{
  "intent_hint": "continue_document",
  "base_artifact_id": "...",
  "document_output_mode": "delta_only"
}
```

### 5.2 修正“请继续补齐”的识别

前端意图识别应把以下表达归为 `continue_document`：

```text
请继续补齐
继续补齐
继续补充
继续完善
接着补齐
补齐剩余
补充剩余
继续扩写
```

这样用户截图里的请求不会再误走 `revise_document` 全文重写 prompt。

### 5.3 流式阶段只展示增量内容

当前 `answer` 事件会持续 append 到 `message.content`。在 `document_output_mode=delta_only` 时，这个行为是合理的，因为 answer 只代表本轮 delta。

前端展示建议：

1. 聊天消息正文显示“本轮新增/修改片段”。
2. Artifact 卡片显示新版本状态，例如 V2 partial。
3. 不在流式阶段把完整文档从头刷到聊天消息里。

### 5.4 complete 后一次性加载完整文档

前端在收到 complete 事件后，根据 payload 执行：

```ts
if (completePayload.final_document_mode === 'inline_snapshot') {
  message.final_document_content = completePayload.final_document;
}

if (completePayload.final_document_mode === 'fetch_artifact_snapshot') {
  const artifact = await getChatDocumentArtifact(completePayload.final_document_artifact_id);
  message.final_document_content = artifact.content_snapshot;
  assignChatDocumentArtifactToMessage(message, artifact);
}
```

展示策略：

1. `message.content` 保留本轮 delta，不强行替换为完整文档，避免聊天列表里突然出现 5 万字长文。
2. 新增“完整文档预览区”或复用 artifact 卡片展开态显示 `final_document_content`。
3. 如果产品要求消息正文就是最终完整文档，则只在 complete 后一次性替换，不在流式阶段重复输出。
4. 推荐保留 delta 与完整文档两个视图：默认展示 delta，用户展开 artifact 查看完整文档。

### 5.5 当前基线自动更新为新 revision

`assignChatDocumentArtifactToMessage()` 当前会 upsert artifact，但不会自动把新 artifact 设为 `selectedBaseArtifact`。建议在以下条件下自动更新：

1. complete payload 带 `chat_document_artifact`。
2. artifact 的 `operation` 为 `continue` 或 `revise`。
3. artifact 的 `status` 为 `available` 或 `partial`。
4. artifact 的 `parent_artifact_id` 等于本轮 `base_artifact_id`，或 source message id 等于当前 assistant message id。

前端执行：

```ts
selectedBaseArtifact.value = normalizedArtifact;
```

页面上的“当前基线”应立即从 V1 更新为 V2 partial 或 V2 available。

## 6. 数据与状态语义修正

### 6.1 completion_status 与 artifact status 的关系

推荐约定：

| 场景 | message.completion_status | message.finish_reason | artifact.status | 是否可作为下一轮基线 |
| --- | --- | --- | --- | --- |
| 正常完整生成 | completed | stop | available | 是 |
| 续写完成但不索引 | partial | fallback_stop | partial | 是 |
| 超时但有可用正文 | partial | timeout | partial | 是 |
| 用户取消但有正文 | cancelled | cancelled | 不自动创建或 partial | 可选 |
| 真正失败无正文 | failed | error | 不创建 | 否 |

核心原则：

1. `completion_status=partial` 不等于不可用。
2. 只要能够合成新的 `ContentSnapshot`，partial artifact 就应允许成为下一轮基线。
3. 是否索引 KB 与是否作为文档基线分开判断。

### 6.2 allow_complete 与可继续能力解耦

当前 recovered 场景通常是：

```text
allow_complete=false
allow_indexing=false
```

这不应阻止 artifact 创建。

建议语义：

1. `allow_complete` 只代表是否把消息视为完整成功。
2. `allow_indexing` 只代表是否进入聊天历史知识库索引。
3. `artifact.CanContinue()` 由 artifact status、kind、snapshot size 决定。
4. partial artifact 可以 `CanContinue=true`。

## 7. 推荐实施步骤

### 7.1 P0：止血闭环

1. 扩展前后端 continue 意图识别，覆盖“请继续补齐”。
2. 增加 `document_output_mode=delta_only`，continue/revise 默认只流式输出 delta。
3. `BuildQuotedContext()` 将 revise 默认 prompt 改为 delta 输出规则。
4. `buildChatDocumentSnapshot()` 支持对 revise delta 合成完整 snapshot。
5. 对存在 base artifact 的 continue/revise，artifact 创建门槛改为校验合成后 snapshot，而不是校验本轮 delta 长度。
6. recovered partial 生成的新 snapshot 应创建 V2 partial artifact。
7. complete payload 带 `final_document_artifact_id` 和 `final_document_mode`。
8. 前端 complete 后自动把新 artifact 设为当前基线。

### 7.2 P1：完整文档一次性加载

1. complete payload 小快照直接带 `final_document`。
2. 大快照走 `getChatDocumentArtifact()` 拉取 `content_snapshot`。
3. 前端新增完整文档预览状态，例如 `message.final_document_content`。
4. Artifact 卡片支持展开查看完整文档。
5. 页面“当前基线”实时更新到最新 revision。

### 7.3 P2：结构化 patch 合并

1. 定义 `<document_patch>` 协议。
2. 支持 `<replace heading="...">`、`<append heading="...">`、`<insert_after heading="...">`。
3. 后端用 Markdown 标题树精确应用 patch。
4. 合并失败时标记 `delta_merge_uncertain`，但仍可创建 partial artifact。
5. 增加 patch 合并单元测试和端到端测试。

## 8. 验收标准

### 8.1 基线推进

1. 复现截图场景后，页面不再停留在“当前基线 V1”。
2. 本轮 recovered partial 如果生成了可合并内容，应创建 V2 partial artifact。
3. 用户下一轮点击“继续补全”或直接输入“继续”，请求应携带 V2 artifact id。
4. 后端注入的 `<document_*_context>` 中 artifact_id 应为 V2，而不是 V1。

### 8.2 增量流式输出

1. 用户对已有长文档说“请继续补齐”时，流式输出只包含新增章节或补齐片段。
2. 不再从 `# 北海电厂...` 这样的文档开头重新刷屏。
3. `document_output_mode=delta_only` 时，answer chunk 不应承载完整 5 万字文档。
4. 模型输出重复 base 开头时，应触发质量标记或降级提示。

### 8.3 完整文档加载

1. complete 事件后，前端能拿到新 artifact id。
2. 前端能一次性加载新 artifact 的完整 `content_snapshot`。
3. 完整文档预览内容应等于后端合成后的 artifact snapshot。
4. 聊天消息中的 delta 与完整文档预览不互相覆盖。
5. 刷新页面后，artifact 列表和消息卡片仍能展示最新 V2/V3 版本。

### 8.4 回归

1. 普通短问答不创建 artifact。
2. 首次生成完整文档仍创建 V1 available。
3. “重新生成”不携带旧 base artifact。
4. `failed/error` 且无正文时不创建 artifact。
5. 长文档翻译任务不受影响。

## 10. 最终建议

本次问题的核心不是“是否还要修 tool_error”，而是需要把“可恢复 partial”从一个 UI 提示提升为完整的数据闭环。

推荐最终形态是：

1. **流式输出只面向本轮变化**：continue/revise 默认 `delta_only`，避免每次从头输出完整文档。
2. **artifact 保存完整文档状态**：后端负责把 base snapshot 与 delta 合成为新的完整 `ContentSnapshot`。
3. **partial revision 也能成为基线**：只要 recovered 内容可合并，就创建 V2/V3 partial artifact，下一轮基于它继续。
4. **complete 后一次性加载完整文档**：前端通过 inline snapshot 或 artifact fetch 加载完整文件，不再依赖流式全文。
5. **聊天展示与文档版本分离**：聊天消息展示本轮新增/修改片段，artifact 预览展示最终完整文档。

这样可以同时解决页面中的三个痛点：

1. “已保留当前生成内容”之后，当前内容真正成为新基线。
2. 续写或修改时不再从文档开头反复流式输出。
3. 修改完成后，前端能一次性加载并展示最终完整文件。
