# 统一模型路由触发条件审查与修正方案

## 1. 审查背景

本次审查围绕三个问题展开：

1. 当前普通 Agent、普通文档、长问答/长文档分别由哪些条件触发。
2. 现有实现是否存在过多前端正则、后端正则和默认值叠加，导致普通问题误入长文档流程。
3. 是否可以统一入口，把普通生成和长文档生成的能力交给大模型判定，由模型决定应该走哪个流程。

审查结论：统一入口是可行的，而且前端不应该继续做自然语言意图判定。前端应只上传用户 prompt 和少量结构化交互信号，模式判定统一由后端完成。但不建议把“是否允许访问知识库、是否允许续写 artifact、是否可以进入专用长文档编排”等硬约束完全交给模型。推荐采用“前端只传 prompt 与显式信号 + 后端统一路由 + 模型结构化辅助判定 + 代码硬约束校验 + 保守降级”的方案。

## 2. 当前触发条件总览

### 2.1 当前前端入口：目前仍在决定 endpoint、intent_hint 和 document_output_mode

位置：`frontend/src/views/chat/index.vue`

当前发送请求时，前端先根据是否启用 Agent 选择接口：

```js
const endpoint = agentEnabled ? '/api/v1/agent-chat' : '/api/v1/knowledge-chat';
```

随后通过 `inferDocumentContinuationIntent` 推断文档续写/修订意图：

```js
const inferDocumentContinuationIntent = (query = '') => {
    const text = query.trim().toLowerCase();
    if (!text) return 'normal';
    if (/(重新生成|从头生成|重写一版|不要基于上一版)/.test(text)) return 'regenerate_document';
    if (/(修改上一版|基于上一个文档修改|把上一份改成|调整上一版|完善上一版)/.test(text)) return 'revise_document';
    if (hasSectionContinuationTarget(query)) return 'revise_document';
    if (/(继续生成|接着写|续写|从上次中断处继续|补全剩余|继续输出|请继续补齐|继续补齐|继续补充|继续完善|接着补齐|补齐剩余|补充剩余|继续扩写)/.test(text)) return 'continue_document';
    return 'normal';
};
```

再通过 `inferDocumentOutputMode` 推断输出模式：

```js
const inferDocumentOutputMode = (query = '', intentHint = 'normal') => {
    const text = query.trim().toLowerCase();
    if (/(输出完整新版全文|给我完整文档|输出完整文档|完整新版全文|重新生成完整方案)/.test(text)) {
        return 'full_document';
    }
    if (intentHint === 'continue_document' || intentHint === 'revise_document') {
        return 'delta_only';
    }
    if (intentHint === 'regenerate_document') {
        return 'full_document';
    }
    return 'full_document';
};
```

这里有两个关键问题：

1. 文档意图判断依赖前端正则，且与后端 `DetectIntent` 重复维护。
2. 普通 `normal` 意图默认返回 `full_document`，导致所有普通问题都会携带完整文档模式。

从架构职责上看，这一层本就不应该存在自然语言意图判断。前端最了解的是用户显式交互状态，而不是语义路由结果。前端可以传：

1. 原始 `query`。
2. 用户是否显式开启 Agent。
3. 用户选择的知识库、文件、MCP、附件、图片。
4. 用户是否点击“继续写”“修改上一版”“长文档模式”等显式按钮。
5. 当前是否绑定某个 `base_artifact_id`。

但前端不应再根据 query 文本自己推断 `continue_document`、`revise_document`、`full_document`。这些都应由后端统一判定。

### 2.2 Handler 入口：再次推断文档 intent，并默认补 full_document

位置：`internal/handler/session/qa.go`

`parseQARequest` 会调用：

```go
documentPrep := h.prepareDocumentRequest(ctx, session, documentContextQuery, request.IntentHint, request.BaseArtifactID, request.DocumentOutputMode, request.DocumentTargetHeading, request.DocumentMergeMode)
reqCtx.documentIntent = documentPrep.intent
reqCtx.documentOperation = documentPrep.operation
reqCtx.baseArtifact = documentPrep.baseArtifact
reqCtx.documentQuotedContext = documentPrep.quotedContext
reqCtx.documentOutputMode = normalizeDocumentOutputModeForRequest(request.DocumentOutputMode, reqCtx.documentIntent)
```

位置：`internal/handler/session/chat_document_artifact.go`

`prepareDocumentRequest` 默认返回：

```go
result := documentRequestPreparation{
    intent:    types.ChatDocumentIntentNormal,
    operation: types.ChatDocumentOperationCreate,
}
```

只有当检测到 `continue_document` 或 `revise_document` 时，才会尝试加载已有 artifact 并构建 quoted context。

`normalizeDocumentOutputModeForRequest` 在没有显式传入模式时默认返回 `full_document`：

```go
switch intent {
case types.ChatDocumentIntentContinue, types.ChatDocumentIntentRevise:
    return types.ChatDocumentOutputModeDelta
case types.ChatDocumentIntentRegenerate:
    return types.ChatDocumentOutputModeFull
default:
    return types.ChatDocumentOutputModeFull
}
```

这意味着即使前端修复了默认值，API、旧前端或 IM 入口不传 `document_output_mode` 时，后端仍可能把普通问题升级为完整文档任务。

### 2.3 普通知识问答：由 handler 的 qaModeNormal 进入 KnowledgeQA

位置：`internal/handler/session/qa.go`

`KnowledgeQA` 固定调用：

```go
h.executeQA(reqCtx, qaModeNormal, !request.DisableTitle)
```

`AgentQA` 会先看 Agent 模式是否启用：

```go
agentModeEnabled := request.AgentEnabled
if reqCtx.customAgent != nil {
    agentModeEnabled = reqCtx.customAgent.IsAgentMode()
}

if agentModeEnabled {
    h.executeQA(reqCtx, qaModeAgent, true)
} else {
    h.executeQA(reqCtx, qaModeNormal, false)
}
```

位置：`internal/application/service/session_knowledge_qa.go`

普通 KnowledgeQA 的实际 pipeline 很清晰：

```go
hasKB := len(knowledgeBaseIDs) > 0 || len(knowledgeIDs) > 0
needsRAG := hasKB || req.WebSearchEnabled

if !needsRAG {
    // Pure chat
    pipeline = LOAD_HISTORY + MEMORY_RETRIEVAL + CHAT_COMPLETION_STREAM + MEMORY_STORAGE
} else {
    // RAG
    pipeline = LOAD_HISTORY + QUERY_UNDERSTAND + CHUNK_SEARCH_PARALLEL + CHUNK_RERANK + WEB_FETCH + CHUNK_MERGE + FILTER_TOP_K + DATA_ANALYSIS + INTO_CHAT_MESSAGE + CHAT_COMPLETION_STREAM
}
```

这条路径本身不直接触发长文档编排。它的问题在于 handler 传入的文档元数据会影响 artifact 注册和索引元数据。

### 2.4 普通 Agent：进入 AgentQA 后先尝试专用文档路径

位置：`internal/application/service/session_agent_qa.go`

AgentQA 的路由顺序如下：

```go
if shouldUseKnowledgeGroundedDocumentContinuationPath(req, agentConfig) {
    return s.runKnowledgeGroundedDocumentContinuationPath(...)
}
if shouldUseDedicatedDocumentEditPath(req) {
    return s.runDedicatedDocumentEditPath(...)
}
if shouldUseKnowledgeGroundedFullDocumentGenerationPath(req, agentConfig) {
    return s.runKnowledgeGroundedFullDocumentGenerationPath(...)
}
if shouldUseDedicatedFullDocumentGenerationPath(req, agentConfig) {
    return s.runDedicatedFullDocumentGenerationPath(...)
}
```

只有上述专用路径都不命中，才创建通用 AgentEngine，进入普通 Agent ReAct/tool calling 流程。

当前专用路径优先级高于普通 Agent，这是合理的；但触发条件过于依赖 `DocumentOutputMode` 和硬编码字段，缺少统一的“任务类型决策”。

### 2.5 专用文档编辑路径触发条件

`shouldUseDedicatedDocumentEditPath` 条件：

1. 存在 `BaseArtifactID`。
2. `DocumentOutputMode != full_document`。
3. `DocumentIntent` 或 `DocumentOperation` 是 continue/revise。
4. 没有显式知识库、知识文件，也没有外部检索关键词。
5. 没有附件、图片或图片描述。
6. `QuotedContext` 不为空。

这条路径适合“基于上一份文档继续/修改”的轻量专用编辑，不适合知识库支撑的长文档续写。

### 2.6 知识库支撑长文档续写路径触发条件

`shouldUseKnowledgeGroundedDocumentContinuationPath` 条件：

1. `AutoContinue == true`。
2. 存在 `BaseArtifactID` 且 `BaseArtifact != nil`。
3. `DocumentOutputMode == delta_only`。
4. 有有效本地知识库范围。
5. intent/operation 是 continue 或 revise。
6. 没有附件、图片或图片描述。

这条路径偏后端自动续写，是长文档生成运行态的一部分，不应由普通用户问题误触发。

### 2.7 知识库支撑完整长文档路径触发条件

`shouldUseKnowledgeGroundedFullDocumentGenerationPath` 条件：

1. `DocumentOutputMode == full_document`。
2. query 不为空，且没有 `BaseArtifactID`。
3. `AutoContinue == false`。
4. intent 是空、normal 或 regenerate。
5. operation 是空、create 或 regenerate。
6. 有有效本地知识库范围。
7. 没有附件、图片或图片描述。

当前没有判断“用户是否真的要生成完整文档”。因此只要普通问题携带 `full_document`，且 Agent 有知识库范围，就会进入这条路径。

### 2.8 无知识库完整长文档路径触发条件

`shouldUseDedicatedFullDocumentGenerationPath` 条件与知识库长文档类似，但要求：

1. 没有有效本地知识库范围。
2. query 不包含 `documentEditRequiresExternalRetrieval` 里的外部检索关键词。

这条路径用于纯模型生成完整文档，如“请输出完整技术方案”。但它同样只依赖 `full_document`，没有可靠的任务类型决策。

### 2.9 普通文档 artifact 注册触发条件

位置：`internal/application/service/chat_document_artifact.go`

`RegisterFromAssistantMessage` 会在 assistant 完成后调用 `shouldRegisterChatDocumentArtifact`。当前普通文档 artifact 的创建是基于输出内容形态和用户 query hint：

```go
if runeLen(trimmed) < 2000 {
    return false
}
headingCount := len(chatDocumentHeadingRE.FindAllStringSubmatch(trimmed, -1))
if headingCount >= 2 {
    return true
}
if headingCount >= 1 && (chatDocumentListRE.MatchString(trimmed) || chatDocumentTableRE.MatchString(trimmed)) {
    return true
}
return chatDocumentQueryHintRE.MatchString(options.UserQuery) && runeLen(trimmed) >= 3000
```

这里的 `chatDocumentQueryHintRE` 也是正则：

```go
regexp.MustCompile(`(?i)(方案|文档|报告|markdown|技术方案|设计方案|plan|report|document)`)
```

也就是说，普通文档 artifact 是否创建，当前并不是由统一路由决定，而是由最终内容形态和 query hint 兜底判断。

## 3. 当前实现的主要问题

### 3.1 同一件事被前后端重复推断

文档意图在前端 `inferDocumentContinuationIntent` 推断一次，在后端 `DetectIntent` 又推断一次。两处规则不完全一致，任何一侧新增关键词都可能造成分歧。

进一步说，前端根本不应该承担这类推断职责。前端天然缺少后端掌握的上下文：

1. 当前 Agent 的真实配置与权限。
2. 当前会话是否存在可续写 artifact。
3. 当前请求是否具备有效知识库范围。
4. 当前附件、图片、自动续写状态是否允许进入专用路径。

因此即使前端正则写得更复杂，也无法可靠做出最终路由判断。

### 3.2 `full_document` 同时承担了“输出形态”和“任务意图”两个职责

`DocumentOutputMode=full_document` 本应只是输出模式，却在 AgentQA 中被当成长文档任务触发器。当前代码没有区分：

1. 用户想要一个完整答案。
2. 用户想要一份普通短文档。
3. 用户想要一份多章节长文档。
4. 用户只是普通事实问答，但前端默认带了 full。

### 3.3 普通请求默认带文档上下文，污染通用 Agent

`prepareDocumentRequest` 默认 `normal/create`，`normalizeDocumentOutputModeForRequest` 默认 `full_document`。这会让 `agentDocumentContextFromQARequest` 对普通请求也返回非空 document context，通用 Agent final synthesis 会收到不必要的文档上下文。

### 3.4 专用长文档路径缺少任务类型闸门

专用路径用硬编码条件截流，但没有独立的“路由决策对象”。这导致任何入口只要把字段拼成 `full_document + has KB`，就会绕过普通 Agent，直接进入长文档编排。

### 3.5 普通文档与长文档边界不清

当前 artifact 注册通过内容长度、标题数量、列表/表格形态判断普通文档；长文档则通过 `full_document` 进入专用编排。二者之间没有统一的“短文档/长文档/普通问答”分类标准。

## 4. 用户需求可行性评估

用户希望“不要通过简单正则判定用户意图”，并考虑“统一入口，把系统普通生成和长文档生成能力交给大模型，由大模型判定应该走哪个流程”。

这个方向可行，而且前端应退出自然语言判定链路，但要拆成两层：

1. 可交给大模型的部分：用户任务类型分类、是否需要文档产物、是否需要长文档编排、是否是续写/修订、是否需要知识库支撑、是否需要普通 Agent 工具能力。
2. 必须由代码掌握的部分：权限、知识库范围、Agent 是否启用、artifact 是否存在且可续写、附件/图片约束、自动续写状态、失败降级、幂等与审计。

不建议让模型直接决定“调用哪个 Go 函数”或绕过后端校验。推荐让前端只传原始 prompt 与显式交互信号，再由后端统一构建路由输入，必要时调用模型输出结构化 JSON 路由决策，最后由后端把这个决策映射到现有执行路径。

## 5. 推荐目标架构：统一模型路由器

### 5.1 新增统一路由对象

建议在 `internal/types` 增加：

```go
type ChatRouteKind string

const (
    ChatRouteNormalQA                    ChatRouteKind = "normal_qa"
    ChatRouteAgentQA                     ChatRouteKind = "agent_qa"
    ChatRouteShortDocument               ChatRouteKind = "short_document"
    ChatRouteFullDocument                ChatRouteKind = "full_document"
    ChatRouteKnowledgeGroundedFullDoc    ChatRouteKind = "knowledge_grounded_full_document"
    ChatRouteDocumentEdit                ChatRouteKind = "document_edit"
    ChatRouteKnowledgeGroundedContinue   ChatRouteKind = "knowledge_grounded_document_continuation"
)

type ChatRouteDecision struct {
    Kind             ChatRouteKind `json:"kind"`
    Intent           string        `json:"intent"`
    Operation        string        `json:"operation"`
    OutputMode       string        `json:"output_mode"`
    UseAgent         bool          `json:"use_agent"`
    UseKnowledge     bool          `json:"use_knowledge"`
    UseLongDocument  bool          `json:"use_long_document"`
    NeedArtifact     bool          `json:"need_artifact"`
    TargetHeading    string        `json:"target_heading,omitempty"`
    MergeMode        string        `json:"merge_mode,omitempty"`
    Confidence       float64       `json:"confidence"`
    Reason           string        `json:"reason"`
}
```

`QARequest` 增加：

```go
RouteDecision *types.ChatRouteDecision
```

后续服务层不再直接根据 `DocumentOutputMode` 做业务路由，而是先看 `RouteDecision.Kind`。

### 5.2 新增 `ChatRouteService`

建议新增文件：

```text
internal/application/service/chat_route_service.go
```

接口建议：

```go
type ChatRouteInput struct {
    Query                    string
    Channel                  string
    EndpointMode             string
    AgentConfigured          bool
    AgentModeEnabledByConfig bool
    HasSelectedKnowledge     bool
    HasEffectiveAgentKB      bool
    WebSearchEnabled         bool
    HasAttachments           bool
    HasImages                bool
    AutoContinue             bool
    ExplicitBaseArtifactID   string
    LatestArtifactSummary    *ChatArtifactRouteSummary
    UserExplicitOutputMode   string
    UserRequestedRoute       string
}

type ChatRouteService interface {
    Decide(ctx context.Context, input ChatRouteInput) (*types.ChatRouteDecision, error)
}
```

### 5.3 模型路由提示词原则

模型路由只做分类，不调用工具，不生成答案。

系统提示建议包含：

```text
你是对话任务路由器，只能返回 JSON。
不要回答用户问题，不要生成正文。
根据用户问题、是否有知识库、是否有可续写文档、是否启用 Agent，选择最合适的执行路线。
普通事实查询、字段查询、包含哪些信息、是什么、多少、何时、谁负责等，默认 normal_qa 或 agent_qa，不要选择 full_document。
只有用户明确要求撰写/生成一份完整文档、报告、方案、标书、长篇 Markdown，才选择 full_document 或 knowledge_grounded_full_document。
如果用户要求修改、补充、续写上一份文档，且存在可续写文档，选择 document_edit 或 knowledge_grounded_document_continuation。
如果不确定，选择 agent_qa，并把 confidence 设为低。
```

输出 schema 必须固定，示例：

```json
{
  "kind": "normal_qa",
  "intent": "normal",
  "operation": "",
  "output_mode": "",
  "use_agent": true,
  "use_knowledge": true,
  "use_long_document": false,
  "need_artifact": false,
  "target_heading": "",
  "merge_mode": "",
  "confidence": 0.86,
  "reason": "用户询问登记表包含哪些信息，是事实查询，不是完整文档生成。"
}
```

### 5.4 后端强校验规则

模型输出不能直接生效，必须通过代码校验：

1. 如果 `AutoContinue=true`，只能进入 continuation 类路线，不能进入普通 full document。
2. 如果 `BaseArtifactID` 为空且没有可用 latest artifact，不能进入 document_edit 或 continuation。
3. 如果 Agent 配置禁用 Agent mode，模型不能把请求升级为 AgentQA。
4. 如果没有有效知识库范围，不能进入 `knowledge_grounded_full_document`。
5. 如果有附件/图片且专用文档路径不支持，不能进入 dedicated document path，应回退通用 Agent 或 KnowledgeQA。
6. 如果模型返回未知 kind、JSON 解析失败、超时、confidence 低于阈值，应保守回退普通问答，而不是长文档。
7. 外部传入的 `document_output_mode=full_document` 只能作为强提示，不能绕过模型路由和代码校验。

### 5.5 保守降级策略

路由失败时建议：

```text
有 Agent 且 Agent mode 开启 -> generic agent
否则有 KB/WebSearch -> KnowledgeQA RAG
否则 -> KnowledgeQA pure chat
```

绝不在路由失败时默认长文档。

## 6. 建议落地步骤

### 6.1 第一阶段：去掉前后端默认 full_document，并增加后端止血闸门

这是 P0 止血项。

这一阶段的目标不是一次性移除前端全部自然语言判定逻辑，而是先把“普通请求默认被升级为完整文档任务”的错误路径切断，避免继续影响线上问答。

第一阶段应包含以下内容：

1. 前端停止对普通 `normal` 请求默认发送 `document_output_mode=full_document`。
2. 前端暂时保留现有 `continue_document`、`revise_document`、`regenerate_document` 相关推断与 `base_artifact_id` 绑定逻辑，避免在统一后端路由尚未落地前直接打断文档续写/修订能力。
3. 后端 `normalizeDocumentOutputModeForRequest` 不再对 `normal/create` 默认返回 `full_document`。
4. 后端 `prepareDocumentRequest` 对普通请求不再填充 `DocumentIntent=normal`、`DocumentOperation=create`，避免给通用 Agent 注入伪造的文档上下文。
5. AgentQA 的 `knowledge_grounded_full_document` 和 `dedicated_full_document` 路径增加临时止血闸门：没有显式完整文档意图时，不能仅凭 `DocumentOutputMode=full_document` 进入长文档编排。
6. 为第一阶段补回归测试：
    - 普通事实查询在有知识库、有 Agent 的情况下不进入长文档。
    - 显式“请输出完整技术方案”仍然能够进入完整文档路径。
    - `continue/revise/regenerate` 现有流程不被本阶段改动打断。

这一阶段不需要引入模型路由，也不要求前端已经完全退出自然语言判定。它的职责是先止血，确保“默认 full_document”不再把普通问答误导入长文档。

前端彻底退出自然语言意图判定，改为只传 `prompt + 显式结构化信号`，应放在统一后端路由落地后的下一阶段，而不是塞进本阶段一次完成。

### 6.2 第二阶段：引入模型路由，但只灰度记录不生效

新增 `ChatRouteService.Decide`，在 handler parse 完成后调用，写入日志和 completion extra，但不改变实际路径。

建议日志：

```text
[ChatRouter] decision kind=normal_qa confidence=0.91 reason="..." requested_output_mode=full_document effective_agent_kb=true
```

用 1-2 周日志对比模型路由与现有硬路由差异，收集误判样本。

### 6.3 第三阶段：模型路由生效，但只控制文档分流

生效范围先限制在文档相关路径：

1. `document_edit`
2. `knowledge_grounded_document_continuation`
3. `full_document`
4. `knowledge_grounded_full_document`
5. `short_document`

普通 KnowledgeQA 与普通 Agent 仍按现有 `agentModeEnabled` 选择，只是不再被 `DocumentOutputMode` 误截流。

### 6.4 第四阶段：统一 Agent/KnowledgeQA 入口

当文档路由稳定后，可以考虑把 `/agent-chat` 与 `/knowledge-chat` 进一步收敛为统一 `/chat` 入口。

但这一步要谨慎：Agent 是否启用目前受 customAgent 配置控制，涉及工具权限、rerank model、MCP、知识库范围和成本。模型可以建议 `use_agent`，但最终是否进入 AgentQA 应以配置和权限为准。

## 7. 对现有函数的修正建议

### 7.1 `frontend/src/views/chat/index.vue`

目标：前端完全退出自然语言意图判定，只传用户 prompt 和显式交互状态。

建议：

1. 删除 `inferDocumentContinuationIntent`、`inferDocumentOutputMode`、`hasSectionContinuationTarget` 这类基于 query 的路由推断逻辑。
2. 保留显式 artifact 选择、自动续写、继续按钮、长文档模式按钮等结构化场景。
3. 请求体默认只传 `query` 和结构化信号，不再传 `intent_hint`、推断出来的 `document_output_mode`。
4. 如果未来有“长文档模式”按钮，前端传 `requested_route=full_document`；如果用户点击“继续上一版”，前端传 `base_artifact_id` 或 `requested_route=document_edit`。这些都是显式 UI 信号，不是自然语言猜测。
5. `/agent-chat` 与 `/knowledge-chat` 的选择如果短期无法统一，可以继续由前端按用户当前选中的 Agent 配置决定；这是配置选择，不是用户意图判定。

### 7.2 `internal/handler/session/chat_document_artifact.go`

目标：handler 接管模式判定入口，不再依赖前端传来的自然语言推断结果。

建议：

1. `prepareDocumentRequest` 默认返回空 intent/operation，而不是 `normal/create`。
2. `normalizeDocumentOutputModeForRequest` 在 normal 情况返回空。
3. `DetectIntent` 可以短期保留作为低优先级 fallback，但它只能运行在后端，不能再要求前端同步维护同一套规则。后续由 `ChatRouteService` 替代。

### 7.3 `internal/handler/session/qa.go`

目标：在后端统一调用路由器，并把路由结果写入 `QARequest`。

建议新增流程：

```go
routeDecision := h.chatRouteService.Decide(ctx, buildChatRouteInput(reqCtx, request))
routeDecision = validateAndNormalizeChatRouteDecision(routeDecision, reqCtx, request)
applyChatRouteDecision(reqCtx, routeDecision)
```

`artifactOptions` 使用 route decision：

```go
artifactOptions := types.RegisterChatDocumentArtifactOptions{
    UserQuery:     reqCtx.query,
    Intent:        routeDecision.Intent,
    Operation:     routeDecision.Operation,
    OutputMode:    routeDecision.OutputMode,
    TargetHeading: routeDecision.TargetHeading,
    MergeMode:     routeDecision.MergeMode,
    BaseArtifact:  reqCtx.baseArtifact,
}
```

### 7.4 `internal/application/service/session_agent_qa.go`

目标：服务层专用路径只信任规范化后的 route decision。

建议替换：

```go
if shouldUseKnowledgeGroundedFullDocumentGenerationPath(req, agentConfig) { ... }
```

为：

```go
switch normalizedAgentExecutionRoute(req, agentConfig) {
case types.ChatRouteKnowledgeGroundedContinue:
    return s.runKnowledgeGroundedDocumentContinuationPath(...)
case types.ChatRouteDocumentEdit:
    return s.runDedicatedDocumentEditPath(...)
case types.ChatRouteKnowledgeGroundedFullDoc:
    return s.runKnowledgeGroundedFullDocumentGenerationPath(...)
case types.ChatRouteFullDocument:
    return s.runDedicatedFullDocumentGenerationPath(...)
default:
    // generic agent
}
```

保留现有 `shouldUse...` 函数作为校验器，而不是主路由器。

### 7.5 `internal/application/service/chat_document_artifact.go`

目标：普通文档 artifact 是否创建优先参考 route decision，而不是仅靠内容形态正则。

建议：

1. `RegisterChatDocumentArtifactOptions` 增加 `NeedArtifact bool` 或 `RouteKind string`。
2. `shouldRegisterChatDocumentArtifact` 先判断 `NeedArtifact`。
3. 现有内容形态判断保留为兜底，但不能把普通问答误标成长文档。

## 8. 模型路由与正则方案对比

| 方案 | 优点 | 风险 | 建议 |
| --- | --- | --- | --- |
| 纯正则 | 快、成本低、可预测 | 维护成本高，语言变体多，容易误判 | 只用于极少数结构化命令或兜底，不作为主判定 |
| 纯模型决定 | 语义理解好，适配表达灵活 | 延迟、成本、不稳定、可能被提示注入影响 | 不建议裸用 |
| 模型结构化路由 + 代码校验 | 兼顾语义和安全边界 | 需要新增路由服务和灰度监控 | 推荐 |
| UI 显式模式按钮 | 最稳定、用户可控 | 需要产品交互改造 | 可作为高优先级强提示，与模型路由并存 |

## 9. 验收用例建议

### 9.1 普通事实问答

输入：

```text
河南省郑州市中原区桐柏路206号5楼32号的火灾警情登记表包含哪些信息？
```

期望路由：

```json
{
  "kind": "agent_qa" 或 "normal_qa",
  "use_long_document": false,
  "output_mode": "",
  "need_artifact": false
}
```

不得出现：

```text
[FullDocument][Router] selected=knowledge_grounded_full_document
[DocumentBudget]
```

### 9.2 普通知识库问答

输入：

```text
请根据知识库说明二级火警调派消防站和车辆的标准是什么？
```

期望：进入普通 RAG 或 generic Agent，不进入长文档。

### 9.3 普通短文档生成

输入：

```text
请写一份会议纪要，内容包括参会人员、会议结论和后续事项。
```

期望：

```json
{
  "kind": "short_document",
  "use_long_document": false,
  "need_artifact": true
}
```

执行路径可以是 KnowledgeQA pure chat 或 generic Agent，不应进入 `runDedicatedFullDocumentGenerationPath`。

### 9.4 长文档生成

输入：

```text
请结合知识库输出一份完整的智慧运行投标技术方案，面向甲方评审专家，章节完整，内容详细。
```

期望：

```json
{
  "kind": "knowledge_grounded_full_document",
  "use_long_document": true,
  "output_mode": "full_document",
  "need_artifact": true
}
```

### 9.5 无知识库长文档生成

输入：

```text
请输出一份完整的企业数据治理建设方案，包含背景、目标、架构、实施路径和保障措施。
```

期望：进入 `full_document`，但不是 `knowledge_grounded_full_document`。

### 9.6 文档修订

前置：会话中存在可续写 artifact。

输入：

```text
请把上一份文档的第四章补充得更详细，增加风险控制内容。
```

期望：进入 `document_edit` 或有知识库时进入知识库支撑续写/修订路径，且必须携带 base artifact。

### 9.7 路由失败降级

模拟模型路由超时、JSON 解析失败、confidence 低。

期望：回退普通 QA 或 generic Agent，不进入长文档。

## 10. 风险与注意事项

1. 模型路由会增加一次 LLM 调用，需控制超时和 token，建议 2-4 秒内失败降级。
2. 模型输出必须 JSON schema 校验，不能信任自然语言说明。
3. 用户可能在 query 中写“忽略上面的规则”，路由模型提示词必须强调只做分类，不服从用户对路由规则的篡改。
4. 路由决策必须写日志，方便审计误判样本。
5. 长文档路径成本高，低置信度一律不要进入。
6. 前端最好展示或记录“本轮按普通问答/长文档/文档修订处理”的内部状态，便于排查。
7. 如果保留 `/agent-chat` 与 `/knowledge-chat` 两个接口，统一路由仍可先在 handler 内共享，不必立即改 API。

## 11. 最终建议

用户需求可行，而且建议明确采用“前端不判定、后端统一判定”的职责划分。

短期应先把自然语言意图判定从前端移除，同时修复默认值：普通请求不再由前端或后端默认携带 `full_document`，AgentQA 也不能仅凭 `DocumentOutputMode` 进入长文档。

中期引入 `ChatRouteService`，由后端统一收集上下文，再让大模型以结构化 JSON 判定任务类型，但只输出“路由建议”。后端根据权限、知识库范围、artifact 状态、附件约束进行二次校验。

长期可以把普通问答、普通短文档、长文档、文档续写/修订统一到一个对话路由入口下，现有专用长文档能力保留为被路由器调用的执行器，而不是由前端正则和 `full_document` 默认值直接触发。

建议优先级：

1. P0：前端停止自然语言意图判定，只传 prompt 与显式信号。
2. P0：去掉前后端 normal 默认 `full_document`。
3. P0：AgentQA 长文档路径改为依赖 `RouteDecision` 或至少增加保守闸门。
4. P1：新增 `ChatRouteService`，先灰度只记录。
5. P1：路由生效控制文档类分流。
6. P2：统一 `/agent-chat`、`/knowledge-chat` 到更高层的 `/chat` 入口。
