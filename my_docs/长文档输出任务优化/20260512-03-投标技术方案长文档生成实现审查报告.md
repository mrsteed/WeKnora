# 投标技术方案长文档生成实现审查报告

## 1. 审查结论

基于后端日志 [my_docs/test/13.md](my_docs/test/13.md#L32) 至 [my_docs/test/13.md](my_docs/test/13.md#L2056) 以及当前后端实现，本次“提交给甲方的投标技术方案，业务部门和技术专家、智慧运行相关技术方案均需详细阐述”的对话主链路已经基本符合长文档生成设计要求：请求被识别为 `full_document`，在智能体配置知识库有效但请求体未显式携带知识库 ID 的情况下，后端正确路由到 `knowledge_grounded_full_document`，并按“本地知识检索 -> 大纲规划 -> 逐章节检索 -> 逐章节流式生成 -> 完成落库 -> 注册文档产物”的受控链路执行。

但实现仍不建议判定为完全达标。主要遗留问题有两类：一是生成正文的小标题 Markdown 规范不足，日志中可见 `###3.1...` 这类缺少空格且标题与正文粘连的输出，影响提交甲方的正式文档质量；二是完成后的聊天历史知识库索引将 50249 字符 Q&A 作为单个 chunk 做 embedding，触发输入长度超过 8192 的错误，虽然本次主响应、消息落库和文档产物注册成功，但会影响历史对话检索质量和后台稳定性。

## 3. 日志事实摘要

本次请求在 [my_docs/test/13.md](my_docs/test/13.md#L32) 中明确携带 `document_output_mode=full_document`、`agent_enabled=true`、`agent_id=22a6c00a-9e10-4776-8465-aec4463d21e2`，且 `knowledge_base_ids=null`。这正是此前设计中特别强调的场景：不能只看请求体知识库 ID，还必须纳入智能体配置中的有效知识库范围。

日志显示后端解析自定义智能体“招投标专家”，模式为 `smart-reasoning`，并在 [my_docs/test/13.md](my_docs/test/13.md#L63) 记录智能体工具与配置，在 [my_docs/test/13.md](my_docs/test/13.md#L73) 明确选择 `[FullDocument][Router] selected=knowledge_grounded_full_document kb_scope=1`。这说明当前路由实现已经覆盖“请求体无 KB，但智能体有 KB”的设计要求。

日志中多次出现章节流式生成自然停止，例如大纲规划后在 [my_docs/test/13.md](my_docs/test/13.md#L300-L301) 出现 `Natural-stop` 和用量统计，第一章在 [my_docs/test/13.md](my_docs/test/13.md#L510-L511) 完成后触发运行时预算反馈，后续章节也持续出现自然停止与 token 统计，如 [my_docs/test/13.md](my_docs/test/13.md#L721-L722)、[my_docs/test/13.md](my_docs/test/13.md#L931-L932)、[my_docs/test/13.md](my_docs/test/13.md#L1141-L1142)、[my_docs/test/13.md](my_docs/test/13.md#L1560-L1561)、[my_docs/test/13.md](my_docs/test/13.md#L1770-L1771) 和 [my_docs/test/13.md](my_docs/test/13.md#L1980-L1981)。

## 4. 当前代码实现对照

### 4.1 路由实现

[internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L105-L123) 在构建智能体配置和 `SearchTargets` 后执行长文档路由，优先判断知识库续写、文档编辑、知识库完整文档生成和纯模型完整文档生成。本次日志命中的就是 `shouldUseKnowledgeGroundedFullDocumentGenerationPath` 分支。

该实现与设计文档 [my_docs/20260511-10-长文档生成未查询本地知识库原因分析与修正方案.md](my_docs/20260511-10-长文档生成未查询本地知识库原因分析与修正方案.md#L280-L300) 的路由矩阵一致：`full_document` 且有效 KB 范围非空，应进入 `runKnowledgeGroundedFullDocumentGenerationPath`。

### 4.2 知识库完整文档路径

[internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L3255-L3492) 实现了知识库完整文档生成路径。它先检索大纲证据，再生成大纲；随后使用 `fullDocumentSectionsForInitialRun(outline)` 取得完整章节列表，逐章检索证据、发出 H2 标题、调用模型生成章节正文，并在最终阶段构建 `evidence_refs`、`completed_sections`、`documentGenerationStatus` 等元数据。

关键点是 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L914-L920) 的 `fullDocumentSectionsForInitialRun` 当前直接返回全部 `outline.Sections`，不再对首轮做固定章节数截断。这与此前“首轮完整文档必须遍历全部大纲章节”的设计要求一致。

### 4.3 章节流与完成判定

[internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L1344-L1459) 的 `consumeFullDocumentSectionStream` 会区分 timeout、channel 关闭、错误响应和 `finish_reason=length`。只有自然完成才返回 completed；length、timeout 或截断会返回 partial，并把 `documentGenerationState` 设置为 continuing。

完整性校验由 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L1729-L1744) 的 `completedFullDocumentIntegrityFailureReason` 负责：它会检查大纲有效性、已完成章节数量、最终渲染标题和缺失章节。该校验能防止 H1/H2 级别的结构不完整被误判为 completed。

### 4.4 完成事件与 artifact 注册

[internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go#L2810-L2854) 的 `emitFullDocumentCompletion` 统一发出 `EventAgentComplete`，并携带 `DocumentGenerationStatus`、`AutoContinueNext`、`KnowledgeRefs`、`AgentSteps` 等字段。

[internal/handler/session/qa.go](internal/handler/session/qa.go#L1218-L1235) 在 Agent 模式下监听 `EventAgentComplete`，调用 `completeAssistantMessage`。该函数在 [internal/handler/session/qa.go](internal/handler/session/qa.go#L1455-L1518) 进行完成状态持久化和无效完整文档兜底校验，并在 [internal/handler/session/qa.go](internal/handler/session/qa.go#L1524-L1568) 调用 `RegisterFromAssistantMessage` 注册文档产物。

[internal/application/service/chat_document_artifact.go](internal/application/service/chat_document_artifact.go#L600-L700) 的注册逻辑接受 completed 或 partial 消息，计算 `DocumentGenerationStatus`，构建 `ContentSnapshot`、状态、标题、checksum 和 evidence refs。因此，本次日志中 artifact 注册成功符合当前设计预期。

## 5. 发现的问题

### 5.1 P1：聊天历史索引对长文档单 chunk embedding 超长

本次主响应完成后，系统异步把 Q&A 写入聊天历史知识库。日志显示 [my_docs/test/13.md](my_docs/test/13.md#L2001) 开始索引到 chat history KB，随后 [my_docs/test/13.md](my_docs/test/13.md#L2030-L2032) 显示 DocReader 只产生 1 个 chunk，且内容长度为 50249；[my_docs/test/13.md](my_docs/test/13.md#L2039) 报错 `INVALID length=50249 (must be [1, 8192])`。

该问题不影响本次用户主响应完成、消息落库和 artifact 注册，但会导致长文档对话历史的向量索引质量不可控。更严重的是，日志中后续仍出现 `Successfully batch saved 1 indices` 和 `processChunks successfully`，容易掩盖 embedding 输入实际无效的问题。建议将长答案的聊天历史索引改为先按 embedding 模型输入上限切分，或对 `full_document` 类型仅索引摘要、标题、大纲和 artifact 元数据，避免把完整 5 万字符 Q&A 作为单段 passage 送入 embedding。

### 5.2 P2：正文内部 Markdown 小标题格式不满足正式投标文档质量

日志中的已生成正文出现 `###3.1时代背景与行业驱动力...`、`###3.2北海电厂面临的挑战与机遇...`、`###3.3建设目标...` 这类格式，见 [my_docs/test/13.md](my_docs/test/13.md#L717)。这类标题缺少 Markdown 标题标记后的空格，并且标题与正文直接粘连，渲染器可能无法识别为标题，也会降低提交甲方文档的专业感。

当前完整性校验主要检查 H1/H2 是否缺失，无法发现 H3 以下标题格式问题。建议在章节生成 Prompt 中明确要求“所有 Markdown 标题必须为 `### 3.1 标题` 格式，标题后必须换行”；同时在后处理或质量评估中加入正则检测，例如发现 `^#{1,6}\S` 或 `^#{1,6}\s*\d+(?:\.\d+)*[^\s]` 时降级为 `needs_review` 或自动修复。

### 5.3 P2：运行耗时较长，用户侧等待压力仍然存在

本次完整生成从 11:37:42 到 11:47:09，主请求耗时约 8 分 58 秒，见 [my_docs/test/13.md](my_docs/test/13.md#L2010)。多章生成首 token 较慢，运行时预算反馈中已经出现 `section_top_k_down_slow_first_token`，例如 [my_docs/test/13.md](my_docs/test/13.md#L511) 和 [my_docs/test/13.md](my_docs/test/13.md#L932)。

当前系统没有失败，但对前端体验来说仍依赖持续进度提示和 SSE 稳定性。建议继续保留 heartbeat，并在 UI 中展示章节级进度、预计剩余章节数和当前阶段，避免用户误以为卡死。

### 5.4 P3：遥测链路存在尾部 deadline 日志

日志尾部 [my_docs/test/13.md](my_docs/test/13.md#L2056) 出现 `traces export: context deadline exceeded`。该错误发生在主响应完成、artifact 注册、聊天历史索引任务之后，更像遥测导出链路问题，不影响本次业务功能。但如果在生产环境频繁出现，会降低链路追踪可用性，需要单独检查 tracing exporter 地址、超时和连接池配置。

## 6. 结论与建议

当前实现已经满足“投标技术方案完整输出”主链路设计：能够从智能体配置知识库推导有效 KB 范围，走知识库 grounded 长文档专用路径，服务端确定性检索本地知识，按完整大纲连续生成全部章节，并在完成后持久化消息与文档 artifact。本次日志中的最终状态是 completed/stop，内容长度 17829，artifact 创建成功，说明用户侧可获得一份完整技术方案。

但为了达到“提交甲方”的交付标准，建议补齐以下两项整改：

1. 修复聊天历史索引对长文档的单 chunk embedding 超长问题。优先在 `MessageService.IndexMessageToKB` 或 `CreateKnowledgeFromPassage` 入口按模型输入上限切分，或者对 `full_document` 消息改为摘要索引。
2. 增强长文档章节正文的 Markdown 质量约束。将 H3/H4 标题格式写入章节 Prompt，并在完成前增加轻量格式校验或自动修复，避免 `###3.1标题正文` 这类格式进入最终 artifact。

综合判定：当前实现“功能主流程符合设计要求，正式交付质量与后台索引健壮性仍需完善”。
