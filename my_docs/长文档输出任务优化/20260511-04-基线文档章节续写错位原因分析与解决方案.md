# 基线文档章节续写错位原因分析与解决方案

## 1. 问题结论

本次“在基线文档版本上，对智慧运行章节进行继写，结果内容被放到三、项目总结后面”的核心原因是：系统把一个“指定章节内补充内容”的请求识别并执行成了“基于上一版文档继续从末尾生成”。

当前文档版本链路里，`continue_document` 的语义非常明确：从上一份文档末尾自然继续。它不是“找到某个章节，然后把新增内容插入该章节”。因此，当用户输入类似“继续补充智慧运行章节，详细阐述每个模块的功能”时，前端和后端会把它归入 `continue_document`，后端提示词要求模型从文档末尾继续，artifact 合并函数也只会把本轮新增内容追加到基线快照末尾。基线文档末尾如果已经是“三、项目总结”，新增的“智慧运行”内容就会被拼到“项目总结”后面，文档架构看起来就乱了。

这个问题和“文档完成后仍自动续写循环”有关联，但不是同一个根因。循环问题主要是完成状态和自动续写停止条件；本问题主要是“章节级补充”与“文尾续写”的意图和合并策略混用。

## 2. 现象复盘

用户预期：

1. 选择某个文档版本作为基线。
2. 要求对“智慧运行章节”继续补充、扩写或细化。
3. 新增内容应该进入“智慧运行”章节内部，通常应位于该章节末尾，且在下一个同级章节或“三、项目总结”之前。

实际结果：

1. 系统生成了与“智慧运行”相关的新内容。
2. 但新增内容没有回填到“智慧运行”章节下。
3. 新内容被追加到基线文档末尾，即“三、项目总结”之后。

这说明模型未获得稳定的“章节插入锚点”，后端合并阶段也没有做章节级插入，只做了尾部追加。

## 3. 当前代码链路

### 3.1 前端意图推断把“继续补充”归为 continue

相关文件：`frontend/src/views/chat/index.vue`

`inferDocumentContinuationIntent()` 的规则中，包含以下触发词：

```text
继续生成、接着写、续写、继续补充、继续完善、继续扩写、补齐剩余等
```

因此，“继续补充智慧运行章节”会命中 `continue_document`。后续请求会携带：

```json
{
  "intent_hint": "continue_document",
  "base_artifact_id": "当前基线文档版本 ID",
  "document_output_mode": "delta_only"
}
```

只有当用户话术没有命中 continue/regenerate/revise，且当前显式选择了基线文档时，前端才会兜底把普通请求转成 `revise_document`。也就是说：

1. “继续补充智慧运行章节”会走 `continue_document`。
2. “在智慧运行章节补充每个模块功能”更可能走显式基线下的 `revise_document` 兜底。
3. “修改上一版，在智慧运行章节补充每个模块功能”会明确走 `revise_document`。

### 3.2 后端 continue 提示词要求从文档末尾继续

相关文件：`internal/application/service/chat_document_artifact.go`

`BuildQuotedContext()` 对 `continue_document` 构造的提示词包含以下关键规则：

```text
你正在继续生成同一会话中的上一份文档。

续写规则：
1. 不要从头重写上一份文档。
2. 从上一份文档末尾自然继续。
3. 保持标题层级、术语、编号、表格和 Markdown 风格一致。
...
```

这对“文章没写完，继续剩余章节”是正确的，但对“补充某个中间章节”是不正确的。即使用户 query 里写了“智慧运行章节”，系统仍没有把它提升为结构化锚点，只是作为自然语言目标放进 `<original_user_goal>`。

因此模型有两种常见输出：

1. 按提示词从文档末尾继续，直接在“项目总结”后继续写。
2. 虽然生成了“智慧运行”相关内容，但由于输出仍被当作 delta，后端还是会把它拼到文档末尾。

### 3.3 continue 合并函数只做文尾追加

相关文件：`internal/application/service/chat_document_artifact.go`

`buildChatDocumentSnapshotResult()` 在 `operation == continue` 时调用：

```go
return chatDocumentSnapshotBuildResult{Snapshot: mergeChatDocumentContinuation(base.ContentSnapshot, trimmed)}
```

`mergeChatDocumentContinuation()` 的逻辑是：

```go
func mergeChatDocumentContinuation(base string, delta string) string {
    baseTrimmed := strings.TrimSpace(base)
    deltaTrimmed := strings.TrimSpace(delta)
    if baseTrimmed == "" {
        return deltaTrimmed
    }
    if deltaTrimmed == "" {
        return baseTrimmed
    }
    overlap := longestChatDocumentOverlap(baseTrimmed, deltaTrimmed, 1024)
    if overlap > 0 {
        return baseTrimmed + deltaTrimmed[overlap:]
    }
    return baseTrimmed + "\n\n" + deltaTrimmed
}
```

这个函数没有章节识别、没有目标 heading、没有“插入到下一个同级标题之前”的逻辑。它唯一的结构处理是做尾部 overlap 去重。只要进入 continue 合并，新增内容就只能出现在基线快照末尾。

### 3.4 revise 模式其实已有章节 patch 能力，但本场景没有进入它

同一个服务里，`revise_document` 支持结构化 patch：

```xml
<document_patch>
<append heading="## 智慧运行">
新增内容
</append>
</document_patch>
```

后端会通过 `applyChatDocumentStructuredPatch()` 查找目标 heading，并把内容追加到该章节范围内。若找不到 heading，则会保守追加到文档末尾，并打上 `delta_merge_uncertain` 质量问题。

也就是说，代码里已经有“章节级合并”的雏形，问题是“继续补充某章节”这类请求没有稳定路由到 `revise_document + patch append`，而是被 `continue_document + tail append` 抢先命中。

## 4. 根因拆解

### 4.1 意图层缺少“有锚点的续写”分类

当前意图只有：

1. `continue_document`：从文尾继续。
2. `revise_document`：修改上一版。
3. `regenerate_document`：重新生成。
4. `normal`：普通问答。

但真实用户话术里有一类请求介于 continue 和 revise 之间：

```text
继续补充智慧运行章节
接着完善 2.5 火电设备运维智能体
把智慧运行这一节继续写细一点
对智能安全监控应急中心章节继续扩写
```

它们不是文尾续写，而是“章节锚点内的增量补充”。当前没有 `section_continue` 或 `section_append` 这样的子意图，导致它被 continue 规则误收。

### 4.2 前后端都把“继续补充”优先解释为文尾续写

前端 `inferDocumentContinuationIntent()` 和后端 `DetectIntent()` 都有类似规则：只要命中“继续补充/续写/继续扩写”，就归入 `continue_document`。

这在“自动续写剩余内容”场景是合理的；但当同一句话里同时出现“章节/小节/智慧运行/2.5”等范围词时，优先级应该反过来：

```text
继续 + 具体章节目标 = 章节补充/修改，不是文尾续写
继续 + 剩余内容/从上次中断/当前文档为基准 = 文尾续写
```

当前没有这个分流。

### 4.3 合并层对 continue 没有任何章节锚点保护

即使模型输出了下面这种内容：

```markdown
## 智慧运行

### 监盘能力
...
```

只要本轮 operation 是 `continue`，后端仍然调用 `mergeChatDocumentContinuation()`，最终就是：

```markdown
...三、项目总结

## 智慧运行

### 监盘能力
...
```

也就是说，模型输出内容本身无法纠正合并策略。必须在服务端合并层提供章节级插入能力，不能只依赖提示词。

### 4.4 超长文档场景会进一步放大错位概率

当基线文档超过 `ChatDocumentArtifactInlineContinuationMaxChars` 后，系统会进入 `outline_tail` 模式，只给模型目录和尾部窗口。对于中间章节“智慧运行”，模型可能看不到完整章节正文，只能看到目录和文档尾部。

在这种模式下，如果没有显式抽取目标章节窗口，模型更容易跟随尾部窗口继续写，最终仍落到“项目总结”之后。

## 5. 推荐解决方案

建议按“先止血、再结构化修复”的方式处理。

### 5.1 短期规避方案：让章节补充走 revise patch

在代码修复前，用户侧提示词可以临时改成“修改上一版”而不是“继续补充”：

```text
修改上一版文档：请在“智慧运行”章节末尾补充智慧运行各模块的功能说明，不要改动其他章节，不要把内容追加到项目总结后面。
```

更稳定的提示词是直接要求 patch：

```text
修改上一版文档，只输出 document_patch：
<document_patch>
<append heading="智慧运行">
请补充智慧运行章节下各模块的功能说明，保持原 Markdown 层级。
</append>
</document_patch>
```

这个规避方案能利用现有 `revise_document` 的 patch 合并能力，但仍依赖用户知道正确话术，不应该作为最终产品方案。

### 5.2 前端修复：识别“章节级继续补充”并改送 revise_document

修改文件：`frontend/src/views/chat/index.vue`

新增一个范围目标识别函数，识别“章节、小节、段落、标题、具体编号、具体模块名称”等目标：

```js
const hasSectionContinuationTarget = (query = '') => {
    const text = query.trim();
    if (!text) return false;
    const hasContinueVerb = /(继续|接着|续写|补充|扩写|完善|细化|补齐)/.test(text);
    const hasScopedTarget = /(章节|小节|段落|标题|模块|部分|第[0-9一二三四五六七八九十百零]+[章节部分]|[0-9]+(?:\.[0-9]+)+|智慧运行|智慧安防|数据湖|算力平台|应急中心|AR眼镜)/.test(text);
    const isTailContinuation = /(剩余内容|从上次中断|文档末尾|继续剩余|当前文档为基准|自动续写)/.test(text);
    return hasContinueVerb && hasScopedTarget && !isTailContinuation;
};
```

然后调整发送前的 intent 推断：

```js
let intentHint = inferDocumentContinuationIntent(value);
if (explicitBaseArtifact && hasSectionContinuationTarget(value)) {
    intentHint = 'revise_document';
}
if (explicitBaseArtifact && intentHint === 'normal') {
    intentHint = 'revise_document';
}
```

这样“继续补充智慧运行章节”会进入 `revise_document`，并由后端 patch 机制负责插入到目标章节，而不是进入文尾续写。

### 5.3 后端修复：DetectIntent 也要做同样兜底

前端判断只能改善 Web 入口，不能覆盖 API、IM 集成和自动化调用。后端 `DetectIntent()` 也需要识别“有范围目标的继续补充”。

修改文件：`internal/application/service/chat_document_artifact.go`

建议新增正则：

```go
chatDocumentScopedTargetRE = regexp.MustCompile(`(?i)(章节|小节|段落|标题|模块|部分|第[0-9一二三四五六七八九十百零]+[章节部分]|[0-9]+(?:\.[0-9]+)+|智慧运行|智慧安防|数据湖|算力平台|应急中心|AR眼镜)`)
chatDocumentTailContinueRE = regexp.MustCompile(`(?i)(剩余内容|从上次中断|文档末尾|继续剩余|当前文档为基准|自动续写)`)
```

意图优先级调整为：

```go
case chatDocumentContinueIntentRE.MatchString(trimmedQuery) &&
    chatDocumentScopedTargetRE.MatchString(trimmedQuery) &&
    !chatDocumentTailContinueRE.MatchString(trimmedQuery):
    intent = types.ChatDocumentIntentRevise
case chatDocumentContinueIntentRE.MatchString(trimmedQuery):
    intent = types.ChatDocumentIntentContinue
```

这样“继续补充某章节”会被后端解释为章节修改/补充，而“继续剩余内容输出”仍保持文尾续写。

### 5.4 提示词修复：revise 模式显式鼓励 append patch

当前 revise 提示词已经要求优先输出 `<document_patch>`，但可以针对“补充/扩写/细化某章节”再强化一条：

```text
如果用户要求补充、扩写、细化某个章节或模块，优先输出：
<document_patch>
<append heading="目标章节标题">
新增内容
</append>
</document_patch>
不要输出完整文档，不要把新增内容放到文档末尾。
```

同时在 `<original_user_goal>` 后增加系统提取的目标锚点，例如：

```text
目标合并位置：智慧运行章节
目标合并方式：append_to_section
```

这样模型不需要自己猜“应该放到哪里”。

### 5.5 服务端合并修复：引入章节目标和模糊 heading 匹配

当前 `applyChatDocumentStructuredPatch()` 已支持：

1. `replace heading="..."`
2. `append heading="..."`
3. `insert_after heading="..."`

但 heading 匹配主要依赖精确标题。如果基线里实际标题是：

```markdown
## 二、五大核心功能模块
### 2.2 智慧运行
```

而 patch 写的是：

```xml
<append heading="智慧运行">
```

现有匹配可能找不到目标，最终又走保守文末追加。因此需要增强 `findMarkdownSectionRangeBySelector()`：

1. 支持去掉编号后的标题匹配，例如 `2.2 智慧运行` 与 `智慧运行` 视为匹配。
2. 支持中文编号归一化，例如 `二、智慧运行` 与 `智慧运行`。
3. 支持 selector 是标题关键词时命中唯一 heading。
4. 如果命中多个 heading，不自动合并，返回 uncertain 并提示用户缩小范围。

建议新增归一化函数：

```go
func normalizeHeadingForMatch(title string) string {
    title = strings.TrimSpace(title)
    title = regexp.MustCompile(`^#{1,6}\s*`).ReplaceAllString(title, "")
    title = regexp.MustCompile(`^(?:[0-9]+(?:\.[0-9]+)*|[一二三四五六七八九十]+)[、.．\s-]*`).ReplaceAllString(title, "")
    title = regexp.MustCompile(`\s+`).ReplaceAllString(title, "")
    return strings.ToLower(title)
}
```

匹配策略：

```go
selectorNorm := normalizeHeadingForMatch(selector)
headingNorm := normalizeHeadingForMatch(currentHeading)
if selectorNorm == headingNorm || strings.Contains(headingNorm, selectorNorm) {
    // 记录候选
}
```

只有候选数量为 1 时才自动合并。

### 5.6 合并层兜底：continue 若检测到章节标题，不应无脑追加

即使前后端意图分流漏掉，也可以在 `buildChatDocumentSnapshotResult()` 增加防错兜底：

```go
if operation == types.ChatDocumentOperationContinue {
    if target := inferSectionTargetFromQuery(options.UserQuery); target != "" {
        merged := mergeChatDocumentRevisionDelta(base.ContentSnapshot, buildAppendPatch(target, trimmed))
        return chatDocumentSnapshotBuildResult{
            Snapshot: merged.Snapshot,
            QualityIssues: merged.QualityIssues,
        }
    }
    return chatDocumentSnapshotBuildResult{Snapshot: mergeChatDocumentContinuation(base.ContentSnapshot, trimmed)}
}
```

这层兜底的意义是：只要用户 query 明确提到章节目标，就不要把 delta 直接拼到全文末尾。即使无法精确定位，也应该打 `delta_merge_uncertain`，并通过 `user_hint` 告诉用户“未能定位目标章节”，而不是静默破坏文档结构。

### 5.7 自动续写要继续保持文尾语义

自动续写的默认 prompt 是：

```text
以当前文档为基准，继续剩余内容输出
```

这类请求应该继续走 `continue_document`，因为它确实表示从文尾继续。不要把自动续写误改成 revise。判断关键是是否包含明确章节目标：

1. `继续剩余内容输出`：文尾续写。
2. `继续补充智慧运行章节`：章节补充。
3. `继续扩写 2.5 火电设备运维智能体`：章节补充。

## 6. 推荐落地步骤

### 第一阶段：快速止血

1. 前端增加 `hasSectionContinuationTarget()`。
2. 显式基线存在且 query 命中“继续 + 章节目标”时，强制发送 `revise_document`。
3. 后端 `DetectIntent()` 增加同样规则，覆盖非 Web 入口。
4. revise 提示词增加“章节补充优先 append patch”的明确规则。

这一阶段改动较小，可以快速避免“智慧运行内容落到项目总结后面”。

### 第二阶段：提升合并稳定性

1. 增强 `findMarkdownSectionRangeBySelector()` 的模糊 heading 匹配。
2. 为无法唯一定位的章节合并返回 `delta_merge_uncertain`。
3. 在 `chatDocumentUserHintForIssues()` 中增加更明确提示：目标章节未唯一定位，已暂停/保守处理。
4. 增加 continue 模式的目标章节兜底，防止误路由后静默 append 到文末。

### 第三阶段：结构化章节编辑能力

1. 在 `RegisterChatDocumentArtifactOptions` 中增加 `TargetHeading`、`MergeMode` 等字段。
2. 前端或后端从 query 中抽取目标章节，随请求链路传递。
3. `BuildQuotedContext()` 针对目标章节抽取完整章节窗口，而不是只给全文或尾部窗口。
4. 对超长文档，优先传入 `document_outline + target_section + target_parent + nearby_siblings`，而不是只给 tail。

4. 不产生 `delta_merge_uncertain`。

### 7.4 误路由兜底测试

即使 options 里 `Operation == continue`，只要 `UserQuery` 明确包含“智慧运行章节”，也不应直接调用 `mergeChatDocumentContinuation()` 把内容追加到文档末尾。期望要么成功插入目标章节，要么标记 `delta_merge_uncertain` 并提示人工检查。

## 8. 最终建议

产品语义上应把“继续生成”拆成两类：

1. 文尾续写：适用于文章未完成、继续剩余章节、自动续写。
2. 章节补充：适用于已有完整文档中，对某一章节继续扩写、补充、细化。

当前系统只实现了第一类的稳定链路，第二类虽然有 `revise_document + document_patch` 的基础能力，但入口分流和 heading 匹配还不够。建议优先把“继续 + 具体章节目标”的请求改走 revise patch，再增强章节定位和合并兜底。这样可以从根上避免“智慧运行内容追加到项目总结后面”的结构错乱问题。
