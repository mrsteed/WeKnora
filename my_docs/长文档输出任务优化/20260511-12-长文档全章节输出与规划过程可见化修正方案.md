# 长文档全章节输出与规划过程可见化修正方案

## 1. 背景

本次复测请求仍是：

```text
这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。
```

页面显示：

```text
已检索 6 条本地知识证据，并规划 7 个章节，先生成前 3 个章节。
```

随后正文确实只生成了前 3 个章节，并在页面底部显示：

```text
正在继续生成剩余章节
```

用户当前明确要求有两点：

1. 已经规划出 7 个章节，就应一次性把 7 个章节全部输出，不要继续保留“先生成前 3 个章节”的限制与批次区分。
2. 规划过程中要持续输出可见的思考与进度，不要 3 分钟只显示一句“已检索 6 条本地知识证据，并规划 7 个章节，先生成前 3 个章节”。

## 2. 当前实现结论

当前 P0 修复解决的是“达到章节批次上限后自动续写”的问题，但没有解决用户这次提出的两个新问题：

```text
1. 后端仍然有人为章节批次上限 dedicatedFullDocumentSectionLimitPerRun = 3。
2. 首轮检索与大纲规划阶段仍是同步阻塞式流程，缺少持续可见进度事件。
```

因此，从用户体验看，系统仍然像是在“先截一段，再续写”，而不是完整执行“请输出完整技术方案”的承诺。

## 3. 日志证据

`my_docs/test/11.md` 中可以看到当前生成链路已经进入知识库支撑长文档路径，并规划出 7 个章节：

```text
# 北海电厂二期智慧电厂项目技术方案

## 项目背景与建设目标
## 总体技术架构设计
## 数据湖与基础算力平台
## 智慧运行中枢系统
## 智慧安防管理方案
## 智能安全监控应急中心
## AR眼镜智能作业辅助体系
```

但服务端仍然按批次限制只生成前 3 章：

```text
已检索 6 条本地知识证据，并规划 7 个章节，先生成前 3 个章节。
```

最终后端完成态仍是：

```text
completion_status: partial
finish_reason: section_batch_limit
allow_complete: false
allow_indexing: false
content_len: 5338
```

这证明本次问题不是知识库未命中，也不是模型自然停止，而是业务逻辑主动触发 `section_batch_limit`。

## 4. 代码定位

### 4.1 硬编码章节批次上限

`internal/application/service/session_agent_qa.go` 当前定义：

```go
dedicatedFullDocumentSectionLimitPerRun = 3
```

知识库支撑完整文档生成路径中继续使用该限制：

```go
sectionLimit := dedicatedFullDocumentSectionLimitPerRun
if sectionLimit <= 0 || sectionLimit > len(outline.Sections) {
    sectionLimit = len(outline.Sections)
}
```

后续只遍历：

```go
for index, section := range outline.Sections[:sectionLimit] {
    ...
}
```

当章节总数超过 `sectionLimit` 时，服务端主动设置：

```go
completionStatus = types.MessageCompletionStatusPartial
finishReason = "section_batch_limit"
documentGenerationStatus = types.ChatDocumentGenerationStatusContinuing
```

这就是“规划 7 章但先生成 3 章”的直接原因。

### 4.2 进度事件太粗

首轮知识库长文档路径只在两个位置发 synthetic thought：

```go
正在检索本地知识库并规划文档大纲。
```

以及：

```go
已检索 %d 条本地知识证据，并规划 %d 个章节，先生成前 %d 个章节。
```

中间包含以下耗时步骤：

```text
1. 多查询检索本地知识库。
2. 调用模型生成大纲。
3. 解析大纲。
4. 创建 generation run。
5. 为当前章节再次检索证据。
6. 调用模型生成章节正文。
```

但其中大部分步骤没有对前端持续发进度事件。

### 4.3 大纲生成是非流式 Chat 调用

当前大纲阶段调用：

```go
outlineResponse, err := chatModel.Chat(ctx, outlineMessages, ...)
```

这是一次性返回的同步调用。在模型返回之前，前端无法看到大纲规划的增量过程，只能看到上一条 synthetic thought。

日志中可以看到模型首个可见正文需要较长时间：

```text
First reasoning_content elapsed_ms=698
First delta.Content elapsed_ms=22676
Natural-stop elapsed_ms=59268
```

也就是说，用户至少有几十秒只能等待一条静态进度。在真实长文档任务中，多章节串行生成会进一步放大这种体验问题。

### 4.4 “思考过程”的边界

用户说的“思考过程”应在产品上实现为可见的规划与执行过程，例如：

```text
正在解析用户目标
正在解析智能体知识库范围
正在检索本地知识库：第 1/3 个查询
已命中 6 条证据，来源包括 xxx.docx
正在生成大纲
已规划 7 个章节
正在生成第 1/7 章：项目背景与建设目标
正在检索第 2/7 章证据：总体技术架构设计
正在生成第 2/7 章正文
```

不建议也不应输出模型隐藏推理链。原因是：隐藏推理属于模型内部过程，不稳定、不可审计，也可能包含无用或误导性内容。正确做法是输出“可解释的执行进度”和“结构化规划摘要”。

## 5. 根因归纳

### 5.1 完整文档语义被批次机制覆盖

`document_output_mode=full_document` 的用户语义是“输出完整文档”。但当前服务端为了规避长文档中途重启和超时，把生成拆成首轮 3 章加自动续写。

这对技术上是稳定策略，但对用户来说不符合“请输出完整”的直觉，尤其当已经规划出 7 个章节时，继续显示“先生成前 3 个章节”会被理解为系统没有完成任务。

### 5.2 自动续写是补丁，不是最终体验

前一轮 P0 修复让 `section_batch_limit` 能自动进入下一轮，避免停在 partial。但它仍会产生多个用户消息、多个 assistant 版本、多个 artifact 修订，用户看到的是“分批生成链路”，不是一个完整输出流程。

本次需求要求取消这种用户可感知的分批边界。

### 5.3 规划阶段没有进度通道

当前 `runKnowledgeGroundedFullDocumentGenerationPath` 没有专门的长文档规划进度管理器，也没有 heartbeat。检索和大纲生成期间只依赖少量 `EventAgentThought`，因此只显示一两句话。

## 6. 修正目标

修正后应满足：

1. 对 `full_document` 首轮请求，后端应生成大纲中的全部章节，不再默认限制为前 3 章。
2. `section_batch_limit` 仅作为异常保护或兼容旧配置使用，不应出现在普通完整文档生成路径。
3. 规划阶段持续输出可见进度，最长 8 到 10 秒必须有新的 thought/progress 事件。
4. 每个章节生成前都应显示章节级进度：章节序号、章节标题、检索状态、生成状态。
5. 仍要保留异常保护：模型流中断、用户停止、网关断开、证据不足时不能误标 completed。
6. 完成后最终状态应是：

```text
completion_status=completed
finish_reason=stop
document_generation_status=completed
allow_complete=true
allow_indexing=true
```

## 7. 推荐修正方案

### 7.1 取消完整文档首轮的人为章节上限

新增一个章节范围决策函数，替代直接读取 `dedicatedFullDocumentSectionLimitPerRun`：

```go
func fullDocumentSectionsForRun(outline dedicatedFullDocumentOutline, req *types.QARequest, continuation bool) []string {
    if len(outline.Sections) == 0 {
        return nil
    }
    if req != nil && req.DocumentOutputMode == types.ChatDocumentOutputModeFull && !continuation {
        return append([]string(nil), outline.Sections...)
    }
    limit := dedicatedFullDocumentSectionLimitPerRun
    if limit <= 0 || limit > len(outline.Sections) {
        limit = len(outline.Sections)
    }
    return append([]string(nil), outline.Sections[:limit]...)
}
```

首轮 `runKnowledgeGroundedFullDocumentGenerationPath` 应使用全部章节：

```go
sectionsToGenerate := fullDocumentSectionsForRun(outline, req, false)
```

并将提示从：

```text
已检索 6 条本地知识证据，并规划 7 个章节，先生成前 3 个章节。
```

改为：

```text
已检索 6 条本地知识证据，并规划 7 个章节，将按大纲连续生成全部章节。
```

对应循环从：

```go
for index, section := range outline.Sections[:sectionLimit]
```

改为：

```go
for index, section := range sectionsToGenerate
```

完成态判断从“章节数超过 sectionLimit 就 partial”改为“只有异常才 partial”：

```go
if completionStatus == types.MessageCompletionStatusCompleted {
    documentGenerationStatus = types.ChatDocumentGenerationStatusCompleted
    finishReason = "stop"
}
```

### 7.2 保留批次能力但降级为保护开关

不建议彻底删除 `dedicatedFullDocumentSectionLimitPerRun`，因为它对以下场景仍有价值：

```text
1. 自动续写兼容旧 generation run。
2. 超长大纲或极小模型上下文的保护模式。
3. 后台任务化生成尚未完成前的兜底策略。
```

建议新增显式配置：

```yaml
conversation:
  full_document:
    section_batch_limit: 0
    enable_section_batching: false
```

语义：

```text
section_batch_limit=0 或 enable_section_batching=false 表示首轮完整输出全部章节。
只有管理员显式开启 enable_section_batching=true 时才按批次生成。
```

当前用户场景应默认关闭分批。

### 7.3 增加长文档规划进度管理器

新增轻量 helper：

```go
type fullDocumentProgressReporter struct {
    ctx       context.Context
    req       *types.QARequest
    eventBus  *event.EventBus
    eventID   string
    iteration int
}
```

提供：

```go
func (r *fullDocumentProgressReporter) Update(content string)
func (r *fullDocumentProgressReporter) Heartbeat(content string)
func (r *fullDocumentProgressReporter) Done()
```

所有长耗时阶段都使用同一个进度 reporter，避免不同 thought event 闪烁或互相覆盖。

### 7.4 检索阶段输出结构化进度

当前 `retrieveKnowledgeGroundedFullDocumentEvidence` 是黑盒同步函数。建议扩展为可选进度回调：

```go
type evidenceProgressCallback func(stage string, current int, total int, detail string)

func retrieveKnowledgeGroundedFullDocumentEvidence(..., progress evidenceProgressCallback) (...)
```

在以下节点回调：

```text
开始检索：共 3 个查询，1 个知识库。
正在检索第 1/3 个查询：用户原始目标。
正在检索第 2/3 个查询：项目背景、总体架构、核心能力。
正在检索第 3/3 个查询：业务价值、实施边界、运维保障。
检索完成：命中 6 条证据，来源 1 个文档。
```

前端显示示例：

```text
正在检索本地知识库（2/3）：项目背景、总体架构、核心能力
```

### 7.5 大纲生成改为可见规划过程

有两种方案。

#### 方案 A：保持 Chat，但加 heartbeat

保留：

```go
chatModel.Chat(ctx, outlineMessages, ...)
```

同时在 goroutine 中等待结果，并每 8 秒发一次 progress：

```text
正在生成文档大纲，已等待 8 秒。
正在生成文档大纲，已等待 16 秒。
```

优点：改动小。缺点：用户仍看不到真实大纲增量。

#### 方案 B：大纲生成改为 ChatStream

将大纲规划从 `Chat` 改成 `ChatStream`：

```go
outlineStream, err := chatModel.ChatStream(ctx, outlineMessages, ...)
```

边接收边累计 outline 文本，并向 thought 输出可见规划摘要：

```text
正在生成大纲：已识别标题“北海电厂二期智慧电厂项目技术方案”
正在生成大纲：已识别 3 个章节
正在生成大纲：已识别 7 个章节
```

注意：不要把大纲 token 作为最终 answer 输出，仍应作为 thought/progress 展示；最终 answer 只输出正式文档正文。

推荐采用方案 B。

### 7.6 章节生成过程输出章节级进度

在每个章节前增加 thought：

```go
progress.Update(fmt.Sprintf("正在生成第 %d/%d 章：%s", index+1, len(sectionsToGenerate), section))
```

章节检索前：

```text
正在检索第 3/7 章“数据湖与基础算力平台”的本地证据。
```

章节生成前：

```text
已检索到 4 条证据，正在生成第 3/7 章“数据湖与基础算力平台”。
```

章节完成后：

```text
第 3/7 章“数据湖与基础算力平台”已完成，继续生成下一章。
```

这样即使某一章模型生成耗时 60 秒，用户也能知道当前卡在哪一步。

### 7.7 章节内模型等待也要 heartbeat

当前章节正文通过 `ChatStream` 生成，只有模型输出正文后前端才能看到内容。如果模型长时间只有 reasoning 或完全未吐正文，用户仍会等待。

建议在每个章节 stream 消费循环中增加首次可见正文超时提示与 heartbeat：

```text
第 4/7 章“智慧运行中枢系统”正在生成，等待模型输出正文。
第 4/7 章“智慧运行中枢系统”仍在生成中，已等待 30 秒。
```

可复用现有文档编辑路径中的 heartbeat 机制：

```go
dedicatedDocumentEditProgressHeartbeatInterval = 8 * time.Second
```

但应抽成通用的 full document stream heartbeat，而不是只服务 document edit。

## 8. 推荐落地步骤

### P0

1. 首轮 `full_document` 取消 `dedicatedFullDocumentSectionLimitPerRun` 限制，默认生成全部大纲章节。
2. 修改用户可见提示：删除“先生成前 N 个章节”，改为“将按大纲连续生成全部章节”。
3. 删除首轮正常完成时的 `section_batch_limit` 分支，只有异常才 partial。
4. 增加检索、大纲、章节生成的 synthetic thought 进度事件。
5. 至少每 8 到 10 秒输出一次可见进度。

### P1

1. 将大纲生成从 `Chat` 改成 `ChatStream`，显示大纲规划增量。
2. `retrieveKnowledgeGroundedFullDocumentEvidence` 增加 progress callback。
3. 每个章节输出“检索证据 -> 生成正文 -> 完成章节”的结构化进度。
4. 前端对 synthetic thought 增加更友好的展示样式，例如“规划中 / 检索中 / 生成中”。

### P2

1. 对超长完整文档引入后端任务化生成，避免单个 HTTP/SSE 请求过长。
2. 支持用户选择“连续输出全文”或“分批可审阅输出”。
3. 增加章节级失败恢复：某一章失败时可从该章重试，而不是重跑全文。

## 9. 验收标准

### 9.1 完整输出验收

同样请求规划出 7 个章节时，首轮应连续输出：

```text
1. 项目背景与建设目标
2. 总体技术架构设计
3. 数据湖与基础算力平台
4. 智慧运行中枢系统
5. 智慧安防管理方案
6. 智能安全监控应急中心
7. AR眼镜智能作业辅助体系
```

最终日志应为：

```text
completion_status=completed
finish_reason=stop
document_generation_status=completed
allow_complete=true
allow_indexing=true
```

不应再出现：

```text
finish_reason=section_batch_limit
先生成前 3 个章节
```

### 9.2 进度可见性验收

从请求开始到第一段正文输出前，前端至少应看到以下进度序列中的多条：

```text
正在解析文档目标与知识库范围。
正在检索本地知识库（1/3）。
正在检索本地知识库（2/3）。
已命中 6 条证据，正在规划完整大纲。
已规划 7 个章节，将连续生成全部章节。
正在生成第 1/7 章：项目背景与建设目标。
```

如果任一步骤耗时超过 8 到 10 秒，应继续输出 heartbeat：

```text
正在规划完整大纲，已等待 16 秒。
```

### 9.3 异常保护验收

如果模型流中断，应仍然返回：

```text
completion_status=partial
finish_reason=section_generation_truncated
auto_continue_next=false
```

不能为了“全部输出”而把半截文档误标 completed。

## 10. 测试建议

### 10.1 后端单测

1. `runKnowledgeGroundedFullDocumentGenerationPath` 大纲 7 章：
   - `streamCalls=7`。
   - 输出包含第 1 到第 7 章标题。
   - `completion_status=completed`。
   - 不出现 `section_batch_limit`。

2. 检索进度测试：
   - mock evidence search 多查询。
   - 断言 `EventAgentThought` 至少包含“正在检索”“已命中证据”。

3. 大纲规划进度测试：
   - mock 慢速 outline model。
   - 断言 heartbeat thought 被持续发送。

4. 章节级进度测试：
   - 7 个章节逐章生成。
   - 断言 thought 包含“第 1/7 章”“第 7/7 章”。

5. 异常保护测试：
   - 第 4 章 stream 关闭但未 Done。
   - 断言 `partial + section_generation_truncated + auto_continue_next=false`。

### 10.2 前端测试

1. 收到 `agent.thought` replace 更新时，思考框内容持续刷新。
2. 收到第 N 章生成进度时，UI 不显示“长时间无输出”。
3. 完整文档 completed 后，不显示“正在继续生成剩余章节”。
4. 不再依赖自动续写用户消息来补齐普通 7 章文档。

## 11. 风险与边界

取消批次限制后，单次请求会更长，需要注意：

```text
1. SSE 连接可能持续更久。
2. 章节越多，服务端 goroutine 占用越久。
3. 浏览器页面刷新会中断实时观看，但后端仍需保证消息和 artifact 状态一致。
4. 如果最终文档非常长，前端渲染和 artifact inline 传输可能变慢。
```

因此，建议同步保留以下保护：

```text
1. 每章单独模型调用，避免一次模型调用超长。
2. 每章完成后立即流式输出并累积 artifact 内容。
3. 仅当所有章节完成后才 allow_complete=true。
4. 若单次总时长超过系统阈值，可降级为后台任务，但不要在普通 7 章方案里默认分批。
```

## 12. 最终建议

当前 P0 自动续写方案只能让系统“继续补完”，但用户要的是“完整文档一次性连续输出”。因此下一步应把修正重点从“前端自动续写”转为“后端 full_document 首轮连续生成全部章节”。

同时，规划阶段不要暴露模型隐藏推理链，而应持续输出可见、可审计的执行过程：检索范围、证据数量、大纲章节、当前章节、等待 heartbeat。这样既满足用户“看到思考过程”的体验诉求，又不会把不可控的模型内部推理暴露给最终用户。
