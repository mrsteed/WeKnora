# 智能推理长文翻译 P1 协议与状态机修复详细设计

## 1. 阶段目标

P1 的目标是把 P0 的止损逻辑升级为稳定协议和稳定状态机，让后端、前端、消息持久化和事件流都共享同一套完成语义。

P1 的交付重点不是任务流重构，而是把现有在线问答链路变成“语义正确、协议清晰、前后端一致”的系统。

## 2. 设计范围

### 2.1 纳入范围

1. internal/event/event_data.go
2. internal/agent/think.go
3. internal/agent/observe.go
4. internal/agent/finalize.go
5. internal/handler/session/agent_stream_handler.go
6. internal/handler/session/qa.go
7. internal/types/message.go
8. 对应 messages 表迁移
9. frontend/src/views/chat/index.vue
10. frontend/src/views/chat/components/AgentStreamDisplay.vue

### 2.2 不纳入范围

1. 长文翻译异步任务服务。
2. markdown 文件产物存储系统。

## 3. 设计目标拆解

### 3.1 后端目标

1. 完成状态持久化。
2. answer、thought、complete 三种事件职责固定。
3. length 截断支持续写与 partial 结束。

### 3.2 前端目标

1. 消息完成态不再只依赖 is_completed。
2. partial、failed、cancelled 可视化。
3. 历史消息回放与流式增量行为一致。

## 4. 数据模型设计

## 4.1 messages 表增强

### 4.1.1 新增字段

建议在 messages 表新增：

1. completion_status varchar(32)
2. finish_reason varchar(32)
3. failure_reason text

### 4.1.2 字段定义

1. completion_status：pending、completed、partial、failed、cancelled。
2. finish_reason：stop、length、tool_calls、content_filter、cancelled、max_iterations。
3. failure_reason：错误描述或系统兜底信息。

### 4.1.3 兼容策略

1. 旧消息默认视为 completed 或根据 is_completed 回填。
2. IsCompleted 暂时保留，作为兼容字段。
3. 新逻辑以 completion_status 为准。

## 4.2 后端状态对象增强

### 4.2.1 AgentState 增强

建议在 AgentState 中新增：

1. CompletionStatus
2. FinishReason
3. AllowIndexing
4. AllowComplete
5. FailureReason
6. PartialAnswer

### 4.2.2 设计目的

1. 让状态不是只在事件里短暂存在，而是成为 Agent 生命周期内的正式数据。
2. 后续 P2 可以复用这些字段，不需要重做语义。

## 5. 协议设计

## 5.1 answer 事件职责

### 5.1.1 唯一职责

answer 事件只负责正文增量输出。

### 5.1.2 明确规则

1. 所有 ResponseTypeAnswer 都走 answer 事件。
2. source=final_answer_tool 只是 metadata，不再决定是否走 answer。
3. answer 事件不负责宣告 completed 语义，只负责正文块。

## 5.2 thought 事件职责

### 5.2.1 唯一职责

thought 事件只负责思考内容和推理过程展示。

### 5.2.2 明确规则

1. 只有 ResponseTypeThinking 才能发 thought 事件。
2. 普通 answer chunk 不允许回落到 thought。
3. thought 流结束不影响 message completion_status。

## 5.3 complete 事件职责

### 5.3.1 唯一职责

complete 事件只负责生命周期结束通知和总结元数据。

### 5.3.2 明确规则

1. complete 不补正文。
2. complete 不从 data.FinalAnswer 反写 assistantMessage.Content。
3. complete 只携带 total_steps、total_duration_ms、completion_status、finish_reason、allow_indexing 等元数据。

这条规则非常关键，因为它直接切断了“complete 阶段再补正文”的隐式副作用。

## 6. 后端实现设计

## 6.1 think.go 分流修复

### 6.1.1 修改目标

把当前“非 final_answer_tool 的普通 answer chunk 落到 thought”的逻辑修掉。

### 6.1.2 新逻辑

1. if chunk.ResponseType == ResponseTypeAnswer，直接发 EventAgentFinalAnswer。
2. if chunk.ResponseType == ResponseTypeThinking，发 EventAgentThought。
3. 默认分支不再对 chunk.Content 做 thought fallback。

### 6.1.3 额外 metadata

建议 answer 事件附带：

1. source
2. event_id
3. finish_reason

## 6.2 observe.go 续写策略

### 6.2.1 修改目标

在 P0 基础上，把 length 识别升级为真实续写机制。

### 6.2.2 续写逻辑

当满足：

1. finish_reason=length
2. len(toolCalls)=0
3. 当前内容非空

则：

1. 把本轮正文追加进 partial 缓冲区。
2. 向 messages 追加 continuation user prompt。
3. continuation prompt 固定为“从中断位置继续，不要重复前文，保持相同格式”。
4. 记录 continuation round 次数。

### 6.2.3 续写上限

建议新增：maxLengthContinuationRounds=3。

超过阈值后：

1. completion_status=partial
2. finish_reason=length
3. allow_indexing=false
4. allow_complete=false

## 6.3 finalize.go 收尾策略

### 6.3.1 修改目标

max iterations 和异常收尾都不能再默认 completed。

### 6.3.2 新规则

1. max iterations 后若 answer 完整性无法证明，则默认 partial。
2. content_filter 默认 failed。
3. 用户 stop 默认 cancelled。

## 6.4 agent_stream_handler.go 修复

### 6.4.1 修改目标

把正文累积职责全部集中到 answer 事件处理器里。

### 6.4.2 新行为

1. handleFinalAnswer 负责累积 h.finalAnswer。
2. handleComplete 不再执行 assistantMessage.Content += data.FinalAnswer。
3. handleComplete 不再用 fallback 补正文，除非明确没有任何 answer 事件且 completion_status=completed。

### 6.4.3 消息更新

在 handleComplete 中写入：

1. assistantMessage.CompletionStatus
2. assistantMessage.FinishReason
3. assistantMessage.IsCompleted = completion_status == completed

## 7. 前端实现设计

## 7.1 chat/index.vue 状态机改造

### 7.1.1 新消息字段

前端消息对象新增：

1. completion_status
2. finish_reason
3. is_partial
4. failure_reason

### 7.1.2 answer 事件处理

新规则：

1. answer.content 只负责追加正文。
2. answer.done 不直接等于 is_completed=true。
3. 只有 complete 事件到达并且 completion_status=completed，才真正置消息 completed。

## 7.2 AgentStreamDisplay 完成判断

### 7.2.1 当前问题

当前 isConversationDone 只要发现 answer.done 或 complete，就视为完成，语义太粗。

### 7.2.2 新规则

1. complete + completion_status=completed：显示已完成。
2. complete + completion_status=partial：显示部分完成。
3. complete + completion_status=failed：显示失败。
4. stop 事件：显示已停止。

### 7.2.3 展示文案建议

1. completed：已完成。
2. partial：输出被截断，结果不完整。
3. failed：生成失败，请重试。
4. cancelled：已停止生成。

## 8. 测试设计

## 8.1 后端测试

建议增加：

1. think.go 普通 answer 分流测试。
2. observe.go 续写流程测试。
3. finalize.go max iterations partial 收尾测试。
4. agent_stream_handler.go 不再在 complete 阶段补正文测试。

## 8.2 前端测试

建议增加：

1. answer.done 但 completion_status=partial 时消息不置 completed。
2. complete.partial 展示部分完成文案。
3. complete.completed 展示已完成。
4. 历史回放消息带 completion_status 时，展示与流式一致。

## 9. 验收标准

P1 完成后，需要满足以下结果：

1. answer、thought、complete 三种事件职责清晰且无正文重复。
2. partial、failed、cancelled 能在前后端统一表达。
3. 长文截断时用户看到的是部分完成，而不是已完成。
4. 普通问答和 final_answer 工具流式输出不回归。

## 10. 本阶段结论

P1 完成后，系统虽然仍未具备真正的长文翻译文件化产出能力，但已经具备了正确的在线问答语义基础。P2 可以在这个基础上安全演进，而不必再背负旧协议的隐式副作用。
