# 数据库智能体跳过长文档路由修正方案

## 1. 背景

当前数据库类型智能体在会话进入 AgentQA 之前，仍会执行统一的长文档路由判定。虽然多数情况下最终不会真正切入长文档生成，但这一层判定会额外触发一次路由模型调用，并在日志中产生“输出/生成/方案”等长文档相关语义判断，干扰数据库查询链路的定位和观测。

本次问题的目标是：

1. 数据库类型智能体不再参与长文档路由判定。
2. 请求直接保留在 agent_qa 路径，进入后续 SQL 查询工具链。
3. 保持非数据库智能体原有的长文档路由行为不变。

## 2. 现状审查

### 2.1 路由判定触发位置

在 internal/handler/session/qa.go 中，parseQARequest 在构造好 qaRequestContext 后，会无条件调用 detectChatRouteDecision。

在 internal/handler/session/chat_route.go 中，detectChatRouteDecision 会：

1. 构造 ChatRouteInput。
2. 解析路由模型 ID。
3. 调用 chatRouteService.Decide。
4. 再根据返回结果决定是否应用文档路由。

这意味着只要进入该请求路径，即使当前智能体本质上是数据库查询型智能体，也会先经历一次长文档路由模型判定。

### 2.2 当前实现缺少数据库智能体短路条件

审查 internal/handler/session/chat_route.go 与 internal/types/chat_route.go 可以确认：

1. detectChatRouteDecision 在调用 ChatRouteService 前，没有判断当前智能体是否为数据库分析型智能体。
2. buildChatRouteInput 也没有向 ChatRouteService 传入“数据库智能体应跳过路由”的显式信号。
3. 因此数据库智能体会被当作普通 agent_qa 请求参与路由模型判定。

### 2.3 数据库智能体的识别线索是存在的

代码中已经存在两类可用识别信息：

1. 显式预设类型：internal/types/custom_agent.go 中的 AgentTypeDatabaseAnalysis。
2. 数据库工具白名单：external_database_schema、external_database_search_tables、external_database_query。

其中第二类对历史数据和自定义智能体尤其重要，因为线上智能体未必都正确回填了 agent_type，但 AllowedTools 已经能表达其数据库型定位。

## 3. 根因分析

根因不在 MySQL 连接器，也不在结构化查询服务本身，而在会话入口的统一路由层：

1. 路由判定发生得过早。
2. 路由判定没有结合智能体类型做豁免。
3. 数据库智能体虽然最终会加载 external_database_query 工具，但在到达工具链之前先走了一次与业务无关的长文档分类。

因此，真正的修复点应放在 detectChatRouteDecision 入口，而不是去改 SQL 查询执行器或 MySQL 连接器。

## 4. 修正方案

### 4.1 修正原则

1. 尽量在最早入口短路，避免无意义的模型路由调用。
2. 兼容显式 database-analysis 预设与历史数据库工具型智能体。
3. 不扩大影响面，不改动普通问答、RAG、Wiki、长文档编辑等现有路径。

### 4.2 具体实现

本次在 internal/handler/session/chat_route.go 中增加了数据库智能体短路逻辑：

1. 当 customAgent 为 smart-reasoning 且 agent_type 为 database-analysis 时，直接跳过 ChatRouteService。
2. 当 customAgent 未设置 agent_type，但 AllowedTools 仅由以下工具集合组成时，也视为数据库型智能体并跳过路由：
   - thinking
   - todo_write
   - final_answer
   - external_database_schema
   - external_database_search_tables
   - external_database_query
3. 被短路的请求会直接写入一个 agent_qa 类型的影子路由结果，用于保留链路观测信息：
   - kind = agent_qa
   - use_long_document = false
   - need_artifact = false
   - reason = database_agent_type_bypass 或 database_tool_only_agent_bypass

这样既能避免长文档误判链路，又不会影响后续 AgentQA 与 SQL 工具执行流程。

## 5. 已落地修改

### 5.1 代码修改

已修改文件：

1. internal/handler/session/chat_route.go
2. internal/handler/session/chat_route_test.go

主要改动包括：

1. 在 detectChatRouteDecision 开头新增数据库智能体旁路判断。
2. 新增数据库工具集识别函数。
3. 新增旁路后的 agent_qa 影子路由结果构造函数。
4. 补充三类单元测试：
   - 显式 database-analysis 智能体跳过路由
   - 仅数据库工具的历史智能体跳过路由
   - 混合工具智能体不跳过，继续走原路由逻辑

### 5.2 未改动部分

以下模块未做行为改动：

1. internal/application/service/chat_route_service.go
2. internal/application/service/structured_query.go
3. internal/databaseconnector/mysql/connector.go

原因是本问题的根因不在 SQL 执行链路，而在会话入口对数据库智能体缺少路由豁免。

## 6. 验证结果

已执行窄范围回归测试：

go test -v ./internal/handler/session -run 'TestDetectChatRouteDecision|TestApplyDocumentRouteDecision|chat_route'

结果：全部通过。

本次验证覆盖了：

1. 原有文档路由相关测试仍通过。
2. 数据库智能体会跳过 ChatRouteService 调用。
3. 混合工具智能体不会被误伤。

## 7. 预期收益

修正后，数据库类型智能体将具备以下收益：

1. 不再进行与 SQL 查询无关的长文档模型路由调用。
2. 降低一次额外模型调用的延迟与成本。
3. 日志链路更加清晰，数据库请求不会再夹杂长文档路由噪音。
4. 对历史数据库智能体和新建 database-analysis 智能体同时生效。

## 8. 后续建议

建议后续补两项增强，但不属于本次最小修复范围：

1. 在 ChatRouteInput 中显式加入 agent_type 或 route_bypass_reason 字段，便于统一观测。
2. 在 Agent 创建或保存时，为数据库智能体强制写入 agent_type，减少后续只能依赖工具白名单兜底的情况。
