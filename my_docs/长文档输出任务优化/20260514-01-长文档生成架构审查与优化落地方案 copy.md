# 长文档生成架构审查与优化落地方案

## 1. 审查范围

本次审查覆盖提交 `7ef8da566cc53a05087eb6e7e9f94fb35405a7d2`，以及当前工作区尚未提交的长文档相关改动。

需要特别说明的是，当前工作区没有 `git diff --cached` 输出，`git status` 显示为未暂存工作区改动。因此本文按“目标提交 + 当前未提交工作区代码”进行审查，而不是按暂存区代码审查。

本次功能目标是：支持一次性生成长文档，包括全文翻译类文档；对生成任务做规划并按批次执行；基于已有文档 artifact 支持续写、修改和自动续写。从功能角度看，当前实现已经覆盖了主要链路。

## 2. 总体评价

当前实现的方向是正确的，尤其是从早期独立的 `long_document_*` 任务表，转向以 `chat_document_artifacts` 为用户可见文档版本、以 `chat_document_generation_runs` 为后台生成运行态的设计。这比单纯把长文档作为一次普通聊天回答更合理，也为版本链、续写、重试、质量状态和最终文档下载提供了统一锚点。

从先进性看，当前方案属于“Artifact First + 运行态补充”的阶段性架构，已经比传统的长文本流式拼接更可靠，但还没有完全达到成熟的“后端任务编排 + 可恢复分段执行 + 事件溯源”的长任务架构。主要原因是：自动续写仍大量依赖前端状态机，后端 `generation_run` 还没有成为唯一权威调度状态；批次输出没有独立持久化；迁移兼容性和 Lite SQLite 路径存在明显缺口；长文档核心逻辑集中在少数大文件中，后续扩展会比较吃力。

结论：可以作为功能验收版本的基础，但建议在正式推广前补齐迁移、任务状态权威和模块边界，否则后续会在长任务恢复、线上升级、问题定位和多端一致性上持续付出较高维护成本。

## 3. 已经做得较好的设计

1. Artifact 作为最终文档权威载体。

   `internal/application/service/chat_document_artifact.go` 将助手消息注册为文档版本，保存完整快照、版本号、父版本、操作类型、生成状态、质量问题和证据引用。前端通过 artifact 获取最终文档，而不是完全信任流式 token，这能显著降低长文档流式错位、截断后状态不一致的问题。

2. 生成状态从普通聊天完成状态中拆出。

   新增 `document_generation_status`，用 `continuing`、`completed`、`needs_review`、`blocked` 表示长文档自身状态，避免只用 `completion_status` 表达“本轮响应完成”和“整篇文档完成”这两个不同语义。这一方向合理。

3. 支持基线续写与结构化 patch。

   当前实现通过 `BuildQuotedContext`、`document_patch`、章节选择、双锚点上下文等机制，让模型可以基于上一版文档做增量修改，而不是每次重写全文。这对大文档续写很关键。

4. 翻译路径已经注意到 chunk overlap 问题。

   `session_document_translation_helpers.go` 使用 `StartAt/EndAt` 去重恢复连续文本，而不是直接把检索分块用空行拼起来。这符合长文档翻译对排版连续性的要求。

5. 路由策略相对保守。

   `chat_route_service.go` 只在用户明确要求完整方案、完整报告、完整译文等情况下路由到长文档，并对普通问答保持保守。这能减少“普通问答误入长文档模式”的回归。

6. 单测覆盖明显增加。

   长文档 Artifact、路由、翻译、SSE complete 元数据、续写中断恢复等路径都有专门测试。聚焦测试通过：

   - `go test ./internal/application/service -run 'Test(ChatDocumentArtifact|ShouldUseLongDocumentTranslation|RunLongDocumentTranslation|BuildChatDocumentTerminalReplayExtra|BindKnowledgeGroundedGenerationRunArtifact|ChatRoute)'`
   - `go test ./internal/handler/session -run 'Test(HandleComplete|GetChatDocumentArtifact|ListChatDocumentArtifactRevisions|DetectChatRouteDecision|ApplyDocumentRoute|.*Document.*|.*AutoContinue.*)'`
   - `go test ./internal/agent ./internal/types`

## 4. 关键问题与风险

### 4.1 P0: SQLite/Lite 迁移缺少新运行态表

Postgres versioned 迁移里新增了：

- `migrations/versioned/000050_chat_document_generation_runs.up.sql`
- `migrations/versioned/000051_chat_document_evidence_refs.up.sql`
- `migrations/versioned/000052_chat_document_generation_run_feedback.up.sql`

但 SQLite 迁移目录目前只看到 `chat_document_artifacts` 的增量列、翻译元数据列和删除旧表迁移，没有对应创建 `chat_document_generation_runs`、`chat_document_evidence_refs` 的 SQLite 迁移。`migrations/sqlite/000000_init.up.sql` 中也未包含这些表。

风险是 WeKnora Lite 或 SQLite 环境下，后端已经注入 `ChatDocumentGenerationRunRepository` 和 `ChatDocumentEvidenceRefRepository`，但执行全文翻译、知识库长文档生成或证据引用持久化时会出现 `no such table: chat_document_generation_runs` 或 `no such table: chat_document_evidence_refs`。

建议立即补齐 SQLite 迁移，并同步更新 Lite 初始化脚本或 consolidated schema。新增迁移应包括：

- 创建 `chat_document_generation_runs`，字段与 Go model 对齐，包括 `budget_json`、`runtime_feedback_json`。
- 创建 `chat_document_evidence_refs`。
- 补齐索引，至少包括 tenant/session、tenant/status、root_message_id、root_artifact_id、tenant/artifact、run_id。
- 增加迁移测试，分别验证 Postgres versioned 和 SQLite migration 集合都能创建当前 model 需要的表与列。

### 4.2 P0: 旧 long_document 表被直接 drop，缺少数据迁移或保留策略

`migrations/versioned/000054_drop_long_document_tasks.up.sql` 和 `migrations/sqlite/000007_drop_long_document_tasks.up.sql` 直接删除：

- `long_document_tasks`
- `long_document_task_batches`
- `long_document_artifacts`

down 迁移只能重建空表，无法恢复线上数据。如果已有用户使用过旧长文档任务，升级会直接丢失任务状态、批次输出和文件 artifact 记录。

建议改为两阶段迁移：

1. 第一阶段保留旧表，新增新表和兼容读取逻辑。
2. 提供一次性 backfill，将旧 task 映射为 `chat_document_generation_runs`，将旧 artifact 映射为 `chat_document_artifacts` 或迁移到统一文件 artifact 存储。
3. 确认一个版本周期后再 drop 旧表。
4. 如果确实确认旧表从未在生产使用，也应在迁移说明中写明，并将 drop 迁移与数据风险确认绑定。

### 4.3 P1: 后端 generation_run 尚未成为长任务调度权威

当前 `chat_document_generation_runs` 已经记录 outline、预算、已完成章节、状态和轮次，但实际自动续写调度主要仍在前端 `frontend/src/views/chat/index.vue` 中完成，例如 `autoContinueState`、`maybeContinueDocumentAutomatically`、`sendAutoContinueMessage`、增长保护和轮次控制。

这会带来几个风险：

- 用户关闭页面、刷新页面、网络短暂断开后，自动续写流程可能停止或重复。
- 多端打开同一会话时，前端状态机之间无法互斥。
- 后端无法独立判断一个 run 是否应该继续、暂停、取消或恢复。
- 自动续写最大轮次、低增长检测、完成判断分散在前端和后端，容易产生状态漂移。

更合理的方向是把 `generation_run` 升级为后端权威状态机，由后端负责批次调度和断点恢复，前端只负责展示状态、发送用户控制指令和订阅事件。

### 4.4 P1: 长文档核心服务过于集中，模块边界不清晰

`internal/application/service/chat_document_artifact.go` 同时承担：

- 意图识别正则。
- 基线文档上下文构造。
- 目标章节解析。
- 结构化 patch 解析和合并。
- 续写重复检测。
- artifact 注册与质量状态判断。
- evidence refs 绑定。
- 版本链列表。

这使得文件职责过宽，后续维护会越来越困难。类似的正则还出现在 handler 和前端，导致同一业务语义被多处复制。

建议拆成独立组件：

- `DocumentIntentClassifier`：只做意图、目标章节、merge mode 的服务端判定。
- `DocumentContextBuilder`：只负责构造续写、修改、双锚点上下文。
- `DocumentPatchMerger`：只负责 patch/章节替换/追加合并。
- `DocumentQualityPolicy`：只负责质量问题、状态和用户提示。
- `DocumentArtifactManager`：只负责 artifact 注册、版本链和证据绑定。

拆分后，前端不再复制核心正则，只接收服务端返回的 intent、target heading、can_continue、user_hint 等结构化字段。

### 4.5 P1: 状态模型分散，缺少统一状态机

当前长文档相关状态至少包括：

- message 的 `completion_status`、`finish_reason`、`failure_reason`。
- artifact 的 `status`、`document_generation_status`、`quality_issues`。
- generation run 的 `status`、`auto_continue_round`、`completed_sections_json`。
- 前端的 `autoContinueState.enabled/stoppedReason/round/lastArtifactId/lowGrowthRounds`。

这些状态各自有转换逻辑，部分转换位于 handler，部分位于 service，部分位于前端。现在单测覆盖了部分组合，但随着场景增加，容易出现“消息 completed、artifact needs_review、run continuing、前端继续续写”的冲突。

建议定义一个统一状态机，例如：

- Run 状态：`planning`、`writing`、`waiting_continue`、`completed`、`needs_review`、`blocked`、`failed`、`cancelled`。
- Round 状态：`pending`、`running`、`succeeded`、`failed`、`skipped`。
- Artifact 状态只表达用户可见版本可用性：`available`、`partial`、`failed`。

所有状态转换必须从一个 service 入口执行，并携带 reason code。前端只根据后端返回的 `run.status` 和 `next_action` 展示，不再自行推导是否续写。

### 4.6 P1: 批次执行仍是请求内同步流程，缺少真正后台任务化

全文翻译 `runLongDocumentTranslationPath` 和长文档章节生成路径目前都在一次请求流程里循环调用模型。虽然会在每个批次后 emit 进度和更新 run，但它仍然依赖当前 HTTP/SSE 请求生命周期。

风险包括：

- 模型单批次耗时过长时，请求上下文或代理层超时仍会中断整个流程。
- 部分批次结果先写入 message/artifact，run 更新和 artifact 注册之间没有事务边界。
- 第 N 批成功但 artifact 注册失败时，run 可能记录已完成，用户却拿不到对应文档快照。
- 无法自然支持后台继续生成、暂停、取消和恢复。

建议将生成任务改为 backend job：

1. 用户请求只创建 run、规划 outline/segments，并返回 run id。
2. 后端通过 asynq 或 Lite 的 SyncTaskExecutor 执行每个 section/segment。
3. 每个批次独立持久化 input snapshot、output delta、status、token budget、model、reason。
4. SSE 只订阅 run event，断线后可以 replay。
5. artifact 由 run 聚合器在批次提交后更新，或按 checkpoint 生成版本。

### 4.7 P1: 翻译路径缺少批次级质量门、超时和重试策略

全文翻译路径已经解决了 chunk overlap，但仍有几个工程缺口：

- 每个 batch 使用 `chatModel.Chat`，没有像 Markdown repair 一样套 `withDocumentGenerationCallTimeout`。
- 初始翻译路径没有使用动态预算，也没有根据模型上下文调整 batch size。
- 输出只做 `sanitizeGeneratedMarkdown`，没有结构完整性校验、页眉页脚残留率检测、重复段落检测。
- 没有批次级重试和降级策略。
- `agentConfig` 在 continuation 中未使用，说明配置与执行链路还未完全打通。

建议翻译任务也走统一 batch runner，每个 batch 带 timeout、重试、质量检查和 checkpoint。质量检查可包括：空输出、原文泄露比例、重复段落、Markdown fence、标题层级、表格行数近似一致性。

### 4.8 P1: artifact 全量快照存储会带来扩展压力

`ChatDocumentArtifact.ContentSnapshot` 保存完整文档快照，当前上限达到 1,200,000 字符。每次续写、修改都会生成新版本全量快照。短期实现简单，但长期会带来：

- 数据库膨胀。
- API 拉取完整 artifact 慢。
- 前端预览和历史版本列表容易携带过大 payload。
- 单条记录超过部分数据库 text/blob 预期或备份成本上升。

建议采用“快照 + delta + 外部对象存储”的折中策略：

- 短文档保留内联 snapshot。
- 超过阈值后将正文写入对象存储或统一文件服务，只在 DB 保存 URI、checksum、size、head/tail preview。
- artifact revision 保存 delta patch 与周期性 compact snapshot。
- 列表接口默认不返回完整 content，只在详情接口或导出接口按需获取。

### 4.9 P2: 证据引用应保存不可变证据快照

`chat_document_evidence_refs` 已保存 `knowledge_id`、`chunk_id`、score、checksum 等信息，但如果知识库重新解析或 chunk 内容变化，历史文档引用会漂移。建议至少保存证据 excerpt、source offset、chunk content hash 和解析版本号，以便长期可审计。

### 4.10 P2: 前端仍存在硬编码文案和重复业务规则

前端新增 `ChatDocumentArtifactCard.vue` 中大量展示文本为中文硬编码，而本次改动同时修改了多语言文件。建议将文档版本、查看版本链、设为基线、完整文档预览、自动续写提示等文案统一进入 i18n。

同时，前端 `index.vue` 复制了大量文档意图、章节目标和自动续写规则。建议前端只保留展示逻辑，核心判断下沉到服务端。

### 4.11 P2: 改动范围中夹杂非长文档文件，提交边界需要收敛

当前未提交改动包含 `.gitignore`、`frontend/pnpm-lock.yaml`、`internal/databaseconnector/mysql/connector.go` 等与本次长文档目标不直接相关的文件。若这些不是本次功能必需，建议拆分提交，避免后续回滚、审查和定位问题时扩大影响面。

## 5. 推荐的优化目标架构

推荐目标是“后端权威 Run + 批次事件溯源 + Artifact 用户视图”。核心对象如下：

1. `document_generation_runs`

   表示一次用户发起的长文档任务，是后端调度权威。保存原始请求、任务类型、模型、知识范围、状态、预算、最大轮次、当前阶段、最终 artifact id。

2. `document_generation_steps`

   表示 outline、检索、section 生成、translation segment、repair、finalize 等步骤。每个步骤保存输入摘要、输出引用、状态、耗时、token、错误原因和重试次数。

3. `document_generation_events`

   追加式事件表，保存规划完成、批次开始、批次完成、质量告警、暂停、恢复、完成等事件。SSE 可从这里 replay。

4. `chat_document_artifacts`

   用户可见文档版本。只表达最终或 checkpoint 文档，不承担任务调度。长文档正文大于阈值时转为外部 content store。

5. `document_generation_evidence_refs`

   保存每个 section/segment 使用的不可变证据快照，支持审计和重新生成。

6. `DocumentGenerationOrchestrator`

   统一入口，负责创建 run、恢复 run、调度下一步、更新状态机、输出事件。

7. `DocumentRenderer/Aggregator`

   负责把 steps 的输出聚合成 artifact，保证 artifact 与 run checkpoint 一致。

前端只做三件事：

- 展示 run 进度和 artifact。
- 发起暂停、继续、取消、设为基线等明确指令。
- 通过 run id 重连并 replay 事件。

## 6. 分阶段落地方案

### 阶段一：补齐上线阻断项

1. 补齐 SQLite 迁移，创建 `chat_document_generation_runs` 和 `chat_document_evidence_refs`。
2. 调整 drop 旧表策略，至少增加数据迁移说明和开关保护；有线上数据时先 backfill，不直接 drop。
3. 增加迁移测试，校验当前 Go model 涉及的表和列在 Postgres/SQLite 都存在。
4. 对全文翻译单批模型调用增加 timeout 和错误 reason 标准化。
5. 将 `.gitignore`、MySQL connector、锁文件等非长文档改动拆出或说明原因。

### 阶段二：统一状态机和服务端续写控制

1. 建立 `DocumentGenerationStateMachine`，集中处理 run/message/artifact 状态转换。
2. 后端 complete 事件返回 `next_action`，例如 `continue_auto`、`wait_user_review`、`blocked`、`done`。
3. 前端删除核心续写判断，只按 `next_action` 触发明确请求或展示按钮。
4. `generation_run` 记录当前 active artifact、已完成批次、失败 reason、下一批次位置。
5. 增加多端、刷新、断线重连后的续写恢复测试。

### 阶段三：批次执行任务化

1. 新增 batch/step 表，将 section 和 translation segment 的输出逐批持久化。
2. 使用 asynq 执行云端/标准版后台任务，Lite 使用现有 `SyncTaskExecutor`。
3. 每个 batch 独立事务提交：step output、evidence refs、run event。
4. artifact 聚合由后端聚合器完成，支持 checkpoint artifact。
5. SSE 改为订阅 run event，支持断线 replay。

### 阶段四：模块拆分和质量体系升级

1. 拆分 `chat_document_artifact.go` 的意图识别、上下文构造、patch 合并、质量策略和 artifact 管理职责。
2. 合并前端、handler、service 中重复的章节目标识别规则，服务端输出结构化结果。
3. 翻译路径加入批次质量门：重复段落、原文泄露、页眉页脚残留、表格结构、Markdown 合法性。
4. 对 evidence refs 保存不可变 excerpt 和 source span。
5. 长文档正文超过阈值后迁移到对象存储或文件服务。

## 7. 验收标准建议

上线前建议至少补齐以下验证：

1. 迁移验证。

   新空库和旧库升级在 Postgres 与 SQLite 下均通过，所有新表和列存在，旧数据按预期保留或迁移。

2. 长文档生成验证。

   50 页、100 页、200 页级别的技术方案生成均能完成、暂停、续写、刷新恢复。

3. 全文翻译验证。

   使用含 chunk overlap、PDF 页眉页脚、表格、标题层级的真实解析文档，验证译文不重复、不丢段、不把解析噪声放大。

4. 状态一致性验证。

   message、artifact、run、前端展示在 completed、continuing、needs_review、blocked、failed 场景下保持一致。

5. 多端与断线验证。

   同一会话多端打开、SSE 断线、浏览器刷新、后端重启后，不重复续写、不丢 checkpoint。

6. 回归验证。

   普通问答、数据库智能体、短翻译、带附件问答不能误入长文档路径。

## 8. 结论

当前代码从功能实现角度已经完成了关键能力：长文档一次性输出、翻译、规划、批次执行、artifact 版本链和基线续写。它已经不是简单的 prompt 拼接方案，而是走向 artifact-first 的工程化方案。

但从整体架构成熟度看，它还处在“功能闭环优先”的阶段。真正稳健的落地方案应把长文档视为后台可恢复任务，而不是一次聊天请求的长尾流程；把 `generation_run` 升级为权威状态机；把前端自动续写变成服务端调度；把每个批次输出、证据和状态事件持久化。这样才能支撑长时间运行、多端重连、生产升级和后续更复杂的文档编辑能力。

建议优先处理 P0 迁移风险，然后分阶段推进后端任务化和模块拆分。这样既能保留当前已实现的功能成果，也能把系统带到更合理、更可维护的长期架构上。
