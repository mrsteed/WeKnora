# Agent 文档修订超时与基线重编辑问题解决方案

## 1. 问题结论

这次失败的直接表现是前端显示 `tool_error`、`生成失败，请重试`，日志中最终报错为：

```text
LLM call failed: LLM stream error: context deadline exceeded
(synthesis also failed: LLM stream error: context deadline exceeded)
```

它已经不是上一轮分析中的 OpenAI 工具消息协议 400 问题。日志明确显示：

```text
[Agent][Round-2] think_start ... protocol_problems=0
```

也就是说，历史 `assistant(tool_calls) -> tool` 的协议链在这一轮进入模型前已经通过校验。真正的失败原因是：在“后续追问 / 重新编辑基线文档”的场景下，系统把上一版文档上下文、运行时上下文、完整工具列表、工具结果和最终合成提示叠加到同一条 Agent 链路里，导致第二轮和失败兜底合成两次进入大上下文、长思考、无可见答案的超时路径。

核心根因可以概括为四点：

1. 文档修订请求仍按通用 ReAct 检索 Agent 执行，未走轻量的“基线文档结构化编辑”路径。
2. `document_revision_context` 被内联进 `query` 后，在最终答案合成阶段又被重复放进 `User question`，形成大段上下文重复。
3. Round-1 工具结果已经追加到 Redis 历史和当前 messages，Round-2 prompt 从约 52k token 增长到约 59k token，但默认上下文压缩阈值是 200k * 0.8，完全不会触发压缩。
4. Round-2 超时后，兜底合成仍使用完整系统提示和原始大 query，再次触发相同超时，最后被标记为 `tool_error`，前端只能展示失败。

要彻底解决后续追问或重新编辑基线文档的问题，不能只调大超时时间。必须把“文档 artifact 编辑”从“通用检索 Agent”中拆出受控路径，限制 prompt 规模，避免失败轮次污染下一轮上下文，并保证失败时仍保留上一份有效基线。

## 2. 日志链路复盘

### 2.1 第一轮并未失败，但已经进入高成本区间

日志显示第一轮请求：

```text
model=deepseek-v4-pro, prompt_tokens=52040, completion_tokens=2365, total_tokens=54405
Completed: chunks=2232, content_len=0, tool_calls=2, stream_duration=74426ms
finish_reason="tool_calls"
```

含义是：

1. 第一轮 prompt 已达到 52k tokens。
2. 模型用了约 72 秒才开始产生工具调用。
3. 第一轮没有输出可见答案，只发起了 `wiki_read_page` 和 `wiki_read_source_doc` 两个工具调用。
4. 工具本身执行很快，分别约 12ms 和 6ms，不是瓶颈。

因此瓶颈不在工具执行，而在模型面对大上下文和复杂系统提示时的思考耗时。

### 2.2 工具结果追加后，第二轮进入超时

工具执行后，系统追加了 3 条工具组消息：

```text
Added 3 tool-group message(s) to context
Round-2 Starting: 40 messages, 24 tools, est_tokens=59468
msg[36]: role=user, len=41632, content=<runtime_context ...>
msg[37]: role=assistant, len=0, tool_calls=[wiki_read_page wiki_read_source_doc]
msg[38]: role=tool, name=wiki_read_page, len=4369
msg[39]: role=tool, name=wiki_read_source_doc, len=8614
```

关键点：

1. 第二轮已经带上当前大 query、assistant tool_calls、两个工具结果。
2. token 估算增长到约 59k。
3. `protocol_problems=0`，说明不是工具协议错误。
4. 24 个工具定义仍然全部随请求发送，文档修订任务没有缩小工具面。

随后第二轮日志为：

```text
First reasoning_content ... elapsed_ms=1612
Completed: chunks=3676, content_len=0, tool_calls=0, stream_duration=118090ms, type_distribution=map[error:1 thinking:3675]
Iteration-2 failed: LLM stream error: context deadline exceeded
```

含义是：模型很快开始输出隐藏思考，但在 118 秒内没有给出 answer，也没有再给出 tool call，最后命中 `LLMCallTimeout`。

### 2.3 兜底合成重复踩同一个坑

`callLLMWithRetry()` 看到已有 1 个 step 和 2 个工具结果后，进入兜底合成：

```text
LLM failed but have 1 steps with 2 tool calls — attempting final answer synthesis from existing results
FinalAnswer Synthesizing from 1 steps, 2 tool calls
Built context: 4 messages, 2 tool results
Starting LLM stream with 5 messages
```

但 `streamFinalAnswerToEventBus()` 当前会重新构造如下消息：

1. 完整 `BuildSystemPromptWithOptions(...)`。
2. 原始 `query`。
3. 文档续写/修订上下文补充。
4. 工具结果摘要。
5. `buildFinalAnswerSynthesisPrompt(query, state)`，其中再次包含完整 `User question: %s`。

在本场景中，原始 `query` 已包含 `<document_revision_context>...</document_revision_context>`。因此最终合成不是轻量总结，而是把大文档修订上下文再次送入模型，且仍带完整通用系统提示。结果是兜底合成也超时，最终抛出：

```text
synthesis also failed: LLM stream error: context deadline exceeded
```

### 2.4 为什么前端显示 tool_error

失败后状态由 `callLLMWithRetry()` 写入：

```go
state.CompletionStatus = "failed"
state.FinishReason = "tool_error"
state.FailureReason = "tool_error"
```

`agent_stream_handler.handleComplete()` 会把这些字段写回 assistant message，并通过 complete 事件传给前端。前端看到 `completion_status=failed` 后停止自动续写，展示“生成失败，请重试”；同时 `failure_reason=tool_error` 展示在错误框中。

这里的 `tool_error` 命名不准确。实际失败不是工具错误，而是“第二轮模型调用超时 + 兜底合成超时”。错误分类应调整为 `llm_timeout` 或 `synthesis_timeout`，否则排障会被误导。

## 3. 代码定位

### 3.1 文档上下文被拼进 Agent query

位置：`internal/application/service/session_agent_qa.go`

当前逻辑：

```go
agentQuery := req.Query
...
if req.QuotedContext != "" {
    agentQuery += "\n\n" + req.QuotedContext
}
...
engine.Execute(ctx, sessionID, req.AssistantMessageID, agentQuery, llmContext, agentImageURLs)
```

同时，`agentDocumentContextFromQARequest()` 又把同一份 `req.QuotedContext` 放入 `AgentDocumentContext`：

```go
QuotedContext: req.QuotedContext,
```

这导致文档上下文既成为用户 query 的一部分，又成为 fallback final synthesis 可再次引用的数据源。

### 3.2 文档修订上下文会内联上一版文档

位置：`internal/application/service/chat_document_artifact.go`

`BuildQuotedContext()` 在 revise delta 模式下生成：

```xml
<document_revision_context>
你正在修改同一会话中的上一份文档。
...
上一份文档内容：
<document>
...
</document>
</document_revision_context>
```

虽然该函数已经有目标章节窗口和超长截断逻辑，但目前只能表达一个 `target_heading`。用户这类请求：

```text
把 2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

实际上包含两个锚点：

1. 源锚点：`2.5.5 火电设备运维智能体——技术实现` 之后的内容。
2. 目标锚点：第二章。

现有 `target_heading` 无法同时表达“源范围 + 目标章节”，因此容易退化为较大的上下文窗口，让模型自行理解结构移动。

### 3.3 上下文压缩阈值过高

位置：`internal/types/agent.go`、`internal/agent/token/compress.go`

当前默认值：

```go
const DefaultMaxContextTokens = 200000
const DefaultContextThresholdRatio = 0.8
```

也就是说，只有估算超过 160k tokens 才压缩旧消息。日志中的 52k / 59k 对模型来说已经足以造成 120 秒超时，但远未达到压缩阈值。

这说明当前 `MaxContextTokens` 是“模型最大可接受上下文”思路，不是“在 120 秒内稳定完成的延迟预算”思路。长文档编辑需要单独的 prompt budget。

### 3.4 兜底合成重复大 query

位置：`internal/agent/finalize.go`

当前合成提示：

```go
messages := []chat.Message{
    {Role: "system", Content: systemPrompt},
    {Role: "user", Content: query},
}
...
finalPrompt := buildFinalAnswerSynthesisPrompt(query, state)
messages = append(messages, chat.Message{Role: "user", Content: finalPrompt})
```

而 `buildFinalAnswerSynthesisPrompt()` 又包含：

```go
User question: %s
```

当 `query` 中包含完整 `<document_revision_context>` 时，最终合成会至少重复一次大文档上下文。

### 3.5 工具结果会在失败前写入 Redis 上下文

位置：`internal/agent/observe.go`

`appendToolResults()` 会在每一轮工具执行后立即写入 ContextManager：

```go
if e.contextManager != nil && len(contextMessages) > 0 {
    e.contextManager.AddMessages(ctx, e.sessionID, contextMessages)
}
```

这对当前运行的 Round-2 不是必须的，因为本地 `messages` 已经持有工具结果；它主要服务跨轮历史恢复。但如果随后 Round-2 和最终合成都失败，这些工具历史仍可能留在 Redis 中，下一次“继续追问 / 重新编辑基线文档”会加载失败轮次产生的检索上下文，增加 prompt 尺寸和误导风险。

## 4. 根治目标

修复后应满足以下目标：

1. 纯文档基线编辑不再默认走 24 个工具的通用 ReAct 检索流程。
2. 同一份 `document_revision_context` 在一次 LLM 请求中最多出现一次。
3. 最终兜底合成必须是轻量提示，不使用完整 Agent 系统提示，不重复大 query。
4. 失败轮次产生的工具结果不能污染下一轮历史上下文。
5. 对“源章节/源范围 -> 目标章节”的编辑请求，系统能构造双锚点上下文，而不是把大文档整体交给模型猜。
6. 如果仍然超时，前端和消息状态应体现 `llm_timeout` / `synthesis_timeout`，并继续指向上一份有效 artifact 作为基线。

## 5. 推荐修复方案

### 5.1 P0：文档修订请求拆出轻量编辑路径

在 `session_agent_qa.go` 或更靠近 chat document pipeline 的位置增加分流：

```text
if req.DocumentIntent == revise
   && req.BaseArtifactID != ""
   && req.DocumentOutputMode == delta_only
   && 本轮没有明确要求检索知识库 / 联网 / 查询数据库:
       走 DocumentEditAgent / DocumentPatchSynthesizer
else:
       走现有通用 Agent
```

轻量编辑路径的行为：

1. 不注册 KB 检索、wiki、数据库、Web search、MCP 等工具。
2. 不进行多轮 ReAct，最多一次 LLM 调用。
3. 默认关闭 Thinking。
4. 输入只包含：用户目标、文档大纲、源章节窗口、目标章节窗口、必要的相邻章节窗口。
5. 输出只允许 `<document_patch>` 或系统内部 patch 结构。
6. 成功后继续复用现有 `RegisterFromAssistantMessage()` 和 `mergeChatDocumentRevisionDelta()` 注册新 artifact。

这样“把某一节后续内容合并到第二章”不会触发知识库读取，也不会进入 Round-2 大上下文。

### 5.2 P0：支持双锚点结构化编辑

当前 `BuildQuotedContext()` 只能表达一个 `target_heading`。建议新增文档编辑计划结构，例如：

```go
type ChatDocumentEditPlan struct {
    Operation          string // append_to_section, replace_section, move_after_heading_to_section
    SourceHeading      string
    SourceRangeMode    string // section, after_heading, until_next_same_or_higher_heading
    DestinationHeading string
    MergeMode          string
}
```

针对用户请求：

```text
把 2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

解析为：

```text
operation: move_after_heading_to_section
source_heading: 2.5.5 火电设备运维智能体——技术实现
destination_heading: 第二章 / ## 第二章 ...
merge_mode: append_to_section
```

构造上下文时只给模型：

1. `<document_outline>`。
2. `<source_anchor>` 与源锚点后的待移动片段。
3. `<destination_section>` 第二章正文。
4. `<nearby_siblings>` 源章节和目标章节相邻窗口。

如果可以确定源片段和目标章节范围，优先直接在服务端生成 patch，而不是让模型重写：

```xml
<document_patch>
<append heading="## 第二章 ...">
待移动片段
</append>
<replace heading="## 2.5.5 火电设备运维智能体——技术实现">
保留到 2.5.5 截止的内容
</replace>
</document_patch>
```

如果现有 patch grammar 不适合表达“移动”，可以新增内部操作 `move_after_heading_to_section`，由后端执行合并；用户侧仍然只看到最终 artifact。

### 5.3 P0：不要把 QuotedContext 同时塞进 query 和 DocumentContext

建议调整 `session_agent_qa.go`：

1. `agentQuery` 保持用户原始问题，不直接拼接 `req.QuotedContext`。
2. 文档上下文只通过 `engine.SetDocumentContext()` 传递。
3. `buildMessagesWithLLMContext()` 根据 `DocumentContext` 决定是否注入紧凑文档上下文。
4. 对普通非文档场景，可保留原有 quoted context 行为，但文档 artifact 场景必须避免重复。

伪代码：

```go
agentQuery := req.Query
if req.QuotedContext != "" && !isChatDocumentRequest(req) {
    agentQuery += "\n\n" + req.QuotedContext
}
```

并在 `AgentDocumentContext` 中增加原始用户目标字段：

```go
type AgentDocumentContext struct {
    ...
    UserGoal string
}
```

最终合成和文档编辑提示都使用 `UserGoal`，而不是携带大段 XML 的 `query`。

### 5.4 P0：最终兜底合成必须瘦身

修改 `internal/agent/finalize.go`：

1. `streamFinalAnswerToEventBus()` 不再使用完整 `BuildSystemPromptWithOptions()`。
2. 新增轻量系统提示，例如 `buildFinalSynthesisSystemPrompt(language, documentContext)`。
3. `buildFinalAnswerSynthesisPrompt()` 使用 stripped query。
4. 对 document revise 场景，禁止把完整 `<document_revision_context>` 写入 `User question`。
5. 对 tool summaries 设置更小总预算，而不是每个工具 1800 chars 叠加。

建议新增工具函数：

```go
func compactQueryForFinalSynthesis(query string, documentContext *types.AgentDocumentContext) string {
    if documentContext != nil && strings.TrimSpace(documentContext.UserGoal) != "" {
        return strings.TrimSpace(documentContext.UserGoal)
    }
    query = stripTaggedBlock(query, "document_revision_context")
    query = stripTaggedBlock(query, "document_continuation_context")
    return strings.TrimSpace(query)
}
```

最终合成消息应类似：

```text
system: 你是最终答复合成器。只根据已给工具摘要和用户目标输出答案；不要调用工具；不要输出思考。
user: 用户目标：把 2.5.5 ... 后续内容合并到第二章。
user: 已检索摘要：...
user: 输出要求：如果是文档修订，只输出 document_patch；如果信息不足，输出可恢复的部分结果和缺口。
```

同时增加总预算保护：

```go
messages = shrinkFinalSynthesisMessages(messages, estimator, maxFinalSynthesisTokens)
```

建议初始值：

```text
maxFinalSynthesisTokens = 12000
maxFinalSynthesisToolSummaryCharsTotal = 4000
maxDocumentRevisionContextCharsForFallback = 6000
```

如果预算仍超限，应直接返回 partial recovery message，不再发起一次注定超时的 LLM 调用。

### 5.5 P0：失败轮次不要污染下一轮上下文

`appendToolResults()` 当前每轮立即写 Redis。建议改为“运行内使用本地 messages，跨轮持久化延后到完成事件”：

1. ReAct 当前运行内部继续向 `messages` 追加工具结果。
2. 不在 `appendToolResults()` 中立即 `contextManager.AddMessages()`。
3. 在 agent 成功完成、生成 partial 可恢复答案，或用户明确停止时，再把本轮可复用的消息组写入上下文。
4. 如果状态是 `failed`，默认不写入本轮工具历史；必要时只写一条短的 assistant 失败摘要，不写 tool messages。

如果短期内不方便改为延迟持久化，也至少要加 run-scoped 回滚：

```text
- appendToolResults 写入时记录本轮 run_id / message_id
- executeLoop 失败时删除本轮写入的 context messages
- 下一轮 buildMessagesWithLLMContext 跳过 failed message_id 对应的 tool group
```

这样下一次“重新编辑基线文档”不会携带失败轮的 wiki/source_doc 工具结果，也不会让 prompt 无意义增长。

### 5.6 P0：错误分类改为模型超时，而不是 tool_error

当前兜底合成失败后固定写：

```go
state.CompletionStatus = "failed"
state.FinishReason = "tool_error"
state.FailureReason = "tool_error"
```

应根据错误分类：

```go
reason := classifyAgentExecutionError(synthErr)
state.FinishReason = reason
state.FailureReason = reason
```

并扩展分类：

```go
func classifyAgentExecutionError(err error) string {
    lower := strings.ToLower(err.Error())
    switch {
    case strings.Contains(lower, "context deadline exceeded"):
        return "llm_timeout"
    case strings.Contains(lower, "synthesis") && strings.Contains(lower, "deadline"):
        return "synthesis_timeout"
    case strings.Contains(lower, "tool"):
        return "tool_error"
    default:
        return "llm_error"
    }
}
```

同时把 `context deadline exceeded` 加入 transient markers 不是根治，但可以避免错误被误判为非 transient：

```go
"deadline exceeded", "context deadline exceeded"
```

注意：对大 prompt 盲目重试会重复消耗 120 秒。更合理的策略是“第一次超时后缩小上下文再重试一次”，而不是原样重试。

### 5.7 P1：引入延迟预算，而不是只看最大上下文

`DefaultMaxContextTokens=200000` 适合表达“模型理论窗口”，不适合控制“120 秒内稳定返回”。建议增加独立配置：

```go
type AgentConfig struct {
    MaxContextTokens int
    PromptLatencyBudgetTokens int
    FinalSynthesisBudgetTokens int
    DocumentEditBudgetTokens int
}
```

建议默认值：

```text
PromptLatencyBudgetTokens: 48000
FinalSynthesisBudgetTokens: 12000
DocumentEditBudgetTokens: 16000
```

在 `runReActIteration()` 发送模型前，如果估算超过 `PromptLatencyBudgetTokens`：

1. 文档修订场景：切换轻量编辑路径或压缩文档上下文。
2. 普通检索场景：压缩历史、裁剪工具结果摘要、减少工具列表。
3. 如果仍超限：返回可恢复失败，不发起 LLM 调用。

这能避免日志中 52k prompt 已经明显慢，但因为没到 160k 压缩线而继续请求的问题。

### 5.8 P1：文档修订时缩小工具面

对于 `DocumentIntentRevise` 且用户没有要求“结合知识库补充事实”的请求，工具列表应为空或仅保留文档编辑相关内部能力。

可在 `buildAgentConfig()` 后增加：

```go
if isPureDocumentRevision(req) {
    agentConfig.AllowedTools = nil
    agentConfig.WebSearchEnabled = false
    agentConfig.MaxIterations = 1
    thinking := false
    agentConfig.Thinking = &thinking
}
```

如果用户请求包含“参考知识库、补充资料、查询原文、根据北海电厂方案补充”等语义，再允许 KB 工具，但仍要限制工具集合和结果预算。

## 6. 建议落地顺序

### 阶段一：先止血

1. 将当前失败会话的 Redis LLM context 清理，避免失败轮工具历史继续影响后续追问。
2. 前端或后端在重新编辑时显式使用最近一条 `completed/partial` 且有 artifact 的基线，不选 failed assistant。
3. 将文档修订请求的 `MaxIterations` 临时降为 1，关闭 Thinking，缩小工具列表。
4. 将 `LLMCallTimeout` 调大只能作为临时缓解，不应作为最终方案。

### 阶段二：修复重复上下文和合成超时

1. 文档请求不再把 `QuotedContext` 直接拼进 `agentQuery`。
2. final synthesis 使用 stripped query 和轻量系统提示。
3. final synthesis 增加 token/char 总预算，超限直接返回 partial recovery，不再盲调模型。
4. 错误分类改为 `llm_timeout` / `synthesis_timeout`。

### 阶段三：拆出文档编辑专用路径

1. 增加 `DocumentPatchSynthesizer` 或 `DocumentEditAgent`。
2. 支持双锚点编辑计划。
3. 可确定的移动/合并操作优先服务端确定性合并。
4. LLM 只负责改写局部段落，不负责读完整大文档。

### 阶段四：上下文持久化治理

1. 工具结果延迟持久化，失败轮次默认不写入跨轮上下文。
2. Redis context 加 message_id/run_id 标记，支持失败回滚或过滤。
3. 历史构造时跳过 failed assistant 对应的工具组。

## 8. 验收标准

修复完成后，用同类请求验证：

1. 日志中不再出现 `Round-2 Starting: ... 24 tools, est_tokens=5xxxx` 的大上下文 ReAct 形态。
2. 文档修订请求优先只跑一次轻量编辑调用。
3. final synthesis 的 request 中不再重复出现完整 `<document_revision_context>`。
4. 超时失败时，`finish_reason/failure_reason` 为 `llm_timeout` 或 `synthesis_timeout`，不是 `tool_error`。
5. 失败后重新追问或重新编辑，基线仍指向上一份成功 artifact。
6. Redis 历史中不保留失败轮次的 wiki/source_doc 工具结果，或构造下一轮消息时会过滤。
7. 前端不会出现“当前基线切到失败答案”的情况。
