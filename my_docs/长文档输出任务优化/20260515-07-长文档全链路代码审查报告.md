# 长文档全链路代码审查报告

审查日期：2026-05-15  
审查依据：`my_docs/长文档需求.md`  
审查范围：后端路由、长文档生成、知识库证据检索、预算协商、artifact/版本基线、自动续写、章节修订、Markdown 质量闸、SSE 完成事件、前端基线选择与全文查看。

## 1. 总体结论

当前代码已经从早期 `long_document_tasks` 迁移到以 `chat_document_artifacts`、`chat_document_generation_runs`、SSE complete payload 和前端 artifact 卡片为核心的新链路。整体方向是正确的：

1. 路由层已尽量保守地区分普通问答、长文档创建、知识库增强长文档、文档编辑和自动续写。
2. 完整文档生成已具备“大纲规划 -> 分章节生成 -> Markdown 规范化 -> artifact 注册 -> 前端查看/续写”的主流程。
3. 知识库增强长文档已经有 generation run，用于持久化大纲、预算、运行反馈、完成章节和 root artifact。
4. 前端已经支持 artifact 卡片、版本链、查看全文/查看修改内容、自动续写和失败重试。

但按需求里的“完整长文档产品能力”审查，目前仍有几个会直接影响生产可用性的关键缺口：

1. **P0：协商出的章节/续写 token 预算没有传给章节和续写模型调用。** 这会导致系统日志和状态显示“预算已协商”，但实际模型请求仍使用模型/客户端默认值，长文档最容易继续出现章节截断、提前结束、续写不完整。
2. **P0：大纲规划仍可能继承 Agent thinking 设置。** 对 JSON-only 大纲任务，thinking 会消耗输出预算并增加 JSON 截断/解析失败概率。
3. **P1：无知识库的完整长文档没有持久化 generation run。** 一旦失败续写，原始大纲、协商预算和运行反馈不能被后续请求稳定复用。
4. **P1：前端手动选定的基线会被新完成 artifact 自动覆盖。** 这和需求“手动指定版本优先”冲突。
5. **P1：`needs_review` artifact 被同时禁止自动续写和手动修订。** 质量复核态应暂停自动续写，但不应阻断用户查看全文或基于该版本人工修订。

## 2. 需求覆盖情况

| 需求项 | 当前实现 | 结论 |
| --- | --- | --- |
| 1. 用户要求生成完整文档时进入长文档模式 | `chat_route_service.go` 与 `session_agent_qa.go` 均有 full document 路由守卫，普通问答有保守兜底 | 基本满足，但依赖关键词和模型路由，仍需回归测试防误触发 |

| 3. 和模型协商上下文参数，后续生成使用协商参数 | `resolveDocumentGenerationBudget` 已协商预算并记录到 extra/run | 部分满足；章节/续写调用未使用 `MaxCompletionTokens`，属于关键缺陷 |

| 9. 完成/重试/修订后自动选择最新版本，手动指定版本优先 | 自动 promote 已有，手动锁也有 | 未完全满足；自动 promote 会清除手动锁 |
| 10. 修正异常 Markdown | 规范化、结构校验、prompt leakage 检测已实现 | 基本满足；质量告警与完成态边界还需继续收口 |
| 11. 不破坏非长文本模式 | 路由层、前端 intent 推断均有 normal guard | 基本满足，需持续回归 |

## 3. 全链路架构观察

当前链路可分为五层：

1. **路由层**
   - 前端根据输入和已选基线推断 `intent_hint`、`base_artifact_id`、`document_output_mode`、`document_target_heading`。
   - 后端 `chat_route_service.go` 负责模型路由和 regex fallback。
   - `session_agent_qa.go` 再根据 `DocumentOutputMode`、`BaseArtifactID`、知识库范围和附件状态选择 dedicated full document、knowledge grounded full document、document edit 或普通 Agent。

2. **规划与预算层**
   - `resolveDocumentGenerationBudget` 先从模型能力和配置得到静态预算，再通过 JSON-only 协商得到 outline/section/continuation tokens、TopK、timeout。
   - 大纲规划用 `generateValidatedFullDocumentOutline`，并支持流式 progress。

3. **生成执行层**
   - 无知识库：`runDedicatedFullDocumentGenerationPath` 按大纲逐章生成。
   - 有知识库：`runKnowledgeGroundedFullDocumentGenerationPath` 先检索 outline evidence，再逐章检索 section evidence 生成。
   - 续写：`runKnowledgeGroundedDocumentContinuationPath` 和 `runKnowledgeGroundedDocumentContinuationRunPath` 负责普通续写与基于 run 的续写。
   - 修订：`runDedicatedDocumentEditPath` 输出 structured patch 或章节片段。

4. **版本与状态层**
   - `chat_document_artifact.go` 负责 artifact 注册、快照合并、Markdown 修复、质量 issue、版本链和 derived fields。
   - `chat_document_generation_runs` 保存知识库增强长文档的大纲、预算、运行反馈、完成章节和 active artifact。
   - complete payload 携带 `document_generation_status`、`auto_continue_next`、`next_action`、`recommended_request`、`final_document_mode`。

5. **前端体验层**
   - `index.vue` 管理 selectedBaseArtifact、autoContinueState、final_document_content、历史恢复、retry。
   - `ChatDocumentArtifactCard.vue` 展示版本、状态、查看全文、设为基线、导出。
   - `documentCompletion.js` 统一显示 continuing / completed / needs_review / blocked 的文案。

这套设计的正确演进方向应当是：**message 只表达本轮对话结果，artifact 表达文档版本快照，generation run 表达长任务的长期计划与运行状态，frontend selection policy 表达用户当前编辑基线。** 当前主要缺陷都来自这四个职责仍有交叉或参数没有贯穿到底。

## 4. 关键问题清单

### P0-1 协商预算没有传入章节/续写模型调用

证据：

1. `DocumentGenerationBudget` 定义了 `SectionMaxCompletionTokens` 和 `ContinuationMaxCompletionTokens`，协商函数也会更新这些字段。
2. 大纲调用传入了 `MaxCompletionTokens: currentBudget.OutlineMaxCompletionTokens`。
3. 但完整章节生成调用只传了 `Temperature` 和 `Thinking`，没有传 `MaxCompletionTokens`：
   - `runKnowledgeGroundedFullDocumentGenerationPath` 的章节 `ChatStream`。
   - `runDedicatedFullDocumentGenerationPath` 的章节 `ChatStream`。
   - `runKnowledgeGroundedDocumentContinuationPath` 的续写 `ChatStream`。
   - `runKnowledgeGroundedDocumentContinuationRunPath` 的后续章节 `ChatStream`。
4. 测试里甚至显式断言章节调用 `MaxCompletionTokens == 0`，说明当前测试把这个缺陷固化成了期望。

影响：

1. 需求第 3 条“后续生成中要使用协商好的参数”没有真正达成。
2. 需求第 4 条“逐章节自动完成到文档结束”会受模型默认输出上限影响。
3. 用户看到的预算日志、运行反馈和真实请求不一致，会造成问题定位困难。
4. 曾经出现的“预算显示 8192，实际截断/输出不完整”类问题容易复发。

建议：

1. 新增统一构造函数，例如：
   - `chatOptionsForDocumentOutline(budget, agentConfig)`
   - `chatOptionsForDocumentSection(budget, agentConfig)`
   - `chatOptionsForDocumentContinuation(budget, agentConfig)`
2. 章节生成必须传：
   - `MaxCompletionTokens: currentBudget.SectionMaxCompletionTokens`
   - `Thinking: false`
3. 续写生成必须传：
   - `MaxCompletionTokens: currentBudget.ContinuationMaxCompletionTokens`
   - `Thinking: false` 或按专门策略控制
4. 测试应从 `assert.Zero(options.MaxCompletionTokens)` 改为断言等于预算值，并增加 runtime feedback 后下一轮使用调整预算的用例。
5. 日志中增加 `request_max_completion_tokens` 与 `budget_section_tokens` / `budget_continuation_tokens` 的一致性字段。

### P0-2 大纲规划可能继承 Agent thinking，导致 JSON 规划不稳定

证据：

1. `runKnowledgeGroundedFullDocumentGenerationPath` 中，outline options 的 `Thinking` 来自 `agentConfig.Thinking`。
2. `runDedicatedFullDocumentGenerationPath` 中也同样从 `agentConfig.Thinking` 继承。
3. 大纲 prompt 明确要求 JSON-only、不要 hidden reasoning，但 thinking 选项为 true 时，部分模型会先输出 reasoning 或消耗 completion budget。
4. `generateValidatedFullDocumentOutline` 只重试两次，最终失败会进入 `outline_parse_failed`。

影响：

1. 对 DeepSeek / reasoning 模型尤其危险，reasoning 内容会挤占 `OutlineMaxCompletionTokens`。
2. 大纲 JSON 可能被截断，触发 `outline_parse_failed`，整篇长文档无法开始。
3. 需求第 2 条和第 3 条都受影响。

建议：

1. 大纲规划一律显式 `thinking := false`，不要继承 Agent 配置。
2. budget negotiation 已经这么做，outline generation 应保持一致。
3. 测试新增：当 agentConfig.Thinking=true 时，outline ChatStream options 仍应 `Thinking=false`。
4. 对 outline stream 增加 finish reason 和 raw length 观测，便于区分 JSON 内容截断与解析错误。

### P1-1 无知识库完整文档没有 generation run，失败续写无法复用原大纲/预算

证据：

1. 知识库增强完整文档会调用 `createKnowledgeGroundedGenerationRun`，后续 completion 和 artifact 会绑定 run。
2. 无知识库 `runDedicatedFullDocumentGenerationPath` 只把 `outline`、`completed_sections`、`budget` 放进 complete extra，没有创建或持久化 generation run。
3. 前端 auto continue 的 `generation_run_id` 只从 complete payload 读取；dedicated path 没有 run id，后续只能用 `base_artifact_id + delta_only`。
4. 后端普通 continuation context 只能从 partial artifact 中提取已有标题和尾部窗口，不能知道尚未生成的大纲章节。

影响：

1. 无知识库长文档一旦章节中断，续写无法稳定知道“计划中剩余章节”。
2. 协商预算不能跨轮复用，下一轮可能重新协商出不同参数。
3. runtime feedback 的 low growth、timeout、short section 经验无法跨轮应用。
4. 需求第 5 条“基于当前基线继续生成文档”只能做到弱恢复，不能做到计划级恢复。

建议：

1. 把 `chat_document_generation_runs` 从“knowledge grounded 专用”提升为“所有 full document run 通用”。
2. dedicated full document 创建 run 时保存：
   - original query
   - outline_json
   - budget_json
   - completed_sections_json
   - runtime_feedback_json
   - model_id
   - root_message_id / root_artifact_id
3. 自动续写时优先走 run path：按 `remainingOutlineSections` 精确选择下一批章节。
4. 前端 recommended_request 应始终带上 `generation_run_id`。
5. artifact 注册后绑定 active/root artifact，保证失败重试和版本链统一。

### P1-2 手动指定基线会被新 artifact 自动覆盖

证据：

1. 用户点击“设为基线”时，`useArtifactAsBase` 会设置：
   - `selectedBaseArtifact.value = normalized`
   - `selectedBaseArtifactDisplayLocked.value = true`
2. 但新 complete payload 到达后，`promoteCompletedArtifactAsBase` 在满足 `shouldAutoSelectCompletedArtifact` 时会直接：
   - `selectedBaseArtifact.value = normalized`
   - `selectedBaseArtifactDisplayLocked.value = false`
3. 该函数没有检查当前是否处于手动锁定状态。

影响：

1. 违反需求第 9 条：“对于手动指定的版本，要以手动选定的版本为主”。
2. 用户正在基于 V2 做对比或准备修订时，系统完成 V4 后会把基线切到 V4。
3. 后续输入会误用最新版本，而不是用户指定版本。

建议：

1. `promoteCompletedArtifactAsBase` 应先判断 `selectedBaseArtifactDisplayLocked`：
   - 如果锁定且当前 selected id 与新 artifact 不同，不覆盖。
   - 如果新 artifact 是用户手动选定版本的直接子版本，可只更新 display hint，不改 selectedBaseArtifact。
2. 把“用于展示的最新版本”和“用于下一次请求的基线版本”拆成两个概念：
   - `latestArtifactForDisplay`
   - `selectedBaseArtifactForRequest`
3. 增加前端测试覆盖：手动选中 V1 后，V2 complete 不应改变下一次请求的 `base_artifact_id`。

### P1-3 `needs_review` 状态阻断了人工修订能力

证据：

1. `ChatDocumentArtifact.CanContinue()` 对 `needs_review` 和 `blocked` 都返回 false。
2. 前端 `canContinueChatDocumentArtifact` 同样把 `needs_review` 判为不可用。
3. 后端 `prepareDocumentRequest` 加载 base artifact 后要求 `artifact.CanContinue()`，否则直接放弃文档修订上下文。
4. 前端 `useArtifactAsBase` 也因为 `canContinueChatDocumentArtifact` 对 `needs_review` 版本提示不可作为基线。

影响：

1. `needs_review` 本意是“正文已生成但需人工复核”，不是“不能被用户继续操作”。
2. 用户无法基于一个有格式告警但内容可见的版本进行章节修订或补充。
3. 需求第 6 条“对已经生成的文档某个章节进行修改或扩写”被质量状态过度限制。

建议：

1. 拆分能力字段：
   - `can_auto_continue`
   - `can_manual_continue`
   - `can_manual_revise`
   - `can_use_as_base`
   - `can_index`
2. `needs_review` 应禁止自动续写和自动索引，但允许用户查看全文、导出、手动选择为修订基线。
3. 后端 `prepareDocumentRequest` 对显式 `base_artifact_id` 的 revise/continue 应允许 `needs_review` artifact，只要有非空 snapshot。
4. UI 上把 `needs_review` 显示成“需复核”，而不是等同“不可继续”。

### P1-4 章节修订路径没有使用长文档预算协商

证据：

1. `runDedicatedDocumentEditPath` 使用 `s.cfg.Conversation.Summary.MaxCompletionTokens`，缺省 4096。
2. 它没有调用 `resolveDocumentGenerationBudget`，也没有根据模型能力或目标章节长度调整预算。
3. 对“章节补充功能清单”“扩写某章”“替换完整章节”这类任务，4096 可能不足。

影响：

1. 需求第 6 条的章节修订/扩写在大章节场景下容易提前截断。
2. 和完整长文档的预算体系割裂，日志与用户体验不一致。

建议：

1. 文档编辑也使用 `DocumentGenerationBudget`，至少对 `ContinuationMaxCompletionTokens` 或单独 `EditMaxCompletionTokens` 建模。
2. 根据 target section 的 rune/token 估算和 merge mode 调整预算。
3. 对 append/replace/insert_after 分别设置不同最小预算。
4. structured patch prompt 中继续要求不要输出全量文档，降低 token 压力。

### P2-1 artifact 状态、文档生成状态、消息完成状态仍有职责交叉

现状：

1. `completion_status` 表示本轮消息是否 completed/partial/failed。
2. `document_generation_status` 表示文档任务是否 continuing/completed/needs_review/blocked。
3. `artifact.status` 表示 artifact available/partial/failed。
4. 质量 issue 又会影响 artifact status 和 generation status。

问题：

1. 质量告警有时应该只是“复核提醒”，不应改变生成完成态。
2. artifact partial 既可能表示“内容真的不完整”，也可能表示“内容完整但需复核”。
3. 前端把 `status=partial` 显示为“部分完成”，对 needs_review artifact 不够准确。

建议：

1. 将状态拆成两组：
   - 生成进度：`writing / continuing / completed / blocked / failed`
   - 质量评级：`clean / repaired / needs_review / unsafe_for_index`
2. artifact.status 只表达快照可用性：`available / unavailable`，不要承载质量语义。
3. 用户文案应优先用 `document_generation_status + quality_issues` 组合，而不是只看 artifact.status。

### P2-2 文档修订合并策略仍偏启发式，需要结构化补丁成为强契约

现状：

1. `mergeChatDocumentRevisionDelta` 优先解析 `<document_patch>`。
2. 如果没有 patch，会退化为找首个 Markdown heading 替换章节。
3. 找不到 heading 时追加到文档尾部，并标记 `delta_merge_uncertain`。

风险：

1. 对长文档里重复标题、相似标题、章节编号变化的场景，启发式匹配仍可能错位。
2. 如果模型没有输出 patch，系统只能靠首标题判断，风险较高。

建议：

1. 对 dedicated edit path 强制 structured patch，非 patch 输出只作为 fallback 并进入 needs_review。
2. patch operation 应允许稳定 section id，例如 outline number / heading path，而不只依赖 heading 文本。
3. artifact 中保存修订元数据：target heading、resolved heading、merge mode、merge confidence。
4. 前端可在 needs_review 卡片展示“合并位置不确定”，允许用户查看 delta 和全文。

## 5. 先进设计建议

### 5.1 建立统一长文档 Run State Machine

建议所有 full document 任务，包括无知识库、有知识库、全文翻译，都统一创建 `chat_document_generation_runs`。状态机建议如下：

1. `planning`：生成/恢复大纲。
2. `writing`：按章节生成。
3. `continuing`：已产生 artifact，但仍有剩余章节。
4. `needs_review`：正文快照存在，质量或合并位置需人工复核。
5. `completed`：正文完整，允许下载/查看/可作为默认下一基线。
6. `blocked`：缺证据、低增长、重复开头、轮次上限等阻断。
7. `failed/cancelled`：执行失败或用户取消。

Run 负责长期计划，artifact 负责版本快照，message 负责一次对话展示。

### 5.2 建立预算契约对象并贯穿所有模型调用

当前预算对象已经有雏形，但应成为强契约：

1. outline call 必须使用 `OutlineMaxCompletionTokens`。
2. section call 必须使用 `SectionMaxCompletionTokens`。
3. continuation call 必须使用 `ContinuationMaxCompletionTokens`。
4. edit call 应有 `EditMaxCompletionTokens` 或复用 continuation budget。
5. 每个模型调用日志都输出：
   - budget source
   - model id
   - request max completion tokens
   - expected budget tokens
   - whether matched
   - timeout seconds
   - thinking enabled

### 5.3 区分自动续写能力和人工修订能力

建议 artifact derived fields 不再只有 `can_continue`：

| 能力 | completed | continuing/partial | needs_review | blocked |
| --- | --- | --- | --- | --- |
| 查看全文 | 是 | 是，有提示 | 是，有提示 | 是，如有 snapshot |
| 自动续写 | 否/按需求 | 是 | 否 | 否 |
| 人工续写 | 是 | 是 | 是 | 视原因 |
| 章节修订 | 是 | 是 | 是 | 视原因 |
| 自动索引 | 是 | 否 | 否 | 否 |
| 默认选为新基线 | 是 | 只在继续场景 | 否，除非用户确认 | 否 |

这样可以满足“安全地暂停自动行为”和“不阻断用户修订”的平衡。

### 5.4 前端基线选择策略分层

建议前端维护三个概念：

1. `latestArtifact`：当前会话最新版本，用于展示和版本链。
2. `selectedBaseArtifact`：用户下一次请求显式使用的基线。
3. `activeDisplayArtifact`：当前消息正在展示全文/修订内容的 artifact。

自动完成只更新 `latestArtifact` 和 display hint；只有在未手动锁定时才更新 `selectedBaseArtifact`。

### 5.5 Markdown 质量闸改成分级策略

建议质量处理分三类：

1. **可自动修复**：标题空格、列表缩进、代码块闭合。生成态保持 completed，记录 `quality_issues`。
2. **需人工复核**：内部标签泄漏、未规划标题、合并位置不确定。生成态 completed 或 needs_review，允许查看全文，禁止自动索引/自动续写。
3. **阻断型错误**：空输出、JSON 大纲无法解析、知识库证据缺失、连续低增长、重复开头。生成态 blocked/continuing，给出继续或重试路径。

## 7. 优先级整改计划

### 第一阶段：修正真实生成能力

1. 所有章节/续写 ChatStream 使用协商预算的 `MaxCompletionTokens`。
2. 大纲规划、章节生成、续写生成统一禁用 thinking，除非未来有专门的 reasoning budget 设计。
3. 修正现有断言 `MaxCompletionTokens == 0` 的测试，改为预算值断言。

### 第二阶段：打通 run 级恢复

1. dedicated full document 也创建 generation run。
2. 自动续写和失败重试统一优先使用 run id。
3. run 中持久化 outline、budget、runtime feedback、completed sections。

### 第三阶段：收口前端版本策略

1. 修复手动锁被自动 promote 清除的问题。
2. 拆分 latest/display/request base 三个状态。
3. `needs_review` 显示为可复核版本，而不是简单“部分完成”。

### 第四阶段：质量与能力模型升级

1. artifact derived fields 拆分自动续写、人工修订、查看、索引能力。
2. Markdown 质量 issue 分级。
3. 修订 patch 增加 merge confidence 和 resolved heading 元数据。

## 8. 结语

当前实现已经具备长文档产品的主干，但还差一个关键动作：**把“计划、预算、运行反馈、版本选择”变成强契约，而不是只作为事件 metadata 或启发式状态存在。**

最先应修的是 P0：预算必须落到每一次章节/续写请求，大纲规划必须禁用 thinking。只要这两点不收口，后续再多的自动续写、Markdown 修复和前端提示都会在真实模型输出上继续摇摆。随后再把 dedicated full document 纳入 generation run，前端基线锁和 needs_review 能力边界就能顺势理清。
