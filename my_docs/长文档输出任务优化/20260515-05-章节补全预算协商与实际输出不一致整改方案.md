# 章节补全预算协商与实际输出不一致整改方案

## 1. 结论

基于本次日志，当前这轮“请为第7章 AR眼镜智能作业方案补充功能清单”的章节补全**没有完整输出**。

这不是“模型明明拿到了 `8192` 却只生成了一半”的问题，而是 dedicated document edit 实际发起流式请求时，`MaxCompletionTokens` 已经被单独改成了 `2048`，因此最终在 `completion_tokens=2048` 处以 `finish_reason=length` 被截断。

换句话说，本次链路里同时存在三组不同含义的数字：

1. `context_window=64000`：模型上下文窗口能力。
2. `max_output=8192`、`section_tokens=8192`、`continuation_tokens=8192`：长文档预算层解析出的章节输出预算。
3. `document_edit_stream_started ... max_completion_tokens=2048`：dedicated document edit 真正发给模型的本次请求上限。

最终生效的是第 3 组，而不是第 2 组。

因此，这次现象的准确判断应当是：

**预算协商/能力映射层已经把章节预算抬到 `8192`，但在 completed continuation run 重定向到 dedicated document edit 后，这份 budget 没有继续下传，实际请求改用 summary 配置的 `2048` 发起，导致章节补全仍然被提前截断。**

## 2. 当前到底输出完整没有

没有完整输出，证据足够明确。

### 2.1 模型是被长度上限截断，不是自然完成

日志显示：

1. `document_edit_stream_started ... max_completion_tokens=2048`
2. `[LLM Usage] ... completion_tokens=2048`
3. `document_edit_complete ... finish_reason=length`

这三行连在一起说明：

1. dedicated edit 请求的实际输出上限就是 `2048`。
2. 模型最终正好跑满了这 `2048`。
3. 结束原因是 `length`，不是 `stop`。

因此本轮输出是**被请求上限截断**，不是完整完成。

### 2.2 持久化状态也已经明确不是 completed

日志继续显示：

1. `completion_status: partial`
2. `finish_reason: length`
3. `failure_reason: section_generation_truncated`
4. `allow_complete: false`
5. `document_generation_status: continuing`

这说明当前后端对本轮结果的最终判定已经不是 completed，而是：

**章节补全未完成，本轮仅生成了部分结果，需要继续补齐。**

### 2.3 artifact 也不是可信的“完整整文权威结果”

日志显示：

1. 本轮注册了新的 revise artifact：`artifact_id: 443322be-1841-4a89-8194-92c148ea2477`
2. persistence summary 中存在 `quality_issues: [delta_merge_uncertain markdown_heading_normalized]`

这意味着：

1. 当前产物仍然是一次 revise/delta 型输出。
2. merge 结果还带有 `delta_merge_uncertain`。
3. 这类结果不能被当成“已经稳定收口的完整章节”或“可靠整文快照”。

因此，从消息状态和 artifact 质量闸两个层面看，这轮输出都**不能判定为完整**。

## 3. 为什么协商的上下文长度和最后模型输出不一致

这里实际混在一起的不是同一层参数，所以表面上看起来“不一致”，本质上是**链路中途切换了预算来源**。

## 3.1 预算协商阶段拿到的是“长文档章节预算”，不是 dedicated edit 实际请求值

日志先出现：

1. `[FullDocument][Router] selected=knowledge_grounded_document_continuation`
2. `[DocumentBudget] source=capability ... section_tokens=8192 continuation_tokens=8192 context_window=64000 max_output=8192`

这说明在 long document / continuation 这条路径上，系统已经把 DeepSeek 的能力解释为：

1. 上下文窗口 `64000`
2. 最大输出 `8192`
3. 章节预算 `8192`
4. 续写预算 `8192`

所以预算层本身并没有继续停留在 `4096`，这一点和之前旧问题不同。

## 3.2 negotiation 两次失败，但这不是本次 2048 截断的主因

日志中两次出现：

1. `negotiateDocumentGenerationBudget ... context deadline exceeded`
2. `fallback_section_tokens=8192`

这说明：

1. 动态协商调用超时了。
2. 预算没有拿到模型返回的精细化建议值。
3. 但 fallback 仍然保留了 capability-derived 的 `8192`。

也就是说，即便 negotiation 失败，本轮已解析出的章节预算仍然是 `8192`，并不是直接退回 `2048`。

因此，`2048` 的来源不在 negotiation 失败，而在**后续真正发起 dedicated edit 请求时重新选了一套 token 来源**。

## 3.3 completed continuation run 被重定向到 dedicated edit 后，预算链路断开了

日志顺序非常关键：

1. `knowledge_grounded_completed_run_redirect_to_document_edit`
2. `document_edit_path_selected`
3. `document_edit_stream_started ... max_completion_tokens=2048`

这说明控制流已经从 knowledge grounded continuation run 切到了 dedicated document edit path。

代码对应关系在 `internal/application/service/session_agent_qa.go`：

1. `runKnowledgeGroundedDocumentContinuationRunPath(...)` 先调用 `resolveDocumentGenerationBudget(...)` 得到 long document budget。
2. 当 `remainingSections == 0` 且命中 redirect 条件时，直接调用 `runDedicatedDocumentEditPath(...)`。
3. `runDedicatedDocumentEditPath(...)` 内部没有接收上一步已经解析好的 `DocumentGenerationBudget`，而是重新决定自己的 `maxCompletionTokens`。

这就是预算链路断开的控制点。

## 3.4 dedicated edit 实际使用的是 summary 配置，不是 document budget

代码里 dedicated edit 的关键逻辑是：

1. `maxCompletionTokens := s.cfg.Conversation.Summary.MaxCompletionTokens`
2. 若该值小于等于 0，则回退到 `4096`
3. `chatModel.ChatStream(... MaxCompletionTokens: maxCompletionTokens ...)`

这意味着：

1. dedicated edit 的实际请求上限根本不是 `resolveDocumentGenerationBudget(...)` 算出的 `SectionMaxCompletionTokens`。
2. 它直接读取的是 summary 配置。
3. 当前环境下 summary 配置显然是 `2048`，所以日志打印出了 `document_edit_stream_started ... max_completion_tokens=2048`。

这也正好解释了后续为什么会出现：

1. 流式请求上限是 `2048`
2. 最终 `completion_tokens=2048`
3. `finish_reason=length`

因此，你看到的“不一致”并不是模型输出和协商结果冲突，而是：

**长文档预算层协商的是 `8192`，dedicated edit 执行层却改用 `2048` 重新下发请求。**

## 4. 本次链路的真实执行过程

按时间顺序，可以收敛为下面这条链：

1. 用户请求带 `base_artifact_id`、`intent_hint=revise_document`、`document_output_mode=delta_only`，属于章节修订。
2. 路由判定 `document_edit`，并进入 agent QA 执行。
3. FullDocument Router 仍先选中了 `knowledge_grounded_document_continuation`。
4. budget 解析阶段得到 `section_tokens=8192`、`continuation_tokens=8192`。
5. negotiation 两次超时，但 fallback 预算仍保持 `8192`。
6. continuation run 发现 `remainingSections == 0`，于是重定向到 dedicated document edit。
7. dedicated edit 没有继承前面的 `budget`，而是从 `Conversation.Summary.MaxCompletionTokens` 取到 `2048`。
8. 实际流式请求以 `MaxCompletionTokens=2048` 发起。
9. 模型跑满 `2048` 后以 `finish_reason=length` 截断。
10. 后端最终把消息正确收口为 `partial`，并禁止当作 completed 使用。

所以，本次链路的关键问题不是“协商没生效”这一句就能概括，而是：

**协商/能力映射在 continuation 路径生效了，但 redirect 到 dedicated edit 后又被另一套 summary token 配置覆盖。**

## 5. 根因归纳

本次故障至少有四个直接根因。

### 5.1 根因一：document budget 和 dedicated edit request budget 分属两套独立来源

当前实现里：

1. long document / continuation 预算由 `resolveDocumentGenerationBudget(...)` 负责。
2. dedicated document edit 请求预算由 `runDedicatedDocumentEditPath(...)` 直接从 summary 配置读取。

两者没有统一的 budget owner，也没有强制传递关系。

### 5.2 根因二：redirect 到 dedicated edit 时，没有把 resolved budget 继续向下传递

这导致：

1. 前半段日志显示 `8192`
2. 真正执行编辑时又变成 `2048`

预算在控制流切换处丢失了。

### 5.3 根因三：negotiation timeout 让问题更难判断，但不是这次截断的第一根因

由于日志里同时出现：

1. `source=capability`
2. `negotiation_failed`
3. `document_edit_stream_started ... 2048`

很容易误判成“协商失败导致只剩 2048”。

但从日志可知，协商失败后的 fallback 仍是 `8192`，所以真正把请求压到 `2048` 的，是 dedicated edit 的独立 token 选择逻辑。

### 5.4 根因四：dedicated edit 缺少与 long document 相同级别的 budget observability

long document section path 已经有：

1. `request_max_completion_tokens`
2. `request_max_completion_tokens_expected`
3. `request_max_completion_tokens_matches_budget`

但 dedicated edit 当前只有：

1. `document_edit_stream_started ... max_completion_tokens=2048`

缺少“预算期望值”和“实际请求值”的并排对照，导致排查时容易把 capability、negotiation、request 三层值混在一起。

## 6. 整改建议

## 6.1 P0：为 dedicated edit 明确引入统一的 DocumentGenerationBudget 入参

建议修改 `internal/application/service/session_agent_qa.go`：

1. `runDedicatedDocumentEditPath(...)` 增加 `DocumentGenerationBudget` 入参，或从 `req` / `extra` 中显式接收 resolved budget。
2. `knowledge_grounded_completed_run_redirect_to_document_edit` 发生时，把当前已解析的 `budget` 原样透传给 dedicated edit。
3. dedicated edit 请求的 `MaxCompletionTokens` 不再直接读取 summary 配置，而是按以下规则选择：
   - revise / delta：优先使用 `budget.SectionMaxCompletionTokens`
   - continuation：优先使用 `budget.ContinuationMaxCompletionTokens`
   - 仅在 budget 缺失时才回退到 summary 配置

这样才能保证日志里的 `8192` 真正变成最终请求上的 `8192`。

## 6.2 P0：summary token 只作为 fallback，不得覆盖已解析的文档预算

当前 dedicated edit 的 `s.cfg.Conversation.Summary.MaxCompletionTokens` 适合作为通用问答默认值，不适合作为章节补全/文档修订的最高优先级预算来源。

建议规则改为：

1. 若存在 document budget，则 summary token 不参与覆盖。
2. 若 dedicated edit 是从 document continuation / revise redirect 而来，则 summary token 只能作为兜底值。
3. 若 summary token 小于 resolved document budget，不得反向压低请求上限。

## 6.3 P0：在 dedicated edit 路径补齐 request-vs-budget 观测日志

建议 dedicated edit 起流前新增与 long document section path 等价的日志字段：

1. `budget_source`
2. `budget_section_tokens`
3. `budget_continuation_tokens`
4. `request_max_completion_tokens`
5. `request_max_completion_tokens_expected`
6. `request_max_completion_tokens_matches_budget`

一旦出现：

1. budget 是 `8192`
2. request 却是 `2048`

就立即打出 warning，并在完成态中追加 `request_max_completion_tokens_mismatch` 质量问题。

## 6.4 P0：negotiation timeout 与 request budget mismatch 要分开判责

建议把本次链路拆成两类独立告警：

1. `document_budget_negotiation_timeout`
2. `document_request_budget_mismatch`

只有这样才能区分：

1. 是预算协商没拿到更优值
2. 还是预算拿到了，但执行层没照着发请求

本次日志中，真正导致截断的是第 2 类。

## 6.5 P1：若 dedicated edit 截断且 merge 不确定，不应暴露“整文已可查看”的语义

本轮虽然状态已经收口为 `partial`，但 artifact 仍带 `delta_merge_uncertain`。

建议：

1. 只有当 patch merge 成功并通过完整性校验时，才允许生成权威 `final_document` / `final_document_artifact_id`。
2. 若 `finish_reason=length` 或 `delta_merge_uncertain`，前端应优先展示“修改片段/继续补齐”，而不是暗示“当前整文已稳定可读”。

## 7. 建议补充回归

至少补以下回归。

### 7.1 redirect 后预算传递一致性

场景：

1. revise_document
2. `base_artifact_id` 存在
3. continuation run 结束后重定向到 dedicated edit
4. capability / fallback budget = `8192`

期望：

1. dedicated edit 请求 `MaxCompletionTokens = 8192`
2. dedicated edit 启动日志能同时打印 budget 与 request
3. `request_max_completion_tokens_matches_budget = true`

### 7.2 negotiation 超时但 request 仍应继承 fallback capability budget

场景：

1. negotiation timeout
2. `fallback_section_tokens = 8192`
3. redirect 到 dedicated edit

期望：

1. 实际请求仍然使用 `8192`
2. 不得因为 negotiation 超时退回 summary token `2048`

### 7.3 dedicated edit length 截断收口

场景：

1. revise delta
2. request `MaxCompletionTokens = 2048`
3. `finish_reason=length`

期望：

1. `completion_status = partial`
2. `failure_reason = section_generation_truncated`
3. `allow_complete = false`
4. 若 budget expected 大于 request，则额外打 `request_max_completion_tokens_mismatch`

## 8. 最终判断

针对这次日志，可以直接给出三个明确结论：

1. **当前这轮章节补全没有完整输出。** 证据是 `completion_tokens=2048`、`finish_reason=length`、`completion_status=partial`、`allow_complete=false`。
2. **`8192` 和最终输出不一致，不是模型没有按 `8192` 输出，而是 dedicated edit 实际请求只按 `2048` 发起。**
3. **造成这种不一致的关键根因，不在模型本身，也不在 capability 映射本身，而在 continuation run 重定向到 dedicated edit 后，resolved document budget 没有继续透传，反而被 summary token 覆盖。**

因此，本次整改重点应当收敛到一件事：

**统一章节修订链路上的 budget owner，让 `resolveDocumentGenerationBudget(...)` 产出的预算成为 redirect 后 dedicated edit 的唯一高优先级请求预算来源。**

只要这一点不改，后续仍然会继续出现：

1. 前半段日志显示 `8192`
2. 真正起流又变成 `2048`
3. 最终在 `finish_reason=length` 处提前截断

这种“预算看起来已经抬上去了，但结果仍然不完整”的表面矛盾。
