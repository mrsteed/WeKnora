# 长文档生成任务通用修正模式完整技术方案

## 0. 方案审查结论

经审查，原方案的总体方向是正确的：它已经从“长文翻译专项修复”抽象到了“长文档生成任务通用治理”，并且选择了 P0 止损、P1 协议收敛、P2 任务化重构的渐进路线。这一路线符合当前系统状态，也能控制上线风险。

但原方案仍存在以下不完整之处：

1. 缺少明确的“什么时候走普通问答、什么时候走长文档任务”的判定标准。
2. 缺少任务状态机和批次状态机，导致 completed、partial、failed、cancelled 的流转边界不够清晰。
3. 数据模型偏概要，缺少幂等、租户隔离、输入快照、产物校验、任务参数和重试控制字段。
4. 接口设计只覆盖创建、查询、下载和重试，缺少取消、事件订阅、批次查看和产物元数据接口。
5. 缺少质量门禁、可观测性、灰度策略和兼容迁移方案。
6. P0、P1、P2 的工程落点还不够明确，研发实施时仍需要二次拆解。

本文档已在这些方面进行补强。修正后的结论是：本方案完整性、准确性和合理性可以满足技术评审要求，但建议严格按 P0、P1、P2 分阶段落地，不建议直接跳到 P2。

## 1. 文档定位

本文档基于当前目录下已有的分阶段方案与详细设计文档进行整合，但不再将问题限定为“长文翻译任务”。

本文档要解决的是更通用的问题：当系统需要处理全文翻译、长文总结、章节改写、结构化提取、报告生成、Markdown 导出等长文档生成任务时，如何避免继续复用当前在线问答链路，从而导致截断、误完成、脏数据入库、前后端状态失真和结果不可交付等问题。

因此，本文档定义的不是某一个业务场景的专项修复，而是一套可复用的长文档生成任务修正模式。

本文档的最终目标是形成一套通用工程模式，使系统后续面对任何“长输入、长输出、可交付文件”的生成任务时，都能复用同一套完成态、任务调度、批次执行、质量校验和产物交付机制。

## 2. 问题定义

## 2.1 现象层问题

当前系统将一类本质上需要分阶段处理、可中断恢复、最终需要文件化交付的长文档生成任务，错误地放进了单轮在线问答模型调用中承载，导致出现以下共性问题：

1. 模型输出因 length 或其他原因被截断，但系统仍将其当成 completed。
2. answer、thought、complete 事件职责不清，正文、推理和生命周期信息互相污染。
3. 前端只要收到 done 或 complete 就展示“已完成”，无法正确表达 partial、failed、cancelled。
4. 聊天历史知识库错误收录半成品、规划性文本或异常输出。
5. 结果只能停留在对话窗口中，无法稳定交付为文件产物。
6. 一旦中途失败，已生成内容难以复用，重试成本高。

## 2.2 根因层问题

这些问题的根因并不是某一种任务提示词写得不好，也不只是 token 不够，而是架构边界错误，主要体现在四个方面：

1. 完成语义错误：把“流结束”错误等同于“任务完成”。
2. 协议语义错误：把 answer、thought、complete 混用，导致前后端无法基于统一协议工作。
3. 存储语义错误：把所有输出都当作可索引、可沉淀、可复用的正式结果。
4. 任务承载错误：把长文档生成任务塞进适合短交互问答的在线 SSE 输出链路。

## 2.3 本方案适用范围

本方案适用于以下任务类型：

1. 全文翻译与双语对照生成。
2. 长文总结、提纲生成和章节摘要。
3. 文档改写、润色、重组和格式统一。
4. 从长文中提取结构化信息并导出 Markdown、JSON、CSV 或报告文件。
5. 基于知识文档生成审计报告、评审意见、技术方案或交付材料。

本方案不直接用于以下场景：

1. 极短回答的普通对话问答。
2. 不需要持久化中间结果和文件交付的一次性临时回答。

## 2.4 普通问答与长文档任务的判定边界

为了避免方案落地后仍然出现误路由，系统必须建立清晰的任务判定边界。

建议满足以下任一条件时，优先进入长文档生成任务链路：

1. 用户明确要求生成“完整文档”“完整报告”“完整 Markdown”“导出文件”“全文处理”。
2. 用户要求保留章节、目录、表格、图片说明、引用、脚注或其他原文结构。
3. 输入内容来自完整文档、多个 chunk、大量知识库片段或长上下文检索结果。
4. 预估输出 token 超过单次模型稳定输出窗口的 50%。
5. 任务需要分段处理、结果汇总、失败重试或后台执行。
6. 任务结果需要后续下载、归档、审计或作为独立知识资产管理。

建议满足以下条件时，仍可保留普通问答链路：

1. 用户只请求简短解释、局部问答或局部摘录。
2. 输出不需要文件化交付。
3. 失败后无需保留中间结果。
4. 结果不需要批次级质量校验。

该判定边界是本方案能否真正通用化的关键。如果入口仍然无法区分短问答和长任务，后续 P0 到 P2 的治理会被持续绕过。

## 3. 通用修正目标

长文档生成任务的通用修正目标应统一为以下五项：

1. 任何未完整完成的结果都不能再被伪装成 completed。
2. 任何 partial、failed、cancelled 结果都不能误入聊天历史知识库。
3. 前后端必须基于同一套 completion_status 和 finish_reason 语义工作。
4. 长文档生成任务必须具备分批执行、可恢复、可重试和可文件化交付的能力。
5. Chat 只负责发起任务、展示进度和查看结果，不再承载整篇正文。

## 4. 通用修正原则

## 4.1 结束不等于完成

所有长文档生成链路都必须遵守如下原则：

1. 流式输出结束，只代表当前会话生命周期结束。
2. 只有 completion_status=completed，才代表业务意义上的真正完成。
3. finish_reason 只能描述结束原因，不能代替 completion_status。

## 4.2 正文、思考、生命周期必须解耦

协议层必须把三类信息拆开：

1. answer 只负责正文增量。
2. thought 只负责推理或思考展示。
3. complete 只负责生命周期收尾和状态元数据。

## 4.3 入库必须有硬门禁

即使上层状态判断失误，最终入库链路也必须强制拦截不合格结果。任何长文档任务的索引或沉淀动作，都必须先经过完成态校验、结果质量校验和任务类型校验。

## 4.4 长任务必须任务化而不是聊天化

只要任务满足以下任一条件，就不应再走普通在线问答主链路：

1. 目标输出明显超过单次 completion 稳定窗口。
2. 结果需要文件交付。
3. 任务需要多轮续写、分批处理或失败恢复。
4. 任务需要结构化输入和结构化输出控制。

## 4.5 中间结果必须可恢复

长文档生成任务的中间结果不是临时噪音，而是执行资产。系统应支持：

1. 批次级落库。
2. 局部重试。
3. 部分结果可交付。
4. 汇总前可审计。

## 5. 通用修正模式总览

建议采用三阶段渐进修正模式。这个模式不只适用于翻译，也适用于任意长文档生成任务。

## 5.1 P0 止损修复

目标：先阻断错误完成和错误入库。

本阶段关注的是语义止血，而不是能力升级。

必须达成：

1. 截断输出不再视为 completed。
2. partial、failed、cancelled 不再入库。
3. 现有短问答主链路不明显回归。

## 5.2 P1 协议与状态机收敛

目标：统一后端完成态、事件协议、持久化字段和前端展示语义。

本阶段关注的是让系统说同一种语言，而不是先做异步任务平台。

必须达成：

1. answer、thought、complete 事件职责固定。
2. completion_status、finish_reason、failure_reason 成为正式模型字段。
3. 前端能正确展示 completed、partial、failed、cancelled。

## 5.3 P2 任务流重构

目标：把长文档生成任务从在线问答链路中拆出，建设成异步任务系统。

本阶段关注的是能力边界重构和文件交付。

必须达成：

1. 长任务不再依赖单次 SSE 正文输出。
2. 任务支持分批执行、重试、恢复和结果汇总。
3. 最终以 Markdown 或其他产物文件交付。

## 6. 通用架构方案

## 6.1 总体架构分层

建议把长文档生成任务统一抽象为四层：

1. 任务入口层：识别用户意图并创建任务。
2. 任务编排层：负责切分、调度、状态推进和失败恢复。
3. 内容执行层：按批次调用模型完成具体生成动作。
4. 产物交付层：负责文件汇总、存储、下载和结果回看。

## 6.2 推荐模块划分

建议引入以下通用模块，而不是为每类任务单独复制实现：

1. LongDocumentTaskService：创建任务、推进状态、汇总结果。
2. LongDocumentBatchPlanner：根据 token 预算和结构边界切分批次。
3. LongDocumentWorker：执行批次调用。
4. LongDocumentAssembler：按顺序汇总中间结果并生成最终文件。
5. LongDocumentArtifactService：管理产物存储、下载和元数据。
6. LongDocumentPromptStrategy：按任务类型注入专用 prompt 模板。

## 6.3 任务类型抽象

建议定义统一任务类型枚举，例如：

1. translation
2. summary
3. rewrite
4. extraction
5. report_generation
6. markdown_export

每种任务类型只改变以下可变部分：

1. prompt 模板。
2. 输出格式约束。
3. 批次汇总策略。
4. 结果质量校验规则。

任务框架本身不因具体任务类型变化而变化。

## 6.4 通用组件职责边界

为了避免后续实现时再次把业务逻辑散落到 Chat、Agent、Handler 和 MessageService 中，建议固定以下职责边界：

1. Chat Handler 只负责识别入口、创建任务和返回任务卡片，不直接执行长文档正文生成。
2. LongDocumentTaskService 只负责任务生命周期和状态推进，不直接拼 prompt。
3. LongDocumentBatchPlanner 只负责切分和批次规划，不直接调用模型。
4. LongDocumentPromptStrategy 只负责把 task_kind 转换为模型输入模板。
5. LongDocumentWorker 只负责执行单个批次并落库结果。
6. LongDocumentAssembler 只负责汇总批次结果并生成产物。
7. LongDocumentArtifactService 只负责产物元数据、存储路径、下载鉴权和校验。

这个拆分可以确保新增任务类型时只扩展策略，不复制整条任务链路。

## 6.5 与现有在线问答链路的关系

长文档任务链路不应替代现有在线问答链路，而是作为并行能力存在。

建议关系如下：

1. 普通 QA 继续走现有 Agent/Chat 流式链路。
2. 中等长度回答可在 P1 之后保留有限 continuation 能力。
3. 长文档生成请求进入 long document task 链路。
4. Chat 中只展示任务卡片、进度和下载入口。
5. 长文档产物如需进入知识库，应通过独立知识资产导入流程完成，而不是复用聊天历史索引。

## 7. P0 止损修复详细方案

## 7.1 completion metadata 最小扩展

在不立即变更数据库的前提下，优先在内存态、事件态和 handler 透传最小完成元数据。

建议统一扩展以下字段：

```go
CompletionStatus string `json:"completion_status,omitempty"`
FinishReason     string `json:"finish_reason,omitempty"`
IsPartial        bool   `json:"is_partial,omitempty"`
AllowIndexing    bool   `json:"allow_indexing,omitempty"`
AllowComplete    bool   `json:"allow_complete,omitempty"`
FailureReason    string `json:"failure_reason,omitempty"`
```

这些字段不是“翻译场景字段”，而是所有长文档生成任务都需要的通用执行语义。

## 7.2 截断识别规则

observe 层必须把以下场景区分开：

1. finish_reason=stop 且结果完整，可判定 completed。
2. finish_reason=length，应判定为 partial。
3. finish_reason=content_filter，应判定为 failed。
4. 用户主动终止，应判定为 cancelled。
5. 达到 max iterations 但无法证明完整性，应优先判定为 partial。

## 7.3 消息收尾门禁

completeAssistantMessage 或等价收尾逻辑必须收敛为以下行为：

1. AllowComplete=true 时才允许设置 IsCompleted=true。
2. AllowIndexing=true 时才允许进入知识库索引。
3. partial、failed、cancelled 结果允许保留消息内容，但不得伪装为正式完成内容。

## 7.4 知识库入库硬门禁

IndexMessageToKB 或等价索引入口建议增加统一 options：

```go
type MessageIndexOptions struct {
    CompletionStatus string
    FinishReason     string
    AllowIndexing    bool
    TaskKind         string
}
```

推荐门禁规则：

1. AllowIndexing=false 直接返回。
2. CompletionStatus!=completed 直接返回。
3. FinishReason=length 直接返回。
4. assistantAnswer 为空直接返回。
5. 长文档任务类型默认不写入聊天历史知识库，除非显式声明允许索引。
6. 命中规划性文本、模板性解释文本或中途续写提示语时直接返回。

## 7.5 临时缓解措施

为了降低轻度截断概率，可在 P0 同步做以下缓解，但必须明确它们不是根治方案：

1. 适度提升相关 Agent 的 max_completion_tokens。
2. 对明显长任务增加意图识别提示，提醒用户改走任务模式。
3. 对超长请求在入口处打告警日志，便于后续统计和治理。

## 8. P1 协议与状态机收敛详细方案

## 8.1 持久化模型增强

建议在 messages 表或等价消息持久化模型中新增：

1. completion_status
2. finish_reason
3. failure_reason

推荐状态取值：

1. pending
2. completed
3. partial
4. failed
5. cancelled

IsCompleted 可在过渡期保留，但新逻辑应以 completion_status 为准。

## 8.2 事件协议收敛

### 8.2.1 answer 事件

answer 事件只负责正文增量输出。

规则：

1. 所有正文 chunk 都必须走 answer。
2. answer 不负责宣布任务完成。
3. answer 事件只附带必要元数据，例如 source、event_id、chunk_index。

### 8.2.2 thought 事件

thought 事件只负责思考展示。

规则：

1. 只有显式推理内容才能走 thought。
2. 普通正文不得 fallback 到 thought。
3. thought 不影响 completion_status。

### 8.2.3 complete 事件

complete 事件只负责生命周期结束和状态汇报。

规则：

1. complete 不补正文。
2. complete 不回写 final answer 到 assistantMessage.Content。
3. complete 只携带 completion_status、finish_reason、failure_reason、duration、steps 等元数据。

## 8.3 续写策略标准化

对在线链路仍允许继续承载的中等长度任务，可定义统一续写规则：

1. 当 finish_reason=length 且当前正文非空时，可进入 continuation round。
2. continuation prompt 固定要求“从中断位置继续，不要重复前文，保持相同格式”。
3. 续写轮次必须有限制，例如最多 3 轮。
4. 超过上限仍未完成时，统一收口为 partial。

续写策略只是一种兼容机制，不应成为长任务的主路径。

## 8.4 前端状态机修正

前端消息模型建议统一增加：

1. completion_status
2. finish_reason
3. is_partial
4. failure_reason

前端展示规则建议统一为：

1. 只有 complete 且 completion_status=completed 时，才显示已完成。
2. complete 且 completion_status=partial 时，显示部分完成或输出被截断。
3. complete 且 completion_status=failed 时，显示失败。
4. complete 且 completion_status=cancelled 时，显示已停止。
5. answer.done 不再直接等于消息完成。

## 8.5 统一状态机定义

P1 必须补齐统一状态机，否则 P2 的任务状态会继续继承旧的 completed 误判问题。

### 8.5.1 消息完成状态机

消息 completion_status 建议流转如下：

1. pending：消息已创建，但尚未开始稳定输出。
2. streaming：正在输出正文或思考过程。
3. completed：正文输出完整，且 finish_reason、质量校验和业务规则均允许完成。
4. partial：有部分可展示内容，但完整性无法保证。
5. failed：生成失败或被安全策略阻断。
6. cancelled：用户主动停止或系统主动取消。

推荐流转规则：

1. pending 可以进入 streaming、failed、cancelled。
2. streaming 可以进入 completed、partial、failed、cancelled。
3. completed、failed、cancelled 为终态。
4. partial 是可交付但不完整的终态，也可以通过“重试”派生新任务。

### 8.5.2 finish_reason 与 completion_status 的映射

建议默认映射如下：

| finish_reason | 默认 completion_status | 是否允许索引 | 说明 |
| --- | --- | --- | --- |
| stop | completed | 是 | 仍需通过质量门禁，stop 不自动证明长任务完整 |
| length | partial | 否 | 输出窗口耗尽，默认不完整 |
| content_filter | failed | 否 | 安全策略阻断 |
| cancelled | cancelled | 否 | 用户或系统主动取消 |
| max_iterations | partial | 否 | Agent 循环耗尽但完整性不可证明 |
| tool_error | failed | 否 | 工具或外部依赖失败 |

对于普通短问答，stop 可以直接映射为 completed。对于长文档任务，stop 只是批次完成信号，最终任务 completed 还必须等待所有批次和汇总质量门禁通过。

## 9. P2 任务流重构详细方案

## 9.1 任务入口识别

当请求满足以下条件时，系统应优先识别为长文档生成任务，而不是普通问答：

1. 用户明确要求“输出完整文档”或“导出文件”。
2. 输入内容已经是整篇文档、整本手册、整批章节或大量 chunk。
3. 用户要求保留结构、目录、表格、图注或章节层级。
4. 目标结果预期远超普通聊天窗口可承载范围。

## 9.2 通用任务模型

建议新增统一任务表 long_document_tasks，至少包含：

1. id
2. tenant_id
3. session_id
4. task_kind
5. source_ref
6. output_format
7. status
8. total_batches
9. completed_batches

## 9.2.1 任务状态机

long_document_tasks.status 建议流转如下：

1. pending：任务已创建，尚未调度。
2. running：至少一个批次已开始执行。
3. assembling：批次已完成，正在生成最终产物。
4. completed：所有批次和产物质量门禁通过。
5. partial：部分批次失败，但已有可交付产物。
6. failed：任务无法生成可用产物。
7. cancelled：用户或系统取消。

终态包括 completed、partial、failed、cancelled。

建议新增批次表 long_document_task_batches，至少包含：

1. id
2. task_id
3. batch_no
4. input_snapshot
5. output_payload
6. status
7. retry_count
8. error_message
9. created_at
10. updated_at

建议进一步补充以下工程字段：

1. input_token_estimate
2. output_token_estimate
3. model_name
4. prompt_version
5. started_at
6. completed_at
7. checksum
8. quality_status
9. parent_batch_id

其中 parent_batch_id 用于支持超长批次自动再切分，避免重试时丢失原始批次关系。

## 9.2.2 批次状态机

long_document_task_batches.status 建议流转如下：

1. pending：批次已规划，尚未执行。
2. running：批次正在执行。
3. completed：批次输出通过质量门禁。
4. retrying：批次准备重试。
5. split：批次过大，已拆分为子批次。
6. failed：批次达到重试上限仍失败。
7. skipped：任务取消或上游条件不满足。

任务最终状态应由批次状态和产物状态共同决定，而不是由任意单次模型调用决定。

## 9.3 批次切分策略

批次切分应同时考虑 token 预算和文档结构边界。

建议规则：

1. 优先按章节、小节、表格、附录、参考文献边界切分。
2. 单批大小应受稳定输出窗口控制。
3. 图表说明和关联正文尽量同批。
4. 对超长单章支持二次再切分。

## 9.4 批次执行策略

每个批次都应执行以下通用流程：

1. 读取输入快照。
2. 根据 task_kind 选择对应 prompt strategy。
3. 调用模型生成批次结果。
4. 落库 output_payload。
5. 进行批次级质量校验。
6. 更新任务进度。

批次执行必须满足幂等要求：同一个 batch_id 重复执行时，不得重复追加输出，不得重复计数，不得覆盖已完成且校验通过的批次结果。建议通过 batch status、checksum、attempt number 和 idempotency_key 共同控制。

## 9.5 汇总与产物生成策略

所有批次执行完成后，应由 Assembler 统一汇总：

1. 按 batch_no 顺序拼接结果。
2. 根据输出格式渲染为 Markdown、JSON、CSV 或报告模板。
3. 追加任务元数据和缺失说明。
4. 生成最终文件。
5. 写入 artifact_path。

汇总阶段必须执行最终质量门禁：

1. 批次数量与规划数量一致。
2. completed 批次按 batch_no 连续可拼接。
3. 输出文件非空且大小合理。
4. 文件 checksum 已生成。
5. Markdown、JSON、CSV 等目标格式可被基础解析器校验。
6. partial 产物必须包含缺失段落说明，不能伪装为完整产物。

当存在失败批次但整体仍可交付时：

1. 任务状态记为 partial。
2. 最终文件中明确标识缺失区段。
3. 前端允许下载 partial 产物。

## 9.6 调度与恢复策略

推荐复用现有任务队列体系，例如 Asynq。

初版建议：

1. 单任务串行批次执行。
2. 多任务可并发，但受租户配额和资源限流控制。
3. 每批次支持有限重试。
4. 超时、长度截断或临时模型错误时，可缩小批次后二次执行。

## 11. 前端交互方案

## 11.1 Chat 入口行为

Chat 在识别到长文档生成任务后，应执行以下统一行为：

1. 不直接等待整篇正文流回。
2. 创建任务。
3. 插入一条“任务已创建”的 assistant 消息。
4. 展示任务进度卡片。

## 11.2 任务进度卡片

建议统一展示以下信息：

1. task_kind
2. 当前状态
3. 批次进度
4. 已完成数量
5. 失败批次数
6. 最近错误原因
7. 下载按钮

## 11.3 结果查看与失败恢复

建议支持：

1. 下载 completed 产物。
2. 下载 partial 产物。
3. 查看失败批次和错误原因。
4. 重试失败批次。

## 12. 索引与知识沉淀策略

长文档生成任务与普通对话问答的索引策略必须分离。

建议规则：

1. 普通聊天知识库只索引稳定、简洁、可复用的问答结果。
2. 长文档任务默认不写入聊天历史知识库。
3. 若业务确需沉淀，应以最终 artifact 或结构化摘要作为独立知识资产入库，而不是直接索引聊天正文。
4. partial 结果不得作为正式知识资产入库，除非显式标记为草稿并进入专门草稿域。

## 12.1 质量门禁策略

长文档任务必须建立任务级和批次级质量门禁。没有通过门禁的内容，不能进入 completed，也不能作为正式知识资产沉淀。

批次级质量门禁建议包含：

1. 输出非空。
2. 输出没有明显模型自述、规划性文本或错误提示。
3. 输出格式符合 task_kind 对应要求。
4. 未命中安全阻断或内容过滤。
5. token 截断时不得判定 completed。

任务级质量门禁建议包含：

1. 所有必须批次均 completed。
2. 汇总产物可被目标格式解析。
3. 产物 checksum 和文件大小有效。
4. partial 任务显式标识缺失内容。
5. 产物下载权限与租户权限一致。

## 12.2 可观测性策略

为了判断方案是否真正解决问题，需要增加以下观测指标：

1. long_document_task_created_total
2. long_document_task_completed_total
3. long_document_task_partial_total
4. long_document_task_failed_total
5. long_document_batch_retry_total
6. long_document_batch_split_total
7. long_document_artifact_generated_total
8. chat_long_task_misrouted_total
9. message_indexing_skipped_total
10. finish_reason_length_total

关键日志必须包含 task_id、batch_id、tenant_id、task_kind、completion_status、finish_reason、retry_count 和 artifact_id，便于线上排查。

## 12.3 安全与权限策略

长文档任务会生成可下载文件，权限边界必须强于普通聊天消息。

建议规则：

1. 创建任务时校验用户对 source_ref 的读取权限。
2. 查询任务时校验 tenant_id、session_id 或任务归属。
3. 下载产物时校验 artifact 与 task 的绑定关系。
4. partial 产物和 failed 任务的错误信息不得泄露系统 prompt、密钥或内部路径。
5. 产物支持过期时间或显式清理策略。

## 13. 风险与控制策略

## 13.1 任务误判风险

风险：应走任务模式的请求仍被错误送入普通聊天链路。

控制：

1. 增加入口识别规则。
2. 对超长任务加告警和观测埋点。
3. 对特定 Agent 或特定场景强制启用任务模式。

## 13.2 批次边界破坏结构连续性

风险：切分不合理导致最终文档结构断裂。

控制：

1. 以结构边界优先切分。
2. 允许批次附带前文摘要作为上下文。
3. 在汇总阶段执行结构一致性校验。

## 13.3 中间结果不可恢复

风险：中途失败后全部重做。

控制：

1. 批次级落库。
2. 有限重试。
3. 支持 partial 结果导出。

## 13.4 用户误解任务状态

风险：用户看到 done 就误以为全文已完成。

控制：

1. 前端只以 completion_status 展示完成态。
2. completed、partial、failed、cancelled 文案必须区分。
3. 只有产物可下载且状态正确时才展示最终完成。

## 14. 测试与验收方案

## 14.1 P0 验收

1. finish_reason=length 的输出不会被标记为 completed。
2. partial、failed、cancelled 不会再进入聊天历史知识库。
3. 普通短问答无明显回归。

## 14.2 P1 验收

1. answer、thought、complete 无职责重叠。
2. 前后端统一展示 completion_status。
3. complete 阶段不再补正文。

## 14.3 P2 验收

1. 长文档生成任务不再依赖单次在线问答全文输出。
2. 任务支持批次执行、失败恢复和文件交付。
3. partial 任务可保留中间成果并可下载。
4. Chat 只展示任务进度和结果入口，不再承载整篇正文。

## 14.4 测试矩阵

建议按以下矩阵补齐测试：

| 测试类型 | 覆盖目标 | 必测场景 |
| --- | --- | --- |
| 单元测试 | 状态判定 | stop、length、content_filter、cancelled、max_iterations |
| 单元测试 | 索引门禁 | completed 可入库，partial/failed/cancelled 不入库 |
| 单元测试 | 批次规划 | 按结构边界切分、超长批次再切分 |
| 集成测试 | 任务执行 | pending 到 running 到 completed |
| 集成测试 | 失败恢复 | 批次失败、重试、partial 产物生成 |
| 前端测试 | 状态展示 | completed、partial、failed、cancelled 卡片展示 |
| 权限测试 | 产物下载 | 越权下载被拒绝 |
| 回归测试 | 普通问答 | 短问答流式输出和历史索引不回归 |

## 14.5 灰度与回滚策略

建议通过配置开关逐步启用：

1. enable_completion_metadata
2. enable_message_index_guard
3. enable_long_document_task_router
4. enable_long_document_task_worker
5. enable_long_document_artifact_download

回滚策略：

1. P0 回滚时只关闭新门禁开关，保留日志。
2. P1 回滚时前端可退回 is_completed 兼容字段，但后端仍保留 completion_status。
3. P2 回滚时关闭任务路由，新任务回到普通链路，但已创建任务继续由 worker 消费或标记 cancelled。

不建议通过删除数据或回滚迁移来处理普通故障，除非字段设计出现严重错误。

## 14.6 数据迁移与兼容策略

P1 新增消息状态字段时建议采用兼容迁移：

1. 新增字段允许为空。
2. 旧消息读取时根据 is_completed 映射 completion_status。
3. 新消息写入时同时写 completion_status 和 is_completed。
4. 前端优先读取 completion_status，不存在时回退 is_completed。
5. 稳定后再评估是否弱化 is_completed。

P2 新增任务表时不影响现有消息表，避免把任务产物和聊天消息强耦合。

## 15. 推荐实施顺序

推荐实施顺序仍然是：

1. 先做 P0 止损修复。
2. 再做 P1 协议与状态机收敛。
3. 最后做 P2 任务流重构。

原因很明确：

1. 不先修 completion 语义，后续任何重构都可能继续制造脏数据。
2. 不先修协议，前后端状态语义会继续错位。
3. 不先收敛边界，异步任务系统也可能复用旧错误语义。

## 15.1 工程落地清单

P0 建议落地项：

1. 扩展 AgentFinalAnswerData 和 AgentCompleteData 的 completion metadata。
2. 在 analyzeResponse 或等价逻辑中识别 length、content_filter、max_iterations。
3. 调整 completeAssistantMessage，不再无条件设置 IsCompleted=true。
4. 为 IndexMessageToKB 增加 MessageIndexOptions 和硬门禁。
5. 增加 P0 单测和基础回归测试。

P1 建议落地项：

1. messages 表新增 completion_status、finish_reason、failure_reason。
2. 修正 answer、thought、complete 事件职责。
3. 修正前端 handleAgentChunk 和 AgentStreamDisplay 的完成态判断。
4. 增加有限 continuation 机制和上限控制。
5. 增加历史消息兼容读取逻辑。

P2 建议落地项：

1. 新增长文档任务路由和 task_kind 判定。
2. 新增 long_document_tasks、long_document_task_batches、long_document_artifacts。
3. 新增 BatchPlanner、Worker、Assembler、ArtifactService。
4. 新增创建、查询、取消、批次查看、下载和事件订阅接口。
5. 新增前端任务进度卡片和产物下载入口。
6. 增加任务级质量门禁、观测指标和灰度开关。

## 15.2 最小可行版本建议

如果需要控制首版范围，建议 P2 的最小可行版本只支持：

1. task_kind=markdown_export 或 report_generation 中的一种。
2. output_format=markdown。
3. 单任务串行执行。
4. completed 和 failed 两种主状态，partial 可先只在后端保留。
5. 支持创建、查询、下载三个接口。

在最小可行版本稳定后，再逐步增加批次重试、partial 下载、事件订阅和更多 task_kind。

## 16. 结论

长文档生成任务的问题，本质上不是某一个任务场景的专项缺陷，而是系统把“需要任务化承载的长流程生成”误当成了“适合在线问答承载的短响应交互”。

因此，正确解法不是继续围绕某一种任务类型做提示词补丁，而是建立一套通用修正模式：先修 completion 语义，再修协议与状态机，最后完成任务化和文件化交付。

一旦这套模式建立起来，后续无论是全文翻译、长文总结、报告生成、结构化提取还是 Markdown 导出，都可以复用同一套治理框架，而不必为每个任务单独重走一遍修复过程。
