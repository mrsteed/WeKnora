# 长文档生成与续写状态收口及 Markdown 修正审查方案

## 1. 审查结论

本次“基于基线版本续写后，正文已经完成，但界面仍显示继续生成/部分完成，且 Markdown 格式没有被修正”的问题，不是单点缺陷，而是长文档生成链路内部存在三类结构性不一致：

1. 不同长文档路径对“是否完成全文”的判定标准不一致。
2. 不同长文档路径对 Markdown 质量修正的层级不一致。
3. 服务层、Handler 落库层、长文档任务转发层都在重复计算 `document_generation_status` 和 `auto_continue_next`，导致终态被后置步骤覆盖。

结合当前代码，可以得出明确结论：

1. 首轮完整长文档生成和带 generation run 的分批续写，已经具备“章节级修正 + 整文完整性校验 + artifact-first 收口”的较完整逻辑。
2. 无 generation run 的知识库续写路径仍然停留在“只修当前 delta、只认 completion marker、没有完整文档收口校验”的简化实现上。
3. 专用文档编辑路径几乎不做整文级 Markdown 修正，只把原始流式结果直接作为最终答案发出，后续 artifact 注册阶段最多只会打质量告警，不会把修正后的快照反写到最终展示结果。
4. 本次日志对应的问题，直接触发点发生在服务层 completion summary 阶段，而不是 artifact 注册后的覆盖阶段；不过 Handler 和长文档任务层仍然保留后置重算 `document_generation_status` 和 `auto_continue_next` 的能力，会把这种错误状态继续放大到最终展示层。

因此，本问题的根因不是“前端显示错了”，而是：

**后端没有为所有长文档生成/续写路径提供统一的完成判定、Markdown 修正和最终状态收口逻辑，前端只是如实显示了一个被多次覆盖后的不一致状态。**

## 2. 当前长文档链路梳理

当前与长文档生成、续写、文档编辑相关的后端路径至少有以下几类：

1. 完整长文档首轮生成
   - 专用路径：`runDedicatedFullDocumentGenerationPath(...)`
   - 知识库路径：`runKnowledgeGroundedFullDocumentGenerationPath(...)`

2. 长文档续写
   - 无 generation run 的知识库续写：`runKnowledgeGroundedDocumentContinuationPath(...)`
   - 有 generation run 的知识库续写：`runKnowledgeGroundedDocumentContinuationRunPath(...)`

3. 基线文档编辑
   - 专用 patch/修订路径：`runDedicatedDocumentEditPath(...)`

这些路径虽然都属于“长文档生成/续写/修订”，但当前并没有共享同一套收尾逻辑。

## 2.1 基于本次日志的实际执行链路

结合本次日志，可以把实际执行顺序还原为以下链路：

1. 大模型在 OpenAI 兼容层自然结束：`finish=stop`，不是超时、取消或 tool call 中断。
2. 服务层立即输出 completion summary：
   - `completion_status: partial`
   - `finish_reason: stop`
   - `document_generation_status: continuing`
   - `auto_continue_next: true`
   - `artifact_id: ""`
   - `generation_run_id: ""`
   - `outline_sections: 9`
   - `runtime_sections: 1`
   - `quality_issues: [markdown_heading_normalized]`
3. 紧接着进入 `completeAssistantMessageInPlace(...)`，先把 message 以 `partial` 状态落库，再注册 artifact。
4. artifact 首次查询出现 `record not found`，这是首次按 `source_message_id` 注册新 revision 的正常现象，不是异常根因。
5. artifact 注册成功后，persistence summary 仍然是：
   - `completion_status: partial`
   - `finish_reason: stop`
   - `document_generation_status: continuing`
   - `artifact_registered: true`
   - `artifact_revision_no: 3`
   - `generation_run_id: ""`

这组日志能够直接证明三件事：

1. 这次请求命中的不是 `runKnowledgeGroundedDocumentContinuationRunPath(...)`，而是无 generation run 的 `runKnowledgeGroundedDocumentContinuationPath(...)`。
2. 当前错误状态在 artifact 注册之前就已经形成，因为 completion summary 和 persistence summary 的核心状态完全一致。
3. 本次并非“完全没有做 Markdown 处理”，而是只记录到了 `markdown_heading_normalized` 这一类轻量归一化结果，但没有完成整文级收口修正。

## 3. 直接问题定位

## 3.1 状态显示不正确的直接原因

### 3.1.1 无 generation run 的知识库续写路径在自然停止后，仍然主要依赖 completion marker 判定是否完成全文

在 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runKnowledgeGroundedDocumentContinuationPath(...)` 中，初始状态被设置为：

1. `completionStatus = partial`
2. `finishReason = continuation_pending`
3. `documentGenerationStatus = continuing`

随后流式结果完成后，只做了两步：

1. 对本轮 delta 结果执行 `applyGeneratedDocumentMarkdownQualityGate(...)`
2. 仅当输出中包含 `<!-- document_complete -->` 时，把状态改成 completed

本次日志正好落在这个分支的典型行为上：

1. 上游流结束原因为 `finish=stop`，说明模型本轮自然结束。
2. 但由于输出中没有 completion marker，服务层仍把结果保留为：
   - `completion_status: partial`
   - `document_generation_status: continuing`
   - `auto_continue_next: true`

这意味着：

1. 在 simple continuation 路径中，`finish=stop` 只代表“本轮自然结束”，不代表“整篇文档已经完成”。
2. 只要模型没有输出 completion marker，即使正文实际上已经写到了总结与展望，也不会自动转成 completed。
2. 该路径没有像完整文档和 generation run 续写路径那样，对“合并后的完整快照”执行完整性收口判断。

这与截图中的现象完全吻合：

1. 正文已经写到“总结与展望”。
2. 界面仍显示“当前轮内容已生成，系统可继续自动续写剩余章节”。

也就是说，本次状态不正确，首先发生在服务层，而不是前端层。

### 3.1.2 generation run 续写路径和完整长文档路径已经有更完整的终态收口

相比之下：

1. [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runKnowledgeGroundedDocumentContinuationRunPath(...)`
2. [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runKnowledgeGroundedFullDocumentGenerationPath(...)`
3. [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runDedicatedFullDocumentGenerationPath(...)`

都在 finalize 阶段执行了：

1. `applyFullDocumentArtifactQualityGate(...)`
2. `applyArtifactFirstFullDocumentIntegrityOutcome(...)`

也就是说，这些路径会对“整篇文档快照”做完整性与质量校验，而不是只依赖 completion marker。

这说明：

**问题不是系统完全没有正确的长文档收尾机制，而是无 generation run 的 continuation 路径没有接入同一套收尾机制。**

## 3.2 状态会在服务层算完后，再被后置链路继续放大

### 3.2.1 本次日志中的 continuing 首发于服务层，但 Handler 仍保留 artifact 覆盖能力

在 [internal/handler/session/qa.go](internal/handler/session/qa.go) 的 `emitAssistantCompleteEvent(...)` 中，逻辑顺序是：

1. 先根据 `options.DocumentGenerationStatus` 生成 `AutoContinueNext`
2. 如果存在 artifact，则再执行：
   - `completeData.DocumentGenerationStatus = artifact.DocumentGenerationStatus`
   - 重新计算 `AutoContinueNext`
   - 重新计算 `AutoContinueReason`
   - 再执行 `applyChatDocumentContinuationDecision(...)`

这意味着：

**只要 artifact 注册结果与服务层 completion 计算结果不一致，前端最终接收到的就是 artifact 侧的状态，而不是服务层最初发出的状态。**

但需要根据本次日志修正文档中的判断边界：

1. 本次样例里，completion summary 与 persistence summary 都是 `partial + continuing`。
2. 这说明当前错误状态不是由 artifact 注册后“覆盖错了”才产生的。
3. 更准确的说法应是：artifact 覆盖能力是现有架构中的放大器，不是这次日志样例的首发点。

### 3.2.2 长文档任务层又会再重算一次 `auto_continue_next`

在 [internal/application/service/long_document_task.go](internal/application/service/long_document_task.go) 中，`populateLongDocumentContinuationPayload(...)` 会再次基于：

1. `document_generation_status`
2. `finish_reason`
3. `failure_reason`
4. `autoContinueRound`

重新构建 continuation decision，并写入：

1. `auto_continue_next`
2. `can_auto_continue`
3. `recommended_request`

也就是说，即使前面某层已经明确给出终态，这一层仍然会再次推导继续态。

这会带来一个典型问题：

1. 如果某个中间层错误地把 `document_generation_status` 保留成 `continuing`
2. 那么后面两层都会继续推导出 `auto_continue_next=true`
3. 最终前端就稳定显示“可继续自动续写剩余章节”

### 3.2.3 前端状态展示逻辑本身没有明显问题

前端相关判断主要在：

1. [frontend/src/views/chat/utils/documentCompletion.js](frontend/src/views/chat/utils/documentCompletion.js)
2. [frontend/src/views/chat/index.vue](frontend/src/views/chat/index.vue)
3. [frontend/src/views/chat/components/botmsg.vue](frontend/src/views/chat/components/botmsg.vue)

其中 `isDocumentCompletionContinuing(...)` 的逻辑非常直接：

1. `document_generation_status === continuing`
2. 且 `auto_continue_next === true`

满足时就展示“当前轮内容已生成，系统可继续自动续写剩余章节”。

因此这次前端只是忠实展示了错误的后端状态，不是根因。

## 3.3 Markdown 并非完全未修正，而是只做了轻量归一化，没有完成整文级修正

本次日志中的 `quality_issues: [markdown_heading_normalized]` 已经说明：

1. 系统确实检测到了 Markdown 标题格式问题。
2. 系统也确实执行了“标题补空格、段落补换行”这一类轻量归一化。
3. 前端提示“检测到 Markdown 标题格式不规范，系统已自动补齐标题空格和段落换行”与后端质量问题码是一致的。

因此，本次问题更准确的表述不是“完全没有修正”，而是：

**只做了局部规范化，没有把“修正后的完整快照”作为权威终稿重新收口。**

### 3.3.1 无 generation run 的知识库续写只修当前 delta，不修合并后的完整快照

在 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runKnowledgeGroundedDocumentContinuationPath(...)` 中，当前只对：

1. 本轮 `finalContent`

执行：

1. `applyGeneratedDocumentMarkdownQualityGate(...)`

但这里的 `finalContent` 只是本轮新增 delta，不是“基线 artifact + 新增 delta”合并后的完整文档。

从本次日志看，`quality_issues: [markdown_heading_normalized]` 就是在这个层级产生的。也就是说：

1. 当前路径会对本轮 delta 做一定程度的标题归一化。
2. 但归一化结果没有提升到“整文合并后的最终快照”层级重新校验。
3. 因此用户仍可能看到章节拼接边界错位、标题层级衔接异常，或者 artifact 卡片持续保留“已自动修复部分标题格式”的提示。

这会导致两个问题：

1. 基线末尾与新增正文的拼接边界不会被整文质量闸校验。
2. 整文层级、章节闭合、标题衔接、尾部结构不一致等问题无法被修正。

而 generation run 续写路径使用的是：

1. `completedSnapshot.String()` 作为完整快照
2. 再跑 `applyFullDocumentArtifactQualityGate(...)`

这是正确层级。当前 simple continuation 路径缺的正是这一层。

### 3.3.2 专用文档编辑路径几乎没有 Markdown 修正

在 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 的 `runDedicatedDocumentEditPath(...)` 和 `consumeDedicatedDocumentEditStream(...)` 中：

1. 流式输出被直接拼接到 `finalContent`
2. 结束后直接发 `AgentComplete`
3. 没有调用 `applyGeneratedDocumentMarkdownQualityGate(...)`
4. 也没有调用 `applyFullDocumentArtifactQualityGate(...)`

这意味着：

**文档编辑路径对最终返回给用户的正文，基本没有任何格式修正。**

### 3.3.3 artifact 注册阶段只做“归一化和告警”，不做真正的修复闭环

在 [internal/application/service/chat_document_artifact.go](internal/application/service/chat_document_artifact.go) 中：

1. `prepareChatDocumentArtifactSnapshot(...)` 会做基础归一化：
   - 标题补空格
   - 修复部分 Markdown heading
   - 关闭未闭合代码块
2. `evaluateArtifactMarkdownQuality(...)` 只做验证：
   - `validateGeneratedDocumentMarkdown(...)`
   - 若失败，则把 `DocumentGenerationStatus` 调整为 `needs_review`
   - 并打 `quality_issues`

它不会：

1. 发起修复模型调用
2. 把修复后的快照再回写到 message 的最终展示正文
3. 把“修复后的 artifact 快照”重新提升为权威 final answer

因此这条链路的效果是：

**发现格式问题并降级状态，但不能真正修好并回写给用户。**

## 4. 根因归纳

综合以上代码，当前长文档生成和续写逻辑存在以下四个根因。

### 4.1 根因一：长文档 continuation 路径分裂，收尾能力不一致

当前至少存在三套不同的收尾逻辑：

1. 完整长文档首轮生成
2. generation run 续写
3. simple continuation

其中第 3 套没有接入“artifact-first 完整性收口”。

### 4.2 根因二：状态收口没有单一可信源，但本次样例的首发点在服务层

当前状态在以下层级都可能被改写：

1. 服务层 completion 生成
2. Handler persistence 后 artifact 回写
3. 长文档任务转发层 continuation decision 再计算
4. 前端历史消息重建逻辑

没有“唯一可信终态来源”，自然会出现 completed 被继续态覆盖的问题。

结合本次日志，还需要补充一个更精确的判断：

1. 当前样例里，错误状态首先由 simple continuation 服务层生成。
2. 后续 persistence、artifact、task payload 只是继续沿用了这个 continuing 状态。
3. 因此整改优先级应先修服务层 finalize，再收紧后置覆盖能力。

### 4.3 根因三：Markdown 修正只覆盖了部分路径、部分层级

当前存在三个层级的格式处理：

1. section 级修正
2. delta 文本级修正
3. artifact 快照级校验

但并没有统一为：

**所有长文档路径都必须在“完整快照”层执行一次最终 Markdown 修正与完整性校验。**

### 4.4 根因四：artifact 注册承担了“告警”和“状态降级”，却没有承担“权威结果修正回写”

artifact 注册当前更像是一个审计和落库步骤，而不是最终结果规范化步骤。它可以把状态降级为 `needs_review`，但不能把修复后的文档提升成最终用户可见正文。

## 5. 整改目标

彻底修复该问题，目标应统一为：

1. 无论首次完整生成还是基线续写，最终是否完成全文必须由同一套收尾规则决定。
2. 所有长文档路径在结束前都必须对“完整文档快照”执行一次最终 Markdown 修正与校验。
3. `document_generation_status`、`auto_continue_next`、`next_action` 只能由一个权威阶段产出，后续层只允许透传，不允许再次推导覆盖。
4. 若最终修复后的快照可用，应优先以该快照作为前端展示和 artifact 持久化的共同权威版本。

## 6. 完整改造方案

## 6.1 P0：统一长文档终态收口为一套共享逻辑

建议抽出统一的长文档 finalize 管线，供以下路径共用：

1. `runDedicatedFullDocumentGenerationPath(...)`
2. `runKnowledgeGroundedFullDocumentGenerationPath(...)`
3. `runKnowledgeGroundedDocumentContinuationRunPath(...)`
4. `runKnowledgeGroundedDocumentContinuationPath(...)`
5. 必要时也供 `runDedicatedDocumentEditPath(...)` 的长文档修订模式复用

统一 finalize 至少包含以下步骤：

1. 生成“完整文档候选快照”
   - 首轮生成：直接使用整篇正文
   - 续写：使用 `BaseArtifact.ContentSnapshot + delta` 合并结果
   - revision/patch：使用 patch merge 后的新快照

2. 对完整快照执行统一修正
   - `normalizeFullDocumentFinalAnswer(...)`
   - `applyGeneratedDocumentMarkdownQualityGate(...)`
   - 若有章节规划，再执行 `applyArtifactFirstFullDocumentIntegrityOutcome(...)`

3. 统一判定终态
   - `completed`
   - `continuing`
   - `needs_review`
   - `blocked`

4. 统一生成 continuation decision
   - `auto_continue_next`
   - `next_action`
   - `recommended_request`

5. 统一产出 authoritative final answer
   - 前端展示用
   - artifact 持久化用
   - completion event 用

当前最需要优先修正的是：

**把 simple continuation 路径接入与 generation run 续写相同的 artifact-first finalize 管线。**

## 6.2 P0：无 generation run 的知识库续写不再只认 completion marker

建议重点改造 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 中的 `runKnowledgeGroundedDocumentContinuationPath(...)`。

当前行为：

1. 只修 delta
2. 只看 completion marker
3. 不做整文完整性判断

建议改为：

1. 用 `mergeChatDocumentContinuation(baseSnapshot, delta)` 生成合并后的完整候选快照
2. 对该完整候选快照执行 `applyFullDocumentArtifactQualityGate(...)`
3. 如果存在 outline 或可从 artifact 提取 outline，再执行 `applyArtifactFirstFullDocumentIntegrityOutcome(...)`
4. 即使模型没有输出 completion marker，只要本轮是自然停止 `finish=stop` 且完整性判断通过，也应收敛为 completed
5. completion marker 只作为“强完成信号”，不再是“唯一完成信号”

## 6.3 P0：统一状态权威源，禁止后置层重复推导覆盖

建议规定：

1. 服务层 finalize 产出的 `DocumentGenerationStatus`、`AutoContinueNext`、`NextAction` 是唯一权威结果。
2. Handler 在 [internal/handler/session/qa.go](internal/handler/session/qa.go) 的 `emitAssistantCompleteEvent(...)` 中：
   - 仅当 completion payload 缺字段时，才允许用 artifact 补齐
   - 不允许 artifact 无条件覆盖 completion payload 的终态
3. 长文档任务层在 [internal/application/service/long_document_task.go](internal/application/service/long_document_task.go) 的 `populateLongDocumentContinuationPayload(...)` 中：
   - 仅在 payload 中不存在 `auto_continue_next` / `next_action` / `recommended_request` 时才补算
   - 不允许再次覆盖明确给出的终态

否则“服务层算 completed，末端又重算为 continuing”的问题会持续出现。

## 6.4 P0：Markdown 修正必须在完整快照层完成，而不是只修 delta

建议把 Markdown 修正明确拆成两层：

1. 章节输出修正
   - 继续保留 `applyGeneratedSectionMarkdownQualityGate(...)`
2. 整文最终修正
   - 所有长文档路径结束前都必须对“完整文档快照”执行 `applyGeneratedDocumentMarkdownQualityGate(...)`

尤其是续写场景，必须避免以下错误用法：

1. 只修本轮 delta
2. 直接把未经整文修正的 delta 当最终结果发出
3. 再在 artifact 注册阶段被动降级为 `needs_review`

正确顺序应为：

1. merge base + delta
2. 修整文
3. 校整文
4. 再决定状态
5. 再发 completion

## 6.5 P0：专用文档编辑路径补齐格式修正和权威结果输出

建议改造 [internal/application/service/session_agent_qa.go](internal/application/service/session_agent_qa.go) 中的 `runDedicatedDocumentEditPath(...)` / `consumeDedicatedDocumentEditStream(...)`：

1. 对最终输出的 patch 或章节片段执行 `normalizeGeneratedMarkdown(...)`
2. 若是整章/整段 Markdown 输出，执行相应的 repair quality gate
3. 当 artifact 已成功注册并生成更规范的快照时，允许以 artifact 的规范化内容作为最终权威结果回写 completion
4. 对于长文档 revision 场景，不应只停留在“发现问题 -> needs_review”，应优先尝试修复后再决定是否 needs_review

## 6.6 P1：弱化 simple continuation，优先统一到 generation run 续写模型

从当前实现质量看：

1. generation run 续写路径更完整
2. simple continuation 路径逻辑简化过度

建议中期目标是：

1. 当显式 base artifact 进入知识库续写时，优先创建或补齐 generation run
2. 后续统一走 `runKnowledgeGroundedDocumentContinuationRunPath(...)`
3. simple continuation 只保留为极小范围的兜底路径

这样可以减少分叉逻辑。

## 6.7 P1：artifact 质量闸从“只告警”提升为“修复优先，告警兜底”

当前 [internal/application/service/chat_document_artifact.go](internal/application/service/chat_document_artifact.go) 中的 `evaluateArtifactMarkdownQuality(...)` 只做校验，不做修复。

建议改造为：

1. 若 snapshot 存在可修复的 Markdown 问题，先尝试 normalize / repair
2. 修复成功后再重新校验
3. 只有修复失败或仍有结构性问题时，才降级为 `needs_review`

同时应把“修复后的 snapshot”回写为权威 artifact 内容，而不是仅记录问题码。

## 7. 前端建议

前端不是本次根因，但为防止历史脏数据继续误导用户，建议做两项增强：

1. [frontend/src/views/chat/utils/documentCompletion.js](frontend/src/views/chat/utils/documentCompletion.js)
   - 当 `completion_status === completed` 时，应优先展示 completed 语义，即使历史 payload 中残留 `auto_continue_next=true`
   - `needs_review` 与 `partial/continuing` 语义必须严格区分

2. [frontend/src/views/chat/index.vue](frontend/src/views/chat/index.vue)
   - `rehydrateHistoricalDocumentContinuationState(...)` 仅对明确中断的 partial/failed 文档做恢复，不要对 terminal completed 文档再补 continuation 提示

这两项不是主修复，但能降低脏历史消息对展示的污染。

## 8. 建议实施顺序

### 8.1 第一阶段

优先完成四项：

1. simple continuation 接入 artifact-first finalize
2. completion payload 成为唯一状态权威源
3. long document task 不再覆盖已有 continuation decision
4. 续写场景改为对 merged snapshot 做整文 Markdown 修正

### 8.2 第二阶段

继续完成三项：

1. dedicated document edit 路径补齐 Markdown 修正
2. artifact 质量闸升级为“修复优先”
3. 显式基线续写逐步统一到 generation run 模型

## 9. 回归验证要求

修复后建议至少覆盖以下回归场景：

1. 首轮完整长文档生成，最后一章结束后未输出 completion marker，但整文结构完整
   - 期望：completed

2. 基于 base artifact 的 knowledge grounded continuation，正文已写到总结/展望，未输出 marker
   - 期望：completed，而不是 continuing

3. continuation 输出存在标题无空格、列表缩进异常、粗体标签黏连
   - 期望：最终 artifact 与最终展示正文均为修正后的 Markdown

4. continuation 结果存在内部提示词泄漏或结构异常
   - 期望：needs_review，但不再错误显示“系统可继续自动续写剩余章节”

5. dedicated document edit 输出章节 patch 时 Markdown 有轻微格式问题
   - 期望：优先修复后展示，而不是直接原样返回

6. 历史 completed 文档重新加载
   - 期望：前端不再重建 continuing 状态

## 10. 最终判断

当前长文档逻辑的核心问题不是“单个状态字段错了”，而是：

1. continuation 路径质量不一致
2. finalize 逻辑不统一
3. completion 与 artifact 双重收口互相覆盖
4. Markdown 修正停留在局部层面，没有统一提升到完整快照层

要彻底解决本问题，必须把“状态收口”和“Markdown 修正”都统一到“完整快照 + 单一权威终态”的架构上。只修前端提示词，或只修某一个 completion 判断，都会继续在其它路径上复发。
