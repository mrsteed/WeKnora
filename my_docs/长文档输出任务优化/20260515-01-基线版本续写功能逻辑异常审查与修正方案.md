# 基线版本续写功能逻辑异常审查与修正方案

## 1. 审查结论

本次“基于基线版本继续续写”异常，不是单点实现缺陷，而是路由层、应用层和执行层对“显式基线 artifact + 已选知识库”这一组合的语义定义不一致，导致请求虽然能识别出续写意图，却无法按“基线文档 + 本地知识库补充”的目标路径稳定落地。

结合当前代码，结论如下：

1. 路由模型已经能够识别这是“已有文档上的续写/编辑请求”，并且会保留知识库参与语义。
2. 处理器层对 `document_edit` 的应用守卫却把“带知识库的显式基线续写”判定为非法组合，直接拦截。
3. 即便阴影路由未应用，请求前置阶段仍会保留旧的基线文档编辑上下文，因此请求不会彻底失败，而是退回旧链路继续执行。
4. 旧链路中的专用文档编辑路径又明确拒绝带知识库的请求，知识库续写路径则强依赖 `autoContinue=true`，导致“显式基线续写 + 知识库”最终无法进入任何专用执行器，只能退回通用 Agent 路径。
5. 现有代码中已经存在可复用的知识库续写执行器，不需要再新增一套新的长文档引擎；问题核心在于接入条件和路由语义没有打通。

因此，本问题的根因可以归纳为一句话：

**系统已经支持“无知识库的基线编辑”与“自动续写的知识库续写”，但不支持“显式指定基线 artifact 的知识库续写”，而当前路由模型又会自然产出这个组合，导致前后层语义断裂。**

## 2. 现象与控制链路复盘

结合本次日志与代码，可还原出如下控制路径。

### 2.1 路由模型的输出是合理的

路由类型定义位于 [../internal/types/chat_route.go](../internal/types/chat_route.go)。当前已有以下几类与文档相关的路由：

1. `document_edit`
2. `knowledge_grounded_document_continuation`
3. `full_document`
4. `knowledge_grounded_full_document`

在本次场景中，请求显式携带了 `base_artifact_id`，同时存在有效知识库范围。日志显示路由模型给出的结果是：

1. `kind=document_edit`
2. `use_knowledge=true`
3. `use_long_document=false`
4. `reason=用户要求继续完成已有文档...使用agent和知识库`

这说明模型理解本轮请求是“编辑已有文档”，不是“新建完整长文档”，这一点本身没有问题。

### 2.2 路由应用阶段把该组合拦截了

在 [../internal/handler/session/chat_route.go](../internal/handler/session/chat_route.go) 中，`detectChatRouteDecision` 会在拿到路由决策后调用 `applyDocumentRouteDecisionWithReason(...)`。

当 `kind=document_edit` 时，代码会进入：

1. `case types.ChatRouteDocumentEdit`
2. 调用 `canApplyDocumentEditRouteDecisionWithReason(reqCtx, false)`

而 `canApplyDocumentEditRouteDecisionWithReason(...)` 当前存在如下关键守卫逻辑：

1. 如果有附件或图片，拦截。
2. 如果 `requireAutoContinue=true`，则强依赖 `reqCtx.autoContinue`。
3. 当 `requireAutoContinue=false` 时，如果存在 `knowledgeBaseIDs` 或 `knowledgeIDs`，直接返回 `has_knowledge_scope`。

这意味着：

**显式基线续写只要带了知识库范围，就无法应用 `document_edit` 路由。**

这正好与本次日志中的 `blocked_by=has_knowledge_scope` 完全吻合。

### 2.3 阴影路由虽然未应用，但旧的基线编辑上下文并没有丢

在 [../internal/handler/session/qa.go](../internal/handler/session/qa.go) 中，`parseQARequest(...)` 会先调用 `prepareDocumentRequest(...)`，再调用 `detectChatRouteDecision(...)`。

也就是说，路由判定之前，请求其实已经基于：

1. `BaseArtifactID`
2. 用户 query
3. 文档意图识别结果
4. quoted context 构建结果

完成了一次旧版文档编辑预处理。

即使后面的阴影路由应用失败，`qaRequestContext` 中仍然保留：

1. `documentIntent`
2. `documentOperation`
3. `baseArtifact`
4. `baseArtifactID`
5. `documentQuotedContext`
6. `documentOutputMode`

随后 `buildQARequest()` 会把这些字段原样带入执行层。

因此，本问题不是“路由未应用导致完全丢失续写能力”，而是：

**路由未应用后，请求会退回旧的基线文档编辑预处理结果继续执行。**

### 2.4 执行层再次把该组合排除在专用路径之外

在 [../internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go) 中，Agent 执行入口会按顺序选择专用路径：

1. `shouldUseKnowledgeGroundedDocumentContinuationPath(...)`
2. `shouldUseDedicatedDocumentEditPath(...)`
3. `shouldUseKnowledgeGroundedFullDocumentGenerationPath(...)`
4. `shouldUseDedicatedFullDocumentGenerationPath(...)`
5. 其余情况退回通用 Agent 路径

对本次场景而言，这两个关键守卫分别存在如下问题：

#### A. 知识库续写路径只接受 `autoContinue=true`

`shouldUseKnowledgeGroundedDocumentContinuationPath(...)` 的前置条件中明确要求：

1. `req.AutoContinue == true`
2. `req.BaseArtifactID != ""`
3. `req.BaseArtifact != nil`
4. `req.DocumentOutputMode == delta_only`
5. 本地知识范围有效

这意味着：

**即使已经显式给了 `BaseArtifactID` 且知识库范围有效，只要不是自动续写请求，就进不了知识库续写执行器。**

#### B. 专用文档编辑路径明确拒绝知识库范围

`shouldUseDedicatedDocumentEditPath(...)` 又有另一条排除条件：

1. 只要 `KnowledgeBaseIDs` 或 `KnowledgeIDs` 非空，直接返回 `false`

也就是说：

**无知识库的显式基线编辑才能进入专用 patch 路径；带知识库时会被拒绝。**

### 2.5 最终退回通用 Agent 路径

综合以上条件，本次“显式基线续写 + 知识库”组合的最终结果是：

1. 路由模型识别到了文档编辑意图。
2. 处理器层因 `has_knowledge_scope` 不应用该路由。
3. 旧的文档编辑上下文被保留下来。
4. 执行层既进不了知识库续写路径，也进不了专用文档编辑路径。
5. 请求只能退回通用 Agent QA。

这就是“功能并非完全不可用，但行为明显异常”的根本原因：

**系统表面上接住了请求，实际却没有走到与路由结论一致的执行器。**

## 3. 根因分析

### 3.1 路由类型语义缺口

当前类型系统只明确支持两类续写：

1. 不带知识库的 `document_edit`
2. 自动续写语义下的 `knowledge_grounded_document_continuation`

但缺少一个明确的中间态：

**显式指定 `base_artifact_id` 的知识库辅助续写/修订。**

这导致同一请求在不同层会被解释成不同含义：

1. 路由模型认为这是“文档编辑，并使用知识库”。
2. 处理器层认为“文档编辑不能带知识库”。
3. 执行层认为“知识库续写必须是 autoContinue”。

### 3.2 路由输入构建也在强化这种语义偏差

在 [../internal/handler/session/chat_route.go](../internal/handler/session/chat_route.go) 的 `buildChatRouteInput(...)` 中：

1. `AutoContinue=true` 时，`UserRequestedRoute=knowledge_grounded_document_continuation`
2. 仅当存在 `BaseArtifactID` 且不是自动续写时，`UserRequestedRoute=document_edit`

也就是说，当前输入构建阶段默认认为：

**显式基线续写等于 `document_edit`，而不是“知识库续写的一种触发方式”。**

这会进一步放大路由和执行层之间的语义不一致。

### 3.3 现有可复用执行器并没有真正缺失

这一点很关键。

在 [../internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go) 中，`runKnowledgeGroundedDocumentContinuationPath(...)` 并不严格依赖已有 generation run 才能工作。

当前实现是：

1. 先尝试加载 generation run。
2. 如果存在 run、outline 和 searchTargets，就走完整的分章节续写链路。
3. 如果没有 run，也会退化到基于 `BaseArtifact` 提取 outline、再结合本地知识继续生成。

这说明：

**知识库续写执行器本身已经具备承接“显式基线 artifact + 本地知识库”的能力。真正缺的是路由和守卫条件放行。**

### 3.4 测试覆盖存在明显空洞

当前测试覆盖了：

1. `document_edit` 在无知识库时可正常应用。
2. `knowledge_grounded_full_document` 的拦截条件。
3. 若干 full document 与 translation 场景。

但缺少一个最关键的回归用例：

**显式 `BaseArtifactID` + 有效知识库范围 + AgentQA + delta 输出**。

没有这条测试，就很难在重构路由、补充知识库续写、或调整守卫条件时及时发现功能回退。

## 4. 推荐修正方向

## 4.1 总体原则

修复建议遵循三条原则：

1. 显式 `BaseArtifactID` 仍然是本轮续写的主锚点，不能因为引入知识库就丢失基线上下文。
2. 知识库参与应被视为“续写增强能力”，而不是“与基线编辑互斥的另一种模式”。
3. 尽量复用现有知识库续写执行器，不再新增第三套相似的文档续写引擎。

## 4.2 推荐方案：复用现有知识库续写执行器，放宽接入条件

建议优先采用“复用执行器 + 对齐路由语义”的方案，而不是新增新的 route kind 和新的执行路径。

核心思路如下：

1. 当请求满足“显式 `BaseArtifactID` + 有效知识库范围 + delta 输出 + 续写/修订意图”时，应统一落到现有的知识库续写执行器。
2. `knowledge_grounded_document_continuation` 不再只表示“自动续写”，而应表示“基于既有文档锚点进行知识库辅助续写”，自动续写只是其中一种触发方式。
3. 对于不带知识库的显式基线编辑，仍然保留现有 `document_edit` 专用 patch 路径。

这条方案的优点是：

1. 变更面相对可控。
2. 可复用现有 evidence retrieval、outline 续写、continuation token budget、质量闸与 artifact 注册逻辑。
3. 能消除当前“路由结论正确，但没有执行器承接”的断层。

## 5. 具体修正建议

### 5.1 P0：放宽知识库续写路由的接入条件

建议调整 [../internal/handler/session/chat_route.go](../internal/handler/session/chat_route.go) 中与知识库续写相关的守卫逻辑。

目标行为应为：

1. `document_edit` 继续只服务于“无知识库的基线编辑”。
2. 当存在有效知识库范围时，不应再把显式基线续写挡在 `has_knowledge_scope` 之外。
3. 对“显式 `BaseArtifactID` + knowledge scope”的请求，应允许其映射到知识库续写语义。

推荐做法：

1. 保留 `canApplyDocumentEditRouteDecisionWithReason(...)` 对纯 `document_edit` 的知识库拦截。
2. 为知识库续写新增或拆分专门的守卫函数，不再使用“是否 autoContinue”作为唯一入口条件。
3. 新守卫应接受如下任一锚点成立：
   - `reqCtx.autoContinue == true`
   - `reqCtx.baseArtifactID != ""`

### 5.2 P0：统一执行层选择条件

建议调整 [../internal/application/service/session_agent_qa.go](../internal/application/service/session_agent_qa.go) 中的 `shouldUseKnowledgeGroundedDocumentContinuationPath(...)`。

当前逻辑硬性要求 `req.AutoContinue == true`，这是本次问题最直接的执行层阻塞点。

建议改为：

1. 只要存在 `BaseArtifactID`、`BaseArtifact`、有效知识库范围、delta 输出，以及 continue/revise 意图，就允许进入知识库续写路径。
2. `AutoContinue` 只作为“是否已有轮次状态、是否允许自动多轮接续”的附加信息，不再是唯一准入条件。

这样，显式基线续写请求就能够直接复用 `runKnowledgeGroundedDocumentContinuationPath(...)`。

### 5.3 P0：对齐路由语义与输入构建

建议调整 [../internal/handler/session/chat_route.go](../internal/handler/session/chat_route.go) 中 `buildChatRouteInput(...)` 的 `UserRequestedRoute` 计算方式。

推荐逻辑：

1. `AutoContinue=true` 时，仍然标记为 `knowledge_grounded_document_continuation`。
2. `BaseArtifactID!=\"\"` 且存在有效知识库范围时，也应优先表达为知识库续写语义，而不是统一标成 `document_edit`。
3. 只有在 `BaseArtifactID!=\"\"` 且无知识库范围时，才标记为 `document_edit`。

这样可以让路由模型看到的“当前系统意图”与后续真实可执行路径保持一致。

### 5.4 P1：路由归一化增加显式语义收敛

如果不希望修改路由模型 prompt 或 route kind 枚举，也可以在 [../internal/application/service/chat_route_service.go](../internal/application/service/chat_route_service.go) 的归一化阶段增加一层语义收敛：

1. 当模型返回 `kind=document_edit`
2. 且 `use_knowledge=true`
3. 且存在 `ExplicitBaseArtifactID`

则将其归一化为知识库续写语义，再由处理器层与执行层按统一守卫落地。

这不是唯一做法，但它能减少“模型会返回一种系统当前无法直接承接的路由”的概率。

### 5.5 P1：保留无知识库 patch 路径，不要互相污染

`shouldUseDedicatedDocumentEditPath(...)` 当前服务于：

1. 基线文档修订
2. 结构化 patch 输出
3. 无知识库、无附件、无图片的轻量编辑

这一路径应该保留，不建议为了快速修复而直接取消它对知识库的拦截，否则会把本地检索、章节续写、patch 生成三种语义混在一起，后续问题会更难收敛。

正确的改法是：

1. 无知识库时走 `document_edit`
2. 有知识库时走 `knowledge_grounded_document_continuation`

而不是让单一路径同时承担两套逻辑。

## 6. 建议修改点清单

建议按以下顺序实施：

### 6.1 第一阶段：打通主路径

1. 修改 `shouldUseKnowledgeGroundedDocumentContinuationPath(...)`，允许显式 `BaseArtifactID` 进入知识库续写。
2. 修改处理器层守卫，允许“显式基线 + knowledge scope”应用知识库续写语义。
3. 调整 `buildChatRouteInput(...)` 的 `UserRequestedRoute` 计算。

完成这三项后，主路径即可从“退回通用 Agent”切换为“进入专用知识库续写执行器”。

### 6.2 第二阶段：语义收敛与观测补齐

1. 在 route normalization 中增加 `document_edit + use_knowledge + explicit_base_artifact_id` 的语义收敛。
2. 补充日志字段，明确打印：
   - 最终选择的文档执行路径
   - 是否因 knowledge scope 触发了知识库续写
   - 是否因 base artifact 触发了显式续写
3. 补充 completion extra，便于前端或后续排查识别这类请求是否真正命中专用续写链路。

## 8. 风险与注意事项

### 8.1 不要把“有知识库”误解成“必须新建长文档”

本次日志已经证明路由模型的这部分判断是对的：

1. 这是已有文档的续写/编辑。
2. 不是重新创建完整长文档。

修复时应继续保持：

1. 输出模式仍以 `delta_only` 为主。
2. 基线 artifact 仍是主锚点。
3. 知识库仅作为续写补充证据来源。

### 8.2 不要把显式续写和自动续写完全绑定

自动续写只是续写的一种触发方式，并不等价于“所有知识库续写”。

如果继续把知识库续写严格绑定到 `autoContinue`，那么只要用户显式选择基线 artifact，就会再次出现本次同类问题。

## 9. 最终判断

当前代码并不是“不支持基线续写”，也不是“不支持知识库续写”，而是：

1. 支持无知识库的基线编辑。
2. 支持自动续写场景下的知识库续写。
3. **不支持显式基线 artifact 驱动的知识库续写。**

而本次用户请求恰好落在第 3 类，因此才出现：

1. 路由结论正确。
2. 路由应用失败。
3. 旧上下文仍在。
4. 专用执行器均无法命中。
5. 最终退回通用 Agent 路径。

建议按本文第 5 节和第 6 节进行修正。优先级最高的改动不是新增功能，而是**统一“显式基线续写 + 知识库”的语义，并放行到现有知识库续写执行器**。
