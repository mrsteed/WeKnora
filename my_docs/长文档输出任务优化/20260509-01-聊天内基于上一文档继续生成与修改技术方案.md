# 聊天内基于上一文档继续生成与修改技术方案

## 1. 审查结论

基于当前代码复核，上一版方案的总体方向“不要改造 `/continue-stream`，而是让下一轮请求显式承接上一份文档”是正确的，但存在一个必须修正的边界问题：上一版把长文档任务产物也纳入统一续写模型，这与最新约束不一致。

本次方案明确排除长文档模式。长文档模式在当前系统中只作为文档翻译入口，不参与本需求的“同一聊天内基于上一个回答继续生成和修改”。因此，本文只覆盖普通 Knowledge Chat 和 Agent Chat 中由 assistant 消息产生的 Markdown/纯文本长回答。

本方案的核心结论如下：

1. `ContinueStream` 继续保持断线恢复语义，不承担“重新调用模型续写”的职责。
2. 当前 `completion_status=partial` 已能表达半成品状态，可以作为识别可续写回答的重要依据。
3. 普通聊天和 Agent 聊天都已有 `QuotedContext` 注入点，P0 阶段可以先复用它完成续写上下文注入，降低改造成本。
4. 需要新增一个轻量的“会话文档快照”模型，专门记录由聊天回答产生的可续写文档版本。
5. 不接入长文档任务、长文档 artifact、长文档批次重试或异步长文档编辑任务。

## 2. 需求边界

### 2.1 覆盖范围

本方案覆盖以下场景：

1. 用户在普通聊天中要求生成技术方案、报告、Markdown 文档等长文本。
2. 用户在 Agent 聊天中要求生成技术方案、审查报告、实施方案等长文本。
3. 上一轮回答因为中断、截断、用户停止、服务异常或模型输出限制，只生成了部分内容。
4. 下一轮用户输入“继续生成”“接着上面写”“基于上一个回答继续输出”“把上一版补充得更详细”等。
5. 下一轮用户输入“修改上一版”“基于上一个方案调整”“把上一份改成更详细版本”等。

### 2.2 排除范围

以下内容不纳入本方案：

1. 长文档翻译任务的批处理流程。
2. 长文档任务的 artifact 下载和重试。
3. 基于原始知识库文件重新分批翻译或重新拼装 Markdown。
4. 超大文档的异步编辑任务。
5. 二进制文件、Office 文件、PDF 文件的在线编辑。

如果用户在长文档翻译入口发起任务，仍沿用当前长文档模式；如果用户在普通聊天中生成了一份方案或报告，才进入本文描述的会话文档续写能力。

## 3. 当前代码结构审查

### 3.1 消息状态已具备半成品表达能力

`internal/types/message.go` 中 `Message` 已有：

1. `CompletionStatus`
2. `FinishReason`
3. `FailureReason`
4. `IsCompleted`

并定义了：

```go
const (
    MessageCompletionStatusPending   = "pending"
    MessageCompletionStatusCompleted = "completed"
    MessageCompletionStatusPartial   = "partial"
    MessageCompletionStatusFailed    = "failed"
    MessageCompletionStatusCancelled = "cancelled"
)
```

`Message.IsTerminal()` 已经把 `partial`、`failed`、`cancelled` 视为终态。这说明后端可以区分“业务流已结束”和“回答完整完成”。续写能力应基于这一状态继续设计，而不是重新发明一套完成态。

### 3.2 ContinueStream 不能作为续写接口

`internal/handler/session/stream.go` 的 `ContinueStream` 当前职责是：

1. 根据 `session_id` 和 `message_id` 获取同一条 assistant 消息的 StreamManager 事件。
2. 回放已有 SSE 事件。
3. 如果流仍未终止，则继续轮询新事件。
4. 如果消息已终态，则发出终态事件让前端收口。

它不创建新的 user message，不调用模型，也不处理用户本轮“继续写”的自然语言指令。因此它只能用于刷新页面或网络断开后的继续接收旧流，不能承担本需求。

正确做法是：用户下一轮说“继续生成”时，仍创建新的 user message 和新的 assistant message，只是在新请求中注入上一份文档快照和续写指令。

### 3.3 历史上下文能看到上一轮，但无法稳定续写

普通聊天的历史加载位于 `internal/application/service/chat_pipeline/common.go`，`loadAndProcessHistory` 会按 `request_id` 组装 Q&A 历史。Agent 聊天的历史恢复位于 `internal/application/service/llmcontext/context_manager.go`，会从消息表恢复上下文。

这能让模型看到上一轮内容，但仍有不足：

1. 历史消息没有告诉模型“不要从头开始”。
2. 历史内容可能被上下文压缩或截断。
3. 用户只说“继续”时，模型不一定知道继续哪一份文档。
4. 修改场景需要完整基线，而普通历史不保证完整保留。

因此需要一个明确的会话文档快照，作为续写和修改的稳定基线。

### 3.4 现有 QuotedContext 可作为 P0 注入点

`internal/types/qa_request.go` 中 `QARequest` 已包含：

```go
QuotedContext string
```

普通 Knowledge Chat 中：

1. `internal/application/service/session_knowledge_qa.go` 会把 `req.QuotedContext` 写入 `chatManage.QuotedContext`。
2. `internal/application/service/chat_pipeline/into_chat_message.go` 会在纯聊路径和 RAG 路径中把 `chatManage.QuotedContext` 追加到 `UserContent`。

Agent Chat 中：

1. `internal/application/service/session_agent_qa.go` 会把 `req.QuotedContext` 追加到 `agentQuery`。

这意味着 P0 阶段无需重写聊天 pipeline，只要在 handler 层识别续写意图并生成 `QuotedContext`，即可让普通聊天和 Agent 聊天都拿到上一份文档快照。

### 3.5 请求结构需要补充续写字段

`internal/handler/session/types.go` 中 `CreateKnowledgeQARequest` 当前没有：

1. `intent_hint`
2. `base_artifact_id`
3. `document_operation`

前端 `frontend/src/api/chat/streame.ts` 也不会发送这些字段。当前 `frontend/src/views/chat/index.vue` 的 `sendMsg` 只负责选择 endpoint、组装知识库、文件、Agent、web search、memory 等参数。

因此需要在请求结构、前端 stream body、handler request context 中补齐续写字段。

## 4. 修正后的总体方案

新增“会话文档快照”能力，只记录聊天回答产生的可续写 Markdown/纯文本结果。

整体流程：

1. assistant 消息完成或 partial 收口后，判断该回答是否是一份可续写文档。
2. 如果命中条件，创建 `ChatDocumentArtifact` 记录，保存回答正文快照和版本信息。
3. 用户下一轮发送消息时，前端根据输入和最近 artifact 给出 `intent_hint`、`base_artifact_id`。
4. 后端再次识别意图，确认 artifact 属于当前 tenant/session。
5. 后端把上一版文档快照组装为 `QuotedContext`，并补充续写或修改规则。
6. 普通聊天和 Agent 聊天复用现有 `QuotedContext` 注入路径生成回答。
7. 本轮 assistant 完成后，保存新的 artifact revision，形成版本链。

该方案不接触长文档任务，不改变长文档翻译入口。

## 5. 数据模型设计

### 5.1 新增 chat_document_artifacts 表

建议新增类型：`types.ChatDocumentArtifact`。

```go
type ChatDocumentArtifact struct {
    ID               string         `json:"id" gorm:"type:varchar(36);primaryKey"`
    TenantID         uint64         `json:"tenant_id" gorm:"index;not null"`
    SessionID        string         `json:"session_id" gorm:"type:varchar(36);index;not null"`
    SourceMessageID  string         `json:"source_message_id" gorm:"type:varchar(36);index;not null"`
    SourceRequestID  string         `json:"source_request_id" gorm:"type:varchar(64);index"`
    ParentArtifactID string         `json:"parent_artifact_id,omitempty" gorm:"type:varchar(36);index"`
    RevisionNo       int            `json:"revision_no" gorm:"not null;default:1"`
    Title            string         `json:"title" gorm:"type:varchar(255)"`
    ArtifactKind     string         `json:"artifact_kind" gorm:"type:varchar(32);index;not null"`
    ContentType      string         `json:"content_type" gorm:"type:varchar(64);not null"`
    ContentSnapshot  string         `json:"content_snapshot" gorm:"type:text;not null"`
    ContentChecksum  string         `json:"content_checksum" gorm:"type:varchar(64);index;not null"`
    Status           string         `json:"status" gorm:"type:varchar(32);index;not null"`
    CompletionStatus string         `json:"completion_status" gorm:"type:varchar(32);index"`
    Operation        string         `json:"operation" gorm:"type:varchar(32);index"`
    CreatedBy        string         `json:"created_by" gorm:"type:varchar(36);index"`
    CreatedAt        time.Time      `json:"created_at"`
    UpdatedAt        time.Time      `json:"updated_at"`
    DeletedAt        gorm.DeletedAt `json:"deleted_at" gorm:"index"`
}
```

字段说明：

1. `SourceMessageID`：产生该版本的 assistant 消息 ID。
2. `SourceRequestID`：方便和同一轮 user/assistant 消息关联。
3. `ParentArtifactID`：续写或修改的上一版 artifact。
4. `RevisionNo`：同一文档链上的版本号。
5. `ArtifactKind`：第一阶段只需要 `chat_markdown`、`chat_text`。
6. `ContentSnapshot`：保存完整文本快照，P0 阶段不引入文件存储。
7. `Status`：建议 `available`、`partial`、`failed`。
8. `CompletionStatus`：同步 assistant 消息完成态，例如 `completed` 或 `partial`。
9. `Operation`：`create`、`continue`、`revise`、`regenerate`。

### 5.2 不新增 Message 外键字段

第一阶段不建议改 `messages` 表。通过 `SourceMessageID` 从 artifact 反查即可。

原因：

1. 降低迁移风险。
2. 不影响历史消息读写。
3. 不影响现有客户端 SDK。
4. 后续如果需要在消息加载时展示 artifact，可由查询接口按 message IDs 批量返回。

### 5.3 数据库迁移

当前迁移目录分为：

1. `migrations/versioned`：PostgreSQL。
2. `migrations/sqlite`：SQLite。

最新 PostgreSQL 版本已到 `000047`，SQLite 版本已到 `000003`，建议新增：

1. `migrations/versioned/000048_chat_document_artifacts.up.sql`
2. `migrations/versioned/000048_chat_document_artifacts.down.sql`
3. `migrations/sqlite/000004_chat_document_artifacts.up.sql`
4. `migrations/sqlite/000004_chat_document_artifacts.down.sql`

索引建议：

1. `(tenant_id, session_id, created_at)`：查询会话最近 artifact。
2. `(tenant_id, parent_artifact_id, revision_no)`：查询版本链。
3. `(source_message_id)`：从消息查 artifact。
4. `(content_checksum)`：避免同一消息重复登记。

## 7. 意图识别方案

### 7.1 意图枚举

新增类型：

```go
const (
    ChatDocumentIntentNormal     = "normal"
    ChatDocumentIntentContinue   = "continue_document"
    ChatDocumentIntentRevise     = "revise_document"
    ChatDocumentIntentRegenerate = "regenerate_document"
)
```

### 7.2 规则识别优先级

后端识别应按以下优先级：

1. 如果 query 包含“重新生成”“从头生成”“重写一版”“不要基于上一版”，直接判定 `regenerate_document`。
2. 如果 query 包含“修改上一版”“基于上一个文档修改”“把上一份改成”“调整上一版”“完善上一版”，判定 `revise_document`。
3. 如果 query 包含“继续生成”“接着写”“续写”“从上次中断处继续”“补全剩余”“继续输出”，判定 `continue_document`。
4. 如果前端传入 `IntentHint`，但后端规则无法确认，只能作为弱信号。
5. 其它情况为 `normal`。

### 7.3 启用续写的条件

只有满足以下条件，才真正启用文档续写上下文：

1. 意图为 `continue_document` 或 `revise_document`。
2. 当前 session 存在可用 artifact，或者 `BaseArtifactID` 指向当前 session 的可用 artifact。
3. artifact 的 `ArtifactKind` 为 `chat_markdown` 或 `chat_text`。
4. artifact 的 `Status` 为 `available` 或 `partial`。
5. 用户没有显式要求重新生成。

如果用户只输入“继续”，但当前会话没有可用 artifact，应按普通聊天处理，或者返回简短提示：“当前会话没有可继续的文档，请先生成一份文档。”推荐 P0 阶段按普通聊天处理，避免打断现有交互。

### 7.4 前端轻量识别

在 `frontend/src/views/chat/index.vue` 增加工具函数：

```ts
const inferDocumentContinuationIntent = (query = '') => {
  const text = query.trim().toLowerCase();
  if (!text) return 'normal';
  if (/(重新生成|从头生成|重写一版|不要基于上一版)/.test(text)) return 'regenerate_document';
  if (/(修改上一版|基于上一个文档修改|把上一份改成|调整上一版|完善上一版)/.test(text)) return 'revise_document';
  if (/(继续生成|接着写|续写|从上次中断处继续|补全剩余|继续输出)/.test(text)) return 'continue_document';
  return 'normal';
};
```

前端识别只负责提升体验，后端仍是权威判断。

## 8. QuotedContext 装配方案

### 8.1 续写模式

对于 `continue_document`，后端构造如下 `QuotedContext`：

```text
<document_continuation_context>
你正在继续生成同一会话中的上一份文档。

续写规则：
1. 不要从头重写上一份文档。
2. 从上一份文档末尾自然继续。
3. 保持标题层级、术语、编号、表格和 Markdown 风格一致。
4. 如果上一份文档末尾句子不完整，先补齐该句，再继续后续内容。
5. 默认只输出新增内容，不要重复上一份文档中已经完整输出的段落。
6. 不要解释“我将继续”，直接输出文档正文。

上一份文档元数据：
- artifact_id: {{.ArtifactID}}
- revision_no: {{.RevisionNo}}
- completion_status: {{.CompletionStatus}}
- operation: {{.Operation}}

上一份文档内容：
<document>
{{.BaseContent}}
</document>
</document_continuation_context>
```

### 8.2 修改模式

对于 `revise_document`，后端构造如下 `QuotedContext`：

```text
<document_revision_context>
你正在修改同一会话中的上一份文档。

修改规则：
1. 以上一份文档为基线进行修改。
2. 不要丢失用户没有要求删除的章节和内容。
3. 按用户本轮要求调整结构、补充细节或修正文案。
4. 输出修改后的完整 Markdown 文档。
5. 不要输出 diff 标记，不要输出修改说明，除非用户明确要求。

上一份文档元数据：
- artifact_id: {{.ArtifactID}}
- revision_no: {{.RevisionNo}}
- completion_status: {{.CompletionStatus}}
- operation: {{.Operation}}

上一份文档内容：
<document>
{{.BaseContent}}
</document>
</document_revision_context>
```

### 8.3 内容长度控制

因为本方案排除长文档模式，P0 阶段应控制在在线聊天可承载范围内：

1. `ContentSnapshot` 小于等于 30k 字符：完整注入。
2. `ContentSnapshot` 大于 30k 且小于等于 80k 字符：续写模式注入文档标题树和末尾 16k 字符；修改模式若用户未指定章节，提示用户缩小修改范围或改用“重新生成精简版”。
3. 超过 80k 字符：不启用自动续写，退回普通聊天提示，避免在线上下文再次截断。

标题树可用简单 Markdown 标题正则提取：

```go
var markdownHeadingRE = regexp.MustCompile(`(?m)^(#{1,6})\s+(.+)$`)
```

P0 阶段不需要复杂 Markdown AST。

## 9. Artifact 创建与版本保存

### 9.1 创建门禁

在 `completeAssistantMessage` 成功更新消息后，调用 artifact service 尝试登记：

```go
h.chatDocumentArtifactService.RegisterFromAssistantMessage(ctx, assistantMessage, options)
```

登记条件：

1. `message.Role == "assistant"`。
2. `message.Content` 去除空白后长度大于等于 2000 字符。
3. `CompletionStatus` 为 `completed` 或 `partial`。
4. `CompletionStatus` 不是 `failed` 或 `cancelled`。
5. 内容命中文档特征：至少 2 个 Markdown 标题，或标题加列表/表格，或用户 query 命中“方案/文档/报告/Markdown/技术方案/设计方案”。
6. 内容不是纯错误信息、不是工具日志、不是空 fallback。

注意：创建 artifact 不等于写入聊天历史知识库。`shouldIndexMessageToKB` 仍保持 completed-only 的质量门禁。

### 9.2 版本链规则

如果本轮请求有 `baseArtifactID` 且实际命中了 `continue_document` 或 `revise_document`：

1. 新 artifact 的 `ParentArtifactID = baseArtifact.ID`。
2. `RevisionNo = baseArtifact.RevisionNo + 1`。
3. `Operation = continue` 或 `revise`。

如果本轮是普通新文档：

1. `ParentArtifactID = ""`。
2. `RevisionNo = 1`。
3. `Operation = create`。

如果用户显式 `regenerate_document`：

1. 不注入 base artifact。
2. 新 artifact 可作为新的根版本，`RevisionNo = 1`。
3. `Operation = regenerate`。

### 9.3 续写结果保存

P0 阶段不建议后端强行合并正文，因为普通聊天前端展示的是本轮 assistant 输出。如果后端把旧正文和新增正文合并后再写回 assistant message，会导致聊天列表重复展示旧内容。

推荐规则：

1. assistant message 保存模型本轮输出内容。
2. artifact 的 `ContentSnapshot` 保存“合并后的完整版本”。
3. 对 `continue_document`：`ContentSnapshot = base.ContentSnapshot + "\n\n" + assistant.Content`。
4. 对 `revise_document`：`ContentSnapshot = assistant.Content`，因为修改模式要求模型输出完整新版。

这样聊天界面不会重复上一版，但 artifact 可承接完整文档版本。

### 9.4 修改结果校验

对 `revise_document`，需要防止模型只输出一句“已修改”。建议保存 artifact 前做轻量校验：

1. 如果新内容长度小于旧内容 40%，且用户没有“精简/摘要/压缩”等要求，则标记 artifact `Status=partial`。
2. 如果旧内容有一级标题，新内容完全没有标题，则标记 `Status=partial`。
3. 如果新内容包含“我已修改”“下面是修改建议”但缺少文档正文特征，则不创建 artifact。

## 10. 后端接入点

### 10.1 Handler 构造阶段

`internal/handler/session/handler.go` 的 `Handler` 需要注入：

```go
chatDocumentArtifactService interfaces.ChatDocumentArtifactService
```

`NewHandler` 增加对应参数，并在 container 中注册服务。

### 10.2 parseQARequest 阶段

在 `parseQARequest` 中读取：

1. `request.IntentHint`
2. `request.BaseArtifactID`

创建 `qaRequestContext` 时保存这两个字段。

### 10.3 创建 user message 前后

推荐在创建 user message 之前完成意图识别和 artifact 解析，但在真正调用模型前再构造 `QuotedContext`。

原因：

1. 意图识别需要 query。
2. artifact 权限校验只依赖 session 和 tenant。
3. `RenderedContent` 持久化时应包含最终注入后的上下文，方便下一轮历史恢复。

### 10.4 buildQARequest 阶段

`qaRequestContext.buildQARequest()` 将文档续写上下文写入 `QuotedContext`：

```go
quotedContext := rc.quotedContext
if rc.documentQuotedContext != "" {
    quotedContext = appendQuotedContext(quotedContext, rc.documentQuotedContext)
}
```

如果当前系统只有 IM quote 使用 `QuotedContext`，需要避免覆盖它，应做追加而不是替换。

### 10.5 completeAssistantMessage 阶段

在 `completeAssistantMessage` 更新消息成功后、`IndexMessageToKB` 前后均可尝试创建 artifact。推荐放在更新消息成功后、索引前：

1. message 已有最终内容和 completion 状态。
2. artifact 创建失败不应阻塞聊天完成。
3. artifact 创建和 KB 索引互不影响。

如果 artifact 创建失败，只记录 warning，不改变用户回答。

## 13. 权限与安全

### 13.1 tenant/session 校验

所有 artifact 查询必须同时校验：

1. `tenant_id` 等于当前租户。
2. `session_id` 属于当前用户可访问会话。
3. `base_artifact_id` 必须属于当前 session。

不能只按 artifact ID 查询后直接使用。

### 13.2 prompt 注入防护

`ContentSnapshot` 是模型生成内容，也可能包含提示注入文本。构造 `QuotedContext` 时必须明确：

1. `<document>` 内是待续写文档内容，不是新指令来源。
2. 真正指令来自系统规则和用户本轮 query。
3. 文档内部出现“忽略上述规则”等内容时，应视为普通文档文本。

### 13.3 数据体积控制

`ContentSnapshot` 建议设置最大保存长度，例如 100k 字符。超过阈值时：

1. artifact 仍可创建，但 `Status=partial`。
2. 保存 `ContentSnapshot` 的截断版本和 `FailureReason`/metadata。
3. 不自动续写超阈值 artifact。

第一阶段也可以直接拒绝创建超阈值 artifact，保持实现简单。

## 16. 风险与处理

### 16.1 用户意图误判

风险：用户说“继续讲一下”可能是普通追问。

处理：后端只有在存在可用 artifact 且命中明确文档续写词时才注入上下文；前端提供取消基于上一版的提示。

### 16.2 模型重复输出上一版

风险：模型看到完整文档后从头复述。

处理：续写模式明确要求“只输出新增内容”，并在 artifact 保存时合并版本；如果本轮输出与 base 开头高度重复，可标记为低质量并提示重试。

### 16.3 修改模式丢内容

风险：模型只输出修改片段。

处理：修改模式要求输出完整新版，并用长度和标题结构做轻量校验。

### 16.4 artifact 表膨胀

风险：长回答多次续写导致数据库增长。

处理：设置创建门禁、最大快照长度和软删除策略。P1 可增加清理任务，只保留最近 N 个版本或最近 N 天版本。

## 17. 最终推荐方案

最终推荐采用“聊天回答文档快照 + 续写意图识别 + QuotedContext 注入 + artifact 版本链”的方案。

这套方案的优点是：

1. 不改变 `ContinueStream` 的断线恢复语义。
2. 不引入长文档模式，符合“长文档模式只作为文档翻译入口”的边界。
3. 复用现有 `QuotedContext`，普通聊天和 Agent 聊天都能快速接入。
4. 通过 artifact 版本链解决“上一份文档是什么”的定位问题。
5. partial 消息可以作为会话内续写基线，但不会污染聊天历史知识库。

P0 完成后即可满足核心需求：同一聊天内，上一份明细技术方案没有完整输出时，用户下一轮说“继续生成”，系统能够基于上一份方案接着写，而不是重新从头生成。

## 18. 基于 2026-05-09 日志的续写失败专项分析与修正方案

本节仅基于当前日志与现有实现分析原因，并给出修正方案，不直接修改代码。

### 18.1 现象概述

本次用户在 Agent Chat 中基于已有 artifact 继续补充文档，前端请求为：

```json
{
  "query": "根据基线文档，继续补充智慧运行（AI智能运行体系）章节，详细阐述每个模块的功能。",
  "agent_enabled": true,
  "intent_hint": "revise_document",
  "base_artifact_id": "63da0a1e-10bf-44f6-b2bf-ae394d99db92"
}
```

从日志可见：

1. LLM 在 22 秒左右已经进入 `final_answer` 工具调用阶段：

```text
First tool_calls delta ... first_name="final_answer" ... elapsed_ms=22367
```

2. 整个流在约 117.5 秒结束，最终报错为：

```text
LLM stream error: context deadline exceeded
```

3. 在真正错误返回前，后端先把 assistant message 以 `completed/stop` 持久化了一次，而且内容长度是 0：

```text
completion_status: completed, finish_reason: stop, content_len: 0, agent_steps_count: 0
```

4. 紧接着同一条 message 又被第二次以 `failed/error` 持久化，且这次内容长度已经是 6018：

```text
completion_status: failed, finish_reason: error, failure_reason: agent_execution, content_len: 6018
```

5. 因为第一次完成态内容为空、第二次是失败态，所以最终用户看到的是“生成失败，请重试”，而不是“已生成部分内容，可继续补全”。

### 18.2 结论

这不是单一的“模型超时”问题，而是以下三层问题叠加：

1. **上游 LLM 单轮调用超时**：在长基线文档 + Agent 推理 + 知识库检索场景下，当前单轮 LLM 调用时限不足，导致流式输出在接近结束时被 `context deadline exceeded` 打断。
2. **失败路径状态收口错误**：超时后，Agent 引擎先发出一个字段为空的 `EventAgentComplete`，Handler 端把空值默认解释成 `completed/stop`，错误地先落了一次成功态。
3. **部分结果没有被归类为 partial**：虽然已经有 6018 字正文流到了 message，但最终又被覆盖成 `failed`，导致 artifact 不创建、版本链不推进、前端只显示失败提示。

换言之，**真正的根因不是 artifact 机制本身，而是 Agent 超时场景下“完成事件”和“失败事件”的状态语义不一致**。

### 18.3 日志对应的根因链

#### 18.3.1 第一层：LLM 调用确实超时

日志明确表明本轮 Agent thinking 流在约 117.5 秒时结束，并抛出：

```text
LLM stream error: context deadline exceeded
```

说明本次请求不是模型正常结束，而是被调用上下文强制截断。

这与当前场景高度一致：

1. 本轮不是纯短问答，而是基于基线文档继续改写长文档。
2. Agent 模式会先经历 thinking、tool_call、final_answer 等多阶段。
3. 本轮还绑定了知识库，提示词和上下文都比普通继续生成更大。

因此，**超时是触发故障的第一事件**。

#### 18.3.2 第二层：空的 complete 事件被错误当成成功完成

日志里第一条持久化是：

```text
completion_status: completed, finish_reason: stop, content_len: 0, agent_steps_count: 0
```

但紧接着 `agent_stream_handler.go:478` 的日志却显示：

```text
agent_steps_count: 0, completion_status: , finish_reason:
```

这说明当时 `EventAgentComplete` 里至少有两个关键字段是空的：

1. `CompletionStatus` 为空。
2. `FinishReason` 为空。

当前实现里，Handler 的 `completionOptionsFromComplete()` 会以默认值初始化 Agent 完成态：

1. `CompletionStatus = completed`
2. `FinishReason = stop`
3. `AllowIndexing = true`
4. `AllowComplete = true`

如果 complete 事件本身字段为空，就会落回这些默认值，进而把“实际上尚未明确成功的事件”写成真正的 completed 成功态。

因此，**这次第一条 completed 持久化不是业务真实成功，而是空 complete 事件被错误默认成成功态的结果**。

#### 18.3.3 第三层：真正错误到来后又把消息覆盖成 failed

后面真正的 Agent 执行错误返回时，又触发了第二次持久化：

```text
completion_status: failed, finish_reason: error, failure_reason: agent_execution, content_len: 6018
```

这说明：

1. 前一阶段其实已经积累了相当长度的正文内容。
2. 但因为最终错误路径使用的是 `failed`，而不是 `partial`，所以系统把“超时前已经产出的正文”也归类成失败答案。

这直接带来三个后果：

1. 前端显示“生成失败，请重试”，而不是“部分生成完成”。
2. artifact service 不会登记当前版本，因为 `RegisterFromAssistantMessage` 不会为 failed 消息建 artifact。
3. 用户下一轮无法基于这次已经生成的 6018 字内容继续，而只能继续沿用旧基线。

因此，**真正的用户体验损失来自“超时后的结果被归类为 failed，而不是 partial”**。

#### 18.3.4 第四层：`record not found` 不是根因，只是副作用日志

日志中的：

```text
SELECT * FROM chat_document_artifacts WHERE ... source_message_id = '7745...'
record not found
```

这是 artifact 去重查询时的正常探测日志，表示“当前 message 还没有关联 artifact”，不是异常根因。

真正的问题是：

1. 第一次 completed 持久化时 `content_len = 0`，自然不会创建 artifact。
2. 第二次 failed 持久化时状态又不允许创建 artifact。

所以最终没有 revision 落库，**这是结果，不是根因**。

### 18.4 对当前实现的准确判断

基于日志和当前代码，可以明确判断：

1. **P0/P1 的文档续写链路本身是通的**：前端正确传入了 `intent_hint=revise_document` 和 `base_artifact_id`。
2. **P2 的质量增强不是本次失败的直接原因**：结构检测、长文档提示、artifact 质量信号与本次报错没有直接冲突。
3. **真正失效的是 Agent 超时场景下的状态机**：
  1. 引擎层没有在超时错误路径上先规范化 `state.CompletionStatus/FinishReason`。
  2. Handler 层把“空 complete 事件”默认为 completed。
  3. 最终失败态又把已产出的正文按 failed 覆盖，导致 artifact 无法承接。

### 18.5 具体修正方案

以下方案按“先止血，再完善”分层执行。

#### 18.5.1 修正一：Agent 引擎在超时错误路径必须先规范化完成态

修正位置建议：

1. `internal/agent/think.go`
2. `internal/agent/engine.go`
3. `internal/agent/finalize.go`

修正目标：

1. 只要 LLM 调用以错误退出，`state.CompletionStatus` 不能保持空值。
2. 只要出现 `context deadline exceeded`，必须在引擎层明确写出：
  1. `CompletionStatus = partial` 或 `failed`
  2. `FinishReason = timeout` 或 `error`
  3. `FailureReason = llm_timeout`
3. `emitCompletionEvent()` 发出的 `EventAgentComplete` 不能再携带空的 `CompletionStatus` 与 `FinishReason`。

建议规则：

1. 如果超时前已经产生了可见正文 chunk，统一归类为：
  1. `CompletionStatus = partial`
  2. `FinishReason = timeout`
  3. `FailureReason = llm_timeout`
  4. `AllowIndexing = false`

## 19. 基于 2026-05-09 18:14 日志的第二类故障分析与修正方案

本节对应的是另一类实际故障。它与上一节的超时问题不同，本次日志显示：**final answer 已经自然生成成功，但最终仍被系统按 `failed/tool_error` 收口**，因此前端才出现截图中的“生成失败，请重试 / tool_error”。

### 19.1 现象概述

本轮请求仍然是基于基线文档做续写：

```json
{
  "query": "根据基线文档，继续补充智慧运行（AI智能运行体系）章节，详细阐述每个模块的功能。",
  "agent_enabled": true,
  "intent_hint": "revise_document",
  "base_artifact_id": "63da0a1e-10bf-44f6-b2bf-ae394d99db92"
}
```

但本次日志与上一节有两个本质差异：

1. 没有出现 `context deadline exceeded`。
2. final answer 流明确是自然停止，而且已经生成出完整答案。

日志证据：

```text
Natural-stop at OpenAI layer (finish=stop ... elapsed_ms=15632)
```

```text
[Agent][Stream] Completed: chunks=469, content_len=621, tool_calls=0
```

```text
[Agent][FinalAnswer] Final answer generated: 621 characters
```

但紧接着 `qa.go` 的持久化日志却是：

```text
completion_status: failed, finish_reason: tool_error, failure_reason: tool_error, content_len: 493, agent_steps_count: 1
```

随后 `agent_stream_handler.go` 下发给前端的 complete 事件也是：

```text
completion_status: failed, finish_reason: tool_error
```

最终前端只能渲染为：

1. 文本内容中断。
2. 消息状态为失败。
3. 底部出现 `tool_error`。

### 19.2 结论

本次故障不是“模型没答出来”，也不是“续写链路没走通”，而是：

1. Agent 某个较早阶段已经把 `state` 标记成 `failed/tool_error`。
2. 后续兜底 final answer 合成虽然成功，但没有把这个失败态覆盖回成功态或 partial 态。
3. `finalize.go` 在发 answer chunk 和 complete 事件时直接复用了旧的失败态。
4. `qa.go` 在持久化时又沿用这个失败态，导致“答案已生成，但消息最终被判失败”。

因此，本次问题的本质是：**状态机中的失败态被提前写入，并在 final answer 成功后没有被纠正，形成 stale failure state 污染最终收口。**

### 19.3 日志对应的根因链

#### 19.3.1 第一层：final answer 实际上已经成功生成

从日志可见，本轮 final answer 阶段具有完整的成功信号：

1. `remote_api.go` 明确记录了 `Natural-stop at OpenAI layer (finish=stop)`。
2. `think.go` 记录本轮流式完成，`content_len=621`。
3. `finalize.go` 明确记录 `Final answer generated: 621 characters`。

这三条日志组合起来已经足以说明：**最终 LLM 合成阶段不是失败，而是成功完成。**

因此，前端出现 `tool_error` 不是因为 final answer 本身报错。

#### 19.3.2 第二层：失败态在 final answer 之前已经被写入 state

当前实现里，`internal/agent/think.go` 在“LLM 失败但已有 tool results，可降级合成 final answer”的路径上，会先把状态写成：

1. `CompletionStatus = failed`
2. `FinishReason = tool_error`
3. `FailureReason = tool_error`
4. `AllowIndexing = false`
5. `AllowComplete = false`

然后才调用 `streamFinalAnswerToEventBus()` 去做最终答案合成。

这意味着：**只要进入这条 graceful degradation 分支，state 在 final answer 开始前就已经被标记成失败。**

#### 19.3.3 第三层：final answer 发流时直接复用了旧失败态

`internal/agent/finalize.go` 中 `buildFinalAnswerEventData()` 会直接从 `state` 读取：

1. `CompletionStatus`
2. `FinishReason`
3. `AllowIndexing`
4. `AllowComplete`
5. `FailureReason`

因此，如果调用 `streamFinalAnswerToEventBus()` 时 `state` 已经是 `failed/tool_error`，那么即使 answer chunk 是正常生成出来的，发给 EventBus 的 `EventAgentFinalAnswer` 也会继续携带失败态。

也就是说，本次日志里的“答案正常流出”和“最终被标记失败”并不矛盾，因为它们共享的是同一个被污染的 state。

#### 19.3.4 第四层：成功生成后的结果没有反向纠正状态

当前 `streamFinalAnswerToEventBus()` 成功后只会：

1. 记录 `state.FinalAnswer = fullAnswer`
2. 返回成功

但它不会在成功后重写：

1. `state.CompletionStatus`
2. `state.FinishReason`
3. `state.FailureReason`
4. `state.AllowIndexing`
5. `state.AllowComplete`

因此，成功的 final answer 只更新了正文，没有更新最终状态。

这就是本次故障的核心控制点。

#### 19.3.5 第五层：qa 持久化优先使用已有 message.Content，导致正文长度进一步缩水

本次日志里还有一个重要副症状：

1. `finalize.go` 记录 `Final answer generated: 621 characters`
2. `qa.go` 持久化时却是 `content_len: 493`

当前 `completeAssistantMessage()` 的逻辑是：

1. 只有在 `updatedMessage.Content == ""` 时，才会用 `options.FinalAnswer` 回填正文。
2. 如果流式过程中 `assistantMessage.Content` 已经累积了一部分内容，就不会再用最终合成后的 `FinalAnswer` 覆盖。

因此，本次 493 与 621 的差值说明：

1. 流式缓冲区中已经积累了一版较短正文。
2. 真正合成完成的 621 字最终答案并没有回写覆盖这份正文。

这不是本次失败态的主因，但它会进一步放大用户感知问题：**不仅状态错了，连最终落库的正文也可能不是最终版。**

### 19.4 对当前实现的准确判断

基于日志和当前实现，可以做出以下准确判断：

1. **续写请求参数本身没有问题**：`intent_hint=revise_document` 与 `base_artifact_id` 都已正确传入。
2. **final answer 模型调用本身没有失败**：日志已经明确给出 `Natural-stop` 与 `Final answer generated`。
3. **真正出错的是 Agent graceful degradation 的状态管理**：
  1. 先把 state 标记成 `failed/tool_error`。
  2. 后续生成成功时没有回写修正这个状态。
  3. complete 事件、持久化状态、前端展示全部沿用了旧失败态。
4. **正文持久化还有次级问题**：最终合成答案与消息正文缓冲区没有做“最终版优先”的统一。

### 19.5 具体修正方案

#### 19.5.1 修正一：final answer 合成成功后必须重写最终状态

修正位置建议：

1. `internal/agent/think.go`
2. `internal/agent/finalize.go`
3. `internal/agent/engine.go`

修正目标：

1. 进入 graceful degradation 分支时，允许先记录“上游 think 失败”的事实。
2. 但如果 `streamFinalAnswerToEventBus()` 最终成功，则必须在返回前重新规范化最终状态。

建议规则：

1. 如果 final answer 合成成功且正文非空，则最终状态至少应改为：
  1. `CompletionStatus = partial` 或 `completed`
  2. `FinishReason = stop` 或 `fallback_stop`
  3. `FailureReason = ""` 或保留单独的 `upstream_failure_reason`
2. 推荐优先落为 `partial`，因为这次答案来源于降级兜底而不是主路径完整成功。
3. `AllowIndexing` 与 `AllowComplete` 应依据业务策略明确赋值，不能继续沿用 `failed` 分支残留值。

如果不做这一步，后面所有 handler 和前端即使逻辑正确，也只能接收到错误的失败态。

#### 19.5.2 修正二：将“上游失败原因”与“最终交付状态”拆开建模

当前实现把“上游 tool_error”直接写进了最终 completion 字段，导致“过程失败但结果可用”的场景无法表达。

建议拆成两层语义：

1. **ExecutionOutcome**：记录中间阶段是否发生过 tool/think 错误。
2. **DeliveryStatus**：记录最终是否成功向用户交付了可展示答案。

建议规则：

1. 上游 tool 失败但最终成功合成答案：
  1. `DeliveryStatus = partial` 或 `completed`
  2. `ExecutionOutcome = tool_error_recovered`
2. 上游 tool 失败且最终也没有答案：
  1. `DeliveryStatus = failed`
  2. `ExecutionOutcome = tool_error`

如果当前阶段不准备调整数据模型，至少也应做到：

1. `completion_status` 代表最终交付态。
2. `failure_reason` 仅表示上游异常原因，不能反向覆盖已经成功交付的结果。

#### 19.5.3 修正三：complete 事件优先使用最终答案态，而不是旧 state

修正位置建议：

1. `internal/agent/finalize.go`
2. `internal/handler/session/agent_stream_handler.go`

修正目标：

1. `buildFinalAnswerEventData()` 不应无条件继承旧 state。
2. 如果当前事件来自“成功完成的 final answer 合成”，则事件状态应由“本次合成结果”决定，而不是由之前某个失败分支决定。

建议规则：

1. 对 final answer chunk 事件增加显式参数，例如：
  1. `eventCompletionStatus`
  2. `eventFinishReason`
  3. `eventFailureReason`
2. `emitCompletionEvent()` 也应在发送前基于最终答案是否存在、是否成功合成来重新决策最终 complete 状态。

#### 19.5.4 修正四：agent completion 持久化时，“最终版正文”必须覆盖流式缓冲正文

修正位置建议：

1. `internal/handler/session/qa.go`

修正目标：

1. 对 Agent 模式，若 `options.FinalAnswer` 非空，应支持按策略覆盖 `assistantMessage.Content`。
2. 至少在以下场景必须使用最终版正文：
  1. final answer 合成成功。
  2. `options.FinalAnswer` 长于当前 message.Content。
  3. 当前 message 状态来自 graceful degradation/fallback 收口。

建议规则：

1. 增加“最终答案优先”的判定。
2. 当 `FinalAnswer` 被认定为权威版本时，用它覆盖流式拼接正文。

这样可以解决本次 `621` 与 `493` 不一致的问题，避免用户看到被截短或非最终版内容。

#### 19.5.5 修正五：前端对 recovered tool error 场景不要显示硬失败

修正位置建议：

1. `frontend/src/views/chat/index.vue`
2. `frontend/src/views/chat/components/botmsg.vue`

修正目标：

1. 当后端已经有完整正文，且最终交付态不是 `failed` 时，前端不应继续显示 `tool_error` 红条。
2. 如果业务上选择把这类结果定义为 `partial`，则前端应展示：
  1. “已基于降级结果生成部分内容。”
  2. “可继续补全或重新生成。”

这样能把“可恢复错误”与“真正不可用失败”区分开。

#### 19.5.6 修正六：增加 graceful degradation 成功后的回归测试

必须增加回归测试，覆盖以下场景：

1. 上游 think/tool 阶段报错。
2. `state` 先被写成 `failed/tool_error`。
3. `streamFinalAnswerToEventBus()` 成功生成正文。
4. 最终 `EventAgentComplete` 必须不是 `failed/tool_error`。
5. `completeAssistantMessage()` 持久化的正文必须等于最终版 `FinalAnswer`。
6. revision 模式下若正文达到门槛，应继续允许 artifact 注册。

### 19.6 建议的修复顺序

建议按以下顺序修：

1. **第一优先级**：修正 graceful degradation 成功后未回写最终状态的问题。
2. **第二优先级**：修正 complete 事件继续复用 stale failure state 的问题。
3. **第三优先级**：修正 Agent 持久化阶段“最终答案不覆盖旧正文”的问题。
4. **第四优先级**：前端区分 recovered tool error 与 hard failure。
5. **第五优先级**：补齐回归测试，覆盖 tool_error -> final answer success 的完整链路。

### 19.7 验收标准

本问题修复后，应满足以下验收：

1. 复现同类场景时，即使上游出现 tool/think 错误，只要 final answer 成功生成，就不再以下游 `failed/tool_error` 收口。
2. `finalize.go` 记录 `Final answer generated` 后，对外发出的 complete 事件状态必须与最终交付态一致。
3. `qa.go` 最终持久化的正文应与最终版 `FinalAnswer` 一致，不再出现明显缩水。
4. 若最终态为 `partial` 或 `completed`，前端不再显示硬错误红条。
5. 若正文达到 artifact 门槛，revision 模式下仍可继续生成版本链，而不是因为 stale failure state 被中断。
