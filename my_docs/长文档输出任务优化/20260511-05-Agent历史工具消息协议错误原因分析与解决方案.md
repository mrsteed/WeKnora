# Agent 历史工具消息协议错误原因分析与解决方案

## 1. 问题结论

本次报错的直接原因是：发送给 `deepseek-v4-pro` 的 OpenAI 兼容 Chat Completions 请求中，历史上下文包含了不符合工具调用协议的 `role=tool` 消息。模型服务在真正生成前即校验失败，返回：

```text
400 Bad Request: Messages with role 'tool' must be a response to a preceding message with 'tool_calls'
```

这不是“文档修改指令”本身的问题，也不是知识库、Agent 分享或 Web 搜索的问题。日志显示系统已经成功解析会话、加载 Agent、注册工具并进入 Agent 第一轮思考；失败发生在第一轮 LLM 流式请求创建阶段。

核心修复方向是：在任何历史上下文进入 LLM 请求前，必须严格保证 OpenAI 工具调用消息链合法，即：

```text
assistant(tool_calls=[call_a, call_b])
tool(tool_call_id=call_a)
tool(tool_call_id=call_b)
```

不能出现孤立 `tool`、缺少 `tool_call_id` 的 `tool`、与最近一组 `tool_calls` 不匹配的 `tool`，也不能把工具调用 assistant 和工具结果拆散后继续发送给模型。

## 2. 日志现象复盘

### 2.1 请求进入了文档修订 Agent 流程

用户请求：

```text
把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

请求参数中包含：

```json
{
  "intent_hint": "revise_document",
  "base_artifact_id": "14a55959-399a-4510-9c0d-9f43e438e7dd",
  "document_output_mode": "delta_only",
  "agent_enabled": true,
  "agent_id": "22a6c00a-9e10-4776-8465-aec4463d21e2"
}
```

这说明本轮是基于上一份文档的结构化修订，不是普通问答。

### 2.2 非根因日志

以下日志看起来显眼，但不是导致本次 400 的主因：

1. `agent_share.go:197 record not found`

   系统随后打印 `Using own agent`，说明只是先查跨租户共享 Agent 未命中，然后回退到用户自己的 Agent，属于正常分支。

2. `Dropped database tools ... because no database-capable KB is in scope`

   当前知识库不支持数据库工具，系统按配置移除了外部数据库工具，不影响普通文档修订。

3. `Duplicate tool registration rejected: final_answer`

   `final_answer` 被注册了两次，注册表采用 first-wins，属于噪声告警，不是本次请求失败原因。

4. `chat_document_artifact.go:41 record not found`

   这是失败后的结果：本轮 assistant 内容为空，没有生成新的文档 artifact，因此按 `source_message_id` 查不到 artifact。

### 2.3 关键失败点

关键日志如下：

```text
Loaded 24 messages from Redis
LLM context stats: messages=24
Ready: 25 messages, 24 tools
[Agent][Round-1/40] Starting: 25 messages, 24 tools
[Agent][Round-1] msg[21]: role=tool, name=wiki_read_source_doc, len=103
[Agent][Round-1] msg[22]: role=tool, name=wiki_read_page, len=103
[Agent][Round-1] msg[23]: role=assistant, len=8092, content=<document_patch>...
[Agent][Round-1] msg[24]: role=user, len=41632, content=<runtime_context ...>
Failed to start LLM stream: Messages with role 'tool' must be a response to a preceding message with 'tool_calls'
```

这说明：

1. 本轮请求没有等到模型开始输出，失败发生在请求体校验阶段。
2. 请求体中携带了上一轮或更早轮次的工具历史消息。
3. 至少有一个 `tool` 消息在最终发送给 provider 的消息序列中，不满足“必须紧跟并响应前一个 assistant 的 tool_calls”这一协议要求。

## 3. 代码链路定位

### 3.1 历史上下文从 Redis 进入 Agent

相关文件：`internal/application/service/llmcontext/context_manager.go`

`GetContext()` 先从 Redis 加载会话上下文：

```go
messages, err := cm.storage.Load(ctx, sessionID)
if len(messages) > 0 {
    return messages, nil
}
```

日志中的 `Loaded 24 messages from Redis` 和 `Cache hit: 24 messages` 证明本轮使用的是 Redis 中的历史上下文，而不是完全从数据库重建。

### 3.2 Agent 构造 LLM messages 时保留了历史 tool 消息

相关文件：`internal/agent/observe.go`

`buildMessagesWithLLMContext()` 会把历史上下文加入本轮请求：

```go
sanitized = redactHistoryKBResults(llmContext)
...
if msg.Role == "user" || msg.Role == "assistant" || msg.Role == "tool" {
    messages = append(messages, msg)
}
```

这里的 `redactHistoryKBResults()` 只会把知识库工具结果的正文替换为占位文本，避免模型复用过期检索结果；它不会校验工具消息链是否合法。

因此，如果 Redis 中已有孤立 `tool`、缺失 `tool_call_id` 的 `tool`，或者工具调用 assistant 被压缩、合并、丢失后留下了 tool result，这些问题会继续进入 LLM 请求。

### 3.3 请求前已有一次清洗，但清洗规则不够严格

相关文件：`internal/agent/think.go`、`internal/agent/tools/sanitize_messages.go`

`callLLMWithRetry()` 在发送请求前调用：

```go
messages = agenttools.SanitizeMessages(messages)
response, err := e.streamThinkingToEventBus(ctx, messages, tools, iteration, sessionID)
```

`SanitizeMessages()` 目前宣称会处理孤立工具消息，但实现仍有缺口：

```go
if msg.Role == "tool" && msg.ToolCallID != "" {
    if !hasMatchingToolCall(messages[:i], msg.ToolCallID) {
        msg.Role = "system"
        ...
    }
}
```

主要问题有四个：

1. 只在 `ToolCallID != ""` 时校验。若历史 `tool` 消息没有 `tool_call_id`，会原样保留，provider 仍会拒绝。
2. `hasMatchingToolCall()` 只检查“任意更早的 assistant 是否有同 ID tool_call”，没有检查它是否是最近一组、是否紧邻、是否仍在清洗后的消息序列中。
3. 清洗连续同角色 assistant 时只拼接 `Content`，可能丢掉后一个 assistant 的 `ToolCalls`，但后续校验仍基于原始 `messages[:i]`，导致清洗后序列与校验依据不一致。
4. 没有处理“assistant(tool_calls) 后缺少部分 tool 结果”或“tool 结果被别的 user/assistant 隔开”的完整协议约束。

因此，日志中的 400 说明：现有 `SanitizeMessages()` 没有把非法工具消息链真正清理掉。

### 3.4 上下文持久化存在部分写入风险

相关文件：`internal/agent/observe.go`

`appendToolResults()` 会先把 assistant tool_calls 写入上下文，再逐个写入 tool result：

```go
contextManager.AddMessage(ctx, sessionID, assistantMsg)
...
contextManager.AddMessage(ctx, sessionID, contextToolMsg)
```

这是多次独立写入。如果进程中断、请求取消、Redis 写入失败或后续逻辑异常，缓存中可能留下半截工具调用链。下一轮请求再加载这段历史，就可能出现：

1. 有 assistant(tool_calls) 但缺少对应 tool result。
2. 有 tool result 但前面的 assistant(tool_calls) 被压缩、合并或丢失。
3. 工具结果顺序与 tool_calls 不一致。

这些历史污染会在 Redis TTL 内持续影响同一会话。

## 4. 失败后的状态问题

本次日志里还有一个伴随问题：Agent 调用失败后，assistant message 被持久化成了 completed。

关键日志：

```text
Agent execution failed: LLM call failed: ... 400 Bad Request ...
Preparing assistant completion persistence ... completion_status: completed, finish_reason: stop, failure_reason: , content_len: 0, agent_steps_count: 0
Agent completion is being persisted without agent_steps
Persisted assistant completion state ... is_completed: true, allow_indexing: true, content_len: 0
```

这会带来三个副作用：

1. 前端看到的是一次“已完成但无内容”的回答，而不是明确失败态。
2. 空 assistant 消息可能参与历史对话恢复或索引逻辑，污染后续上下文。
3. 文档 artifact 注册阶段查不到本轮产物，只能出现 `record not found`。

因此除了修复工具消息协议，还要修正失败状态持久化：LLM 400 这类不可恢复错误不应保存为 `completed/stop`，也不应允许索引。

## 5. 推荐解决方案

建议分三层处理：先止血清理受污染会话，再修复消息协议清洗，最后补齐失败状态和测试。

### 5.1 线上止血方案

对已经受影响的会话，优先清理 Redis 中的 LLM 上下文缓存，让下一轮不要继续携带坏的 tool history。

当前代码中 Redis key 前缀为：

```go
llmcontext.NewRedisStorage(redisClient, 24*time.Hour, "context:")
```

因此本次会话可清理：

```bash
redis-cli DEL context:ecb4120a-df09-4c4f-ba88-e8d85b552b95
```

如果部署环境使用了非默认 Redis DB、密码或集群，需要按实际连接参数执行。清理后可以让用户在同一会话重试；如果数据库回放仍重建出非法 AgentSteps，则建议新建会话或临时过滤该会话的异常 assistant 消息。

同时应把本轮空 assistant 消息标记为失败或排除索引。对应 message_id：

```text
ed0ed81e-8551-440b-85ac-16b60485da65
```

建议修正为：

```text
completion_status = failed
finish_reason     = model_error
failure_reason    = LLM 400: invalid tool message protocol
is_completed      = true 或 false，按现有失败态约定统一
allow_indexing    = false
content           = 可保留为空，或写入用户可见失败提示
```

### 5.2 严格修复 SanitizeMessages

修改文件：`internal/agent/tools/sanitize_messages.go`

目标不是简单“找任意前置 tool_call_id”，而是按 OpenAI 协议重新构造合法消息序列。

建议新增 `ValidateAndSanitizeToolProtocol()`，按以下规则从左到右处理：

1. `system/user/assistant` 普通消息正常保留。
2. `tool` 消息必须满足：
   - `ToolCallID` 非空。
   - 前一个未闭合工具组来自最近的 `assistant`。
   - `ToolCallID` 属于该 assistant 的 `ToolCalls`。
   - 没有被 user/assistant/system 打断。
3. `assistant` 如果带 `ToolCalls`：
   - 所有 tool_call 必须有稳定 ID；空 ID 应生成或整组降级为普通 assistant 摘要。
   - 后续必须紧跟对应 tool result。
   - 如果历史中结果不完整，应丢弃整组工具协议字段，或把整组转换为普通 assistant 历史摘要，不能留下半组工具消息。
4. `tool` 如果非法：
   - 推荐转换成 `assistant` 或 `system` 摘要，例如 `[Historical tool result omitted: wiki_read_page]`。
   - 更保守的做法是直接丢弃，因为历史检索结果本来已经要求每轮重新检索。
5. 禁止在合并连续 assistant 时丢失 `ToolCalls`。
   - 如果前后两个 assistant 任一方包含 `ToolCalls`，不要做普通 content merge。
   - 或者先完成工具组校验，再做普通消息合并。
6. 校验必须基于清洗后的 `result`，不能基于原始 `messages[:i]`。

伪代码示意：

```go
func SanitizeMessages(messages []chat.Message) []chat.Message {
    result := make([]chat.Message, 0, len(messages))
    pending := map[string]bool{}
    pendingAssistantIndex := -1

    closeBrokenPending := func() {
        if len(pending) > 0 && pendingAssistantIndex >= 0 {
            // 历史工具组不完整，不能继续以 tool_calls 协议发送。
            result[pendingAssistantIndex].ToolCalls = nil
            result[pendingAssistantIndex].Content = strings.TrimSpace(
                result[pendingAssistantIndex].Content + "\n[Historical tool calls omitted because results were incomplete.]",
            )
            pending = map[string]bool{}
            pendingAssistantIndex = -1
        }
    }

    for _, msg := range messages {
        switch msg.Role {
        case "assistant":
            closeBrokenPending()
            // 保留普通 assistant；若有 tool_calls，建立 pending 集合。
        case "tool":
            if msg.ToolCallID == "" || !pending[msg.ToolCallID] {
                // 非法历史 tool，丢弃或转摘要，不能保留 role=tool。
                continue
            }
            result = append(result, msg)
            delete(pending, msg.ToolCallID)
            if len(pending) == 0 {
                pendingAssistantIndex = -1
            }
        case "user", "system":
            closeBrokenPending()
            result = append(result, msg)
        }
    }
    closeBrokenPending()
    return result
}
```

实际实现时需要兼顾多工具并行调用：一个 assistant 带多个 `tool_calls` 时，后面可以连续出现多个 `tool` 消息，但必须全部属于这一组。

### 5.3 在构造 LLM 请求前增加最终断言

修改文件：`internal/agent/think.go`

在 `SanitizeMessages()` 后增加协议校验：

```go
messages = agenttools.SanitizeMessages(messages)
if problems := agenttools.ValidateToolMessageProtocol(messages); len(problems) > 0 {
    logger.Warnf(ctx, "[Agent][Round-%d] invalid tool protocol after sanitize: %v", round, problems)
    messages = agenttools.DropInvalidToolProtocolMessages(messages)
}
```

原则是：任何情况下都不能把非法 `role=tool` 发给 provider。宁可丢弃历史工具结果，也不要让整轮请求 400。

同时建议把日志中 “Calling LLM” 的消息摘要改为清洗后再打印，或同时打印清洗前后数量：

```text
messages_before_sanitize=25
messages_after_sanitize=23
invalid_tool_messages_dropped=2
```

这样下次排查能直接看到清洗是否生效。

### 5.4 上下文持久化改为工具组原子写入

修改文件：`internal/application/service/llmcontext/context_manager.go`、`internal/agent/observe.go`

新增批量追加接口，例如：

```go
AddMessages(ctx, sessionID string, messages []chat.Message) error
```

`appendToolResults()` 不再多次调用 `AddMessage()`，而是组装完整工具组后一次性写入：

```go
historyMessages := []chat.Message{assistantMsg}
historyMessages = append(historyMessages, toolMsgs...)
contextManager.AddMessages(ctx, sessionID, historyMessages)
```

这样可以减少 Redis 中出现半截工具调用链的概率。

如果短期不想改接口，也至少需要在写入后读取最近一组消息做校验；发现不完整时回滚或清除该组。

### 5.5 修复失败完成态

修改文件：`internal/application/service/session_agent_qa.go`、`internal/handler/session/qa.go`、`internal/handler/session/agent_stream_handler.go`

当 Agent 在第一轮 LLM 请求阶段收到 400 时，应把状态映射为失败，而不是 completed：

```text
completion_status = failed
finish_reason     = model_error 或 llm_error
failure_reason    = 原始错误摘要
allow_complete    = false 或按失败态约定处理
allow_indexing    = false
agent_steps       = 保留已有步骤；如果 0 步也不告警为 completed
```

前端 SSE 应收到明确失败事件，展示“模型请求失败，请重试或新建会话”，而不是空白完成。

### 5.6 DB fallback 和历史恢复也要过滤坏数据

修改文件：`internal/application/service/llmcontext/context_manager.go`

`rebuildAgentStepMessages()` 会根据数据库中的 `AgentSteps` 重建 assistant/tool 消息。这里也应复用同一套协议校验：

```go
result = agenttools.SanitizeMessages(result)
```

更理想的是在 `rebuildAgentStepMessages()` 内部就保证：

1. `ToolCall.ID` 为空时生成稳定 ID。
2. `ToolCall.Result == nil` 时不生成 `role=tool`。
3. 工具调用组不完整时转换为普通 assistant 摘要。

否则 Redis 清理后，DB fallback 仍可能把坏历史重新写回 Redis。

## 6. 建议测试用例

### 6.1 Sanitizer 单元测试

文件：`internal/agent/tools/sanitize_messages_test.go`

新增用例：

1. `tool` 没有 `tool_call_id`：必须被丢弃或转为非 tool 角色。
2. `tool_call_id` 只匹配很早之前的 assistant，但最近消息不是该 assistant：必须被丢弃或降级。
3. `assistant` 无 tool_calls，紧接 `assistant` 有 tool_calls，再接 `tool`：不能因为合并 assistant 而丢掉 tool_calls。
4. `assistant(tool_calls=[a,b])` 后只出现 `tool(a)`，缺少 `tool(b)`：整组不能以半合法状态发出。
5. `assistant(tool_calls=[a,b])` 后出现 `tool(c)`：`tool(c)` 必须被清理。
6. 多个连续合法 tool result：必须保留。

### 6.2 Agent 请求构造测试

文件：`internal/agent/observe_test.go` 或新增 `internal/agent/think_test.go`

构造 Redis 历史中包含坏 tool 消息，调用 `buildMessagesWithLLMContext()` + `SanitizeMessages()`，断言最终发送给 mock chat 的消息满足：

```text
不存在孤立 role=tool
所有 role=tool 都有 tool_call_id
每组 tool 都紧跟最近 assistant(tool_calls)
```

### 6.3 失败态持久化测试

文件：`internal/handler/session/qa_completion_test.go` 或 `internal/handler/session/agent_stream_handler_test.go`

模拟 LLM 返回 400，断言：

```text
completion_status != completed
allow_indexing == false
failure_reason != ""
content 不会被当作正常文档 artifact 注册
```

## 7. 建议实施顺序

1. 先清理受影响会话 Redis key，恢复用户当前会话可用性。
2. 修复 `SanitizeMessages()`，确保非法工具历史不会再打到 provider。
3. 增加发送前最终断言和清洗日志。
4. 将工具组写入上下文改为批量原子追加。
5. 修复 LLM 调用失败后的完成态持久化。
6. 补齐 sanitizer、Agent 请求构造、失败态持久化三类测试。

## 8. 最小可落地补丁范围

## 9. 结论

本次故障的本质是历史 Agent 工具调用消息进入下一轮 LLM 请求时，没有被严格重建为合法 OpenAI tool-calling 消息序列。Provider 的 400 是正确拦截，说明请求体中确实存在非法 `role=tool`。

短期通过清理 Redis 上下文可以解除当前会话阻塞；长期必须修复消息清洗、上下文原子写入和失败态持久化。修复完成后，即使历史缓存中混入坏数据，也应在发送模型前被丢弃或降级，不能再让用户请求因历史工具消息污染而直接失败。
