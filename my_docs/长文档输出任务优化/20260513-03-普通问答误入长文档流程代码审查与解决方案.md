# 普通问答误入长文档流程代码审查与解决方案

## 1. 审查背景

用户问题：

```text
河南省郑州市中原区桐柏路206号5楼32号的火灾警情登记表包含哪些信息？
```

这是一个典型的知识库事实查询问题，期望系统检索对应资料后直接回答登记表字段或内容摘要。但从截图和日志看，系统进入了“文档规划 / 长文档生成”体验，并生成了类似《火灾警情登记表信息说明》的章节大纲。

本次审查目标是结合日志和代码定位：为什么普通问答会进入 `knowledge_grounded_full_document` 长文档路径，并给出可落地的修复方案。

补充说明：用户提供的 `f6bb69135` 在当前本地仓库不是有效 Git 提交号；当前 HEAD 为 `7ef8da56`。本次审查按当前工作区代码和用户提供日志进行。

## 2. 结论摘要

这是一个“前端默认值 + 后端默认值 + 后端路由条件过宽”共同造成的问题。

前端 `inferDocumentOutputMode` 对普通 `normal` 意图默认返回 `full_document`。后端 `normalizeDocumentOutputModeForRequest` 在请求未显式传入输出模式时，也会把普通意图默认归一化为 `full_document`。进入 `AgentQA` 后，只要请求是 `full_document`，且智能体有有效本地知识库范围，`shouldUseKnowledgeGroundedFullDocumentGenerationPath` 就会直接返回 `true`，没有再判断用户问题是否真的在要求“生成完整文档/方案/报告”。

因此，普通问题被误标记为完整文档任务后，后端按长文档流程执行预算协商、大纲检索、大纲规划和章节生成，最终出现截图中的“文档规划 生成中”。

## 3. 日志证据链

### 3.1 普通问题进入 AgentQA

日志中用户原始问题是事实查询：

```text
session_agent_qa.go:68[AgentQA] | Start agent-based question answering ... query: 河南省郑州市中原区桐柏路206号5楼32号的火灾警情登记表包含哪些信息？
```

问题关键词是“包含哪些信息”，并没有“生成完整文档、输出技术方案、撰写报告、形成 Markdown”等长文档意图。

### 3.2 后端路由直接选择长文档路径

紧接着，后端选择了知识库支撑的完整文档生成路径：

```text
session_agent_qa.go:126[AgentQA] | [FullDocument][Router] selected=knowledge_grounded_full_document kb_scope=5
```

这说明在进入普通 agent 执行前，路由已经被长文档分支截获。

### 3.3 长文档预算协商被触发

随后出现长文档预算计算和协商：

```text
session_agent_qa.go:1697[resolveStaticDocumentGenerationBudget] | [DocumentBudget] source=capability ... outline_tokens=1536 section_tokens=4096 ...
remote_api.go:309[logRequest] | You are estimating safe generation budgets for a long markdown document
```

预算协商请求中明确写入：

```text

expected_document_type: long_markdown
expected_sections: 6
```

这不是问题根因，而是误入长文档路径后的结果。系统已经把一个查询问题包装成了 `long_markdown` 文档任务。

### 3.4 检索查询开始偏离用户意图

第一轮检索使用原问题，没有命中：

```text

Vector retrieval found 0 results

```

随后长文档流程继续使用模板化的大纲检索语句：

```text
query text: 请检索与该文档目标相关的项目背景、总体架构、核心能力、实施方案和保障措施。
```

这类查询适用于技术方案或项目方案，不适合“火灾警情登记表包含哪些信息”的事实问答。截图中的 8 章大纲也体现了长文档模板化倾向，而不是直接回答登记表字段。

## 4. 代码级根因

### 4.1 前端普通问题默认发送 `full_document`

位置：`frontend/src/views/chat/index.vue`

当前逻辑：

```js
const inferDocumentOutputMode = (query = '', intentHint = 'normal') => {
    const text = query.trim().toLowerCase();
    if (/(输出完整新版全文|给我完整文档|输出完整文档|完整新版全文|重新生成完整方案)/.test(text)) {
        return 'full_document';
    }
    if (intentHint === 'continue_document' || intentHint === 'revise_document') {
        return 'delta_only';
    }
    if (intentHint === 'regenerate_document') {
        return 'full_document';
    }
    return 'full_document';
};
```

问题在最后一行：普通 `normal` 意图默认返回 `full_document`。因此用户只要在聊天框输入普通问题，前端就会构造：

```js
document_output_mode: 'full_document'
```

而 `frontend/src/api/chat/streame.ts` 会在字段存在时原样放入请求体：

```ts
if (params.document_output_mode) {
  postBody.document_output_mode = params.document_output_mode;
}
```

### 4.2 后端归一化也默认普通请求为 `full_document`

位置：`internal/handler/session/chat_document_artifact.go`

当前逻辑：

```go
func normalizeDocumentOutputModeForRequest(outputMode string, intent string) string {
    switch strings.TrimSpace(outputMode) {
    case types.ChatDocumentOutputModeFull:
        return types.ChatDocumentOutputModeFull
    case types.ChatDocumentOutputModeDelta:
        return types.ChatDocumentOutputModeDelta
    }

    switch intent {
    case types.ChatDocumentIntentContinue, types.ChatDocumentIntentRevise:
        return types.ChatDocumentOutputModeDelta
    case types.ChatDocumentIntentRegenerate:
        return types.ChatDocumentOutputModeFull
    default:
        return types.ChatDocumentOutputModeFull
    }
}
```

即使前端不传 `document_output_mode`，后端也会把普通 `normal/create` 请求补成 `full_document`。这使 API 调用、旧前端或第三方 IM 入口也可能复现同类问题。

### 4.3 AgentQA 长文档路由只检查模式，不检查真实文档意图

位置：`internal/application/service/session_agent_qa.go`

当前路由入口：

```go
if shouldUseKnowledgeGroundedFullDocumentGenerationPath(req, agentConfig) {
    logger.Infof(ctx, "[FullDocument][Router] selected=knowledge_grounded_full_document kb_scope=%d", len(agentConfig.SearchTargets.GetAllKnowledgeBaseIDs()))
    return s.runKnowledgeGroundedFullDocumentGenerationPath(ctx, req, eventBus, summaryModel, agentConfig)
}
```

核心判断：

```go
func shouldUseKnowledgeGroundedFullDocumentGenerationPath(req *types.QARequest, agentConfig *types.AgentConfig) bool {
    if req == nil || req.DocumentOutputMode != types.ChatDocumentOutputModeFull {
        return false
    }
    if strings.TrimSpace(req.Query) == "" || strings.TrimSpace(req.BaseArtifactID) != "" {
        return false
    }
    if req.AutoContinue {
        return false
    }
    switch req.DocumentIntent {
    case "", types.ChatDocumentIntentNormal, types.ChatDocumentIntentRegenerate:
    default:
        return false
    }
    switch req.DocumentOperation {
    case "", types.ChatDocumentOperationCreate, types.ChatDocumentOperationRegenerate:
    default:
        return false
    }
    if !hasEffectiveLocalKnowledgeScope(req, agentConfig) {
        return false
    }
    if len(req.Attachments) > 0 || len(req.ImageURLs) > 0 || strings.TrimSpace(req.ImageDescription) != "" {
        return false
    }
    return true
}
```

只要满足以下条件就会进入长文档：

1. `DocumentOutputMode == full_document`。
2. 问题不为空。
3. 不是续写，不带基础文档。
4. 意图是空、normal 或 regenerate。
5. 有有效本地知识库范围。
6. 没有附件或图片。

用户日志完全满足这些条件：智能体加载了 5 个知识库，前端/后端又把请求标成 `full_document`，所以普通查询被路由到 `knowledge_grounded_full_document`。

### 4.4 单测覆盖固化了错误边界

位置：`internal/application/service/session_agent_qa_test.go`

当前测试只覆盖“完整技术方案 + 知识库范围”应进入长文档：

```go
req := &types.QARequest{
    Query:              "请输出完整的投标技术方案",
    DocumentOutputMode: types.ChatDocumentOutputModeFull,
}
```

缺少反例：

```text
河南省郑州市中原区桐柏路206号5楼32号的火灾警情登记表包含哪些信息？
```

也就是说，测试没有保护“事实查询不能因为带 full_document 就进入长文档”这个边界。

## 5. 影响范围

1. 所有普通聊天问题都有机会被前端标记为 `full_document`。
2. 使用智能推理 Agent 且关联了知识库时，普通问答更容易被 `knowledge_grounded_full_document` 截获。
3. 进入长文档路径后，会额外触发预算协商、大纲检索、章节生成，增加延迟和模型成本。
4. 检索 query 会从用户事实问题扩展为“项目背景、总体架构、实施方案”等模板化查询，答案相关性下降。
5. 用户界面会出现“文档规划/章节生成”而非普通回答，体验与预期明显不一致。

## 6. 修复原则

长文档路径不能只由 `DocumentOutputMode=full_document` 决定。`full_document` 最多表示输出通道或客户端期望，不能单独代表用户真实意图。

建议把触发条件收紧为：

```text
显式完整文档意图 + full_document 输出模式 + 非续写/非修订 + 有效知识库范围
```

其中“显式完整文档意图”应来自明确关键词或结构化入口，而不是普通 `normal` 意图默认值。

## 7. 推荐解决方案

### 7.1 前端：普通问题不再默认 `full_document`

修改 `frontend/src/views/chat/index.vue` 的 `inferDocumentOutputMode`。

建议逻辑：

```js
const hasExplicitFullDocumentIntent = (query = '') => {
    const text = query.trim().toLowerCase();
    if (!text) return false;
    return /(?:生成|输出|撰写|编写|写一份|形成|整理|给我).{0,20}(?:完整)?(?:文档|全文|方案|报告|技术方案|设计方案|投标方案|标书|markdown)/i.test(text) ||
        /(?:输出完整新版全文|给我完整文档|输出完整文档|完整新版全文|重新生成完整方案)/i.test(text);
};

const inferDocumentOutputMode = (query = '', intentHint = 'normal') => {
    if (intentHint === 'continue_document' || intentHint === 'revise_document') {
        return 'delta_only';
    }
    if (intentHint === 'regenerate_document') {
        return 'full_document';
    }
    if (hasExplicitFullDocumentIntent(query)) {
        return 'full_document';
    }
    return '';
};
```

`streame.ts` 已经只在字段 truthy 时才写入 `postBody.document_output_mode`，所以普通问题返回空字符串后，请求体不会携带该字段。

对本次问题，`包含哪些信息` 不会命中显式完整文档意图，因此前端不应再发送 `full_document`。

### 7.2 后端：归一化函数不要把 normal 默认成 full

修改 `internal/handler/session/chat_document_artifact.go`。

建议把函数签名扩展为：

```go
func normalizeDocumentOutputModeForRequest(outputMode string, intent string, query string) string
```

建议逻辑：

```go
func normalizeDocumentOutputModeForRequest(outputMode string, intent string, query string) string {
    switch strings.TrimSpace(outputMode) {
    case types.ChatDocumentOutputModeFull:
        return types.ChatDocumentOutputModeFull
    case types.ChatDocumentOutputModeDelta:
        return types.ChatDocumentOutputModeDelta
    }

    switch intent {
    case types.ChatDocumentIntentContinue, types.ChatDocumentIntentRevise:
        return types.ChatDocumentOutputModeDelta
    case types.ChatDocumentIntentRegenerate:
        return types.ChatDocumentOutputModeFull
    }

    if isExplicitFullDocumentCreationQuery(query) {
        return types.ChatDocumentOutputModeFull
    }
    return ""
}
```

对应调用位置从：

```go
reqCtx.documentOutputMode = normalizeDocumentOutputModeForRequest(request.DocumentOutputMode, reqCtx.documentIntent)
```

调整为：

```go
reqCtx.documentOutputMode = normalizeDocumentOutputModeForRequest(request.DocumentOutputMode, reqCtx.documentIntent, reqCtx.query)
```

这样可以避免 API 或旧前端未传字段时被后端自动升级为长文档任务。

### 7.3 后端路由：增加长文档意图闸门

仅修前端和 handler 默认值还不够，因为外部调用方仍可能显式传入 `document_output_mode=full_document`。`AgentQA` 路由必须有自己的保护。

建议在 `internal/application/service/session_agent_qa.go` 增加共享判断：

```go
var fullDocumentCreationIntentRE = regexp.MustCompile(`(?i)(生成|输出|撰写|编写|写一份|形成|整理|给我).{0,30}(完整)?(文档|全文|方案|报告|技术方案|设计方案|投标方案|标书|markdown)`)

func isExplicitFullDocumentCreationQuery(query string) bool {
    trimmed := strings.TrimSpace(query)
    if trimmed == "" {
        return false
    }
    return fullDocumentCreationIntentRE.MatchString(trimmed)
}
```

然后在两个完整文档路由判断中增加闸门：

```go
if req.DocumentIntent != types.ChatDocumentIntentRegenerate && !isExplicitFullDocumentCreationQuery(req.Query) {
    return false
}
```

需要同时覆盖：

1. `shouldUseKnowledgeGroundedFullDocumentGenerationPath`
2. `shouldUseDedicatedFullDocumentGenerationPath`

这样即使请求体误带 `full_document`，普通问题也会落回通用 Agent 或知识库问答链路。

### 7.4 后端：普通请求不要携带文档上下文

当前 `prepareDocumentRequest` 默认返回：

```go
intent: normal
operation: create
```

即使不是文档任务，`QARequest` 也会携带 `DocumentIntent=normal`、`DocumentOperation=create`。这对长文档路由误判有放大作用，也会让通用 Agent final synthesis 看到不必要的 document context。

建议在 handler 组装阶段收口：

```go
if reqCtx.documentOutputMode == "" &&
    reqCtx.documentIntent == types.ChatDocumentIntentNormal &&
    reqCtx.documentOperation == types.ChatDocumentOperationCreate &&
    reqCtx.baseArtifact == nil &&
    !reqCtx.autoContinue {
    reqCtx.documentIntent = ""
    reqCtx.documentOperation = ""
}
```

这样普通问答在服务层就是纯问答请求，而不是“normal/create 文档请求”。

### 7.5 检索策略：长文档路径不要在零命中时继续模板化跑偏

本次日志中，精确问题零命中后，长文档路径继续使用：

```text
请检索与该文档目标相关的项目背景、总体架构、核心能力、实施方案和保障措施。
```

这类泛化 query 对技术方案有效，但对事实查询会跑偏。路由修复后该问题会大幅减少，但建议额外加保护：

1. `buildKnowledgeGroundedOutlineQueries` 生成模板化 query 前，先确认 `isExplicitFullDocumentCreationQuery(req.Query)`。
2. 如果不是明确长文档任务，不要构造“项目背景/总体架构/实施方案”类 query。
3. 对知识库长文档路径，如果首轮所有 query 都零命中，应尽快返回 `local_knowledge_not_found` 或回退普通 QA，而不是继续规划无证据文档。

该项为 P1，可以在 P0 路由收紧后单独实施。

## 10. 建议实施顺序

1. 先改前端 `inferDocumentOutputMode`，立即减少 Web 端误触发。
2. 同步改后端 `normalizeDocumentOutputModeForRequest`，防止未传字段时后端兜底误判。
3. 再改 `shouldUseKnowledgeGroundedFullDocumentGenerationPath` 和 `shouldUseDedicatedFullDocumentGenerationPath`，防止外部调用方显式误传 `full_document`。
4. 补 P0 回归测试，确保“包含哪些信息/是什么/有哪些字段”类问题不会再进入长文档路径。
5. 最后优化长文档 outline 检索模板，降低零命中后跑偏的概率。

## 11. 风险与兼容性

1. 旧行为中，普通问题默认都会创建 full document artifact。修复后，只有明确长文档意图才会创建完整文档 artifact。这符合用户预期，但如果已有业务依赖“所有回答都以文档 artifact 注册”，需要单独确认。
2. API 调用方如果想生成完整文档，应显式传 `document_output_mode=full_document`，并在 query 中表达清楚“生成/输出完整文档/方案/报告”。
3. 正则意图判断只能作为第一阶段方案。后续如果有明确的“文档模式”按钮或结构化入口，建议前端传入更明确的 `document_task=true` 或新增 `DocumentIntentCreateFull`，减少自然语言正则误判。

## 12. 最终建议

本问题应按 P0 处理。它不是模型回答质量问题，也不是知识库检索单点问题，而是请求被过早路由到了错误任务类型。

最小可行修复是：

1. 前端普通问题不再默认发送 `full_document`。
2. 后端普通请求不再默认补成 `full_document`。
3. AgentQA 长文档路由必须要求显式完整文档意图。

修复后，本次问题应回到普通知识库问答流程，直接回答“火灾警情登记表包含哪些信息”，而不是进入“文档规划/章节生成”。
