# 长文档模型预算动态协商设计方案

## 一、背景与目标

当前长文档生成链路中，部分与模型能力相关的限制值仍然是服务端固定值，例如：

- `fullDocumentSectionMinCompletionTokens = 2048`
- `fullDocumentSectionEvidenceLimit = 6`
- `Conversation.Summary.MaxCompletionTokens`
- `EmbeddingTopK` / `RerankTopK`
- `AgentConfig.LLMCallTimeout`
- `AgentConfig.MaxContextTokens`

这些固定值能提供稳定兜底，但它们无法准确适配不同模型的上下文窗口、输出上限、推理速度、thinking 行为、供应商参数限制和实际可承载 token 数。对于完整长文档生成，固定预算过小会导致章节过短或频繁进入 `partial + continuing`；固定预算过大则可能触发供应商报错、超时、成本过高或首 token 等待过久。

本方案目标是在对话初期增加一次“模型预算协商”流程，由系统结合模型声明能力、运行时探测和模型自评建议，生成本轮长文档生成的动态预算。协商成功时使用动态预算；协商失败、超时或结果不可信时回退到现有固定兜底值，保证可用性不下降。

## 二、设计原则

1. 协商结果只作为服务端预算输入，不允许模型直接控制安全边界。
2. 所有动态值必须经过服务端 clamp，不能超过租户配置、模型供应商上限、系统硬上限和成本上限。
3. 协商失败必须无感回退到固定值，不能阻断用户正常对话。
4. 长文档预算优先按“章节级”生效，避免影响普通问答链路。
5. 协商应尽量早发生，但不能明显增加首轮等待；需要短超时、可缓存、可复用。
6. 协商过程和结果要可观测，便于复盘为什么某次文档生成预算偏大或偏小。

## 三、需要动态协商的预算项

### 1. 输出 token 预算

建议拆成三类，而不是继续只依赖一个 `MaxCompletionTokens`：

| 预算项 | 当前来源 | 动态含义 | 兜底值 |
| --- | --- | --- | --- |
| `outline_max_completion_tokens` | `min(maxCompletionTokens, 1024)` | 大纲阶段最大输出 token | 1024 |
| `section_max_completion_tokens` | `effectiveFullDocumentSectionMaxCompletionTokens` | 单章节正文最大输出 token | `max(config, 2048)`，默认 4096 |
| `continuation_max_completion_tokens` | `Summary.MaxCompletionTokens` | 自动续写单轮最大输出 token | 默认 4096 |

协商时应根据文档目标、计划章节数、模型上下文窗口和供应商输出上限，给出章节级 token 建议。例如技术方案类长文档每章可建议 2048-4096，摘要类文档可建议 1024-2048。

### 2. 检索证据预算

| 预算项 | 当前来源 | 动态含义 | 兜底值 |
| --- | --- | --- | --- |
| `outline_evidence_top_k` | 6 | 大纲规划证据数量 | 6 |
| `section_evidence_top_k` | `fullDocumentSectionEvidenceLimit` | 每章节证据数量 | 6 |
| `continuation_evidence_top_k` | 6 | 自动续写证据数量 | 6 |

动态策略应考虑知识库数量、查询复杂度、章节目标粒度和模型上下文窗口。上下文窗口较大且知识库证据充分时，可以提高到 8-12；窗口较小或成本敏感时，保持 4-6。

### 3. 上下文与拼接预算

| 预算项 | 当前来源 | 动态含义 | 兜底值 |
| --- | --- | --- | --- |
| `section_context_char_limit` | prompt 拼接逻辑中的隐式截断 | 每章可带入的证据字符上限 | 现有截断规则 |
| `document_snapshot_char_limit` | artifact 上下文构建 | 续写时带入的文档快照长度 | 现有 head/tail 策略 |
| `max_context_tokens` | `AgentConfig.MaxContextTokens` | Agent 运行时上下文压缩目标 | 现有配置或 200000 |

这部分不建议完全交给模型决定。模型可以给出建议，但服务端必须按 token 估算器和供应商上下文窗口裁剪。

### 4. 超时与阶段节奏

| 预算项 | 当前来源 | 动态含义 | 兜底值 |
| --- | --- | --- | --- |
| `outline_first_token_timeout` | progress heartbeat + LLM timeout | 大纲阶段首 token 等待阈值 | 30 秒或现有配置 |
| `section_call_timeout` | `LLMCallTimeout` | 单章节 LLM 调用超时 | 120 秒或配置值 |
| `progress_heartbeat_interval` | 8 秒 | 规划/章节生成进度心跳 | 8 秒 |

超时值可以基于模型历史速度动态调整，但不能无限放大。建议限制在 30-300 秒之间。

## 四、协商输入来源

协商不是单纯“问模型你能写多少 token”，而是综合三类信息。

### 1. 静态模型能力

优先使用系统已知元数据：

- 模型 ID、供应商、部署方式。
- 上下文窗口大小。
- 最大输出 token 上限。
- 是否支持 `max_completion_tokens`。
- 是否支持 `thinking=false`。
- 是否支持流式输出。
- 是否存在供应商特定限制，例如 Qwen thinking 模型、OpenAI 兼容接口差异。

推荐新增 `ModelCapabilityProvider` 或扩展现有模型配置，提供统一能力结构：

```go
type ModelCapability struct {
    ModelID                 string
    Provider                string
    ContextWindowTokens     int
    MaxOutputTokens         int
    SupportsStreaming       bool
    SupportsThinkingControl bool
    DefaultThinkingEnabled  bool
    RecommendedTimeoutSec   int
}
```

### 2. 运行时探测

对模型能力缺失或供应商上限不可靠的模型，可以做轻量探测：

- 使用很短 prompt 检查流式首 token 延迟。
- 使用低 token 请求确认 `max_completion_tokens` 是否被接受。
- 如果供应商返回上限错误，记录实际允许范围。
- 探测结果按模型 ID、租户、endpoint 缓存。

探测必须是短超时、低成本、可跳过的。不能每次对话都进行重探测。

### 3. 模型自评协商

在长文档生成开始前，可以发送一次短 prompt，让模型基于任务目标和计划章节规模给出预算建议。该调用必须要求 JSON 输出，并且服务端只接受白名单字段。

示例 prompt：

```text
You are estimating safe generation budgets for a long markdown document.
Return JSON only. Do not write the document.

Known constraints:
- model_context_window_tokens: 32768
- model_max_output_tokens: 8192
- user_goal: ...
- expected_document_type: technical proposal
- expected_sections: 8
- local_knowledge_available: true

Return:
{
  "outline_max_completion_tokens": number,
  "section_max_completion_tokens": number,
  "continuation_max_completion_tokens": number,
  "section_evidence_top_k": number,
  "section_call_timeout_seconds": number,
  "reason": string
}
```

服务端解析后必须做严格校验：

- JSON 必须可解析。
- 所有数字必须为整数。
- 不接受未声明字段作为行为依据。
- 不允许模型建议超过能力上限。
- 不允许低于系统最低可用值。
- `reason` 只用于日志，不参与控制。

## 五、预算决策流程

完整流程建议如下：

```text
用户请求进入
  ↓
识别 document_output_mode=full_document 或长文档续写
  ↓
读取租户配置、Agent 配置、模型静态能力
  ↓
查询模型能力缓存 / 探测缓存
  ↓
如果缓存缺失且允许探测：执行短超时能力探测
  ↓
执行预算协商调用（可选，短超时）
  ↓
服务端 clamp 与可信度校验
  ↓
生成 DocumentGenerationBudget
  ↓
写入 req/runtime context/generation_run extra
  ↓
后续 outline、section、continuation 全部读取该预算
  ↓
失败则回退 fallback budget
```

推荐新增统一预算结构：

```go
type DocumentGenerationBudget struct {
    Source                         string // negotiated / capability_cache / fallback
    ModelID                        string
    OutlineMaxCompletionTokens      int
    SectionMaxCompletionTokens      int
    ContinuationMaxCompletionTokens int
    OutlineEvidenceTopK             int
    SectionEvidenceTopK             int
    ContinuationEvidenceTopK        int
    SectionCallTimeoutSeconds       int
    ProgressHeartbeatSeconds        int
    ContextWindowTokens             int
    MaxOutputTokens                 int
    NegotiationReason               string
}
```

预算来源优先级：

1. 服务端安全硬上限。
2. 租户/管理员显式配置。
3. 模型能力元数据。
4. 模型协商建议。
5. 运行时历史统计。
6. 固定兜底值。

注意：优先级不是简单覆盖关系。最终值应由 clamp 生成，例如：

```text
final_section_tokens = clamp(
  negotiated_section_tokens,
  min = fallback_section_min_tokens,
  max = min(tenant_max_completion_tokens, model_max_output_tokens, system_hard_max)
)
```

如果租户配置低于系统长文档最小值，需要明确策略：

- 管理员配置是硬限制时，不能突破，应降级为 `partial + continuing`。
- 当前实现更偏向“长文档最低可用保障”，即低于 2048 时提升到 2048。
- 建议后续增加配置项区分 `tenant_hard_limit` 与 `tenant_default_limit`，避免语义冲突。

## 六、协商失败兜底策略

协商失败包括以下情况：

- 协商调用超时。
- 模型返回非 JSON。
- JSON 字段缺失或类型错误。
- 建议值超过模型/租户/系统硬上限。
- 建议值明显不合理，例如 section token 小于 512 或 top-k 大于 50。
- 模型不支持该探测调用或接口返回参数错误。

失败时应使用固定兜底：

```go
fallback := DocumentGenerationBudget{
    Source:                         "fallback",
    OutlineMaxCompletionTokens:      1024,
    SectionMaxCompletionTokens:      effectiveFullDocumentSectionMaxCompletionTokens(cfg),
    ContinuationMaxCompletionTokens: effectiveFullDocumentSectionMaxCompletionTokens(cfg),
    OutlineEvidenceTopK:             6,
    SectionEvidenceTopK:             fullDocumentSectionEvidenceLimit,
    ContinuationEvidenceTopK:        6,
    SectionCallTimeoutSeconds:       120,
    ProgressHeartbeatSeconds:        8,
}
```

需要注意：协商失败不应返回错误给用户。前端最多可以在开发日志或调试面板看到预算来源是 fallback。

## 七、与当前代码的集成点

### 1. `session_agent_qa.go`

当前应替换的硬编码点包括：

- `fullDocumentSectionEvidenceLimit`
- `fullDocumentSectionMinCompletionTokens`
- `effectiveFullDocumentSectionMaxCompletionTokens`
- `generateValidatedFullDocumentOutline` 的 `MaxCompletionTokens: min(maxCompletionTokens, 1024)`
- `retrieveKnowledgeGroundedFullDocumentEvidence(..., 6)`
- `retrieveKnowledgeGroundedFullDocumentEvidence(..., fullDocumentSectionEvidenceLimit)`
- 章节 `ChatStream` 的 `MaxCompletionTokens`

建议改为：

```go
budget := negotiateDocumentGenerationBudget(ctx, req, chatModel, agentConfig, s.cfg, documentProfile)

outlineOptions.MaxCompletionTokens = budget.OutlineMaxCompletionTokens
sectionOptions.MaxCompletionTokens = budget.SectionMaxCompletionTokens
retrieveKnowledgeGroundedFullDocumentEvidence(..., budget.SectionEvidenceTopK, ...)
```

### 2. `ChatDocumentGenerationRun`

长文档自动续写需要跨轮复用同一预算，否则首轮和续写轮可能预算不一致。建议在 generation run 中持久化预算 JSON：

```go
BudgetJSON types.JSON `json:"budget_json" gorm:"type:json"`
```

如果不想立刻改数据库，也可以先放在 run extra 或 complete extra 中，但长期应持久化为结构化字段。

### 3. `agent_complete` payload

complete 事件建议携带预算摘要，便于调试和前端展示：

```json
{
  "budget": {
    "source": "negotiated",
    "section_max_completion_tokens": 4096,
    "section_evidence_top_k": 8
  }
}
```

默认前端不需要展示给普通用户，但日志和诊断文档可以使用。

### 4. 日志与可观测性

每次预算决策应打印结构化日志：

```text
[DocumentBudget] source=negotiated model=qwen section_tokens=4096 section_top_k=8 timeout=180 reason="technical proposal with 8 sections"
```

协商失败时记录失败原因：

```text
[DocumentBudget] negotiation_failed model=xxx err=json_parse_failed fallback_section_tokens=2048
```

## 八、Prompt 与模型协商格式设计

建议新增一个专用 prompt builder：

```go
func buildDocumentBudgetNegotiationMessages(req *types.QARequest, profile DocumentProfile, capability ModelCapability) []chat.Message
```

`DocumentProfile` 可由用户请求、大纲或初步分类得到：

```go
type DocumentProfile struct {
    Goal                  string
    OutputMode            string
    EstimatedDocumentType  string // technical_proposal / report / summary / manual
    ExpectedSectionCount   int
    KnowledgeGrounded      bool
    EvidenceScopeKBCount   int
    AutoContinue           bool
}
```

协商 prompt 必须强调：

1. 只输出 JSON。
2. 不输出文档正文。
3. 不能建议超过给定模型能力。
4. 预算建议以“安全完成任务”为目标，而不是越大越好。
5. 如果不确定，返回保守值和原因。

服务端还应使用 schema 或手写解析器校验字段。

## 九、缓存策略

协商结果建议分两层缓存：

### 1. 模型能力缓存

按以下 key 缓存：

```text
tenant_id + model_id + provider + endpoint_hash
```

缓存内容：上下文窗口、最大输出、thinking 控制能力、流式能力、探测时间。

### 2. 预算建议缓存

预算建议不应只按模型缓存，因为不同任务差异很大。建议按粗粒度 profile 缓存：

```text
tenant_id + model_id + document_type + section_count_bucket + knowledge_grounded
```

缓存时间可以较短，例如 6-24 小时。

## 十、安全边界与风险控制

### 1. 防止模型自我放大预算

模型可能建议过大的 token 或 top-k。必须使用服务端 clamp：

- `section_max_completion_tokens <= model_max_output_tokens`
- `section_max_completion_tokens <= system_hard_max_output_tokens`
- `section_evidence_top_k <= system_hard_max_evidence_top_k`
- `section_call_timeout_seconds <= system_hard_timeout_seconds`

### 2. 防止预算过小导致短文档

模型也可能建议过小预算。完整文档模式应有最低可用值：

- outline 不低于 512。
- section 不低于 2048，或至少不低于当前固定兜底。
- section evidence 不低于 4。

如果租户强制限制更低，应明确记录 `budget_limited_by_tenant`，并将最终状态更倾向于 `partial + continuing`。

### 3. 防止协商增加首包等待

协商调用必须满足：

- timeout 建议 3-5 秒。
- 可被缓存跳过。
- 协商期间前端发 `planning` synthetic thought，例如“正在评估模型预算”。
- 协商失败立即 fallback。

### 4. 防止供应商参数错误

部分模型不支持 `max_completion_tokens` 或 thinking 控制。预算协商不能只听模型输出，还要结合供应商适配层能力。参数错误应被记录到模型能力缓存，后续避免重复失败。

## 十一、分阶段实施方案

### 阶段 1：预算结构与兜底统一

目标：不引入真实模型协商，只先把硬编码值集中到一个预算结构中。

改动：

1. 新增 `DocumentGenerationBudget`。
2. 新增 `fallbackDocumentGenerationBudget(cfg)`。
3. `runDedicatedFullDocumentGenerationPath`、`runKnowledgeGroundedFullDocumentGenerationPath`、`runKnowledgeGroundedDocumentContinuationRunPath` 全部从 budget 读取 token 与 top-k。
4. complete extra 中输出 budget summary。
5. 测试覆盖 fallback budget。

收益：先消除散落硬编码，为后续协商做铺垫。

### 阶段 2：静态模型能力接入

目标：基于模型配置和供应商能力做第一次动态裁剪。

改动：

1. 新增或扩展模型能力 provider。
2. 读取上下文窗口、最大输出 token、thinking 支持等。
3. budget fallback 经过 capability clamp。
4. 对未知模型继续使用阶段 1 兜底。

收益：避免对小模型设置过大 token，也避免对大模型预算过保守。

### 阶段 3：短超时模型协商

目标：在长文档对话初期让模型给出任务级预算建议。

改动：

1. 新增 `negotiateDocumentGenerationBudget`。
2. 新增 JSON prompt 与解析校验。
3. 设置 3-5 秒超时。
4. 协商失败回退 fallback。
5. 记录 `source=negotiated/fallback`。

收益：同一个模型可根据任务类型动态选择章节长度和证据数量。

### 阶段 4：运行时反馈闭环

目标：用真实生成结果修正预算。

可采集指标：

- 首 token 延迟。
- 单章输出 token 估算。
- 是否出现 `finish_reason=length`。
- 是否频繁进入 `partial + continuing`。
- 每章证据命中数量。
- 每章平均生成耗时。

策略：

- 如果多次 length 截断，下次提高 section token 或降低每轮章节数。
- 如果每章都很短且 stop，保持或降低预算。
- 如果首 token 过慢，降低 evidence top-k 或关闭 thinking。
- 如果证据不足，增加 section evidence top-k 或引导用户补知识库。

## 十二、推荐的数据结构与接口草案

```go
type DocumentBudgetNegotiator interface {
    Negotiate(ctx context.Context, input DocumentBudgetInput) (DocumentGenerationBudget, error)
}

type DocumentBudgetInput struct {
    Request       *types.QARequest
    AgentConfig   *types.AgentConfig
    ModelID       string
    ModelName     string
    Capability    ModelCapability
    Profile       DocumentProfile
    TenantConfig  *config.Config
}
```

预算生成伪代码：

```go
func negotiateDocumentGenerationBudget(ctx context.Context, input DocumentBudgetInput) DocumentGenerationBudget {
    fallback := fallbackDocumentGenerationBudget(input.TenantConfig)
    capability := loadModelCapability(input)
    fallback = clampBudget(fallback, capability, input.TenantConfig)

    if !shouldNegotiate(input) {
        return fallback.WithSource("fallback")
    }

    negotiated, err := callBudgetNegotiationModel(ctx, input, fallback, capability)
    if err != nil {
        return fallback.WithSource("fallback")
    }

    normalized, err := validateAndClampBudget(negotiated, fallback, capability, input.TenantConfig)
    if err != nil {
        return fallback.WithSource("fallback")
    }
    normalized.Source = "negotiated"
    return normalized
}
```

## 十三、测试方案

### 单元测试

1. 协商成功：模型返回合法 JSON，最终 budget 使用模型建议但经过 clamp。
2. 协商返回非 JSON：回退 fallback。
3. 协商超时：回退 fallback。
4. 协商建议过大：被 clamp 到模型或系统上限。
5. 协商建议过小：被提升到完整文档最低可用值。
6. 未知模型能力：使用 fallback。
7. 租户低 token 配置：根据 hard/default 语义验证是否允许突破。

### 服务层回归测试

1. `runKnowledgeGroundedFullDocumentGenerationPath` 的章节 `ChatStream` 使用 budget token。
2. 章节检索使用 budget top-k。
3. 自动续写轮复用 generation run 中的 budget。
4. 协商失败不影响文档生成。

### 集成测试

1. 大模型：预算提高，首轮章节更完整。
2. 小模型：预算被 clamp，不触发供应商参数错误。
3. 慢模型：协商超时后 fallback，前端仍能看到 planning 进度。

## 十四、与当前修复链路的关系

当前 P0/P1 已经提供了固定兜底：

- P0 保证不会错误完成。
- P1 保证章节逐章生成，并提升最小章节 token 和证据数量。

动态协商不应替代这两层保护，而是在它们之上做优化：

- 协商成功：根据模型能力和任务目标提高完整输出质量。
- 协商失败：回到 P1 固定预算。
- 任何情况下：P0 完整性闸门仍然最终裁决是否可以 `completed`。

## 十五、建议结论

可以在对话初期与模型进行预算协商，但不能把协商结果作为直接执行值。推荐采用“模型能力元数据 + 短超时协商 + 服务端 clamp + 固定兜底”的组合方案。

最优落地路径是先把当前散落硬编码收敛成 `DocumentGenerationBudget`，再逐步接入模型能力和协商调用。这样即使协商功能暂时失败，现有长文档生成能力也不会退化；协商成功时，则可以针对不同模型和不同文档任务动态调整 token、证据数量、上下文窗口和超时，提升完整输出质量。
