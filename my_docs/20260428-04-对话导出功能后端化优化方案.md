# 20260428-04 对话导出功能后端化优化方案

## 1. 背景与结论

针对当前对话框中的导出功能进行了实现审查，结论如下：

1. 现状并不是“一个统一的导出功能异常”，而是四种导出格式分别走了不同实现路径。
2. Markdown 是唯一完全不依赖外部系统命令、也不依赖复杂浏览器转换链路的导出格式，因此稳定性最高。
3. PDF 和 Word 虽然已经有后端接口，但当前实现强依赖运行环境中安装 `wkhtmltopdf` 与 `pandoc`，环境不满足时会自动回退到前端降级实现。
4. XLSX 当前没有后端实现，完全依赖前端 `xlsx` 包和浏览器下载能力。
5. 从可维护性、稳定性、可观测性和部署一致性角度看，除 Markdown 外，其余格式应尽快统一收口到后端实现；若条件允许，Markdown 也建议纳入同一套后端导出协议。

结论：

可以后端实现，而且当前仓库已经具备较好的基础。推荐将导出能力重构为“后端统一生成，前端只负责发起请求与下载结果”的单一模式。

## 3. 当前实现现状

### 3.1 前端入口现状

`ExportDropdown.vue` 中直接根据用户选择分发到四个导出函数：

1. `exportAsPDF`
2. `exportAsMarkdown`
3. `exportAsWord`
4. `exportAsXLSX`

这意味着前端实际承担了“格式路由器”的角色，而不是单纯的下载触发器。

### 3.2 四种格式的真实链路

#### Markdown

当前链路：

1. 前端直接创建 Blob。
2. 浏览器直接下载 `.md` 文件。

特点：

1. 无后端依赖。
2. 无外部系统命令依赖。
3. 无 HTML 转换或格式重建。

这是当前最稳定的一条导出链路。

#### PDF

当前链路：

1. 前端调用 `/api/v1/export/document`。
2. 后端将 Markdown 转为 HTML。
3. 后端调用 `wkhtmltopdf` 生成 PDF。
4. 如果后端失败，则前端自动回退到 `html2pdf.js`。

问题在于：

1. 成功与否取决于后端环境是否安装 `wkhtmltopdf`。
2. 后端失败后会悄悄切回前端生成，导致行为不可预测。
3. 前端 PDF 生成对复杂表格、图片、分页、代码块、字体和跨域资源更敏感。

#### Word

当前链路：

1. 前端调用 `/api/v1/export/document`。
2. 后端调用 `pandoc` 直接将 Markdown 转为 DOCX。
3. 如果后端失败，则前端自动回退到 `html-docx-js-typescript`。

问题在于：

1. 成功与否取决于后端环境是否安装 `pandoc`。
2. 前端降级方案生成的 DOCX 兼容性、样式完整度、复杂内容保真度都弱于后端转换。
3. 前端和后端产物不是同一标准，导致用户结果不稳定。

#### XLSX

当前链路：

1. 完全不走后端。
2. 前端通过正则式和字符串扫描提取 Markdown 表格。
3. 前端调用 `XLSX.writeFile` 直接生成本地文件。

问题在于：

1. 后端没有 XLSX 能力，无法在服务端做统一控制。
2. 当前表格提取逻辑只覆盖最简单的 `| ... |` Markdown 表格。
3. 无法识别代码块中的伪表格、复杂列表中的表格、标题结构、非表格正文等更完整语义。
4. 一旦浏览器环境、WebView 环境或沙箱环境对文件下载能力有限，XLSX 会直接失败或行为异常。

## 4. 已确认的核心问题

### 4.1 技术路径分裂

当前导出能力不是一个系统，而是三套系统拼接：

1. Markdown：纯前端本地下载。
2. PDF / DOCX：后端优先，失败后前端兜底。
3. XLSX：纯前端表格解析。

这种设计的直接问题是：

1. 可用性依赖不同层。
2. 质量由不同引擎决定。
3. 错误路径不一致。
4. 测试面膨胀。
5. 出问题时很难定位是前端、后端还是环境依赖。

### 4.2 后端能力存在，但前端没有真正消费能力探测结果

前端工具层里虽然实现了 `getExportCapabilities`，但当前导出组件并没有消费这套能力结果来控制菜单显示、禁用状态或错误提示。

后果是：

1. 用户总能看到 PDF / Word / XLSX 入口。
2. 即使后端能力缺失，界面也不会提前提示。
3. 用户只能在点击后才发现失败。

### 4.3 后端导出依赖运行环境，不具备“默认可用”特性

当前后端 PDF / DOCX 实现依赖：

1. `wkhtmltopdf`
2. `pandoc`

虽然 `docker/Dockerfile.app` 已安装这两个命令，但这并不代表所有运行环境都具备：

1. 本地裸机开发环境可能没有。
2. 某些历史部署镜像可能没有。
3. 某些桌面端或定制部署环境可能没有。

本次在当前环境中检查时，这两个命令均未找到，说明“后端优先”在当前环境里实际上会直接落入前端降级路径。

### 4.4 XLSX 当前设计本质上不是“文档导出”，而是“前端文本表格抽取”

现有 `extractTablesFromMarkdown` 的思路过于简化，只能解决最基本场景：

1. 逐行扫描。
2. 识别以 `|` 开始并以 `|` 结束的行。
3. 跳过 `|---|---|` 这类分隔行。

这会带来几个明显缺陷：

1. 文本主体无法结构化保留。
2. 多个表格之间的标题、上下文说明全部丢失。
3. 非标准 Markdown 表格会误判或漏判。
4. 无法为不同逻辑区块生成稳定 sheet。

### 4.5 自动前端降级掩盖真实故障

PDF / DOCX 当前策略是：

1. 先请求后端。
2. 后端失败后自动回退前端。

这会造成两个问题：

1. 用户看到的是“有时成功、有时失败、有时格式错乱”，但不知道实际走的是哪条链路。
2. 运维侧无法快速判断问题是接口故障、命令缺失、内容异常，还是浏览器导出兼容性问题。

这类“静默降级”在导出场景里不适合长期保留。

## 5. 是否适合后端实现

结论：适合，而且当前仓库已经具备落地条件。

### 5.1 已具备的基础

#### 已有后端接口骨架

`internal/handler/export.go` 已经提供：

1. `/api/v1/export/document`
2. `/api/v1/export/capabilities`
3. `/api/v1/export/html`

说明导出领域对象、路由分组、错误处理和下载响应头都已经成型。

#### 已有 Markdown 渲染基础

`internal/utils/export/markdown.go` 已经用 `goldmark` 完成：

1. GFM 表格支持。
2. 代码高亮。
3. HTML 样式包装。

这意味着后端已经具备“统一内容中间层”的基础能力。

#### 已有 XLSX 依赖

根模块 `go.mod` 中已包含：

1. `github.com/xuri/excelize/v2`

这说明后端实现 XLSX 不需要额外引入新的重量级依赖，直接可以复用现有依赖栈。

### 5.2 为什么后端更适合承担导出职责

后端化后可以统一处理以下问题：

1. 文件生成逻辑统一。
2. 依赖检查统一。
3. 错误码和错误消息统一。
4. 日志与观测统一。
5. 内容结构解析统一。
6. 不再受浏览器下载 API 差异影响。
7. 可以逐步扩展为异步导出、大文件导出、审计留痕。

## 6. 推荐优化方向

推荐目标是：

**将对话导出收敛为“后端统一生成 + 前端纯下载”的单一路径。**

### 6.1 目标架构

建议导出体系统一为：

1. 前端只负责发起导出请求、展示状态、下载结果。
2. 后端负责 Markdown 解析、格式转换、二进制生成、能力探测和错误返回。
3. 所有格式通过同一个导出服务层实现，不再按格式散落在前端工具函数中。

目标数据流：

1. 前端获取原始 Markdown 内容。
2. 调用统一导出接口，携带 `format`、`content`、`filename_prefix`。
3. 后端按格式策略生成文件。
4. 后端直接返回文件流。
5. 前端只做下载，不做格式降级生成。

### 6.2 格式支持建议

建议接口支持：

1. `markdown`
2. `pdf`
3. `docx`
4. `xlsx`

这样做有两个好处：

1. 所有导出都经过同一套协议和能力探测。
2. 前端实现可以极度简化，只保留一个“调用接口并下载 Blob”的统一函数。

---

## 7. 推荐后端方案设计

### 7.1 服务分层建议

建议在后端形成以下结构：

1. `handler`：参数校验、鉴权、响应头设置。
2. `service/export`：导出编排层，负责按格式路由到具体策略。
3. `utils/export`：纯格式转换和文件生成工具。
4. `parser`：Markdown 结构提取能力，供 XLSX 与其他结构化导出复用。

建议抽象为：

1. `ExportFormat`
2. `Exporter`
3. `CapabilitiesProvider`

这样可避免后续继续在 `switch format` 中堆逻辑。

### 7.2 API 设计建议

建议扩展现有接口，而不是另起新接口。

#### `POST /api/v1/export/document`

请求建议支持：

1. `content`
2. `format`，支持 `markdown/pdf/docx/xlsx`
3. `filename_prefix`
4. 可选 `options`

可选 `options` 可预留：

1. PDF 页面尺寸。
2. 是否导出目录。
3. XLSX 是否将非表格正文落到概览页。
4. DOCX 是否应用 reference docx 模板。

#### `GET /api/v1/export/capabilities`

建议返回：

1. 每个格式是否可用。
2. 不可用原因。
3. 当前后端使用的生成引擎。
4. 最大内容限制。

示例：

1. `markdown.available=true`
2. `pdf.available=false, reason=wkhtmltopdf missing`
3. `docx.available=true, engine=pandoc`
4. `xlsx.available=true, engine=excelize`

这样前端可以把禁用原因直接展示给用户。

### 7.3 PDF 方案建议

#### 短期方案

短期可以继续保留当前 `goldmark -> styled HTML -> wkhtmltopdf` 流程。

优点：

1. 改动小。
2. 已有代码可复用。
3. 适合先完成“统一后端化”目标。

缺点：

1. `wkhtmltopdf` 较老。
2. 对现代 CSS 支持一般。
3. 某些中文字体、复杂布局和打印样式控制能力有限。

#### 中期推荐方案

中期建议评估替换为基于 Chromium 的 PDF 渲染方案，例如：

1. headless Chromium
2. chromedp
3. Playwright 驱动的服务端渲染

原因：

1. 与前端最终显示更接近。
2. CSS 兼容性更好。
3. 对现代布局、分页和字体控制更稳定。

建议策略：

1. 先完成后端统一。
2. 再把 PDF 引擎从 `wkhtmltopdf` 平滑切换到 Chromium。

### 7.4 DOCX 方案建议

当前 `pandoc` 方案可继续保留，但要补充以下能力：

1. 启动期能力检查。
2. `reference.docx` 模板支持。
3. 图片、代码块、表格和标题层级的回归测试。
4. 临时文件清理与超时控制。

如果后续要做品牌化样式输出，建议基于 `pandoc + reference.docx` 的方式统一模板，而不是在前端用 HTML 包一层 DOCX。

### 7.5 XLSX 方案建议

这是本次最值得后端化的一块，因为当前完全没有后端实现。

建议基于 `excelize` 实现服务端 XLSX 生成，并采用“结构化内容导出”而不是“纯文本正则抽表”。

推荐处理策略：

1. 使用 `goldmark` AST 解析 Markdown。
2. 抽取标题、段落、列表、代码块、表格等结构块。
3. 对于表格：
   1. 每个表格生成独立 sheet，或按一级标题分组生成 sheet。
   2. 首行自动应用表头样式。
   3. 自动列宽。
4. 对于非表格正文：
   1. 生成 `概览` sheet。
   2. 保留标题层级与正文内容。
5. 对于代码块：
   1. 可写入单独 sheet，或写入概览页并标记语言。

这样生成的 XLSX 不再只是“能打开”，而是具备可读性和结构稳定性。

## 8. 前端配套优化建议

### 8.1 前端不再承担格式生成职责

建议将前端 `exportUtils.ts` 的职责收敛为：

1. 生成文件名。
2. 调用后端接口。
3. 处理下载 Blob。
4. 展示明确错误。

前端不再负责：

1. `html2pdf.js` 生成 PDF。
2. `html-docx-js-typescript` 生成 DOCX。
3. 正则抽取 Markdown 表格并生成 XLSX。

### 8.2 能力探测必须前置接入 UI

`ExportDropdown` 打开时或组件挂载后，应主动读取 `/api/v1/export/capabilities`。

建议行为：

1. 不可用格式置灰。
2. Hover 显示原因，例如“当前服务器未安装 pandoc”。
3. 点击不可用项时不给出泛化失败提示，而是给出明确原因。

### 8.3 去掉静默前端降级

建议默认关闭 PDF / DOCX 的静默前端降级。

原因：

1. 导出属于结果交付场景，稳定性比“尽量兜底”更重要。
2. 静默降级会掩盖环境和后端问题。
3. 前后端双路径会放大测试复杂度。

如果业务上必须保留降级，只建议用于开发模式，并在界面上明确提示“当前已使用前端降级导出，结果可能与正式导出不同”。

## 9. 建议实施步骤

### 第一阶段：低风险收口

目标：先让导出能力从“多链路混跑”收口为“以后端为主，前端只负责下载”。

建议任务：

1. 扩展 `ExportDocumentRequest.Format`，增加 `xlsx` 与 `markdown`。
2. 在后端新增 `MarkdownToXLSX`。
3. 基于 `excelize` 实现基础版 XLSX 导出。
4. 前端移除 `html2pdf.js`、`html-docx-js-typescript`、前端 XLSX 生成逻辑。
5. `ExportDropdown` 接入 capabilities，做格式可用性控制。

阶段目标：

1. 所有格式都能走统一接口。
2. 前端不再承担格式生成。

### 第二阶段：提升质量与可观测性

建议任务：

1. 为 `capabilities` 增加 `reason` 字段。
2. 为导出接口增加 request id、耗时日志、失败原因分类。
3. 为 PDF / DOCX / XLSX 增加集成测试样例。
4. 为导出内容增加大小限制与超时控制。

阶段目标：

1. 用户知道为什么不可用。
2. 研发知道为什么失败。
3. 运维知道环境缺什么。

### 第三阶段：替换老旧 PDF 引擎与增强样式模板

建议任务：

1. 评估基于 Chromium 的 PDF 渲染。
2. 为 DOCX 增加 reference docx 模板。
3. 为 XLSX 增加标题分组、多 sheet 与样式增强。

阶段目标：

1. 提升导出保真度。
2. 提升用户感知质量。

## 10. 风险与注意事项

### 10.1 环境依赖风险

当前 PDF / DOCX 依赖外部命令，必须在以下层面做显式保障：

1. Docker 镜像。
2. 裸机部署文档。
3. 启动健康检查。
4. capabilities 接口。

sudo apt-get update && sudo apt-get install -y chromium pandoc libfontconfig1 fonts-noto-cjk fonts-noto-cjk-extra fonts-noto-color-emoji fonts-liberation

### 10.2 资源消耗风险

服务端导出会消耗：

1. CPU
2. 内存
3. 临时磁盘

建议控制：

1. 单次导出内容大小上限。
2. 命令执行超时。
3. 临时文件清理。
4. 并发导出上限。

### 10.3 内容安全风险

导出内容本质上来自对话文本，建议在服务端处理时明确：

1. HTML 仅使用受控模板。
2. 避免任意远程资源抓取。
3. 对图片和链接做白名单或安全策略。

### 10.4 测试覆盖风险

导出链路非常容易出现“能生成但结果不正确”的情况，因此不能只做接口 200 校验。

至少应覆盖：

1. 中文标题。
2. 多级列表。
3. 代码块。
4. 表格。
5. 图片。
6. 超长内容。
7. 无表格内容导出 XLSX。

---

## 11. 验收建议

建议以以下标准验收：

1. Markdown / PDF / DOCX / XLSX 四种格式都走统一后端接口。
2. 前端不再生成 PDF / DOCX / XLSX。
3. capabilities 能准确反映每种格式是否可用及原因。
4. 环境缺少命令时，界面可以提前提示，不再靠点击后失败发现。
5. XLSX 至少能正确输出：
   1. 概览页。
   2. 标准 Markdown 表格。
   3. 多表格场景。
6. PDF / DOCX 能稳定处理中文、代码块、表格和长文本。

---

## 12. 最终建议

如果只做一个最有价值的调整，建议优先做下面这件事：

**把 PDF / DOCX / XLSX 三种格式统一改为后端生成，并让前端接入 capabilities 后再展示可用性。**

原因很直接：

1. 这一步能先消除当前最大的架构分裂问题。
2. 这一步可以最明显改善“用户能点但导不出来”的体验。
3. 这一步实现成本可控，因为后端接口骨架、Markdown 渲染基础和 `excelize` 依赖都已经在仓库中存在。

简化后的推荐路线是：

1. 先补后端 XLSX。
