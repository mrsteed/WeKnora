# 私有知识库创建后列表不显示问题审查与解决方案

审查日期：2026-05-06

## 一、结论

账号 `admin` 创建的知识库 `移动端集成库` 已经真实写入数据库，没有被软删除，也不是临时知识库。它在数据库中的关键字段如下：

| 字段 | 实际值 |
| --- | --- |
| id | `5a5477e3-27b3-4d61-9738-49a46b0b688c` |
| name | `移动端集成库` |
| type | `document` |
| tenant_id | `10000` |
| created_by | `63952d33-2c57-49fe-8af6-593674bdbcd7` |
| visibility | `private` |
| organization_id | 空 |
| is_temporary | `false` |
| deleted_at | 空 |

`created_by` 对应的用户就是 `admin`，并且 `admin` 是超级管理员：

| 字段 | 实际值 |
| --- | --- |
| id | `63952d33-2c57-49fe-8af6-593674bdbcd7` |
| username | `admin` |
| email | `admin@hlsa.com` |
| tenant_id | `10000` |
| is_super_admin | `true` |
| can_access_all_tenants | `true` |

因此，这不是数据库未创建成功，也不是私有可见性权限本身把 admin 拦住。根因是前端知识库列表页在刷新列表时，无条件把当前组织上下文 `orgStore.currentOrganizationId` 带到了 `GET /api/v1/knowledge-bases?organization_id=...`。后端收到 `organization_id` 后会再次按 `kb.organization_id == organization_id` 精确过滤，而私有知识库创建时 `organization_id` 为空，所以被过滤掉。

另外，数据库核验还发现：`移动端集成库` 当前类型是 `document`，不是 `database`，并且 `data_sources` 表中没有它的绑定数据源。如果预期创建的是“数据库知识库”，需要按数据库类型重新创建或补齐类型/数据源创建流程，不能只看名称判断。

## 二、实际数据库核验

### 1. 知识库记录存在

执行的只读 SQL：

```sql
SELECT id, name, type, tenant_id, created_by, visibility, organization_id,
       is_temporary, deleted_at, created_at, updated_at
FROM knowledge_bases
WHERE name = '移动端集成库'
ORDER BY created_at DESC;
```

结果确认：

- 记录存在 1 条。
- `visibility = 'private'`。
- `created_by = admin 用户 id`。
- `organization_id` 为空。
- `is_temporary = false`。
- `deleted_at` 为空。

### 2. admin 用户匹配创建者

执行的只读 SQL：

```sql
SELECT id, username, email, tenant_id, is_super_admin,
       can_access_all_tenants, is_active, deleted_at, created_at
FROM users
WHERE username = 'admin'
   OR id = '63952d33-2c57-49fe-8af6-593674bdbcd7'
ORDER BY created_at;
```

结果确认：

- `admin.id = 63952d33-2c57-49fe-8af6-593674bdbcd7`。
- 与 `knowledge_bases.created_by` 完全一致。
- `admin.is_super_admin = true`。
- `admin.can_access_all_tenants = true`。
- 用户未删除且处于启用状态。

### 3. 不带组织过滤时可以查到

模拟后端列表查询的基础结果：

```sql
SELECT id, name, type, visibility, organization_id, created_by
FROM knowledge_bases
WHERE tenant_id = 10000
  AND is_temporary = false
  AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT 20;
```

结果中包含：

```text
5a5477e3-27b3-4d61-9738-49a46b0b688c | 移动端集成库 | document | private | 空 | 63952d33-2c57-49fe-8af6-593674bdbcd7
```

### 4. 带当前组织过滤时查不到

当前库中存在组织 `慧联世安`：

```text
6607c299-b93e-4bcf-8e07-c4fe32144a09 | 慧联世安 | tenant_id=10000
```

模拟带组织过滤：

```sql
SELECT id, name, type, visibility, organization_id, created_by
FROM knowledge_bases
WHERE tenant_id = 10000
  AND is_temporary = false
  AND deleted_at IS NULL
  AND organization_id = '6607c299-b93e-4bcf-8e07-c4fe32144a09'
ORDER BY created_at DESC
LIMIT 20;
```

结果只返回绑定到该组织的知识库，不包含 `移动端集成库`。这与前端现象一致。

### 5. 不是数据库类型知识库

执行的只读 SQL：

```sql
SELECT id, name, type, knowledge_base_id, tenant_id, status, created_at
FROM data_sources
WHERE knowledge_base_id = '5a5477e3-27b3-4d61-9738-49a46b0b688c'
ORDER BY created_at DESC;
```

结果为空，说明该知识库没有数据库数据源绑定；再结合 `knowledge_bases.type = 'document'`，它当前不是数据库知识库。

## 三、代码链路审查

### 1. 创建链路正常写入私有可见性和创建人

相关文件：`internal/handler/knowledgebase.go`

创建接口 `CreateKnowledgeBase` 会从登录上下文读取用户 ID，并写入 `kbReq.CreatedBy`：

```go
userIDVal, _ := c.Get(types.UserIDContextKey.String())
if userID, ok := userIDVal.(string); ok {
    kbReq.CreatedBy = userID
}
```

如果请求没有显式传 `visibility`，后端会默认设为私有：

```go
if kbReq.Visibility == "" {
    kbReq.Visibility = types.KBVisibilityPrivate
}
```

数据库中的实际值也证明创建链路已经写入：

- `created_by = admin.id`
- `visibility = private`
- `tenant_id = 10000`

### 2. 后端可见性服务不会隐藏 admin 的私有库

相关文件：`internal/application/service/kb_visibility.go`

`ListAccessibleKBs` 对超级管理员有绕过逻辑：

```go
if isSuperAdmin {
    kbs, err := s.kbRepo.ListKnowledgeBasesByTenantID(ctx, tenantID)
    ...
    return kbs, nil
}
```

由于 `admin.is_super_admin = true`，理论上 `GET /api/v1/knowledge-bases` 不带组织过滤时应该返回租户内全部非临时知识库。SQL 模拟也确认了这一点。

### 3. 后端列表接口会额外执行 organization_id 过滤

相关文件：`internal/handler/knowledgebase.go`

列表接口读取查询参数：

```go
organizationID := c.Query("organization_id")
```

然后在拿到可访问知识库之后再次过滤：

```go
kbs = filterByOrganization(kbs)
```

过滤逻辑是精确匹配：

```go
if kb != nil && kb.OrganizationID == organizationID {
    filtered = append(filtered, kb)
}
```

这段逻辑本身适合“查看某个组织空间下的知识库”，但如果前端在“我的/全部/私有”列表里也传 `organization_id`，私有库会因为 `organization_id` 为空而被排除。

### 4. 前端列表页无条件传 currentOrganizationId

相关文件：`frontend/src/views/knowledge/KnowledgeBaseList.vue`

`fetchList` 当前逻辑：

```ts
const listParams: { organization_id?: string } = {}
if (orgStore.currentOrganizationId) {
  listParams.organization_id = orgStore.currentOrganizationId
}
listKnowledgeBases(listParams)
```

这意味着只要全局组织切换器中有当前组织，知识库列表页就会被组织 ID 收窄。即使左侧选择的是“我的”，请求仍然带 `organization_id`。

### 5. 组织 store 会自动选择并持久化第一个组织

相关文件：`frontend/src/stores/organization.ts`

`currentOrganizationId` 来自 localStorage：

```ts
const currentOrganizationId = ref<string>(localStorage.getItem('weknora_current_org_id') || '')
```

当组织列表加载后，如果当前没有选中组织，会自动选择第一个组织：

```ts
if (!currentOrganizationId.value && response.data.length > 0) {
  switchOrganization(response.data[0].id)
}
```

所以 admin 一旦有组织成员关系，前端很容易自动进入“带组织上下文”的状态。数据库里 admin 至少属于：

| organization_id | name | role |
| --- | --- | --- |
| `6607c299-b93e-4bcf-8e07-c4fe32144a09` | 慧联世安 | admin |
| `9fde03b5-e74d-4ee3-a930-b0507f615fdd` | 张磊测试 | admin |

## 四、根因

根因不是权限字段错误，而是“全局当前组织”和“知识库列表筛选条件”耦合过度。

具体表现：

1. 创建私有知识库时，后端正确写入 `visibility=private`、`created_by=admin.id`、`organization_id=''`。
2. 创建成功后，前端调用 `fetchList()` 刷新列表。
3. `fetchList()` 发现 `orgStore.currentOrganizationId` 不为空，就请求 `GET /api/v1/knowledge-bases?organization_id=...`。
4. 后端先按可见性返回 admin 可访问的知识库，再按 `organization_id` 精确过滤。
5. 私有知识库没有组织 ID，于是被过滤掉，导致“创建成功但列表不显示”。

## 五、立即恢复方案

### 方案 A：前端清空当前组织选择

如果只是当前环境需要立刻看到该知识库，可以清空顶部/侧边的组织选择器，然后刷新知识库列表。

浏览器控制台可执行：

```js
localStorage.removeItem('weknora_current_org_id')
location.reload()
```

预期结果：前端不再请求 `organization_id`，`移动端集成库` 会出现在“我的”或“全部”列表中。

### 方案 B：直接访问不带 organization_id 的列表接口验证

使用 admin 登录态访问：

```text
GET /api/v1/knowledge-bases
```

应包含 `移动端集成库`。

对比访问：

```text
GET /api/v1/knowledge-bases?organization_id=6607c299-b93e-4bcf-8e07-c4fe32144a09
```

不会包含 `移动端集成库`，因为它是私有库且 `organization_id` 为空。

### 方案 C：不要直接改 organization_id 作为临时修复

不建议把私有库的 `organization_id` 直接改成某个组织 ID。原因是当前产品语义里：

- `private` 表示仅创建者可见。
- `organization_id` 主要服务于 `org` 可见性或组织空间视图。
- 私有库强行绑定组织会让数据语义变混乱，也可能影响后续权限判断和空间计数。

如果业务上希望该知识库属于组织，应通过“组织可见”流程创建或更新为：

```text
visibility = org
organization_id = 目标组织 ID
```

但当前编辑流程还存在可见性更新未提交的问题，见下方“附带问题”。

## 六、正式修复建议

### 推荐修复 1：知识库列表页不要无条件带 currentOrganizationId

修改文件：`frontend/src/views/knowledge/KnowledgeBaseList.vue`

当前逻辑：

```ts
const listParams: { organization_id?: string } = {}
if (orgStore.currentOrganizationId) {
  listParams.organization_id = orgStore.currentOrganizationId
}
```

建议改为：默认列表请求不带 `organization_id`，只有用户明确进入某个组织空间视图时才按组织请求。

建议实现方向：

```ts
const fetchList = () => {
  loading.value = true
  return Promise.all([
    listKnowledgeBases().then((res: any) => {
      const data = res.data || []
      kbs.value = data.map((kb: any) => ({
        ...kb,
        updated_at: kb.updated_at ? formatStringDate(new Date(kb.updated_at)) : '',
        showMore: false,
        isProcessing: kb.is_processing || false,
        processing_count: kb.processing_count || 0,
      }))
    }),
    orgStore.fetchSharedKnowledgeBases(),
    orgStore.fetchOrganizations(),
  ]).finally(() => { loading.value = false }).then(() => {
    const counts = orgStore.resourceCounts?.knowledge_bases?.by_organization
    if (counts) spaceCountByOrg.value = { ...counts }
  })
}
```

原因：该页面已经有 `spaceSelection` 负责“全部/全局/组织/我的/共享”的展示分组；全局 `currentOrganizationId` 只适合作为创建组织知识库时的默认组织，不应该自动收窄“我的知识库”列表。

### 推荐修复 2：创建成功后根据返回值切到正确分组并高亮

创建成功后，如果返回的知识库是私有库，可以主动切到“我的”分组：

```ts
const handleKBEditorSuccess = (kbId: string) => {
  spaceSelection.value = 'mine'
  fetchList().then(() => triggerHighlightFlash(kbId))
}
```

更完整的做法是让 `KnowledgeBaseEditorModal` 的 success 事件把创建结果对象返回给列表页，例如：

```ts
emit('success', result.data)
```

列表页根据 `result.data.visibility` 决定切到：

- `private` -> `mine`
- `global` -> `global`
- `org` -> `org` 或具体组织视图

这样可以减少“创建成功但当前筛选视图看不到”的误解。

### 推荐修复 3：保留后端 organization_id 过滤语义，但补接口测试

后端 `organization_id` 精确过滤本身可以保留，因为它表达的是“查询某组织下的知识库”。但需要补测试覆盖：

1. `GET /knowledge-bases` 不带组织 ID，超级管理员能看到私有库。
2. `GET /knowledge-bases?organization_id=xxx` 只返回该组织下的知识库，不返回 `organization_id` 为空的私有库。
3. 前端“我的”视图请求不应带 `organization_id`。

如果后端希望降低误用风险，也可以增加一个更明确的参数，例如：

```text
GET /api/v1/knowledge-bases?scope=organization&organization_id=xxx
```

只有 `scope=organization` 时才执行组织过滤。这样可以避免单个 `organization_id` 参数被全局上下文误带后改变列表语义。

## 七、附带问题

### 1. 编辑知识库时可见性不会提交

`KnowledgeBaseEditorModal.vue` 编辑模式中，表单有 `visibility` 和 `organization_id` 字段，但 `doSubmit()` 调用 `updateKnowledgeBase` 时只传了：

```ts
await updateKnowledgeBase(props.kbId, {
  name: data.name,
  description: data.description,
  config: updateConfig
})
```

没有传：

```ts
visibility: data.visibility,
organization_id: data.organization_id,
```

这会导致用户在编辑弹窗里修改可见性，看似可以操作，但后端不会收到字段。建议同步修复。

### 2. “移动端集成库”不是数据库知识库

数据库实际值是：

```text
knowledge_bases.type = document
data_sources 无绑定记录
```

如果业务预期它是数据库知识库，需要重新走数据库类型创建流程，在基础信息中选择 `数据库` 类型并填写连接信息。当前这条记录仅从名称上看像“数据库”，但系统类型不是 `database`。

### 3. 创建入口默认打开 document 类型

`KnowledgeBaseList.vue` 的创建入口当前调用：

```ts
uiStore.openCreateKB('document', visibility)
```

虽然弹窗内可以切换类型，但如果用户从“创建数据库知识库”的心智进入，默认 `document` 容易误操作。建议后续增加明确的创建类型菜单：

- 创建文档知识库
- 创建 FAQ 知识库
- 创建数据库知识库

或者在数据库相关页面提供 `openCreateKB('database', visibility)` 的入口。

## 八、建议验证清单

修复后按以下步骤验证：

1. admin 登录，顶部组织选择器选中 `慧联世安`。
2. 在“我的”视图创建私有知识库。
3. 创建成功后，列表仍能显示新知识库。
4. 请求日志中 `GET /api/v1/knowledge-bases` 不应带 `organization_id`。
5. 切换到具体组织空间视图时，才允许请求组织维度接口或显式带组织筛选。
6. 创建组织可见知识库时，`visibility=org` 且 `organization_id` 正确写入。
7. 创建数据库知识库时，`knowledge_bases.type=database`，并且 `data_sources.knowledge_base_id` 有对应绑定记录。

## 九、最终建议

优先修复前端 `KnowledgeBaseList.vue` 的 `fetchList()`，去掉对 `orgStore.currentOrganizationId` 的无条件列表过滤。这是导致私有知识库创建后不显示的直接原因。

随后补上两项体验和一致性修复：

1. 创建成功后根据新知识库可见性切换到对应列表分组并高亮。
2. 编辑模式提交 `visibility` 和 `organization_id`，避免可见性配置界面和实际保存行为不一致。

当前数据库中的 `移动端集成库` 不需要数据修复即可显示；只要前端不带组织过滤，admin 就能看到它。如果它本应是数据库知识库，则应重新按 `database` 类型创建，因为当前记录没有数据库数据源绑定。
