# todo_write 与长文档模式统一可行性分析与方案

## 审查目标

本次目标不是直接改代码，而是全面排查当前实现，回答下面三个问题：

1. 0.3.6 新增的 `todo_write` 模式，与 dev 分支 0.3.1 自行加入的长文档模式，当前分别是怎么工作的。
2. 二者是否可以统一。
3. 如果可以统一，应该统一到哪一层、怎么做、风险是什么。

本次重点审查范围：

- Agent 工具与工具注册链路。
- 长文档路由与专用执行链路。
- SSE 事件协议。
- 前端聊天流渲染与计划显示。
- 历史回放与完成态持久化。

重点文件：

- `internal/agent/tools/todo_write.go`
- `internal/application/service/agent_service.go`
- `internal/application/service/session_agent_qa.go`
- `internal/application/service/chat_route_service.go`
- `internal/event/event.go`
- `internal/event/event_data.go`
- `internal/types/chat_route.go`
- `internal/types/qa_request.go`
- `frontend/src/views/chat/components/AgentStreamDisplay.vue`
- `frontend/src/views/chat/components/ToolResultRenderer.vue`
- `frontend/src/views/chat/components/tool-results/PlanDisplay.vue`
- `frontend/src/views/chat/utils/thinkingEvent.js`
- `frontend/src/views/chat/utils/planningOutline.js`

## 结论摘要

结论一：二者当前不是同一个层级的能力。

- `todo_write` 是普通 Agent 循环内部的一个“计划工具”，本质是检索任务清单管理。
- 长文档模式是后端在路由阶段就分流出去的专用生成流水线，本质是“完整文档规划 + 分章生成 + artifact 持久化 + 自动续写”。

结论二：二者不适合粗暴合并成“同一个执行模式”。

原因不是名字不同，而是职责边界不同：

- `todo_write` 管的是“我要查什么、按什么顺序查”。
- 长文档模式管的是“我要生成什么文档、章节怎么规划、正文怎么持续输出、何时自动续写、最终如何落 artifact”。

结论三：二者可以统一，但统一点不应该放在执行引擎层，而应该放在“计划协议层 + 计划展示层 + 计划状态持久化层”。

最合理的方向不是“让长文档去调用 `todo_write`”，也不是“让普通 Agent 都走长文档模式”，而是：

- 保留两套执行引擎。
- 统一输出一套计划状态协议。
- 前端统一渲染成同一类计划/规划时间线。
- 历史回放和调试信息统一使用同一套 plan snapshot 数据结构。

结论四：仓库里其实已经有一个很适合承接统一方案的未启用抽象：

- `internal/event/event.go` 里已有 `EventAgentPlan = "agent.plan"`。
- `internal/event/event_data.go` 里已有 `AgentPlanData`。
- 但当前前后端几乎没有真正使用它。

这意味着“统一计划协议”不需要凭空新造概念，完全可以优先复用现有但闲置的 `agent.plan` 事件体系，再扩展字段。

## 一、当前两套模式分别是什么

### 1.1 todo_write 模式的真实定位

`todo_write` 不是长文档生成器，也不是文档规划器。

它在 `internal/agent/tools/todo_write.go` 里的定义非常明确：

- 它是一个 Agent 工具。
- 它用于创建和更新“结构化任务列表”。
- 它强调任务必须是“检索 / 研究任务”。
- 它显式要求“不要把总结和最终答案放进 todo_write，最终综合交给 thinking 工具”。

从提示词约束看，它的核心语义是：

- 管理检索步骤。
- 追踪 pending / in_progress / completed。
- 帮模型在复杂检索型问题里显式列出任务。

它返回的数据结构也很轻量：

- `task`
- `steps[]`
- `display_type = plan`

换句话说，`todo_write` 当前是“任务清单工具”，不是“完整文档规划协议”。

### 1.2 长文档模式的真实定位

长文档模式不是 Agent 工具。

它在 `internal/application/service/session_agent_qa.go` 里是专门的路由分支：

- `shouldUseKnowledgeGroundedFullDocumentGenerationPath()`
- `shouldUseDedicatedFullDocumentGenerationPath()`
- `runKnowledgeGroundedFullDocumentGenerationPath()`
- `runDedicatedFullDocumentGenerationPath()`

也就是说，长文档模式在请求进入普通 Agent 循环之前，就已经被后端识别并切走了。

这条链路除了“规划”，还承担了以下专用职责：

- 识别是否是完整文档任务。
- 判断是否有本地知识范围。
- 生成大纲。
- 按章节拉证据。
- 按章节持续流式输出正文。
- 生成 artifact。
- 决定是否自动续写。
- 记录 generation run、budget、quality issues。

这已经不是一个普通“计划工具”能替代的范围。

## 二、当前代码里二者分别怎么工作

## 2.1 todo_write 的后端链路

### 2.1.1 工具可用性

`todo_write` 并不是每次都可用。

在 `internal/application/service/agent_service.go` 里，后端会根据知识范围与 web search 决定是否注册它：

- 当没有知识范围，且没有 web search 时，会把 `todo_write` 过滤掉。
- 只有在“复杂检索任务”具备执行土壤时，它才会真正进入 Agent 可用工具集。

这意味着 `todo_write` 的前提条件是“普通 Agent 模式 + 可用检索能力”。

### 2.1.2 工具执行与返回

`todo_write` 作为工具执行时，返回的是结构化工具结果，而不是专用事件流：

- `ToolResult.Success`
- `Output`
- `Data.task`
- `Data.steps`
- `Data.total_steps`
- `Data.display_type = plan`

它没有章节、artifact、generation run、自动续写这些概念。

### 2.1.3 持久化方式

因为它是普通 Agent 工具，所以它最终是以“tool_call / tool_result”嵌在 AgentSteps 里的。

也就是说，`todo_write` 当前的持久化载体是：

- Agent 工具调用历史。
- 工具结果数据。

它并没有独立的计划生命周期对象。

## 2.2 todo_write 的前端链路

前端对 `todo_write` 的显示是明确的工具结果渲染路径。

### 2.2.1 基础展示入口

`frontend/src/views/chat/components/ToolResultRenderer.vue` 中：

- `display_type === 'plan'` 时，进入 `PlanDisplay.vue`。

`PlanDisplay.vue` 本身只认一套简单数据：

- `steps[]`
- `status`
- `description`

它的设计语义是“任务清单卡片”，不是“大纲规划器”。

### 2.2.2 Agent 时间线上的附加表现

`frontend/src/views/chat/components/AgentStreamDisplay.vue` 里又给 `todo_write` 做了额外增强：

- 当 `tool_name === 'todo_write'` 时，标题显示成“更新计划”。
- 会显示计划状态摘要：in progress / pending / completed 数量。
- 会通过 `buildFullEventList()` 注入 `plan_task_change` 事件，把不同 task 的切换插入时间线。

注意这里的“计划变化”是从工具结果里推导出来的，不是后端有专门的计划事件。

这说明 todo_write 的前端展示，本质上是：

- 工具结果卡片。
- 再叠加少量时间线增强。

## 2.3 长文档模式的后端链路

### 2.3.1 路由阶段决定，不进普通 Agent 工具循环

长文档模式的入口判断在 `internal/application/service/session_agent_qa.go`：

- 先判断 translation continuation。
- 再判断 translation full document。
- 再判断 knowledge grounded continuation。
- 再判断 dedicated document edit。
- 再判断 knowledge grounded full document。
- 再判断 dedicated full document。

只有都不命中，才落回普通 AgentQA。

这和 `todo_write` 完全不同：

- `todo_write` 是普通 Agent 循环中的一个工具。
- 长文档模式是绕开普通 Agent 循环的专用服务路径。

### 2.3.2 规划与正文输出共用 Agent 风格流事件

虽然执行引擎不同，但长文档模式对外发事件时，已经在复用 Agent 风格协议：

- 大纲规划通过 `EventAgentThought` 发出。
- 正文流式内容通过 `EventAgentFinalAnswer` 发出。
- 完成态通过 `EventAgentComplete` 发出。

具体来看：

- `fullDocumentProgressReporter.PublishOutline()` 把大纲作为 `AgentThoughtData`，并打上 `Synthetic=true`、`Stage="planning"`、`Outline=...`。
- `emitDedicatedFullDocumentAnswerChunk()` 内部发的是 `EventAgentFinalAnswer`。
- `emitFullDocumentCompletion()` 发的是 `EventAgentComplete`。

这说明长文档模式在“事件协议层”已经和聊天流做了一定统一。

### 2.3.3 长文档模式除了规划，还承担很多专属语义

长文档模式额外携带的语义明显超过 `todo_write`：

- `document_generation_status`
- `generation_run_id`
- `final_document_mode`
- `final_document_artifact_id`
- `auto_continue_next`
- `auto_continue_reason`
- `recommended_request`
- 章节级进度
- 证据查询进度
- 预算与质量信息

因此它不是“计划工具加强版”，而是“完整文档执行协议”。

## 2.4 长文档模式的前端链路

长文档 planning 在前端不是通过 `PlanDisplay` 呈现的，而是走 thinking / outline 分支。

### 2.4.1 thinking 事件承载 planning

`frontend/src/views/chat/utils/thinkingEvent.js` 会把以下信息挂到 thinkingEvent 上：

- `synthetic`
- `stage`
- `outline`
- `planning_outline`
- `section_current`
- `section_total`
- `query_current`
- `query_total`
- `progress_label`

这已经是一套“规划过程事件”的雏形。

### 2.4.2 planning outline 的解析逻辑是独立的

`frontend/src/views/chat/utils/planningOutline.js` 专门负责：

- 从 `outline / planning_outline / thinking content` 中提取结构化大纲。
- 区分 `outlineOnly`。
- 区分 `base_document` 与 planning outline。

这条前端逻辑与 `todo_write.steps[]` 完全不共用。

### 2.4.3 AgentStreamDisplay 对二者的处理完全不同

在 `frontend/src/views/chat/components/AgentStreamDisplay.vue`：

- `todo_write` 被视为 `tool_call/tool_result`。
- 长文档 planning 被视为 `thinking` 事件，甚至是 synthetic planning thought。
- `plan_task_change` 是从 `todo_write` 推导出来的。
- planning outline 则是从 `thinking/outline` 推导出来的。

所以当前前端实际上已经维护了两套“计划 UI 语义”：

- 任务清单语义。
- 文档大纲语义。

## 三、为什么当前不能直接说“统一成一个模式”

## 3.1 入口层不同

`todo_write` 的入口是：

- 普通 Agent 模式。
- 模型在中间回合自行决定调用该工具。

长文档模式的入口是：

- 请求进入服务后，由后端路由器或显式输出模式决定。
- 然后直接走专用 full document path。

这两者的控制权不在同一个位置。

## 3.2 计划对象不同

`todo_write` 的计划对象是：

- “检索任务列表”。

长文档模式的计划对象是：

- “文档大纲 + 章节顺序 + 证据检索阶段 + 正文生成阶段 + 自动续写阶段”。

前者偏任务单，后者偏执行蓝图。

## 3.3 终局目标不同

`todo_write` 本身不产出正文，只负责组织任务。

长文档模式的规划阶段必须服务于：

- 章节结构稳定。
- 后续章节连续生成。
- artifact 可落库。
- auto continue 可恢复。

这要求长文档的 planning 输出必须是“可执行合同”，不是普通任务清单。

## 3.4 状态机复杂度不同

`todo_write` 当前状态机很简单：

- pending
- in_progress
- completed
- skipped

长文档模式的状态机至少包含：

- planning
- retrieving
- generating
- finalizing
- completed
- partial
- blocked
- needs_review
- continuing

如果强行用 `todo_write` 的四状态去承载长文档，会损失大量语义。

## 3.5 持久化需求不同

`todo_write` 只要能回放出工具结果卡片就够了。

长文档模式需要持久化：

- generation run
- artifact id
- final document delivery mode
- auto continue recommendation
- quality issues
- budget source

这不是 `PlanDisplay` 级别的数据结构能覆盖的。

## 四、统一的可行性评估

为了避免“统一”这个词过于宽泛，建议把它拆成五层来判断。

## 4.1 入口路由层统一

结论：低可行，不建议统一。

原因：

- 长文档模式必须在路由早期就分流，否则普通 Agent 工具循环无法稳定承担 artifact、auto continue、章节生成预算控制。
- `todo_write` 是工具层语义，只有进入普通 Agent 循环后才可能出现。

因此不能把“是否长文档”交给 `todo_write` 再决定，也不应该让所有 `todo_write` 都升级成长文档路由。

## 4.2 执行编排层统一

结论：中低可行，不建议做成单引擎。

原因：

- 长文档执行器是“规划 + 检索 + 分章生成 + 质量门控 + artifact 持久化 + 自动续写”的专用流水线。
- `todo_write` 只是 Agent 的一个辅助工具，不具备章节生成、artifact 生命周期和 continuation 策略。

如果强行统一成一个执行引擎，最终大概率会把长文档专用能力重新塞回 Agent 工具栈，复杂度更高，稳定性更差。

## 4.3 事件协议层统一

结论：高可行，强烈建议统一。

这是当前最值得做、收益最高、风险最低的统一层。

理由：

- 长文档模式已经在复用 `EventAgentThought / EventAgentFinalAnswer / EventAgentComplete`。
- 仓库里已经存在未启用的 `EventAgentPlan` 与 `AgentPlanData`。
- `todo_write` 当前通过工具结果卡片表达计划，长文档通过 thinking/outline 表达计划，实际上都在向前端传递“规划状态”。

统一后可以做到：

- 后端不管计划来自 `todo_write`，还是来自 full document outline，都额外发一份标准 `agent.plan` snapshot。
- 前端统一消费 `agent.plan`，而不是一会儿读 `tool_result.display_type=plan`，一会儿读 `thinking.stage=planning`。

## 4.4 持久化 / 回放层统一

结论：中高可行，建议随协议统一一起做。

当前两个模式的历史回放来源不同：

- `todo_write` 回放来自 tool_call / tool_result。
- 长文档 planning 回放来自 thought / outline / complete extra。

如果统一 plan snapshot：

- 历史回放可以直接从标准计划快照恢复。
- 不必在前端继续维持两套推导逻辑。
- 调试成本会明显下降。

## 4.5 前端展示层统一

结论：高可行，强烈建议统一。

当前前端的计划展示已经分裂成两套：

- `PlanDisplay` 负责任务清单。
- `thinking + planningOutline` 负责大纲规划。

最合理的方向是：

- 新建统一的 `PlanTimeline` / `PlanStateCard`。
- 支持两种子形态：
  - `task_list`
  - `document_outline`
- 二者共用同一套头部、阶段、状态、进度、折叠展开、历史回放入口。

## 五、推荐的统一方案

## 5.1 统一目标

目标不是把 `todo_write` 和长文档变成同一个后端执行器。

目标应该是：

- 统一成“同一类计划对象”。
- 统一成“同一套计划事件协议”。
- 统一成“同一套前端计划展示框架”。
- 保留不同执行引擎。

可以概括为：

统一计划协议，不统一业务引擎。

## 5.2 统一数据模型建议

建议基于现有 `EventAgentPlan` / `AgentPlanData` 扩展为标准计划快照协议。

建议字段如下：

- `plan_id`
- `source`
  - `todo_write`
  - `full_document_outline`
  - `document_continuation`
  - `translation_outline`
- `plan_kind`
  - `research_task_list`
  - `document_outline`
  - `document_execution_plan`
- `goal`
- `status`
  - `pending`
  - `planning`
  - `retrieving`
  - `generating`
  - `finalizing`
  - `completed`
  - `partial`
  - `blocked`
- `steps[]`
  - 兼容 `todo_write`
- `outline`
  - `title`
  - `sections[]`
- `section_progress`
  - `current`
  - `total`
  - `title`
- `query_progress`
  - `current`
  - `total`
  - `label`
- `generation_run_id`
- `artifact_id`
- `auto_continue`
- `extra`

其中要注意：

- `steps[]` 不是所有 plan_kind 都需要。
- `outline` 也不是所有 plan_kind 都需要。
- 所以统一协议要做成“同一外壳、不同负载”，而不是强制二者字段完全一致。

## 5.3 后端改造建议

### 第一阶段：双写，不破坏现有行为

建议先做兼容性最好的双写方案：

- `todo_write` 继续保留原来的 tool_result `display_type=plan`。
- 同时在工具结果落地后，新增发一条 `EventAgentPlan` snapshot。
- 长文档模式继续保留原来的 `EventAgentThought(stage=planning)` 与 `outline` 逻辑。
- 同时在 `PublishOutline()`、阶段更新、章节推进时，同步发 `EventAgentPlan` snapshot。

这样前端可以逐步切换，不会一下子推翻现有历史回放逻辑。

### 第二阶段：建立 PlanAdapter

建议新增一个统一的后端适配层，例如：

- `BuildPlanSnapshotFromTodoWrite(...)`
- `BuildPlanSnapshotFromFullDocumentProgress(...)`
- `BuildPlanSnapshotFromDocumentContinuation(...)`

这样所有计划类能力都先转换成统一 snapshot，再决定发给前端什么事件。

### 第三阶段：完整回放统一

等前端完成迁移后：

- 历史回放优先使用 plan snapshot。
- 旧的 `todo_write tool_result -> 前端推导 plan_task_change` 可逐步降级为兼容逻辑。
- 旧的 `thinking.stage=planning -> 前端提取 outline` 也可逐步转成兼容回放逻辑。

## 5.4 前端改造建议

### 当前问题

前端现在至少维护了三套和计划相关的特殊逻辑：

- `PlanDisplay.vue` 渲染 `todo_write.steps[]`。
- `AgentStreamDisplay.vue` 里针对 `todo_write` 写了大量标题、状态摘要、`plan_task_change` 推导逻辑。
- `thinkingEvent.js` 与 `planningOutline.js` 又专门维护长文档 planning 解析。

这会让后续继续加模式时，逻辑越来越散。

### 推荐做法

新建统一计划渲染组件，例如：

- `PlanTimeline.vue`

由它统一接收标准 `plan snapshot`，内部再按 `plan_kind` 分两种渲染：

- `research_task_list`
- `document_outline`

统一后的前端收益：

- 计划头部、状态条、阶段标签、折叠展开、历史回放、调试入口全共用。
- `todo_write` 与长文档 planning 看起来是“同一类交互”，不会再像现在这样割裂。
- 后续如果要新增“数据库分析计划”“迁移计划”“翻译计划”，只要接同一协议即可。

## 5.5 执行层的保留策略

建议明确保留两条执行引擎：

- 普通 Agent 引擎：继续负责 `todo_write`、thinking、tool call、普通问答。
- 长文档专用引擎：继续负责 full document、continuation、translation、artifact、quality gate。

不要尝试在第一阶段把两者强行收敛成一个执行器。

原因很直接：

- 协议统一和 UI 统一是低风险高收益。
- 引擎统一是高风险低收益。

## 六、不建议采用的统一方式

## 6.1 不建议让长文档模式直接依赖 todo_write

原因：

- `todo_write` 只会返回任务列表，不会保证章节结构稳定。
- 它没有 artifact、generation run、budget、quality、auto continue 语义。
- 让长文档依赖 `todo_write`，等于把专用流水线重新拆散，稳定性会下降。

## 6.2 不建议让普通 Agent 问答都升级成“先规划后长文档”

原因：

- 普通问答里很多只是复杂检索，不是完整文档生成。
- 如果统一成 full document 路由，会引入过重的规划与 artifact 成本。

## 6.3 不建议只做 UI 统一，不做协议统一

原因：

- 如果后端仍然是两套完全不同的表达方式，前端适配层只会越来越厚。
- 只统一组件外观，不能解决历史回放和调试协议分裂问题。

## 七、落地实施建议

## 7.1 第一阶段：协议补齐

目标：先让后端能发统一计划快照。

建议任务：

1. 扩展 `AgentPlanData`，补足计划快照字段。
2. 在 `todo_write` 结果落地后同步发 `EventAgentPlan`。
3. 在 `fullDocumentProgressReporter.PublishOutline()` 和阶段推进逻辑中同步发 `EventAgentPlan`。
4. 在 `complete` 事件里附带最后一次计划快照引用或摘要。

## 7.2 第二阶段：前端统一适配

目标：把两套计划展示逻辑并到统一计划组件。

建议任务：

1. 新增统一计划视图组件。
2. 为旧事件流写 PlanAdapter：
   - `todo_write tool_result -> plan snapshot`
   - `thinking.stage=planning + outline -> plan snapshot`
3. AgentStreamDisplay 优先渲染 plan snapshot，旧逻辑只保留兼容路径。

## 7.3 第三阶段：历史回放统一

目标：让计划回放不再依赖多套前端推导。

建议任务：

1. 在 stream / message persistence 中保留标准计划快照。
2. 历史打开会话时优先恢复 plan snapshot。
3. 逐步减少 `plan_task_change` 临时推导与 `planningOutline.js` 的特殊兜底路径。

## 7.4 第四阶段：观测与调试统一

目标：让开发和排查时能快速看懂当前进入了哪类计划。

建议任务：

1. 在前端调试信息里标明：
   - `plan_kind`
   - `source`
   - `generation_run_id`
   - `route kind`
2. 在后端日志里统一输出 plan snapshot 摘要。
3. 对 `todo_write` 和长文档 planning 的关键时刻补统一埋点。

## 八、风险与注意事项

## 8.1 语义压平风险

如果把长文档 planning 强行压成 `steps[]`，会损失：

- 大纲结构。
- 章节层次。
- 证据检索与章节生成的阶段差异。

所以统一协议不能只围绕 `PlanDisplay` 的当前数据结构设计。

## 8.2 回放兼容风险

当前历史消息里：

- `todo_write` 是 tool_result。
- 长文档 planning 是 thinking + outline + complete extra。

统一时必须保留旧数据兼容，否则已有会话回放会断。

## 8.3 auto continue 风险

长文档模式的 `auto_continue_next`、`recommended_request`、`generation_run_id` 都是续写闭环的关键。

如果统一计划协议时忽略这些字段，前端虽然看起来统一了，但自动续写链会失去可观测性。

## 8.4 IM / 其他端展示风险

仓库里 IM 侧对 `todo_write` 也有独立展示映射。

如果统一计划协议只改 Web 前端，不同步检查 IM 映射，后续会出现不同端体验再次分裂。

## 九、最终建议

### 最终判断

`todo_write` 模式与长文档模式不能直接统一成“同一个模式”。

准确说法应该是：

- 二者不适合统一执行引擎。
- 但非常适合统一为同一类“计划协议”和“计划展示框架”。

### 推荐路线

建议采用：

- 执行层分离。
- 协议层统一。
- 展示层统一。
- 历史回放统一。

### 一句话方案

不要让长文档去复用 `todo_write` 的执行器；要让 `todo_write` 和长文档都输出同一种 plan snapshot，再由前端统一渲染。

### 实施优先级

优先级建议如下：

1. 先统一计划事件协议，复用现有但未启用的 `EventAgentPlan` / `AgentPlanData`。
2. 再统一前端计划组件与历史回放。
3. 最后才考虑是否抽象出共享的 PlanningCoordinator。

如果一开始就尝试把两条执行链硬并成一个模式，风险远高于收益。
