# 文档编辑请求误走知识检索原因分析与修正方案

## 1. 问题现象

用户请求为：

```text
把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

前端已经显示当前基线为：

```text
北海电厂二期智慧电厂项目技术方案 V4
```

也就是说，本轮请求具备明确的基线 artifact，理论上应该进入阶段三实现的“文档编辑专用路径”：只读取上一版 artifact 的结构化上下文，构造 `<document_patch>`，不再走通用 Agent 检索链路。

但实际输出为：

```text
很抱歉，我无法完成您要求的操作。

原因：用于检索文档上下文的工具调用失败（knowledge_search 不可用），未能获取原文内容。
```

这说明请求没有进入专用文档编辑路径，而是又回到了通用 Agent 推理流程，并且模型尝试调用了 `knowledge_search`。

## 2. 日志证据

请求入口日志显示：

```json
{
  "query": "把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。",
  "knowledge_base_ids": null,
  "knowledge_ids": null,
  "intent_hint": "revise_document",
  "base_artifact_id": "14a55959-399a-4510-9c0d-9f43e438e7dd",
  "document_output_mode": "delta_only",
  "web_search_enabled": true
}
```

关键点：

1. `intent_hint=revise_document`。
2. `base_artifact_id` 存在。
3. `document_output_mode=delta_only`。
4. `knowledge_base_ids=null`、`knowledge_ids=null`，用户本轮没有显式选择知识库文档。
5. `web_search_enabled=true`，这是 UI 或会话层面的开关，不等价于用户明确要求联网检索。

随后日志显示阶段一止血逻辑确实生效：

```text
Custom agent config applied: MaxIterations=1, AllowedTools=[final_answer], WebSearchEnabled=false
Applied document-edit stopgap: cleared session context before agent execution
Registered tools: [final_answer]
```

但是请求仍然进入了通用 Agent：

```text
[Agent] SystemPrompt: 12649 chars
[Agent] Ready: 2 messages, 3 tools [final_answer, read_skill, execute_skill_script]
```

之后模型生成了不存在的工具调用：

```text
LLM responded: finish=tool_calls, tools=[knowledge_search]
Tool knowledge_search Args: {"query":"2.5.5 火电设备运维智能体 技术实现 第二章", ...}
execute_failed error="tool not found: knowledge_search"
```

最终合成阶段只看到了工具失败摘要：

```text
Tool knowledge_search summary:
Error: tool not found: knowledge_search
```

因此模型给出了“无法完成，检索工具不可用”的解释。

## 3. 根因分析

根因不是基线文档中没有相关内容，也不是 artifact 选择失败。根因是阶段三的专用文档编辑路径分流条件过严。

原先逻辑中，`shouldUseDedicatedDocumentEditPath()` 包含以下排除条件：

```go
if req.WebSearchEnabled || len(req.KnowledgeBaseIDs) > 0 || len(req.KnowledgeIDs) > 0 {
    return false
}
```

这导致只要请求层带着 `web_search_enabled=true`，即使用户没有要求联网、没有要求检索知识库、也没有显式选择知识库文件，纯文档编辑请求仍然会被踢回通用 Agent。

本次日志正是这种情况：

1. 请求满足文档编辑条件：`revise_document + base_artifact_id + delta_only`。
2. 用户目标是移动/合并基线文档内部章节，不需要外部检索。
3. 但 `web_search_enabled=true` 阻断了专用路径。
4. 通用 Agent 使用完整检索型系统提示，系统提示中仍包含 `knowledge_search` 工作流说明。
5. 阶段一 stopgap 已把实际注册工具限制为 `final_answer`，因此模型发出的 `knowledge_search` 调用无法执行。
6. 最终合成基于工具失败摘要输出道歉，而不是生成文档 patch。

换句话说，阶段一和阶段二的止血逻辑没有失效；真正缺口在阶段三分流入口：把“WebSearch 开关打开”误判成“用户要求外部检索”。

## 4. 已实施修正

已修正 `internal/application/service/session_agent_qa.go` 中的分流条件。

修正后的策略：

1. 纯文档增量修订/续写请求，只要有 `BaseArtifactID` 和 `QuotedContext`，默认进入专用文档编辑路径。
2. `WebSearchEnabled=true` 不再单独阻断专用路径。
3. 只有出现以下情况时才回到通用 Agent：
   - 请求显式携带 `KnowledgeBaseIDs` 或 `KnowledgeIDs`。
   - 用户文本明确要求外部检索，例如“联网”“搜索知识库”“结合知识库”“查询数据库”“引用来源”等。
   - 请求包含附件、图片或图片描述，需要额外上下文处理。

新增辅助函数：

```go
func documentEditRequiresExternalRetrieval(query string) bool
```

它用于区分“UI 开关打开”与“用户明确要求检索”。

核心修正从：

```go
if req.WebSearchEnabled || len(req.KnowledgeBaseIDs) > 0 || len(req.KnowledgeIDs) > 0 {
    return false
}
```

调整为：

```go
if len(req.KnowledgeBaseIDs) > 0 || len(req.KnowledgeIDs) > 0 || documentEditRequiresExternalRetrieval(req.Query) {
    return false
}
```

## 5. 新增测试覆盖

已在 `internal/application/service/session_agent_qa_test.go` 补充以下测试场景：

1. `WebSearchEnabled=true` 但用户是纯文档编辑请求时，应进入专用路径。
2. 用户明确写出“结合知识库资料”时，应回到通用 Agent 路径。
3. 原有显式 `KnowledgeBaseIDs` 的请求仍然不会进入专用路径。

关键断言：

```go
assert.True(t, shouldUseDedicatedDocumentEditPath(req))
```

用于覆盖本次日志中的真实输入形态。

## 7. 修正后的预期链路

对同类请求：

```text
把2.5.5 火电设备运维智能体——技术实现 后续的内容，合并到第二章。
```

后端应走以下路径：

1. `prepareDocumentRequest()` 根据基线 artifact 构造 `document_revision_context`。
2. `shouldUseDedicatedDocumentEditPath()` 返回 `true`。
3. `AgentQA()` 直接调用 `runDedicatedDocumentEditPath()`。
4. 不创建通用 Agent Engine。
5. 不注册 `knowledge_search`、Wiki、MCP、技能工具。
6. 不发送完整检索型 system prompt。
7. 模型只看到：用户目标 + 双锚点文档编辑上下文。
8. 输出 `<document_patch>`，再由现有 artifact 注册与 merge 逻辑合并生成新版本。

日志上应不再出现：

```text
Agent Ready: ... tools [final_answer, read_skill, execute_skill_script]
Tool knowledge_search ... tool not found
```

而应出现专用路径相关行为，例如一次性文档编辑流式调用。

## 8. 后续建议

这次问题说明阶段三的“纯文档编辑判定”不能只看功能开关，还要区分用户意图。

建议后续继续补强：

1. 为专用文档编辑路径增加更明确的日志，例如：`Dedicated document edit path selected`，方便线上快速确认是否分流成功。
2. 将 `documentEditRequiresExternalRetrieval()` 的关键词规则沉淀为可测试的意图判定函数，后续支持更多表达，例如“参考上传资料”“根据某某方案补充”。
3. 增加一条接近本次日志的端到端测试：`web_search_enabled=true + revise_document + base_artifact_id + delta_only`，断言不会创建 Agent Engine，不会注册检索工具。
4. 如果专用路径成功生成 patch，但 merge 出现不确定，应保留上一份有效 artifact，并将新版本标记为 `partial/needs_review`，避免错误覆盖基线。

## 9. 结论

本次失败不是基线文档缺内容，而是阶段三专用文档编辑路径被 `web_search_enabled=true` 误阻断，导致请求回落到通用检索 Agent。通用 Agent 的系统提示仍然诱导模型调用 `knowledge_search`，但阶段一 stopgap 已经没有注册该工具，于是产生 `tool not found: knowledge_search`，最终只能输出无法完成。

修正后，WebSearch 开关不再单独阻断纯文档编辑请求。只要用户没有明确要求外部检索，同类基线重编辑请求会进入专用文档编辑路径，避免再次因为不存在的检索工具而失败。
