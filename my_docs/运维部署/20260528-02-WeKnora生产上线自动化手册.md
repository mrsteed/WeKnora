# WeKnora 生产上线自动化手册

## 1. 文档目标

本文是在《20260528-01-WeKnora生产级编译打包部署清单.md》的基础上，结合项目现有 `Makefile`、`scripts/build_images.sh`、`scripts/start_all.sh`、`scripts/package-lite.sh`、`scripts/migrate.sh` 和 Helm Chart，给出一套可重复执行的生产上线自动化流程。

配套文件位于同一目录：

1. `20260528-02-weknora-prod-automation.sh`：生产构建、打包、发布、备份、验证总控脚本。
2. `20260528-02-weknora-prod-automation.env.example`：自动化配置样例。

自动化脚本不会替代现有仓库脚本，而是做一层生产编排：

1. 调用当前代码的 Dockerfile、Makefile 和包脚本生成产物。
2. 统一注入版本、commit、构建时间、Go 版本和镜像 tag。
3. 自动构建 App、docreader、frontend、sandbox 四个生产镜像。
4. 自动归档 CLI、MCP Server、Client SDK、Lite 包等非镜像产物。
5. 自动生成 Docker Compose `.env` 和 Helm `values-production.yaml`。
6. 自动执行 Compose 或 Helm 上线、验证、备份、回滚入口。

## 2. 文件清单

| 文件 | 用途 | 是否需要修改 |
|------|------|--------------|
| `20260528-02-weknora-prod-automation.sh` | 自动化执行脚本 | 通常不改，直接执行 |
| `20260528-02-weknora-prod-automation.env.example` | 配置样例 | 复制为 `.env` 后修改 |
| `20260528-02-WeKnora生产上线自动化手册.md` | 本文档 | 不需要 |
| `20260528-01-WeKnora生产级编译打包部署清单.md` | 详细人工部署清单 | 作为背景和排障依据 |

建议先复制配置文件：

```bash
cd /home/xmkp/workspace/WeKnora

cp my_docs/运维部署/20260528-02-weknora-prod-automation.env.example \
   my_docs/运维部署/20260528-02-weknora-prod-automation.env

chmod 600 my_docs/运维部署/20260528-02-weknora-prod-automation.env
```

然后编辑真实值：

```bash
vi my_docs/运维部署/20260528-02-weknora-prod-automation.env
```

必须替换所有 `change-me-*` 字段，尤其是数据库、Redis、JWT、AES、对象存储和镜像仓库配置。

## 3. 自动化脚本总体设计

脚本按三类场景组织命令。

构建机命令：

| 命令 | 说明 |
|------|------|
| `check` | 检查构建工具、Docker、仓库结构和版本变量 |
| `version` | 输出解析后的版本、镜像名和构建元数据 |
| `test-source` | 按配置执行 Go、CLI、Client、前端、MCP 源码检查 |
| `build-images` | 构建四个生产镜像 |
| `push-images` | 推送四个生产镜像 |
| `package-artifacts` | 归档二进制、CLI、MCP、Lite、SDK 和 SHA256 |
| `build-release` | 执行 `check`、可选测试、构建镜像、归档产物 |
| `publish-release` | 执行 `build-release` 后推送镜像 |

Docker Compose 命令：

| 命令 | 说明 |
|------|------|
| `prepare-compose-dir` | 把运行所需文件复制到 `COMPOSE_PROJECT_DIR` |
| `render-compose-env` | 根据自动化配置生成生产 `.env` |
| `deploy-compose` | 执行 Compose 启动或升级 |
| `verify-compose` | 检查容器状态、App `/health` 和前端入口 |
| `backup-compose` | 从 Compose PostgreSQL 容器执行 `pg_dump` |
| `rollback-compose` | 修改 `WEKNORA_VERSION` 并重启 Compose |
| `compose-online` | 串联准备目录、渲染 `.env`、部署、验证 |

Kubernetes/Helm 命令：

| 命令 | 说明 |
|------|------|
| `generate-helm-values` | 生成生产 `values-production.yaml` |
| `apply-helm-secrets` | 创建 namespace、应用 Secret、可选 imagePullSecret、可选 TLS Secret |
| `deploy-helm` | 执行 `helm upgrade --install` |
| `verify-helm` | 检查 Deployment rollout、资源和 App 日志 |
| `backup-helm` | 从 Helm PostgreSQL Deployment 执行 `pg_dump` |
| `rollback-helm` | 执行 `helm rollback` |
| `helm-online` | 串联 values、Secret、部署、验证 |

## 4. 配置文件关键字段

### 4.1 发布与构建字段

| 字段 | 示例 | 说明 |
|------|------|------|
| `REGISTRY` | `registry.example.com/weknora` | 企业镜像仓库命名空间 |
| `RELEASE_VERSION` | `0.6.0-20260528.1` | 本次生产版本，不建议使用 `latest` |
| `PLATFORM` | `linux/amd64` | Docker 构建平台 |
| `TARGETARCH` | `amd64` | docreader 下载 grpc_health_probe/protoc 时使用 |
| `MAX_FILE_SIZE_MB` | `50` | 前端、后端、docreader 文件大小限制 |
| `RUN_SOURCE_TESTS` | `false` | 是否在 `build-release` 中执行源码检查 |

### 4.2 运行时字段

| 字段 | 说明 |
|------|------|
| `DOMAIN` | 生产访问域名 |
| `APP_EXTERNAL_URL` | WeKnora 外部访问地址 |
| `DB_USER`、`DB_PASSWORD`、`DB_NAME` | 生产数据库配置 |
| `REDIS_PASSWORD` | Redis 密码 |
| `MINIO_*` | 对象存储配置 |
| `JWT_SECRET` | JWT 签名密钥 |
| `TENANT_AES_KEY` | 租户 API Key 等历史数据依赖的稳定密钥 |
| `SYSTEM_AES_KEY` | 系统级敏感字段 AES 密钥，必须稳定保存 |
| `CRYPTO_MASTER_KEY`、`CRYPTO_SALT` | 新加密体系主密钥和盐值，建议显式配置并备份 |
| `WEKNORA_SANDBOX_DOCKER_IMAGE` | 脚本会自动按当前 `REGISTRY/RELEASE_VERSION` 生成 |

密钥生成示例：

```bash
openssl rand -base64 48
openssl rand -hex 32
openssl rand -base64 32
```

## 5. 构建机自动化发布流程

### 5.1 检查环境

```bash
cd /home/xmkp/workspace/WeKnora

bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh check
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh version
```

预期输出：

1. 能识别仓库根目录。
2. 能解析 `VERSION`、commit、构建时间和 Go 版本。
3. 能得到四个生产镜像名。
4. Docker daemon 可用。

### 5.2 可选源码检查

如果希望自动化构建前执行测试，把配置里的 `RUN_SOURCE_TESTS` 改为：

```dotenv
RUN_SOURCE_TESTS=true
```

然后执行：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh test-source
```

注意：如果当前仓库存在既有测试失败，应先在发布记录中说明并决定是否豁免，不建议在脚本里静默忽略。

### 5.3 构建镜像和归档产物

只构建镜像：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh build-images
```

只归档非镜像产物：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh package-artifacts
```

完整构建：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh build-release
```

输出位置：

```text
dist/release-${RELEASE_VERSION}/
```

其中包含：

1. `bin/WeKnora_<goos>_<goarch>`
2. `bin/WeKnora-lite_*.tar.gz`
3. `cli/weknora_<goos>_<goarch>`
4. `mcp/*.whl` 和 `mcp/*.tar.gz`
5. `sdk/weknora-client-go_*.tar.gz`
6. `release.env`
7. `checksums/SHA256SUMS`

### 5.4 推送镜像

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh push-images
```

或者完整发布：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh publish-release
```

推送完成后，建议保存镜像 digest：

```bash
docker inspect --format='{{index .RepoDigests 0}}' registry.example.com/weknora/weknora-app:${RELEASE_VERSION}
docker inspect --format='{{index .RepoDigests 0}}' registry.example.com/weknora/weknora-docreader:${RELEASE_VERSION}
docker inspect --format='{{index .RepoDigests 0}}' registry.example.com/weknora/weknora-ui:${RELEASE_VERSION}
docker inspect --format='{{index .RepoDigests 0}}' registry.example.com/weknora/weknora-sandbox:${RELEASE_VERSION}
```

## 6. Docker Compose 自动化上线

适用于单机或小规模生产。

### 6.1 准备生产目录

配置中默认：

```dotenv
COMPOSE_PROJECT_DIR=/opt/weknora
```

执行：

```bash
sudo mkdir -p /opt/weknora
sudo chown -R $USER:$USER /opt/weknora

bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh prepare-compose-dir
```

该命令会复制：

1. `docker-compose.yml`
2. `.env.example`
3. `config/`
4. `scripts/`
5. `migrations/`
6. `skills/`
7. `docker/searxng/`

### 6.2 生成生产 `.env`

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh render-compose-env
```

输出：

```text
/opt/weknora/.env
```

该文件权限会被设置为 `600`。上线前务必人工复核一次：

```bash
sudo sed -n '1,220p' /opt/weknora/.env
```

### 6.3 启动或升级

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh deploy-compose
```

一键上线：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh compose-online
```

如需启用 Compose profile，在配置中设置：

```dotenv
COMPOSE_PROFILES=full
```

多个 profile 使用逗号分隔：

```dotenv
COMPOSE_PROFILES=searxng,jaeger
```

### 6.4 验证

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh verify-compose
```

脚本会执行：

1. `docker compose ps`
2. `docker compose logs app --tail=120`
3. `curl http://127.0.0.1:${APP_PORT}/health`
4. `curl -I http://127.0.0.1:${FRONTEND_PORT}/`

### 6.5 备份

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh backup-compose
```

默认输出：

```text
/opt/weknora/backups/weknora_${DB_NAME}_YYYYmmddHHMMSS.dump
```

对象存储仍需使用 MinIO/S3 工具单独备份，例如：

```bash
mc mirror minio/weknora-prod ./backup-object/weknora-prod-$(date +%Y%m%d%H%M%S)
```

### 6.6 回滚

回滚前先确认数据库迁移是否兼容旧镜像。若新版本已执行不可逆迁移，应优先恢复数据库备份。

```bash
ROLLBACK_VERSION=0.6.0-previous \
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh rollback-compose
```

脚本会修改 `/opt/weknora/.env` 中的 `WEKNORA_VERSION` 并重启 Compose。

## 7. Kubernetes/Helm 自动化上线

适用于正式多节点生产。

### 7.1 生成 Helm values

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh generate-helm-values
```

默认输出：

```text
/tmp/weknora-values-production.yaml
```

这个 values 文件遵循当前 Helm Chart 的真实模板规则：

1. `app.env` 只放 Chart 模板明确渲染的固定字段。
2. `DISABLE_REGISTRATION`、`WEKNORA_LANGUAGE`、`APP_EXTERNAL_URL`、`WEKNORA_SANDBOX_*`、`CRYPTO_MASTER_KEY`、`CRYPTO_SALT` 放入 `app.extraEnv`。
3. 外部 MinIO 凭证通过 `values.minio.accessKeyId` 和 `values.minio.accessKeySecret` 注入。
4. `secrets.existingSecret=weknora-secrets`。

### 7.2 应用 Kubernetes Secret

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh apply-helm-secrets
```

该命令会：

1. 创建或更新 `HELM_NAMESPACE`。
2. 创建或更新 `weknora-secrets`。
3. 如果配置了 `REGISTRY_USERNAME` 和 `REGISTRY_PASSWORD`，创建或更新 imagePullSecret。
4. 如果配置了 `TLS_CERT_FILE` 和 `TLS_KEY_FILE`，创建或更新 TLS Secret。

说明：脚本使用临时文件方式传入主要应用 Secret，避免把数据库密码、JWT、AES 密钥直接放在命令参数里。

### 7.3 执行 Helm 发布

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh deploy-helm
```

该命令会先执行：

```bash
helm lint ./helm -f ${HELM_VALUES_FILE}
```

再执行：

```bash
helm upgrade --install ${HELM_RELEASE} ./helm -n ${HELM_NAMESPACE} -f ${HELM_VALUES_FILE}
```

一键上线：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh helm-online
```

### 7.4 验证

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh verify-helm
```

脚本会检查：

1. `deploy/${HELM_FULLNAME}-app`
2. `deploy/${HELM_FULLNAME}-frontend`
3. `deploy/${HELM_FULLNAME}-docreader`
4. `pods`、`svc`、`ingress`、`pvc`
5. App 最近日志
6. 如果配置了 `APP_EXTERNAL_URL`，会对首页执行 HTTP HEAD 检查

默认 `HELM_FULLNAME=weknora`。如果你的 `HELM_RELEASE` 不是 `weknora`，请按 Chart fullname 规则设置 `HELM_FULLNAME`。

### 7.5 备份

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh backup-helm
```

默认从：

```text
deploy/${HELM_FULLNAME}-postgres
```

执行：

```bash
pg_dump -U ${DB_USER} -d ${DB_NAME} -Fc
```

并保存到：

```text
${K8S_BACKUP_DIR}/weknora_${DB_NAME}_YYYYmmddHHMMSS.dump
```

### 7.6 回滚

先查看历史：

```bash
helm -n weknora history weknora
```

设置回滚版本：

```bash
HELM_ROLLBACK_REVISION=3 \
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh rollback-helm
```

注意：Helm 回滚只回滚 Kubernetes 资源和镜像版本，不会自动回滚数据库结构。涉及迁移的版本必须结合数据库备份恢复策略执行。

## 8. dry-run 预演

配置中可以设置：

```dotenv
DRY_RUN=true
```

然后执行：

```bash
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh build-images
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh generate-helm-values
bash my_docs/运维部署/20260528-02-weknora-prod-automation.sh deploy-compose
```

`dry-run` 会打印大部分将执行的命令，但不会构建镜像、复制生产目录或部署服务。涉及部分配置校验仍会执行，因此未替换 `change-me-*` 时仍会失败，这是预期行为。

## 10. 上线完成标准

当以下条件全部满足时，可以认为自动化上线完成：

1. 四个镜像均已推送到企业镜像仓库。
2. `dist/release-${RELEASE_VERSION}/checksums/SHA256SUMS` 已生成。
3. Compose 或 Helm 部署成功。
4. App `/health` 返回 200。
5. 前端入口可访问。
6. App 日志没有持续迁移失败。
7. docreader Pod/容器健康。
8. 可以登录、创建知识库、上传文档、完成解析、发起问答。
9. 数据库备份和对象存储备份已完成。
10. 发布记录中保存了版本号、commit、镜像 digest、配置文件位置和回滚方案。

## 11. 注意事项

1. 自动化脚本不会自动生成生产密钥；密钥必须由运维人员生成、保存和备份。
2. `.env`、Helm values、Secret 文件不得提交到 Git。
3. 当前 App 默认自动执行 `migrations/versioned`；迁移失败会阻断启动。
4. 升级前必须备份数据库和对象存储。
5. 回滚镜像前必须确认数据库迁移是否兼容旧版本。
6. 如果 Kubernetes 使用外部数据库，需要扩展 Helm values 和 Chart 的数据库连接配置，不能只修改本文脚本。
7. 如果镜像仓库不允许使用路径型 namespace，请把 `REGISTRY` 调整为目标仓库的真实镜像前缀。

至此，WeKnora 从生产编译、打包、发布到 Compose/Helm 上线的自动化流程已具备可执行入口。
