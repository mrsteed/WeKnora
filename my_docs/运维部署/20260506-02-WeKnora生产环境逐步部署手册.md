# WeKnora 生产环境逐步部署手册

## 1. 文档说明

本文不是方案分析，而是一份可以直接按顺序执行的部署手册。

适用范围：

1. 当前仓库代码不做二次改造。
2. 采用当前最优生产方案：Kubernetes + Helm。
3. 使用 Helm Chart 内置的 ParadeDB 和 Redis。
4. 文件存储使用独立 S3/MinIO 兼容对象存储。
5. 文档解析使用内置 `docreader` 服务。

如果你严格按照本文步骤执行，最终应当得到一套可访问、可登录、可上传文档、可完成解析与问答的生产环境。

## 2. 部署目标

本手册最终交付以下生产组件：

1. `frontend`：WeKnora Web 前端
2. `app`：WeKnora 主后端
3. `docreader`：文档解析服务
4. `postgres`：ParadeDB，承担业务库与默认检索能力
5. `redis`：异步流和任务队列
6. 外部 MinIO/S3 兼容对象存储
7. Ingress + HTTPS 外部访问入口

## 3. 执行前约定

本文统一假设以下值，请在执行前替换成你自己的真实值：

| 变量 | 示例值 | 说明 |
|------|--------|------|
| 域名 | `weknora.example.com` | WeKnora 对外访问域名 |
| 命名空间 | `weknora` | Kubernetes Namespace |
| 镜像仓库 | `registry.example.com/weknora` | 企业私有镜像仓库 |
| 版本号 | `v0.0.0` | 本次发布版本 |
| 对象存储地址 | `https://minio.example.com` | 外部 MinIO/S3 兼容地址 |
| 对象存储桶 | `weknora-prod` | 上传文件与解析结果桶 |
| 对象存储前缀 | `prod` | 对象键前缀 |
| StorageClass | `fast-ssd` | 集群内高速存储类 |
| Ollama 地址 | `http://ollama.ollama.svc.cluster.local:11434` | 外部模型服务示例 |

## 6. 准备 WeKnora 发布镜像

这里给两种方式。

如果你当前希望快速上线，可以直接使用：

1. `wechatopenai/weknora-app:<tag>`
2. `wechatopenai/weknora-docreader:<tag>`

### 方式 B：构建并推送企业内部镜像

正式生产推荐方式 B。

### 步骤 5：进入代码仓库

```bash
cd /home/xmkp/workspace/WeKnora
```

### 步骤 6：生成版本变量

```bash
eval "$(./scripts/get_version.sh env)"
echo "$VERSION"
```

如果你有企业内部版本号规范，也可以手工覆盖：

```bash
export VERSION=v0.0.0
```

```bash

  -t registry.example.com/weknora/weknora-app:${VERSION} .

  -t registry.example.com/weknora/weknora-docreader:${VERSION} .

docker build -f frontend/Dockerfile \
  -t registry.example.com/weknora/weknora-ui:${VERSION} frontend/
```

### 步骤 8：推送镜像到私有仓库

```bash
docker push registry.example.com/weknora/weknora-app:${VERSION}
docker push registry.example.com/weknora/weknora-docreader:${VERSION}
docker push registry.example.com/weknora/weknora-ui:${VERSION}
```

### 步骤 9：确认镜像已存在

在镜像仓库控制台或使用仓库查询命令确认以下标签均已存在：

1. `weknora-app:${VERSION}`
2. `weknora-docreader:${VERSION}`
3. `weknora-ui:${VERSION}`

## 7. 准备对象存储

当前仓库默认生产更适合使用 `STORAGE_TYPE=minio`，因此这一步必须先完成。

### 步骤 10：准备外部对象存储账号

你需要提前拿到：

1. `MINIO_ENDPOINT`
2. `MINIO_ACCESS_KEY_ID`
3. `MINIO_SECRET_ACCESS_KEY`
4. 桶名和前缀规划

### 步骤 11：创建对象存储桶

以下以 `mc` 为例：

```bash
mc alias set weknora-prod https://minio.example.com <ACCESS_KEY> <SECRET_KEY>
mc mb weknora-prod/weknora-prod
mc anonymous set none weknora-prod/weknora-prod
```

建议值：

1. 桶名：`weknora-prod`
2. 路径前缀：`prod`

### 步骤 12：验证桶可用

```bash
mc ls weknora-prod
```

预期结果：

1. 能看到 `weknora-prod` 桶
2. 没有认证报错

## 10. 生成生产密钥

这一步非常关键，不能使用仓库里的默认示例值。

### 步骤 18：在终端生成密钥变量

```bash
export DB_PASSWORD="$(openssl rand -base64 24 | tr -d '\n')"
export REDIS_PASSWORD="$(openssl rand -base64 24 | tr -d '\n')"
export JWT_SECRET="$(openssl rand -base64 32 | tr -d '\n')"
export TENANT_AES_KEY="$(openssl rand -hex 16)"
export SYSTEM_AES_KEY="$(openssl rand -hex 16)"
export MINIO_ACCESS_KEY_ID="weknora"
export MINIO_SECRET_ACCESS_KEY="change-me"
```

注意：

1. `TENANT_AES_KEY` 和 `SYSTEM_AES_KEY` 必须长期保留，不能随意变更
2. 一旦丢失，历史加密数据会不可恢复

### 步骤 19：创建 Kubernetes Secret

```bash
kubectl -n weknora create secret generic weknora-secrets \
  --from-literal=DB_USER=postgres \
  --from-literal=DB_PASSWORD="$DB_PASSWORD" \
  --from-literal=DB_NAME=weknora \
  --from-literal=REDIS_USERNAME= \
  --from-literal=REDIS_PASSWORD="$REDIS_PASSWORD" \
  --from-literal=JWT_SECRET="$JWT_SECRET" \
  --from-literal=TENANT_AES_KEY="$TENANT_AES_KEY" \
  --from-literal=SYSTEM_AES_KEY="$SYSTEM_AES_KEY"
```

如果 Secret 已存在，请先删除旧 Secret 再创建，或者改为 `kubectl apply -f` 管理方式。

### 步骤 20：确认 Secret 创建成功

```bash
kubectl -n weknora get secret weknora-secrets
```

```bash
cat > values-production.yaml <<'EOF'

      cpu: "2"
      memory: 4Gi
  env:

    STORAGE_TYPE: minio
    LOCAL_STORAGE_BASE_DIR: /data/files
    AUTO_RECOVER_DIRTY: "true"
    STREAM_MANAGER_TYPE: redis
    CONCURRENCY_POOL_SIZE: "10"
    ENABLE_GRAPH_RAG: "false"
    TZ: Asia/Shanghai
  extraEnv:
    - name: DISABLE_REGISTRATION
      value: "true"
    - name: APP_EXTERNAL_URL
      value: https://weknora.example.com
    - name: OLLAMA_BASE_URL
      value: http://ollama.ollama.svc.cluster.local:11434

frontend:
  replicaCount: 2
  image:
    repository: registry.example.com/weknora/weknora-ui
    tag: v0.0.0
  resources:
    requests:
      cpu: 100m
      memory: 128Mi
    limits:
      cpu: 500m
      memory: 512Mi

docreader:
  replicaCount: 2
  image:
    repository: registry.example.com/weknora/weknora-docreader
    tag: v0.0.0
  resources:
    requests:
      cpu: 500m
      memory: 1Gi
    limits:
      cpu: "2"
      memory: 2Gi
  env:
    STORAGE_TYPE: minio

postgresql:
  enabled: true
  persistence:
    enabled: true
    size: 200Gi
  resources:
    requests:
      cpu: "1"
      memory: 2Gi
    limits:
      cpu: "4"

redis:
  enabled: true
  persistence:
    enabled: true
    size: 20Gi
  resources:
    requests:
      cpu: 200m
      memory: 256Mi
    limits:
      cpu: "1"
      memory: 1Gi

dataFiles:
  persistence:
    enabled: true
    size: 20Gi

ingress:
  enabled: true
  className: nginx
  host: weknora.example.com
  tls:
    enabled: true

  existingSecret: weknora-secrets

  enabled: false

  enabled: false

  enabled: false
```

## 14. 执行数据库迁移

由于当前镜像不会自动迁移数据库，这一步不能跳过。

### 步骤 32：创建迁移 Job 文件

```bash
cd ~/deploy/weknora-prod

cat > weknora-migrate.yaml <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: weknora-migrate
  namespace: weknora
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: registry.example.com/weknora/weknora-app:v0.0.0
          imagePullPolicy: IfNotPresent
          envFrom:
            - secretRef:
                name: weknora-secrets
          env:
            - name: DB_HOST
              value: postgres
            - name: DB_PORT
              value: "5432"
          command:
            - /bin/bash
            - -lc
            - /app/scripts/migrate.sh up
  backoffLimit: 1
EOF
```

### 步骤 33：按实际镜像版本修改迁移 Job

将 `registry.example.com/weknora/weknora-app:v0.0.0` 替换成你的真实镜像地址和 tag。

### 步骤 34：执行迁移

```bash
kubectl apply -f ~/deploy/weknora-prod/weknora-migrate.yaml
```

### 步骤 35：查看迁移日志

```bash
kubectl -n weknora logs job/weknora-migrate -f
```

预期结果：

1. 日志中执行 `migrate.sh up`

### 步骤 36：确认 Job 执行完成

## 21. 常见失败点与处理

### 问题 1：Pod 一直起不来

排查命令：

```bash
kubectl -n weknora describe pod <pod-name>
kubectl -n weknora logs <pod-name>
```

优先检查：

1. 镜像地址是否正确
2. StorageClass 是否存在
3. Secret 是否创建成功
4. MinIO endpoint 是否可达

### 问题 2：数据库迁移失败

优先检查：

1. `weknora-secrets` 中数据库账号密码是否正确
2. `DB_HOST=postgres` 是否可解析
3. `postgres` Pod 是否 Running

### 问题 3：上传成功但图片访问失败

优先检查：

1. MinIO `publicEndpoint` 是否为浏览器可访问地址
2. `APP_EXTERNAL_URL` 是否正确
3. 桶权限和对象路径前缀是否匹配

### 问题 4：首页能打开但 API 失败

优先检查：

1. Ingress 转发是否正常
2. `app` Service 是否正常
3. `app` `/health` 是否返回 200

## 22. 本手册执行完成的判定标准

当以下条件全部满足时，可以视为生产环境部署完成：

1. Helm Release 状态为 `deployed`
2. `frontend`、`app`、`docreader`、`postgres`、`redis` Pod 全部 Running
3. 迁移 Job 成功完成
4. `https://weknora.example.com` 可访问
5. 可以登录系统
6. 可以上传文档并完成解析
7. 可以发起问答
8. 对象存储中已生成上传对象
