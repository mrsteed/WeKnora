# WeKnora 生产级编译、打包、部署详细清单

## 1. 文档定位

本文基于 2026-05-28 当前仓库代码整理，目标是给出一份可落地执行的生产级清单，覆盖从源码编译、产物归档、镜像打包，到生产环境上线、验证、升级和回滚的完整流程。

当前仓库推荐的生产运行形态是容器化部署：

1. 主后端 `app`：Go/Gin 服务，入口为 `cmd/server`。
2. 前端 `frontend`：Vue/Vite 构建后由 Nginx 提供静态资源和反向代理。
3. 文档解析 `docreader`：Python gRPC 服务。
4. Agent Skills 沙箱 `sandbox`：按需执行技能脚本的隔离镜像。
5. PostgreSQL/ParadeDB：业务库和默认混合检索能力。
6. Redis：流式响应和异步任务队列。
7. 可选组件：MinIO、Qdrant、Milvus、Weaviate、Doris、Neo4j、SearXNG、Jaeger、Langfuse、Dex。
8. 独立交付模块：CLI、MCP Server、Go Client SDK、Lite 单机包、macOS 桌面包、小程序插件。

生产推荐路径：先在构建机生成版本化产物和镜像，推送到企业镜像仓库，再通过 Docker Compose 或 Kubernetes/Helm 上线。小规模单机可用 Docker Compose；多节点、正式生产推荐 Kubernetes + Helm。

## 2. 当前代码构建入口总览

| 模块 | 技术栈 | 构建入口 | 推荐生产产物 | 默认输出位置 |
|------|--------|----------|--------------|--------------|
| 主后端 App | Go 1.26、CGO | `make build-prod` 或 `docker/Dockerfile.app` | `WeKnora` 二进制、`weknora-app` 镜像 | 本地 `./WeKnora`；镜像内 `/app/WeKnora` |
| Lite 单机版 | Go + SQLite + 前端静态资源 | `make build-lite`、`make package-lite` | `WeKnora-lite`、tarball | `./WeKnora-lite`；`dist/WeKnora-lite_<version>_<goos>_<goarch>.tar.gz` |
| macOS 桌面版 | Wails + Lite | `make package-mac-app` | `.app` 应用包 | `dist/WeKnora Lite.app` |
| CLI | 独立 Go module | `cd cli && make build` | `weknora` 二进制 | `cli/bin/weknora` |
| Go Client SDK | 独立 Go module | `cd client && go test ./...` | SDK 源码包，不单独出二进制 | 无固定二进制产物 |
| MCP Server | Python 3.10+ | `cd mcp-server && python -m build` | wheel、sdist、控制台命令 | `mcp-server/dist/` |
| 前端 Web | Node 24、Vite、Vue | `cd frontend && npm ci && npm run build` 或 `frontend/Dockerfile` | 静态资源、`weknora-ui` 镜像 | `frontend/dist/`；镜像内 `/usr/share/nginx/html` |
| docreader | Python 3.10.18、uv、gRPC | `docker/Dockerfile.docreader` | `weknora-docreader` 镜像 | 镜像产物 |
| sandbox | Python 3.11 + Node 20 | `docker/Dockerfile.sandbox` | `weknora-sandbox` 镜像 | 镜像产物 |
| 小程序插件 | Node 测试脚本 | `cd miniprogram && npm test` | 小程序源码包 | 无自动打包脚本 |
| 数据库迁移 | golang-migrate | App 启动自动执行，或 `scripts/migrate.sh` | 迁移状态写入数据库 | `migrations/versioned` |

说明：`mcp-server` 是 Python 包，不是 Go 模块；`client` 是 Go SDK，不建议在生产发布中当作服务启动；`docreader` 当前生产交付方式是镜像，不生成本地可执行二进制。

## 3. 构建机准备清单

### 3.1 基础工具

构建机建议使用 Linux x86_64 或 arm64，至少准备：

```bash
git --version
go version
node --version
npm --version
python3 --version
pip --version
docker version
docker compose version || docker-compose version
openssl version
```

推荐版本和约束：

| 工具 | 推荐要求 | 原因 |
|------|----------|------|
| Go | 1.26.x | 根模块和 CLI 模块均声明 Go 1.26 |
| Node.js | 24.x | `frontend/Dockerfile` 使用 `node:24-alpine` |
| npm | 随 Node 24 | 前端 Docker 构建使用 `npm ci` |
| Python | 3.10+ | `mcp-server` 要求 Python >= 3.10；`docreader` 镜像使用 3.10.18 |
| Docker | 支持 BuildKit 更佳 | 构建多阶段镜像 |
| Docker Compose | 优先 v2 插件 | `scripts/start_all.sh` 优先使用 `docker compose` |
| Helm | 3.10+ | Kubernetes 生产部署使用 Helm Chart |
| kubectl | 与目标集群兼容 | Kubernetes 发布、验证和回滚 |

### 3.2 拉取源码并确认版本

```bash
cd /opt/build

git clone <your-weknora-repo-url> WeKnora
cd WeKnora

git status --short
git rev-parse --short HEAD
cat VERSION
./scripts/get_version.sh info
```

当前仓库 `VERSION` 文件为 `0.6.0`。生产发布建议使用不可变版本号，例如 `0.6.0-20260528.1` 或企业内部 release tag。

### 3.3 配置构建代理和私有仓库变量

如在内网构建，建议提前设置：

```bash
export GOPROXY=https://goproxy.cn,direct
export GOSUMDB=off
export GOPRIVATE=<your-private-go-module-pattern>
export APT_MIRROR=http://mirrors.tencent.com/debian
export APK_MIRROR_ARG=mirrors.tencent.com
export REGISTRY=registry.example.com/weknora
export RELEASE_VERSION=0.6.0-20260528.1
```

`docker/Dockerfile.app` 支持 `GOPRIVATE_ARG`、`GOPROXY_ARG`、`GOSUMDB_ARG`、`VERSION_ARG`、`COMMIT_ID_ARG`、`BUILD_TIME_ARG`、`GO_VERSION_ARG` 等构建参数；`docker/Dockerfile.docreader` 支持 `APT_MIRROR`。

## 4. 生产级编译流程

### 4.1 统一生成版本变量

```bash
cd /opt/build/WeKnora

eval "$(./scripts/get_version.sh env)"
export VERSION="${RELEASE_VERSION:-$VERSION}"
export COMMIT_ID="$(git rev-parse --short HEAD 2>/dev/null || echo unknown)"
export BUILD_TIME="$(date -u '+%Y-%m-%d %H:%M:%S UTC')"
export GO_VERSION="$(go version 2>/dev/null || echo unknown)"

printf 'VERSION=%s\nCOMMIT_ID=%s\nBUILD_TIME=%s\nGO_VERSION=%s\n' \
  "$VERSION" "$COMMIT_ID" "$BUILD_TIME" "$GO_VERSION"
```

建议把上述信息写入发布记录，后续排障时可和系统信息页、镜像标签、Git commit 对齐。

### 4.2 编译主后端 App 二进制

生产标准版二进制：

```bash
cd /opt/build/WeKnora
make build-prod
ls -lh ./WeKnora
```

产物说明：

1. 输出二进制：`./WeKnora`。
2. 源码入口：`./cmd/server`。
3. 版本注入：`internal/handler.Version`、`Edition`、`CommitID`、`BuildTime`、`GoVersion`。
4. 链接参数包含 `-w -s`，适合生产发布。
5. CGO 已启用，构建机需要基础编译工具和 sqlite 相关库。
6. protobuf 注册冲突策略通过 `google.golang.org/protobuf/reflect/protoregistry.conflictPolicy=warn` 降级为告警。

如需手工复现 Makefile 行为：

```bash
cd /opt/build/WeKnora

eval "$(./scripts/get_version.sh env)"
LDFLAGS="$(./scripts/get_version.sh ldflags) -X 'google.golang.org/protobuf/reflect/protoregistry.conflictPolicy=warn'"

CGO_ENABLED=1 \
CGO_CFLAGS="-Wno-deprecated-declarations" \
go build -ldflags="-w -s ${LDFLAGS}" -o WeKnora ./cmd/server
```

### 4.3 编译 Lite 单机版

Lite 版会先构建前端到 `web/`，再构建带 SQLite FTS5 tag 的 Go 二进制：

```bash
cd /opt/build/WeKnora
make build-lite
ls -lh ./WeKnora-lite
```

输出：

1. `./WeKnora-lite`：单机二进制。
2. `./web/`：从 `frontend/dist` 同步来的前端静态资源。
3. 使用 `-tags sqlite_fts5`。
4. 适合演示、小规模离线部署或桌面场景，不是多节点生产的首选。

如果已有 `web/` 且需要跳过前端构建：

```bash
SKIP_FRONTEND=1 make build-lite
```

### 4.4 编译 CLI

CLI 是 `cli/` 下的独立 Go module，依赖本仓库 `client/` module。

```bash
cd /opt/build/WeKnora/cli
make tidy
make test
make build
ls -lh ./bin/weknora
./bin/weknora --help
```

输出：

1. 二进制目录：`cli/bin/`。
2. 二进制文件：`cli/bin/weknora`。
3. 版本信息由 `cli/Makefile` 注入到 `github.com/Tencent/WeKnora/cli/internal/build`。
4. 发布时建议重命名为包含平台的文件，例如 `weknora-cli_linux_amd64`。

跨平台构建示例：

```bash
cd /opt/build/WeKnora/cli
mkdir -p dist

GOOS=linux GOARCH=amd64 go build -ldflags="${LDFLAGS:-}" -o dist/weknora_linux_amd64 .
GOOS=linux GOARCH=arm64 go build -ldflags="${LDFLAGS:-}" -o dist/weknora_linux_arm64 .
GOOS=darwin GOARCH=arm64 go build -ldflags="${LDFLAGS:-}" -o dist/weknora_darwin_arm64 .
```

### 4.5 验证 Go Client SDK

`client/` 是 SDK，不是生产常驻服务。发布前至少执行测试和模块校验：

```bash
cd /opt/build/WeKnora/client
go mod tidy
go test ./...
```

如企业内部需要 SDK 源码包，可归档：

```bash
cd /opt/build/WeKnora
mkdir -p dist/sdk

tar --exclude='.git' --exclude='client/.git' \
  -czf dist/sdk/weknora-client-go_${VERSION}.tar.gz client
```

### 4.6 构建 MCP Server Python 包

`mcp-server/` 是独立 Python 包，入口命令在 `pyproject.toml` 中定义：

1. `weknora-mcp-server = main:sync_main`
2. `weknora-server = run_server:main`

构建步骤：

```bash
cd /opt/build/WeKnora/mcp-server
python3 -m venv .venv
. .venv/bin/activate
python -m pip install --upgrade pip build wheel
python -m build
ls -lh dist/
```

输出：

1. `mcp-server/dist/weknora_mcp_server-1.0.0-py3-none-any.whl`
2. `mcp-server/dist/weknora_mcp_server-1.0.0.tar.gz`

安装验证：

```bash
python -m pip install dist/weknora_mcp_server-1.0.0-py3-none-any.whl
WEKNORA_BASE_URL=https://weknora.example.com/api/v1 \
WEKNORA_API_KEY=<your-api-key> \
weknora-mcp-server --check-only
```

生产使用时，MCP Server 通常部署在调用方机器或 MCP Host 所在环境，不建议和 WeKnora App 混在同一个容器中。

### 4.7 构建前端静态资源

```bash
cd /opt/build/WeKnora/frontend
npm ci
npm run build
ls -lh dist/
```

输出：

1. 前端静态资源目录：`frontend/dist/`。
2. Docker 镜像内会复制到 `/usr/share/nginx/html`。
3. Nginx 模板在 `frontend/nginx.conf`，通过环境变量设置 `MAX_FILE_SIZE_MB`、`APP_HOST`、`APP_PORT`、`APP_SCHEME`。

生产镜像构建使用 `frontend/Dockerfile`，构建阶段会执行 `npm ci` 和 `npm run build`。

### 4.8 构建 docreader

本地源码方式主要用于开发；生产推荐直接构建镜像：

```bash
cd /opt/build/WeKnora

docker build \
  --platform linux/amd64 \
  --build-arg TARGETARCH=amd64 \
  --build-arg APT_MIRROR="${APT_MIRROR:-}" \
  -f docker/Dockerfile.docreader \
  -t ${REGISTRY}/weknora-docreader:${VERSION} \
  .
```

镜像内容：

1. Python 3.10.18 运行时。
2. `uv` 创建的依赖环境。
3. `docreader/scripts/generate_proto.sh` 生成的 protobuf 代码。
4. LibreOffice、antiword、Playwright WebKit、grpc_health_probe。
5. 默认启动命令：`uv run -m docreader.main`。
6. 端口：`50051`。

### 4.9 构建 sandbox

Agent Skills 默认 `WEKNORA_SANDBOX_MODE=docker`，因此生产环境需要准备 sandbox 镜像：

```bash
cd /opt/build/WeKnora

docker build \
  --platform linux/amd64 \
  -f docker/Dockerfile.sandbox \
  -t ${REGISTRY}/weknora-sandbox:${VERSION} \
  .
```

`sandbox` 镜像包含 Python 3.11、Node 20、npm/npx、jq，并以 UID 1000 的非 root 用户执行。Compose 中 `sandbox` 是 profile 服务，不常驻；App 执行 Skills 时按需拉起。

### 4.10 小程序插件

当前 `miniprogram/` 只有测试脚本，没有自动生产打包脚本：

```bash
cd /opt/build/WeKnora/miniprogram
npm test
```

如果需要生产发布，应按微信小程序开发者工具或企业 CI 插件的流程上传，本文不把它作为后端生产部署阻断项。

## 5. 生产镜像打包流程

### 5.1 推荐一键构建脚本

当前仓库提供 `scripts/build_images.sh`：

```bash
cd /opt/build/WeKnora
./scripts/build_images.sh --all
```

默认构建以下本地镜像：

1. `wechatopenai/weknora-app:latest`
2. `wechatopenai/weknora-docreader:latest`
3. `wechatopenai/weknora-ui:latest`
4. `wechatopenai/weknora-sandbox:latest`

脚本可选参数：

```bash
./scripts/build_images.sh --app
./scripts/build_images.sh --docreader
./scripts/build_images.sh --frontend
./scripts/build_images.sh --sandbox
./scripts/build_images.sh --clean
```

生产发布建议不要只用 `latest`，应重新打企业仓库版本标签。

### 5.2 手工构建版本化镜像

```bash
cd /opt/build/WeKnora

eval "$(./scripts/get_version.sh env)"
export VERSION="${RELEASE_VERSION:-$VERSION}"
export COMMIT_ID="$(git rev-parse --short HEAD 2>/dev/null || echo unknown)"
export BUILD_TIME="$(date -u '+%Y-%m-%d %H:%M:%S UTC')"
export GO_VERSION="$(go version 2>/dev/null || echo unknown)"
export REGISTRY=registry.example.com/weknora
export PLATFORM=linux/amd64
export TARGETARCH=amd64

# App
docker build \
  --platform ${PLATFORM} \
  --build-arg GOPRIVATE_ARG="${GOPRIVATE:-}" \
  --build-arg GOPROXY_ARG="${GOPROXY:-https://goproxy.cn,direct}" \
  --build-arg GOSUMDB_ARG="${GOSUMDB:-off}" \
  --build-arg VERSION_ARG="${VERSION}" \
  --build-arg COMMIT_ID_ARG="${COMMIT_ID}" \
  --build-arg BUILD_TIME_ARG="${BUILD_TIME}" \
  --build-arg GO_VERSION_ARG="${GO_VERSION}" \
  --build-arg APK_MIRROR_ARG="${APK_MIRROR_ARG:-}" \
  -f docker/Dockerfile.app \
  -t ${REGISTRY}/weknora-app:${VERSION} \
  .

# docreader
docker build \
  --platform ${PLATFORM} \
  --build-arg TARGETARCH=${TARGETARCH} \
  --build-arg APT_MIRROR="${APT_MIRROR:-}" \
  -f docker/Dockerfile.docreader \
  -t ${REGISTRY}/weknora-docreader:${VERSION} \
  .

# frontend
docker build \
  --platform ${PLATFORM} \
  --build-arg MAX_FILE_SIZE_MB="${MAX_FILE_SIZE_MB:-50}" \
  -f frontend/Dockerfile \
  -t ${REGISTRY}/weknora-ui:${VERSION} \
  frontend/

# sandbox
docker build \
  --platform ${PLATFORM} \
  -f docker/Dockerfile.sandbox \
  -t ${REGISTRY}/weknora-sandbox:${VERSION} \
  .
```

### 5.3 镜像标记和推送

```bash
docker tag ${REGISTRY}/weknora-app:${VERSION} ${REGISTRY}/weknora-app:release-${VERSION}
docker tag ${REGISTRY}/weknora-docreader:${VERSION} ${REGISTRY}/weknora-docreader:release-${VERSION}
docker tag ${REGISTRY}/weknora-ui:${VERSION} ${REGISTRY}/weknora-ui:release-${VERSION}
docker tag ${REGISTRY}/weknora-sandbox:${VERSION} ${REGISTRY}/weknora-sandbox:release-${VERSION}

docker push ${REGISTRY}/weknora-app:${VERSION}
docker push ${REGISTRY}/weknora-docreader:${VERSION}
docker push ${REGISTRY}/weknora-ui:${VERSION}
docker push ${REGISTRY}/weknora-sandbox:${VERSION}
```

推送完成后记录镜像 digest：

```bash
docker inspect --format='{{index .RepoDigests 0}}' ${REGISTRY}/weknora-app:${VERSION} || true
docker inspect --format='{{index .RepoDigests 0}}' ${REGISTRY}/weknora-docreader:${VERSION} || true
docker inspect --format='{{index .RepoDigests 0}}' ${REGISTRY}/weknora-ui:${VERSION} || true
docker inspect --format='{{index .RepoDigests 0}}' ${REGISTRY}/weknora-sandbox:${VERSION} || true
```

### 5.4 归档非镜像产物

建议统一放到 `dist/release-${VERSION}/`：

```bash
cd /opt/build/WeKnora
mkdir -p dist/release-${VERSION}/{bin,cli,mcp,sdk,checksums}

# 主后端二进制
make build-prod
cp WeKnora dist/release-${VERSION}/bin/WeKnora_linux_amd64

# Lite 包
make package-lite ${VERSION}
cp dist/WeKnora-lite_${VERSION}_*.tar.gz* dist/release-${VERSION}/bin/ 2>/dev/null || true

# CLI
(cd cli && make build)
cp cli/bin/weknora dist/release-${VERSION}/cli/weknora_linux_amd64

# MCP Server
(cd mcp-server && python3 -m pip install --upgrade build && python3 -m build)
cp mcp-server/dist/* dist/release-${VERSION}/mcp/

# Go Client SDK 源码包
tar --exclude='.git' -czf dist/release-${VERSION}/sdk/weknora-client-go_${VERSION}.tar.gz client

# 校验和
(cd dist/release-${VERSION} && find . -type f ! -path './checksums/*' -print0 | xargs -0 sha256sum > checksums/SHA256SUMS)
```

```bash
cd /opt/build/WeKnora

go test ./...
(cd cli && go test ./...)
(cd client && go test ./...)
(cd frontend && npm ci && npm run build)
(cd mcp-server && python3 test_module.py)
```

### 6.2 镜像级冒烟检查

```bash
cd /opt/build/WeKnora

docker run --rm ${REGISTRY}/weknora-app:${VERSION} /app/WeKnora --help || true
docker run --rm ${REGISTRY}/weknora-docreader:${VERSION} python -m docreader.main --help || true
```

### 6.3 配置和密钥检查

生产 `.env` 或 Helm Secret 必须至少确认：

| 配置项 | 建议值/要求 | 说明 |
|--------|-------------|------|
| `GIN_MODE` | `release` | 生产模式 |
| `DISABLE_REGISTRATION` | `true` | 生产建议关闭开放注册 |
| `DB_DRIVER` | `postgres` | 生产推荐 PostgreSQL/ParadeDB |
| `RETRIEVE_DRIVER` | `postgres` 或专用向量库 | 默认 postgres 走 ParadeDB/pgvector 能力 |
| `DB_USER`、`DB_PASSWORD`、`DB_NAME` | 强密码，独立库 | 不要使用示例密码 |
| `REDIS_PASSWORD` | 强密码 | Compose Redis 使用 `--requirepass` |
| `JWT_SECRET` | 随机高强度 | 用 `openssl rand -base64 48` 生成 |
| `TENANT_AES_KEY` | 固定保存 | 已加密数据依赖该值 |
| `SYSTEM_AES_KEY` | 32 字节 | 用于 API Key 等敏感字段加密 |
| `CRYPTO_MASTER_KEY`、`CRYPTO_SALT` | 推荐显式生成并备份 | 不设置时会持久化到数据卷中的 `.crypto_state.json` |
| `STORAGE_TYPE` | `minio`/`cos`/`tos`/`s3` | 生产不建议只依赖容器本地盘 |
| `APP_EXTERNAL_URL` | `https://weknora.example.com` | IM/图片外链等场景需要 |
| `DOCREADER_ADDR` | `docreader:50051` | Compose/Helm 内部服务名 |
| `AUTO_RECOVER_DIRTY` | `true` | 当前默认即 true，迁移 dirty 状态自动恢复 |
| `WEKNORA_SANDBOX_MODE` | `docker` | Agent Skills 推荐 Docker 沙箱 |
| `WEKNORA_SANDBOX_DOCKER_IMAGE` | 企业仓库 sandbox 镜像 | 使用私有仓库时必须覆盖 |
| `MAX_FILE_SIZE_MB` | 按业务设置 | 修改后需重启 App、frontend、docreader |

密钥生成示例：

```bash
openssl rand -base64 48   # JWT_SECRET
openssl rand -hex 32      # CRYPTO_MASTER_KEY
openssl rand -base64 32   # CRYPTO_SALT
openssl rand -base64 32   # LANGFUSE_SALT
openssl rand -hex 32      # LANGFUSE_ENCRYPTION_KEY
openssl rand -base64 32   # LANGFUSE_NEXTAUTH_SECRET
```

## 7. Docker Compose 生产上线步骤

适用场景：单机、小规模内网环境、演示生产环境。正式多节点或高可用生产建议跳到第 8 章 Kubernetes/Helm。

### 7.1 准备服务器目录

```bash
sudo mkdir -p /opt/weknora
sudo chown -R $USER:$USER /opt/weknora
cd /opt/weknora
```

复制以下文件到服务器：

1. `docker-compose.yml`
2. `.env.example`，改名为 `.env`
3. `config/`
4. `skills/preloaded/`

### 7.5 数据库迁移策略

当前代码在 App 启动时默认自动执行迁移：

1. `AUTO_MIGRATE` 不等于 `false` 时执行。
2. 迁移目录为 `migrations/versioned`。
3. `AUTO_RECOVER_DIRTY` 不等于 `false` 时会尝试自动恢复 dirty 状态。
4. 如果迁移最终失败，当前 App 启动会返回 `database migration failed` 并阻断启动。

生产建议：

1. 首次上线保持自动迁移开启。
2. 升级前先备份数据库。
3. 对强管控环境，可在维护窗口手工执行迁移，再启动 App。

手工迁移命令：

```bash
cd /opt/weknora

docker compose exec app /app/scripts/migrate.sh version
docker compose exec app /app/scripts/migrate.sh up
```

如果 App 尚未启动，可临时运行一次性容器执行迁移：

```bash
docker compose run --rm app /app/scripts/migrate.sh up
```

### 8.6 迁移执行方式

当前 App 启动默认自动迁移。如果需要单独控制迁移窗口，可设置 `AUTO_MIGRATE=false`，先部署数据库和一次性迁移 Job，再启动 App。

一次性迁移 Job 示例：

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: weknora-migrate
  namespace: weknora
spec:
  backoffLimit: 1
  template:
    spec:
      restartPolicy: Never
      imagePullSecrets:
        - name: regcred
      containers:
        - name: migrate
          image: registry.example.com/weknora/weknora-app:0.6.0-20260528.1
          command: ["/bin/bash", "-lc"]
          args:
            - |
              set -e
              cd /app
              /app/scripts/migrate.sh version || true
              /app/scripts/migrate.sh up
          envFrom:
            - secretRef:
                name: weknora-secrets
          env:
            - name: DB_DRIVER
              value: postgres
            - name: DB_HOST
              value: postgres
            - name: DB_PORT
              value: "5432"
            - name: RETRIEVE_DRIVER
              value: postgres
            - name: MIGRATIONS_DIR
              value: /app/migrations/versioned
```

执行：

```bash
kubectl apply -f ~/deploy/weknora-prod/weknora-migrate.yaml
kubectl -n weknora logs job/weknora-migrate -f
kubectl -n weknora get job weknora-migrate
```

如果保留 App 自动迁移，则迁移 Job 可不执行，但升级前仍必须备份数据库。

## 15. 常见故障处理

### 15.1 App 启动失败并提示迁移失败

排查：

```bash
# Compose
docker compose logs app --tail=300

# Kubernetes
kubectl -n weknora logs deploy/weknora-app --tail=300
```

处理顺序：

1. 查看失败 migration 版本。
2. 检查 PostgreSQL 扩展：`pg_trgm`、`vector`、`pg_search`。
3. 检查数据库账号是否有创建扩展、建表、建索引权限。
4. 检查磁盘空间。
5. dirty 状态可执行 `scripts/migrate.sh version`、`scripts/migrate.sh force <N-1>`、`scripts/migrate.sh up`。

### 15.2 上传大文件返回 413

`MAX_FILE_SIZE_MB` 同时影响前端 Nginx、Go 后端、docreader、浏览器侧校验。修改后必须同步重启：

```bash
docker compose up -d frontend app docreader
```

Kubernetes 需要更新 values 后 `helm upgrade`。

### 15.3 docreader 不健康

检查：

```bash
docker compose logs docreader --tail=200
# 或
kubectl -n weknora logs deploy/weknora-docreader --tail=200
```

重点确认：

1. `DOCREADER_ADDR` 是否为 `docreader:50051`。
2. 50051 只需容器/集群内部访问，不建议暴露公网。
3. 如果启用 `GRPC_AUTH_TOKEN`，建议同时启用 TLS。
4. PDF/Office 解析依赖 LibreOffice、Playwright WebKit，镜像构建不能跳过相关步骤。

### 15.4 Agent Skills 执行失败

检查：

1. `WEKNORA_SANDBOX_MODE=docker`。
2. `WEKNORA_SANDBOX_DOCKER_IMAGE` 指向可拉取的 sandbox 镜像。
3. 生产节点允许 App 调用 Docker，或在 Kubernetes 中有对应的沙箱执行方案。
4. `skills/preloaded` 挂载不会覆盖内置技能；entrypoint 会从 `/app/skills/_builtin` 合并缺失项。

### 15.5 前端可访问但 API 失败

检查：

1. `frontend/nginx.conf` 中 `/api/` 代理到 `APP_SCHEME://APP_HOST:APP_PORT`。
2. Compose 中默认 `APP_HOST=app`、`APP_BACKEND_PORT=8080`。
3. Ingress 是否把 `/api/` 转发到正确服务。
4. App `/health` 是否 200。

## 17. 推荐最终执行顺序

1. 从 Git tag 或指定 commit 拉取代码。
2. 生成版本变量并记录 `VERSION`、`COMMIT_ID`、`BUILD_TIME`。
3. 执行源码测试：Go、CLI、Client、Frontend、MCP。
4. 构建 App、docreader、frontend、sandbox 四个生产镜像。
5. 构建 CLI、MCP wheel、Lite 包等独立交付产物。
6. 为所有产物生成 SHA256。
7. 推送镜像到企业镜像仓库并记录 digest。
8. 准备生产 `.env` 或 Helm `values-production.yaml`。
9. 创建数据库、Redis、对象存储、TLS 和 Secret。
10. 首次上线前备份已有数据；全新环境则记录空库初始化时间。
11. 使用 Compose 或 Helm 部署。
12. 观察 App 迁移日志，确认没有 `database migration failed`。
13. 检查 `/health`、Pod/容器状态、Ingress/前端访问。
14. 执行业务验收：登录、模型配置、上传、解析、问答。
15. 验证 CLI 和 MCP Server 能连接生产 API。
16. 固化发布记录：镜像 digest、配置版本、数据库迁移版本、验收截图或日志。
17. 将备份、密钥、回滚命令交接给运维值班人员。

完成以上步骤后，当前代码版本的 WeKnora 即达到可上线生产环境的编译、打包和部署要求。
