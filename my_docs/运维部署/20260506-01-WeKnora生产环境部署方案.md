# WeKnora 生产环境部署方案

## 1. 文档目的

本文基于当前仓库代码、Makefile、Docker Compose、Dockerfile 以及 Helm Chart 的真实实现，给出一套适用于生产环境的部署方案，重点回答以下问题：

1. 当前代码在生产环境最适合采用什么部署方式。
2. 依赖组件分别如何部署安装。
3. 主程序如何部署安装、升级和校验。

本文结论不是泛化建议，而是针对当前仓库可直接落地的方案。

## 2. 基于当前代码的结论

### 2.1 当前代码已经提供了三条运行路径

1. 开发模式：`make dev-start` + `make dev-app` + `make dev-frontend`
2. 容器模式：`docker-compose.yml`
3. Kubernetes 模式：`helm/`

### 2.2 不建议在生产环境沿用 `make dev-app`

原因如下：

1. `make dev-app` 的定位是“本地运行后端应用，需先运行 dev-start”，本质是开发调试模式。
2. 开发模式依赖本地代码目录和本地进程，不具备镜像不可变发布、滚动升级、标准化探活和副本扩缩容能力。
3. 前后端分离在本机直接启动，适合热更新和断点调试，不适合运维托管。

### 2.3 当前代码下的最优生产方案

在“不改代码、不补新部署能力”的前提下，推荐以下主方案：

**Kubernetes + Helm 部署 WeKnora 主程序，使用内置 ParadeDB 和 Redis，外接 S3/MinIO 兼容对象存储。**

这是当前仓库最优而且最稳的生产方案，原因如下：

1. 仓库已经维护 Helm Chart，说明 Kubernetes 是明确支持的正式发布路径。
2. Helm 模板已经把 `app`、`frontend`、`docreader`、`postgres`、`redis` 组织成完整发布单元。
3. 当前默认检索驱动是 `RETRIEVE_DRIVER=postgres`，意味着最少组件方案就是直接使用 ParadeDB，同时承担业务库、关键词检索和向量检索，避免额外引入 Qdrant/Milvus/ES。
4. 当前默认消息流管理是 `STREAM_MANAGER_TYPE=redis`，Redis 属于核心依赖，无法省略。
5. 当前默认文件存储是 `STORAGE_TYPE=minio`，生产环境更适合使用独立对象存储，而不是依赖容器本地文件系统。

## 3. 生产部署总览

### 3.1 推荐架构

```text
用户/浏览器
    |
    v
Ingress / SLB / HTTPS
    |
    +---------------------> frontend (Nginx)
    |
    +---------------------> app (Go/Gin)
                               |
                               +--> docreader (gRPC)
                               +--> ParadeDB(PostgreSQL)
                               +--> Redis
                               +--> S3/MinIO 对象存储
                               +--> 外部模型服务 / Ollama / OpenAI 兼容接口
                               +--> 可选：Neo4j / Langfuse
```

### 3.2 组件分级

#### 核心必需组件

1. `frontend`：Web UI
2. `app`：主业务后端
3. `docreader`：文档解析服务
4. `ParadeDB(PostgreSQL)`：主数据库，同时承担当前默认检索能力
5. `Redis`：流管理与异步任务队列

#### 强烈建议生产启用的外围组件

1. S3/MinIO 兼容对象存储：用于文件存储与文档解析结果访问
2. Ingress + TLS：统一对外入口
3. 私有镜像仓库：固定镜像版本，避免直接拉 `latest`

#### 按功能启用的可选组件

1. `Neo4j`：仅当启用 GraphRAG 时需要
2. `Langfuse`：仅当需要链路追踪和模型调用观测时需要
3. `Qdrant` / `Milvus` / `Weaviate` / `Elasticsearch`：只有明确要替换默认 `postgres` 检索方案时才需要

## 4. 为什么这套方案是“当前代码最优”

### 4.1 组件数最少

当前代码默认 `RETRIEVE_DRIVER=postgres`。这意味着只要部署 ParadeDB，就可以同时覆盖：

1. 主业务数据库
2. 关键词检索
3. 向量检索

这样可以少维护一套独立向量数据库，生产复杂度最低。

### 4.2 与现有 Helm Chart 完全对齐

当前 Helm 模板中：

1. `app` 直接使用服务名 `postgres` 和 `redis`
2. `docreader` 直接使用服务名 `docreader`
3. MinIO 使用 values 注入外部 endpoint，适合接入独立对象存储

这说明当前 chart 对“外部对象存储”支持较好，但对“外部 PostgreSQL/Redis”并未做成通用参数化。基于这一点，零改造的最佳部署路径就是：

1. PostgreSQL/Redis 跟随 chart 一起部署
2. 对象存储独立部署
3. 模型服务独立部署

### 4.3 当前镜像不会自动执行数据库迁移

当前 `docker-entrypoint.sh` 只处理目录权限和内置技能目录合并，不会自动执行迁移。

这意味着生产环境必须把数据库迁移当成单独步骤，而不能假设应用启动后自动完成。

## 5. 资源规划建议

以下为中小规模生产环境的起步配置：

### 5.1 Kubernetes 集群建议

1. Kubernetes 版本：1.25+
2. Helm 版本：3.10+
3. 节点数：至少 3 个工作节点
4. 存储：SSD 类型 StorageClass

### 5.2 建议资源

| 组件 | 副本 | CPU Request | CPU Limit | Memory Request | Memory Limit | 存储 |
|------|------|-------------|-----------|----------------|--------------|------|
| frontend | 2 | 100m | 500m | 128Mi | 512Mi | 无 |
| app | 3 | 500m | 2 | 1Gi | 4Gi | 20Gi |
| docreader | 2 | 500m | 2 | 1Gi | 2Gi | 临时盘 |
| ParadeDB | 1 | 1 | 4 | 2Gi | 8Gi | 200Gi |
| Redis | 1 | 200m | 1 | 256Mi | 1Gi | 20Gi |

说明：

1. `app` 和 `docreader` 可以横向扩容。
2. 当前 chart 中 `postgres` 和 `redis` 都是单副本，属于中小规模生产可用方案，不是多活高可用方案。
3. 如果后续要做数据库高可用，应先扩展 chart 支持外部数据库和外部 Redis，再接入托管服务或专用集群。

## 6. 依赖组件部署安装

## 6.1 Kubernetes 基础设施

### 6.1.1 安装 Ingress NGINX

```bash
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

kubectl create namespace ingress-nginx

helm upgrade --install ingress-nginx ingress-nginx/ingress-nginx \
  -n ingress-nginx
```

如果你们已有公司统一 Ingress Controller，可直接复用，不需要重复安装。

### 6.1.2 准备命名空间

```bash
kubectl create namespace weknora
```

## 6.2 生产密钥准备

以下密钥必须在生产环境重新生成，不能使用 `.env.example` 默认值：

1. `DB_PASSWORD`
2. `REDIS_PASSWORD`
3. `JWT_SECRET`
4. `TENANT_AES_KEY`
5. `SYSTEM_AES_KEY`

生成示例：

```bash
export DB_PASSWORD="$(openssl rand -base64 24 | tr -d '\n')"
export REDIS_PASSWORD="$(openssl rand -base64 24 | tr -d '\n')"
export JWT_SECRET="$(openssl rand -base64 32 | tr -d '\n')"
export TENANT_AES_KEY="$(openssl rand -hex 16)"
export SYSTEM_AES_KEY="$(openssl rand -hex 16)"
```

创建 Kubernetes Secret：

```bash
kubectl -n weknora create secret generic weknora-secrets \
  --from-literal=DB_USER=postgres \
  --from-literal=DB_PASSWORD="$DB_PASSWORD" \
  --from-literal=DB_NAME=weknora \
  --from-literal=REDIS_USERNAME= \
  --from-literal=REDIS_PASSWORD="$REDIS_PASSWORD" \
  --from-literal=JWT_SECRET="$JWT_SECRET" \
  --from-literal=TENANT_AES_KEY="$TENANT_AES_KEY" \
  --from-literal=SYSTEM_AES_KEY="$SYSTEM_AES_KEY"
```

## 6.3 对象存储部署安装

### 6.3.1 推荐方式

生产环境推荐使用独立 S3/MinIO 兼容对象存储，不建议把用户上传文件长期放在 `app` 容器挂载目录中。原因如下：

1. `app` 与 `docreader` 扩容后需要共享文件访问。
2. 文档解析产物需要稳定的外部可访问地址。
3. 对象存储更利于备份、生命周期管理和跨节点访问。

### 6.3.2 推荐选择

1. 企业已有对象存储：优先复用 S3 兼容服务
2. 私有化环境：独立部署 MinIO 集群或单实例 MinIO

### 6.3.3 创建桶

以 MinIO 为例：

```bash
mc alias set weknora-prod https://minio.example.com <ACCESS_KEY> <SECRET_KEY>
mc mb weknora-prod/weknora-prod
mc anonymous set none weknora-prod/weknora-prod
```

建议：

1. 桶名使用 `weknora-prod`
2. 路径前缀使用 `prod/`
3. 使用 HTTPS 对外暴露对象存储

## 6.4 ParadeDB 部署安装

### 6.4.1 为什么必须部署 ParadeDB

当前代码默认：

1. `RETRIEVE_DRIVER=postgres`
2. Helm Chart 使用 `paradedb/paradedb`

因此在当前默认生产方案下，ParadeDB 是必需组件。

### 6.4.2 当前代码下的推荐做法

直接启用 Helm Chart 自带的 `postgres` 组件，由 chart 创建 PVC 持久化数据。

这样做的优点：

1. 与当前模板完全一致
2. 服务发现最简单
3. 不需要额外改 chart 才能接入外部数据库

### 6.4.3 运维注意事项

1. 把 PostgreSQL PVC 放在 SSD 存储类
2. 必须配置定时逻辑备份和存储快照
3. 当前模板是单副本 Deployment，不是主从 HA

## 6.5 Redis 部署安装

Redis 在当前代码里承担 Stream Manager 和异步任务队列能力，属于核心依赖。

当前代码下的推荐做法：

1. 直接使用 Helm Chart 自带的 `redis` 组件
2. 开启 AOF 持久化
3. 使用独立 PVC

运维建议：

1. 生产密码必须更换
2. 避免暴露公网端口
3. 定期持久化备份 AOF 文件或卷快照

## 6.6 模型服务部署安装

WeKnora 本身不强绑某一种模型服务，但生产上线前必须准备好以下外部能力中的至少一组：

1. 对话模型
2. Embedding 模型
3. Rerank 模型

如果使用 Ollama：

1. 将 Ollama 独立部署，不要与 WeKnora `app` 容器混跑
2. 通过 `OLLAMA_BASE_URL` 指向独立服务地址

如果使用 OpenAI 兼容网关或商用模型平台：

1. 优先使用平台统一网关地址
2. 将模型密钥存入 WeKnora 系统配置或 Secret 管理系统

## 6.7 可选组件部署安装

### 6.7.1 Neo4j

只有在启用 GraphRAG 时才需要。如果 `ENABLE_GRAPH_RAG=false`，可以不部署。

### 6.7.2 Langfuse

只有在需要模型调用追踪、成本观测、链路调试时才需要。若上线初期只关注稳定运行，可以暂缓部署。

## 7. 主程序部署安装

## 7.1 镜像策略

生产部署建议固定镜像版本，不要直接长期使用 `latest`。

### 方案 A：直接使用官方镜像

适用于快速上线：

1. `wechatopenai/weknora-app:<tag>`
2. `wechatopenai/weknora-docreader:<tag>`
3. `wechatopenai/weknora-ui:<tag>`

### 方案 B：构建企业内部镜像

适用于正式生产：

```bash
cd WeKnora

eval "$(./scripts/get_version.sh env)"

docker build -f docker/Dockerfile.app \
  -t registry.example.com/weknora/weknora-app:${VERSION} .

docker build -f docker/Dockerfile.docreader \
  -t registry.example.com/weknora/weknora-docreader:${VERSION} .

docker build -f frontend/Dockerfile \
  -t registry.example.com/weknora/weknora-ui:${VERSION} frontend/

docker push registry.example.com/weknora/weknora-app:${VERSION}
docker push registry.example.com/weknora/weknora-docreader:${VERSION}
docker push registry.example.com/weknora/weknora-ui:${VERSION}
```

## 7.2 生产 values 文件

在集群外准备 `values-production.yaml`，示例如下：

```yaml
global:
  storageClass: fast-ssd

app:
  replicaCount: 3
  image:
    repository: registry.example.com/weknora/weknora-app
    tag: v0.0.0
  resources:
    requests:
      cpu: 500m
      memory: 1Gi
    limits:
      cpu: "2"
      memory: 4Gi
  env:
    GIN_MODE: release
    RETRIEVE_DRIVER: postgres
    STORAGE_TYPE: minio
    LOCAL_STORAGE_BASE_DIR: /data/files
    AUTO_RECOVER_DIRTY: "true"
    STREAM_MANAGER_TYPE: redis
    CONCURRENCY_POOL_SIZE: "10"
    ENABLE_GRAPH_RAG: "false"
    TZ: Asia/Shanghai
  extraEnv:
    - name: DISABLE_REGISTRATION
      value: "true"
    - name: APP_EXTERNAL_URL
      value: https://weknora.example.com
    - name: OLLAMA_BASE_URL
      value: http://ollama.ollama.svc.cluster.local:11434

frontend:
  replicaCount: 2
  image:
    repository: registry.example.com/weknora/weknora-ui
    tag: v0.0.0
  resources:
    requests:
      cpu: 100m
      memory: 128Mi
    limits:
      cpu: 500m
      memory: 512Mi

docreader:
  replicaCount: 2
  image:
    repository: registry.example.com/weknora/weknora-docreader
    tag: v0.0.0
  resources:
    requests:
      cpu: 500m
      memory: 1Gi
    limits:
      cpu: "2"
      memory: 2Gi
  env:
    STORAGE_TYPE: minio

postgresql:
  enabled: true
  persistence:
    enabled: true
    size: 200Gi
  resources:
    requests:
      cpu: "1"
      memory: 2Gi
    limits:
      cpu: "4"
      memory: 8Gi

redis:
  enabled: true
  persistence:
    enabled: true
    size: 20Gi
  resources:
    requests:
      cpu: 200m
      memory: 256Mi
    limits:
      cpu: "1"
      memory: 1Gi

dataFiles:
  persistence:
    enabled: true
    size: 20Gi

ingress:
  enabled: true
  className: nginx
  host: weknora.example.com
  tls:
    enabled: true
    secretName: weknora-tls

secrets:
  existingSecret: weknora-secrets

minio:
  enabled: true
  endpoint: https://minio.example.com
  publicEndpoint: https://minio.example.com
  accessKeyId: weknora
  accessKeySecret: change-me
  bucketName: weknora-prod
  pathPrefix: prod
  useSSL: true

neo4j:
  enabled: false

qdrant:
  enabled: false

jaeger:
  enabled: false
```

说明：

1. `RETRIEVE_DRIVER` 保持 `postgres`，这是当前代码下最省运维成本的生产配置。
2. `STORAGE_TYPE` 设为 `minio`，让上传文件和解析结果落到对象存储。
3. `DISABLE_REGISTRATION` 生产建议设为 `true`。
4. `APP_EXTERNAL_URL` 建议显式设置，避免外链文件地址错误。


## 7.4 执行数据库迁移

由于当前镜像不会自动迁移数据库，首次安装和每次涉及表结构变更的升级都要单独执行迁移。

建议使用 Job：

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: weknora-migrate
  namespace: weknora
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: registry.example.com/weknora/weknora-app:v0.0.0
          imagePullPolicy: IfNotPresent
          envFrom:
            - secretRef:
                name: weknora-secrets
          env:
            - name: DB_HOST
              value: postgres
            - name: DB_PORT
              value: "5432"
          command:
            - /bin/bash
            - -lc
            - /app/scripts/migrate.sh up
```

执行：

```bash
kubectl apply -f weknora-migrate.yaml
kubectl -n weknora logs job/weknora-migrate -f
```

迁移成功后，如需重复执行升级迁移，请删除旧 Job 后重新创建：

```bash
kubectl -n weknora delete job weknora-migrate
kubectl apply -f weknora-migrate.yaml
```

## 8. 升级策略

推荐顺序：

1. 备份 ParadeDB 数据卷和对象存储关键数据
2. 推送新版本镜像
3. 更新 `values-production.yaml` 中镜像 tag
4. 执行 `helm upgrade`
5. 执行数据库迁移 Job
6. 验证健康检查和关键业务流程

## 9. 备份与恢复建议

### 9.1 ParadeDB

至少保留以下两类备份：

1. 逻辑备份：`pg_dump`
2. 存储卷快照：PVC 或底层云盘快照

### 9.2 Redis

保留 AOF 文件和卷快照。

### 9.3 对象存储

对 `weknora-prod` 桶开启版本控制、生命周期和跨区备份策略。

## 10. 当前方案的边界

这份方案是“当前代码不改造前提下的最优生产方案”，也有明确边界：

1. Helm 模板当前把 `DB_HOST` 固定为 `postgres`、`REDIS_ADDR` 固定为 `redis:6379`，不适合直接接入外部 PostgreSQL/Redis。
2. `postgres` 和 `redis` 当前是单副本 `Deployment`，适合中小规模生产，但不是严格高可用架构。
3. 数据库迁移目前需要人工或流水线显式执行，未内建为 Helm Hook。

如果后续要继续提升生产等级，建议下一阶段做三项增强：

1. 将 PostgreSQL 和 Redis 改为 external 模式可配置
2. 将数据库类组件升级为专用高可用部署方案
3. 将数据库迁移做成 Helm Hook Job 或 CI/CD 发布步骤

## 11. 备选方案：单机 Docker Compose 生产部署

如果当前没有 Kubernetes 条件，可以使用单机 Docker Compose 作为过渡方案，但不建议作为长期主方案。

### 11.1 适用场景

1. 单机部署
2. 中低并发
3. 以快速上线为目标

### 11.2 基本步骤

```bash
cp .env.example .env
```

至少修改以下配置：

1. `GIN_MODE=release`
2. `DISABLE_REGISTRATION=true`
3. `DB_PASSWORD`
4. `REDIS_PASSWORD`
5. `JWT_SECRET`
6. `TENANT_AES_KEY`
7. `SYSTEM_AES_KEY`
8. `APP_EXTERNAL_URL`
9. `MINIO_*` 或云对象存储参数

启动：

```bash
docker compose --profile minio up -d
```

迁移：

```bash
docker compose exec app /app/scripts/migrate.sh up
```

前置条件：

1. 反向代理独立配置 HTTPS
2. 数据目录挂载到稳定磁盘
3. 做好容器自动重启与备份

## 12. 最终建议

如果现在就要上生产，且目标是“少改代码、稳定上线、后续可扩容”，建议直接采用：

1. Kubernetes + Helm
2. 默认 `postgres` 检索方案
3. 内置 ParadeDB + 内置 Redis
4. 外接 S3/MinIO 对象存储
5. 外接模型服务
6. 单独执行数据库迁移 Job

这条路径与当前仓库实现最一致，运维复杂度最低，也最适合作为当前版本的正式生产部署方案。
