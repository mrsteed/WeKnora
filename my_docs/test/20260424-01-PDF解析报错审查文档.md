# PDF 解析报错审查文档

## 1. 审查范围

本次审查基于以下输入材料：

1. 日志文件：`my_docs/test/pdf解析报错.md`
2. 上传文件：`my_docs/test/20260420-01-AI_OA_PC端与H5页面功能清单-精简AI.pdf`
3. 当前代码实现：MinerU 解析链路、MinIO 文件服务初始化链路、本地开发启动脚本链路

本次目标不是复现问题，而是回答以下几个问题：

1. 上传该 PDF 后，为什么会出现“磁盘读取速度非常大”与服务报错现象。
2. 为什么后续服务无法正常启动。
3. 这两个现象之间的关系是什么。
4. 根因更接近 PDF 本身、MinerU 解析、还是存储/基础设施配置。

## 2. 结论摘要

先给结论：

1. 从日志证据看，**上传的 5.11MB PDF 已成功入 MinIO，并成功进入 MinerU 解析流程**，并不是“上传即失败”。
2. “磁盘读取速度非常大”更像是 **MinerU OCR/版面解析阶段触发了大量本地临时文件读写与系统资源争用**，但日志中**没有直接证据**表明根因是 PDF 大小异常；相反，日志更能证明是 **解析阶段引发了系统级资源抖动**。
3. 随后出现的直接故障是 **Redis I/O timeout、慢 SQL、任务队列异常**，说明当时系统已经进入明显的 **资源阻塞或系统负载异常状态**。
4. “服务无法正常启动”的**直接根因不是 MinerU 把服务逻辑打坏了**，而是 **服务启动时仍按 `STORAGE_TYPE=minio` 初始化文件服务，并强制连接 `localhost:9000` 的 MinIO；而此时本地 MinIO 不可达，导致启动阶段 panic 退出**。
5. 因此，这次问题应拆成两个阶段看：
   - **阶段 A：PDF 触发 MinerU 解析，造成系统资源抖动，进而拖垮 Redis/任务处理链路。**
   - **阶段 B：服务重启时，MinIO 存储仍被设为必需依赖，但 `localhost:9000` 无法连通，导致应用启动失败。**

一句话概括：

**PDF 解析阶段的问题更像“MinerU 解析过程资源占用过高”，而重启失败阶段的问题是“MinIO 启动依赖未满足”。后者是明确、可直接定位的启动失败根因。**

## 3. 关键时间线

### 3.1 文档进入解析链路

日志显示：

1. `15:36:30.035` 开始处理文档任务：
   - `file_path=minio://weknora/10000/.../a2a8d175-9609-461d-ba19-c3620e53dc69.pdf`
2. `15:36:30.044` 解析引擎命中 `mineru`：
   - `engine="mineru"`
3. `15:36:30.060` 存储服务解析为 `provider=minio`
4. `15:36:30.535` MinerU 开始解析文件：
   - `Parsing file=20260420-01-AI_OA_PC端与H5页面功能清单-精简AI.pdf size=5365539 via http://127.0.0.1:8000`

这说明：

1. PDF 文件已经成功上传到 MinIO。
2. WeKnora 已经成功从 MinIO 取到文件内容。
3. MinerU 服务地址 `http://127.0.0.1:8000` 当时是可用的。
4. 进入 MinerU 的文件大小是 `5365539` 字节，约 `5.11MB`。

### 3.2 解析期间系统开始异常

从 `15:36:49` 到 `15:37:35`，日志出现一组连续异常：

1. `SLOW SQL >= 200ms`，认证 token 查询耗时达到 `1439.905ms`

3. Redis 客户端连接被丢弃：
   - `discarding bad PubSub connection: write tcp ... i/o timeout`

这组日志的意义非常明确：

1. 问题已经不再局限于单个 PDF 解析调用。
2. 当时的宿主机或进程所在环境出现了明显的系统级阻塞。
3. Redis、任务队列、数据库访问都被连带拖慢或超时。

### 3.3 服务被中断并退出

日志中出现：

1. `^C`
2. `Received signal: interrupt, starting server shutdown...`
3. `Failed to run application: server error: accept tcp [::]:8081: use of closed network connection`

这里说明：

1. 服务并不是自然恢复，而是被人工中断了。
2. `use of closed network connection` 是中断关机阶段的伴随报错，不是最早根因。

### 3.4 重启失败

随后执行 `make dev-app` 时，服务初始化过程在文件服务构建阶段 panic：

```text
panic: ... initFileService ... failed to check bucket:
Get "http://localhost:9000/weknora/?location=": dial tcp 127.0.0.1:9000: i/o timeout
```

这条日志已经足够下结论：

1. 当前服务启动配置仍然要求使用 MinIO 作为文件存储。
2. 服务在启动阶段会主动连接 MinIO 并检查 bucket。
3. 本地 `localhost:9000` 当时不可达。
4. 因此应用在容器构建阶段直接失败，导致无法启动。

## 4. 代码链路审查

### 4.1 MinerU 解析链路本身

`internal/infrastructure/docparser/mineru_converter.go` 显示：

1. WeKnora 会把整个文件内容读入内存后，通过 HTTP `POST /file_parse` 发给 MinerU。
2. 请求字段中默认：
   - `parse_method=ocr`
   - `return_images=true`
   - `return_content_list=true`
3. `mineruTimeout = 1000 * time.Second`

这意味着：

1. 当前链路会让 MinerU 执行 OCR 路线，而不是轻量文本提取。
2. 返回图片与内容列表，会进一步放大解析阶段的 CPU、内存和 I/O 压力。
3. 这解释了为什么一个“只有 5.11MB”的 PDF，依然可能在 MinerU 阶段造成很高的资源消耗。

也就是说：

**PDF 文件体积小，不代表 OCR 解析成本低。**

如果 PDF 页面多、页面复杂、包含大量截图/扫描页/嵌图，MinerU 仍可能产生大量图像解码、临时页渲染、OCR 中间文件和图片输出。

### 4.2 服务启动为何会被 MinIO 卡死

`internal/container/container.go` 的 `initFileService()` 逻辑表明：

1. `STORAGE_TYPE` 如果是 `minio`，就必须读取：
   - `MINIO_ENDPOINT`
   - `MINIO_ACCESS_KEY_ID`
   - `MINIO_SECRET_ACCESS_KEY`
   - `MINIO_BUCKET_NAME`
2. 然后调用 `file.NewMinioFileService(...)`

`internal/application/service/file/minio.go` 的 `NewMinioFileService()` 又会在初始化阶段执行：

1. `BucketExists(...)`
2. 若 bucket 不存在则 `MakeBucket(...)`

所以这不是“懒加载存储服务”，而是“**启动时强校验 MinIO 可用性**”。

只要 MinIO 不可达，服务就起不来。

### 4.3 本地开发脚本为何会访问 localhost:9000

`.env` 显示：

1. `STORAGE_TYPE=minio`
2. `.env` 里写的是 `MINIO_ENDPOINT=minio:9000`

但本地开发脚本 `scripts/dev.sh` 会在本地运行模式下导出：

1. `DOCREADER_ADDR=localhost:50051`
2. `MINIO_ENDPOINT=localhost:9000`
3. `REDIS_ADDR=localhost:6379`

这说明：

1. 用 `make dev-app` 本地启动时，程序不会使用容器内的 `minio:9000`。
2. 它会改连宿主机上的 `localhost:9000`。
3. 只要本地 MinIO 容器没起来，或者端口未监听，启动就必然失败。

## 5. 根因判断

## 5.1 不是 PDF 文件大小本身导致

从日志看，这个 PDF 大小约 `5.11MB`，不能直接算“大文件”。

但 MinerU 处理成本不由原始文件大小单独决定，更关键的是：

1. 页数
2. 是否扫描版
3. 页面是否为整页图片
4. 图表、截图、复杂排版数量
5. OCR 与图片回传是否开启

因此：

**不能因为 PDF 只有 5.11MB，就排除 MinerU 高负载。**

## 5.2 第一阶段根因：MinerU 解析触发资源争用

从现有日志能支持的最强结论是：

1. 文档在进入 MinerU 解析后不久，系统开始出现 Redis I/O timeout、慢 SQL、任务处理异常。
2. 这更符合“宿主机 I/O、CPU 或内存资源被明显拖垮”的现象。
3. 由于日志没有直接打印 MinerU 容器内部资源明细，因此不能把根因绝对化到某一个单项资源。

本次最合理的审查结论是：

**MinerU 解析过程大概率触发了高资源占用，进而造成整机或相关进程链路阻塞。**

## 5.3 第二阶段根因：本地 MinIO 不可达导致服务无法启动

这个结论是明确且已经被日志与代码双重证实的：

1. 应用配置仍然是 `STORAGE_TYPE=minio`
2. 本地开发模式把 `MINIO_ENDPOINT` 改成 `localhost:9000`
3. 启动时会立即 `BucketExists`
4. 日志明确报：

```text
Get "http://localhost:9000/weknora/?location=": dial tcp 127.0.0.1:9000: i/o timeout
```

所以：

**服务无法启动的直接原因是 MinIO 不可达，而不是数据库迁移失败、Redis 配置错误或 MinerU endpoint 缺失。**

## 6. 为什么会出现“看起来像 MinerU 导致服务起不来”的错觉

因为两个问题发生得很接近：

1. 上传 PDF 后，MinerU 解析阶段先把系统拖入异常状态。
2. 运维或开发者随后中断服务并重启。
3. 重启时 MinIO 端口不可达，应用又在启动阶段强依赖 MinIO。

最终表现成：

1. 先是 PDF 解析出事；
2. 然后服务起不来；
3. 于是容易把“起不来”也归咎给 MinerU 本身。

但严格按证据拆解后：

1. **MinerU 负责触发资源异常。**
2. **MinIO 不可达负责导致重启失败。**

这两者有关联，但不是同一个根因。

## 7. 修复建议

### 7.1 先修复“无法启动”问题

这是当前最直接的问题，优先级最高。

建议二选一：

1. 如果当前仍要使用 MinIO：
   - 确保本地 `localhost:9000` 确实有 MinIO 服务监听；
   - 确保 bucket `weknora` 可达；
   - 再启动 `make dev-app`。
2. 如果当前不再使用本地 MinIO，而改用远端 MinIO：
   - 把 `.env` / 启动环境中的 `MINIO_ENDPOINT` 改为真实远端地址；
   - 不要再让 `scripts/dev.sh` 把 endpoint 覆盖成 `localhost:9000`；
   - 或直接把 `STORAGE_TYPE` 改成 `local` 做临时恢复。

### 7.2 限制 MinerU 对系统资源的冲击

建议从下面几项着手：

1. 对 MinerU 解析任务增加并发限制，避免多个 OCR 任务同时跑满磁盘与 CPU。
2. 对 PDF OCR 使用更保守的策略：
   - 非扫描 PDF 优先走文本提取而不是 OCR；
   - 对大页数文档加页数上限或分段解析。
3. 关闭不必要的返回项，例如评估是否必须 `return_images=true`。
4. 对 MinerU 容器加资源隔离：
   - CPU quota
   - 内存上限
   - 临时目录独立挂载
5. 监控 `/tmp`、MinerU 输出目录、磁盘吞吐与 iowait，避免再次把 Redis/数据库一起拖慢。

### 7.3 加强启动阶段的容错性

当前 `initFileService()` 对 MinIO 是强依赖启动校验，这在生产环境可以接受，但在本地开发阶段较脆弱。

可考虑后续优化：

1. 启动时只做配置合法性校验，不立即 `BucketExists`。
2. 将 MinIO 连通性检查改为延迟到首次真实读写时。
3. 或提供 `SKIP_STORAGE_HEALTHCHECK=true` 一类的开发模式开关。

这样即使 MinIO 暂时未就绪，也不会让整个应用完全起不来。

## 8. 最终审查结论

本次事件的最终判断如下：

1. 该 PDF 文件已成功上传到 MinIO，并成功进入 MinerU 解析，不是上传阶段失败。
2. MinerU OCR 解析阶段大概率触发了明显的资源争用，导致 Redis、任务队列和数据库访问出现超时与抖动。
3. 服务后续无法正常启动的直接根因，是本地开发模式仍要求连接 `localhost:9000` 的 MinIO，并在启动阶段做 bucket 检查；而当时 MinIO 不可达，于是应用 panic 退出。
4. 因此，“PDF 解析异常”和“服务重启失败”是连续发生的两个问题：
   - 前者偏资源问题；
   - 后者偏存储依赖配置问题。

如果只修 MinerU，不修 MinIO 启动依赖，服务仍然起不来；
如果只恢复 MinIO，不限制 MinerU 资源占用，后续仍可能再次出现类似抖动。

## 9. 建议的排查顺序

建议按下面顺序继续处理：

1. 先恢复 MinIO 可达性，保证服务能启动。
2. 确认当前是否真的还要使用本地 MinIO；如果不用，立即切换为远端 MinIO 或 local 存储。
3. 单独复测这份 PDF 的 MinerU 解析，记录：
   - 解析耗时
   - MinerU 容器 CPU / 内存 / 磁盘吞吐
   - `/tmp` 或输出目录增长情况
4. 若复测再次出现高 I/O，再对 MinerU 策略做限流、降级 OCR、减少返回图片等优化。
