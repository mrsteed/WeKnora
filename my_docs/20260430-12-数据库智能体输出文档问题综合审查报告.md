# 数据库智能体输出文档问题综合审查报告

## 1. 文档说明

本报告用于整合以下两份已有审查文档，并给出统一后的最终结论：

1. [my_docs/20260430-10-数据库类型智能体查询输出文档卡住问题代码审查报告.md](my_docs/20260430-10-数据库类型智能体查询输出文档卡住问题代码审查报告.md)
2. [my_docs/20260430-11-数据库智能体完整结论丢失问题复审报告.md](my_docs/20260430-11-数据库智能体完整结论丢失问题复审报告.md)

整合后的结论是：

- “卡住感”确实存在，但它不是本次问题的第一根因。
- 本次主故障是“完整 final_answer 已经生成，但在后端协议链路中没有被正确下发和回填”。
- 数据库 schema 过大、最终收尾上下文过重、前端终态提示不足，这些问题依然成立，但应归类为放大器，不应再作为主结论。
- 与提交 `8eae3a537c1edefb36cfab32e4db8c4332170ceb` 高度相关，但相关点不只是“长文任务入口”，而是该提交同时改动了 Agent 最终答案流式下发、complete 收尾和前端完成态消费。

## 2. 最终统一结论

这次数据库智能体“输出完整文档时卡住”的现象，经过日志复审后，应统一表述为：

1. 模型已经生成了完整文档。
2. 完整文档主要存在于 `final_answer` 工具参数中。
3. 当前后端 Agent 协议在 `tool_calls` 轮次里，错误地把过渡性 `answer` chunk 当成最终答案流给前端。
4. 随后在识别到 `final_answer` 工具后，又错误地假设“完整答案已经流过了”，因此只补了 `done=true`，没有把权威完整正文再次下发。
5. complete 阶段又没有优先用 `AgentCompleteData.FinalAnswer` 覆盖前面那份半成品内容。

因此，用户最终看到的是：

- 半成品文档
- 长时间保持“生成中”的状态
- 或者消息最终结束了，但正文依旧残缺

所以，本次问题的主故障不是“模型没生成出来”，也不是“单纯慢收尾”，而是：

**完整文档已经生成，但在 final_answer -> complete 的后端协议链路里被截断并丢失了。**

## 3. 统一证据链

## 3.1 第一组日志已经说明“普通流式内容”与“完整 final_answer”不一致

你最初提供的数据库日志里，已经存在关键数字反差：

1. `content_len=92`
2. `final_answer tool: answer=5835 chars`

这意味着：

- 这一轮模型在普通流式 `content` 里只留下了 92 个字符。
- 但 `final_answer` 工具参数里却有 5835 字符的完整正文。

如果完整文档真的已经通过普通 `answer` chunk 流给前端，那么 `content_len` 不会只有 92。

所以从第一组日志开始，其实就已经说明：

- 权威答案在 `final_answer` 工具里。
- 前面流出来的并不是完整成品。

## 3.2 第二组日志把这个结论彻底坐实

补充日志文件：

- [my_docs/test/1.md](my_docs/test/1.md)

这份日志里有更强对照：

1. `content_len=360`
2. `final_answer tool: answer=7887 chars`

并且日志中的预览文本只是：

- “All data has been collected. Let me now synthesize the final answer.”
- 接下来几条概括性 bullet

这清楚说明：

- 前面流出的 360 字符只是工具调用前的总结性前言。
- 真正的完整 Markdown 文档在 `final_answer` 工具参数里，长度为 7887 字符。

因此，“完整内容没生成出来”这个解释已经被日志直接推翻。

## 3.3 之前报告中的“慢收尾”判断仍然有事实基础，但只能降级为次要结论

此前文档 [my_docs/20260430-10-数据库类型智能体查询输出文档卡住问题代码审查报告.md](my_docs/20260430-10-数据库类型智能体查询输出文档卡住问题代码审查报告.md) 中提到的这些观察仍然成立：

1. `external_database_schema` 默认会输出全量 schema。
2. 最后一轮 prompt token 很大，例如 32633。
3. 最后一轮耗时很长，例如 53 秒、63 秒。
4. 前端在终态到达前只显示生成中，容易让用户误判为卡死。

这些问题解释的是：

- 为什么最后一轮很慢
- 为什么用户容易感知成“卡住”

但它们不能解释：

- 为什么后端已经拿到了 5835 字符或 7887 字符完整 final_answer，前端却没有看到它

所以它们应被保留为性能与体验放大器，而不再作为主根因。

## 3.4 与提交 8eae3a537c1edefb36cfab32e4db8c4332170ceb 的关联性判断

在补充审查该提交后，需要对本报告增加一条版本关联说明。

结论是：

1. 本次数据库智能体“输出完整文档时卡住且正文残缺”的问题，与提交 `8eae3a537c1edefb36cfab32e4db8c4332170ceb` 高度相关。
2. 但相关点不应狭义理解为“新增了长文输出模式，所以数据库智能体也被影响”。
3. 更准确地说，是该提交在引入长文输出能力时，同时改动了 Agent 收尾链路中的多个关键判断，导致数据库智能体所走的同一条协议链也被影响。

具体来看：

1. [internal/agent/think.go](internal/agent/think.go) 中，`ResponseTypeAnswer` 的流出范围被放宽。此前逻辑更偏向只处理明确来源的最终答案流；该提交后的逻辑只要遇到 `answer` chunk 就直接发 `EventAgentFinalAnswer`。这会把 tool-calling 轮次中的过渡前言提前流给前端。
2. [internal/agent/observe.go](internal/agent/observe.go) 中，引入了“如果答案已经流过，则 `final_answer` 成功解析后只补 done”的假设。但数据库智能体日志已经证明，前面流过的只是短前言，不是权威全文。
3. [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go) 中，complete 阶段仍然先信任 `h.finalAnswer`，只有为空才回退到 `data.FinalAnswer`。这使得前面被污染的半成品内容在 complete 阶段无法被完整正文覆盖。
4. [frontend/src/views/chat/index.vue](frontend/src/views/chat/index.vue) 中，`complete` 事件主要用于更新 completion 状态和 `agent_complete` 元数据，不负责用 complete 包里的最终正文回填消息内容。因此一旦后端正文漏发，前端无法兜底修正。

因此，这个提交和当前故障的关系应定性为：

- **强相关**
- **大概率是引入当前回归的关键版本**
- **问题根源在共享的 Agent 收尾协议改动，而不是“长文任务”这个功能入口本身**

换句话说，如果只说“这和长文输出模式无关”，结论会过轻；如果直接说“数据库智能体问题就是长文任务功能本身导致”，结论又会过重。

最准确的表述应是：

**该提交在实现长文输出能力时，顺带改动了 Agent 最终答案的共享收尾链路，而数据库智能体正好复用了这条链路，所以被同一批协议改动一并影响。**

## 4. 主故障链路

当前真正的故障链路可以分成三刀。

## 4.1 第一刀：tool-calling 轮次中，系统过早把过渡性 `answer` chunk 当作最终答案流出

相关代码：

- [internal/agent/think.go](internal/agent/think.go)

当前实现中，只要模型流里出现 `ResponseTypeAnswer`，就立刻发 `EventAgentFinalAnswer` 给前端。

问题在于，这里没有区分两种场景：

1. 自然结束场景：模型真的在直接写最终答案。
2. tool-calling 场景：模型只是先说几句“我来总结一下”，随后真正调用 `final_answer` 工具提交最终结果。

数据库智能体当前落在第二种场景。

因此：

- “现在所有数据都已获取，让我汇总结果”
- “All data has been collected. Let me now synthesize the final answer”

这类过渡文本，被错误地当作最终答案片段流给前端和消息累积链路。

这一步污染了后续 `h.finalAnswer` 的内容基础。

## 4.2 第二刀：识别到 `final_answer` 工具后，只发 done，不发权威正文

相关代码：

- [internal/agent/observe.go](internal/agent/observe.go)

当前实现中，当 `ParseFinalAnswerArgs(rawArgs)` 成功时，逻辑默认认为：

- 完整答案此前已经通过流式 `answer chunk` 发出
- 这里不必再重复发送正文，只需要补一个 `Done=true`

所以 `recovered == true` 时，系统只发：

1. `Content: ""`
2. `Done: true`

而不会把 `answer=5835 chars` 或 `answer=7887 chars` 的正文再次下发。

但新增日志已经证明这个前提是错的：

- 前面流过的不是完整答案
- 真正的完整答案还在 `final_answer` 工具参数里

于是系统出现了最关键的问题：

**后端已经拿到了权威完整结果，但协议层选择不发。**

## 4.3 第三刀：complete 阶段拒绝使用权威 `FinalAnswer` 覆盖半成品

相关代码：

- [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)

`handleComplete` 当前逻辑是：

1. 先把 `h.finalAnswer` 写到 `assistantMessage.Content`
2. 只有当 `h.finalAnswer == ""` 时，才用 `data.FinalAnswer` 做 fallback

这里的核心错误在于：

- `h.finalAnswer` 并不保证是权威最终答案
- 它只是前面通过 `handleFinalAnswer` 收到的流式内容累积
- 一旦前面流入的是半成品，那么 `h.finalAnswer` 就会变成“非空但不完整”

这时 complete 阶段会错误地认为：

- “已经有内容了，不需要 `data.FinalAnswer`”

结果就是：

1. 消息存储保留半成品
2. fallback 不触发
3. 前端 complete 事件只更新状态，不更新完整正文
4. 用户最终看到的仍然是残缺版本

## 5. 次要放大器

下面这些问题依旧需要修，但它们的定位应明确为“放大器”。

## 5.1 Schema 默认全量展开，导致 prompt 体积失控

相关代码：

- [internal/agent/tools/external_database_schema.go](internal/agent/tools/external_database_schema.go)
- [internal/application/service/database_schema.go](internal/application/service/database_schema.go)

当前在未指定 `tables` 时，系统会把数据库下所有命中表的主键、所有列、类型、注释完整拼成 prompt 文本。

它带来的问题是：

1. 第一轮上下文很大
2. 模型更容易在工具调用前生成长篇“中间总结”
3. 最终收尾阶段更慢，用户更容易觉得卡住

但它本身并不会直接导致“7887 字符 final_answer 不显示”。

## 5.2 final answer 阶段再次原样回灌工具输出，继续拉高最后一轮耗时

相关代码：

- [internal/agent/finalize.go](internal/agent/finalize.go)

当前 `streamFinalAnswerToEventBus` 会把每个工具的 `toolCall.Result.Output` 原样拼回给模型。

这会导致：

1. schema 工具输出再次膨胀 prompt
2. query 工具逐行逐列的明细再次进入上下文
3. 最后一轮又慢又重

这会放大用户看到半成品后长期停留的“卡住感”，但仍不是完整答案丢失的主因。

## 5.3 前端终态提示不足，加剧“卡住”感知

相关代码：

- [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue)
- [frontend/src/views/chat/index.vue](frontend/src/views/chat/index.vue)

当前前端逻辑是：

1. 只要 `answer` 事件开始流出，就渲染正文
2. 只有收到终态 `completion_status` 或 `stop` 才真正结束生成态

这意味着：

- 一旦半成品先流出
- 用户就会看到文档正文已经有了
- 但界面仍处于生成中

这会非常自然地被理解成“卡住”。

所以前端提示问题是真实存在的，但它放大的是故障观感，不是故障根源。

## 6. 修复方案

## 6.1 P0 根修：tool-calling 轮次不要立刻把 `answer` chunk 当成最终答案流出

建议改造：

- [internal/agent/think.go](internal/agent/think.go)

建议策略：

1. 在 `streamThinkingToEventBus` 内先缓存本轮 `ResponseTypeAnswer` chunk
2. 等整轮结束后再判断：
   - 如果本轮没有 tool calls，且是自然结束，可以把缓存内容作为最终答案流出
   - 如果本轮出现了 `final_answer` 工具，就不要把这些缓存 chunk 当成最终答案

这个改动直接切断“过渡前言污染最终答案流”的问题。

## 6.2 P0 根修：`final_answer` 工具成功解析后，必须下发权威正文

建议改造：

- [internal/agent/observe.go](internal/agent/observe.go)

建议逻辑：

1. 当 `ParseFinalAnswerArgs(rawArgs)` 成功时，不能再默认认为正文已经流过
2. 应明确把解析出的 `answer` 作为权威最终答案正文发送给前端
3. 如需避免重复，可增加协议字段，例如：
   - `authoritative: true`
   - `replace_content: true`

否则，当前“只发 done”会持续让完整结果丢失。

## 6.3 P0 根修：complete 阶段必须优先信任 `AgentCompleteData.FinalAnswer`

建议改造：

- [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)

建议把当前“只有 `h.finalAnswer == ""` 才 fallback”的判断，改成：

1. 只要 `data.FinalAnswer` 非空且与 `h.finalAnswer` 不一致，就优先使用 `data.FinalAnswer`
2. 如果 `data.FinalAnswer` 更长、更完整，也应覆盖当前内容
3. fallback 条件应从“是否为空”升级为“是否与权威最终答案一致”

这样可以避免“已经有一点半成品，所以拒绝回填完整正文”的错误。

## 6.4 P1 协议兜底：complete 事件携带 `final_answer`，前端可回填正文

建议：

1. complete 事件数据中显式加入 `final_answer`
2. 前端在收到 complete 时，如果发现：
   - 当前 `message.content` 明显短于 `final_answer`
   - 或当前内容与 `final_answer` 不一致
   则直接回填或替换为权威正文

这样即使上游某一环漏发正文，前端也能在 complete 阶段收口为正确内容。

## 6.5 P2 性能治理：保留 schema 与 query 输出压缩

此前两份报告里提到的性能治理仍然建议保留：

1. `external_database_schema` 默认摘要模式
2. `external_database_query` 面向 LLM 的紧凑摘要输出
3. `finalize.go` 不再原样回灌工具全文
4. 前端增加“正在整理最终文档”状态提示

这些项依然重要，因为它们能显著降低最后一轮耗时和“卡住感”，但应作为第二优先级处理。

## 7. 推荐落地顺序

### 第一阶段：先修主故障

1. 修 [internal/agent/think.go](internal/agent/think.go)，停止过早流出 tool-calling 轮次的 answer chunk
2. 修 [internal/agent/observe.go](internal/agent/observe.go)，`final_answer` 成功解析后下发权威正文
3. 修 [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)，complete 阶段优先信任 `FinalAnswer`

### 第二阶段：补协议兜底

1. complete 事件增加 `final_answer`
2. 前端在 complete 时支持正文回填替换
3. 对终态信号丢失或正文不一致增加兜底校验

### 第三阶段：补性能与体验

1. schema 输出摘要化
2. query 输出预算压缩
3. final answer 上下文压缩
4. 前端增加“正在整理最终文档”阶段提示

## 8. 最终判断

统一后的最终判断如下：

1. 数据库智能体当前确实存在“输出文档时像卡住”的用户感知问题
2. 但这次问题的主根因不是“模型生成太慢”
3. 主根因是：完整 final_answer 已经生成，但被后端协议链路错误地丢失了
4. 该故障与提交 `8eae3a537c1edefb36cfab32e4db8c4332170ceb` 高度相关，且大概率就是由该提交对共享 Agent 收尾链路的改动引入或显著放大
5. schema 过大、上下文过重、前端终态提示不足，只是把这个问题放大得更明显

因此，如果只做性能优化或只改前端 loading 文案，问题会缓解，但不会根治。

真正的根修点在于：

- 修正 final_answer 的权威正文下发链路
- 修正 complete 阶段的最终答案覆盖逻辑

## 9. 一句话结论

这次问题不应再定义为“完整文档生成太慢导致卡住”，而应定义为：

**完整文档已经生成，但在 final_answer -> complete 的后端协议链路中被截断并丢失了；而大 schema 与长尾收尾只是进一步放大了用户的卡住感。**
