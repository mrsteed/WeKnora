# 数据库源扩展代码审查与实施方案

> 日期：2026-05-06  
> 目标：基于当前仓库真实实现，审查“数据库型知识库 / 数据库数据源”现状，明确当前实际支持范围，并给出一套可落地、可分阶段推进的数据库源扩展方案。

---

## 一、审查结论

### 1.1 结论摘要

先给结论：**当前代码并不是只支持 PostgreSQL。数据库型知识库这条链路，后端与前端都已经同时支持 MySQL 和 PostgreSQL。**

现有实现已经具备以下能力：

1. 创建 `type=database` 的知识库。
2. 为数据库型知识库创建外部数据库数据源。
3. 通过数据库连接器做连接校验、Schema 发现、表/视图资源枚举。
4. 为数据库型知识库保存 Schema 快照。
5. 基于 Schema 快照做只读 SQL 校验与实时查询。
6. 在前端配置页中选择 MySQL 或 PostgreSQL 作为数据库引擎。

但如果目标是继续扩展到 SQL Server、ClickHouse、Oracle 等数据库，当前实现还有几个明确瓶颈：

1. **数据库类型识别在多处是硬编码**，并没有完全收敛到注册表能力。
2. **SQL 方言层只正式覆盖 MySQL / PostgreSQL**，新数据库接入不只是“加一个连接器”。
3. **前端数据库连接器定义是写死的**，新增数据库需要前后端双改。
4. **连接配置模型偏向 MySQL / PostgreSQL 通用场景**，对 Oracle / SQL Server 这类有额外连接语义的引擎承载不足。
5. **Schema 模型目前更适合“单 database + 单 schema”视角**，对 catalog / owner / 多 schema 数据库的表达能力不够强。

因此，合理路线不是“一次性接所有数据库”，而是：

1. 先把数据库源能力从“二库特化”整理成“元数据驱动 + 方言驱动”的扩展骨架。
2. 第一批新增数据库优先选择 **SQL Server** 和 **ClickHouse**。
3. **Oracle 放到第二阶段**，因为连接、权限、驱动与运维复杂度都明显更高。

## 三、当前代码现状

### 3.1 当前已经落地的数据库型知识库链路

当前数据库源能力不是概念稿，而是已经打通了一条完整主链路：

#### 3.1.1 运行时数据库连接器层

`internal/databaseconnector/connector.go` 定义了统一的 `DatabaseConnector` 接口，包含：

1. `Type()`
2. `Validate()`
3. `DiscoverSchema()`
4. `Query()`
5. `Dialect()`

这说明数据库源能力的核心抽象已经存在，后续扩展数据库类型时，不需要推翻架构。

#### 3.1.2 已实现的数据库连接器

当前已经有两个正式实现：

1. `internal/databaseconnector/mysql/connector.go`
2. `internal/databaseconnector/postgres/connector.go`

两者都实现了：

1. 连接校验。
2. 连接池缓存。
3. 表、列、主键、外键、索引的元数据发现。
4. 只读查询执行。

这意味着当前仓库对“外部业务数据库实时访问”的支持，已经不是 PostgreSQL-only，而是 **MySQL + PostgreSQL 双实现**。

#### 3.1.3 数据源管理适配层

`internal/datasource/connector/database/adapter.go` 把数据库连接器桥接进现有 `datasource.Connector` 体系中，使数据库数据源可以复用数据源管理 API 做：

1. 创建时校验。
2. 资源发现。
3. 在前端显示为可选数据源。

但它故意拒绝 `FetchAll` / `FetchIncremental`，说明当前数据库源不是“同步导入型”，而是**实时连接 + Schema 快照 + 实时查询型**。

#### 3.1.4 Schema 快照服务

`internal/application/service/database_schema.go` 负责：

1. 根据 DataSource 找到对应连接器。
2. 发现 Schema。
3. 应用表白名单、列黑名单等策略。
4. 保存快照。
5. 生成 Prompt Schema 供后续 NL2SQL 或 Agent 使用。

这层说明系统已经把“数据库结构”当成正式的一类知识上下文，而不是只做临时查询。

#### 3.1.5 结构化查询服务

`internal/application/service/structured_query.go` 负责：

1. 读取数据库知识库对应的 Schema 快照。
2. 根据数据库类型获取连接器。
3. 做 SQL 校验。
4. 发起只读查询。
5. 记录查询审计日志。

这是一条标准的“受控实时查询”链路，说明数据库型知识库已经是产品级能力，而不是实验代码。

#### 3.1.6 前端能力也已同步到位

`frontend/src/views/knowledge/settings/DataSourceEditorDialog.vue` 中已经明确写入两个数据库型连接器定义：

1. `mysql`
2. `postgresql`

`frontend/src/i18n/locales/zh-CN.ts` 里也写明了“当前支持 MySQL 和 PostgreSQL”。

因此，当前系统的真实状态应描述为：

> **数据库型知识库当前正式支持 MySQL 与 PostgreSQL 两种数据库源。**

---

### 3.2 当前能力边界

虽然链路已经存在，但当前数据库源能力有非常明确的边界：

#### 3.2.1 当前是“实时查询型”，不是“全量导入型”

数据库适配器明确拒绝同步导入接口，说明当前方案的核心模式是：

1. 只同步 Schema 元数据。
2. 不把业务表行数据抽到本地知识库。
3. 真正查数时走远程数据库实时查询。

这条路线本身是合理的，也应继续保持，不建议为了扩展多数据库而把数据库源改造成“大规模离线抽数系统”。

#### 3.2.2 当前是“只读查询型”

`sql_guard.go` 明确限制：

1. 只允许单条 `SELECT`。
2. 禁止危险函数。
3. 禁止越界表。
4. 禁止敏感列。
5. 限制超时时间与最大返回行数。

这也是合理方向。后续扩展到更多数据库时，应该继续坚持**只读、受限、可审计**，而不是开放执行任意 SQL。

#### 3.2.3 当前更偏向单连接、单库、单 schema 视角

从 `DatabaseConnectionConfig` 与 `DatabaseSchema` 的结构看，当前模型天然更适合：

1. 一个数据源绑定一个目标数据库。
2. PostgreSQL 额外支持一个 schema。
3. 返回一个主要 schema 快照。

对于 MySQL / PostgreSQL，这种模型基本成立；但对 SQL Server / Oracle 这类数据库，命名空间模型会更复杂。

## 四、当前扩展的真实阻塞点

### 4.1 数据库类型识别存在硬编码

当前不少地方并没有“只依赖连接器注册表”，而是直接写死了支持的数据库类型。

典型位置包括：

1. `internal/application/service/datasource_service.go` 中的 `isDatabaseDataSourceType()`。
2. `internal/datasource/scheduler.go` 中的 `isDatabaseDataSource()`。
3. `internal/container/container.go` 中对数据库连接器与数据库 datasource adapter 的逐个注册。
4. `internal/datasource/connector.go` 中的 `ConnectorMetadataRegistry` 仅列出 MySQL / PostgreSQL。

这类硬编码会造成一个后果：

> 新增数据库并不是“只写一个 connector”就结束，而是要在多个后端文件里补齐同类判断。

这会持续拉高维护成本，也容易漏改。

### 4.2 SQL 方言支持仍然是 MySQL / PostgreSQL 特化

这是扩展里最容易被低估的点。

`internal/types/database_schema.go` 当前只有两个 `SQLDialect`：

1. `mysql`
2. `postgresql`

`internal/application/service/sql_guard.go` 虽然大量规则是通用的，但仍有明显方言耦合：

1. 只对 MySQL 做了反引号和 `INTERVAL` 归一化。
2. 当前的行数限制语义基本围绕 `LIMIT`。
3. SQL 校验与后续执行 SQL 之间没有独立的“方言策略对象”。

这对新增数据库是关键问题：

1. SQL Server 常见语法涉及 `TOP`、`OFFSET FETCH`。
2. Oracle 常见限制语法是 `FETCH FIRST n ROWS ONLY`。
3. ClickHouse 虽然也支持 `LIMIT`，但函数、类型、系统表与普通 OLTP 库不同。

所以，新数据库扩展不能停留在“目录下多一个 connector.go”，必须把 SQL 守卫抽成**可插拔方言策略**。

### 4.3 连接配置模型承载力不足

当前 `internal/types/database_connection.go` 的配置模型非常简洁，适合 MySQL / PostgreSQL 通用场景：

1. `host`
2. `port`
3. `database`
4. `schema`
5. `ssl_mode`
6. 查询限制类设置

这对后续数据库支持存在几个不足：

1. Oracle 常见需要 `service_name` / `sid` / `tns` 一类配置。
2. SQL Server 可能需要 `instance`、`encrypt`、`trust_server_certificate` 等配置。
3. ClickHouse 可能涉及 `secure`、`compress`、`cluster`、HTTP/TCP 协议选择。
4. 证书类 TLS 配置当前只有一个 `ssl_mode`，表达能力偏弱。

因此，建议保留现有通用字段，但新增**数据库引擎专属扩展配置块**，而不是把所有数据库都强行塞进一套最小公共字段里。

### 4.4 Schema 模型对多命名空间数据库表达不够充分

当前 `DatabaseSchema` 只有：

1. `database_name`
2. `schema_name`
3. `tables`

这对 MySQL / PostgreSQL 足够，但对以下场景不够理想：

1. SQL Server 的 `database.schema.table`。
2. Oracle 的 `owner.table` / `schema.table` 视角。
3. ClickHouse 的 `database.table` 虽然简单，但系统库与业务库也需要区分。

建议后续把命名空间表达从“固定 database + schema”升级为“通用 namespace 结构”，至少增加下面两种能力中的一种：

1. 在 `TableSchema` 级别补充 `catalog` / `schema` / `owner` 字段。
2. 在 `DatabaseSchema` 中引入更通用的 `Namespaces` 结构。

否则，后面为了兼容多数据库，很容易在字符串拼接和大小写归一化上出现边界问题。

### 4.5 前端连接器定义是写死的

`DataSourceEditorDialog.vue` 当前直接写死了 MySQL / PostgreSQL 的字段定义、默认端口、描述与表单逻辑。

这意味着新增一个数据库至少要同步修改：

1. 连接器定义。
2. 默认值生成。
3. 表单字段展示。
4. 端口默认逻辑。
5. 国际化提示语。

短期这样改没问题，但如果目标是支持 4 到 6 种数据库，前端最好转向**后端驱动的数据库引擎元数据**，否则 UI 层会越来越像“数据库枚举硬编码表”。

## 五、推荐扩展目标与优先级

### 5.1 推荐目标数据库顺序

结合当前架构、实现成本和业务合理性，建议按以下顺序扩展：

#### 第一优先级：SQL Server

原因：

1. 企业内部业务库场景常见。
2. 元数据发现路径相对清晰，可通过 `sys.tables`、`sys.columns`、`sys.indexes` 等系统视图完成。
3. 读权限模型虽然比 MySQL / PostgreSQL 略复杂，但仍在可控范围内。
4. 与“数据库型知识库”的业务定位高度匹配。

#### 第二优先级：ClickHouse

原因：

1. 典型分析型数据库，天然契合“数据问答 / 报表分析 / 聚合分析”场景。
2. 读多写少、只读查询的产品模式与当前链路一致。
3. `LIMIT` 语义与现有守卫兼容度较高，改造成本通常低于 SQL Server / Oracle。

#### 第三优先级：Oracle

原因：

1. 客户需求可能存在，但驱动、连接串、权限、部署依赖都更复杂。
2. schema / owner / synonym / 大小写规则复杂度高。
3. 一旦前两个数据库的扩展骨架没有打稳，Oracle 很容易成为“例外堆积点”。

结论：**建议不要并行一次性接入 SQL Server、ClickHouse、Oracle。** 先把扩展骨架做成通用能力，再按 SQL Server、ClickHouse、Oracle 的顺序落地。

## 六、合理的扩展方案

### 6.1 总体思路

建议采用“三层改造 + 分阶段扩库”的方式：

1. **第一层：数据库引擎元数据化**
2. **第二层：SQL 方言策略化**
3. **第三层：连接配置与 Schema 模型泛化**

在这三层完成之后，再去加具体数据库连接器，扩展成本会明显下降。

---

### 6.2 方案一：数据库引擎元数据化

#### 6.2.1 目标

把当前“多处 if/switch 判断数据库类型”的模式，收敛为“数据库引擎描述符 + 注册表能力查询”。

#### 6.2.2 推荐做法

新增统一描述结构，例如：

```go
type DatabaseEngineDescriptor struct {
    Type                 string
    Dialect              types.SQLDialect
    DisplayName          string
    DefaultPort          int
    SupportsSchema       bool
    SupportsCatalog      bool
    SupportsSSLMode      bool
    SupportsReadonlyPing bool
    UIFields             []DatabaseFieldDescriptor
}
```

再由数据库连接器注册表或独立的 engine registry 提供：

1. `ListEngines()`
2. `HasEngine(type)`
3. `GetDescriptor(type)`

#### 6.2.3 要解决的问题

这样可以替换掉当前这些硬编码点：

1. `isDatabaseDataSourceType()`
2. scheduler 里的数据库类型判断
3. container 里的逐个注册说明
4. `ConnectorMetadataRegistry` 中写死的数据库条目

#### 6.2.4 直接收益

新增数据库时，系统只需要：

1. 注册一个新的 engine descriptor。
2. 注册一个新的 database connector。
3. 前端通过接口拉取 descriptor 自动渲染表单。

而不是到处补 switch。

---

### 6.3 方案二：SQL 方言策略化

#### 6.3.1 目标

把当前 `sql_guard.go` 里的“少量方言分支”提升为完整的方言策略对象。

#### 6.3.2 推荐做法

新增方言策略接口，例如：

```go
type SQLDialectPolicy interface {
    Dialect() types.SQLDialect
    NormalizeForValidation(sqlText string) string
    ValidateReadOnly(sqlText string) error
    DetectLimit(sqlText string) (limit int, ok bool)
    ApplyLimit(sqlText string, maxRows int) (string, error)
    NormalizeIdentifier(name string) string
}
```

再由 `structuredQueryService` / `sqlGuard` 使用该策略来处理：

1. 标识符引用。
2. limit 检查。
3. 兼容不同数据库的分页 / 限流语法。
4. 方言差异导致的 SQL 归一化。

#### 6.3.3 为什么这是必须项

如果不拆方言策略，后续会出现两个问题：

1. SQL Server / Oracle 特例不断塞进 `sql_guard.go`，文件复杂度会快速上升。
2. 某些数据库即使连接器接上了，也会因为行数限制、方言识别、标识符规则问题，导致查询链路无法稳定上线。

#### 6.3.4 方案建议

第一步至少新增三个策略实现：

1. `mysql_policy.go`
2. `postgres_policy.go`
3. `sqlserver_policy.go`

如果随后接 ClickHouse，再新增：

4. `clickhouse_policy.go`

Oracle 留在后续阶段再补。

---

### 6.4 方案三：连接配置与 Schema 模型泛化

#### 6.4.1 连接配置改造建议

保留当前通用字段，同时增加引擎扩展字段，例如：

```go
type DatabaseSourceSettings struct {
    Host              string         `json:"host"`
    Port              int            `json:"port"`
    Database          string         `json:"database"`
    Schema            string         `json:"schema,omitempty"`
    SSLMode           string         `json:"ssl_mode,omitempty"`
    EngineOptions     map[string]any `json:"engine_options,omitempty"`
    TableAllowlist    []string       `json:"table_allowlist,omitempty"`
    ColumnDenylist    []string       `json:"column_denylist,omitempty"`
    BusinessJoinHints []string       `json:"business_join_hints,omitempty"`
    MaxRows           int            `json:"max_rows,omitempty"`
    QueryTimeoutSec   int            `json:"query_timeout_sec,omitempty"`
    SampleRows        int            `json:"sample_rows,omitempty"`
    SchemaRefreshCron string         `json:"schema_refresh_cron,omitempty"`
}
```

示例用法：

1. SQL Server: `instance`, `encrypt`, `trust_server_certificate`
2. Oracle: `service_name`, `sid`, `connect_string`
3. ClickHouse: `secure`, `compress`, `protocol`

这样能兼容不同数据库，而不破坏当前 MySQL / PostgreSQL 的存量数据结构。

#### 6.4.2 Schema 模型改造建议

建议优先做轻量泛化，而不是一次性重构成特别复杂的层级模型。

比较稳妥的改法是：

1. 在 `TableSchema` 增加 `CatalogName`、`SchemaName` 或 `Namespace` 字段。
2. 让 `DatabaseSchema` 继续保留顶层 `DatabaseName` / `SchemaName`，用于兼容当前 API。
3. 新数据库连接器按实际命名空间填充表级字段。

这样做可以兼顾：

1. 不破坏当前前端展示与历史快照。
2. 为 SQL Server / Oracle 的多层命名空间留下空间。

---

### 6.5 方案四：前端改为后端驱动的数据库引擎定义

#### 6.5.1 现状问题

前端当前直接硬编码数据库连接器定义，短期可用，长期不适合扩数据库种类。

#### 6.5.2 推荐做法

新增后端数据库引擎描述接口，返回：

1. 引擎类型
2. 显示名称
3. 默认端口
4. 字段定义
5. 字段是否必填
6. 字段说明与占位符

前端的 `DataSourceEditorDialog.vue` 改为根据该接口动态渲染数据库引擎表单。

#### 6.5.3 收益

1. 新增数据库时前端几乎不需要再改业务逻辑。
2. 不同数据库的字段差异可以在后端统一维护。
3. 国际化和权限控制也更容易统一处理。

---

## 七、推荐实施阶段

### 7.1 第一阶段：扩展骨架整理，不新增数据库

目标：先把现有 MySQL / PostgreSQL 代码整理成“可继续扩展”的结构。

建议任务：

1. 引入数据库引擎描述符与 registry。
2. 用 registry 能力替换 `isDatabaseDataSourceType()` 与 scheduler 的硬编码判断。
3. 将 SQL 守卫拆出方言策略接口。
4. 在 `DatabaseSourceSettings` 中增加 `EngineOptions`。
5. 调整前后端接口，使前端能读取数据库引擎元数据。

这一阶段完成后，现有功能不应发生行为变化，但新增数据库的成本会明显下降。

### 7.2 第二阶段：新增 SQL Server 支持

建议任务：

1. 新增 `internal/databaseconnector/sqlserver/connector.go`。
2. 新增 `types.DatabaseTypeSQLServer` 对应的 engine descriptor。
3. 新增 `SQLDialectSQLServer` 与方言策略。
4. 补齐 schema 发现、资源枚举、读查询执行、审计、表单字段。
5. 增加连接校验与 schema/query 测试。

验收标准：

1. 能创建 SQL Server 数据库型知识库。
2. 能生成 schema 快照。
3. 能执行受限只读查询。
4. 能记录查询审计日志。

### 7.3 第三阶段：新增 ClickHouse 支持

建议任务：

1. 新增 ClickHouse connector。
2. 实现系统表元数据发现。
3. 补齐只读查询与 limit 策略。
4. 针对分析型查询场景增加更合理的默认超时与最大行数。

ClickHouse 更适合作为“数据分析型数据库源”，不建议一开始就与 OLTP 数据库使用完全相同的默认策略。

### 7.4 第四阶段：评估 Oracle 支持

只有在前三阶段稳定后再进入 Oracle。

建议在进入 Oracle 前先确认：

1. 驱动依赖是否能接受。
2. 部署环境是否允许 Oracle 客户端要求。
3. 权限模型和数据字典查询范围是否已明确。
4. 是否需要单独的运维说明文档。

## 九、测试建议

数据库源扩展不能只做单元测试，建议至少覆盖三层验证。

### 9.1 单元测试

1. engine descriptor 注册与查找。
2. 不同数据库方言的 SQL normalize / limit / 只读校验。
3. 数据库连接配置的序列化、脱敏、加密存储。
4. Schema 快照过滤逻辑。

### 9.2 集成测试

1. 创建数据库型知识库。
2. 创建对应数据库数据源。
3. 触发 schema refresh。
4. 执行只读查询。
5. 验证审计日志落库。

### 9.3 回归测试

必须确保以下能力不回退：

1. MySQL 数据库知识库创建与查询。
2. PostgreSQL 数据库知识库创建与查询。
3. 前端数据库连接表单渲染。
4. 旧数据源配置的兼容读取。

## 十、最终建议

### 10.1 推荐方案

**推荐采用“先抽象、后扩库”的渐进方案。**

最合理的实施顺序是：

1. 先整理数据库引擎元数据、SQL 方言策略、连接配置泛化。
2. 然后先接入 SQL Server。
3. 再接入 ClickHouse。
4. Oracle 作为后续阶段单独推进。

### 10.2 不推荐方案

不建议以下做法：

1. 不整理骨架，直接在现有代码上继续堆第三、第四种数据库。
2. 一次性并行接入 SQL Server、ClickHouse、Oracle。
3. 为了兼容多数据库，把当前“只读实时查询”链路改造成“大规模行数据同步”。

### 10.3 最终判断

当前代码基础是好的，说明数据库型知识库不是从零开始：

1. MySQL / PostgreSQL 已经落地。
2. 连接器、Schema、查询、审计链路都已存在。
3. 真正缺的是“扩展骨架通用化”，不是重新设计整套系统。

所以，本次审查的最终判断是：

> **WeKnora 当前已经具备继续扩展数据库源的工程基础。合理路线不是重做，而是围绕“数据库引擎元数据化 + SQL 方言策略化 + 配置与 Schema 泛化”做一次中等规模整理，然后按 SQL Server、ClickHouse、Oracle 的顺序逐步扩展。**
