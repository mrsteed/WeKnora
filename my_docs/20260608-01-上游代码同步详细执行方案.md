# 20260608-01 上游代码同步详细执行方案

## 1. 文档目标

本文档基于 `my_docs/implementation_plan.md` 细化形成，可直接作为本轮 Git 上游代码同步的执行手册使用。目标如下：

1. 将 `Tencent/WeKnora` 的最新 `upstream/main` 同步到本地仓库。
2. 保留本地 `mrsteed/WeKnora` 中已经存在的业务能力与界面定制。
3. 以一次性 `merge` 为主线完成同步，避免在大规模分叉场景下逐提交 `rebase` 带来的高冲突成本。
4. 将验证通过后的结果受控推送到 `origin/dev`。

本文档不是通用手册，而是面向本轮同步基线的专项执行方案。

## 2. 本轮同步基线

### 2.1 分支与远端约定

本轮同步固定遵循以下远端和分支约定：

```bash
origin   https://github.com/mrsteed/WeKnora.git
upstream https://github.com/Tencent/WeKnora.git
```

主开发分支约定如下：

```bash
本地工作分支: dev
目标推送分支: origin/dev
上游吸收来源: upstream/main
```

必须保证：

1. `dev` 跟踪的是 `origin/dev`，不是 `upstream/main`。
2. 默认推送目标是 `origin`，不能误推到 `upstream`。
3. 所有同步操作都在临时同步分支完成，不能直接在 `dev` 上边拉上游边解决冲突。

建议在正式执行前确认：

```bash
git remote -v
git branch -vv
git config --get branch.dev.remote
git config --get branch.dev.merge
git config --get remote.pushDefault
```

推荐结果：

```bash
branch.dev.remote=origin
branch.dev.merge=refs/heads/dev
remote.pushDefault=origin
```

## 3. 同步策略与裁决原则

### 3.1 采用 merge 的原因

本轮明确采用：

```bash
git merge upstream/main
```

而不是：

```bash
git rebase upstream/main
```

原因如下：

1. 上下游差异规模过大，逐提交 `rebase` 极可能在多个历史提交点反复冲突。
2. 本地有大量业务定制，很多冲突并不是单行代码冲突，而是整条调用链语义冲突。
3. 使用 `merge` 可以把冲突集中到一次处理窗口内，利于按模块统一裁决。
4. `merge` 更适合保留本地开发历史，并把本次同步作为可审计的单次整合事件记录下来。

### 3.2 核心裁决原则

本轮同步以“本地功能优先、上游修复补入”为总原则，具体解释如下：

1. 本地产品行为、本地业务流程、本地权限模型冲突时，以本地为准。
2. 上游纯新增能力、通用修复、依赖升级、兼容性增强，只要不破坏本地语义，应吸收。
3. 上游安全修复不能简单放弃；若与本地逻辑冲突，必须以本地代码为基底手动补入安全逻辑。
4. 对配置、依赖、国际化、版本文件等“非单边 ownership 文件”，必须做显式合并，不能整文件覆盖。

### 3.3 禁止事项

执行过程中禁止以下做法：

1. 直接在 `dev` 上执行合并并提交。
2. 直接使用 `git checkout --theirs` 或 `git checkout --ours` 批量覆盖全量冲突文件。
3. 在未建立备份锚点前开始同步。
4. 未完成验证就直接推送覆盖 `origin/dev`。
5. 在不核对远端配置的情况下执行 `git push`。

## 4. 执行前检查清单

正式同步前，必须按以下顺序完成检查。

### 4.1 校验仓库状态

```bash
git status --short --branch
git remote -v
git branch -vv
```

通过标准：

1. 当前位于本仓库根目录。
2. 远端配置正确。
3. `dev` 存在且跟踪 `origin/dev`。

### 4.2 固化工作区未提交修改

根据当前计划，工作区存在以下未提交文件：

1. `frontend/src/i18n/locales/zh-CN.ts`
2. `frontend/src/views/settings/StorageEngineSettings.vue`

先固化：

```bash
git add frontend/src/i18n/locales/zh-CN.ts frontend/src/views/settings/StorageEngineSettings.vue
git commit -m "chore: save local changes before upstream sync 20260608"
```

如果执行前额外出现其他未提交文件，必须一并处理。原则是：

1. 能明确归属本轮前置保存的，直接提交。
2. 临时实验改动不想入历史的，单独 stash，但要记录名称。

如需暂存临时实验改动：

```bash
git stash push -u -m "pre-sync-temp-20260608"
```

### 4.3 建立回退锚点

```bash
git checkout dev
git pull --ff-only origin dev
git branch backup/dev-20260608-before-sync
git tag sync-dev-20260608-pre
```

此处必须满足两个目标：

1. 给 `dev` 建立完整回退点。
2. 保证同步基线是最新的 `origin/dev`，而不是某个过期本地提交。

### 4.4 拉取最新上下游引用

```bash
git fetch origin --prune
git fetch upstream --prune
```

拉取后建议立即复核差异：

```bash
git rev-list --left-right --count dev...upstream/main
git log --oneline --decorate dev..upstream/main -n 20
git log --oneline --decorate upstream/main..dev -n 20
```

目的是确认：

1. 差异规模是否仍与实施计划接近。
2. 上游是否在同步窗口内又新增了高风险提交。
3. 本地独有提交是否仍然集中在预期业务主题。

## 5. 正式同步步骤

### 5.1 创建临时同步分支

```bash
git checkout -b sync/dev-20260608-main dev
```

要求：

1. 临时分支必须从当前最新 `dev` 切出。
2. 后续所有冲突解决、验证、补丁修复都只在该分支进行。

### 5.2 执行合并

```bash
git merge upstream/main --no-edit
```

可能出现三种结果：

1. 无冲突直接合并成功。
2. 存在冲突，进入手工裁决阶段。
3. 少量自动合并但仍需要后续人工补丁修复和验证。

### 5.3 记录冲突清单

进入冲突状态后，先不要急着修改文件，先记录全量冲突清单：

```bash
git diff --name-only --diff-filter=U
git status --short
```

建议再按主题做一次快速归类：

```bash
git diff --name-only --diff-filter=U | rg "organization|rbac|agent|share|knowledge|duckdb|export|timeline|mcp|opensearch|StorageEngine|zh-CN|go.mod|go.sum|package.json|pnpm-lock"
```

归类目的不是为了拖延处理，而是为了先判断冲突是否集中在以下高风险域：

1. 组织与权限。
2. 知识库与共享。
3. 智能体与 MCP。
4. DuckDB / OpenSearch / 检索引擎。
5. 前端设置页与国际化。

## 6. 冲突处理执行规则

### 6.1 总体处理顺序

建议严格按下列顺序解决冲突，避免前面修完、后面又被依赖改写：

1. 依赖与构建文件。
2. 后端基础设施与中间件。
3. 权限、组织、知识库、智能体主链路。
4. 配置模板与默认值。
5. 前端页面、状态管理、国际化文案。
6. 文档、版本号、示例文件。

### 6.2 文件类型裁决矩阵

| 文件类型 | 处理策略 |
|---|---|
| 本地业务逻辑文件 | 以本地实现为基底，补入上游新增字段、参数、守卫和修复 |
| 上游纯新增文件 | 直接吸收上游 |
| 安全修复相关文件 | 不能直接丢弃上游改动，需在本地逻辑中手动融合 |
| `go.mod` / `go.sum` | 保留双方必要依赖，执行依赖整理后再验证 |
| `package.json` / `pnpm-lock.yaml` | 合并依赖版本与脚本，禁止无理由回退上游升级 |
| 国际化文案文件 | 合并双方 key，不能整文件覆盖 |
| 配置样例文件 | 保留本地定制项，同时补入上游新配置项 |
| `CHANGELOG.md` / `VERSION` | 采用上游基线信息，同时保留本地需要继续维护的记录 |

### 6.3 本地优先的重点模块

以下模块出现冲突时，应优先保护本地语义：

1. 长文档输出链路。
2. 结构化数据库、DuckDB、数据库知识库接入链路。
3. 对话导出链路。
4. 组织、共享空间、成员、权限、跨租户协作链路。
5. 智能体分享、共享 Agent 访问、前端分享页。
6. 管理员能力与定制路由。
7. 登录页、知识库列表和系统设置等已定制前端页面。

### 6.4 上游必须补入的重点模块

以下模块即便与本地产生冲突，也必须做“基于本地的手工补入”：

1. SSRF 和文件访问安全限制。
2. 租户验证与权限边界补丁。
3. DuckDB 启动修复。
4. OpenSearch、新的向量存储或索引能力。
5. MCP 多传输支持。
6. 解析追踪时间线和相关前端展示。

### 6.5 单文件冲突处理建议流程

对每个冲突文件，统一按以下流程处理：

1. 先阅读冲突双方代码，识别“本地业务语义”和“上游修复语义”。
2. 如果上游只是重构位置或参数命名变化，应把本地逻辑迁移到上游新结构上。
3. 如果上游引入新字段、新 guard、新配置项，先接入结构，再补回本地行为。
4. 处理完单文件后立即 `git add`，不要把大量未暂存冲突堆在一起。

操作模板如下：

```bash
# 编辑并解决冲突
git add <resolved-file>

# 查看剩余冲突
git diff --name-only --diff-filter=U
```

### 6.6 推荐的冲突分批方式

建议按以下批次推进：

#### 批次 A：依赖和配置

优先处理：

1. `go.mod`
2. `go.sum`
3. `package.json`
4. `pnpm-lock.yaml`
5. `config/` 下配置模板
6. `docker-compose*.yml`

目标：先让后续代码层合并有可编译的依赖基础。

#### 批次 B：权限、组织、共享主链路

优先处理：

1. `internal/application/service/organization*`
2. `internal/application/service/kbshare*`
3. `internal/application/service/agent_share*`
4. `internal/handler/knowledge*`
5. `internal/middleware/auth*`
6. 所有 RBAC 相关中间件与 handler

目标：先把“最容易误伤本地业务”的链路稳定下来。

#### 批次 C：知识库、检索、数据库与文档解析

优先处理：

1. DuckDB 相关逻辑
2. OpenSearch / 向量存储相关逻辑
3. 文档解析 timeline 相关后端和前端逻辑
4. HNSW / embedding / rerank 相关适配

#### 批次 D：Agent、MCP 与前端设置

优先处理：

1. Agent 分享与可见性
2. MCP server 传输协议支持
3. 系统设置页
4. 模型编辑器、思维模式、新手引导

#### 批次 E：国际化、文档和版本尾项

优先处理：

1. `frontend/src/i18n/locales/zh-CN.ts`
2. 其他语言包
3. `README*`
4. `CHANGELOG.md`
5. `VERSION`

## 7. 合并后的验证门禁

### 7.1 完成冲突解决并提交合并结果

当所有冲突文件都已处理并暂存后，执行：

```bash
git status
git commit
```

建议提交信息使用：

```text
merge: sync upstream main into dev 20260608
```

### 7.2 后端编译验证

```bash
go build ./...
```

如果失败，优先修复以下典型问题：

1. 依赖版本变化导致的接口签名不兼容。
2. 新旧结构体字段不一致。
3. 中间件、服务接口、路由注册的参数变化。
4. 上游新增安全字段、本地调用未补齐。

### 7.3 后端聚焦测试

```bash
go test ./internal/agent/...
go test ./internal/handler/...
go test ./internal/container
```

建议先跑这三组，因为它们最容易暴露：

1. 智能体能力是否被误伤。
2. 核心 API 权限和 handler 行为是否回归。
3. 依赖注入或容器注册是否因上游改动失配。

必要时补充：

```bash
go test ./internal/application/...
go test ./internal/middleware/...
```

### 7.4 前端安装与构建验证

```bash
cd frontend
pnpm install
pnpm build
cd ..
```

前端验证重点关注：

1. 设置页是否正常编译。
2. 国际化 key 是否缺失。
3. 分享、权限、知识库列表、模型配置等页面是否存在类型错误。
4. 上游新增页面依赖是否已被本地路由和状态树接住。

### 7.5 人工冒烟检查建议

如果时间允许，建议至少完成以下人工冒烟：

1. 登录与管理员入口。
2. 知识库创建、文档上传、检索设置。
3. DuckDB 或数据库知识库相关入口。
4. 组织创建、成员管理、共享知识库。
5. 智能体分享页与共享 Agent 访问。
6. 长文档模式与导出功能入口。

## 8. 推送与主分支落地

### 8.1 先推送临时同步分支

在所有验证通过前，不要直接动 `origin/dev`，先推送临时分支：

```bash
git push -u origin sync/dev-20260608-main
```

推送后建议做两件事：

1. 远端查看提交差异和文件变更是否符合预期。
2. 如有 CI，先观察临时分支构建结果。

### 8.2 再快进更新 dev

确认临时分支结果稳定后：

```bash
git checkout dev
git pull --ff-only origin dev
git merge sync/dev-20260608-main --ff-only
git push origin dev
```

要求：

1. `dev` 只接受来自已验证同步分支的快进合并。
2. 如果 `dev` 在同步期间又有新提交进入，先重新评估是否需要把新提交并入同步分支后再落地。

## 9. 回滚方案

### 9.1 合并未提交时回滚

如果 `git merge upstream/main` 后还没提交，直接放弃本次合并：

```bash
git merge --abort
```

### 9.2 临时同步分支验证失败时回滚

如果已经在同步分支产生合并提交，但验证失败，不要污染 `dev`：

```bash
git checkout dev
git branch -D sync/dev-20260608-main
git checkout -b sync/dev-20260608-main dev
```

如果希望保留失败现场供后续排查，可不要删除分支，改为重新建新分支继续。

### 9.3 dev 已被更新后的回滚

如果 `origin/dev` 已被本次同步更新，但需要回退到同步前状态，则使用备份锚点：

```bash
git checkout dev
git reset --hard backup/dev-20260608-before-sync
git push --force-with-lease origin dev
```

执行该步骤前必须确认：

1. `origin/dev` 上没有其他人新提交需要保留。
2. 强推回滚已完成团队通知。

## 10. 验收标准

本轮同步完成必须同时满足以下条件：

1. `sync/dev-20260608-main` 已成功吸收 `upstream/main`。
2. 本地关键业务能力未丢失。
3. 上游新增重点能力和关键安全修复已经补入。
4. `go build ./...` 通过。
5. 指定的聚焦后端测试通过。
6. 前端 `pnpm build` 通过。
7. 结果已先推送到临时同步分支，再受控落地到 `origin/dev`。

## 11. 推荐执行节奏

为降低长时间冲突处理带来的误判风险，建议按以下节奏推进：

1. 先用 30 分钟完成准备、备份和合并触发。
2. 用 1 到 2 个工作段集中处理冲突，不要在处理中穿插其他需求改动。
3. 每完成一个批次就做一次局部编译或快速检查，避免最后集中暴露几十个问题。
4. 所有冲突解决完后，优先执行编译与聚焦测试，再做前端构建。
5. 临时分支推送后先观察，再更新 `dev`。

此节奏的目的，是把风险从“一次性大爆炸”拆成“可观测、可回退、可复核”的几段动作。
