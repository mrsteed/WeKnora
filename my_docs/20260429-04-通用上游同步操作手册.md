# 20260429 通用上游同步操作手册

## 1. 文档目标

本文档提供一套可重复执行的上游同步流程，用于在以下前提下完成代码同步：

1. 同步 `Tencent/WeKnora` 的最新主线代码。
2. 保证本地已有修改和业务定制不被上游代码直接覆盖。
3. 将最终同步结果提交到 `https://github.com/mrsteed/WeKnora.git` 的 `dev` 分支。

本文档适用于日常增量同步，也适用于本地分支已经长期偏离上游后的集中补齐。

---

## 2. 基本原则

要同时满足“同步上游最新代码”和“本地修改不被覆盖”，必须遵循以下原则：

1. 不直接在当前工作分支上硬拉上游代码。
2. 先把本地修改固化为提交、备份分支或 tag，确保可回退。
3. 在独立的临时同步分支上吸收 `upstream/main`。
4. 发生冲突时，以“保留上游修复 + 补回本地业务语义”为准，而不是无条件保留某一侧。
5. 验证通过后，再受控更新远端 `origin/dev`。

如果跳过备份、直接覆盖，或者在工作区脏状态下同步，都会显著放大误覆盖风险。

---

## 3. 远端约定

本流程默认使用以下远端约定：

```bash
origin   https://github.com/mrsteed/WeKnora.git
upstream https://github.com/Tencent/WeKnora.git
```

建议先检查当前仓库是否符合该约定：

```bash
git remote -v
```

如果 `origin` 或 `upstream` 未按上述配置指向，先修正远端，再继续后续步骤。

---

## 4. 同步前准备

### 4.1 切回本地主开发分支

```bash
git checkout dev
git pull --ff-only origin dev
```

目的：

1. 保证本地 `dev` 先和远端 `origin/dev` 对齐。
2. 避免后续同步把本地过期分支当成基线。

### 4.2 检查工作区是否干净

```bash
git status --short --branch
```

要求：

1. 如果工作区是 clean，可以继续。
2. 如果有未提交改动，必须先处理，再继续同步。

### 4.3 固化本地修改

如果当前有尚未提交但必须保留的本地修改，使用以下任一方式先固化：

方式一：提交到当前分支

```bash
git add .
git commit -m "chore: save local changes before upstream sync"
```

方式二：临时 stash

```bash
git stash push -u -m "pre-upstream-sync"
```

建议：

1. 能形成明确提交的改动，优先提交。
2. 只有临时实验性改动，才使用 `stash`。

原因：

真正能保证“本地修改不被覆盖”的，不是口头约束，而是先把本地状态变成 Git 可追踪对象。

---

## 5. 标准备份动作

在开始同步前，先建立明确的回退锚点：

```bash
git checkout dev
git branch backup/dev-$(date +%Y%m%d)-before-sync
git tag sync-dev-$(date +%Y%m%d)-pre
```

这一步的作用：

1. 备份分支用于快速恢复同步前的 `dev`。
2. tag 用于记录同步前的精确提交点。
3. 即使后续 `origin/dev` 被重写，也仍然可以通过备份分支完整找回。

---

## 6. 推荐同步流程

### 6.1 获取最新上游代码

```bash
git fetch origin --prune
git fetch upstream --prune
```

必要时可以查看差异规模：

```bash
git rev-list --left-right --count dev...upstream/main
git log --oneline dev..upstream/main -n 20
git log --oneline upstream/main..dev -n 20
```

这些命令用于判断：

1. 上游新增了多少提交。
2. 本地分支保留了哪些必须继续存在的能力。

### 6.2 创建临时同步分支

```bash
git checkout -b sync/dev-$(date +%Y%m%d)-main dev
```

说明：

1. 临时同步分支始终从当前 `dev` 切出。
2. 这能保证同步是以现有本地能力为基线，而不是从空白分支重新拼装。

### 6.3 优先使用 rebase 吸收上游主线

```bash
git rebase upstream/main
```

这样做的含义是：

1. 先把 `upstream/main` 作为新的历史基线。
2. 再把本地 `dev` 的提交逐个重放到最新基线之上。

这正是“同步上游最新代码，同时保留本地修改”的核心机制。

---

## 7. 冲突处理规则

`rebase` 过程中如果出现冲突，不要简单地整文件选 ours 或 theirs，而应按下面的规则处理。

### 7.1 优先保留上游的内容

以下类型通常应优先保留上游实现，再把本地逻辑补回：

1. 安全修复。
2. 超时控制、重试、容错和错误处理。
3. 初始化流程、配置结构、数据校验逻辑。
4. 上游已经修复的线上问题或已发布兼容性问题。

### 7.2 必须保留本地的内容

以下类型不能在冲突处理时被简单覆盖：

1. 已上线或正在使用的业务功能。
2. 本地 fork 独有的产品行为。
3. 与组织、租户、权限、导出、定制模型、知识库筛选等直接相关的本地扩展。
4. 已经验证通过的兼容性适配。

### 7.3 正确的合并方式

正确做法通常是：

1. 先接受上游的新接口和新控制流程。
2. 再把本地业务语义迁移到新的接口之上。
3. 最终得到“上游基线 + 本地能力”的组合结果。

而不是：

1. 直接用本地旧实现覆盖上游新实现。
2. 或者直接让上游代码把本地业务功能全部冲掉。

### 7.4 rebase 冲突处理命令

解决完冲突后执行：

```bash
git add <冲突文件>
git rebase --continue
```

如果当前这次同步需要放弃：

```bash
git rebase --abort
git checkout dev
```

---

## 8. rebase 不可控时的兜底方案

如果遇到以下情况：

1. 冲突文件过多。
2. 同一条业务链路已被上游完全改写。
3. 继续 rebase 的成本明显高于重做整合。

则改用兜底方案：

### 8.1 从最新上游新建整合分支

```bash
git checkout -b sync/dev-$(date +%Y%m%d)-rebuild upstream/main
```

### 8.2 按主题回放本地改动

```bash
git cherry-pick <commit1>
git cherry-pick <commit2>
```

建议按功能主题分批回放，例如：

1. 后端基础能力。
2. Agent 与知识库能力。
3. Data Analysis / DuckDB 适配。
4. 前端页面和多语言。
5. 导出功能与文档处理能力。

这种方式的代价更高，但可控性更强，适合大跨度同步。

---

## 9. 推荐验证步骤

同步完成后，至少要完成以下验证。

### 9.1 后端验证

优先执行与本轮热点相关的聚焦验证：

```bash
go test ./internal/agent/...
go test ./internal/handler/...
go test ./internal/container
go test ./internal/agent/tools
```

如果需要做更大范围检查：

```bash
make test
```

说明：

1. 如果某些测试依赖外部服务，需单独记录环境依赖。
2. 环境依赖失败不能直接等同于同步失败，但必须在结果说明中写清楚。

### 9.2 前端验证

```bash
cd frontend
pnpm install --frozen-lockfile
pnpm build
```

如果本机没有启用 `pnpm`：

```bash
corepack enable
pnpm install --frozen-lockfile
pnpm build
```

### 9.3 关键手工回归

至少回归以下能力：

1. 聊天流式输出。
2. 智能体编辑与保存。
3. 数据分析相关流程。
4. 对话导出功能。
5. 受本轮冲突影响的前端页面或后端接口。

## 10. 推送到 mrsteed/WeKnora 的 dev 分支

完成验证后，不建议第一步就直接改写 `origin/dev`，而应先推送临时同步分支：

```bash
git push -u origin sync/dev-$(date +%Y%m%d)-main
```

确认无误后，再把同步结果更新到目标分支 `origin/dev`。

如果当前就在同步分支上：

```bash
git push --force-with-lease origin HEAD:dev
```

或者显式指定分支名：

```bash
git push --force-with-lease origin sync/dev-$(date +%Y%m%d)-main:dev
```

之所以使用 `--force-with-lease`，而不是裸 `-f`，是因为：

1. `rebase` 会改写提交历史。
2. 需要覆盖远端 `dev` 的旧历史。
3. `--force-with-lease` 能在远端分支被其他人意外推进时提供额外保护。

这一步完成后，最终代码就会落到：

```bash
https://github.com/mrsteed/WeKnora.git 的 dev 分支
```

---

## 11. 回滚方案

### 11.1 尚未推送到 origin/dev

```bash
git rebase --abort
git checkout dev
git branch -D sync/dev-$(date +%Y%m%d)-main
```

### 11.2 已推送临时同步分支，但未覆盖 origin/dev

```bash
git push origin --delete sync/dev-$(date +%Y%m%d)-main
git checkout dev
git branch -D sync/dev-$(date +%Y%m%d)-main
```

### 11.3 已覆盖 origin/dev

使用同步前备份分支恢复：

```bash
git checkout backup/dev-$(date +%Y%m%d)-before-sync
git push --force-with-lease origin HEAD:dev
```

---

## 12. 完成判定标准

满足以下条件，才算一次同步完整结束：

1. 本地同步分支已经建立在最新 `upstream/main` 之上。
2. 本地业务修改仍然存在，并通过了关键验证。
3. 后端聚焦测试通过，或失败仅限已确认的环境依赖问题。
4. 前端构建通过。
5. 关键手工回归通过。
6. 同步结果已经成功推送到 `https://github.com/mrsteed/WeKnora.git` 的 `dev` 分支。

---

## 13. 一次性执行示例

下面是一套可直接参考的执行顺序：

```bash
git checkout dev
git pull --ff-only origin dev
git status --short --branch

git branch backup/dev-$(date +%Y%m%d)-before-sync
git tag sync-dev-$(date +%Y%m%d)-pre

git fetch origin --prune
git fetch upstream --prune

git checkout -b sync/dev-$(date +%Y%m%d)-main dev
git rebase upstream/main

# 处理冲突并完成验证后
git push -u origin sync/dev-$(date +%Y%m%d)-main
git push --force-with-lease origin HEAD:dev
```

如果同步过程严格按上述步骤执行，就可以同时满足：

1. 吸收最新上游代码。
2. 保留本地已提交的改动。
3. 将最终结果安全提交到 `mrsteed/WeKnora` 的 `dev` 分支。
