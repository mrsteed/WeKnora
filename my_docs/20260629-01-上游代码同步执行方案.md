## 3. 总体裁决原则

### 3.1 三条总要求

本轮同步始终围绕以下三条总要求执行：

1. 吸收上游最新公共能力和修复。
2. 本地业务语义、本地产品行为和本地权限设计优先。
3. 最终结果验证通过后进入 `dev`。

### 3.2 冲突裁决顺序

发生冲突时，按以下顺序判断：

1. 先判断上游改动是否属于安全修复、兼容性修复或公共基础能力。
2. 若是，则优先吸收上游结构与防护逻辑。
3. 再把本地业务行为、本地界面语义和本地权限模型补回到新结构上。
4. 如果上游默认实现会覆盖本地已落地能力，则以本地行为为准。

### 3.3 明确保留的本地能力

以下能力属于本轮同步中的保留项，不能被上游默认实现直接替换：

1. Organization 共享空间及组织成员管理。
2. 知识库共享、智能体共享、共享 Agent 停用能力。
3. 本地长文档链路与分阶段输出能力。
4. 数据库查询知识库、私有数据库集成和相关智能体能力。
5. 对话导出和本地定制前端交互。
6. 本地认证兼容、API Key 兼容和跨租户切换能力。
7. `WEKNORA_DEV_DATA_ROOT` 对开发环境数据目录的本地约定。

### 3.4 权限与 RBAC 冲突处理原则

虽然本次用户要求不再单独展开完整 RBAC 改造计划，但权限冲突仍是本轮同步的高风险区域，处理原则保持不变：

1. 上游 RBAC 结构可以吸收，但不能直接替换本地 Organization 共享空间语义。
2. 本地组织成员、共享关系、知识库归属和智能体归属控制必须保留。
3. 任何会导致历史组织成员、共享成员、API Key 或跨租户管理路径直接变成 `403` 的改动，都不能未经兼容层直接落地。
4. 如果上游要求开启强制鉴权，必须在角色回填、历史兼容和回归验证之后再启用。

## 4. 本轮同步策略

### 4.1 本轮继续采用 merge，而不是 rebase

虽然通用手册默认优先考虑 `rebase upstream/main`，但结合当前分支状态，本轮更适合继续采用 `merge`，原因如下：

1. `dev` 近历史中已经包含一次上游同步整合提交，继续使用 `merge` 与现有历史形态一致。
2. 当前差异规模为 `63 / 158`，已明显超出轻量级逐提交重放场景。
3. 本地保留项分布在权限、前端、长文档、数据库和开发运维路径，多数冲突不是单个提交可以简单重放的线性差异。
4. `merge` 可以把冲突集中在一次整合窗口内，便于按模块统一裁决。

因此本轮主路径明确为：

```bash
git checkout -b sync/dev-20260629-main dev
git merge upstream/main --no-ff --no-edit
```

### 4.2 兜底路径

如果 `merge` 冲突范围超出可控阈值，例如：

1. 核心权限链路在多层文件上同时大面积改写。
2. 前后端同一业务链路被上游整体重构。
3. 合并后编译恢复成本明显高于重建整合分支。

则立即切换到兜底路径：

```bash
git merge --abort
git checkout -b sync/dev-20260629-rebuild upstream/main
```

随后按主题回放本地能力，而不是机械按时间顺序逐个 `cherry-pick`。

## 5. 可执行同步步骤

### 5.1 修正远端配置

先把当前仓库补齐到标准远端布局：

```bash
git remote -v
git branch -vv
git config --get branch.dev.remote
git config --get branch.dev.merge
git config --get remote.pushDefault
```

如果 `upstream` 不存在，则执行：

```bash
git remote add upstream https://github.com/Tencent/WeKnora.git
git config remote.pushDefault origin
git branch --set-upstream-to=origin/dev dev
```

如果 `upstream` 已存在但 URL 错误，则执行：

```bash
git remote set-url upstream https://github.com/Tencent/WeKnora.git
git config remote.pushDefault origin
```

完成后预期结果：

1. `dev` 跟踪 `origin/dev`。
2. 默认推送目标为 `origin`。
3. `upstream` 只用于拉取，不用于推送。

### 5.2 冻结当前工作区

正式同步前，工作区必须 clean。

先检查：

```bash
git status --short --branch
```

建议先把当前文档和已有改动固化为提交：

```bash
git add .gitignore \
  my_docs/20260608-01-上游代码同步详细执行方案.md \
  my_docs/20260629-01-上游代码同步执行方案.md
git commit -m "docs: save sync planning changes before upstream sync 20260629"
```

如果其中包含临时实验内容，不适合直接入历史，则改用：

```bash
git stash push -u -m "pre-sync-20260629"
```

原则：

1. 能形成明确提交的改动，优先提交。
2. 临时试验性内容，才使用 `stash`。

### 5.3 建立回退锚点

```bash
git checkout dev
git pull --ff-only origin dev
git branch backup/dev-20260629-before-sync
git tag sync-dev-20260629-pre
```

这一步不能省略，作用如下：

1. 形成同步前完整回退点。
2. 避免在过期本地基线之上做同步。
3. 为后续问题排查提供精确对比点。

### 5.4 拉取最新上下游引用并复核差异

```bash
git fetch origin --prune
git fetch upstream --prune
git rev-list --left-right --count dev...upstream/main
git log --oneline --decorate dev..upstream/main -n 20
git log --oneline --decorate upstream/main..dev -n 20
```

执行结果应与本文档当前基线大体接近。如果差异数量明显变化，例如上游又出现大量新增提交，则应先更新本方案中的基线说明，再继续执行。

### 5.5 创建临时同步分支并执行合并

```bash
git checkout -b sync/dev-20260629-main dev
git merge upstream/main --no-ff --no-edit
```

说明：

1. 不直接在 `dev` 上合并。
2. 所有冲突处理、验证和补丁修复都只在同步分支完成。
3. 合并提交本身就是本轮同步的审计锚点。

### 5.6 合并后冲突处理顺序

不要一开始就对所有冲突文件做批量 `ours` 或 `theirs`，必须分层处理。

建议顺序如下：

1. `go.mod`、`go.sum`、`package.json`、锁文件和基础依赖。
2. `config`、`internal/container`、`internal/router`、迁移编号等编译级冲突。
3. `internal/middleware`、`internal/handler`、`internal/application/service` 等权限与业务链路冲突。
4. `frontend` 中权限按钮、页面路由、共享设置、组织页面和定制交互冲突。
5. `docs`、`my_docs`、国际化和低风险展示层文件。

### 5.7 权限相关冲突的特殊裁决

以下文件或模块属于本轮冲突重点，处理时必须优先保本地权限语义：

1. `internal/application/service/organization.go`
2. `internal/application/service/kbshare.go`
3. `internal/application/service/agent_share.go`
4. `internal/handler/knowledge.go`
5. `internal/middleware/auth.go`
6. 上游新增或变更的 `internal/middleware/rbac.go`
7. 上游新增或变更的 `internal/middleware/kb_access.go`
8. 上游新增或变更的 `internal/router/rbac.go`
9. Organization、共享设置、权限按钮相关前端页面

处理规则：

1. 上游公共守卫、审计、角色枚举和安全修复可以吸收。
2. 本地 Organization 共享空间、成员角色、共享权限计算和共享 Agent 访问路径必须保留。
3. 若上游同租户 RBAC 与本地共享链路冲突，先保持本地可用，再逐步兼容。

### 5.8 `merge` 不可控时切换重建分支

若出现以下任一情况，即可放弃当前合并窗口并改走重建路径：

1. 同一业务链路需要在 10 个以上核心文件里同时做大规模人工拼接。
2. 合并后无法在合理成本内恢复后端编译。
3. 多个权限路径互相覆盖，难以确认哪一侧才是真正控制点。

切换命令：

```bash
git merge --abort
git checkout -b sync/dev-20260629-rebuild upstream/main
```

重建路径按以下主题回放本地能力：

1. 认证、租户、API Key 和跨租户兼容。
2. Organization 与共享空间。
3. 知识库共享、智能体共享和共享 Agent 停用。
4. 长文档和导出链路。
5. 数据库智能体与本地数据源扩展。
6. 前端定制页面和交互。

## 7. 收口与提交流程

### 7.1 先推送同步分支

验证通过后，先推送临时同步分支供复核：

```bash
git push -u origin sync/dev-20260629-main
```

如果最终走的是重建路径，则推送对应分支：

```bash
git push -u origin sync/dev-20260629-rebuild
```

### 7.2 再更新 `dev`

原则上不直接对 `dev` 强推，优先采用可审计、可快进的更新方式：

```bash
git checkout dev
git pull --ff-only origin dev
git merge --ff-only sync/dev-20260629-main
git push origin dev
```

如果在同步期间 `origin/dev` 又发生推进，则先把同步分支与最新 `origin/dev` 再次对齐并重新验证，然后再更新 `dev`。

### 7.3 仅在明确审批后才允许历史重写

只有在团队明确确认需要重写 `dev` 历史时，才考虑：

```bash
git push --force-with-lease origin HEAD:dev
```

本轮默认不采用此路径。
