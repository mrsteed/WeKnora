# WeKnora 项目 Docker 依赖与数据挂载现状清单（排除模型 Docker）

## 1. 文档目的

本文用于固化当前代码基线下 WeKnora 项目的 Docker 依赖现状，重点记录以下内容：

- 当前由仓库直接编排的 Docker 服务。
- 每个服务对应的 compose 入口、镜像或 Dockerfile 路径。
- 当前代码里声明的数据卷、配置挂载和挂载目标路径。
- 后续如果代码更新改动了服务启动方式、profile、Dockerfile 路径或挂载路径，可按本文恢复到 2026-06-30 这份基线状态。

本文依据以下代码文件整理：

- `docker-compose.yml`
- `docker-compose.dev.yml`
- `README.md`
- `scripts/cloud-image/README.md`

## 2. 排除范围

本文排除“模型 Docker”，包括但不限于用户自行部署的 Ollama、vLLM、Xinference、llama.cpp、Embedding 服务、Rerank 服务等模型推理容器。

当前仓库代码并没有在 compose 中直接内置这些模型容器，模型侧主要通过环境变量接入外部地址，例如：

- `OLLAMA_BASE_URL=${OLLAMA_BASE_URL:-http://host.docker.internal:11434}`

因此，本文只记录项目自身和基础依赖服务，不记录外部模型容器。

## 3. 当前启动基线

### 3.1 容器化部署基线

- 主编排文件：`docker-compose.yml`
- README 当前默认启动命令：`docker compose --profile minio up -d`
- 如果把 `.env` 中的 `STORAGE_TYPE` 改回 `local`，则可以直接使用：`docker compose up -d`

默认常驻核心容器为：

- `frontend`
- `app`
- `docreader`
- `postgres`
- `redis`

`minio` 不是默认无 profile 启动的服务，但当前 README 的默认启动命令会显式带上 `--profile minio`。

### 3.2 本地开发基线

- 开发编排文件：`docker-compose.dev.yml`
- 该文件只启动基础设施和可选依赖。
- `app` 与 `frontend` 在本地开发模式下不通过该 compose 启动，而是在宿主机本地运行。

## 9. 后续代码更新后如何恢复到当前状态

如果后续更新代码影响了 Docker 启动或数据挂载，建议按下面顺序恢复：

1. 先核对服务集合。
   - `docker-compose.yml` 默认核心服务必须保持为：`frontend`、`app`、`docreader`、`postgres`、`redis`。
   - `docker-compose.dev.yml` 不应额外加入 `frontend`、`app`、`mcp`。

2. 再核对 profile 归属。
   - `minio` 当前属于 `minio`、`full`。
   - `qdrant` 当前属于 `qdrant`、`full`。
   - `milvus` 当前只属于 `milvus`。
   - `weaviate` 当前只属于 `weaviate`。
   - `doris-fe` / `doris-be` 当前只属于 `doris`。
   - `mcp` 当前只属于 `full`。

3. 再恢复 build context 和 Dockerfile 路径。
   - 尤其检查：`frontend/Dockerfile`、`docker/Dockerfile.app`、`docker/Dockerfile.docreader`、`docker/Dockerfile.sandbox`、`docker/Dockerfile.odl-hybrid`、`mcp-server/Dockerfile`。

4. 再恢复绑定挂载和命名卷映射。
   - `app` 必须保留：`data-files -> /data/files`。
   - `app` 必须保留：`docreader-tmp -> /tmp/docreader:ro`。
   - `app` 必须保留：`./config/config.yaml -> /app/config/config.yaml`。
   - `app` 必须保留：`./skills/preloaded -> /app/skills/preloaded`。
   - `docreader` 必须保留：`docreader-tmp -> /tmp/docreader`。
   - `postgres` 必须保留：`postgres-data -> /var/lib/postgresql/data`。
   - `minio` 必须保留：`minio_data -> /data`。
   - `qdrant` 必须保留：`qdrant_data -> /qdrant/storage`。
   - `milvus` 必须保留：`milvus_data -> /var/lib/milvus`。
   - `weaviate` 必须保留：`weaviate_data -> /var/lib/weaviate`。
   - `doris-fe` / `doris-be` 必须保留当前 FE/BE 数据与日志卷。
   - `langfuse-clickhouse` / `langfuse-minio` 必须保留当前数据卷。
   - `searxng-init` / `searxng` 必须保留 `searxng_config` 卷和 `./docker/searxng/settings.yml` 绑定挂载。
   - `dex` 必须保留 `./misc/dex-config.yaml -> /etc/dex/config.yaml`。

5. 特别注意当前与直觉不一致、最容易被“顺手改掉”的地方。
   - 生产 compose 的 `redis` 当前**没有**卷挂载。
   - 开发 compose 的 `redis` 当前**有**卷挂载：`redis_data_dev -> /data`。
   - `milvus`、`weaviate`、`doris` 当前都**不自动属于** `full` profile。
   - README 默认命令会带 `--profile minio`，因此默认演示部署通常会把 `minio` 启起来。

## 10. 建议的恢复核对方式

若后续真的发生 compose 变更，建议把最新代码与本文逐项对比，最低限度要核对：

- 服务名是否一致。
- profile 是否一致。
- build context 和 Dockerfile 路径是否一致。
- image tag 是否被意外改动。
- volumes 的 source / target 对是否一致。
- 文件型 bind mount 的仓库相对路径是否一致。
- 文件底部 `volumes:` 声明是否仍保留对应卷名。

只要上述几项恢复一致，通常就能把 Docker 启动行为和数据挂载路径恢复到当前基线。
