# dev 分支对话问题审查与修正方案

## 审查范围

本次只做代码审查与修正方案整理，不改动任何前后端代码。

对比范围以 `main...dev` 为准，重点检查了以下链路：

- 前端聊天页消息流处理与滚动控制。
- Agent SSE 事件桥接。
- Agent 终局收口方式。
- 长文档 / 规划路由与普通 Agent 路由的分流条件。

- `frontend/src/composables/useChatStreamHandler.ts`
- `frontend/src/composables/useStickyBottomOnResize.ts`

- `internal/agent/tools/todo_write.go`

## 结论摘要

结论一：截图里的绿色三点不是额外的“神秘 loading 条”，而是前端全局 typing 指示器 `loading-typing`。它出现在 dev 的根因不是后端多发了事件，而是 `agent_query` 到首个 assistant 行真正创建之间，dev 当前聊天页没有像 main 那样立即创建占位 assistant 消息，导致全局 typing 指示器短暂暴露。

结论二：`我已经获取了完整内容……` 后停顿很久，再整段蹦出正文，核心根因在后端普通 Agent 链路。dev 相比 main 新增了 `final_answer` 终局工具收口；当模型选择这条路径时，正文往往不会像 main 那样靠自然 stop 持续流出，而是等到 `final_answer` 工具参数就绪后一次性送到前端，所以用户感知就是“长停顿 + 大段落突变出现”。

结论三：后续文字不能自动滚动，核心不是 CSS，而是聊天页丢失了 main 分支的“粘底滚动”机制。dev 当前页只会在 SSE 事件到达时执行一次 `scrollToBottom()`，但 `AgentStreamDisplay` / `botmsg` 内部又使用 typewriter、Markdown 增量渲染、Mermaid/图片后处理，DOM 高度会在事件之后继续增长。main 用 `ResizeObserver` 追随这些延迟高度变化，dev 当前聊天页没有接这条能力，所以自动滚动失效。

结论四：你看到的“带规划”并不是单一来源，当前 dev 至少存在两类“规划模式”：

- 普通 Agent 的 `todo_write` 计划卡模式。
- 长文档生成链路的 outline / planning 模式。

这两类模式的触发条件完全不同。它们之所以“有时正常”，是因为这两条链路本身更偏结构化流式输出；而普通 Agent 如果走 `final_answer` 终局工具，就更容易出现停顿和整段落屏。

## 一、问题 1：对话开始多出绿色 loading 三点，是什么过程

### 现象对应的真实代码路径

dev 当前聊天页的全局 typing 指示器在 `frontend/src/views/chat/index.vue:62` 渲染，显示条件在 `frontend/src/views/chat/index.vue:1740` 开始的 `showGlobalTypingIndicator`。

它的判断逻辑并不复杂：

- 只要 `loading.value === true`，就有机会显示。
- 只有当最后一条消息已经是“未完成的 assistant Agent 消息”时，才会隐藏这个全局三点。

因此，决定它会不会闪出来的关键，不是后端有没有 `thinking`，而是前端什么时候把 assistant 占位行放进 `messagesList`。

### dev 为什么会显示，main 为什么不明显显示

dev 当前实现中，`agent_query` 分支位于 `frontend/src/views/chat/index.vue:2613`：

- 它只保存 `assistant_message_id`。
- 如果列表里还没有对应 assistant 消息，它不会立刻创建占位消息。
- 代码反而继续保持 `loading.value = true`，见 `frontend/src/views/chat/index.vue:2622`。

这意味着在以下窗口期内：

- 用户消息已经发送。
- SSE 已经回来了 `agent_query`。
- 但第一个 `thinking / tool_call / answer` 事件还没到。

页面最后一条仍然是 user 消息，于是 `showGlobalTypingIndicator` 为 true，全局三点就暴露出来了。

真正创建 assistant 行发生在 `handleAgentChunk()`，位于 `frontend/src/views/chat/index.vue:2820`，并在 `frontend/src/views/chat/index.vue:2843` 才把 `loading.value` 关掉。也就是说，绿色三点正是 `agent_query -> 首个 assistant chunk` 这段空窗期的视觉产物。

main 分支的行为之所以更合理，不是因为没有这段空窗，而是因为主聊天页当时并没有把这套逻辑散落在 `index.vue` 里，而是走了现成的 helper：

- `frontend/src/composables/useChatStreamHandler.ts:205` 提供统一的 `shouldShowGlobalTypingIndicator`。
- `frontend/src/composables/useChatStreamHandler.ts:861-900` 在收到 `agent_query` 时，如果还没有消息，会立刻创建一个 assistant 占位消息，并马上 `loading.value = false`、`scrollToBottom(true)`。

这就是 main 里全局三点不容易暴露，而 dev 会在开始阶段看到绿色三点的直接原因。

### 结论

这不是后端新增了一个异常 loading 流程。

这是前端 dev 当前聊天页把 main 中的“`agent_query` 立刻落占位 assistant 行”能力丢了，导致全局 typing 指示器在空窗期直接暴露。

### 修正办法

前端必须恢复与 main 等价的 `agent_query` 收口策略，至少包含这三点：

- 收到 `agent_query` 时，立即创建 assistant 占位消息，而不是等到 `thinking / tool_call / answer` 才创建。
- 创建占位后立刻关闭 `loading.value`，让全局三点退出。
- 复用现有 `useChatStreamHandler.ts`，不要继续在 `index.vue` 手抄一套近似但不等价的流处理。

如果只想做最小修补，不重构到 helper，也必须把 `frontend/src/views/chat/index.vue:2613-2622` 的逻辑补齐到与 `frontend/src/composables/useChatStreamHandler.ts:861-900` 同等语义。

## 二、问题 2：获取完整内容后停顿很久，再突然蹦出大段正文

### 先给结论

这个问题的根因在后端普通 Agent 终局收口方式，而不是前端 Markdown 渲染。

main 分支的普通 Agent 终局以“自然 stop + 普通 answer 流”为主；dev 在 `internal/agent/observe.go` 新增了 `final_answer` 终局工具分支，导致模型在某些回合会把最终答案放到工具参数里一次性交回。这样一来，在最终答案真正落到前端之前，用户只会看到检索/思考过程或者一句过渡话，正文则会在终局瞬间整段出现。

### main 的终局方式

main 分支 `observe.go` 的核心思路是：

- 如果模型自然停止，且没有工具调用，就把普通文本当最终答案。
- 如果 think 阶段已经把 answer 逐步流给前端，就只补一个 done，不再重复整段正文。

换句话说，main 的默认契约是“最终答案尽量走自然文本流，不依赖额外的 `final_answer` 工具”。

### dev 新增的终局方式

dev 在 `internal/agent/observe.go:383` 新增了明确的 `Case 2: final_answer tool call present`。

这段逻辑的效果是：

- 如果本轮工具调用中出现 `final_answer`。
- 后端先解析这个工具参数里的 `answer` 字段。
- 如果此前没有 `FinalAnswerStreamed`，就会在 `internal/agent/observe.go:425` 一次性把完整 `answer` 发成一条最终答案事件。
- 随后立刻发 done。

对于前端来说，这就是典型的“前面没有正文流，最后一下整段落屏”。

### 为什么 dev 更容易走到这条路径

仅仅“支持 `final_answer` 工具”还不一定导致问题，但 dev 还额外做了两件会提高命中率的事情：

- `internal/agent/engine.go:747` 的空回答重试提示，明确告诉模型：`Please provide your answer by calling the final_answer tool.`
- `internal/agent/tools/final_answer.go` / `internal/agent/tools/definitions.go` 让这条工具路径成为正式终局能力。

也就是说，dev 不只是“兼容 `final_answer`”，而是在某些情况下主动把模型往这条路径上推。

一旦模型这样做，正文就不再像 main 那样靠自然 answer 持续流出，而是更像“等工具结果解析完成后一次性交付”。

### 为什么你会先看到“我已经获取了完整内容……”这类一句话

这句话非常像 dev 当前 `thinking` / `sequential thinking` 工具提示词鼓励模型输出的“面向用户的过程说明”。

也就是说，实际发生的更可能是：

- 检索和思考过程先持续输出。
- 模型给了一句“我已经拿到资料，可以回答了”的过渡说明。
- 但真正正文不是自然流式 answer，而是后续走 `final_answer` 工具回传。

所以用户感知是：

- 已经“看起来快要回答了”。
- 但正文还要再等一个终局工具回合。
- 最后整段一起落下。

### 为什么“带规划”的时候看起来反而更正常

如果进入的是长文档生成链路，dev 的输出方式反而是连续的。

`internal/application/service/session_agent_qa.go` 里新增了完整的 full document 路由：

- `internal/application/service/session_agent_qa.go:135-140` 决定是否进入 `knowledge_grounded_full_document` 或 `dedicated_full_document`。
- `internal/application/service/session_agent_qa.go:1394` 的 `PublishOutline()` 会先发 planning / outline。
- `internal/application/service/session_agent_qa.go:4235` 起的 `emitDedicatedFullDocumentAnswerChunk()` 会按标题、章节、段落持续发 answer chunk。
- 例如 `internal/application/service/session_agent_qa.go:5526`、`5596`、`5621`、`5771`、`5809`、`5834` 都是在逐段流式发送标题、章节和正文。

这条链路天然更接近“先规划，再一段一段生成正文”，所以用户会觉得过程更连贯。

换句话说：

- 普通 Agent + `final_answer` 工具，更容易出现长停顿后整段输出。
- full document 规划链路，本身就是分阶段、分章节持续流式，更容易显得正常。

### 修正办法

后端必须统一“普通 Agent 的最终回答契约”。建议按下面顺序处理：

- 对普通智能问答，优先恢复 main 的自然 stop 文本流作为默认终局，不要把 `final_answer` 工具作为常态终局。
- `final_answer` 工具只保留给极少数必须结构化兜底的场景，不应成为普通检索型问答的主收口方式。
- 删除或弱化 `internal/agent/engine.go:747` 这类明确提示模型“调用 final_answer 工具”的重试文案，避免把模型引向一次性交付路径。
- 如果暂时必须保留 `final_answer` 工具，也要确保这条路径的内容能增量可见，而不是只在工具参数完整后一次性发出。

单改前端解决不了这个问题。只修前端，最终也只能把“最后整段出现”滚得更平滑，不能消除“长时间没有正文”的根因。

## 三、问题 2 的后半段：后续文字不能自动滚动，需要手动向下拖

### 根因不是 `scrollToBottom()` 没写，而是只写了一半

dev 当前聊天页里当然有 `scrollToBottom()`，`frontend/src/views/chat/index.vue:2140` 定义，且消息更新后多处都会调用。

但它只覆盖了“收到 SSE 事件的那一刻”。

当前页面真正缺失的是“事件之后，内容继续长高时还能保持粘底”。

### 为什么现在会失效

当前正文显示并不是简单的 `textContent += chunk`：

- `frontend/src/views/chat/components/AgentStreamDisplay.vue:948` 使用了 `useTypewriter()`。
- `frontend/src/views/chat/components/AgentStreamDisplay.vue:981`、`995-998` 还会在答案继续展开、图片补水合、Mermaid 渲染完成后再次触发 DOM 高度变化。
- 非 Agent 路径里，`frontend/src/views/chat/components/botmsg.vue:200-287` 同样存在 typewriter 与 Markdown 二次渲染。

这意味着即使某一刻已经执行过 `scrollToBottom()`，后续几百毫秒到几秒内，正文区域仍会继续变高。

如果父级没有额外监听这些“延迟增高”，视口就会停在旧的底部位置，用户看到的就是：

- 一开始还能往下跟。
- 过一会儿越跟越慢。
- 最终正文继续长，但页面不再自动贴底。

### main 为什么正常

main 分支这块有专门补偿机制：

- `frontend/src/composables/useStickyBottomOnResize.ts:11` 定义了 `useStickyBottomOnResize()`。
- 它用 `ResizeObserver` 监听消息列表高度变化，只要用户没有主动上滑，就在高度变化后通过 `requestAnimationFrame` 再追一次底部。

同时，main 的聊天页确实接入了这条能力。当前仓库里 helper 仍然存在，但 dev 当前 `frontend/src/views/chat/index.vue` 已经没有调用它。

### 这和“大段蹦出”为什么会叠加放大

如果后端已经因为 `final_answer` 工具把正文变成“大块到达”，那前端又恰好缺少 `ResizeObserver` 粘底：

- 一次 SSE 事件会带来很大一段正文。
- `AgentStreamDisplay` 再把这一大段通过 typewriter 平滑展开。
- 展开期间高度持续增加，但父级没有持续跟随。

于是“不能自动滚动”的体感就会特别明显。

### 修正办法

前端必须恢复 main 的粘底机制，而不是只靠一次性 `scrollToBottom()`：

- 在主聊天页重新接入 `useStickyBottomOnResize()`。
- 保持“用户主动上滑就停止追底、回到底部后恢复追底”的语义。
- 对 `AgentStreamDisplay` / `botmsg` 的 typewriter 与图片补水合场景统一走 ResizeObserver，而不是在子组件里零散 emit。

这是必修项。否则即使后端恢复持续流式，图片、Mermaid、Markdown 复杂布局仍然会让自动滚动间歇性失效。

## 四、问题 3：为什么有时会进入“带规划”的对话流程

这个问题必须拆成两类，否则会误判。

### 规划模式 A：普通 Agent 的 `todo_write` 计划卡

这类规划的触发点不是前端，而是普通 Agent 工具集。

关键代码：

- `internal/application/service/agent_service.go:503-505`：如果没有知识库范围且没有 web search，会直接禁用 `todo_write`。
- `internal/application/service/agent_service.go:615`：真正注册 `todo_write` 工具。
- `internal/agent/tools/todo_write.go:14`、`22-33`：工具描述明确要求它用于复杂、多步骤、检索型任务，尤其是 3 步以上的复杂检索计划。

因此，这类“带计划”的触发条件是：

- 当前回合必须先具备 `todo_write` 工具可用性。
- 也就是通常要有知识库检索范围，或者启用了 web search。
- 模型还要判断当前任务足够复杂，值得先列计划。

它不是前端 deterministic 开关，而是“工具可用性 + 模型判断”的组合结果，所以会表现为“有时触发，有时不触发”。

### 规划模式 B：长文档生成链路的 outline / planning

这类规划不是普通 Agent 的 todo 卡，而是完整文档生成前的专用规划阶段。

触发链路如下：

- 前端普通发送时，`frontend/src/views/chat/index.vue:1492` 的 `inferDocumentOutputMode()` 对普通问题返回空字符串，不会主动把普通请求标成 `full_document`。
- 也就是说，如果一个普通输入最后进入了 full document 模式，根因不在前端，而在后端判路。
- `internal/application/service/chat_route_service.go:120` 的 `chatRouteSystemPrompt` 要求路由模型只在“明确要求完整方案、完整报告、完整标书、完整长篇 Markdown、全文翻译”等情况下，才选择 `full_document` 或 `knowledge_grounded_full_document`。
- `internal/application/service/chat_route_service.go:32`、`210-220` 还提供了正则兜底，只要 query 命中“输出/生成/撰写 + 完整方案/报告/技术方案/Markdown”等表达，也可能在路由模型失败时走 full document fallback。
- `internal/handler/session/chat_route.go:150-193` 会把路由决定真正写回请求上下文。
- 最终 `internal/application/service/session_agent_qa.go:135-140` 决定是否进入 `knowledge_grounded_full_document` 或 `dedicated_full_document`。
- `internal/application/service/session_agent_qa.go:386-469` 又进一步限制：不能有附件、图片、base artifact、auto continue；有知识范围时走 knowledge grounded，无知识范围时才走 dedicated。

因此，这类“带规划”的触发条件是：

- query 语义被 chat route 判成完整长文档生成。
- 当前回合不带附件/图片。
- 不是续写，不带 base artifact。
- 是否有知识范围决定走 `knowledge_grounded_full_document` 还是 `full_document`。

### 为什么触发原因会让人觉得“不明”

原因有三个：

- 普通输入时，前端并不显式发送 `full_document`，所以用户表面上看不到模式开关。
- 服务端 chat route 既有 LLM 路由判定，也有 regex fallback，两套机制叠加。
- 同一类问题，是否带知识库、是否带附件、是否已有 artifact、当前是否 auto continue，都会改变最终路由。

所以，当前 dev 里“规划模式”看起来像随机触发，本质上是“隐藏在后端的判路结果没有被前端透明展示”。

## 五、main 与 dev 的关键差异总结

| 维度 | main 的关键行为 | dev 当前行为 | 直接影响 |
| --- | --- | --- | --- |
| 聊天页流事件编排 | 主聊天页复用 `useChatStreamHandler`，`agent_query` 立即创建 assistant 占位 | `index.vue` 手写流逻辑，`agent_query` 仅保存 id，不立即建占位行 | 开场绿色三点暴露 |
| 粘底滚动 | 主聊天页使用 `useStickyBottomOnResize` 追随延迟高度变化 | 当前聊天页未接入该 hook | typewriter / 图片 / Markdown 增高后不再自动滚动 |
| 普通 Agent 终局 | 以自然 stop 文本流为主，无显式 `final_answer` 工具终局分支 | `observe.go` 新增 `final_answer` 工具终局 | 更容易长停顿后整段输出 |
| 长文档生成 | main 整体效果更接近普通流式回答 | dev 新增 full document 路由与 outline/planning 链路 | 有时流程更好，有时普通 Agent 路径更差，体验割裂 |
| 规划触发透明度 | 规划来源相对单一 | 同时存在 `todo_write` 计划卡和 full document planning | 用户难以判断当前到底进入了哪条链路 |

## 六、彻底修正 dev 当前问题的建议方案

### 前端必须做的事

- 恢复 main 级别的聊天流处理抽象。优先直接回用 `frontend/src/composables/useChatStreamHandler.ts`，不要继续在 `frontend/src/views/chat/index.vue` 手抄近似逻辑。
- 在 `agent_query` 阶段立即创建 assistant 占位消息，并同步关闭全局 `loading`。
- 重新接入 `frontend/src/composables/useStickyBottomOnResize.ts`，让粘底逻辑覆盖 typewriter、Markdown 重排、图片补水合、Mermaid 渲染等延迟高度变化。
- 对前端 UI 明确区分“普通 Agent 计划卡”和“长文档规划阶段”，避免用户把两者都理解成同一种“规划模式”。

### 后端必须做的事

- 收敛普通 Agent 的最终回答契约。默认回到 main 的自然 stop 文本流，不把 `final_answer` 作为常态终局。
- 若保留 `final_answer` 工具，至少不要再通过 `internal/agent/engine.go:747` 主动提示模型调用它；否则普通检索问答会越来越偏向一次性交付。
- 明确把 full document 路由、普通 Agent 路由、document edit 路由三者在返回事件里打出前端可见标记，方便排查“为什么进入规划模式”。
- 保证同一类问题在“有知识范围 / 无知识范围 / 有附件 / 无附件”之间的路由差异是可观察、可解释的，而不是只能从代码里反推。

### 推荐的修复顺序

- 第一步：前端先恢复 `agent_query` 占位消息与 `useStickyBottomOnResize`，先把绿色三点和自动滚动问题修掉。
- 第二步：后端回退或限缩普通 Agent 的 `final_answer` 工具常态使用，把普通问答重新拉回自然文本流。
- 第三步：把 chat route 决策结果、最终链路类型、是否命中 regex fallback 显式打到前端调试信息里。
- 第四步：针对“普通 Agent”“todo_write 计划卡”“full document planning”“auto continue”四条路径分别补验收用例。

## 七、验收建议

修正完成后，至少要覆盖以下场景：

- 普通 Agent 问答，有知识库，无附件，无规划卡，确认开场不再出现绿色三点，正文能尽早开始流式出现。
- 普通 Agent 问答，命中工具检索多轮后，确认不会在终局长时间静默后整段落屏。
- `todo_write` 计划卡场景，确认计划卡与正文切换自然，滚动持续贴底。
- full document planning 场景，确认 outline、章节标题、章节正文持续流式输出，且自动滚动在整轮生成过程中稳定。
- 图片、Mermaid、长 Markdown 表格场景，确认内容在后处理增高后仍然保持粘底，除非用户主动上滑。

## 最终判断

这三个现象不是三个孤立 bug，而是 dev 当前把 main 的稳定聊天页交互契约拆散后，又叠加了一套新的长文档与 `final_answer` 终局能力，结果前后端语义没有重新对齐。

要彻底解决，必须同时处理两件事：

- 前端恢复 main 的占位消息与粘底滚动能力。
- 后端收敛普通 Agent 的终局输出契约，避免普通问答频繁走 `final_answer` 一次性交付。

只修前端，会留下“长停顿后整段出现”；只修后端，会留下“绿色三点暴露”和“自动滚动失效”。两侧都要改，问题才算彻底关闭。
