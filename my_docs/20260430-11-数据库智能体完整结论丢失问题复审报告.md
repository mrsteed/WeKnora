# 数据库智能体完整结论丢失问题复审报告

## 1. 复审结论

基于新增日志再次审查后，之前那份“主要是长尾收尾慢、前端看起来像卡住”的结论不够准确。

更准确的结论是：

1. 后端确实生成了完整结论。
2. 完整结论主要存在于 `final_answer` 工具参数中。
3. 但当前 Agent 流式协议在 `tool_calls` 轮次里，先把一段不完整的 `answer` chunk 当成最终答案流给前端。
4. 随后在识别到 `final_answer` 工具后，又错误地假设“完整答案已经流过了”，因此只补了一个 `done=true` 标记，没有把真正的完整答案再次下发。
5. 最终导致前端看到的是“半成品文档 + 一直像在生成 / 或最终只保留半成品”，而不是后端已经生成的完整文档。

所以，这次问题的主故障不是“模型还没写完”，而是“完整 final_answer 被后端协议链路丢了”。

此前报告里关于“大 schema / 大上下文 / 收尾长尾”的判断并非完全错误，但它们属于放大器，不是这次“有完整结论却输出不完整”的第一根因。

## 2. 新日志给出的关键反证

## 2.1 第一组日志已经反证“完整答案已流给前端”这个前提不成立

你之前提供的数据库日志里有两个核心数字：

1. `content_len=92`
2. `final_answer tool: answer=5835 chars`

这两个数字放在一起，含义非常明确：

- 这一轮模型在普通流式 `content` 里只留下了 92 个字符。
- 但 `final_answer` 工具参数里却有 5835 字符的完整答案。

如果完整文档真的已经通过普通流式 `answer` chunk 发给前端，那么 `content_len` 不应该只有 92。

因此，这已经能说明：

- `final_answer` 工具里的 `answer` 才是权威完整结果。
- 当前协议层“完整答案已经流过了”的假设是错的。

## 2.2 my_docs/test/1.md 中的新日志把这个模式再次坐实

文件：

- [my_docs/test/1.md](my_docs/test/1.md)

这份日志里同样出现了更强的对照：

1. `content_len=360`
2. `final_answer tool: answer=7887 chars`

并且在同一轮里，日志预览内容只是：

- “All data has been collected. Let me now synthesize the final answer...”
- 以及几条概括性 bullet

这说明这一轮普通流式 `content` 只是一个总结性前言，根本不是最终完整文档。

但真正的完整答案已经被 `final_answer` 工具成功解析出来，长度 7887 字符。

因此，当前现象不是“模型卡在输出完整文档时没写完”，而是：

- 模型写完了。
- 后端把写完的结果拿到了。
- 但后端没有把这个权威结果正确下发给前端和消息持久化链路。

## 3. 真实故障链路

这次问题的完整链路，可以从三个代码点串起来。

## 3.1 第一个故障点：tool-calling 轮次中，think.go 过早把 `ResponseTypeAnswer` 当成最终答案流出去

相关代码：

- [internal/agent/think.go](internal/agent/think.go)

当前逻辑里，只要模型流里出现 `ResponseTypeAnswer`，就立刻发出 `EventAgentFinalAnswer`：

1. 不等整轮结束。
2. 不判断这轮最终是否还会出现 `tool_calls`。
3. 不判断这些 chunk 是真正 final answer，还是工具调用前的过渡性自然语言。

这在普通“自然结束”场景下可能没问题，但在数据库智能体这类会调用 `final_answer` 工具的模型上，就会出错。

因为某些模型会在调用工具前先流出一段自然语言，例如：

- “现在所有数据都已获取，让我汇总结果。”
- “The organizational structure includes...”

这些内容看起来像答案，但实际上只是工具调用前的过渡文本，不是最后那份完整 Markdown 文档。

当前实现却直接把它们作为最终答案片段推给了前端，污染了 `h.finalAnswer` 的累计结果。

## 3.2 第二个故障点：observe.go 在 `final_answer` 工具成功解析后，只补 done 标记，不补真正正文

相关代码：

- [internal/agent/observe.go](internal/agent/observe.go)

当前实现的关键假设是：

- 如果 `final_answer` 工具参数已经成功解析，说明这轮真正的答案内容之前已经通过流式 `answer chunk` 发给前端了。
- 因此这里只需要再发一个 `Done=true` 的完成标记即可。

所以代码在 `recovered == true` 时，只发了：

1. `Content: ""`
2. `Done: true`

而不会再把 `answer=5835 chars` 或 `answer=7887 chars` 的正文内容发出去。

这正是本次故障最核心的一刀。

因为从新增日志可以确认，这个前提是错的：

- 前面流出来的只有 92 字符或 360 字符。
- 真正的完整答案还躺在 `final_answer` 工具参数里。

换句话说，当前后端已经“拿到了完整答案”，却又“决定不发”。

## 3.3 第三个故障点：handleComplete 在有部分答案时拒绝用 `data.FinalAnswer` 覆盖，导致持久化内容也残缺

相关代码：

- [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)

`handleComplete` 当前做了这件事：

1. 先把 `h.finalAnswer` 写入 `assistantMessage.Content`。
2. 只有在 `h.finalAnswer == ""` 时，才用 `data.FinalAnswer` 做 fallback。

这里的问题非常关键：

- `h.finalAnswer` 并不是“权威最终答案”。
- 它只是前面通过 `handleFinalAnswer` 收到的流式 answer chunk 累积值。
- 一旦前面误流出的是半成品前言，那么 `h.finalAnswer` 就会变成“非空但不完整”。

这时 `handleComplete` 会认为：

- “既然已经有内容了，就不需要 data.FinalAnswer 了。”

于是：

1. 数据库消息内容保留的是半成品。
2. fallback answer 也不会触发。
3. complete 事件只同步状态，不同步完整正文。
4. 前端最终拿到的依旧是残缺版本。

这就是为什么用户会看到：

- 明明已经有完整结论。
- 但页面只停在前半截或骨架标题处。

## 4. 当前结论应该如何修正

此前结论需要改成下面这样：

### 4.1 主故障

数据库智能体在 `final_answer` 工具路径下，存在“完整最终答案丢失”的协议缺陷：

1. 工具调用轮次中过早流出了非权威 answer chunk。
2. 识别到 `final_answer` 后又没有把权威答案正文重新发出去。
3. complete 阶段也没有用 `data.FinalAnswer` 覆盖或回填。

因此，后端虽然完成了最终结论生成，但前端和持久化消息只保留了半成品。

### 4.2 次要放大器

以下问题仍然存在，但它们不是这次“完整结论丢失”的主因：

1. `external_database_schema` 默认全量展开，导致 prompt 很大。
2. `finalize.go` 会把工具输出再次原样回灌，拉高最后一轮耗时。
3. 前端在终态到来前只显示 loading，放大了“卡住感”。

它们解释的是“为什么这一轮很慢”，不是“为什么完整答案没显示出来”。

## 5. 修复建议

## 5.1 P0 根修：tool-calling 轮次不要立刻把 answer chunk 当成最终答案流出去

建议改造：

- [internal/agent/think.go](internal/agent/think.go)

建议策略：

1. 在单轮 `streamThinkingToEventBus` 内先缓存 `ResponseTypeAnswer` chunk。
2. 等整轮流式结束后，再判断：
   - 如果本轮没有 tool calls，且是自然结束，才把缓存内容作为最终答案流出。
   - 如果本轮出现了 `final_answer` 工具，就不要把这些缓存 chunk 当成最终答案发给前端。

这是最干净的修法，因为它直接避免了“工具调用前的过渡文本污染最终答案流”。

## 5.2 P0 根修：`final_answer` 工具成功解析后，必须下发权威正文，而不是只发 done

建议改造：

- [internal/agent/observe.go](internal/agent/observe.go)

建议逻辑：

1. 当 `ParseFinalAnswerArgs(rawArgs)` 成功时，不要再默认认为答案已经流过。
2. 应明确把解析出的 `answer` 作为权威最终答案下发。
3. 如果需要避免重复，可以配套增加协议字段，例如：
   - `authoritative: true`
   - `replace_content: true`

否则，只发 `Done=true` 的实现会持续把完整答案丢掉。

## 5.3 P0 根修：complete 阶段必须优先信任 `data.FinalAnswer`

建议改造：

- [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)

建议把当前判断：

- 只有 `h.finalAnswer == ""` 时才使用 `data.FinalAnswer`

改成：

1. 只要 `data.FinalAnswer` 非空，且与 `h.finalAnswer` 不一致，优先使用 `data.FinalAnswer`。
2. 如果 `data.FinalAnswer` 更长、更完整，也应覆盖当前 content。
3. fallback 条件不应只看“是否为空”，还应看“是否与权威 final answer 不一致”。

否则，任何“前面流过一点半成品”的情况都会让完整结论永久丢失。

## 5.4 P1：complete 事件建议直接携带可供前端回填的 final_answer

目前 complete 事件只同步状态信息，前端在 `case 'complete'` 时也只做：

1. `syncMessageCompletionState`
2. `resetReplyState`

不会回填正文。

建议：

1. 在 complete 事件数据中显式带上 `final_answer`。
2. 前端在收到 complete 时，如果发现：
   - 当前 message.content 明显短于 `final_answer`
   - 或内容不一致
   则直接替换为权威 `final_answer`。

这能作为协议兜底，避免上游遗漏时前端永远停留在半成品。

## 5.5 P2：保留此前的大上下文治理

此前报告里的这些治理仍建议继续做：

1. `external_database_schema` 默认摘要模式。
2. `external_database_query` 输出预算压缩。
3. `finalize.go` 不再原样回灌工具全文。
4. 前端增加“正在整理最终文档”阶段提示。

这些改动仍然重要，因为它们能减少最后一轮耗时，降低“看起来卡住”的概率。

但它们应被归类为性能与体验优化，而不是此次主故障的根修。

## 6. 最终判断

根据新增日志，当前最准确的判断是：

1. 系统已经生成了完整文档。
2. 完整文档主要在 `final_answer` 工具参数里。
3. 当前后端协议错误地把部分过渡性 answer chunk 当作最终答案。
4. 又在 complete 阶段拒绝用权威 `data.FinalAnswer` 覆盖它。

所以用户看到的不是“模型没写完”，而是“完整答案已经生成，但被协议链路截断并丢失了”。

这次问题如果只按“慢收尾”处理，会修偏。

正确修复方向应当是：

1. 先修 `final_answer` 权威正文下发链路。
2. 再修 complete 阶段的最终答案覆盖逻辑。
3. 最后再做数据库工具输出裁剪和前端状态优化。

## 7. 一句话结论

这不是“完整文档生成太慢导致看起来卡死”，而是“完整文档已经生成，但在 final_answer -> complete 的后端协议链路里被丢了”。
