# 长文档规划标题与输出截断问题分析及修正方案

## 一、问题背景

用户在 Agent 模式下发起请求：

> 这是一份提交给甲方的投标技术方案，业务部门和技术专家，智慧运行相关的技术方均需要详细阐述，请输出完整的技术方案。

请求参数中开启了 `agent_enabled=true`，并指定 `document_output_mode=full_document`。从日志和截图看，系统进入了“知识库支撑的完整长文档生成”链路，但最终出现三个问题：

1. 规划过程没有可见展示，只显示“已等待 xx 秒”。
2. 标题显示错误，H1 标题后直接拼接了多个 `##` 章节标题。
3. 文档只输出了很小一部分就停止，并被标记为完成。

本次分析依据主要来自 `my_docs/test/12.md` 日志、截图现象，以及当前长文档生成相关代码。

## 二、日志关键证据

### 1. 大纲阶段长期无可见内容

日志显示第一次模型调用约 39.890 秒后才出现首个可见 `delta.Content`：

```text
[LLM Stream] First delta.Content ... preview="#" ... thinking_seen=true, elapsed_ms=39890
[LLM Stream] Natural-stop ... elapsed_ms=42245
```

这说明模型在大纲规划阶段先进行了较长时间的隐藏思考或内部推理，前端无法展示真实大纲 token，只能展示后端心跳文案，例如“正在规划完整文档大纲，已等待 32 秒”。

### 2. 大纲解析后只剩一个“正文”章节

大纲阶段结束后，后续检索 query 已经变成：

```text
query text: ... 请输出完整的技术方案。 正文
```

紧接着第二次检索 query 中的“标题”变成：

```text
北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##数据湖与基础算力平台建设方案##智慧运行系统方案##智慧安防系统方案##智能安全监控应急中心方案## AR眼镜智能作业方案##证据缺口与待补充内容 正文
```

随后章节生成请求中也明确显示：

```text
Document title:
# 北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##...

Full outline:
# 北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##...

Current section heading:
## 正文
```

这证明系统没有正确识别多个 `##` 章节，而是把整串内容当成了 H1 标题；由于解析不到 H2，代码兜底生成了唯一章节 `正文`。

### 3. 单章节输出后被误判为全文完成

章节生成阶段模型请求的 `max_completion_tokens` 为 2048，模型自然停止：

```text
[LLM Stream] Natural-stop ... finish=stop ... completion_tokens=1909
```

最终持久化日志显示：

```text
completion_status: completed
finish_reason: stop
content_len: 1426
agent_steps_count: 0
Agent completion is being persisted without agent_steps
```

这说明系统认为唯一章节 `正文` 已经生成完毕，于是将整个文档标记为 `completed`。但由于原本应生成的章节树已经在大纲解析阶段丢失，后端没有再判断“用户期望的完整技术方案是否真的覆盖全部章节”。

## 三、问题一：规划过程没有显示，只显示等待时间

### 现象

截图中规划区域只显示类似：

```text
正在规划完整文档大纲，已等待 32 秒。
```

用户看不到实际规划出来的章节，也看不到检索、章节规划、章节生成等过程。

### 直接原因

1. 大纲生成阶段首个可见 token 出现在 39 秒之后，期间模型可能在输出 `reasoning_content` 或内部 thinking，普通正文 token 没有到达前端。
2. 长文档大纲生成代码会忽略 `ResponseTypeThinking`，不会把模型隐藏推理透出给用户。
3. 当前进度展示主要依赖 synthetic thought 心跳，且是同一个事件 ID 的 `Replace` 更新，所以用户看到的是一条被反复替换的等待文案。
4. 大纲内容只有在解析到标题或章节时才会更新为“已识别标题/章节”。本次大纲被模型压缩为一行，章节解析失败，因此没有形成丰富的规划进度。
5. 完成事件里没有带 `agent_steps`，持久化时 `agent_steps_count=0`，历史重载后也无法复原完整规划步骤。

### 根因判断

这不是单纯的前端渲染问题，而是“长文档专用链路”的进度事件设计不完整：规划、检索、生成过程没有形成可持久化的结构化步骤；当模型长期不输出可见 token 或大纲解析失败时，前端只能展示等待心跳。

### 修正方案

1. 后端在完整文档链路中引入结构化进度步骤：`planning`、`retrieving`、`generating`、`finalizing`。
2. `fullDocumentProgressReporter` 不只发送一条 `Replace=true` 的 thought，还应维护步骤列表，并在 `AgentCompleteData.AgentSteps` 中返回。
3. 大纲阶段在以下节点主动发送 synthetic progress：
   - 已解析用户目标。
   - 已确认知识库范围。
   - 正在检索大纲证据。
   - 已命中多少条证据。
   - 正在生成大纲。
   - 已识别标题。
   - 已识别 N 个章节。
   - 开始生成第 X/Y 章。
   - 第 X/Y 章完成。
4. 大纲阶段建议强制关闭模型 thinking：`Thinking=false`。规划任务目标是稳定输出结构化大纲，不需要长时间隐藏推理。
5. 如果仍需保留 thinking 模型能力，也要保证 8 秒左右心跳持续更新，并明确展示“正在分析文档结构/正在检索证据/正在等待模型输出大纲”。
6. 前端 `AgentStreamDisplay` 优先使用事件 metadata 中的 `stage` 展示阶段标签，历史消息优先使用 `agent_steps` 还原步骤。

## 四、问题二：标题显示有误

### 现象

页面标题显示为：

```markdown
# 北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##数据湖与基础算力平台建设方案##...
```

H1 标题和所有 H2 章节标题被拼成了一行。

### 直接原因

当前大纲解析逻辑只识别“独占一行”的 Markdown 标题：

```text
# 标题
## 章节一
## 章节二
```

但日志中的模型输出等价于：

```text
# 北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##...
```

解析器看到这一行以 `# ` 开头，就把整行作为 `outline.Title`；后面的 `##项目背景...` 没有位于新行行首，因此没有被识别为章节。

### 根因判断

根因是大纲解析器对模型输出格式过于乐观，缺少“标题粘连”容错。提示词要求“使用 Markdown headings only”，但模型仍可能省略换行；后端没有对 `# title##section` 这类常见异常格式做归一化，也没有在发现标题中包含 `##` 时触发重试或失败保护。

### 修正方案

1. 在 `parseDedicatedFullDocumentOutline` 前增加归一化函数，例如 `normalizeDedicatedFullDocumentOutlineMarkdown`。
2. 归一化规则应覆盖：
   - `# 标题##章节一##章节二` 自动拆成多行。
   - `##章节` 自动规范为 `## 章节`。
   - 多个标题连续粘连时，在每个 `##` 前插入换行。
   - 去掉代码块包裹、列表前缀、额外空白。
3. 如果解析后的 `outline.Title` 仍包含 `##`，判定大纲异常，不允许继续用这个标题进入正文生成。
4. 如果 `len(outline.Sections)==0`，不要直接兜底为 `正文`。完整文档模式下应先重试一次大纲生成，仍失败则返回 `partial/failed`，提示“大纲解析失败”，避免误生成一篇单章节短文。
5. 优化大纲提示词，增加强约束和示例：

```text
Each heading must be on its own line.
The H1 title must occupy the first line only.
Every H2 section must start on a new line with "## ".
Never concatenate headings on the same line.
```

6. 增加单元测试，至少覆盖以下样例：

```markdown
# 北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##智慧运行系统方案
```

期望解析结果：

```text
Title = 北海电厂二期智慧电厂项目投标技术方案
Sections = [项目背景与建设目标, 总体技术架构与模块划分, 智慧运行系统方案]
```

## 五、问题三：文档没有输出完，只输出很小一部分就停止

### 现象

最终输出只包含标题、`正文`，以及少量模块段落。日志中最终内容长度只有 `content_len=1426`，但状态是 `completed`。

### 直接原因

1. 大纲解析失败后，章节列表为空。
2. 代码用 `正文` 作为兜底章节。
3. 完整文档生成循环只遍历 `sectionsToGenerate`，此时只有一个章节 `正文`。
4. 这个章节的模型调用自然结束，`finish_reason=stop`。
5. 后端认为所有计划章节已经生成完毕，于是 `completion_status=completed`，`document_generation_status=completed`。
6. 没有额外校验最终文档是否包含原始大纲中的多个 H2 章节，也没有校验“技术方案”类文档的最小结构完整性。

### 根因判断

文档短不是因为前端没有继续展示，也不是简单的 SSE 中断。日志中服务端已经自然完成并持久化。真正根因是：大纲解析失败导致章节树坍缩为单章节，后端缺少完整性校验，错误地把“单章节完成”当成“全文完成”。

### 次要影响因素

1. 章节生成请求 `max_completion_tokens=2048`，如果真的把全部内容塞进一个 `正文` 章节，即使模型正常停止，也很难输出完整投标技术方案。
2. 知识库证据每个 chunk 被截到约 1200 字，章节证据数有限。对于“智慧运行详细阐述”这种长技术方案，证据不足时模型会倾向输出摘要式内容。
3. 当前章节生成提示词要求“只写当前 section body”，但当前 section 是泛化的 `正文`，缺少明确章节目标，导致输出聚焦度不足。

### 修正方案

1. 完整文档模式下禁止把大纲空章节直接兜底为 `正文`。
2. 建立最终完整性检查：
   - `completedSections` 必须覆盖 `outline.Sections`。
   - 最终内容必须包含 H1 和所有计划 H2。
   - 标题不得包含 `##`。
   - 如果任一章节生成失败或流关闭未收到 `Done=true`，状态必须是 `partial` 或 `continuing`。
3. 如果模型返回 `finish_reason=length`、流关闭、章节内容为空、或章节数量不完整，应设置：

```text
completion_status = partial
document_generation_status = continuing
auto_continue_next = true
finish_reason = section_generation_truncated / outline_or_section_incomplete
```

4. 对技术方案类长文档，建议按章节逐段生成，每章独立 token budget，而不是让一个 `正文` 承担全部内容。
5. 对知识库支撑的技术方案，每个章节检索时可适度提高证据数量，例如从 4 条提升到 6-8 条，并优先保留与章节标题强相关的 chunk。
6. 当证据不足时，应在对应章节输出“证据缺口/待补充资料”，但不能把全文标记为已经完整交付。

## 六、建议修改点

### 后端修改点

1. `internal/application/service/session_agent_qa.go`
   - 增强 `parseDedicatedFullDocumentOutline` 的容错能力。
   - 增加大纲归一化函数。
   - 大纲异常时重试或失败，不直接降级为单个 `正文` 章节。
   - `runKnowledgeGroundedFullDocumentGenerationPath` 和 `runDedicatedFullDocumentGenerationPath` 在 finalize 前执行完整性校验。
   - `emitFullDocumentCompletion` 支持携带结构化 `AgentSteps`。
   - `fullDocumentProgressReporter` 维护进度步骤列表，并将 stage、summary、duration 写入 complete 事件。

2. `internal/handler/session/agent_stream_handler.go`
   - complete 事件中保留并转发 `agent_steps`。
   - 若长文档完成事件没有 `agent_steps`，至少从 synthetic thought 事件回填步骤，避免 `agent_steps_count=0`。

3. `internal/handler/session/qa.go`
   - 对完整文档 `completed` 做二次保护：当 artifact 或 completion metadata 显示结构不完整时，不允许持久化为 completed。
   - 保留已有的 `document_generation_status`、`auto_continue_next`，确保前端可自动续写。

4. `config/prompt_templates` 或长文档专用 prompt 构造函数
   - 明确要求每个 Markdown heading 独占一行。
   - 增加输出示例。
   - 对大纲阶段关闭或弱化 thinking，减少 30-40 秒无可见输出。

### 前端修改点

1. `frontend/src/views/chat/components/AgentStreamDisplay.vue`
   - 对 synthetic progress 按 `stage` 展示“规划中/检索中/生成中/收尾中”。
   - 当 complete 事件包含 `agent_steps` 时，用它恢复历史步骤。
   - 不要只显示“已完成 1 个步骤”，应显示长文档阶段摘要，例如“已规划 8 个章节，正在生成第 2/8 章”。

2. `frontend/src/views/chat/index.vue`
   - complete 事件中若存在 `chat_document_artifact.content_snapshot` 或 `final_document`，优先作为最终文档内容。
   - 当 `document_generation_status=continuing` 或 `auto_continue_next=true` 时，保持自动续写调度，不把 UI 误标为最终完成。

## 七、优先级修复计划

### P0：阻断错误完成

1. 先修复大纲归一化与解析失败保护。
   - 在进入 `parseDedicatedFullDocumentOutline` 前，先把 `# 标题##章节一##章节二` 这类粘连格式拆成标准多行 Markdown。
   - 若归一化后 `outline.Title` 仍包含 `##`，或解析不到有效 H2，直接判定为 `outline_parse_failed`，不得进入正文生成。
2. 禁止完整文档模式降级为单个 `正文` 章节。
   - `full_document` 链路下，`len(outline.Sections)==0` 时不再兜底写入 `正文`。
   - 大纲异常时应优先重试一次；重试仍失败则以 `partial/failed` 结束，并向前端返回明确失败原因。
3. 在完成事件发出前增加“结构完整性闸门”。
   - 只有当 `completedSections` 覆盖 `outline.Sections`、最终内容包含全部计划 H2、且标题不含 `##` 时，才允许发出 `document_generation_status=completed`。
   - 只要缺章、章节流截断、章节正文为空、或实际输出结构与大纲不一致，一律降级为 `partial + continuing`，禁止误报完成。
4. 在消息持久化层再加一道 completed 保护。
   - `qa.go` / `agent_stream_handler.go` 在写入 `completion_status=completed` 前，要复核完整文档元数据是否满足完成条件。
   - 若上游误传了 `completed`，但 artifact 或 completion metadata 仍显示结构不完整，持久化层必须拦截并改写为非 completed，避免错误状态进入历史消息和前端回放。

### P1：恢复完整输出能力

1. 每个章节独立生成，确保 `sectionsToGenerate` 覆盖全部 H2。
2. 章节失败或 token 截断时返回 `partial + continuing`。
3. 前端根据 `auto_continue_next` 自动续写后续章节。
4. 适度提高章节证据数量和 token budget。

### P2：优化规划过程可见性

1. 完整记录 planning/retrieving/generating/finalizing 阶段。
2. complete 事件携带 `agent_steps`。
3. 历史消息重载后能看到规划与生成过程。
4. 前端展示章节进度和阶段 badge。

```markdown
# 北海电厂二期智慧电厂项目投标技术方案##项目背景与建设目标##总体技术架构与模块划分##数据湖与基础算力平台建设方案##智慧运行系统方案##智慧安防系统方案##智能安全监控应急中心方案## AR眼镜智能作业方案##证据缺口与待补充内容
```

期望：

```text
Title = 北海电厂二期智慧电厂项目投标技术方案
Sections count = 8
Title 不包含 ##
Sections 不包含空字符串
```

### 2. 完整文档生成单元测试

模拟大纲模型返回粘连标题，断言：

1. 系统能拆出多个章节。
2. 生成循环会遍历所有章节。
3. final answer 包含所有 `##` 标题。
4. `document_generation_status=completed` 只在全部章节完成后出现。

### 3. 异常大纲保护测试

模拟模型只返回普通文本或无法解析的大纲，断言：

1. 不会生成 `## 正文` 单章节短文。
2. 返回 `partial/failed`。
3. `finish_reason=outline_parse_failed`。
4. 前端展示可理解的失败原因。

### 4. 进度展示测试

断言 SSE 至少包含以下阶段事件：

```text
planning -> retrieving -> generating -> finalizing
```

并且 complete payload 中 `agent_steps_count > 0`。

### 5. 端到端复测

使用同一用户请求复测，预期页面表现：

1. 8 秒内出现明确规划进度，而不是只显示等待时间。
2. 标题仅为 `北海电厂二期智慧电厂项目投标技术方案`。
3. 下方依次出现项目背景、总体架构、数据湖、智慧运行、智慧安防、应急中心、AR 眼镜、证据缺口等章节。
4. 如果单轮无法写完，状态显示“继续生成中”，并触发自动续写，而不是显示“已完成”。
5. 日志中不再出现 `Agent completion is being persisted without agent_steps`。

## 九、结论

三个问题是同一条链路上的连锁故障：

```text
大纲阶段长时间无可见 token
  -> 只能显示等待心跳
模型输出粘连 Markdown 标题
  -> 后端解析器把全部章节吞进 H1 标题
章节列表为空后兜底为“正文”
  -> 只生成一个泛化章节
缺少全文完整性校验
  -> 单章节 stop 被误判为完整文档 completed
```

最关键的修复是：增强大纲解析与异常保护，禁止完整文档降级成单个 `正文` 章节，并在完成前校验计划章节是否全部生成。进度展示和 `agent_steps` 持久化属于体验与可追踪性修复，应与根因修复同步落地。
