# 人员管理编辑登录密码功能设计方案

## 1. 背景与目标

当前系统已经具备以下能力：

1. 人员管理页可在组织节点下创建用户、编辑用户基础资料、设置组织管理员、移除成员。
2. 当前登录用户可通过 `/api/v1/auth/change-password` 修改自己的登录密码。
3. 系统尚未提供“组织管理员在人员管理页直接修改某个成员登录密码”的能力。

本次需求是在“人员管理”功能中增加“编辑人员登录密码”能力，满足组织管理员或上级可管辖管理员在不掌握用户旧密码的前提下，为指定成员重置新的登录密码。

本设计文档只覆盖前后端代码设计，不直接修改业务代码。

## 2. 现状梳理

### 2.1 前端现状

当前人员管理页位于以下位置：

1. `frontend/src/views/admin/MemberManage.vue`
2. `frontend/src/views/admin/components/MemberTable.vue`
3. `frontend/src/views/admin/components/EditUserDialog.vue`
4. `frontend/src/api/org-tree/index.ts`

现有页面行为：

1. 左侧按组织树选择组织节点。
2. 右侧展示当前组织直辖成员与继承管理员。
3. 每个成员当前支持“编辑”“设为/撤销管理员”“移除”操作。
4. “编辑”会打开 `EditUserDialog.vue`，仅允许修改用户名、邮箱、手机号、角色，不涉及密码。

### 2.2 后端现状

当前组织树与用户相关能力位于以下位置：

1. `internal/router/router.go`
2. `internal/handler/org_tree.go`
3. `internal/handler/auth.go`
4. `internal/application/service/user.go`
5. `internal/types/user.go`
6. `internal/types/interfaces/user.go`

现有相关接口：

1. `POST /api/v1/org-tree/{id}/create-user`：在组织下创建用户。
2. `PUT /api/v1/org-tree/{id}/users/{user_id}`：更新组织内用户资料。
3. `GET /api/v1/org-tree/{id}/members`：获取组织成员。
4. `POST /api/v1/auth/change-password`：当前登录用户自助改密，需要提供旧密码。

现有密码处理方式：

1. 密码以 `bcrypt` 哈希形式存储在 `types.User.PasswordHash`。
2. `userService.ChangePassword` 仅支持用户本人改密，要求校验旧密码。
3. 当前不存在管理员绕过旧密码校验、直接为目标用户设置新密码的服务接口。

## 3. 设计原则

### 3.1 功能原则

1. 该功能属于“人员管理”场景，应挂载在组织成员管理链路下，而不是复用“当前用户自助改密”接口。
2. 管理员修改他人密码时，不要求输入旧密码，只输入新密码和确认密码。
3. 只能修改当前组织管理范围内成员的密码，不能越权修改其他租户或无管理关系成员的密码。
4. 不在成员列表直接明文展示任何密码信息，也不返回任何密码相关数据。

### 3.2 安全原则

1. 新密码服务端统一做长度和强度校验，不能只依赖前端。
2. 日志中不能记录明文密码。
3. 不能允许管理员修改自己的密码来绕过当前“自助改密需旧密码”的安全约束。
4. 不能允许普通成员调用该接口。
5. 不能允许跨租户修改密码。

## 4. 目标方案概览

新增一条组织成员管理接口，由组织管理员在人员管理页对成员执行“重置密码”操作。

完整链路如下：

1. 管理员在 `MemberTable.vue` 点击“重置密码”。
2. 前端弹出独立的“编辑登录密码”对话框。
3. 对话框提交新密码、确认密码到新的 org-tree 接口。
4. 后端基于当前登录用户、组织节点、目标用户三者关系做权限校验。
5. 后端生成新的 bcrypt 哈希并更新目标用户 `PasswordHash`。
6. 前端提示修改成功并关闭弹窗。

## 5. 后端设计

### 5.1 路由设计

建议新增接口：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `PUT` | `/api/v1/org-tree/{id}/users/{user_id}/password` | 组织管理员修改指定成员登录密码 |

路由归属建议：

1. 挂在 `RegisterOrgTreeRoutes` 下。
2. 与 `PUT /org-tree/{id}/users/{user_id}` 并列，保持“组织下用户管理”语义一致。
3. 不放在 `/auth` 组下，避免与“当前用户自助改密”语义混淆。

建议在 `internal/router/router.go` 中新增：

```go
orgTree.PUT("/:id/users/:user_id/password", orgTreeHandler.UpdateUserPasswordInOrg)
```

### 5.2 请求与响应类型设计

建议在 `internal/types/user.go` 中新增请求结构：

```go
type UpdateUserPasswordInOrgRequest struct {
    NewPassword     string `json:"new_password" binding:"required,min=8,max=32"`
    ConfirmPassword string `json:"confirm_password" binding:"required,min=8,max=32"`
}
```

说明：

1. 复用当前系统创建用户时的长度上限 `8~32`，保持口径一致。
2. `ConfirmPassword` 主要用于接口一致性与防误操作，避免前端绕过确认逻辑。
3. 如果团队倾向于简化接口，也可只传 `new_password`，但不建议，因为服务端无法兜底确认一致性。

响应体建议保持轻量：

```json
{
  "success": true,
  "message": "User password updated successfully"
}
```

不返回用户详情，避免误导前端把该接口当成资料更新接口使用。

### 5.3 Handler 设计

建议在 `internal/handler/org_tree.go` 中新增：

```go
func (h *OrgTreeHandler) UpdateUserPasswordInOrg(c *gin.Context)
```

处理流程建议如下：

1. 从路径参数获取 `orgID`、`userID`。
2. 从上下文获取 `tenantID` 和当前登录用户。
3. 绑定请求体 `UpdateUserPasswordInOrgRequest`。
4. 校验 `new_password == confirm_password`。
5. 校验当前登录用户是否为该组织的管理员，复用现有 `h.isOrgAdminOf(...)`。
6. 查询目标用户是否存在。
7. 校验目标用户是否属于当前租户。
8. 校验目标用户是否在当前组织的可管理成员范围内。
9. 禁止修改当前登录用户本人密码，提示改用 `/auth/change-password`。
10. 调用用户服务执行密码更新。
11. 返回成功响应。

建议错误语义：

1. 参数错误：`400 Bad Request`
2. 无权限：`403 Forbidden`
3. 目标用户不存在：`404 Not Found`
4. 目标用户不在当前租户或不在当前组织可管理范围：`403 Forbidden`

### 5.4 权限校验设计

这是本功能最关键的部分。

建议满足以下权限规则：

1. 只有当前组织管理员或继承而来的上级管理员可以修改本组织成员密码。
2. 只能修改当前组织直辖成员密码，不能修改仅因别的组织关系关联的用户密码。
3. 超级管理员若本身可访问该页面，可继续沿用组织管理权限模型，不单独开后门。
4. 当前用户不能通过该接口修改自己密码，必须走自助改密接口。

推荐实现方式：

1. 先用现有 `isOrgAdminOf` 判断当前操作者是否具备组织管理权限。
2. 再新增一个组织成员归属校验，例如复用或补充 `orgTreeService` 查询，确认目标用户是该组织的直辖成员。
3. 如果当前后端没有现成“校验用户是否在组织内”的服务接口，建议在 `orgTreeService` 增加只读判断方法，而不是在 handler 中直接拼 repository 逻辑。

建议新增服务方法之一：

```go
IsDirectMember(ctx context.Context, orgID string, tenantID uint64, userID string) (bool, error)
```

如果当前服务层已有获取成员列表能力，也可以先复用，但从长期维护看，提供明确的布尔型校验方法更清晰。

### 5.5 UserService 设计

当前 `userService.ChangePassword` 的职责是“用户本人改密并校验旧密码”。

不建议直接改造该方法为可选旧密码模式，否则会让接口语义混乱。建议新增独立能力：

```go
func (s *userService) AdminSetPassword(ctx context.Context, userID string, newPassword string) error
```

并在接口定义 `internal/types/interfaces/user.go` 中追加：

```go
AdminSetPassword(ctx context.Context, userID string, newPassword string) error
```

该方法职责：

1. 根据 `userID` 查询用户。
2. 对 `newPassword` 做服务端规则校验。
3. 生成新的 `bcrypt` 哈希。
4. 更新 `PasswordHash` 与 `UpdatedAt`。
5. 持久化到 `userRepo.UpdateUser`。

这样可以把两类密码修改能力彻底分开：

1. `ChangePassword`：本人改密，必须提供旧密码。
2. `AdminSetPassword`：管理员改他人密码，不验证旧密码。

### 5.6 密码校验规则设计

建议后端统一新增一个小型校验函数，供“创建用户”“管理员重置密码”“用户自助改密”共用，避免三处规则漂移。

建议规则：

1. 长度 8 到 32 位。
2. 至少包含字母和数字。
3. 不允许全空格。
4. 建议拒绝与用户名完全相同。

建议提炼为：

```go
func validateLoginPassword(password string, username string) error
```

优点：

1. 未来可在创建用户和当前用户改密中一并复用。
2. 降低“新建用户允许弱密码、编辑密码不允许弱密码”这种不一致问题。

### 5.7 会话与令牌影响

本次需求的最低可行版本建议只更新密码，不强制踢下线已有会话。

原因：

1. 当前用户服务已经区分 access token 与 refresh token，但本需求没有现成的“按用户批量吊销全部 token”对外业务动作。
2. 如果本次连带加入“重置密码后强制目标用户重新登录”，会扩大范围到鉴权与 token 生命周期策略。

因此本期建议：

1. 文档明确“重置后对新登录生效”。
2. 不在本期强制清理已有 token。

可选增强项：

1. 后续增加“管理员重置密码后吊销目标用户全部 token”的能力。
2. 若后续要做，可在 `AuthTokenRepository` 层增加按 `userID` 批量 revoke 的业务调用。

## 6. 前端设计

### 6.1 API 层设计

建议在 `frontend/src/api/org-tree/index.ts` 中新增：

```ts
export interface UpdateUserPasswordInOrgRequest {
  new_password: string
  confirm_password: string
}

export async function updateUserPasswordInOrg(
  orgId: string,
  userId: string,
  data: UpdateUserPasswordInOrgRequest,
): Promise<ApiResponse<null>>
```

请求路径：

```ts
put(`/api/v1/org-tree/${orgId}/users/${userId}/password`, data)
```

### 6.2 人员列表操作入口设计

建议在 `frontend/src/views/admin/components/MemberTable.vue` 的操作区增加一个按钮：

1. 按钮文案：`重置密码`
2. 位置：放在“编辑”和“设为管理员/撤销管理员”之间，保证操作心智连贯
3. 仅对可操作成员显示
4. 对当前登录用户本人禁用，并给出提示“请在个人设置中修改自己的密码”

建议新增事件：

```ts
(e: 'resetPassword', member: OrgMember): void
```

### 6.3 页面编排设计

建议在 `frontend/src/views/admin/MemberManage.vue` 中新增：

1. `showResetPasswordDialog`
2. `passwordEditingUser`
3. `handleResetPassword(member)`
4. `handleResetPasswordSuccess()`

并挂载新的密码编辑弹窗组件。

### 6.4 弹窗组件设计

建议新增组件：

1. `frontend/src/views/admin/components/ResetUserPasswordDialog.vue`

不建议把密码编辑混入现有 `EditUserDialog.vue`，原因如下：

1. 资料编辑与密码编辑属于两个风险等级不同的动作。
2. 资料编辑是普通表单提交，密码编辑需要更严格校验与更明确提醒。
3. 独立弹窗更容易控制字段清空、关闭即销毁、禁用浏览器自动回填等行为。

弹窗字段建议：

1. 用户名，只读展示。
2. 新密码。
3. 确认新密码。
4. 风险提示文案，例如“修改后该成员需使用新密码登录”。

前端校验建议：

1. 必填。
2. 长度 8 到 32 位。
3. 至少包含字母和数字。
4. 两次输入一致。
5. 弹窗关闭时清空密码字段。

交互建议：

1. 使用 `type="password"`。
2. 增加 `autocomplete="new-password"`，避免浏览器误填旧密码。
3. 成功后只提示成功，不在界面中再次展示新密码。

## 8. 兼容性与风险分析

### 8.1 兼容性

该方案为增量改动，不影响现有接口：

1. 不修改当前 `/auth/change-password` 行为。
2. 不修改现有创建用户与编辑资料接口的数据结构。
3. 人员管理页原有编辑、移除、设管理员能力保持不变。

### 8.2 风险点

主要风险如下：

1. 若只校验“操作者是组织管理员”，但不校验“目标用户属于该组织”，可能造成越权改密。
2. 若允许修改自己密码，会绕开旧密码校验策略。
3. 若前端把密码字段混入资料编辑弹窗，容易造成误提交和状态污染。
4. 若后端未统一密码规则，可能出现创建用户与重置密码口径不一致。
5. 若日志框架打印原始请求体，需要再次确认密码字段脱敏规则是否覆盖 `new_password` 和 `confirm_password`。

### 8.3 日志脱敏建议

当前中间件已对 `password` 字段做脱敏，但建议补充确认是否同时覆盖：

1. `new_password`
2. `confirm_password`
3. `old_password`

如果未覆盖，建议在日志脱敏规则中一并补齐，避免密码以新字段名泄漏。

## 9. 测试设计

### 9.1 后端测试点

建议至少覆盖以下测试：

1. 组织管理员可成功修改直辖成员密码。
2. 非组织管理员调用接口返回 403。
3. 修改不属于当前组织的用户密码返回 403。
4. 修改跨租户用户密码返回 403。
5. 修改本人密码返回 400 或 403，并提示走自助改密入口。
6. 两次密码不一致返回 400。
7. 弱密码返回 400。
8. 更新后目标用户可用新密码登录。
9. 更新后旧密码登录失败。

### 9.2 前端测试点

建议至少覆盖以下测试：

1. 成员列表出现“重置密码”按钮。
2. 点击后弹出独立对话框。
3. 当前用户本人按钮禁用。
4. 两次密码不一致时前端阻止提交。
5. 提交成功后显示成功提示并关闭弹窗。
6. 弹窗关闭后重新打开，密码字段已清空。
7. 接口失败时正确展示后端错误消息。

## 10. 实施顺序建议

建议按以下顺序落地：

1. 后端新增类型、服务、handler、路由。
2. 后端补齐权限校验与密码规则校验。
3. 前端新增 API 方法。
4. 前端新增独立密码重置弹窗。
5. 前端把“重置密码”按钮挂到人员列表。
6. 联调并补齐国际化文案。
7. 最后补充回归测试。

## 11. 最终建议

本需求建议采用“组织成员独立重置密码接口 + 前端独立密码弹窗”的实现方案，不建议把该功能混入现有资料编辑接口。

原因如下：

1. 接口语义清晰，便于权限审计。
2. 前后端边界更稳定，不会污染现有编辑资料流程。
3. 更容易扩展后续能力，例如“重置密码后强制下线”或“首次登录强制改密”。

如果下一步进入实现阶段，可以直接按本设计文档落代码，改动范围清晰且对现有功能影响较小。
