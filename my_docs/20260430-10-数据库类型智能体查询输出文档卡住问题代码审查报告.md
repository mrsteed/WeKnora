# 数据库类型智能体查询输出文档卡住问题代码审查报告

## 1. 问题结论

这次现象更准确地说，不是后端真正死锁，也不是请求没有返回，而是“数据库工具返回内容过大 + Agent 收尾阶段再次放大上下文 + 前端只在终态信号到达后才收起加载态”叠加后，形成了明显的“卡住不动”观感。

从你提供的日志看，后端最终是正常完成的：

- Agent 共执行 11 轮，12 个步骤。
- 最后一轮模型返回 `finish_reason="tool_calls"`，并触发 `final_answer` 工具。
- `final_answer` 产出了 5835 字符答案。
- 整体请求在 2 分 40 秒左右结束，HTTP 状态码为 200。

也就是说，这次更接近“超长尾收尾导致前端长期处于生成中”，不是“服务没有结束”。

## 2. 现象与日志对应关系

你给出的日志有几个关键信号：

1. 数据库 schema 内容被大量写入上下文。
   日志正文已经出现超长表结构输出，包含大量表、字段、注释，截断前总长度达到 112065 字符。

2. 最后一轮 prompt 非常大。
   `prompt_tokens=32633`，说明模型在收尾阶段仍然要重新消化一大批历史工具结果，而不是只处理一个小型摘要。

3. 最后一轮本身耗时很长。
   `stream_duration=53552ms`，也就是最后一轮单独又跑了约 53.5 秒。

4. 后端已完成，但前端在完成信号到达前只能继续显示“生成中”。
   日志里已经有：
   - `final_answer tool: answer=5835 chars`
   - `Agent execution completed in 11 rounds`
   - `Stream completed ... closing immediately without waiting for title event`

这说明服务端执行链路最终闭环了，但用户在看到正文主体后，到真正收到终态事件之间，仍经历了一段较长的无明显进展窗口。

## 3. 根因分析

### 3.1 一级根因：数据库 schema 默认全量展开，导致一次工具调用就把全库结构灌进模型上下文

相关实现：

- [internal/agent/tools/external_database_schema.go](internal/agent/tools/external_database_schema.go)
- [internal/application/service/database_schema.go](internal/application/service/database_schema.go)

当前链路是：

1. `external_database_schema` 工具接收 `knowledge_base_id` 和可选的 `tables`。
2. 如果 `tables` 为空，就直接调用 `BuildPromptSchema(ctx, kbID, selectedTables)`。
3. `BuildPromptSchema` 会把命中的每张表都展开成完整 prompt 文本：
   - 数据库名
   - schema 名
   - 所有表名
   - 每张表的主键
   - 每张表的所有列
   - 每列类型、可空性、注释、敏感标记

问题在于：这里没有任何默认摘要模式，也没有字符预算、表数量预算、列数量预算。只要智能体第一次为了“先看数据库结构”调用该工具，且没有显式传 `tables`，它就会把整个数据库快照全部返回给模型。

这和你这次日志是完全对得上的。日志里出现的大段 `sys_dept / sys_personnel / sys_role / sys_role_dept / sys_role_menu ...` 就是这条路径的结果。

影响：

- 首轮 schema 探查就会把上下文迅速做大。
- 后续每一轮推理都在背着大上下文继续运行。
- 对“组织架构分析”这种本来只需要少数表的任务，也会先被迫阅读大量无关表结构。

### 3.2 二级根因：final answer 阶段把历史工具输出原样再次回灌给模型，放大了第二次上下文成本

相关实现：

- [internal/agent/finalize.go](internal/agent/finalize.go)

`streamFinalAnswerToEventBus` 在生成最终答案时，会重新构造一组 messages，并把每个工具调用结果按下面的形式追加进去：

`Tool <name> returned: <toolCall.Result.Output>`

这里的问题是，回灌的是工具的文本输出原文，而不是压缩后的结构摘要。

对于数据库工具，这意味着：

- schema 工具返回的大段表结构文本，会再次被拼进最终回答 prompt。
- query 工具返回的逐行逐列明细，也会再次被拼进去。

这就是为什么你已经看到答案主体在前端开始出现了，但最后一轮仍然能消耗 32633 个 prompt token，并继续跑 53 秒以上。Agent 不是没结束，而是在做一次“超重的最终整理”。

影响：

- 数据库工具的体积问题被放大两次。
- 越是长文档式回答，越容易在“正文看起来已经差不多了”之后，继续卡在最终收尾。

### 3.3 三级根因：数据库查询结果的 LLM 可见文本也偏冗长，缺少面向 Agent 的输出预算

相关实现：

- [internal/agent/tools/external_database_query.go](internal/agent/tools/external_database_query.go)
- [internal/application/service/structured_query.go](internal/application/service/structured_query.go)

当前查询工具虽然有 `LIMIT` / `MaxRows` 约束，但它给 LLM 的文本输出格式仍然比较重：

- 先打印返回行数与耗时。
- 然后按 `Record #1 / #2 / #3 ...` 逐条展开。
- 每条记录再把每个字段完整打印一遍。

这对于“让前端展示结果”没问题，但对于“再喂给模型做多轮推理和最终成文”并不经济。

也就是说，当前实现更像“面向人类终端直接阅读的 verbose 文本”，而不是“面向 Agent 二次推理的 compact 上下文”。

当用户问题要求“输出成一篇结构化文档”时，这种 verbose 输出会进一步拉高最终整理成本。

### 3.4 观感放大器：前端在正文开始出现后，仍要等终态信号才结束“生成中”状态

相关实现：

- [frontend/src/views/chat/components/AgentStreamDisplay.vue](frontend/src/views/chat/components/AgentStreamDisplay.vue)
- [frontend/src/views/chat/index.vue](frontend/src/views/chat/index.vue)
- [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)
- [internal/handler/session/stream.go](internal/handler/session/stream.go)

前端的状态逻辑是：

1. 只要 `answer` 开始流出，就会把正文展示出来。
2. 但对话是否“完成”，仍依赖终态：
   - `completion_status` 进入 `completed / partial / failed / cancelled`
   - 或收到 `stop` 事件
3. 在终态到来前，组件仍会保留流式中的视觉状态。

这个逻辑本身没错，但它会把后端最后的长尾等待直接暴露给用户。尤其当答案主体已经出来、页面长时间只剩底部三个点时，用户会非常自然地认为“卡死了”。

这不是本次问题的首要根因，但它明显放大了感知问题。

### 3.5 架构脆弱点：SSE 完成依赖 `complete` 事件单点，缺少兜底完成包

相关实现：

- [internal/handler/session/helpers.go](internal/handler/session/helpers.go)
- [internal/handler/session/agent_stream_handler.go](internal/handler/session/agent_stream_handler.go)

当前 `sendCompletionEvent` 已经被改成 no-op，流结束完全依赖 `complete` 事件本身写入成功并被轮询到。

这意味着：

- 一旦 `handleComplete` 追加 `complete` 事件失败，SSE 就没有额外兜底完成包。
- 前端也不会有第二条明确的结束信号。

这次日志里可以确认 `complete` 事件是成功的，因为后端打印了 `Stream completed ...`。所以它不是这次事故的直接触发点。

但从架构稳健性上看，它会把任何一次尾部事件写入失败直接放大成“前端一直不结束”的真卡住问题，属于必须补的薄弱点。

## 4. 为什么这次会表现为“文档输出到一半像卡住”

把这次行为串起来，就是下面这条链：

1. 用户问“请分析北下街的组织架构”。
2. 数据库智能体先看 schema。
3. schema 工具默认返回全库表结构，而不是只返回组织架构相关的表。
4. Agent 又做了多轮人员、部门、角色关联查询。
5. 最后一轮开始整理文档时，系统把前面几轮的大块工具输出再次整包送给模型。
6. 前端已经开始展示 Markdown 正文，但还没有收到终态完成事件。
7. 用户此时看到页面底部持续显示生成中，于是感知为“卡住不动”。

所以，本质不是“最后一段 Markdown 渲染失败”，而是“最后一轮的收尾成本远超用户预期”。

## 5. 修复方案

### 5.1 P0：把 `external_database_schema` 改成默认摘要模式，禁止首次全库全字段展开

建议直接修改策略，而不是只靠提示词约束模型。

建议方案：

1. 当 `tables` 为空时，默认只返回表级摘要：
   - 表名
   - 表注释
   - 主键
   - 列数量
   - 前若干个高价值字段

2. 真正的字段级全量展开，只允许在显式传入 `tables` 时进行。

3. 给 schema 输出增加硬预算：
   - 最大表数
   - 每表最大列数
   - 总字符数上限

4. 在输出末尾明确下一步引导：
   - “如果需要查看某些表的完整字段，请再次调用并传入 `tables`。”

这样做的收益最大，因为它能在根部把 11 万字符级别的上下文膨胀直接砍掉。

### 5.2 P0：final answer 阶段不要再原样拼接工具输出，改为压缩摘要回灌

建议修改 [internal/agent/finalize.go](internal/agent/finalize.go) 的收尾拼装逻辑。

建议做法：

1. 为每类工具提供单独的“最终回答摘要器”：
   - `external_database_schema` 只保留相关表、主键、连接线索、敏感字段提示。
   - `external_database_query` 只保留查询目的、SQL 摘要、行数、关键统计、前若干条样本。

2. `streamFinalAnswerToEventBus` 不再使用 `toolCall.Result.Output` 原文，而是使用压缩后的 summary。

3. 对单个工具回灌内容设置字符预算，例如 2 KB 到 4 KB，上限后自动截断并附带“已省略 N 字符”。

这是第二个必须优先做的点。否则即使 schema 首次缩小了，只要 query 结果一多，最终成文阶段还是会被拖慢。

### 5.3 P0：给数据库查询工具增加“Agent 视角输出”和“前端展示输出”双通道

当前一个 `Output` 同时承担两件事：

- 给模型继续推理
- 给前端展示

这会导致两边都不够理想。建议拆成两层：

1. `Data` 保留完整结构化数据，供前端富展示。
2. `Output` 改成面向 LLM 的紧凑摘要文本。

例如：

- 返回 62 行时，`Output` 不要逐行全打印。
- 可以输出：
  - 查询目的
  - 实际执行 SQL
  - 行数
  - 关键字段分布
  - 前 5 行样本
  - 是否截断

这样既保留展示能力，又不会把模型上下文拖爆。

### 5.4 P1：前端增加“正在整理最终文档”的显式阶段提示，避免误判为卡死

建议在 Agent 流式协议里新增一个更细的阶段信号，例如：

- `finalizing`
- `assembling_answer`
- `synthesizing_report`

前端收到该事件后，不再只显示三个点，而是显示明确状态文案，例如：

- 正在整理最终文档
- 正在汇总查询结果并生成报告

这样即使最后 30 秒到 60 秒仍在跑，用户也知道系统没有卡死，而是在收尾。

### 5.5 P1：为 `complete` 事件增加兜底发送策略，避免单点失败导致真卡住

建议补两件事：

1. `handleComplete` 追加尾事件时使用不可取消上下文，避免尾部写入被请求上下文取消干掉。
2. 保留一个兜底 completion 包，而不是把 `sendCompletionEvent` 完全置空。

可以不再发送旧版空 `answer done` 包，但至少应该保留一个明确的 transport-level completion sentinel，避免 `complete` 单点丢失后前端永远不结束。

### 5.6 P2：给数据库 Agent 加“表范围收敛”策略

这是提示词与工具协同层面的优化：

1. 首次只能做 schema 摘要探查。
2. 第二步必须先圈定候选表。
3. 第三步才允许查详细字段或跑 SQL。

例如对于“组织架构分析”，应优先收敛到：

- `sys_dept`
- `sys_personnel`
- `sys_user`
- `sys_role`
- `sys_role_dept`

而不是把 `sys_menu`、登录日志等无关表一并带入模型上下文。

## 6. 推荐落地顺序

### 第一阶段：立刻止血

1. 改 `external_database_schema` 默认输出模式，未指定 `tables` 时只返回摘要。
2. 改 `streamFinalAnswerToEventBus`，不要原样拼接大工具输出。
3. 改 `external_database_query` 的 LLM 可见文本，加入字符预算。

### 第二阶段：修复用户感知

1. 新增 `finalizing` 类事件。
2. 前端把“正在整理最终文档”展示出来。
3. 在超长收尾阶段展示已耗时和当前阶段。

### 第三阶段：补协议韧性

1. `complete` 尾事件改为不可取消上下文写入。
2. 增加 transport 层 completion 兜底信号。
3. 为完成信号缺失增加超时自恢复或自动 partial 收敛策略。

## 7. 最终判断

这次“数据库类型智能体输出文档时卡住不动”的主因，不是前端渲染卡死，也不是 SSE 没回包，而是数据库工具结果过大，导致 Agent 在最终成文阶段承担了过重的二次上下文整理成本。

前端之所以让用户感觉更明显，是因为正文已经开始显示，但真正的完成态要等最后的 `complete` 信号，期间缺少“正在整理最终文档”的状态说明。

如果只修前端提示，问题会变成“看起来没那么像卡住”，但后端长尾仍在。

如果只修后端工具裁剪，不补完成态协议和阶段提示，问题虽然会缓解，但在异常链路里仍可能表现成真正的流式挂住。

因此正确的处理方式是：

- 先从数据库工具输出裁剪和 final answer 回灌裁剪入手解决根因。
- 再补前端阶段提示改善用户感知。
- 最后补 SSE 完成协议兜底，提升尾部可靠性。

## 8. 一句话建议

这类问题不要按“前端卡住”处理，要按“数据库 Agent 上下文失控 + 收尾协议可观测性不足”处理。根修点在工具输出预算和 final answer 组装策略，不在 Markdown 渲染层。
