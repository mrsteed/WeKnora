# 开发环境数据目录挂载说明

## 目标

`make dev-start` 与 `./scripts/dev.sh start` 支持两种开发数据模式：

- 未设置 `WEKNORA_DEV_DATA_ROOT`：沿用历史 Docker named volumes。
- 显式设置 `WEKNORA_DEV_DATA_ROOT`：把持久化数据绑定到宿主机目录，推荐默认值为 `/data/weknora`。

这样做的目的有两点：

- 开发数据落盘位置固定，便于排查、备份与清理。
- 宿主机可以直接查看 PostgreSQL、Redis、MinIO、DocReader 等开发数据目录。

## 默认规则

- 默认行为：不设置 `WEKNORA_DEV_DATA_ROOT`，继续使用 Docker named volumes
- 推荐宿主机目录：`/data/weknora`
- 可覆盖环境变量：`WEKNORA_DEV_DATA_ROOT`
- `make dev-stop` 只停止容器，不会删除宿主机数据目录，也不会删除 named volume

示例：

```bash
make dev-start

make dev-start WEKNORA_DEV_DATA_ROOT=/data/weknora

WEKNORA_DEV_DATA_ROOT=/data/weknora ./scripts/dev.sh start --qdrant
```

## 目录映射

| 服务 | 容器内路径 | 宿主机路径 |
|------|------------|------------|
| PostgreSQL | `/var/lib/postgresql/data` | `${WEKNORA_DEV_DATA_ROOT}/postgres` |
| Redis | `/data` | `${WEKNORA_DEV_DATA_ROOT}/redis` |
| MinIO | `/data` | `${WEKNORA_DEV_DATA_ROOT}/minio` |
| Qdrant | `/qdrant/storage` | `${WEKNORA_DEV_DATA_ROOT}/qdrant` |
| OpenSearch | `/usr/share/opensearch/data` | `${WEKNORA_DEV_DATA_ROOT}/opensearch` |
| Milvus | `/var/lib/milvus` | `${WEKNORA_DEV_DATA_ROOT}/milvus` |
| Neo4j | `/data` | `${WEKNORA_DEV_DATA_ROOT}/neo4j` |
| DocReader | `/tmp/docreader` | `${WEKNORA_DEV_DATA_ROOT}/docreader/tmp` |
| Jaeger | `/var/lib/jaeger` | `${WEKNORA_DEV_DATA_ROOT}/jaeger` |
| SearXNG 配置 | `/etc/searxng` | `${WEKNORA_DEV_DATA_ROOT}/searxng/config` |
| Langfuse ClickHouse 数据 | `/var/lib/clickhouse` | `${WEKNORA_DEV_DATA_ROOT}/langfuse/clickhouse/data` |
| Langfuse ClickHouse 日志 | `/var/log/clickhouse-server` | `${WEKNORA_DEV_DATA_ROOT}/langfuse/clickhouse/logs` |
| Langfuse MinIO | `/data` | `${WEKNORA_DEV_DATA_ROOT}/langfuse/minio` |

说明：

- `app` 与 `frontend` 仍然是本地进程，不通过 Docker 数据卷持久化。
- 只有启用了对应 profile 的可选服务，才会实际使用对应的宿主机目录。

## 目录准备与权限

执行 `make dev-start` 或 `./scripts/dev.sh start` 时，脚本会自动创建上述目录，并对创建出的子目录执行 `chmod 0777`，以降低不同镜像用户 UID/GID 不一致导致的开发环境写入失败问题。

如果推荐目录不可写，有两种处理方式：

1. 预先给 `/data/weknora` 配置好写权限。
2. 启动时改用自定义目录，例如 `WEKNORA_DEV_DATA_ROOT=/data/weknora/my-dev make dev-start`。

如果连根目录都无法创建，脚本会直接报错并终止启动。

## 与历史 named volume 的关系

当前实现不会自动迁移旧的 Docker named volume 数据，也不会自动删除旧 volume。

如果你此前已经使用过旧版开发环境：

1. 旧 named volume 仍会保留在 Docker 本地。
2. 未设置 `WEKNORA_DEV_DATA_ROOT` 时，当前脚本仍会继续使用这些 named volume。
3. 显式设置 `WEKNORA_DEV_DATA_ROOT` 后，才会切到宿主机目录挂载。
4. 如果需要保留旧数据，请在首次启动宿主机目录模式前自行迁移。

## 快速核对

启动后可用以下方式确认挂载是否生效：

```bash
make dev-status

docker inspect WeKnora-postgres-dev --format '{{ range .Mounts }}{{ .Source }} -> {{ .Destination }}{{ println }}{{ end }}'
```

如果输出中出现 `/data/weknora/...` 或你自定义的根目录，说明宿主机目录挂载已生效；如果显示 `postgres-data-dev`、`redis_data_dev` 等 volume 名，则仍在使用历史 named volume 模式。
